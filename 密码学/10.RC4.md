**RC4 (Rivest Cipher 4)** 是密码学历史上最著名、应用最广泛，但最终跌落神坛的**流密码 (Stream Cipher)** 算法。

它由密码学大牛 **Ron Rivest** (也就是 RSA 算法里的那个 'R') 在 1987 年设计。

结合你刚才学的 **PRG (伪随机数生成器)** 和 **XOR** 的知识，RC4 可以说是把这两个概念工程化的典型代表。

### 1. RC4 的核心机制

RC4 的原理非常简单，它本质上是一个**状态机**。它的核心不是复杂的数学公式（像 RSA 那样的模幂运算），而是对一个数组进行不断的**洗牌 (Shuffling)**。

#### A. 内部状态 (Internal State)

RC4 维护一个包含 256 个字节的数组（称为 **S-Box**），初始时填满 $0$ 到 $255$。

$$S = [0, 1, 2, \dots, 255]$$

此外还有两个索引指针 $i$ 和 $j$，初始为 0。

#### B. 两个阶段

1. **密钥调度算法 (KSA - Key Scheduling Algorithm)**：
   - 利用你的密钥 (Key)，对这个 $S$ 数组进行“初始洗牌”。
   - 就像洗扑克牌一样，根据密钥的内容，把 $0-255$ 的顺序彻底打乱。
2. **伪随机生成算法 (PRGA - Pseudo-Random Generation Algorithm)**：
   - 这是生成**密钥流 (Keystream)** 的过程。
   - 每生成一个字节，就做以下几步：
     1. $i = (i + 1) \pmod{256}$
     2. $j = (j + S[i]) \pmod{256}$
     3. **交换** $S[i]$ 和 $S[j]$ 的值（继续洗牌）。
     4. 计算输出下标 $t = (S[i] + S[j]) \pmod{256}$。
     5. 输出密钥流字节 $K = S[t]$。

#### C. 加密

最后，把你生成的密钥流字节 $K$ 与明文 $P$ 进行 **异或 (XOR)**：

$$C = P \oplus K$$

------

### 2. RC4 的优点 (为什么它曾统治世界？)

在 90 年代和 00 年代初，RC4 几乎无处不在（WEP WiFi 加密、SSL/TLS HTTPS、PDF 加密等），原因如下：

1. **极度简单 (Simplicity)**：
   - 代码极短。核心逻辑用 C 语言写甚至不超过 10 行。
   - 非常容易在软件中实现，不容易写出 Bug。
2. **速度极快 (Speed)**：
   - 它只需要数组访问、加法和异或。没有乘法，没有除法，没有复杂的矩阵变换。
   - 在当年的老旧 CPU 上，它比 DES 或早期的 AES 快得多。
3. **可变密钥长度**：
   - 它支持从 40 位到 2048 位的任意长度密钥（虽然当年美国出口限制强制要求用弱的 40 位，埋下了祸根）。

------

### 3. RC4 的缺点 (为什么它被彻底弃用？)

虽然 RC4 设计很精妙，但它在统计学上是不完美的。根据你之前学的 **"PRG 必须不可预测"**，RC4 恰恰在这里翻车了。

#### A. 密钥流偏差 (Biased Output) - 致命伤

这是 RC4 最大的弱点。理论上，PRG 的输出每个字节出现的概率应该是 $\frac{1}{256}$。

但是研究发现，RC4 生成的**前几个字节**有严重的统计偏差。

- 例如，第二个字节倾向于为 0。
- 某些特定的字节组合出现的概率微小地高于其他组合。
- **后果：** 只要攻击者收集足够多的密文（几百万个包），利用这些微小的统计偏差，就能反向推算出密钥。

#### B. 弱密钥 (Weak Keys)

存在一些特定的密钥（Weak Keys），会导致 S-Box 的洗牌不够乱，从而在输出流中泄露大量密钥信息。

#### C. WEP 的惨案 (IV 使用不当)

早期的 WiFi 加密协议 **WEP** 使用了 RC4。

- **错误实现：** 它把一个 24 位的 IV 和主密钥**直接拼接**在一起作为 RC4 的种子。
- **攻击：** 由于 IV 只有 24 位，很快就会重复（两次一密攻击）。加上 RC4 前几个字节的偏差，导致 WEP 密码可以在几分钟内被 100% 破解。这是密码学教材中的经典反面教材。

#### D. 对状态的依赖

RC4 无法并行计算。你要算第 100 个字节，必须先算完前 99 个，因为 $S$ 数组的状态是时刻变化的。这在现代多核 CPU 上反而成了劣势（相比之下，AES-CTR 可以并行加密）。

------

### 4. 总结

| **特性**     | **描述**                                                     |
| ------------ | ------------------------------------------------------------ |
| **地位**     | 曾经的流密码之王，现已是**“加密禁术”**。                     |
| **数学原理** | 基于置换的随机洗牌 + XOR。                                   |
| **主要漏洞** | **输出偏差 (Bias)**。前几个字节不是真随机。                  |
| **现状**     | **RFC 7465** 明确禁止在 TLS (HTTPS) 中使用 RC4。Windows、浏览器都已默认禁用。 |
| **继任者**   | **ChaCha20**。它也是流密码，也很快，但设计上更现代，彻底解决了 RC4 的偏差问题，且支持 CPU 并行计算。 |

**一句话总结：** RC4 就像一辆 90 年代的跑车，当年很快、很酷、结构简单，但后来发现它的刹车系统（随机性）有设计缺陷，容易在高速行驶时失控（被破解），所以现在已经被强制报废，换成了 ChaCha20 这辆现代超跑。
