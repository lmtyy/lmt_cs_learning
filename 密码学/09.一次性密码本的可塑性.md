**一次性密码本（OTP）的可塑性 (Malleability)**，或者更广泛地说是**流密码的可塑性**，是这类加密体制最著名、也最危险的缺陷之一。

简单来说：**OTP 提供了完美的“保密性”（Confidentiality），但在“完整性”（Integrity）上是零分。**

虽然攻击者看不懂你在说什么（读不出明文），但他可以随心所欲地**篡改**你的话，而且能够**精确控制**篡改后的结果。这被称为 **比特翻转攻击 (Bit-Flipping Attack)**。

以下是详细的数学原理和攻击演示。

------

### 1. 核心数学原理：XOR 的透明性

我们回顾一下 OTP 的解密公式。

假设：

- $P$ 是明文。
- $K$ 是密钥。
- $C$ 是密文。

$$C = P \oplus K$$

$$P = C \oplus K$$

**可塑性（Malleability）** 指的是：如果攻击者修改密文 $C$，变成 $C'$，那么解密出来的明文 $P'$ 会发生什么变化？

假设攻击者将密文 $C$ 异或上一个“修改掩码” $\Delta$（Delta）：

$$C' = C \oplus \Delta$$

接收方并不知情，继续用密钥 $K$ 解密：

$$\begin{aligned} P' &= C' \oplus K \\ &= (C \oplus \Delta) \oplus K \\ &= (C \oplus K) \oplus \Delta \\ &= P \oplus \Delta \end{aligned}$$

**结论：**

$$P' = P \oplus \Delta$$

这意味着：**你在密文对应位置翻转了哪个比特，明文对应位置的比特也会准确地翻转，而且不会影响其他位。**这种修改是完全可预测的。

------

### 2. 攻击演示：篡改银行转账

假设你是一个中间人攻击者。你截获了一段加密数据包。虽然你没有密钥，不知道明文的具体内容，但通过流量分析（或者已知明文攻击），你猜测这段密文对应的是：“**Send $100 to Bob**”。

你想把它改成：“**Send $900 to Bob**”。

#### 步骤 1：准备修改掩码 ($\Delta$)

你需要计算字符 `'1'` 和字符 `'9'` 在 ASCII 码上的差异。

- `'1'` 的 ASCII 是 `0011 0001` (0x31)

- `'9'` 的 ASCII 是 `0011 1001` (0x39)

- **计算 $\Delta$：**

  $$\Delta = \text{'1'} \oplus \text{'9'} = 0000\ 1000$$

#### 步骤 2：篡改密文

你找到密文中对应数字位置的那个字节 $C_{byte}$，将其与 $\Delta$ 进行异或：

$$C'_{byte} = C_{byte} \oplus 0000\ 1000$$

#### 步骤 3：发送给银行

银行接收到 $C'$。银行完全不知道数据被改过（因为 OTP 没有校验机制），直接解密。

#### 步骤 4：解密结果

银行解密时，密钥 $K$ 会消掉原来的 $C$，但留下了 $\Delta$：

$$P' = P \oplus \Delta = \text{'1'} \oplus (\text{'1'} \oplus \text{'9'}) = \text{'9'}$$

**结果：** 银行看到的明文变成了 “**Send $900 to Bob**”。攻击成功，且没有触发任何解密错误（不像分组密码可能会导致乱码）。

------

### 3. 为什么这叫“可塑性”？

“可塑性”这个词来源于材料科学，意为“可以被塑造成任何形状”。

在密码学中，如果在不知道密钥的情况下，攻击者可以通过对密文进行特定的变换，使得解密后的明文也发生**有意义的、预期的变换**，我们就说这个加密算法是“可塑的”。

- **OTP/流密码：** 具有完全的可塑性（Bit-level Malleability）。
- **CBC 模式（分组密码）：** 具有部分可塑性（修改一个密文块，会彻底打乱当前明文块，但会可预测地修改下一个明文块的对应位）。

### 4. 解决方案：认证 (Authentication)

正因为 OTP 和流密码（如 ChaCha20, RC4）都有极强的可塑性，所以现代密码学定下一条铁律：

> **Encryption without Authentication is useless.**
>
> **（没有认证的加密是无用的。）**

为了防止这种攻击，我们必须引入 **消息验证码 (MAC, Message Authentication Code)**。

- 在加密之前，先计算明文的哈希/MAC。
- 或者使用 **AEAD (Authenticated Encryption with Associated Data)** 算法（如 **ChaCha20-Poly1305** 或 **AES-GCM**）。

**机制：** 如果攻击者翻转了密文的某一位，解密时的 MAC 校验就会失败（因为 $MAC(P') \neq MAC(P)$），系统会直接报错并丢弃消息，而不是执行错误的指令。

### 总结

一次性密码本的可塑性告诉我们：**“保密”不等于“安全”。**

即使攻击者看不懂你的信（保密性），他依然可以把你信里的“同意”改成“拒绝”（破坏完整性），而唯一的防御手段就是**完整性校验 (Integrity Check)**。
