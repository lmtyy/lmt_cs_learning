这组PPT开始讲解 **Layered Structure（分层结构）**，这是继“单内核（Monolithic）”之后的一种更规范、更模块化的OS设计方法。

不同于单内核把所有功能堆在一起，分层结构像叠积木一样，每一层都构建在更低层之上，并且只能调用更低层的服务。

---

### 第一张图：Layered Structure 概念

**核心思想**：
*   操作系统被划分为**若干层**。
*   **Layer 0（最底层）**：硬件。
*   **Layer N（最高层）**：用户界面（UI / Shell）。
*   **关键规则**：每一层只能调用**比自己更低层**的功能和服务。不能跨越层级调用。

**Modularity（模块化）**：
*   这是分层结构带来的最大好处。因为每一层都只依赖下面的一层，所以可以独立开发、测试和修改某一层，而不影响其他层。
*   这种“洋葱圈”式的结构，也常被称为 **Hierarchical Structure**。

---

### 第二张图：Layered Structure 图解（局部放大）

这张图展示了 **Layer M** 和 **Layer M-1** 之间的依赖关系。

*   **Layer M（上层）**：提供了 **new operations**（新功能）。
*   **Layer M-1（下层）**：提供了 **existing operations**（已有功能）。
*   **箭头方向**：**只能从 M 指向 M-1**。Layer M 调用 Layer M-1 提供的服务。
*   **Hidden operations（隐藏操作）**：
    *   在这个设计中，Layer M 无法直接调用 Layer M-1 内部未公开的函数。这就是封装的体现。

---

### 第三张图：Layered Structure (OS/2) 实例

这张图展示了 **OS/2**（IBM曾经开发的操作系统）的层次化设计。

*   **最上层**：**Applications**（用户应用程序）。
*   **API层**：**Application-programming interface**（应用程序编程接口）和 **API extension**。这是用户程序进入操作系统的入口。
*   **中间层**：**Subsystems**（子系统）。
    *   这里通常包含：文件系统、网络协议栈等高层服务。它们在API下，在内核上。
*   **底层内核（System Kernel）**：
    *   这是操作系统的核心，负责最底层的工作：
    *   Memory management（内存管理）
    *   Task dispatching（任务调度 / 进程调度）
    *   Device management（设备管理）
*   **最底层**：**Device drivers**（设备驱动程序）。它们直接与硬件交互。

---

### 💡 对比：Monolithic vs Layered

| 结构 | Monolithic（单内核） | Layered（分层） |
| :--- | :--- | :--- |
| **组织方式** | 所有核心功能位于同一层级，同一地址空间 | 功能被划分为多个层级，每层依赖下层 |
| **调用方式** | 直接函数调用（速度快） | 必须通过层层接口（可能产生开销） |
| **模块化** | 低（高度耦合） | 高（解耦，便于开发维护） |
| **安全性** | 较低（崩溃容易波及整个内核） | 较高（某一层崩溃可能导致下层不可用，但比单内核更容易隔离） |
| **实际应用** | Linux, Windows（现代内核） | **OS/2**, **早期 UNIX**, **THE操作系统** |

**💡 分层结构的局限性（常见考点）：**
虽然分层结构提高了模块化，但在实际实现中有一个问题：**层与层之间的交互可能产生额外开销**（每穿过一层都需要消息传递或函数调用）。而且，**如何正确地划分层级**也是一门艺术（例如，文件系统放哪一层？虚存放哪一层？），如果划分不合理，会导致性能下降。

**接下来的课件大概率会讲解**：
*   **Microkernel（微内核）**：这是分层结构的“极端版”——把内核做到最小，多数功能移到用户空间。
*   **Virtual Machine（虚拟机）**：这是一种完全不同的结构，用于在同一硬件上运行多个操作系统。
