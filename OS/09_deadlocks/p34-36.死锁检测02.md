这组幻灯片是**死锁检测算法**的**完整计算实例**。通过这个具体例子，你可以清楚地看到“检测算法”和之前学的“安全算法（银行家算法）”在计算步骤上的区别。

我们一步一步来拆解这个过程。

## 第一张图：初始状态 \(T_0\)

### 题目信息
- 5个进程 (\(P_0 \sim P_4\))，3种资源 (A:7个, B:2个, C:6个)
- 给出了 **Allocation**（已分配）矩阵和 **Request**（当前请求）矩阵
- 当前 **Available**（可用资源）为 **`(0, 0, 0)`**！

> ⚠️ **重要观察**：当前系统没有任何空闲资源！这通常意味着死锁风险很高。

### 表格内容
| 进程 | Allocation (A B C) | Request (A B C) |
|------|--------------------|-----------------|
| \(P_0\) | 0 1 0 | 0 0 0 |
| \(P_1\) | 2 0 0 | 2 0 2 |
| \(P_2\) | 3 0 3 | 0 0 0 |
| \(P_3\) | 2 1 1 | 1 0 0 |
| \(P_4\) | 0 0 2 | 0 0 2 |

**Available** = `(0, 0, 0)`

---

### 手动模拟检测算法（验证幻灯片底部的结论）

幻灯片底部说：**序列 `<P_0, P_2, P_3, P_1, P_4>` 能使所有 `Finish[i] = true`**。

我们手动验算一遍：

**Step 1: 初始化**
- `Work = Available = (0, 0, 0)`
- `Finish` 数组：所有进程初始为 `false`

**Step 2: 找能执行的进程**
- 检查谁能被满足？
  - \(P_0\): Request = `(0,0,0)`。\(\le\) `Work = (0,0,0)` ✅
  - 执行 \(P_0\)，释放资源 `Allocation(0,1,0)`。
  - 新 `Work` = `(0,0,0) + (0,1,0) = (0,1,0)`
  - `Finish[0] = true`

**Step 3: 继续找**
- 当前 `Work = (0,1,0)`
  - \(P_1\): Request = `(2,0,2)`，需要2个A，但Work只有0个A ❌
  - \(P_2\): Request = `(0,0,0)` \(\le\) `(0,1,0)` ✅
  - 执行 \(P_2\)，释放 `Allocation(3,0,3)`。
  - 新 `Work` = `(0,1,0) + (3,0,3) = (3,1,3)`
  - `Finish[2] = true`

**Step 4: 继续找**
- 当前 `Work = (3,1,3)`
  - \(P_3\): Request = `(1,0,0)` \(\le\) `(3,1,3)` ✅
  - 执行 \(P_3\)，释放 `Allocation(2,1,1)`。
  - 新 `Work` = `(3,1,3) + (2,1,1) = (5,2,4)`
  - `Finish[3] = true`

**Step 5: 继续找**
- 当前 `Work = (5,2,4)`
  - \(P_1\): Request = `(2,0,2)` \(\le\) `(5,2,4)` ✅
  - 执行 \(P_1\)，释放 `Allocation(2,0,0)`。
  - 新 `Work` = `(5,2,4) + (2,0,0) = (7,2,4)`
  - `Finish[1] = true`

**Step 6: 继续找**
- 当前 `Work = (7,2,4)`
  - \(P_4\): Request = `(0,0,2)` \(\le\) `(7,2,4)` ✅
  - 执行 \(P_4\)，释放 `Allocation(0,0,2)`。
  - 新 `Work` = `(7,2,4) + (0,0,2) = (7,2,6)`
  - `Finish[4] = true`

**结论**：所有 `Finish` 都为 `true` ✅。系统当前**没有死锁**。

## 第二张图：状态变化 → 死锁发生

### 新请求
> **\(P_2\) 请求额外的1个C资源**

更新 `Request` 矩阵：
- \(P_2\) 原本 Request 是 `(0,0,0)`，现在变成 **`(0,0,1)`**

### 重新运行检测算法

**Step 1: 初始化**
- `Work = (0,0,0)`
- 所有 `Finish = false`

**Step 2: 找能执行的进程**
- \(P_0\): Request = `(0,0,0)` \(\le\) `(0,0,0)` ✅
- 执行 \(P_0\)，释放 `(0,1,0)`。
- 新 `Work` = `(0,1,0)`
- `Finish[0] = true`

**Step 3: 继续找**
- 当前 `Work = (0,1,0)`
  - \(P_1\): Request = `(2,0,2)` ❌ (缺A)
  - \(P_2\): Request = `(0,0,1)` ❌ (缺C)
  - \(P_3\): Request = `(1,0,0)` ❌ (缺A)
  - \(P_4\): Request = `(0,0,2)` ❌ (缺C)

**结果**：找不到任何能执行的进程了！

### 最终状态
- `Finish[1] = false`, `Finish[2] = false`, `Finish[3] = false`, `Finish[4] = false`
- **存在 `false` 的进程！** → **死锁发生**

### 幻灯片结论
> **死锁存在，包含进程 \(P_1, P_2, P_3, P_4\)**

## 第三张图：何时调用检测算法？

这一页是**工程实践问题**：检测算法不能总运行（太耗资源），什么时候运行？

### 两个决策因素
1. **死锁发生的频率**：如果死锁很少发生 → 不用频繁检测
2. **需要回滚的进程数量**：如果回滚代价高 → 要尽早检测

### 常见的触发策略
- **定期运行**：比如每10分钟扫描一次
- **资源请求失败时触发**：当进程请求资源但得不到满足时，触发检查
- **CPU利用率骤降时触发**：系统突然变慢 → 可能死锁了

## 💡 关键对比：避免 vs 检测（算题区别）

| | 死锁避免（银行家算法） | 死锁检测 |
|---|---|---|
| **判断矩阵** | \(Need = Max - Allocation\) | **Request**（当前实际请求） |
| **比较对象** | Need vs Available | Request vs Available |
| **预知需求** | 必须预知 Max | 不需要 |
| **测试条件** | 是否能完成所有进程 | 是否能完成部分进程 |
| **是否主动阻止** | 是（拒绝分配） | 否（事后发现） |

## 🎯 做题技巧

**死锁检测的计算题，直接套5步**：

1. **列矩阵**：Allocation, Request, Available
2. **初始化**：`Work = Available`，`Finish = false`（所有进程）
3. **找可满足进程**：`Request[i] ≤ Work`
4. **更新**：`Work += Allocation[i]`，`Finish[i] = true`
5. **判断**：如果全部 `true` → **无死锁**；否则 → **死锁**（`false`的那些进程就是死锁进程）
