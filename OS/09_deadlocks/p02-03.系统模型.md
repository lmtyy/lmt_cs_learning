### 第一张图：系统模型（定义与示例）

#### 1. 死锁的通用定义
幻灯片顶部的第一段话给出了死锁的**教科书定义**：
> 一组被阻塞（blocked）的进程，每一个进程都持有一种资源，并且正在等待另一个进程手中的资源。

这就是死锁的**核心状态**：**我在等你，你在等他，他在等我**。

#### 2. 硬件资源示例（磁带机）
这是一个经典的物理资源死锁场景：
*   **系统资源**：有 **2台** 磁带机。
*   **进程 P1 和 P2**：
    *   P1 拿走了1台，想再拿1台（但已被 P2 占用）。
    *   P2 拿走了1台，想再拿1台（但已被 P1 占用）。

这直观地展示了**循环等待**：P1 等 P2，P2 等 P1，谁也动不了。

#### 3. 软件资源示例（信号量代码）
这是考试中**最容易考到**的代码段。它展示了死锁在代码层面是如何发生的：

*   **信号量 A 和 B** 初始化为 1（代表互斥锁）。
*   **P0 进程**：
    ```c
    wait(A);  // P0 锁住 A
    wait(B);  // P0 尝试锁 B，但此时 B 被 P1 锁着，P0 阻塞！
    ```
*   **P1 进程**：
    ```c
    wait(B);  // P1 锁住 B
    wait(A);  // P1 尝试锁 A，但此时 A 被 P0 锁着，P1 阻塞！
    ```

**结论**：P0 和 P1 互相拿着对方想要的钥匙，双双卡死。这就是软件层面最典型的死锁。

### 第二张图：通用的系统模型（General Model）

这张幻灯片将死锁模型抽象化，用于数学和算法层面的描述。

#### 1. 资源模型
*   **资源类型**：\(R_1, R_2, ..., R_m\)
    *   包括：CPU周期、内存空间、I/O设备（打印机、磁盘等）。
*   **实例数量**：每种资源 \(R_i\) 有 \(W_i\) 个实例（比如3台打印机）。
    *   **注意**：如果只有1个实例（\(W_i=1\)），互斥冲突会更明显；如果有多个实例，需要依赖复杂的算法来检测。

#### 2. 进程操作资源的三阶段
进程使用资源遵循固定的生命周期：
1.  **Request（申请）**：请求资源。如果请求不到，进程就会阻塞（wait）。
2.  **Use（使用）**：实际占用资源。
3.  **Release（释放）**：用完后释放，让给别人用。

> **死锁就在 Request 阶段发生**——因为申请不到资源，才会僵持。

#### 3. 死锁状态的精确定义（重点！）
幻灯片最下方给出了死锁状态的数学定义：
> 当集合中的每一个进程都在等待**只能由该集合中另一个进程触发**的事件时，该进程集合就处于**死锁**状态。

**通俗解释**：A在等B干活，B在等C干活，C在等A干活。只要能打破这个“闭环”，死锁就会解除。


*(答案：会的。因为 P0 抢了 A，P1 抢了 B，然后 P0 等 B，P1 等 A。代码顺序只是掩盖了逻辑，本质上还是循环等待)*
