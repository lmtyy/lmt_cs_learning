这四张 PPT 展示了进程同步问题的**具体案例**——**有界缓冲区问题 (Bounded-Buffer Problem)**，也就是常说的**生产者-消费者问题**。

课件从这里开始从理论（为什么需要同步）转向具体的代码实现，并揭示了**若不使用信号量/PV操作，直接并发运行代码会导致的严重错误**。

我来为你一步步拆解这四张图，帮你理清思路。

---

### 图1：背景 (BACKGROUND)
**核心观点：并发访问共享数据会造成数据不一致。**

*   这页重申了上一节课的理论基础。
*   如果不加控制，多个进程同时读写同一个变量，数据会出错。
*   为了保持数据一致性，我们需要机制来保证**有序执行 (orderly execution)**。

---

### 图2：共享数据定义 (Shared Data)
这是我们要操作的数据结构，相当于共享的“仓库”。

```c
#define BUFFER_SIZE 10       // 缓冲区大小，最多存10个数据
typedef struct { ... } item; // 数据项的结构
item buffer[BUFFER_SIZE];    // 存放数据的数组（环形缓冲区）
int in = 0;     // 生产者下一个应该存放数据的位置（写指针）
int out = 0;    // 消费者下一个应该取走数据的位置（读指针）
int counter = 0;// 缓冲区里当前存了多少个数据
```

*   **理解 `in` 和 `out`**：利用 `% BUFFER_SIZE` 实现循环使用的环形缓冲区。写数据往 `in` 写，取数据从 `out` 取。
*   **理解 `counter`**：它是**共享变量**。`counter++` 代表存入一个，`counter--` 代表取走一个。

---

### 图3：生产者进程 (Producer)
这是一段**“有问题”**的代码。请仔细看它怎么操作 `counter` 的。

```c
while (1) {
    // 1. 忙等待：如果满了（counter == 10），就一直空转，啥也不干
    while (counter == BUFFER_SIZE)
        ; /* do nothing */

    // 2. 生产一个数据放入缓冲区
    buffer[in] = nextProduced;
    in = (in + 1) % BUFFER_SIZE; // 写指针后移

    // 3. 关键动作：计数器加1
    counter++; 
}
```

---

### 图4：消费者进程 (Consumer)
同样也是**“有问题”**的代码。

```c
while (1) {
    // 1. 忙等待：如果空了（counter == 0），就一直空转
    while (counter == 0)
        ; /* do nothing */

    // 2. 从缓冲区取出一个数据
    nextConsumed = buffer[out];
    out = (out + 1) % BUFFER_SIZE; // 读指针后移

    // 3. 关键动作：计数器减1
    counter--;
}
```

---

### 🚨 核心考点：这段代码为什么出错了？（Race Condition）

虽然单独看生产者或消费者的逻辑都没问题（满了就不放，空了就不取），但**当它们并发运行时，问题就出在 `counter++` 和 `counter--` 这一行代码上**。

`counter++` 在汇编层面通常对应三条指令：
1.  `LOAD` (把 counter 的值从内存读到寄存器)
2.  `INCREMENT` (在寄存器里加 1)
3.  `STORE` (把寄存器的新值写回内存)

**假设当前 `counter = 5`。**
*   **生产者 (P)** 执行了 `LOAD`，寄存器拿到 5。就在它还没来得及执行 `INCREMENT` 和 `STORE` 时，CPU 调度器切到了**消费者 (C)**。
*   **消费者 (C)** 也执行了 `LOAD`，由于内存里的 `counter` 还是 5（P还没写回去），C 拿到的也是 5。
*   **消费者 (C)** 完成减 1 操作，把 4 写回内存。
*   **生产者 (P)** 恢复运行，继续执行 `INCREMENT` (5变成6) 和 `STORE`，把 6 写回内存。

**结果**：一个加 1，一个减 1。`5+1-1` 应该等于 `5`。但实际结果变成了 **6**！
这就是最经典的**竞态条件 (Race Condition)**。数据丢失了（消费者的减 1 被生产者的覆盖了，或者反之）。

### 💡 考试/做题重点提示
这两页 PPT 展示的是**未加同步机制的“原始”代码**。在后续的章节（Semaphores / 信号量）中，老师会教你如何修改这段代码：

1.  你需要定义两个**信号量**：
    *   `mutex` (互斥锁)：初始化为 1，保护 `counter` 的加减。
    *   `empty` (空槽位数量)：初始化为 `BUFFER_SIZE`。
    *   `full` (已填数据数量)：初始化为 0。
2.  你会用 `P(mutex)` 和 `V(mutex)` 把 `counter++` 和 `counter--` 包裹起来，确保它们是原子操作。
3.  你会用 `P(empty)` 代替 `while(counter==BUFFER_SIZE)` 的忙等待（让进程让出CPU睡眠）。
4.  你会用 `P(full)` 代替 `while(counter==0)` 的忙等待。
