### 图 1：死锁 (Deadlock) 与 饥饿 (Starvation)

**关键点**：信号量虽然解决了竞态条件，但如果程序员使用不当，会引入新的问题。

#### 1. 死锁 (Deadlock)
*   **定义**：两个或多个进程在无限期地等待一个事件，而这个事件只能由等待进程中的一个产生。
*   **代码示例（经典死锁代码）**：
    *   两个信号量 `S` 和 `Q`，都初始化为 1。
    *   **进程 P0**：申请 `S` (成功)，然后申请 `Q`。
    *   **进程 P1**：申请 `Q` (成功)，然后申请 `S`。
    *   **死锁发生**：P0 拿着 S，等 Q；P1 拿着 Q，等 S。两人互相握着对方需要的锁不放。谁也动不了。
*   **危害**：程序“卡死”，永远无法执行下去。

#### 2. 饥饿 (Starvation)
*   **定义**：无限期阻塞。一个进程可能永远无法从信号量队列中被移除（永远轮不到它）。
*   **原因**：信号量队列的实现方式可能不公平（比如后进先出 LIFO 队列，或者优先级调度）。导致某些进程永远被插队。

**结论**：好的同步算法不仅要解决竞态条件，还要确保**不产生死锁**和**避免饥饿**。

---

### 图 2：Semaphores: Types (信号量的两种类型)

这张 PPT 区分了两种信号量：

1.  **计数信号量 (Counting Semaphore)**：取值范围是**任意整数**（0, 1, 2, 3...）。用于资源数量管理（如 10 个空位的缓冲区）。
2.  **二进制信号量 (Binary Semaphore)**：取值范围只能是 **0 或 1**。其实就是我们常说的“互斥锁 (Mutex)”。

**关键结论**：计数信号量可以用二进制信号量来实现。

### 图 3 & 图 4：Implementing Counting Semaphore by Binary Semaphore
*(用二进制信号量实现计数信号量)*

这部分就是**具体的实现代码**。虽然操作系统内核通常直接支持计数信号量，但掌握这个原理能加深你对并发编程的理解。

#### 图 3：定义数据结构
*   `S1`：保护内部变量 `C` 的互斥锁 (初始为 1)。
*   `S2`：用于阻塞和唤醒进程的同步信号量 (初始为 0)。
*   `C`：整型计数器 (初始为计数信号量的初始值)。

#### 图 4：实现代码 (注意两个操作的区别)

**`wait` 操作 (申请资源)**：
```c
wait(S1);      // 1. 上锁，保护 C
C--;           // 2. 资源数减 1
if (C < 0) {   // 3. 如果没有资源了
    signal(S1);    //   先释放锁，给别人机会操作
    wait(S2);      //   然后阻塞自己，等待唤醒
}
signal(S1);    // 4. 释放锁
```

**`signal` 操作 (释放资源)**：
```c
wait(S1);      // 1. 上锁，保护 C
C++;           // 2. 资源数加 1
if (C <= 0)    // 3. 如果之前有进程在等 (C <= 0 说明有人排队)
    signal(S2);    //   唤醒一个等待的进程
else
    signal(S1);    //   没有进程等，直接释放锁
```

**原理**：
*   `wait(S1)` 确保 `C` 的操作（加减）是**原子**的，不会有竞态条件。
*   `S2` 充当了**排队等待区**。当 `C` 为负数时，表示队列里有进程在等，这时候通过 `wait(S2)` 挂起进程。
*   当 `C` 增加时，如果 `C <= 0`，说明有人在排队，通过 `signal(S2)` 唤醒一个。
