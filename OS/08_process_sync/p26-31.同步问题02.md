这组 PPT 详细讲解了经典同步问题中的 **“读者-写者问题” (Readers-Writers Problem)**，并重点展示了 **“读者优先” (Readers First)** 的解决方案。

这个问题的核心在于：**允许多人同时读，但绝对不允许有人写的时候还有人读（或写）。**

我为你拆解这 6 张图，梳理逻辑并画重点：

---

### 图 1：问题定义 (是什么？)

这张 PPT 定义了同步规则：
1.  **共享对象**：一个数据对象（比如一个文件、一个数据库记录）。
2.  **两类进程**：
    *   **Reader (读者)**：只读，不修改。
    *   **Writer (写者)**：会修改（读+写）。
3.  **同步要求**：
    *   **写者必须互斥**：一个写者正在写时，**任何其他进程（无论是读者还是写者）都不能访问**。 (写->写互斥，写->读互斥)
    *   **读者可以共存**：多个读者可以**同时读**。 (读->读允许)

---

### 图 2：两种解决方案 (优先级)

这个问题并没有绝对正确的解，而是取决于**你想要照顾谁**：

1.  **Readers First (读者优先)**：只要有一个读者在读，后续来的读者都可以直接进来，而**写者必须等待所有读者读完**。
    *   *缺点*：如果读者络绎不绝，写者可能**永远等不到（饥饿）**。
2.  **Writers First (写者优先)**：一旦有写者准备写，**不允许新的读者进入**，让写者尽快完成。
    *   *缺点*：读者可能被长时间阻塞。

**本组 PPT 讲解的是“读者优先”方案。**

---

### 图 3  & 图 4：Shared Data (共享变量初始化)

这是解决本问题的**数据结构核心**。

```c
semaphore mutex = 1;   // 保护 readcount 变量的互斥锁
semaphore wrt = 1;     // 控制写者互斥的锁 (也控制读者与写者的互斥)
int readcount = 0;     // 当前有多少个读者正在读
```

**特别注意：**
*   **`wrt`** 是控制“能不能写”的锁。**`wrt` 被占住时，写者不能写，新来的读者也不能进。**
*   **`readcount`** 决定了谁去抢 `wrt`。这是读者优先策略的基石。

---

### 图 5：Writers Process (写者进程)

**写者的代码非常简单：**

```c
wait(wrt);      // 申请写锁 (P操作)
    // 执行写操作
    writing is performed;
signal(wrt);    // 释放写锁 (V操作)
```

**逻辑**：只要 `wrt` 被占住（比如有读者在读），写者就会在 `wait(wrt)` 处**阻塞**，直到所有读者都读完释放 `wrt`。

---

### 图 6：Readers Process (读者进程)

这部分是**实现“读者优先”最关键的代码**，它通过**计数器**和**第一个/最后一个读者**来控制 `wrt` 锁。

#### 进入部分（Entry Section）：
```c
wait(mutex);          // 1. 保护 readcount，防止并发修改出问题
readcount++;          // 2. 来了一个读者，计数器+1
if (readcount == 1)   // 3. **关键点**：如果是第一个读者
    wait(wrt);        //   抢写锁！阻止写者进入。
signal(mutex);        // 4. 释放 readcount 锁
```

#### 核心执行部分：
```c
reading is performed; // 这里是真正的读操作，多个读者可以同时执行
```

#### 离开部分（Exit Section）：
```c
wait(mutex);          // 1. 保护 readcount
readcount--;          // 2. 读者走了，计数器-1
if (readcount == 0)   // 3. **关键点**：如果是最后一个读者
    signal(wrt);      //   释放写锁！允许写者进入。
signal(mutex);        // 4. 释放 readcount 锁
```

---

### 💡 核心逻辑分析 (最重要的考点)

请务必理解这个逻辑，考试特别爱考：

1.  **为什么 `wrt` 控制整个流程？**
    *   `wrt` 是一个**写者锁**，但**第一个读者**和**最后一个读者**会去操作它。
    *   **第一个读者来了**：他锁住 `wrt`。从此所有写者被堵在外面。但后续的读者因为 `readcount` 变大了，不会再去抢 `wrt`，所以可以顺利进入。
    *   **最后一个读者走了**：他释放 `wrt`。此时如果有写者在等，就能进去了。

2.  **为什么要有 `mutex`？**
    *   如果没有 `mutex`，两个读者同时执行 `readcount++` 会导致竞态条件（之前讲过的 `counter++` 问题）。所以必须用信号量保护这个计数器。

3.  **为什么这叫“读者优先”？**
    *   当写者在等待时，只要有新的读者到来（`readcount++`），只要之前有人还在读（`readcount > 0`），写者就必须继续等待。所以**源源不断的读者可以饿死写者**。
