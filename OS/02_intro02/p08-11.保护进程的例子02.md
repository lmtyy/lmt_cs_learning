## 1. Address Translation（地址翻译）

这张图讲的是硬件机制：

- **地址空间**：每个程序（进程）和内核都有自己的地址空间（一组可用的内存地址）。
- **虚拟地址**：CPU 执行指令时发出的地址（比如 `mov [0x1000], 5` 中的 `0x1000`）。
- **物理地址**：内存芯片上真实的行/列地址。
- **MMU（内存管理单元）**：硬件单元，自动在 CPU 和内存之间做翻译。

> 关键点：程序自己看到的地址是假的、连续的、从 0 开始；  
> 真正的物理内存可能是分散的、受保护的。

---

## 2. Example of Address Translation（具体例子）

这张图展示了两个程序（Prog 1 和 Prog 2）如何共享物理内存但互相看不见。

### 左边：
- **Prog 1 的虚拟地址空间**：包含 Code, Data, Heap, Stack（连续布局）
- **Prog 2 类似的虚拟地址空间**

### 右边：
- **物理内存**实际被分成很多块：
  - OS code / data / heap & stacks（内核占用）
  - Stack 1, Stack 2（两个程序的栈）
  - Heap 1, Heap 2
  - Code 1, Code 2
  - Data 1, Data 2

### 翻译示例：
- Prog 1 访问虚拟地址 `Code` 区域 → MMU 映射到物理 `Code 1`
- Prog 2 访问同样的虚拟地址 `Code` 区域 → MMU 映射到物理 `Code 2`

> 相同虚拟地址，不同物理地址 → 隔离。  
> 这就是“地址翻译实现进程间保护”的直观例子。

## 3. Dual Mode Operation（双模式操作）

这张图强调了硬件提供的两种模式，以及切换机制。

### 两种模式：
| 模式 | 可执行指令 | 目的 |
|------|-----------|------|
| 内核态 | 所有指令（包括特权指令） | OS 运行 |
| 用户态 | 普通指令（禁止修改页表、直接 I/O 等） | 应用程序运行 |

### 禁止示例：
- 用户态尝试修改页表 → CPU 产生异常 → OS 接管并杀死进程。

### 从用户态切换到内核态的唯一方式（硬件自动完成）：
- **系统调用**（程序主动请求，如 `read()`）
- **中断**（外部设备信号，如键盘输入）
- **其他异常**（如除零、页错误）

> 切换时：CPU 保存当前状态，模式位改为内核态，跳转到 OS 预先设定的处理函数。

## 4. UNIX System Structure（UNIX 系统结构）

这张图把前面所有概念整合成一个完整的分层架构。

### 结构（从上到下）：

#### User Mode（用户态）
- Applications
- Standard Libs / shells / commands / compilers

#### Kernel Mode（内核态）
- **系统调用接口**（System-call interface）—— 这是用户进入内核的唯一入口
- 内部模块：
  - 信号、终端处理、字符 I/O
  - 文件系统、交换、块 I/O、磁盘/磁带驱动
  - CPU 调度、页替换、请求调页、虚拟内存

#### Hardware（硬件）
- 终端控制器、设备控制器、内存控制器

### 关键点：
- **系统调用接口是分隔线**：上面是用户态，下面是内核态。
- **所有对硬件、内存管理、进程调度的操作都在内核态**。
- 应用程序通过系统调用（如 `open()`, `read()`）请求服务，陷入内核，执行完后返回用户态。

---

## 最终关系图

```
应用程序 (用户态)
    ↓ 系统调用
系统调用接口 (内核态入口)
    ↓
内核模块：调度 / 内存管理 / 文件系统 / 驱动
    ↓
硬件 (MMU 做地址翻译 + 双模式强制保护)
```

**MMU + 双模式 + 系统调用** 三者协同：
- MMU → 隔离进程地址空间
- 双模式 → 禁止用户态做危险操作
- 系统调用 → 唯一合法进入内核的通道

这就是现代 OS 能够同时运行多个程序、互不影响、且无法搞垮系统的底层原理。
