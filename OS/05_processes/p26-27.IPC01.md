## 一、IPC vs 共享内存（第1张图开头）

| 对比维度 | 共享内存 | IPC（消息传递） |
|----------|----------|------------------|
| **效率** | 高（不经过内核） | 低（每次都要内核介入） |
| **适用场景** | 同一台计算机 | 单机或分布式都行 |
| **同步责任** | 程序员自己负责 | 操作系统可以帮忙 |
| **编程复杂度** | 难（要自己加锁） | 相对简单 |

> 💡 一句话：**共享内存快但难，IPC慢但简单且支持跨机器。**

---

## 二、两个基本操作：send 和 receive

```c
send(message);   // 发送消息
receive(message); // 接收消息
```

所有 IPC 模型都是这两个操作的变种。

---

## 三、Naming：怎么找到对方？（第2张图）

这是 IPC 最核心的设计问题。

### 3.1 直接通信（Direct Communication）

> 发消息时**直接指定**对方的进程 ID

```c
send(P, message);   // 发给进程 P
receive(Q, message); // 从进程 Q 收
```

| 特点 | 说明 |
|------|------|
| 自动建立连接 | 每对进程之间只有一条链接 |
| 一对一 | 一个链接只能对应两个进程 |

### 3.2 间接通信（Indirect Communication）

> 通过**信箱/端口**（mailbox/port）来收发消息

```c
send(A, message);   // 发到信箱 A
receive(A, message); // 从信箱 A 收
```

| 特点 | 说明 |
|------|------|
| 灵活 | 多个进程可以共享同一个信箱 |
| 一对多 | 一个信箱可以被多个进程收发 |

**类比：**
- 直接通信 = 直接打电话给某人
- 间接通信 = 往快递柜里放/取快递

## 四、同步方式：阻塞 vs 非阻塞（第2张图）

这是考试**非常爱考**的点。

| 类型 | 发送方（send） | 接收方（receive） |
|------|---------------|-------------------|
| **阻塞（同步）** | 发完等对方接收完才返回 | 没消息就一直等 |
| **非阻塞（异步）** | 发完立刻继续执行 | 没消息就立刻返回空或错误 |

### 4.1 四种组合

| 发送 | 接收 | 效果 |
|------|------|------|
| 阻塞 | 阻塞 | 双方同步，最常用（如 RPC） |
| 阻塞 | 非阻塞 | 发送方等，接收方立即返回 |
| 非阻塞 | 阻塞 | 发送方不等，接收方等 |
| 非阻塞 | 非阻塞 | 双方都不等，需要自己检查 |

### 4.2 容易混淆的术语

课件里的 `Blocking / Nonblocking` 和 `Synchronous / Asynchronous` 在很多教材里混用。你记住：

> **阻塞 = 等**  
> **非阻塞 = 不等**

## 五、其他 IPC 问题（第1张图后半）

| 问题 | 含义 |
|------|------|
| **自动缓冲 vs 显式缓冲** | 系统自动管理消息队列 / 程序员自己指定缓冲区大小 |
| **传副本 vs 传引用** | 复制整个消息 / 只传指针（共享内存就是传引用） |
| **固定大小 vs 可变大小** | 每条消息固定长度 / 可变长度 |

## 六、帮你总结一页纸

| 概念 | 一句话 |
|------|--------|
| **IPC vs 共享内存** | 共享内存快，IPC 简单且可跨机器 |
| **直接通信** | 指定对方 PID |
| **间接通信** | 通过信箱 |
| **阻塞 send** | 等对方收完再返回 |
| **非阻塞 send** | 发完就跑 |
| **阻塞 receive** | 没消息就一直等 |
| **非阻塞 receive** | 没消息立刻返回 |

## 七、小测验

1. 共享内存和 IPC 哪个效率更高？为什么？
2. 直接通信和间接通信的区别是什么？
3. 阻塞 receive 是什么意思？

> 答案：  
> 1. 共享内存（不经过内核）  
> 2. 直接指定 PID vs 通过信箱  
> 3. 没消息时进程会等待，不执行后续代码
