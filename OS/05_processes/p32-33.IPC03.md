## 一、同步（第 1 张图）

### 1.1 四种基本组合

| 类型 | send 行为 | receive 行为 |
|------|----------|--------------|
| **阻塞 send** | 发完消息后，**等对方收完**才返回 | — |
| **非阻塞 send** | 发完消息，**立刻继续执行** | — |
| **阻塞 receive** | — | 没消息时**一直等**，直到收到 |
| **非阻塞 receive** | — | 没消息时**立刻返回**空或错误 |

### 1.2 图示理解

```
阻塞 send：
发送方：发消息 ──┬── 等 ──┬── 继续
接收方：         └── 收 ──┘

非阻塞 send：
发送方：发消息 ──→ 立刻继续
接收方：       （不知道什么时候收）

阻塞 receive：
接收方：等......等......收到消息 → 继续

非阻塞 receive：
接收方：检查 → 没消息 → 立刻返回
        检查 → 有消息 → 取走 → 继续
```

### 1.3 课件里说的 "most common one"

> **非阻塞 send + 阻塞 receive**

这是最常用的组合，为什么？

| 为什么 send 用非阻塞？ | 为什么 receive 用阻塞？ |
|----------------------|------------------------|
| 发送方发完就继续干活，不用干等 | 接收方本来就是为了等数据，阻塞省 CPU |
| 提高系统吞吐量 | 自然同步 |

**实际例子**：打印任务
- 你的程序（发送方）：丢一个打印任务，继续干别的
- 打印驱动（接收方）：没任务就阻塞等着，有任务就处理

## 二、缓冲（第 2 张图）

这是说：**消息在传送过程中，存在哪里？能存多少？**

### 2.1 三种缓冲方式

| 类型 | 容量 | send 行为（接收方没准备好时） |
|------|------|------------------------------|
| **零容量** | 0 条消息 | 发送方**必须等**接收方收完（阻塞 send） |
| **有界容量** | n 条消息（有限） | 队列满时发送方**必须等** |
| **无界容量** | 无限 | 发送方**永远不等** |

### 2.2 图示理解

```
零容量（0）：
发送方 → [无缓冲区] → 接收方
          ↑
       不能暂存，必须当面交接

有界容量（n）：
发送方 → [队列：最多n条] → 接收方
          ↑
       满了就不能再放

无界容量（∞）：
发送方 → [无限队列] → 接收方
          ↑
       永远有地方，总能往里放
```

### 2.3 三种缓冲和同步的关系

| 缓冲 | send 自动变成 | 说明 |
|------|--------------|------|
| 零容量 | **阻塞 send** | 没地方存，必须等对方收 |
| 有界容量 | 可能阻塞 | 只有满的时候才阻塞 |
| 无界容量 | **非阻塞 send** | 永远有地方，不用等 |

## 三、帮你把两个概念串起来

```
IPC 通信的两大属性：

1. 同步（谁等谁）
   ├── send：阻塞 / 非阻塞
   └── receive：阻塞 / 非阻塞

2. 缓冲（消息存哪里）
   ├── 零容量（0）
   ├── 有界容量（n）
   └── 无界容量（∞）

两者互相影响：
   - 零容量 → send 必须是阻塞的
   - 无界容量 → send 可以是非阻塞的
```

## 四、考试常考对比表

| 场景 | send 行为 | receive 行为 | 缓冲类型 |
|------|----------|-------------|----------|
| 快递柜 | 非阻塞（放下就走） | 阻塞（没件就等） | 有界容量 |
| 面对面交接 | 阻塞（必须给到手上） | 阻塞 | 零容量 |
| 发送大量日志 | 非阻塞（丢了也无所谓） | 非阻塞（有空才处理） | 无界容量 |

## 五、小测验

1. 零容量缓冲区下，send 一定是阻塞还是非阻塞？
2. 最常用的 send/receive 组合是什么？
3. 有界容量缓冲区在什么情况下 send 会阻塞？

> 答案：  
> 1. 阻塞（没地方存，必须等对方收）  
> 2. 非阻塞 send + 阻塞 receive  
> 3. 队列满的时候
