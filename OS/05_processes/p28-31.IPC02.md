---

## 一、直接通信（第 1 张图）

### 1.1 核心定义

> **直接通信 = 发消息时直接点名道姓**

```c
send(P, message);    // 发给进程 P
receive(Q, message); // 从进程 Q 收
```

### 1.2 三个性质（考试常考）

| 性质 | 含义 |
|------|------|
| 链接自动建立 | 不需要提前创建通道，第一次通信时自动生成 |
| 一对一 | 一条链接只连接两个进程 |
| 唯一性 | 每对进程之间只有一条链接 |

### 1.3 非对称寻址（Asymmetry in addressing）

课件里特别提了这个概念：

| 模式 | send | receive |
|------|------|---------|
| **对称** | send(P, msg) | receive(Q, msg) |
| **非对称** | send(P, msg) | receive(id, msg) ← id 自动填成发送者的名字 |

> 非对称的好处：接收方不需要提前知道发送方是谁，操作系统帮你记。

**例子：**
- 服务器调用 `receive(id, msg)`，不管哪个客户端发来，id 自动设为客户端 PID
- 服务器就知道“是客户端 1234 发来的”

---

## 二、间接通信（第 2 张图）

### 2.1 核心定义

> **间接通信 = 往信箱里放/取，不直接指定对方**

```c
send(A, message);    // 往信箱 A 发
receive(A, message); // 从信箱 A 收
```

### 2.2 类比

| 概念 | 类比 |
|------|------|
| 直接通信 | 面对面说：张三，给你100块钱 |
| 间接通信 | 放到快递柜（信箱 A），李四自己去取 |

### 2.3 间接通信的性质（第 4 张图）

| 性质 | 含义 |
|------|------|
| 需要共享信箱 | 没有共享信箱就不能通信 |
| 多对多 | 一个信箱可以被多个进程共享 |
| 多链接 | 一对进程之间可以有多个信箱（多条链接） |
| 信箱可属于进程或 OS | 进程创建的信箱随进程消失；OS 管理的信箱持久存在 |

## 三、经典问题：三个人共享一个信箱（第 3 张图）

**场景：**
- P₁ 发送
- P₂ 和 P₃ 都在 `receive(A, msg)`
- 消息给谁？

### 三种解决方案

| 方案 | 解释 |
|------|------|
| **方案 1** | 一个信箱最多链接 2 个进程（强制一对一，P₂ 和 P₃ 不能同时等） |
| **方案 2** | 一次只允许一个进程执行 receive（互斥，谁先拿到锁谁收） |
| **方案 3** | 系统随机选一个接收者，并通知发送者被谁收了 |

> 💡 实际系统中常用方案 3 或方案 2 的变种。

---

## 四、直接 vs 间接对比总结

| 对比维度 | 直接通信 | 间接通信 |
|----------|----------|----------|
| 指定方式 | 指名道姓（进程 ID） | 投递到信箱 |
| 链接数量 | 每对进程一条 | 可有多个 |
| 链接关联的进程数 | 正好 2 个 | 可以是多个 |
| 灵活性 | 低 | 高 |
| 管理复杂度 | 低 | 高（需要管理信箱） |
| 典型应用 | 简单点对点 | 消息队列、端口 |

---

## 五、帮你总结一页纸

| 概念 | 一句话 |
|------|--------|
| 直接通信 | send 和 receive 直接写对方 PID |
| 间接通信 | 通过信箱（mailbox/port）中转 |
| 对称寻址 | receive(Q, msg)，知道从谁收 |
| 非对称寻址 | receive(id, msg)，系统填 id |
| 信箱 | 间接通信的中转站 |
| 三进程共享信箱 | 需要特殊机制决定谁收 |

## 六、小测验

1. 直接通信中，send(P, msg) 和 receive(Q, msg)，Q 必须是谁？
2. 非对称寻址的好处是什么？
3. 三个进程共享一个信箱，可能有什么问题？

> 答案：  
> 1. Q 必须是发送方 P（因为链接是一对一的）  
> 2. 接收方不需要提前知道发送方是谁  
> 3. 不知道谁收到消息，可能造成混乱
