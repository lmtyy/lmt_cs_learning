### 第1张图：文件结构（File Structure）

这里讲的是文件内部数据是如何组织的，操作系统如何看待文件的形状：
1.  **None（无结构 / 字节流）**：
    *   这是**现代操作系统（如 Unix/Linux）** 采用的模式。
    *   操作系统把文件视为**一堆连续的字节**。它不知道也不关心里面是文本还是代码。由**应用程序**自己决定怎么解释这些字节（比如 Word 知道怎么读出段落）。**这是考点：操作系统越简单，灵活性越高。**
2.  **Simple record structure（简单记录结构）**：
    *   比如由定长记录或行组成的文件。早期系统（如卡片机、磁带机）常用。
3.  **Complex Structures（复杂结构）**：
    *   比如可重定位加载文件（编译后的目标文件，有特定头部和节区）。
4.  **Who decides?**：
    *   **操作系统**决定：文件是不是字节流。
    *   **应用程序/程序员**决定：字节流里的结构怎么解析。
5.  **结论**：PPT 最后一行总结——如果操作系统支持太多结构，会让系统变得臃肿（Bloat）；如果太少（比如全是字节流），程序员就得自己写代码解析，不方便。

> **知识点**：现代操作系统大多选择“字节流”结构。这是**Unix/Linux 哲学**：保持内核简单。

### 第2张图：文件操作（基础）

列出了基本的文件操作及其**系统调用内部发生的事**：
*   **Create**：分配磁盘空间，并分配目录项。
*   **Read/Write/Seek**：读/写/跳到文件特定位置。注意，这里的“Seek”只是改变文件指针的位置，不涉及磁盘读写操作。
*   **Delete**：释放文件占用的所有空间，并从目录中删除该项。
*   **Truncate**：如果不想要文件内容但想保留文件属性（比如清空日志文件），可以用这个操作。释放文件空间但不删除目录项。
*   **Open(F)**：这是一个非常关键的操作。操作系统会去磁盘上寻找目录项，复制到内存的**打开文件表（open-file table）**，并分配**文件描述符（file descriptor）**。
*   **Close(F)**：更新磁盘上的目录信息，并释放内存中的描述符。

### 第3-5张图：文件表机制（核心难点）

这组幻灯片解释了操作系统如何在内存中管理**被打开的文件**，防止每次读写都去慢吞吞的磁盘上搜索目录。

#### 单用户环境（第3张）
*   **问题**：如果每次 Read 都要去搜一遍目录找文件在哪，速度会极慢。
*   **方案**：使用 `open` 系统调用把文件信息拉到内存的 **打开文件表（Open-file table）** 中。后续 `read` 时，只需用**索引（index / file descriptor）**来定位表里的项，不需要再搜磁盘。

#### 多用户环境（第4-5张）- **高频考点**
在多用户系统（如 Unix/Linux）中，多个进程可以同时打开同一个文件。系统使用**两级表**来管理：

1.  **Per-process open-file table（进程级打开文件表）**
    *   **在哪里**：每个进程（如你的浏览器、文本编辑器）都有自己的此表。
    *   **存什么**：这个进程独享的信息。
        *   **File pointer**：当前读到了文件哪个位置（比如进程A读到第100行，进程B读到第0行，各不影响）。
        *   **Access rights**：这个进程对文件的访问权限（比如只读、读写）。
    *   **如何指向**：表里每一条目指向**系统级表**的对应项。

2.  **System-wide open-file table（系统级打开文件表）**
    *   **在哪里**：整个操作系统内核只有一张。
    *   **存什么**：文件本身的不变信息。
        *   **Disk location**：文件在硬盘上的物理地址。
        *   **File size**：文件大小。
        *   **Open count**：有多少个进程正在打开这个文件（关键！用于判断何时真的可以关闭）。
    *   **作用**：当第一个进程打开文件时，系统级表建立信息。后续进程再打开同一个文件，只需在进程级表里新增一个指针，指向系统级表里已有的那个项，**避免重复加载磁盘信息**。

3.  **Open count（打开计数）**：
    *   当 `Open count = 1` 时（最后一个进程关闭文件），系统才会真正把文件信息从内存系统级表中移除，更新磁盘元数据。

### 🔴 第6张图：核心图表解析（必考！）

这张图完美展示了上述的两级表机制（来自教材原图）。

**【图(a) - Open 阶段】**
1.  **User space**：程序发出 `open(filename)`。
2.  **Kernel memory**：系统去**directory structure**（目录结构）寻找文件名。
3.  一旦找到，它将**file control block（FCB，即文件控制块，存储文件属性）** 从磁盘复制到内存。
4.  系统级表（System-wide table）里新增一项，指向这个 FCB。
5.  进程级表（Per-process table）里新增一项，指向系统级表的那一项。
6.  返回给用户程序一个整数索引（Index/文件描述符）。

**【图(b) - Read 阶段】**
1.  **User space**：程序发出 `read(index)`。
2.  **Kernel memory**：直接通过**index**索引到进程级表。
3.  进程级表顺着指针，找到系统级表。
4.  系统级表找到**Data blocks**（数据块）和**File Control Block**。
5.  操作系统根据**File pointer**（进程级表中的值）找到对应的数据块，读回用户空间。

### 💡 为什么这套机制重要？考试怎么考？
*   **简答题**：请描述当两个进程同时打开同一个文件进行读取时，操作系统内部是如何工作的？（答案：两层表 + Open count）
*   **选择题**：当前文件指针（File pointer）存储在哪里？
    *   ❌ 正确答案：存储在每个进程的**Per-process open-file table**里。因为两个进程可以读同一个文件的不同位置。
*   **选择题**：文件的物理位置（Disk location）存储在哪里？
    *   ✅ 正确答案：存储在**System-wide open-file table**（或指向的FCB）。因为文件不管谁读，它在硬盘上的位置是不变的。
