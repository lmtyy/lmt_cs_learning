### 第二张图：文件的定义与内容
*   **What is a file?**
    *   **定义**：文件是**命名的**、**相关信息的集合**，它被记录在**二级存储（Secondary Storage，如硬盘、SSD）**上。
    *   *理解重点*：必须有个名字（给用户看的），必须是个合集（不是孤立的零散数据），必须能持久化存储（断电后还在）。
*   **Content types**：
    *   **Data**：可以是数字、字母、二进制（如图片、音频）。
    *   **Program**：可以是源代码（人读的）或 目标代码/可执行文件（机器读的）。

### 第三张图：文件的属性（Attributes）- ⭐非常重要
这是操作系统管理文件的“户口本”。每个文件除了数据内容外，系统还维护了这些**元数据**：
1.  **Name**：用户可读的名字，比如 `report.docx`。
2.  **Identifier**：系统内部的唯一标识号（通常叫 inode 号或文件句柄编号），操作系统通过它而不是名字去找文件。
3.  **Type**：文件类型，支持不同类型系统的识别。
4.  **Location**：在硬盘/SSD上的物理位置（存放数据的地址）。
5.  **Size**：当前大小（或最大允许大小）。
6.  **Protection**：权限控制，决定谁能读、写、执行（rwx）。
7.  **Time, date, user identification**：创建时间、最后修改时间、最后访问时间、创建者是谁。

⚠️ **结尾一段话非常关键（Bottom Text）**：
> “典型情况下，目录项（directory entry）由文件名及其唯一标识符组成。标识符进而定位其他的文件属性。其中一个文件属性就是文件的位置。”

**通俗解释**：
目录里只存 **[文件名 -> 编号]**。系统拿着编号去另一个地方（比如 Inode 表）查找文件的大小、位置、权限。这样可以灵活管理，比如重命名文件只需要改目录里的名字，不需要动底层的属性表。

### 第四张图：文件类型的识别机制
这里讲的是操作系统怎么知道这个文件是图片、是文本、还是可执行程序？
*   **原则**：如果 OS 能识别文件类型，就能用正确的方式去操作它（比如点击一个 `.txt` 打开记事本，而不是运行它）。
*   **不同的OS处理方式不同**：
    *   **Macintosh OS**：文件内部存储了类型（Type）和 创建者（Creator）。系统直接看文件头。
    *   **Linux/Unix**：文件类型**只是提示（Hints）**。Linux 通常不依赖后缀名，而是看文件内容（通过 magic number）或者看文件的权限位。但为了方便人类，依然用 `.txt`, `.c` 等后缀。

### 💡 课后考点提炼：
1.  **文件 vs. 目录**：文件是存储信息的容器；目录是存储文件“名字到地址映射”的容器。
2.  **属性 vs. 内容**：属性是**关于**文件的数据；内容是文件**真正**的数据。打开文件时先读属性，确认权限和位置，再读内容。
3.  **后缀名的作用**：在 Windows 中决定怎么打开文件；在 Linux 中只是给人看的，对系统是暗示。
4.  **Identifier**：面试或考试常问“如果我要在一个几万文件的目录里找一个文件，为什么系统不先读文件名再找位置？”——因为系统找的是 Identifier（inode），不是通过文件名逐个比对（存储结构是哈希或树，文件名只是人类的索引）。

如果第五张图中表格的具体后缀名需要硬背，可以只记 **Executable (.exe)**, **Source (.c)**, **Object (.o)**, **Library (.dll/.so)**，这几个在后续章节（内存管理、编译原理）经常出现。
