这组PPT深入讲解了**分段系统在硬件层面是如何实现的**，包括具体的硬件寄存器、地址翻译流程、具体的计算示例，以及段的共享。

这部分和之前学的“分页硬件实现”逻辑非常相似，但关键在于**分段引入了“越界检查”机制**（这也是分段相比分页在安全性上的巨大优势）。

---

### 1. 分段系统的硬件支持 (图1)

要实现分段，硬件需要做两件事：记录段的**位置**和记录段的**大小**。

#### **段表 (Segment Table)**
操作系统维护一张段表，每个条目包含两个关键信息：
1.  **段基址 (Base)**：该段在物理内存中的**起始地址**。
2.  **段界限 (Limit)**：该段的**长度**（大小）。

#### **关键寄存器**
*   **STBR (Segment-Table Base Register)**：段表基址寄存器。指向段表在内存中的起始位置。
*   **STLR (Segment-Table Length Register)**：段表长度寄存器。指示当前进程的段表有多长（即包含几个段）。

### 2. 地址翻译与越界保护 (图2 & 图3)

这是分段系统最精彩的部分。它不仅完成地址翻译，还实现了**硬件级别的内存保护**。

#### **地址结构** (图2)
CPU 发出的逻辑地址是：`<s, d>`
*   **s**：段号 (Segment Number)
*   **d**：段内偏移 (Offset)

#### **完整的翻译流程** (参考图3流程图)
1.  **合法性检查**：
    *   检查段号 `s` 是否小于 `STLR`（段表长度）。如果 `s >= STLR`，说明访问了不存在的段，抛出错误。
2.  **查表**：
    *   用 `s` 作为索引，在段表中查找该段对应的 `base`（基址）和 `limit`（界限）。
3.  **越界检查**：
    *   检查偏移量 `d` 是否小于 `limit`。
    *   如果 `d >= limit`，说明试图访问该段之外的内存（比如数组越界），触发**异常/陷阱 (Trap)**。这是极其重要的安全机制！
4.  **计算物理地址**：
    *   如果检查通过：`物理地址 = Base + d`。
    *   直接送入内存单元。

**核心区别**：分页系统因为页大小固定，没有这种“偏移量是否超出边界”的检查，而分段系统有。因此**分段天然支持内存保护**。

### 3. 具体计算实例 (图4 & 图5)

这两张图是一组的。图4定义了所有数据，图5展示了计算结果。

#### **前提条件 (图4)**
*   一个进程有 5 个段 (0~4)。
*   **段表内容**：
    *   段0: limit=1000, base=1400
    *   段1: limit=400, base=6300
    *   段2: limit=400, base=4300
    *   段3: limit=1100, base=3200
    *   段4: limit=1000, base=4700

#### **计算演示 (图5)**
**例1：访问 Segment 1, Byte 1222**
*   `s=1` (在段表范围内)。
*   `d=1222`。查表知 segment 1 的 `limit=400`。
*   **检查**：`1222 < 400`？不成立！
*   **结果**：**`invalid` (非法访问)**。触发越界保护。

**例2：访问 Segment 2, Byte 53**
*   `s=2` (合法)。
*   `d=53`。`limit=400`。
*   **检查**：`53 < 400` (成立)。
*   **计算**：物理地址 = `base(4300) + d(53) = 4353`。

**例3：访问 Segment 3, Byte 852**
*   `s=3` (合法)。
*   `d=852`。`limit=1100`。
*   **检查**：`852 < 1100` (成立)。
*   **计算**：物理地址 = `base(3200) + d(852) = 4052`。

### 4. 段的共享 (图6)

和分页一样，分段也支持共享。这张图展示了两个进程共享编辑器代码的场景。

*   **进程 P1 和 P2** 都在逻辑上需要 `editor` 段 (段0)。
*   **段表映射**：
    *   P1 的 `editor` 段：`base=43062`。
    *   P2 的 `editor` 段：`base=43062`。
*   **结论**：两个进程的段表项里，`editor` 段的 **base 指向了物理内存的同一个位置**。
*   **数据段私有**：P1 的 `data 1` 和 P2 的 `data 2` 则指向了物理内存中不同的、各自独立的位置。

### 💡 总结：分段 vs 分页实现对比

| 特性 | 分页 (Paging) | 分段 (Segmentation) |
| :--- | :--- | :--- |
| 地址结构 | `<页码, 页偏移>` | `<段号, 段内偏移>` |
| 单元大小 | **固定** (如4KB) | **可变** (由程序逻辑决定) |
| 查表内容 | 查 `<帧号>` | 查 `<基址, 界限>` |
| 越界检查 | 无 (通常不需要) | **有** (硬件级内存保护) |
| 外部碎片 | **没有** | **有** (同连续分配) |
| 内存利用率 | 高 | 较低 |

**一句话记忆**：分段用**基址+界限**实现了**地址翻译+内存保护**，但牺牲了物理内存的连续性，留下了外部碎片的隐患。
