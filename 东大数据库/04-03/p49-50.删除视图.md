这节课讲的是**删除视图（DROP VIEW）**，包括基本语法、级联删除、以及删除基表对视图的影响。内容不多但有几个重要考点，我帮你梳理清楚。

---

## 一、基本语法

```sql
DROP VIEW <视图名>;
```

### 作用
- 从**数据字典**中删除指定的视图定义
- **不删除任何基表数据**（视图本身不存储数据）

### 示例1
```sql
DROP VIEW BT_S;
```
- 删除之前创建的 `BT_S` 视图
- 对 `Student` 基表没有任何影响

---

## 二、视图依赖问题（重要！）

### 场景
假设有视图依赖关系：
```
sf_S1（基础视图）
   ↓
sf_S2（基于 sf_S1 创建的视图）
```

### 示例2：尝试删除被依赖的视图

```sql
DROP VIEW sf_S1;
```

### 结果
❗ **执行被拒绝**

**原因**：`sf_S1` 视图上还导出了 `sf_S2` 视图。如果直接删除 `sf_S1`，`sf_S2` 会变成**无效视图**（因为它依赖的视图不存在了）。

---

## 三、级联删除（CASCADE）

### 语法
```sql
DROP VIEW sf_S1 CASCADE;
```

### 效果
- 删除 `sf_S1` 视图
- **同时删除**由 `sf_S1` 导出的所有视图（即 `sf_S2`）

### 执行前后对比

| 操作 | 执行前 | 执行后 |
|------|--------|--------|
| `DROP VIEW sf_S1;` | sf_S1, sf_S2 都存在 | ❌ 拒绝执行 |
| `DROP VIEW sf_S1 CASCADE;` | sf_S1, sf_S2 都存在 | ✅ sf_S1 和 sf_S2 都被删除 |

---

## 四、替代方案（非级联）

课件也提到了另一种方式：

```sql
-- 先删除依赖的视图
DROP VIEW sf_S2;

-- 再删除基础视图
DROP VIEW sf_S1;
```

**优点**：更可控，不会误删其他视图

---

## 五、删除基表对视图的影响（课件未完整，我补充）

### 问题
如果删除基表（例如 `DROP TABLE Student`），由该基表导出的视图会怎样？

### 答案
| 数据库 | 行为 |
|--------|------|
| 大多数 DBMS | 视图变成**无效视图**（仍存在于数据字典，但查询会报错） |
| 部分 DBMS（如 Oracle） | 可以选择保留视图定义，但查询时报错 |

### 处理方式
- 手动删除这些无效视图：`DROP VIEW ...`
- 或者用级联删除基表（如果支持）：`DROP TABLE Student CASCADE;`

---

## 六、知识点总结

| 知识点 | 说明 |
|--------|------|
| `DROP VIEW` 作用 | 只删除视图定义，不删基表数据 |
| 有依赖视图时 | 普通删除被拒绝 |
| `CASCADE` | 级联删除本视图及其导出的所有视图 |
| 替代方案 | 先删子视图，再删父视图 |
| 删除基表 | 相关视图会变成无效视图 |

---

## 七、速查表

```sql
-- 删除视图（无依赖）
DROP VIEW view_name;

-- 删除视图（有依赖，级联删除）
DROP VIEW view_name CASCADE;

-- 先删子视图再删父视图
DROP VIEW child_view;
DROP VIEW parent_view;
```

## 八、常见考题示例

**题目**：假设有视图 `V1`，基于 `V1` 创建了 `V2`，基于 `V2` 创建了 `V3`。执行 `DROP VIEW V2 CASCADE;` 后，哪些视图还存在？

**答案**：
- `V1` 还存在（因为它不依赖 V2）
- `V2` 和 `V3` 都被删除

**理由**：`CASCADE` 会删除目标视图及其**所有后代视图**，但不影响它的祖先视图。
