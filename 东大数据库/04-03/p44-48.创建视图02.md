这节课讲的是**创建视图的各种类型**，包括：基于多基表、基于视图、带表达式、分组视图，以及一个**重要避坑指南**（不要用 `SELECT *` 创建视图）。我来帮你系统整理。

## 一、基于多个基表的视图（例3）

### 业务需求
> 建立**软件学院**选修了**1号课程**的学生视图

### SQL 语句

```sql
CREATE VIEW sf_S1 (Sno, Sname, Grade)
AS
SELECT Student.Sno, Sname, Grade
FROM Student, SC
WHERE Student.Sno = SC.Sno
  AND Sdept = '软件学院'
  AND SC.Cno = '1';
```

### 涉及的表

| 表名 | 作用 | 使用的字段 |
|------|------|-----------|
| `Student` | 学生信息 | Sno, Sname, Sdept |
| `SC` | 选课成绩 | Sno, Cno, Grade |

### 视图结果
- 只包含：学号、姓名、成绩
- 条件筛选：软件学院 + 课程1号

### 关键点
- 视图可以**连接多个基表**
- 视图的列名可以**重新指定**（这里指定了 Sno, Sname, Grade）
- 查询视图时会动态执行这个多表连接

## 二、基于视图的视图（例4）

### 业务需求
> 建立**软件学院选修了1号课程且成绩≥90分**的学生视图

### SQL 语句

```sql
CREATE VIEW sf_S2
AS
SELECT Sno, Sname, Grade
FROM sf_S1
WHERE Grade >= 90;
```

### 视图的层次结构

```
Student表 + SC表
       ↓
   sf_S1视图（软件学院 + 课程1）
       ↓
   sf_S2视图（再加成绩≥90）
```

### 特点
- 视图可以**建立在其它的视图之上**
- 形成视图的**层次结构**
- 查询 `sf_S2` 时，会层层展开，最终访问基表

### 优点
- **模块化**：先定义基础视图，再在其上定义更细的视图
- **复用**：多个上层视图可以共用同一个底层视图

## 三、带表达式（虚拟列）的视图（例5）

### 业务需求
> 定义一个反映学生**出生年份**的视图

### SQL 语句

```sql
CREATE VIEW BT_S (Sno, Sname, Sbirth)
AS
SELECT Sno, Sname, 2012 - Sage
FROM Student;
```

### 虚拟列（派生列）

| 概念 | 说明 |
|------|------|
| **虚拟列** | 视图中由表达式计算得出的列 |
| **是否存储** | 不存储在数据库中，查询时动态计算 |
| **基表中有吗** | 没有对应的物理列 |
| **好处** | 减少数据冗余，避免存储派生数据 |

### 例子说明
- 假设 `Student` 表中有 `Sage`（年龄）列
- 视图通过 `2012 - Sage` 计算出 `Sbirth`（出生年份）
- 如果换成今年是 2026 年，应该写成 `2026 - Sage`

## 四、分组视图（例6）

### 业务需求
> 将学生的**学号**及他的**平均成绩**定义为一个视图

### SQL 语句

```sql
CREATE VIEW S_G (Sno, Gavg)
AS
SELECT Sno, AVG(Grade)
FROM SC
GROUP BY Sno;
```

### 特点

| 特点 | 说明 |
|------|------|
| 包含聚集函数 | `AVG(Grade)` |
| 包含 GROUP BY | 按学号分组 |
| 必须指定列名 | 因为 `AVG(Grade)` 没有名字 |

### 视图用途
- 快速查询每个学生的平均成绩
- 避免重复写分组查询

### ⚠️ 重要限制
这种**分组视图通常是只读的**，不能进行 INSERT/UPDATE/DELETE 操作，因为：
- 无法反向映射回基表的哪一行
- 聚集函数无法反向计算

## 五、⚠️ 重点：避免用 `SELECT *` 创建视图（例7）

### 业务需求
> 将 Student 表中所有**女生**记录定义为一个视图

### ❌ 不好的写法

```sql
CREATE VIEW F_Student (F_Sno, name, sex, age, dept) AS
SELECT *
FROM Student
WHERE Ssex = '女';
```

### 缺点

> **可扩充性差**：修改基表 Student 的结构后，视图与基表的映像关系被破坏，导致视图不能正确工作。

### 问题演示

| 步骤 | 操作 | 结果 |
|------|------|------|
| 1 | 创建视图 `SELECT *` | 视图记住了当时 Student 表的所有列 |
| 2 | 向 Student 表增加一列（如 `phone`） | 基表变了 |
| 3 | 查询视图 | ❌ 可能报错或列不匹配 |

### ✅ 推荐做法

```sql
-- 明确列出需要的列名
CREATE VIEW F_Student AS
SELECT Sno, Sname, Ssex, Sage, Sdept
FROM Student
WHERE Ssex = '女';
```

### 原因总结

| 写法 | 优点 | 缺点 |
|------|------|------|
| `SELECT *` | 写起来快 | 基表结构变化时视图容易出问题 |
| 明确列名 | 稳定、可控 | 稍微多打几个字 |

## 六、五种视图类型汇总

| 类型 | 例子 | 能否更新 | 是否需要指定列名 |
|------|------|---------|-----------------|
| 单基表视图 | `sf_stu` | ✅ 有限制 | 不一定 |
| 多基表视图 | `sf_S1` | ❌ 通常不能 | 建议指定 |
| 视图上的视图 | `sf_S2` | ❌ 取决于底层 | 不需要 |
| 带表达式视图 | `BT_S` | ❌ 不能 | ✅ 必须指定 |
| 分组视图 | `S_G` | ❌ 不能 | ✅ 必须指定 |

## 七、知识点速查

```sql
-- 1. 多基表视图
CREATE VIEW view1 AS
SELECT ... FROM table1, table2 WHERE ...;

-- 2. 视图上的视图
CREATE VIEW view2 AS
SELECT ... FROM view1 WHERE ...;

-- 3. 带表达式的视图
CREATE VIEW view3 (col1, col2, virtual_col) AS
SELECT a, b, expression FROM table;

-- 4. 分组视图
CREATE VIEW view4 (group_col, agg_val) AS
SELECT col, AGG(func) FROM table GROUP BY col;

-- 5. 正确的视图创建（明确列名）
CREATE VIEW view5 AS
SELECT col1, col2, col3 FROM table WHERE condition;
```
