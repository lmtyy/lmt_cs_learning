这节课讲的是**更新视图**，包括基本规则、`WITH CHECK OPTION` 的作用、以及**不可更新视图**的限制。我帮你系统梳理。

---

## 一、核心观点

> **从用户角度：更新视图与更新基本表相同**

用户可以对视图执行 `INSERT`、`UPDATE`、`DELETE` 操作，RDBMS 会通过**视图消解法**将其转换为对基表的操作。

---

## 二、基本更新操作示例

### 例12：UPDATE（更新）

**用户查询：**
```sql
UPDATE sf_stu
SET Sname = '刘辰'
WHERE Sno = '200215122';
```

**视图定义（回顾）：**
```sql
CREATE VIEW sf_stu AS
SELECT Sno, Sname, Sage
FROM Student
WHERE Sdept = '软件学院';
```

**视图消解后的语句：**
```sql
UPDATE Student
SET Sname = '刘辰'
WHERE Sno = '200215122' 
  AND Sdept = '软件学院';
```

**关键点**：系统自动在 `WHERE` 条件中**加上了视图定义条件** `Sdept='软件学院'`

---

### 例13：INSERT（插入）

**用户查询：**
```sql
INSERT INTO sf_stu
VALUES('200215129', '赵新', 20);
```

**视图消解后的语句：**
```sql
INSERT INTO Student (Sno, Sname, Sage, Sdept)
VALUES('200215129', '赵新', 20, '软件学院');
```

**关键点**：系统自动给**未指定的视图列**（`Sdept`）补上视图定义中的值 `'软件学院'`

---

### 例14：DELETE（删除）

**用户查询：**
```sql
DELETE FROM sf_stu
WHERE Sno = '200215129';
```

**视图消解后的语句：**
```sql
DELETE FROM Student
WHERE Sno = '200215129' 
  AND Sdept = '软件学院';
```

**关键点**：同样自动加上视图定义条件

---

## 三、WITH CHECK OPTION 的作用

### 使用 WITH CHECK OPTION 创建视图

```sql
CREATE VIEW sf_stu AS
SELECT Sno, Sname, Sage
FROM Student
WHERE Sdept = '软件学院'
WITH CHECK OPTION;
```

### 区别对比

| 操作 | 不使用 WITH CHECK OPTION | 使用 WITH CHECK OPTION |
|------|------------------------|----------------------|
| **插入**软件学院学生 | ✅ 成功 | ✅ 成功 |
| **插入**信息学院学生 | ✅ 成功（但视图中看不到） | ❌ **拒绝** |
| **更新**学生学院从软件→信息 | ✅ 成功（但视图中消失） | ❌ **拒绝** |
| **删除**软件学院学生 | ✅ 成功 | ✅ 成功 |

### 思考题答案

> **如果通过软件学院学生视图插入信息学院学生信息，是否成功？**

| WITH CHECK OPTION | 结果 |
|-------------------|------|
| 无 | ✅ 插入成功（但视图中看不到这条记录） |
| 有 | ❌ 插入失败，报错 |

## 四、不可更新视图（重要！）

### 什么视图不可更新？

> 当视图的更新**不能唯一地、有意义地转换成对基本表的更新**时，该视图不可更新。

### 典型例子：分组视图

```sql
CREATE VIEW S_G (Sno, Gavg)
AS
SELECT Sno, AVG(Grade)
FROM SC
GROUP BY Sno;
```

### 为什么不可更新？

用户尝试执行：
```sql
UPDATE S_G
SET Gavg = 90
WHERE Sno = '200215121';
```

**问题**：无法转换成对 `SC` 表的更新，因为：
1. `Gavg` 是 `AVG(Grade)` 的计算结果，不是基本表的列
2. 一个 `Sno` 对应 `SC` 中的**多条记录**（多门课）
3. 无法确定要把哪门课的成绩改成多少，才能让平均分变成 90

---

## 五、常见不可更新视图类型

| 视图类型 | 原因 | 举例 |
|----------|------|------|
| **包含聚集函数** | 无法反向计算 | `AVG`, `SUM`, `COUNT`, `MAX`, `MIN` |
| **包含 GROUP BY** | 一对多映射 | 分组后的统计结果 |
| **包含 DISTINCT** | 无法确定原始行 | `SELECT DISTINCT ...` |
| **多表连接（某些情况下）** | 无法确定更新哪个表 | 除非使用 INSTEAD OF 触发器 |
| **包含表达式/虚拟列** | 无法反向计算 | `balance * 1.05` |

---

## 六、可更新视图的条件（大致标准）

| 条件 | 说明 |
|------|------|
| 基于单个基表 | 不是多表连接 |
| 不包含聚集函数 | 无 `SUM`, `AVG` 等 |
| 不包含 GROUP BY | 无分组 |
| 不包含 DISTINCT | — |
| 不包含表达式列 | 可以用简单列 |
| 视图定义中包含基表的**主键** | 推荐 |

---

## 七、知识点总结

| 操作 | 视图消解后的处理 |
|------|-----------------|
| `UPDATE` | `WHERE` 条件中**加视图定义条件** |
| `INSERT` | **自动补全**视图定义中隐含的列 |
| `DELETE` | `WHERE` 条件中**加视图定义条件** |
| `WITH CHECK OPTION` | 检查更新后的行**是否仍满足视图条件** |
| 分组视图 | **不可更新**，因为无法唯一映射到基表 |

---

## 八、速查表

```sql
-- 可更新视图（单表、无聚合）
CREATE VIEW good_view AS
SELECT col1, col2 FROM table WHERE condition;

-- 不可更新视图（有聚合）
CREATE VIEW bad_view AS
SELECT col, AVG(col2) FROM table GROUP BY col;

-- 带检查选项
CREATE VIEW safe_view AS
SELECT col1, col2 FROM table WHERE condition
WITH CHECK OPTION;
```
