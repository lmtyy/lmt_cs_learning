## 一、触发器的引入背景

在数据库设计中，我们通常通过 `CREATE TABLE` 语句定义**静态约束**，例如：
- 列约束（`NOT NULL`、`UNIQUE`、`CHECK`）
- 表约束（主键、外键、唯一约束）

这些约束大多是**非过程性**的，即它们只是声明了数据必须满足的条件，但不包含如何处理复杂业务逻辑的步骤。而且它们主要针对**单一列或单一元组**（虽然外键涉及多表，但也是声明式的）。

然而，现实中的业务往往需要**动态的、跨表的完整性规则**，例如：
- 当更新某个表时，自动更新相关联的另一个表。
- 当删除一条记录时，自动将相关数据归档到历史表。
- 当插入数据时，自动检查某些条件并可能拒绝操作或级联修改。

这些需求无法通过简单的表约束实现，于是引入了**触发器（Trigger）**——一种**过程性完整性约束**，它是一段自动执行的程序，能够响应特定的数据修改事件。

---

## 二、触发器的定义与特点

- **触发器是一种特殊的存储过程**，但与普通存储过程不同，它**不需要手动调用**，而是在特定事件发生时由数据库自动触发执行。
- 触发器具备**事务的功能**，即它通常与触发它的操作在同一个事务中执行，要么全部成功，要么全部回滚。
- 触发器可以跨越多个表，实现复杂的业务规则，确保数据的一致性和完整性。

---

## 三、为什么需要触发器——典型应用场景

### 1. **银行取款机系统**
假设有一个银行账户表 `bank`：
| customerName | cardID   | currentMoney |
| ------------ | -------- | ------------ |
| 张三         | 10010001 | 1000.0000    |
| 李四         | 10010002 | 1.0000       |

当张三在ATM上取款200元时，需要执行以下操作：
- 检查账户余额是否足够（`currentMoney >= 200`）。
- 如果足够，更新 `currentMoney = currentMoney - 200`。
- 同时可能需要记录交易日志（插入一条交易记录到 `transaction_log` 表）。
- 还可能更新其他相关表（如每日限额统计）。

如果不使用触发器，应用程序必须编写多条SQL语句并处理事务，这增加了网络交互和代码复杂度。而使用触发器，可以在对 `bank` 表执行 `UPDATE` 操作之前或之后，**自动**检查余额并记录日志，确保业务规则始终被遵守，且代码集中在数据库端，提高了安全性和性能。

### 2. **员工退休管理**
PPT中给出了一个图示：员工表中有张三、王三、李四、宋二、赵二、王五、刘五等。当赵二退休时，需要：
- 从员工表中**删除**赵二的记录。
- 同时将赵二的信息**插入**到退休员工表。

这个场景非常适合使用触发器：在 `员工表` 上定义一个 `DELETE` 触发器，当删除某员工时，自动将该员工的数据插入到 `退休员工表` 中。这样，应用程序只需执行一条 `DELETE` 语句，触发器就会自动完成归档工作，确保数据不丢失，并且整个过程是原子性的。

## 四、触发器的工作原理

触发器基于**事件-条件-动作**规则：
- **事件**：对特定表的 `INSERT`、`UPDATE`、`DELETE` 操作。
- **时机**：可以在事件之前（`BEFORE`）或之后（`AFTER`）触发。
- **粒度**：通常为**行级触发器**（`FOR EACH ROW`），即每影响一行就执行一次；也支持语句级触发器（但MySQL只支持行级）。
- **动作**：触发器体内可以包含复杂的SQL语句，并可以访问被修改行的**新旧数据**（通过 `OLD` 和 `NEW` 引用）。例如在 `UPDATE` 触发器中，`OLD` 表示修改前的行，`NEW` 表示修改后的行。

---

## 五、创建触发器的基本语法（以MySQL为例）

```sql
CREATE TRIGGER trigger_name
    {BEFORE | AFTER} {INSERT | UPDATE | DELETE} ON table_name
    FOR EACH ROW
    trigger_body;
```

- **trigger_name**：触发器名称。
- **触发时机**：`BEFORE` 或 `AFTER`。
- **触发事件**：`INSERT`、`UPDATE` 或 `DELETE`。
- **ON table_name**：触发器基于哪个表。
- **FOR EACH ROW**：行级触发器，每行受影响时触发一次。
- **trigger_body**：要执行的SQL语句块，通常以 `BEGIN ... END` 包围，可以包含多条语句，并可以使用 `OLD` 和 `NEW` 关键字。

### 示例：银行取款触发器（简化版）
```sql
DELIMITER //
CREATE TRIGGER trg_withdraw_check
    BEFORE UPDATE ON bank
    FOR EACH ROW
BEGIN
    -- 检查余额是否足够（假设只处理取款，即金额减少的情况）
    IF NEW.currentMoney < 0 THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '余额不足';
    END IF;
    
    -- 记录交易日志
    INSERT INTO transaction_log(cardID, old_balance, new_balance, trans_time)
    VALUES (OLD.cardID, OLD.currentMoney, NEW.currentMoney, NOW());
END //
DELIMITER ;
```
当执行 `UPDATE bank SET currentMoney = currentMoney - 200 WHERE cardID = '10010001';` 时，触发器自动执行上述检查并记录日志。

### 示例：员工退休触发器
```sql
DELIMITER //
CREATE TRIGGER trg_employee_retire
    AFTER DELETE ON employee
    FOR EACH ROW
BEGIN
    INSERT INTO retired_employee (emp_id, name, retire_date)
    VALUES (OLD.emp_id, OLD.name, CURDATE());
END //
DELIMITER ;
```
当从 `employee` 表删除一条记录时，被删除的行数据（通过 `OLD` 访问）自动插入到 `retired_employee` 表。

## 六、触发器的注意事项

- **性能影响**：触发器在每次数据修改时都会执行额外的操作，可能增加开销，需谨慎设计。
- **递归触发**：触发器内的操作可能再次触发同一触发器（例如在 `AFTER INSERT` 中又插入同一表），需要避免无限循环。多数数据库提供了递归触发限制或检测机制。
- **调试困难**：触发器隐式执行，出错时问题定位可能比显式代码困难。
- **可移植性**：不同数据库的触发器语法差异较大，迁移时需重写。

---

## 七、总结

- 触发器是一种**过程性完整性约束**，弥补了表约束的静态性和局限性。
- 它能够自动响应数据修改事件，实现复杂的跨表业务逻辑，保证数据一致性和完整性。
- 典型应用包括：审计日志、数据归档、级联更新、业务规则检查等。
- 创建触发器时需要指定触发时机、事件、表名以及要执行的动作，并利用 `OLD` 和 `NEW` 访问行数据。

通过银行取款和员工退休两个例子，可以直观地理解触发器如何简化应用程序开发，并将关键业务逻辑集中在数据库层，提高系统的可靠性和安全性。
