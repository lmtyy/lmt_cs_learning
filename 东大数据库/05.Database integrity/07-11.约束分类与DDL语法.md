## 第一页：完整性约束条件的类别（域完整性 vs 关系完整性）

### PPT原文
- **域完整性约束条件**  
  - 施加于某一列上，对给定列上所要更新的某一候选值是否可以接受进行约束条件判断，这是孤立进行的。
- **关系完整性约束条件**  
  - 施加于关系/table上，对给定table上所要更新的某一候选元组是否可以接受进行约束条件判断，或是对一个关系中的若干元组和另一个关系中的若干元组间的联系是否可以接受进行约束条件判断。

### 详细讲解

#### 1. 域完整性约束条件
- **对象**：单个列。
- **检查方式**：只考虑该列的新值是否满足条件，不涉及其他列，因此是“孤立进行”的。
- **常见形式**：
  - 数据类型本身（如 `INT`、`DATE`）—— 隐式约束。
  - `NOT NULL` —— 禁止空值。
  - `CHECK` 限定取值范围，如 `age > 0`。
  - `DEFAULT` 默认值（虽然不是约束，但有助于维护完整性）。
- **举例**：`salary DECIMAL(10,2) CHECK (salary >= 0)`，试图插入负工资会被拒绝。

#### 2. 关系完整性约束条件
- **对象**：整个表（关系）或多个表。
- **检查方式**：可能涉及单行中的多列，也可能涉及多行甚至跨表。
- **分类**：
  - **表内关系约束**：同一表中多个列之间的约束，如 `CHECK (end_date > start_date)`。
  - **表间关系约束**：不同表之间的联系，如外键约束 `FOREIGN KEY`，它要求某个表的列值必须在另一表的主键中存在。
- **举例**：
  - 选课表 `SC` 中，学号 `S#` 必须存在于学生表 `Student` 中（参照完整性）。
  - 订单表中，下单日期不能大于发货日期（涉及两列）。

### 易错点与常考点
- **易错**：将 `NOT NULL` 误认为是关系完整性，其实它是域完整性。
- **常考**：给出一条约束，问属于哪种类型。如“员工的工资不能超过其经理的工资”——这涉及同一表内两行比较（员工与经理），属于关系完整性（表内多行）。
- **注意**：外键约束虽然定义在列上，但因为它参照其他表，所以属于关系完整性。

## 第二页：完整性约束条件的类别（按约束来源分类）

### PPT原文
- **约束来源分类**
  - **结构约束**：来自于模型的约束，例如函数依赖约束、主键约束(实体完整性)、外键约束(参照完整性)，只关心数值相等与否，是否允许空值等；
  - **内容约束**：来自于用户的约束，如用户自定义完整性、关心元组或属性的取值范围。例如Student表的Sage属性值在15岁至40岁之间等。

### 详细讲解

#### 1. 结构约束
- **来源**：关系数据模型本身的要求，与具体业务无关。
- **特点**：只关注**结构上的正确性**，如值是否唯一、是否为空、是否相等。
- **典型例子**：
  - 主键约束（唯一且非空）
  - 唯一约束（值唯一，可空）
  - 外键约束（值必须存在于参照表）
  - 函数依赖（通常通过主键或唯一约束来体现）

#### 2. 内容约束
- **来源**：用户业务规则（应用语义）。
- **特点**：关注**数据的具体内容**是否在允许范围内。
- **典型例子**：
  - 年龄在18~60岁之间
  - 性别只能为'男'或'女'
  - 工资不能超过100000
- **实现方式**：主要通过 `CHECK` 约束，有时也用触发器。

### 易错点与常考点
- **易错**：把外键约束当成内容约束，因为觉得“部门号必须存在”是业务规则，但它是模型层面的参照完整性，属于结构约束。
- **常考**：判断某个约束是结构约束还是内容约束。如“每个部门最多有10名员工”——这涉及计数，通常无法用简单约束实现，但若用断言或触发器，其本质是用户定义，属于内容约束（因为不是模型固有）。

## 第三页：完整性约束条件的类别（按约束状态分类）

### PPT原文
- **约束状态分类**
  - **静态约束**：要求DB在任一时候均应满足的约束；例如Sage在任何时候都应满足大于0而小于150(假定人活最大年龄是150)。
  - **动态约束**：要求DB从一状态变为另一状态时应满足的约束；例如工资只能升，不能降。工资可以是800元，也可以是1000元；可以从800元更改为1000元，但不能从1000元更改为800元。

### 详细讲解

#### 1. 静态约束
- **定义**：描述数据库在**任何时刻**都必须保持的不变性质。
- **特点**：不关心数据如何变化，只关心每个状态本身是否合法。
- **实现**：通常用 `CHECK`、`NOT NULL`、`UNIQUE`、主外键等声明式约束即可。
- **举例**：`age >= 0`，任何时候年龄不能为负。

#### 2. 动态约束
- **定义**：描述数据库**状态变迁**时必须满足的条件，通常涉及新旧值的比较。
- **特点**：无法用简单的声明式约束表达，因为声明式约束只能看到新值，看不到旧值。
- **实现**：必须通过**触发器**（在触发器中可以引用 `OLD` 和 `NEW`）或应用程序逻辑。
- **举例**：
  - 工资只能增加：`NEW.salary >= OLD.salary`
  - 职位升级时不能降级：`NEW.job_level >= OLD.job_level`
  - 库存量不能低于安全库存后再进货（涉及时间和状态变化）

### 易错点与常考点
- **易错**：试图用 `CHECK` 实现动态约束，如 `CHECK (salary >= 旧值)` 是不行的，因为 `CHECK` 中无法引用旧值。
- **常考**：
  - 判断给定规则是静态还是动态。
  - 问：如何实现动态约束？答：触发器。
  - 给出一段业务描述，要求选择合适的实现方式。

## 第四页：SQL-DDL中关于完整性的命令（列级约束、表级约束、约束类型）

### PPT原文
- **SQL-DDL** 中关于完整性的命令  
  - 列级约束  
  - 表级约束  
  - 约束：unique, primary key, not null, check, foreign key

### 详细讲解

#### 1. 列级约束
- **定义**：在定义列的同时直接附加在该列上的约束。
- **位置**：列的数据类型之后。
- **适用范围**：只能约束该列本身，不能涉及其他列。
- **语法示例**：
  ```sql
  CREATE TABLE student (
      sno CHAR(10) PRIMARY KEY,            -- 列级主键
      sname VARCHAR(20) NOT NULL,          -- 列级非空
      age INT CHECK (age >= 0)             -- 列级CHECK
  );
  ```

#### 2. 表级约束
- **定义**：在所有列定义完之后，独立定义的约束。
- **适用范围**：可以涉及多列，也可以定义复合主键、复合唯一、外键等。
- **语法示例**：
  ```sql
  CREATE TABLE sc (
      sno CHAR(10),
      cno CHAR(3),
      score DECIMAL(4,1),
      PRIMARY KEY (sno, cno),                       -- 表级复合主键
      FOREIGN KEY (sno) REFERENCES student(sno),    -- 表级外键
      CHECK (score >= 0 AND score <= 100)           -- 表级CHECK（也可列级）
  );
  ```

#### 3. 五种约束类型详解
| 约束类型        | 作用                     | 能否列级  | 能否表级    | 关键点                                                       |
| --------------- | ------------------------ | --------- | ----------- | ------------------------------------------------------------ |
| **NOT NULL**    | 列值不能为空             | √         | ×           | 只能列级，因为非空是列的属性                                 |
| **UNIQUE**      | 列值或组合值唯一         | √         | √           | 可多列组合唯一；可有一个空值（标准允许一个空值，具体看DBMS） |
| **PRIMARY KEY** | 主键，唯一且非空         | √（单列） | √（可复合） | 一个表只能有一个主键                                         |
| **FOREIGN KEY** | 参照另一表的主键或唯一键 | √（单列） | √（可复合） | 可指定 `ON DELETE/UPDATE` 行为（CASCADE, SET NULL, NO ACTION等） |
| **CHECK**       | 检查条件（布尔表达式）   | √（单列） | √（可多列） | 可包含子查询（但实际支持有限）                               |

### 易错点与常考点
- **易错**：
  - 试图用列级约束定义复合主键（不行）。
  - 忘记外键约束可以定义在表级以支持复合外键。
  - 混淆 `UNIQUE` 与 `PRIMARY KEY`：主键隐含非空，唯一键允许空。
- **常考**：
  - 写出建表语句，包含各种约束。
  - 给出一组列，要求定义合适的约束（如复合主键）。
  - 区分列级与表级约束的适用场景。

## 第五页：SQL-DDL中关于完整性的命令（CHECK子查询示例）

### PPT原文
- **check** 中的条件可以是 **Select-From-Where** 内任何 **Where** 后的语句，包含子查询。比如 **SC** 定义中不使用外键，则可如下约束：

```sql
Create Table SC (
    S# char(8) check( S# in (select S# from student)),
    C# char(3) check( C# in (select C# from course)),
    Score decimal(4,1) check (Score>=0.0 and score<=100.0)
);
```

### 详细讲解

#### 1. CHECK中使用子查询的含义
- SQL标准允许 `CHECK` 条件中包含子查询，这意味着约束可以跨表、使用聚合等，实现非常复杂的完整性规则。
- 该例中，用 `CHECK` 加子查询模仿了外键参照完整性：确保 `S#` 必须在 `student` 表中存在，`C#` 必须在 `course` 表中存在。

#### 2. 与外键约束的区别
- **外键约束** 不仅检查插入/更新时的存在性，还能在删除/更新被参照表时自动维护参照完整性（通过 `ON DELETE/UPDATE` 选项）。而 `CHECK` 子查询**只检查修改SC表时**，对 `student` 或 `course` 表的删除操作不会自动影响 `SC`，可能导致悬挂引用。
- **性能**：每次插入/更新 `SC` 都要执行子查询，开销较大。
- **灵活性**：`CHECK` 子查询可以实现外键无法表达的复杂条件，比如“每门课的选课人数不超过教室容量”等，这时必须用子查询。

#### 3. 数据库实现支持
- 并非所有DBMS都支持在 `CHECK` 中使用子查询。例如：
  - **PostgreSQL**：支持，但要求子查询不能引用被插入表的其他行（避免不确定性）。
  - **Oracle**：支持，但有限制（子查询必须是确定性的）。
  - **MySQL**：8.0.16之前忽略 `CHECK`，之后支持但仍有部分限制。
  - **SQL Server**：不支持在 `CHECK` 中使用子查询（只能引用本表的列），可用触发器替代。

### 易错点与常考点
- **易错**：
  - 认为 `CHECK` 子查询可以完全替代外键，忽略了参照完整性的维护义务。
  - 不知道不同数据库对 `CHECK` 子查询的支持程度不同，在考试中可能默认标准SQL。
- **常考**：
  - 问：如何用SQL实现跨表约束（如学生年龄不能大于系主任的年龄）？可以用 `CHECK` 子查询，或者用触发器。
  - 比较 `CHECK` 子查询与外键的优缺点。
  - 分析 `CHECK` 子查询可能带来的性能问题。

这五页PPT从三个维度（对象、来源、状态）对完整性约束进行了分类，然后介绍了SQL-DDL中定义约束的方法和具体类型，最后用一个子查询例子展示了 `CHECK` 的强大灵活性。掌握这些内容需要：
- 理解不同分类标准的含义，能正确归类给定的约束。
