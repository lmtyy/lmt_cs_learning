## 1. 数据库完整性的定义

### 内容回顾
- **数据库完整性**是指 **DBMS** 应保证数据库在任何情况下的**正确性、有效性和一致性**。
- 广义完整性：包括语义完整性、并发控制、安全控制、数据库故障恢复等。
- 狭义完整性：专指**语义完整性**（即数据符合现实世界语义），也是DBMS完整性管理机制主要处理的对象。本讲义特指语义完整性。

### 详细解释
- **正确性**：数据必须符合现实世界的定义，例如“性别只能为男或女”“年龄不能为负数”。
- **有效性**：数据必须在有效范围内，符合逻辑约束。
- **一致性**：数据之间不能相互矛盾，例如“一个学生的所在系必须是学校已存在的系”。

#### 广义与狭义的区别
- **广义完整性**涵盖了数据库正确运行的方方面面：
  - **并发控制**：防止多个事务同时操作数据导致不一致。
  - **安全控制**：防止非法用户访问数据。
  - **故障恢复**：保证在系统崩溃后数据能够恢复到一致状态。
- **狭义完整性**特指**数据的语义正确性**，即数据内容本身要符合业务规则，与并发、安全、恢复这些外围机制区分开来。数据库课程中通常讨论的“完整性”都是指狭义完整性（语义完整性）。

### 易错点
- **混淆完整性与安全性**：安全性涉及用户权限、防止恶意访问；完整性关心数据内容是否“合情合理”。例如：A学生可以修改自己的成绩（安全机制允许），但修改后的成绩不能超过100分（完整性约束）。两者是不同的概念。
- **忽略一致性**：有时只注意到单条数据的正确性，忽略了数据之间的参照关系（如外键必须存在）。

## 2. 为什么会引发完整性问题？完整性管理的作用

### 内容回顾
- **引发原因**：不正当的数据库操作，如输入错误、操作失误、程序处理失误等。
- **完整性管理的作用**：
  1. 防止和避免数据库中不合理数据的出现。
  2. DBMS应尽可能自动防止不合理现象。
  3. 如果DBMS不能自动防止，则需要应用程序员和用户在操作时处处小心，负担沉重，因此应尽可能让DBMS承担。

### 详细解释
#### 引发原因举例
- **输入错误**：用户将年龄输入为200岁。
- **操作失误**：删除某个部门时，忘记将该部门下的员工部门号置空或更新，导致员工参照了不存在的部门。
- **程序处理失误**：应用程序计算工资时出现逻辑错误，导致工资为负数。

#### 管理作用的意义
- **降低应用负担**：如果没有DBMS的自动检查，每个应用程序都要自己编写检查代码，极易遗漏或出错。
- **保证数据质量**：自动约束确保任何不符合规则的数据更新都被拒绝，从源头保证数据的正确性。

### 常考点
- **选择题**：以下哪个不属于完整性管理的优点？（答案可能是“提高查询速度”之类无关的选项。）
- **简答题**：为什么要由DBMS来保证完整性，而不是完全交给应用程序？

## 3. DBMS怎样自动保证完整性？

### 内容回顾
- **方法**：DBMS允许用户定义一些完整性约束规则（用SQL-DDL定义）。
- **过程**：当有数据库更新操作时，DBMS自动按照完整性约束条件进行检查，确保更新操作符合语义完整性。

### 详细解释
#### 定义约束
- 用户在创建表（`CREATE TABLE`）或修改表（`ALTER TABLE`）时，使用`PRIMARY KEY`、`FOREIGN KEY`、`CHECK`、`UNIQUE`、`NOT NULL`等关键字定义规则。
- 这些规则存储在**数据字典**中，由DBMS维护。

#### 自动检查
- 当执行`INSERT`、`UPDATE`、`DELETE`语句时，DBMS会触发约束检查。
- 如果发现违反约束，DBMS会**拒绝操作**并返回错误信息（违约响应）。

### 易错点
- **检查时机**：有些约束（如`CHECK`）只在插入或更新时检查，删除时一般不检查（除非涉及外键参照完整性）。
- **约束延迟检查**：有些DBMS支持延迟约束检查（`DEFERRABLE`），即事务提交时才检查，这容易让人误以为约束没起作用。

## 4. 完整性约束条件的一般形式

### 内容回顾
- **Integrity Constraint ::= (O, P, A, R)**
  - **O — 数据集合**：约束的对象，如列、多列（元组）、元组集合（表、多表）。
  - **P — 谓词条件**：什么样的约束，即逻辑表达式。
  - **A — 触发条件**：什么时候检查？通常是更新操作发生时。
  - **R — 违约响应**：不满足时怎么办？如拒绝操作、级联操作、置空等。

### 详细解释
这个四元组是完整性约束的抽象模型，帮助我们理解约束的各个要素。

#### O — 数据集合（对象）
- **列级约束**：只针对某一列，如`SAL > 0`。
- **元组约束**：涉及一行中的多个列，如`CHECK (end_date > start_date)`。
- **表级约束**：涉及多行甚至多个表，如外键约束（参照另一张表）、断言（涉及多个表或聚合）。

#### P — 谓词条件
- 可以是简单的比较，也可以是复杂的子查询。
- 例：`age BETWEEN 0 AND 150`；`deptno IN (SELECT deptno FROM department)`。

#### A — 触发条件
- 通常为更新操作（INSERT、UPDATE、DELETE）。有时可以指定具体列，如`UPDATE OF sal`。
- 对于断言，触发条件通常是任何可能影响断言结果的操作。
- 在SQL标准中，约束的检查时机可以是立即执行（IMMEDIATE）或延迟到事务结束（DEFERRED）。

#### R — 违约响应
- **拒绝执行**（最常见）：报错，操作回滚。
- **级联操作**：如`ON DELETE CASCADE`，删除主表时自动删除从表相关行。
- **置空**：`ON DELETE SET NULL`，删除主表时将外键置空。
- **设置为默认值**：`ON DELETE SET DEFAULT`。
- **触发其他动作**：可以通过触发器实现更复杂的响应。

### 与SQL约束的对应关系
| 约束类型    | O (对象)   | P (谓词)    | A (触发)                                   | R (违约响应)                 |
| ----------- | ---------- | ----------- | ------------------------------------------ | ---------------------------- |
| NOT NULL    | 单列       | IS NOT NULL | INSERT/UPDATE                              | 拒绝                         |
| UNIQUE      | 单列或多列 | 值唯一      | INSERT/UPDATE                              | 拒绝                         |
| PRIMARY KEY | 单列或多列 | 唯一且非空  | INSERT/UPDATE                              | 拒绝                         |
| FOREIGN KEY | 外键列     | 参照存在    | INSERT/UPDATE（从表）DELETE/UPDATE（主表） | 拒绝 / CASCADE / SET NULL 等 |
| CHECK       | 列或表     | 自定义条件  | INSERT/UPDATE                              | 拒绝                         |
| 断言        | 表/多表    | 复杂条件    | 相关操作                                   | 拒绝                         |

### 易错点
- **忽略触发条件**：有些考生只记得约束条件，但忘记了外键约束在删除主表时也会触发检查。
- **混淆O和P**：O是约束作用的“数据范围”，P是具体的条件，例如`CHECK (age>0)`中O是`age`列，P是`age>0`。
- **违约响应的多样性**：外键约束的`ON DELETE`和`ON UPDATE`可以有多种选项，必须理解每种选项的含义（特别是`NO ACTION`和`RESTRICT`的区别，虽然多数DBMS中两者等同）。

### 常考点
- **填空题/简答题**：完整性约束的四个组成部分是什么？分别说明其含义。
- **分析题**：给定一个业务规则，用SQL写出相应的约束，并指出其O、P、A、R分别是什么。
- **选择题**：关于外键约束的违约响应，哪种情况会导致数据被自动删除？（CASCADE）
- **判断题**：检查约束只在插入数据时执行，更新时不执行。（错误，更新时也会执行）

## 补充：完整性的其他重要考点（结合后续内容）

虽然这几页PPT主要讲概念，但后续的完整性管理通常会涉及以下内容，也是考试重点：

1. **实体完整性**（PRIMARY KEY）
   - 主键列必须唯一且非空。
   - 复合主键必须整体唯一，且每列非空。

2. **参照完整性**（FOREIGN KEY）
   - 外键值要么为空，要么等于被参照表的主键值。
   - 删除/更新主表时的策略：`CASCADE`、`SET NULL`、`SET DEFAULT`、`RESTRICT`、`NO ACTION`。

3. **用户定义完整性**（CHECK、触发器）
   - `CHECK`可以定义简单的域约束。
   - 触发器可实现更复杂的动态约束。

4. **断言**
   - 定义涉及多表或聚合的复杂约束，但实际使用较少，常被触发器替代。

5. **约束的命名与维护**
   - 使用`CONSTRAINT`关键字为约束命名，便于删除或修改。
   - `ALTER TABLE ... DROP CONSTRAINT` 删除约束；`ADD CONSTRAINT` 添加约束。

6. **约束检查的时机**
   - 立即检查：默认方式。
   - 延迟检查：`INITIALLY DEFERRED`或`INITIALLY IMMEDIATE`，需要事务支持。

### 常见易错题举例
- **判断对错**：一个表可以有多个外键，但只能有一个主键。（正确）
- **选择**：以下哪种情况不会违反参照完整性？  
  A. 向从表插入一个外键值，该值在主表中不存在  
  B. 删除主表中被从表引用的行，且外键约束为`NO ACTION`  
  C. 将从表的外键列更新为一个不存在于主表的值  
  D. 删除主表中未被从表引用的行  
  答案：D

- **设计题**：写出创建学生选课表的SQL语句，要求学号参照学生表，课程号参照课程表，成绩在0~100之间，并定义删除课程时选课记录级联删除。

### 总结
这几页PPT从完整性概念出发，引出了完整性管理的必要性和实现方式，并提供了一个抽象模型(O,P,A,R)来理解约束的组成。掌握这些基础，后续学习具体的SQL约束定义、触发器、断言等将更加轻松。在考试中，重点在于理解完整性的分类、各种约束的语义以及违约处理方式，并能够根据业务规则正确使用SQL定义约束。
