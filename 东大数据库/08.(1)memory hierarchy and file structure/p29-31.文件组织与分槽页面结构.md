### 第一页 PPT 详解：文件组织的基本概念

#### 标题：File Organization

#### 知识点 1：数据库的物理存储层次定义

**原文：**
> *The database is stored as a collection of **files**. Each file is a sequence of **records**. A record is a sequence of **fields**.*

这是一个自上而下的三层逻辑结构：

| 层次 | 名称     | 英文   | 含义                                          | 类比               |
| :--- | :------- | :----- | :-------------------------------------------- | :----------------- |
| L1   | **文件** | File   | 操作系统中的一个物理文件（如 `students.dat`） | 一本**书**         |
| L2   | **记录** | Record | 表中的一行数据（如某个学生的完整信息）        | 书中的一**行文字** |
| L3   | **字段** | Field  | 一行中的某一列属性（如学号、姓名）            | 行中的一个**单词** |

**关键理解：**
- 文件是 DBMS 向操作系统申请存储空间的**最大容器单位**。
- DBMS 内部将文件进一步划分为页（Page），页是 I/O 单位；而记录是用户看到的逻辑单位。

#### 知识点 2：最常见的简化实现方式——定长记录假设

**原文：**
> *Most common approach:*
> - *assume record size is fixed*
> - *each file has records of one particular type only*
> - *different files are used for different relations*

**详细解释：**

1. **假设记录大小固定：**
    - 每一行的字节数完全一样。
    - 典型场景：全定长字段（如 `CHAR(10)`, `INT`, `DATE`），不含 `VARCHAR`。

2. **一个文件只存一种记录类型：**
    - 一个文件对应数据库中的**一个表**（Relation）。
    - 文件 `students.dat` 只存学生表的行，不混入其他表的数据。

3. **为什么这是“最容易实现”的？**
    - **定位第 N 条记录极快：**
        - 因为每条记录大小固定（设为 `R` 字节），第 `i` 条记录的起始位置 = **`i × R`**。
        - 这是 **O(1)** 的数组式随机访问。
    - **删除记录处理简单：**
        - 可以用一个 **位图（Bitmap）** 标记该槽位是否被占用，或者将最后一条记录移动到被删除的位置。
    - **空间管理简单：**
        - 文件大小 = 记录数 × 固定记录长度。新增记录直接追加到文件末尾。

**但现实是：** 真实世界的表几乎必然包含变长字段（如 `VARCHAR`、`TEXT`），所以后续必须考虑变长记录。

### 第二页 PPT 详解：页面内部组织结构（分槽页面）

#### 标题：页面组织

这一页深入到了一个 **8KB 或 16KB** 的页内部，展示了记录是如何被收纳的。

#### 知识点 1：页面大小与基本属性

- **大小：** 8KB 或 16KB，且为 **2 的整数次幂**。
    - 原因：与操作系统页大小对齐（通常 4KB），便于内存映射（mmap）和直接 I/O 对齐；位运算取模极快。
- **一个页面包含多条记录：** 页是多个记录共享的容器，减少了外部碎片。

#### 知识点 2：分槽页面结构（Slotted Page Structure）详解

PPT 中的表格展示了经典的 **分槽页面（Slotted Page）** 布局，这是几乎所有现代 DBMS（PostgreSQL, MySQL, SQL Server）管理变长记录的标准方案。

| 区域         | 内容                                    | 说明                               |
| :----------- | :-------------------------------------- | :--------------------------------- |
| **页面头**   | 记录数量、checksum、上下页指针等        | 页的元数据区                       |
| **（补充）** | **记录删除状态位向量** `(11011...)_2`   | 标记记录槽位是否有效               |
| **记录地址** | 地址指针 0, 地址指针 1, 地址指针 2, ... | **槽数组（Slot Array）**，向下增长 |
| **记录数据** | 记录 0, 记录 1, 记录 2, ...             | **实际数据区**，向上增长           |

**核心设计原理：动态双向增长**

- **槽数组（Slot Array）**：位于页头之后，存储每个记录的**起始偏移量**（和长度信息）。随着记录插入，槽数组**从页的顶部向底部增长**。
- **记录数据区**：存储实际的记录字节。随着记录插入，数据区**从页的底部向顶部增长**。
- **中间空闲空间**：位于槽数组和记录数据区之间，是未使用的空间。

**这种设计的精妙之处（支持变长记录的核心）：**

1. **记录移动时无需重建整个页：**
    - 当一条记录因为 `UPDATE` 变长（如 `VARCHAR` 从 'abc' 改成 'abcdefghijk'）时：
        - 原有位置放不下了。
        - 只需在数据区找新的空闲位置存放新记录，然后**更新对应的槽指针**指向新地址即可。
        - 旧空间被标记为“碎片”，未来可能被其他小记录复用，或者等待页面重组（Reorganize / Vacuum）。

2. **删除记录开销极低：**
    - 不需要物理移动数据来填补空洞。
    - 只需在**记录删除状态位向量**中将对应位设为 `0`（表示无效），并可选地将槽指针置为 NULL。
    - 这种标记为删除的记录称为 **“墓碑”（Tombstone）**。

3. **记录 ID（Record ID / TID）的稳定性：**
    - 记录的逻辑标识符通常是 `(PageID, SlotIndex)`。
    - 即使记录在页面内的**物理位置**变了，只要**槽索引（Slot Index）**不变，外部索引（如 B+ 树叶子节点）指向该记录的指针就**无需更新**。这对索引维护至关重要。

**知识点 3：记录删除状态位向量**

- 示例 `(11011...)_2` 表示：第 0、1、3、4 号槽的记录是有效的；第 2 号槽的记录已被删除（或未使用）。
- 这是一个紧凑的位图，用于快速判断槽位状态。

### 第三页 PPT 详解：定长记录 vs 变长记录

#### 标题：页面组织：定长记录 vs 变长记录

这一页将前两页的概念落地到具体的字节布局对比上。

#### 对比一：定长记录

**示例表结构：** 学号 `CHAR(10)`，姓名 `CHAR(?)` 假设定长，性别 `CHAR(2)`，院系 `CHAR(?)` 定长，绩点 `DECIMAL`。

**物理存储布局：**

| 记录头 | 学号 (10B) | 姓名 (定长) | 性别 (2B) | 院系 (定长) | 绩点 (定长) |
| :----- | :--------- | :---------- | :-------- | :---------- | :---------- |

**特点：**

- **字段位置固定：** 第 N 个字段的起始偏移量是**编译期确定的常数**。
- **定位极快：** 要读“绩点”字段，直接从记录头跳过 `10 + 姓名长度 + 2 + 院系长度` 个字节即可。
- **缺点：** 空间浪费严重。如果“姓名”定义成 `CHAR(50)`，但实际只存了“张三”（4 字节），剩余 46 字节全被填空白字符浪费掉。

#### 对比二：变长记录

**示例表结构：** 学号 `CHAR(10)` 定长，姓名 `VARCHAR` 变长，性别 `CHAR(2)` 定长，院系 `VARCHAR` 变长，绩点 `DECIMAL` 定长。

**物理存储布局（回顾第二讲内容）：**

| 记录头   | 学号 (10B) | 性别 (2B) | 绩点 (定长) | 姓名偏移/长度 | 院系偏移/长度 | 空值向量 | ...后面跟着变长数据 |
| :------- | :--------- | :-------- | :---------- | :------------ | :------------ | :------- | :------------------ |
| 字节偏移 | 0          | 10        | 20          | 24            | 28            | 32       | 实际变长数据区      |

**PPT 中的具体字节表示例：**

| 字段     | 学号         | 性别 | 绩点  | 姓名   | 院系    | 空值向量 |
| :------- | :----------- | :--- | :---- | :----- | :------ | :------- |
| 内容     | `2021310001` | `男` | `3.9` | `37,6` | `43,12` | `0`      |
| 字节偏移 | 0            | 10   | 20    | 24     | 28      | 32       |

**详细解读：**

1. **定长字段前置：**
    - 学号（10 字节）、性别（2 字节？图中偏移 10→20 是 10 字节，可能用了更多填充）、绩点（4 字节？偏移 20→24）被放在记录的**固定部分**。
    - 这样无论变长字段多长，这些固定字段都能通过**固定偏移量**快速访问。

2. **变长字段用“偏移/长度”指针对替代：**
    - 姓名字段实际存的是 `(37, 6)`。
        - `37`：姓名数据的起始位置在**记录内偏移 37 字节处**。
        - `6`：姓名字符串长度为 6 字节（假设编码下“张三”占 6 字节）。
    - 院系字段实际存的是 `(43, 12)`。
        - `43`：院系数据起始偏移。
        - `12`：院系名称长度。

3. **空值向量（Null Bitmap）：**
    - 如果某字段允许 `NULL`，比如绩点未录入，就在位向量中标记，而不是在绩点位置存一个特殊的“-1”值。这更节省空间且语义清晰。

4. **实际数据存放：**
    - 变长字段的真实内容（`张三`、`计算机系`）被放在记录的**尾部**（或者页面的数据区尾部，取决于具体实现）。

### 总结：三页 PPT 的逻辑关系

1. **第一页（宏观策略）：** 物理存储 = 文件 → 记录 → 字段。为简化先假设定长记录。
2. **第二页（中观结构）：** 无论定长变长，记录都存在**页**里。页采用**分槽结构**，通过槽数组和双向增长空间来灵活管理记录。
3. **第三页（微观实现）：** 定长记录在页内是密集排列的数组；变长记录则需要**定长部分前置 + 偏移量表 + 变长数据后置**的复杂结构，而分槽页面正好为此提供了完美的容器支持。
