### 第一页 PPT 详解：访问磁盘页的时间构成

#### 标题：Accessing a Disk Page

这一页的核心是解释 **“磁盘 I/O 的高昂代价究竟由哪些部分组成”**。

#### 知识点 1：Time to access (read/write) a disk block

PPT 将访问一个磁盘块的时间分解为三个部分。**这里的“磁盘块”与“数据库页”在物理上通常是对应的**。

**1. Seek Time (寻道时间)**

- **原文定义：** *moving arms to position disk head on track*
- **详细机制：**
    - 机械硬盘内部有一摞高速旋转的盘片（Platter），每个盘片表面都有密密麻麻的同心圆，称为**磁道（Track）**。
    - 读写数据依靠**磁头（Disk Head）**。所有的磁头都固定在同一个**磁头臂（Actuator Arm）**上，它们只能同步左右摆动。
    - **动作过程：** 当你要读第 100 号磁道的数据，而磁头当前停在第 50 号磁道时，磁头臂必须**物理移动**过去。
    - **为什么是瓶颈？** 这是**纯机械运动**。相对于电信号的纳秒级传播速度，金属臂的惯性移动慢得令人发指。这是磁盘 I/O 延迟中**最大、最难消除**的部分。
    - **典型数值：** 普通消费级硬盘 ~ 8-12 ms；企业级高速硬盘 ~ 3-5 ms。

**2. Rotational Delay (旋转等待时间)**

- **原文定义：** *waiting for block to rotate under head*
- **详细机制：**
    - 磁头已经移动到正确的磁道上方了，但数据是分布在圆周上的**扇区（Sector）**里的。
    - 此时磁头必须**等待**盘片旋转，直到目标扇区恰好转到磁头正下方。
- **为什么是“平均”等待？**
    - 最好情况：刚到位，扇区就来了 -> 延迟 0 ms。
    - 最坏情况：刚到位，扇区刚刚转过去 -> 需要等盘片转**一整圈**（约 8.3 ms @ 7200 RPM）。
    - **平均情况：** 转半圈的时间。这是概率期望值。
    - **计算公式：** 平均延迟 = 1 / (2 * 每秒转数)。
    - **举例：** 7200 RPM 的硬盘，每秒转 120 圈，平均旋转延迟 = 1 / 240 秒 ≈ **4.17 ms**。

**3. Transfer Time (传输时间)**

- **原文定义：** *actually moving data to/from disk surface*
- **详细机制：**
    - 数据终于到了磁头下面，此时开始通过磁头感应磁场变化，将数据转化为电信号传回内存（或反向写入）。
- **相对速度：** 这部分速度取决于**盘片旋转线速度**和**数据密度**。
- **现代磁盘中：** 随着存储密度提升，传输速度越来越快（现在的硬盘连续读取能到 200 MB/s 以上）。
- **耗时比例：** 相比前两项机械等待的毫秒级时间，传输一个 8KB 的页可能只需要**几十微秒（0.0x ms）**。**这部分时间几乎可以忽略不计。**

#### 知识点 2：Therefore, relative placement of pages on disk has major impact on DBMS performance!

**逻辑推导：**

- 访问时间 = **寻道（大头）** + **旋转（中头）** + 传输（小头）。
- 如果 DBMS 需要的数据页在磁盘上**东一个、西一个**（随机分布）：
    - 读第 1 页：寻道 5ms + 旋转 4ms = 9ms。
    - 读第 2 页：又要寻道 5ms + 旋转 4ms = 9ms。
    - 结果：CPU 大部分时间都在**空转等待磁盘**。
- **结论：** DBMS 在磁盘上摆放数据页的**相对物理位置**，直接决定了磁头需要**移动多少次**，从而决定了系统的生死存亡。

### 第二页 PPT 详解：顺序 I/O 的巨大优势与 DBMS 的控制权

#### 标题：Accessing a Disk Page (续)

这一页是对上一页结论的延伸和解决方案的提出。

#### 知识点 1：Sequential I/O is much faster than random.

- **量化数据：** *10 and 100 times faster than random access.*

**详细解释为什么会快 10-100 倍：**

- **回想第一页的公式：** 访问时间 = 寻道 + 旋转 + 传输。
- **随机访问（Random I/O）：** 每读一个 4KB 页，付一次 **寻道+旋转** 的“门票费”（~10ms）。
- **顺序访问（Sequential I/O）：** 你要读 100 个连续的页。
    - **第 1 个页：** 付一次门票费（寻道+旋转 = 10ms）。
    - **第 2 ~ 100 页：** **不需要再付门票费！**
    - 磁头读完第 1 页时，第 2 页刚好随着盘片旋转转到磁头下方，直接读。
    - 此时的时间主要就是 **传输时间**。
- **算一笔账：**
    - 读 1MB 随机数据（256 个 4KB 页）：256 * 10ms = **2.56 秒**。
    - 读 1MB 连续数据（磁盘顺序读速度 200MB/s）：1MB / 200MB = **0.005 秒**。
    - **速度差：512 倍。** 这就是 PPT 强调 "10 to 100 times" 的原因（实际上可能更夸张）。

#### 知识点 2：DBMS 为何要绕过 OS 控制物理布局？

- **原文引述：** *Since the DBMS can understand its workload access patterns more deeply than the underlying OS, it makes sense for DBMS architects to exercise full control over the spatial positioning of database blocks on disk.*

**这是一个非常深刻且重要的 DBMS 设计哲学问题。逐句拆解：**

**1. OS (操作系统) 的无知：**
现代操作系统都有**文件系统缓存（Page Cache）**和**磁盘调度算法（如电梯算法）**。OS 会尝试帮你合并 I/O 请求。
但是，OS 看到的只是 **“进程 A 请求了文件 X 的字节偏移量 Y”**。
OS **不知道**：

- 这 100 个读取请求其实是 **同一个 SQL 查询的全表扫描**。
- 这 100 个读取请求中，第 3 个比第 1 个更紧急（例如用户界面等待的数据 vs 后台预读的数据）。

**2. DBMS 的全知：**
DBMS **知道**：

- 这是一个全表扫描 -> **应该启用大规模预读（Prefetch）**。
- 这是一个索引嵌套循环连接 -> 接下来会有**大量的随机跳跃读**，现在应该把缓存策略切换为 LRU-2 或者更激进的淘汰策略。

**3. 为什么要“Full Control”（完全控制）？**
PPT 里提到的 *spatial positioning*（空间定位）具体指以下 DBMS 的关键技术：

- **原始设备（Raw Device / Direct I/O）：** DBMS 不经过 OS 的文件系统缓存，直接读写磁盘分区。**理由：** 避免数据被 **双重缓存**（OS 缓存一份，DBMS 自己的 Buffer Pool 也缓存一份），浪费内存；同时避免 OS 不可预测的刷脏页时机导致性能抖动。
- **数据聚集（Clustering）：** 建表时指定 `CLUSTER` 索引，强制表中的行**物理存储顺序**与索引键顺序一致。这样范围扫描 `WHERE id BETWEEN 1 AND 100` 就变成了**一次顺序 I/O**，而不是 100 次随机 I/O。
- **日志先行（WAL - Write-Ahead Logging）：** 不直接把修改的数据页写回原位置（那是随机写），而是先把修改记录**顺序追加**写到一个单独的日志文件尾部。**这是把随机写转化为顺序写的魔术。**

#### 知识点 3：Key to lower I/O cost

这一行是小标题，总结了两页 PPT 的核心主旨：

- **降低 I/O 成本的关键** = **把随机 I/O 转化为顺序 I/O** + **减少磁头机械运动**。
- 这也是为什么 B+ 树索引要把叶子节点用**双向链表**串起来，即使逻辑上是树，物理扫描时却是沿着链表**顺序读取**磁盘页。

### 总结这两页 PPT 的逻辑链

1. **第一页**：I/O 为什么慢？ -> 因为 **机械运动**（寻道 + 旋转）。
2. **第二页**：怎么变快？ -> **避免机械运动**（顺序 I/O）。
3. **怎么做？** -> DBMS **不能依赖 OS**，必须**亲自管理**数据在磁盘上的物理摆放位置（Spatial Positioning）。
