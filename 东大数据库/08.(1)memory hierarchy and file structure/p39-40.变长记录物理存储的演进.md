好的，我们按照 PPT 顺序，详细讲解这两页关于 **变长记录（Variable-Length Records）** 的内容。这两页从问题场景出发，先介绍一种朴素但有缺陷的方案，再引出实际 DBMS 采用的更优存储结构。

### 第一页 PPT 详解：变长记录的产生场景与字节串表示法

#### 标题：Variable-Length Records / Byte String Representation

#### 知识点 1：变长记录在数据库中的几种产生方式

**原文：**
> *Variable-length records arise in database systems in several ways:*
> - *Storage of multiple record types in a file.*
> - *Record types that allow variable lengths for one or more fields.*
> - *Record types that allow repeating fields (used in some older data models).*

**详细解释：**

1. **一个文件中存储多种记录类型（Multiple record types in a file）：**
    - 在某些系统设计中，一个物理文件可能混合存放不同类型的记录。例如，一个文件既存“学生”记录，又存“教师”记录。
    - 由于学生和教师的字段结构不同（学生有“班级”，教师有“职称”），它们的记录长度自然不同。
    - 这种设计在**面向对象数据库**或早期的**网状/层次数据库**中较常见，现代关系型数据库通常一个表一个文件，避免了这种混合。

2. **记录类型中允许一个或多个字段变长（Variable lengths for fields）：**
    - 这是**最常见**的原因。任何包含 `VARCHAR`、`VARBINARY`、`TEXT`、`BLOB` 类型字段的表，其记录都是变长的。
    - 例如：`name VARCHAR(30)` 实际存储“张三”只占很少字节，存储“亚历山德罗维奇”则占用较多字节。
    - 这导致同一张表中的不同行，物理长度差异显著。

3. **记录类型中允许重复字段（Repeating fields）：**
    - 某些旧数据模型（如 COBOL 的 OCCURS 子句、层次模型的“子记录”数组）允许一条记录内包含**数量不定的重复组**。
    - 例如：一个“订单”记录可能包含不定数量的“订单明细”子项。这也会导致记录长度不固定。
    - 现代关系模型中，这种结构通过**垂直拆分**为两张表（订单表 + 订单明细表）并建立外键来规范化，从而避免了单条记录内的重复组。

#### 知识点 2：字节串表示法（Byte String Representation）——一种简单但糟糕的方案

**原文：**
> *Simple (but bad) solution: Byte string representation*
> - *Attach an `end-of-record` (⊥) control character to the end of each record*
> - *Difficulty with deletion (fragmentation of free space)*
> - *Difficulty with growth (movement of records is difficult)*

**方案描述：**
- 将每条记录视为一个连续的字节序列（Byte String）。
- 在记录的最后一个字节后面，附加一个**特殊的“记录结束”控制字符**（例如 ASCII 码中的 `EOT` 或自定义的哨兵值，PPT 中记为 `⊥`）。
- 读取记录时，从记录起始地址开始逐个字节读取，直到遇到 `⊥` 为止，即认为一条记录结束。

**这种方法的严重缺陷：**

**缺陷 1：删除导致空间碎片化（Fragmentation of free space）**

- 假设页面中紧密排列着三条变长记录：`[R1]⊥[R2]⊥[R3]⊥`。
- 当删除中间的 `R2` 后，空间变为：`[R1]⊥[空洞]⊥[R3]⊥`。
- **问题**：留下的“空洞”长度不一定恰好等于新插入记录的长度。
    - 如果新记录比空洞**小**：插入后留下一个更小的碎片空洞。
    - 如果新记录比空洞**大**：放不下，只能找别处或等待页面压缩。
- 随着增删改操作的进行，页面内会布满大量不连续的小空洞（**外部碎片**），空间利用率极低。DBMS 不得不定期执行**页面重组（Reorganization）**来整理碎片，开销巨大。

**缺陷 2：记录变长后难以原地扩展（Difficulty with growth）**

- 假设某条记录的某个 `VARCHAR` 字段被 `UPDATE` 更新为更长的字符串。
- 例如：`name` 从 `'Bob'` 改为 `'Bartholomew'`。
- 由于字节串表示法下，记录之间是**紧密相邻**的，`R1` 后面紧跟着 `R2`。
- 如果 `R1` 想要变长，它身后的 `R2` **挡在了路上**。
- **后果**：必须将整条 `R1` **移动到**页面中有足够连续空间的其他位置（或者移动到新页面）。
- 这种移动会引发连锁反应：
    1. 在原位置留下一个空洞（加剧碎片化）。
    2. 所有指向 `R1` 的外部索引指针（RID）**必须全部更新**，否则会指到错误数据。这对于高并发系统来说是灾难性的维护成本。

**总结：** 字节串表示法虽然实现简单（早期的文件系统或磁带存储可能用过），但它完全无法应对数据库频繁增删改、字段长度动态变化的现实需求。因此，现代 DBMS 无一例外地采用了下一张 PPT 介绍的结构化布局。

### 第二页 PPT 详解：变长记录的结构化存储布局

#### 标题：Variable-Length Records

#### 知识点 1：示例表结构

**PPT 给出 SQL DDL：**
```sql
CREATE TABLE MovieStar(
  name VARCHAR(30) PRIMARY KEY,
  address VARCHAR(255),
  gender CHAR(1),
  birthdate DATE
);
```

**字段分析：**
- `name`：`VARCHAR(30)`，变长。
- `address`：`VARCHAR(255)`，变长。
- `gender`：`CHAR(1)`，定长（1 字节）。
- `birthdate`：`DATE`，定长（通常 4 或 8 字节）。

**结果：** 由于存在两个变长字段，该表的每条记录都是变长的。

#### 知识点 2：实际 DBMS 采用的记录布局

PPT 用一张示意图展示了记录在内存/磁盘上的物理布局：

```
┌──────────────────────┬───────────────┬──────────┬──────────┬───────────┬─────────┬─────────┐
│ other header info    │ record length │ to address│ gender   │ birthdate │ name    │ address │
│ (其他头部信息)        │ (记录长度)     │ (指向地址) │ (定长)    │ (定长)     │ (变长)   │ (变长)   │
└──────────────────────┴───────────────┴──────────┴──────────┴───────────┴─────────┴─────────┘
```

**关键设计原则：定长部分前置 + 偏移量表 + 变长数据后置**

让我们逐块解析这张图：

**1. 记录头（Record Header）—— 前半部分**

| 组件                  | 说明                                                         |
| :-------------------- | :----------------------------------------------------------- |
| **other header info** | 事务信息（创建/删除该记录的事务 ID、回滚指针等，用于 MVCC）、NULL 值位图等。 |
| **record length**     | 该条记录的**总字节长度**。有了这个值，DBMS 在处理时无需扫描记录内容即可知道这条记录在哪里结束、下一条记录从哪里开始。这解决了字节串表示法需要哨兵字符的问题。 |
| **to address**        | 这是一个**偏移量/指针**。具体含义取决于实现：<br>- 可能是指向 `address` 字段数据在记录内起始位置的**偏移量**。<br>- 也可能是整个变长数据区的起始偏移量。<br>- 无论如何，它允许 DBMS **直接跳转**到变长字段的数据位置。 |

**2. 定长字段区（Fixed-Length Fields Area）**

- **gender**（1 字节）和 **birthdate**（4/8 字节）紧跟在记录头的定长部分之后。
- **核心思想**：无论变长字段内容有多长，定长字段的**偏移量是编译期固定**的。
- 例如，在给定记录头大小固定的情况下，`gender` 字段永远位于记录起始地址 + `X` 字节处，`birthdate` 永远位于 + `Y` 字节处。
- 这保证了 DBMS 可以 **O(1) 直接访问定长字段**，无需任何跳转计算。

**3. 变长字段数据区（Variable-Length Data Area）**

- **name** 和 **address** 的实际字符串内容存放在记录的**最后部分**。
- **关键问题**：既然变长字段长度不固定，如何找到 `address` 字段的起始位置？
    - **方案 A（偏移量数组）**：在记录头中存储一个小型数组，记录每个变长字段的 `<起始偏移，长度>`。如之前讲过的 `(37,6)` 代表姓名。
    - **方案 B（目录指针）**：记录头中存一个指针指向记录尾部，变长数据在尾部按逆序或顺序存放，并在尾部前放置一个小的偏移量表。
- PPT 中的 `to address` 就起到了这个导航作用。

#### 知识点 3：这种布局如何解决字节串表示法的问题

| 字节串表示法的问题         | 结构化布局的解决方案                                         |
| :------------------------- | :----------------------------------------------------------- |
| **删除造成碎片**           | 配合**分槽页面（Slotted Page）**使用。记录在页面内通过槽数组管理，删除时只需修改槽指针和位图，不需要物理移动相邻记录。碎片可以通过后台 Vacuum 进程在低负载时整理。 |
| **记录变长后无法原地扩展** | **移动记录但不更新索引**。当记录变长需要移动时，DBMS 在页面内或跨页寻找新位置存放整条记录，然后**只更新槽数组中的指针**指向新位置，而外部索引持有的 **RID (PageID, SlotIndex)** 中的 SlotIndex **保持不变**。这称为**间接层**的威力。 |
| **无法快速定位记录边界**   | 记录头中存储 **record length**，下一条记录的地址 = 当前地址 + record length。无需扫描哨兵字符。 |
| **无法快速访问内部字段**   | 定长字段固定偏移，变长字段通过偏移量表定位。所有访问都是 **O(1)** 的指针运算，而非 **O(N)** 的扫描。 |

---

### 总结与对比

| 特性           | 字节串表示法                       | 结构化变长记录布局                                     |
| :------------- | :--------------------------------- | :----------------------------------------------------- |
| **记录分隔**   | 哨兵字符 `⊥`                       | 记录头中的 `record length`                             |
| **字段访问**   | 必须从前往后扫描解析               | 定长字段 O(1) 直接偏移；变长字段通过偏移量表 O(1) 跳转 |
| **删除操作**   | 留下不连续空洞，碎片严重           | 槽数组管理，逻辑删除，碎片可后台整理                   |
| **更新变长**   | 原地几乎不可能，必须移动且更新索引 | 移动记录，槽指针重定向，RID 不变，索引无需更新         |
| **空间利用率** | 低，外部碎片多                     | 高，页面内灵活分配，复用空洞                           |
