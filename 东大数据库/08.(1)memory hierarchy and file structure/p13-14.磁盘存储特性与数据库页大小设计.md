### 第一页 PPT 详解：磁盘作为二级存储及其对 DBMS 设计的根本影响

#### 知识点 1：Disks - Secondary storage device of choice. (磁盘——首选的二级存储设备)

**详细解释：**

1.  **存储层级结构（Memory Hierarchy）：**
    在计算机体系结构中，存储设备是分层的，自上而下速度越来越慢、容量越来越大、单位成本越来越低。
    - **一级存储（Primary Storage）：** 通常指 **主存（RAM，随机存取存储器）**。CPU 可以直接、快速地访问 RAM 中的数据。它的速度是纳秒级的。但是，RAM 是**易失性**的（断电数据丢失），且每 GB 成本高昂，容量相对较小。
    - **二级存储（Secondary Storage）：** 通常指 **磁盘（HDD，机械硬盘 / SSD，固态硬盘）**。它用于持久化存储数据，即使断电数据也不会丢失。CPU **不能**直接访问磁盘上的数据，必须先将其加载到 RAM 中。

2.  **为什么是“首选”？**
    PPT 说磁盘是 “device of choice”，原因有三：
    - **持久性（Non-volatility）：** 数据库必须保证数据不会因掉电而丢失（ACID 中的 D，Durability 持久性）。磁盘是提供这种保障的最经济手段。
    - **大容量与低成本：** 相比内存，磁盘能够以低廉得多的成本存储 TB 甚至 PB 级别的数据。对于存储海量业务数据的 DBMS 来说，这是硬性要求。
    - **对比磁带（Tape）：** 在更早的时代，磁带也是二级存储，但它是顺序访问的（只能倒带快进），无法随机读取第 N 块数据。**磁盘支持随机访问（Random Access）**，磁头可以快速移动到任意位置读取特定数据块，这是 DBMS 高效检索的基础。

#### 知识点 2：This has major implications for DBMS design! (这对 DBMS 设计有重大影响！)

**详细解释：**
为什么选了磁盘做存储，DBMS 的设计就得跟着变？因为**磁盘的速度比内存慢几个数量级**。

- **时间尺度对比（关键直觉）：**
    - CPU 执行一条指令：~0.3 纳秒
    - 访问 L1 缓存：~1 纳秒
    - 访问主存 RAM：~100 纳秒
    - **访问机械硬盘（HDD）随机数据：~5,000,000 - 10,000,000 纳秒（5-10 毫秒）**
    - **打个比方：** 如果访问内存是去楼下的便利店买水（1 分钟），那么访问机械硬盘就是飞去纽约买水（几个月）。

由于这种巨大的速度鸿沟，DBMS 的核心算法（如查询优化、索引设计、缓存管理）不再是单纯追求减少 CPU 计算次数，而是**首要目标变为：减少磁盘 I/O 次数**。

#### 知识点 3：READ 和 WRITE 操作的定义与高代价分析

- **READ：transfer data from disk to main memory (RAM).**
- **WRITE：transfer data from RAM to disk.**

**“Both are high-cost operations ... must be planned carefully!”**

**详细解释高成本的物理原因（针对机械硬盘 HDD）：**

1.  **寻道时间（Seek Time）：** 磁头臂需要移动到数据所在的磁道上方。这是机械运动，最耗时（通常 3-9 毫秒）。
2.  **旋转延迟（Rotational Latency）：** 磁头到位后，需要等待磁盘旋转，直到目标扇区转到磁头下方。平均是转半圈的时间（通常 2-4 毫秒）。
3.  **传输时间（Transfer Time）：** 真正读写数据的时间，相对于前两者很快（微秒级）。

**为什么 SSD（固态硬盘）改变了局面但仍需谨慎？**

- SSD 没有机械部件，随机读取延迟极低（~0.1 毫秒）。
- 但是，SSD 的 **写入** 操作存在 **写放大（Write Amplification）** 和 **单元磨损（Wear-out）** 问题。
- 此外，即使 SSD 快了，它**仍然比内存慢 100 倍以上**，且通过系统总线、I/O 控制器的软件开销依然存在。

**DBMS 如何“仔细规划”（Careful Planning）来应对？**

- **缓冲区管理（Buffer Pool）：** 在内存中开辟一大块区域缓存经常访问的磁盘页。如果数据已经在 Buffer Pool 里，就不用读盘了（缓存命中）。
- **顺序访问 vs 随机访问：** DBMS 会尽量把随机写请求转化为顺序写请求（例如使用预写日志 WAL），因为顺序写磁盘比随机写快几十倍。
- **预读（Prefetching）：** 扫描全表时，DBMS 会预测用户即将请求接下来的页，提前异步地把它们从磁盘读到内存。
- **索引结构（B+ Tree）：** B+ 树的矮胖结构设计就是为了**最小化磁盘 I/O 次数**。树的高度低，意味着查找一个键只需要很少次数的磁盘读取。

### 第二页 PPT 详解：数据库页面大小的定制化选择

#### 知识点 1：Database Page Size (数据库页大小) 概念

**详细解释：**
这是 DBMS 管理磁盘空间的**最小逻辑单位**，也是内存（Buffer Pool）和磁盘之间一次 I/O 传输的**最小单位**。

- **类比：** 就像货运公司运货，不会一根一根针地运，而是装在**集装箱**里运。数据库的“页”就是这个集装箱。
- **典型大小：** 4 KB, 8 KB, 16 KB, 32 KB。

#### 知识点 2：Why Custom-Set Page Size? (为什么允许自定义页大小？)

**详细解释：**
PPT 提到某些 DBMS 允许根据“预期工作负载”调整页大小。这背后的权衡非常微妙：

**场景 A：大页（如 32 KB）的优点与适用场景**

- **优点：**
    1. **减少 I/O 次数：** 一次读取能带回更多数据。对于**顺序扫描**（如数据仓库报表分析、全表统计），大页吞吐量极高。
    2. **降低索引树高度：** 大页能容纳更多的键-指针对，B+ 树更矮，查找路径更短。
    3. **适合大字段：** 对于存储大文本或部分 BLOB 数据，一行就能放进一页，减少行溢出（Row Overflow）的碎片管理开销。
- **缺点：**
    1. **空间浪费：** 如果一行只有 100 字节，一个 32KB 的页里会有大量未使用空间，导致**内部碎片**严重。
    2. **缓存污染：** 在 Buffer Pool 中，你读取了 32KB 但可能只用到了其中 100 字节的数据，剩下的 31.9KB 占据了宝贵的内存缓存位置。

**场景 B：小页（如 4 KB）的优点与适用场景**

- **优点：**
    1. **减少碎片：** 适合**高并发 OLTP（联机事务处理）**，每次交易只涉及几行小记录。小页能更紧凑地利用空间。
    2. **写放大较小：** 修改 100 字节，只需要写回 4KB 的脏页，而大页需要写回 32KB。
    3. **内存利用精细：** Buffer Pool 中缓存的数据更具针对性。
- **缺点：**
    1. **I/O 次数增加：** 扫描大量数据时，需要发出更多的读取请求，系统调用开销变大。

#### 知识点 3：商业产品对比案例（IBM DB2 & Oracle vs. SQL Server）

PPT 给出了非常具体的工程哲学对比，这是理解 DBMS 内部架构差异的好例子。

**1. IBM DB2 和 Oracle：支持自定义（Flexibility Philosophy）**

- **详细机制：**
    - **Oracle：** 支持 `BLOCK SIZE` 参数（如 2k, 4k, 8k, 16k, 32k）。甚至可以在同一个数据库中创建**不同块大小的表空间**。
    - **适用场景：** 可以把索引建在 32k 的表空间（为了降低树高），把存细小日志流的表建在 4k 的表空间。
- **代价：**
    - 管理复杂性高。DBA（数据库管理员）需要非常懂业务，知道哪里的数据是顺序读多还是随机读多，选错了块大小可能导致性能灾难。
    - 在 Buffer Pool 管理上，Oracle 需要配置不同大小的 Buffer Cache 区域来对应不同的块大小（例如 `db_2k_cache_size`, `db_32k_cache_size`），这增加了内存规划的难度。

**2. Microsoft SQL Server：不支持多页大小（Simplicity Philosophy）**

- **详细解释：**
    - SQL Server 强制使用 **8 KB** 的页大小。
    - **理由：** "due to the increased administrative complexity this brings." (由于带来的管理复杂性增加)。
    - **深层逻辑（为什么 8KB 对 SQL Server 是“甜点区”？）：**
        1. **NTFS 文件系统契合度：** Windows NTFS 文件系统的分配单元默认就是 4KB 或 8KB，SQL Server 使用 8KB 可以与操作系统底层 I/O 完美对齐，减少跨存储单元边界的 I/O 开销。
        2. **工程专注度：** 微软的策略是“我们帮你选好了最优解，你别乱改”。这使得 SQL Server 的默认配置在 90% 的场景下表现都不差，且 DBA 不需要纠结页大小配置，降低了运维门槛和出错概率。
        3. **Row-Overflow 机制：** 遇到一行大于 8KB 怎么办？SQL Server 会将大的可变长度列（如 `VARCHAR(MAX)`）自动移动到单独的 **ROW_OVERFLOW_DATA** 分配单元（即行外存储），从而优雅地解决了小页存大行的问题，消除了对大页的硬性需求。

### 总结这两页 PPT 的内在逻辑

1.  **第一页是“因”**：因为磁盘慢，所以 I/O 是瓶颈，所以设计必须围绕减少 I/O。
2.  **第二页是“果”之一**：为了针对不同场景优化 I/O 效率（减少次数 vs 减少传输量），我们需要调整**页大小**这个集装箱的尺寸。但调整尺寸会带来管理开销，所以不同厂商基于不同的设计哲学（灵活 vs 稳定）做出了不同的产品选择。
