这三页PPT围绕一个核心思考题展开：**InnoDB 存储引擎默认的页大小为什么是 16KB，既不能太大也不能太小？**

这是一个非常经典的数据库系统设计权衡问题，涉及**硬件特性、I/O 效率、内存利用率和数据结构开销**四个维度的平衡。

我们按PPT顺序逐页解析其中的工程逻辑。

---

### 第一页：背景引入与硬件对齐

这一页提出了问题，并给出了第一个基础理由——**与硬件物理读写单位对齐**。

#### 1. 硬件的最小读写单位

| 硬盘类型         | 最小读写单位（物理扇区/页） |
| :--------------- | :-------------------------- |
| **机械硬盘**     | 512 字节                    |
| **SSD 固态硬盘** | 4KB（4096 字节）            |

- 操作系统和硬盘控制器在交换数据时，**至少**以这个单位进行读写。即使只想读 1 个字节，硬盘也必须把整个 512B 或 4KB 的物理块读到内存。

#### 2. InnoDB 的选择：16KB

- **16KB = 16384 字节**。
- 它正好是机械硬盘 512B 的 **32 倍**，也是 SSD 4KB 的 **4 倍**。
- **对齐的好处**：避免**跨物理块读写**带来的额外开销。如果页大小不是物理块的整数倍，读一个逻辑页可能触发两次物理 I/O，降低效率。

> 📌 **第一层理由总结**：16KB 是硬件物理块的**整数倍**，保证了 I/O 的**对齐和高效**。

---

### 第二页：为什么页不能太大？（如 16MB）

这一页从“过大”的角度分析弊端，给出了两个核心问题：**I/O 压力**和**查找效率**。

#### 1. I/O 放大与资源浪费

- **假设场景**：页大小设为 16MB（即 16KB 的 1024 倍）。
- **问题一：单次 I/O 压力极大**
    - 每次读取一行数据，都必须将整个 16MB 的页从磁盘加载到内存缓冲池。
    - 16MB 的读取时间远大于 16KB，对于高并发系统，这种 I/O 压力是灾难性的。
- **问题二：空间浪费与缓存污染**
    - 如果这 16MB 的页中只有 1 条记录是需要的，其余 99.99% 的空间都是无用数据。
    - 这些无用数据白白占用了宝贵的内存缓冲池，挤出了其他本可以缓存的热点数据，降低整体命中率。

#### 2. 页内链表查找效率低

- PPT 明确指出：**InnoDB 页中存储的结构是链表**。
- 页内记录通过单向链表按主键顺序连接。
- 在一个 **16MB 的巨大页**内，即使使用二分查找定位到页内某个槽位，后续沿链表遍历仍然可能需要扫描很长的链表。
- 页越大，页内链表越长，**CPU 缓存失效**的概率越高，查找效率越低。

> 📌 **第二层理由总结**：页太大 → **单次 I/O 成本高** + **缓存利用率低** + **页内搜索效率差**。

---

### 第三页：为什么页不能太小？（如 16B）

这一页从“过小”的角度分析弊端，核心问题是**物理块浪费**和**逻辑复杂度增加**。

#### 1. 无法用满一个物理块

- **假设场景**：页大小设为 16 字节。
- **硬件限制**：机械硬盘最小读写单位 512B，SSD 最小 4KB。
- **尴尬局面**：
    - 如果 16B 页**独占**一个 512B 物理扇区，则每个页浪费了 496 字节（96.8% 的空间浪费）。
    - 如果**多个 16B 页共用**一个 512B 物理扇区，则逻辑复杂度急剧上升：InnoDB 需要额外维护“哪些逻辑页在同一个物理块中”的映射关系，且修改其中一个页时必须进行 **Read-Modify-Write** 操作（读出整个 512B → 修改其中 16B → 写回整个 512B），写放大严重。

#### 2. 元数据开销占比过高

- 每个页都有固定的**页头部**和**页尾部**信息（如页号、校验和、上一页/下一页指针等）。
- 如果页大小为 16B，而页头部就占了十几个字节，那么**用户数据几乎存不下**，空间利用率极低。

#### 3. I/O 次数激增

- 如果页很小，读取同样多的数据需要**更多次 I/O**。
- 例如读取 1MB 数据：
    - 使用 16KB 页需要约 64 次 I/O。
    - 使用 16B 页需要约 **65,536 次 I/O**，完全不可接受。

> 📌 **第三层理由总结**：页太小 → **物理块对齐失效** + **元数据开销占比高** + **I/O 次数爆炸**。

---

### 总结：16KB 是如何成为“甜点”的？

| 考量维度     | 页太小（如 16B）           | 页太大（如 16MB）    | **16KB（默认值）**                |
| :----------- | :------------------------- | :------------------- | :-------------------------------- |
| **硬件对齐** | 难以对齐，浪费物理块       | 对齐，但粒度太粗     | **完美对齐（512B/4KB 的整数倍）** |
| **I/O 成本** | I/O 次数过多               | 单次 I/O 太重        | **单次 I/O 轻量，次数可控**       |
| **缓存利用** | 元数据开销大，缓存利用率低 | 大量无用数据污染缓存 | **粒度适中，缓存命中率高**        |
| **页内搜索** | 页内数据太少，跨页搜索多   | 页内链表太长，搜索慢 | **页内记录数适中，链表遍历高效**  |
| **写放大**   | Read-Modify-Write 严重     | 写放大严重           | **写放大可控**                    |

16KB 是 MySQL InnoDB 团队经过长期实践，在 **机械硬盘时代** 找到的通用最优解。随着 SSD 普及，部分数据库（如 PostgreSQL）开始支持更大的页（如 32KB），但 16KB 至今仍是平衡各类场景的**黄金值**。
