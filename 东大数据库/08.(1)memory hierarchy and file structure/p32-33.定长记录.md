### 第一页 PPT 详解：定长记录的定义与表结构示例

#### 标题：定长记录

#### 知识点 1：记录的长度（占用空间）固定

**PPT 给出的表结构定义：**

| 字段 | 类型      | 占用字节数 |
| :--- | :-------- | :--------- |
| 学号 | `char`    | 10 字节    |
| 姓名 | `char`    | 20 字节    |
| 性别 | `char`    | 4 字节     |
| 院系 | `char`    | 40 字节    |
| 绩点 | `decimal` | 4 字节     |

**关键理解：**

- 所有字段均使用 **定长字符类型**（`char`），而非变长类型（`varchar`）。
- 无论实际存储的内容有多长（例如“张三”只有 4 字节），每条记录在磁盘上都**固定占用** `10 + 20 + 4 + 40 + 4 = 78` 字节（加上记录头可能更多）。
- 这种设计牺牲了空间效率，但换来了 **极简的寻址计算**。

**示例数据表：**

| 学号       | 姓名 | 性别 | 院系     | 绩点 |
| :--------- | :--- | :--- | :------- | :--- |
| 2021310001 | 张三 | 男   | 计算机系 | 3.9  |
| 2021310002 | 张四 | 女   | 计算机系 | 3.7  |
| ...        | ...  | ...  | ...      | ...  |

表格中每一行都是一条定长记录，所有行的物理长度完全相等。

#### 知识点 2：记录组织方式

**PPT 给出的记录结构：**

| 记录头 | 学号 | 姓名 | 性别 | 院系 | 绩点 |
| :----- | :--- | :--- | :--- | :--- | :--- |

**记录头（Record Header）包含的信息：**

- **事务信息：** 支持 MVCC（多版本并发控制）所需的元数据，例如创建该记录的事务 ID、删除该记录的事务 ID、回滚指针等。
- **NULL 值位图（Null Bitmap）：** 标记哪些字段的值为 `NULL`。对于定长字段，即使值为 `NULL`，依然会占用完整的字段空间（填充默认值或零），位图只是用于语义上的区分。
- **其他元数据：** 如记录长度（尽管定长时可以省略）、指向溢出页的指针等。

#### 知识点 3：页面内整体布局示意

PPT 下方给出了一张页面布局的示意图：

| 页面头       | 记录数量、checksum、上下页指针等 | 记录删除状态位向量 `(11011...)_2` |
| :----------- | :------------------------------- | :-------------------------------- |
| **记录数据** | 记录0                            | 记录1                             |
|              | 记录5                            | 记录6                             |
|              | 记录10                           | 记录11                            |
|              | ...                              |                                   |

**注意：** 这里展示的是**纯定长记录页面**的一个简化版本。与之前讲过的“分槽页面（Slotted Page）”不同，**定长记录的页面不需要槽数组（Slot Array）**，因为每个记录的位置可以通过**固定偏移量**直接计算出来。

**页面中的三个逻辑区域：**

1. **页面头（Page Header）：** 存放页面级的元数据。
2. **记录删除状态位向量（Deleted Record Bitmap）：** 用于标记哪些记录槽位已被逻辑删除。
3. **记录数据区：** 紧挨着排列的定长记录数组。

**页码标注“3 2”：**

这是 PPT 的页码标识，可能表示“第 3 章第 2 节”或“第 3 部分第 2 页”，帮助讲者定位内容。

### 第二页 PPT 详解：页面内组织形式的详细拆解

#### 标题：定长记录——页面内组织形式

这一页对第一页中的“页面头”部分进行了**放大和细化**，详细解释了每个元数据字段的作用。

#### 知识点 1：页面头（Page Header）的详细组成

PPT 列出了页面头包含的关键字段：

| 字段                     | 作用说明                                                     |
| :----------------------- | :----------------------------------------------------------- |
| **记录数量**             | 当前页面内存储的**有效记录数**（不含已删除记录）。           |
| **Checksum（校验和）**   | 防止磁盘读写过程中发生数据损坏。写入页时计算并存储校验和，读取页时重新计算并比对。如果不一致，说明发生了位翻转或 I/O 错误，DBMS 可据此触发恢复机制（如从副本读取或报错）。 |
| **上下页指针（PageID）** | 用于将多个页面**逻辑串联**成一个链表或 B+ 树结构。例如，在堆文件（Heap File）中，所有数据页通过双向链表连接以支持全表扫描；在 B+ 树索引中，叶子节点页通过左右指针形成有序链表，支持范围查询。 |
| **页面类型**             | 标识该页的用途，常见值包括：数据页（存储表记录）、索引页（存储 B+ 树节点）、Undo 页（存储回滚信息）、系统页（存储元数据）等。Buffer Manager 可根据类型采取不同的缓存策略。 |
| **空闲空间**             | 记录该页当前剩余的可用空间大小（以字节为单位）。对于定长记录页面，空闲空间通常只在页面未满时存在；当页面装满记录后，空闲空间为 0，后续插入将分配新页。 |
| **其他**                 | 可能还包括：所属表空间 ID、文件 ID、页面 LSN（日志序列号，用于 WAL 恢复）等。 |

#### 知识点 2：记录删除状态位向量

- **格式：** 一串二进制位，如 `(11011...)_2`。
- **含义：** 每一位对应页面中的一个**记录槽位（Slot）**。
    - `1`：该槽位当前有效，存放着一条未删除的记录。
    - `0`：该槽位已被逻辑删除（或从未被使用），可以被新插入的记录**覆盖复用**。
- **定长记录场景下的优势：**
    - 删除记录时，只需翻转一个比特位，**不需要物理移动任何数据**。
    - 插入新记录时，**优先寻找位向量中为 `0` 的槽位**（空闲槽位），将其复用；若所有槽位都是 `1` 且页面未满，则追加到末尾；若页面已满，则分配新页。

#### 知识点 3：记录数据区的定长数组结构

**图示再现：**

| 记录数据 | 记录0  | 记录1  | 记录2  | 记录3  | 记录4  |
| :------- | :----- | :----- | :----- | :----- | :----- |
|          | 记录5  | 记录6  | 记录7  | 记录8  | 记录9  |
|          | 记录10 | 记录11 | 记录12 | 记录13 | 记录14 |

**定长记录数组的寻址公式：**

假设：
- 页面数据区起始地址为 `base`（紧接页面头和位向量之后）。
- 每条记录固定占用 `R` 字节（含记录头）。

那么，第 `i` 条记录的**起始内存地址**为：

> **`Address(i) = base + i × R`**

**对比变长记录页面：**
- 变长记录需要先查槽数组（`slot_array[i]` 获取偏移量），再跳转。
- 定长记录无需查表，**一次乘法加一次加法**即可定位任意记录，CPU 开销极低。

**局限性：**

- 如果表结构中有任何一个变长字段，就无法使用这种纯数组布局。
- 定长字段的 `char` 类型如果定义过大（如 `char(255)`），会造成严重空间浪费。
- 因此，现代数据库在存储层通常采用 **分槽页面** 作为通用方案，既能高效处理定长记录，也能灵活支持变长记录。纯定长数组布局更多见于**列式存储**或**嵌入式数据库**（如 SQLite 的某些模式）。

1. **第一页** 通过具体表结构展示了定长记录的概念：所有字段类型均为 `char` 等固定长度类型，每行占用空间完全一致。
2. **第二页** 深入页面内部，解析了**页面头**的关键字段（校验和、上下页指针、页面类型等），以及定长记录页面如何利用**位向量**管理删除与插入，如何利用**固定偏移量数组**实现 O(1) 记录定位。
