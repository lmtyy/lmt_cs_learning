这两页PPT讲的是 **MySQL InnoDB 存储引擎的逻辑存储结构**。这是一个从大到小的五层层级体系：**表空间 → 段 → 区 → 页 → 行**。

第一页是结构概览图，第二页是对各层级的文字补充说明。我将按照从大到小的逻辑顺序，将两页内容融合在一起进行逐层讲解。

---

### 第一层：表空间
**对应第一页顶部 & 第二页顶部文字**

#### 定义与作用
- **表空间** 是 InnoDB 存储引擎的最高逻辑存储单位，它对应的是操作系统层面上的**一个或多个物理文件**（如 `.ibd` 文件）。
- 它就像一个**虚拟的容器**，将表相关的所有数据（索引、记录、回滚信息等）都装在里面。

#### 两种表空间类型（第二页重点）

| 类型           | 说明                                                         | 文件表现                               |
| :------------- | :----------------------------------------------------------- | :------------------------------------- |
| **共享表空间** | 默认方式。**多个表**的数据、索引、元数据等都存储在**同一个**或几个共同的文件中。 | 早期版本著名的 `ibdata1` 文件。        |
| **独占表空间** | 推荐方式。**每个表**拥有自己独立的表空间文件。               | 表名 `.ibd` 文件（如 `student.ibd`）。 |

#### 图例解读（第一页表空间下方的小字）
- **Leaf node segment**：叶子节点段（存放 B+Tree 的叶子节点，即实际数据行）。
- **Non-Leaf node segment**：非叶子节点段（存放 B+Tree 的枝干节点，即索引目录项）。
- **Rollback segment**：回滚段（存放旧版本数据，用于事务回滚和 MVCC 多版本并发控制）。

---

### 第二层：段
**对应第一页中部左侧 & 第二页图例**

#### 定义与作用
- **段** 是为了区分**不同用途的数据**而在表空间内划分的逻辑分区。
- 即使是一个简单的表，InnoDB 也会为其分配不同的段来管理索引树和回滚信息。

#### 常见段类型与图例对应
| 段类型           | 图例对应              | 作用                                           |
| :--------------- | :-------------------- | :--------------------------------------------- |
| **叶子节点段**   | Leaf node segment     | 存放 B+Tree 的最底层数据页（存放具体记录行）。 |
| **非叶子节点段** | Non-Leaf node segment | 存放 B+Tree 的上层索引页（存放指针和键值）。   |
| **回滚段**       | Rollback segment      | 存放 Undo Log（撤销日志），用于事务回滚。      |

> **设计原因**：将索引的上下层分开管理，是为了优化磁盘扫描效率，防止数据页和索引页混杂在一起。

---

### 第三层：区
**对应第一页右侧 & 第二页图例**

#### 定义与作用
- **区** 是 InnoDB 为**段**分配空间的**基本单位**。
- 当一个段需要增长时，它不会一页一页地申请，而是一次性申请一个**区**。

#### 关键特征
- **固定大小**：每个区由 **64 个连续的页** 组成。
- **大小换算**：InnoDB 默认页大小 16KB × 64 = **1MB**。
- **图例解读**：第一页右侧画了 3 个 `Extent` 框，每个里面画了多个 `Page`，表示一个区包含连续多个页。

---

### 第四层：页
**对应第一页底部 & 第二页图例**

#### 定义与作用
- **页** 是 InnoDB 磁盘 I/O 的最小单位。
- 无论读取一行还是多行数据，InnoDB 都会将包含该数据所在的**整个页**加载到内存缓冲池中。

#### 关键特征
- **默认大小**：**16KB**。
- **内部结构**：每个页除了存放用户记录（行），还包含：
    - **Infimum / Supremum**：页内虚拟的最小记录和最大记录（图例中的 `Inf.` 和 `Sup.`），用于协助页内查找。
    - **Page Header / File Trailer**：页头尾校验信息。

---

### 第五层：行
**对应第一页底部左侧 & 第二页图例**

#### 定义与作用
- **行** 是存储用户实际数据的**最小逻辑单位**。

#### InnoDB 行格式中的隐藏字段
PPT 第一页底部的行结构中列出了 `Trx id` 和 `Roll Pointer`，这两个**不属于用户定义的列**，是 InnoDB 自动添加的隐藏字段。

| 字段名              | 含义       | 作用                                                         |
| :------------------ | :--------- | :----------------------------------------------------------- |
| **Trx ID**          | 事务ID     | 记录最近一次修改（插入/更新）该行记录的**事务编号**。用于实现 **MVCC**（多版本并发控制）。 |
| **Roll Pointer**    | 回滚指针   | 指向 **Undo Log**（回滚段）中的旧版本数据。如果事务需要回滚，或者有旧查询需要读取修改前的数据，就通过这个指针去找。 |
| **Col 1 ... Col n** | 用户数据列 | 这才是用户 `CREATE TABLE` 定义的字段（如 `id`, `name`）。    |

> 💡 **深度理解**：为什么第二页图中 `Roll Pointer` 出现了两次？这可能暗示一行记录中包含了多个回滚指针（不同列的数据类型可能有不同的 Undo 处理逻辑），但在常规讲解中，我们只需要知道它是用于访问旧数据的指针即可。

---

### 总结：五层结构的逻辑关系图

```
MySQL InnoDB 表
    └── 表空间 (Tablespace)  ——  物理文件 .ibd / ibdata1
           └── 段 (Segment)    ——  分类存放 (索引数据 / 回滚数据)
                  └── 区 (Extent)   ——  连续空间分配单位 (1MB = 64页)
                         └── 页 (Page)    ——  最小 I/O 单位 (16KB)
                                └── 行 (Row)     ——  具体数据 + 隐藏事务字段
```
