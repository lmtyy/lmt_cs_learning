这三页PPT是对上一讲 **InnoDB 逻辑存储结构**中“段、区、页”三个核心概念的深入细化。如果说上一讲是俯瞰五层结构，这三页就是用放大镜逐一观察每一层的内部构造和运行机制。

我们按PPT顺序逐页深入。

---

### 第一页：段

这一页精确定义了 InnoDB 中**段**的两种类型及其与 B+Tree 的对应关系。

#### 1. 段的两种类型

| 段类型     | PPT 名称              | 对应 B+Tree 部分 | 存储内容                                                     |
| :--------- | :-------------------- | :--------------- | :----------------------------------------------------------- |
| **数据段** | Leaf node segment     | **叶子节点**     | 存放表的**实际数据行**（若为主键索引，则存放完整行记录；若为二级索引，则存放索引键和主键值）。 |
| **索引段** | Non-Leaf node segment | **非叶子节点**   | 存放 B+Tree 的**枝干节点**，即索引目录项（键值 + 指向下一层页的指针）。 |

#### 2. 为什么要把叶子节点和非叶子节点分成两个段？

- **物理隔离，提升性能**：叶子节点（数据页）和非叶子节点（索引页）在磁盘上分开存储。当进行全表扫描时，只需要顺序读取叶子节点段，不会夹杂非叶子节点，提高扫描效率。
- **独立扩展，便于管理**：随着数据量增长，叶子节点段和非叶子节点段可以各自独立申请区，互不干扰。

#### 3. 段的自动管理

- PPT 明确指出：**段由存储引擎自动管理**。
- 用户不需要（也无法）手动指定某个表要多少个段、每个段多大。InnoDB 会根据索引树的增长动态分配和回收段内的区。

---

### 第二页：区

这一页详细说明了**区**的空间构成和分配策略。

#### 1. 区的定义

- **区是由连续页组成的空间**。在磁盘上，一个区内的 64 个页的物理地址是连续的。
- **固定大小**：每个区大小为 **1MB**。
- **换算公式**：64 页 × 16KB/页 = 1024KB = 1MB。

#### 2. 区与 B+Tree 的关系

- PPT 提到：“**此时的页就是 B+Tree 的一个页节点**”。
- 一个区包含 64 个页，这些页可以是 B+Tree 中同一层或相邻层的页节点。

#### 3. 区的分配策略：批量申请

- PPT 关键信息：“**向操作系统申请时通常一次申请 4~5 个区**”。
- **原因**：
    - 减少磁盘 I/O 次数。一次性申请 4~5MB 的连续空间，比每次需要一页时临时申请 16KB 高效得多。
    - 预分配空间，避免频繁的磁盘分配系统调用。
- **图例解读**：PPT 底部画了一长串 `Page` 图标，表示一个区内包含连续的 64 个页。

> 💡 **记忆要点**：**段是逻辑管理单元，区是物理分配单元**。段向表空间要空间时，是以“区”为单位申请的，并且为了效率会一次要 4~5 个区。

### 第三页：页

这一页聚焦于 InnoDB 最核心的 I/O 单位——**页**，并展示了页内部的微观结构。

#### 1. 页的核心属性

| 属性                 | 值 / 说明                                                    |
| :------------------- | :----------------------------------------------------------- |
| **最小 I/O 单位**    | InnoDB 每次从磁盘读取或写入磁盘的数据量至少是一页。即使只想查一行数据，也会把包含该行的整个页加载到内存。 |
| **默认大小**         | **16KB**（可通过 `innodb_page_size` 参数在初始化时调整）。   |
| **与 B+Tree 的关系** | **一个数据页就是 B+Tree 的一个节点**。                       |

#### 2. 页内部结构详解（结合 PPT 右侧图例）

PPT 右侧的图例是一个页的剖面图，从上到下依次是：

| 区域           | 图例标记                                                     | 含义与作用                                                   |
| :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **页头部**     | `Inf.` (Infimum)                                             | **虚拟的最小记录**。页内所有实际记录的逻辑下边界，用于辅助页内查找和链表维护。 |
| **用户记录区** | `R0 W`<br>`R1 W`<br>`R2 W`                                   | 实际存储的用户数据行。<br>`R0`、`R1`、`R2` 代表记录 0、记录 1、记录 2。<br>`W` 可能表示记录的状态标识（如是否已删除）。 |
| **页尾部**     | `Sup.` (Supremum)                                            | **虚拟的最大记录**。页内所有实际记录的逻辑上边界。           |
| **行记录结构** | 左侧图例：<br>`Trx id`<br>`Roll Pointer`<br>`Col 1 ... Col n` | 这是页内单条记录的内部构成（与上一讲一致）：<br>- **Trx id**：最后修改该行的事务ID。<br>- **Roll Pointer**：回滚指针，指向 Undo Log。<br>- **Col 1...n**：用户定义的列数据。 |

#### 3. 页内记录的组织方式

- 在一个页内部，记录**并非物理有序存放**（不按主键值从小到大紧密排列）。
- 而是通过每个记录头部的**指针**形成一条**单向链表**，链表按主键顺序连接 `Infimum → 最小记录 → ... → 最大记录 → Supremum`。
- 这种结构使得插入新记录时只需修改指针，无需搬移大量数据（页内重组只在页满时发生）。

---

### 三页总结与关系串联

| 层级   | 关键属性                  | 管理方式                   |
| :----- | :------------------------ | :------------------------- |
| **段** | 逻辑分类（数据段/索引段） | 自动管理，与索引树一一对应 |
| **区** | 物理连续 1MB（64页）      | 批量申请（一次4~5个区）    |
| **页** | 最小 I/O 单位 16KB        | 页内记录通过链表组织       |

**逻辑链条**：
> 一个 **段** 需要扩展空间 → 向表空间申请 **区**（一次申请 4~5 个）→ 每个区包含 64 个连续 **页** → 每个页内部存放若干 **行**，并通过 `Infimum` 和 `Supremum` 维护页内有序链表。
