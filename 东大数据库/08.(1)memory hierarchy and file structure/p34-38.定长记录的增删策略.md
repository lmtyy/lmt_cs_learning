好的，我们按照 PPT 顺序，详细讲解这五页关于 **定长记录的增删操作** 的内容。这几页展示了在定长记录页面中，当插入新记录或删除旧记录时，DBMS 可以采用的不同策略及其优劣。

### 第一页 PPT 详解：记录添加——追加到页面末尾

#### 标题：记录添加

#### 核心策略：直接追加（Append）

**原文：**
> *如果页面可以容纳新纪录，添加到页面最后*

**操作图示：**

**添加前（6 条记录）：**

| 槽位  | 学号       | 姓名 | 性别 | 院系     | 绩点 |

| 记录1 | 2021310002 | 张四 | 女   | 计算机系 | 3.7  |
| 记录2 | 2021310003 | 张五 | 男   | 计算机系 | 4.0  |
| 记录3 | 2021310004 | 张六 | 男   | 法学系   | 3.7  |
| 记录4 | 2021310006 | 张七 | 女   | 物理系   | 3.4  |
| 记录5 | 2021310009 | 张八 | 女   | 物理系   | 3.6  |

**添加后（7 条记录）：**

| 槽位      | 学号           | 姓名     | 性别   | 院系       | 绩点    |
| :-------- | :------------- | :------- | :----- | :--------- | :------ |
| 记录0     | 2021310001     | 张三     | 男     | 计算机系   | 3.9     |
| 记录1     | 2021310002     | 张四     | 女     | 计算机系   | 3.7     |
| 记录2     | 2021310003     | 张五     | 男     | 计算机系   | 4.0     |
| 记录3     | 2021310004     | 张六     | 男     | 法学系     | 3.7     |
| 记录4     | 2021310006     | 张七     | 女     | 物理系     | 3.4     |
| 记录5     | 2021310009     | 张八     | 女     | 物理系     | 3.6     |
| **记录6** | **2021310013** | **张九** | **女** | **法学系** | **3.7** |

**操作步骤分析：**

1. 检查页面空闲空间是否 ≥ 单条记录长度。
2. 如果空间足够，计算新记录的写入位置：
    - 当前有效记录数为 N（此处为 6）。
    - 新记录写入偏移量 = **页面数据区起始地址 + N × 记录长度**。
3. 将新记录字节拷贝到该位置。
4. 页面头中的 **记录数量** 字段更新为 N+1。

**时间复杂度：** **O(1)**。无需移动任何现有数据，仅需一次内存拷贝和一次计数器更新。

**适用条件：**
- 页面有足够空闲空间。
- 记录槽位是**连续、无空洞**的。这意味着之前从未发生过删除，或者删除时采用了某种“空洞填充”策略。

### 第二页 PPT 详解：记录删除——依次前移法

#### 标题：记录删除

#### 核心策略：物理删除 + 向前移动所有后续记录

**原文：**
> *删除记录，并依次将后续记录向前移动*

**操作图示：**

**删除前（7 条记录，欲删除“记录2”——张五）：**

| 槽位      | 学号           | 姓名     | 性别   | 院系         | 绩点    |
| :-------- | :------------- | :------- | :----- | :----------- | :------ |
| 记录0     | 2021310001     | 张三     | 男     | 计算机系     | 3.9     |
| 记录1     | 2021310002     | 张四     | 女     | 计算机系     | 3.7     |
| **记录2** | **2021310003** | **张五** | **男** | **计算机系** | **4.0** |
| 记录3     | 2021310004     | 张六     | 男     | 法学系       | 3.7     |
| 记录4     | 2021310006     | 张七     | 女     | 物理系       | 3.4     |
| 记录5     | 2021310009     | 张八     | 女     | 物理系       | 3.6     |
| 记录6     | 2021310013     | 张九     | 女     | 法学系       | 3.7     |

**删除后（6 条记录）：**

| 槽位      | 学号           | 姓名     | 性别   | 院系       | 绩点    |
| :-------- | :------------- | :------- | :----- | :--------- | :------ |
| 记录0     | 2021310001     | 张三     | 男     | 计算机系   | 3.9     |
| 记录1     | 2021310002     | 张四     | 女     | 计算机系   | 3.7     |
| **记录2** | **2021310004** | **张六** | **男** | **法学系** | **3.7** |
| **记录3** | **2021310006** | **张七** | **女** | **物理系** | **3.4** |
| **记录4** | **2021310009** | **张八** | **女** | **物理系** | **3.6** |
| **记录5** | **2021310013** | **张九** | **女** | **法学系** | **3.7** |

**操作步骤分析：**

1. 定位到待删除记录（记录2）。
2. 将 **记录3 至 记录6** 的**全部数据**向前复制一条记录的长度。
    - 记录3 的内容覆盖 记录2 的位置。
    - 记录4 的内容覆盖 记录3 的位置。
    - ...
    - 记录6 的内容覆盖 记录5 的位置。
3. 页面头中的 **记录数量** 从 7 减为 6。

**时间复杂度：** **O(K)**，其中 K 是被删除记录**之后**的记录数。在最坏情况下（删除第一条记录），需要移动页内几乎所有记录。

**优点：**
- 页面记录保持**物理连续**，无空洞。
- 插入新记录时依然可以沿用简单的**直接追加**逻辑（如第一页所示）。

**缺点：**
- **移动开销巨大**。如果页面内有数百条记录，删除中间一条就会引发大量内存拷贝。
- **索引失效问题**：如果外部有索引指向这些记录（通过 `PageID, SlotIndex`），移动会导致记录的槽位索引发生变化。原来指向“记录3”的索引，在删除后如果不更新，就会错误地指向了“记录4”的内容（因为记录3物理前移了）。因此，**这种方案在实际数据库系统中很少使用**，除非配合额外的间接层（如索引存的是 Record ID 而非槽位，且 Record ID 不随物理移动而变）。

**删除后（6 条记录）：**

| 槽位      | 学号           | 姓名     | 性别   | 院系       | 绩点    |

| 记录5     | 2021310009     | 张八     | 女     | 物理系     | 3.6     |

**操作步骤分析：**

1. 定位到待删除记录（记录2）。
2. 将 **最后一条记录（记录6）** 的**全部数据**直接复制到 记录2 的位置。
3. 页面头中的 **记录数量** 从 7 减为 6。

**时间复杂度：** **O(1)**。无论删除哪条记录，只复制一条记录的数据（最后那条）。

**优点：**
- **移动开销极小**，恒定时间。
- 页面记录依然保持**物理连续**（无空洞），插入新记录时仍可直接追加。

**缺点：**
- **记录顺序被打乱**：原来在末尾的张九，现在跑到了记录2的位置。如果表需要按插入顺序或主键顺序存储（如聚集索引），这种操作会破坏物理顺序。
- **索引更新问题**：指向“记录6”的外部索引现在需要更新为指向“记录2”的槽位。这仍然需要额外的索引维护开销。相比之下，下一节的位图法完美规避了这个问题。

### 第四页 PPT 详解：记录删除——空闲位置链表法

#### 标题：记录删除

#### 核心策略：维护空闲槽位链表，实现快速复用

**原文：**
> *可用空闲位置链表*
> *插入记录时快速定位第一个空位*

**操作图示：**

**删除前（7 条记录，假设此前已删除过记录1和记录5）：**

PPT 图示显示经过一系列删除后，表变成了：

| 槽位  | 学号       | 姓名 | 性别 | 院系     | 绩点 |
| :---- | :--------- | :--- | :--- | :------- | :--- |
| 记录0 | 2021310001 | 张三 | 男   | 计算机系 | 3.9  |
| 记录1 | 2021310003 | 张五 | 男   | 计算机系 | 4.0  |
| 记录2 | 2021310004 | 张六 | 男   | 计算机系 | 3.7  |
| 记录3 | 2021310009 | 张八 | 女   | 物理系   | 3.6  |
| 记录4 | 2021310013 | 张九 | 女   | 法学系   | 3.7  |

**观察变化：**
- 原记录1（张四）、原记录5（张八？注意此处图示有跳跃，实际展示的是空洞被跳跃填满的结果）。
- 实际上，这张图展示的是**使用了空闲链表管理后**，插入新记录时**跳跃填充**空洞的效果。

**空闲位置链表的工作机制：**

1. **数据结构：**
    - 在页面头或专门的区域，维护一个**单向链表**，链接所有**空闲槽位**。
    - 每个空闲槽位的前几个字节（被当做指针）存储**下一个空闲槽位的索引**。

2. **删除操作：**
    - 将待删除记录的槽位标记为“空闲”。
    - 将该槽位**插入到空闲链表头部**。
    - **不移动任何其他记录数据**。

3. **插入操作：**
    - 检查空闲链表是否为空。
    - 若不空，取出链表头部指向的空闲槽位索引。
    - 将新记录写入该槽位。
    - 从空闲链表中移除该槽位。

**优点：**
- **删除 O(1)，插入 O(1)**。
- **完全不移动现有记录**，外部索引无需更新（只要索引存的是 `PageID + SlotIndex`，而槽位索引在逻辑删除后保持不变，只是内容被新记录覆盖）。

**缺点：**
- **空间碎片化**：记录不再保持插入顺序，物理上可能是跳跃排布的。但这对于大部分 OLTP 场景不是问题。
- **需要额外的空闲空间管理开销**（链表指针）。

### 第五页 PPT 详解：定长记录的 Bitmap 位图管理法

#### 标题：定长记录（bitmap）

这是对第四页链表法的**轻量级替代方案**，也是实际数据库系统中最常见的定长记录空闲槽管理方式。

#### 知识点 1：Bitmap 位图原理

**原文：**
> *Bitmap位图：标记记录是否存在*
> *Insert插入：找到第一个空槽*
> *Delete删除：清除槽标志位*
> *Bitmap数目根据页面大小和记录长度计算*

**工作机制：**

1. **位图结构：**
    - 在页面头中分配一段连续的比特位，**每一位对应页面中一个记录槽位**。
    - 位长度 = **页面最大可容纳记录数**。
    - 例如：页面大小 8KB，每条定长记录 80 字节，最多容纳约 100 条记录，则位图占 100 位 ≈ 13 字节。

2. **位图语义：**
    - `1`：该槽位当前被**有效记录占用**。
    - `0`：该槽位**空闲**（从未使用或已被删除）。

3. **插入操作（Insert）：**
    - 在位图中**从低位向高位扫描**，寻找第一个值为 `0` 的位。
    - 将该位对应的槽位索引 `i` 作为新记录的存放位置。
    - 计算物理地址：`base + i × record_size`，写入记录。
    - 将位图中的该位**置为 1**。
    - 时间复杂度：O(M)，M 为槽位数（通常很小，可借助 CPU 位扫描指令加速，如 `BSF` 指令实现 O(1)）。

4. **删除操作（Delete）：**
    - 根据要删除记录的槽位索引 `i`，直接将位图中第 `i` 位**置为 0**。
    - 无需任何数据移动。
    - 时间复杂度：O(1)。

#### 知识点 2：记录定位示例

**原文：**
> *记录：(Page 2, Record 4)*

PPT 图示：

```
页头
Bitmap
记录1 → 记录2 → 记录3 → 记录4 → 记录5
```

**解读：**

- 一条记录的物理地址由 **(PageID, SlotIndex)** 二元组唯一确定。
- 例如 `(Page 2, Record 4)` 表示该记录位于第 2 号数据页的第 4 号槽位。
- 外部索引（如 B+ 树叶节点）存储的就是这个二元组，称为 **Record ID (RID)** 或 **Tuple ID (TID)**。
- 由于位图法删除时不移动记录，只是翻转位，**RID 在记录的整个生命周期内保持稳定**（除非发生页面重组或跨页移动）。这对于索引维护是**巨大优势**。

#### 知识点 3：Bitmap 大小计算

**公式：**
> 位图字节数 = ⌈(页面有效数据区大小 / 单条记录长度) / 8⌉

**示例：**
- 页面有效数据区 8000 字节。
- 每条定长记录 80 字节。
- 最大记录数 = 8000 / 80 = 100。
- 位图位数 = 100。
- 位图字节数 = 100 / 8 = 12.5 → **13 字节**。

### 五页 PPT 总结对比

| 策略           | 删除开销 | 插入开销                  | 数据移动 | 索引影响                | 空间连续性 |
| :------------- | :------- | :------------------------ | :------- | :---------------------- | :--------- |
| **依次前移法** | O(K)     | O(1) 追加                 | 大量移动 | RID 变化，需更新索引    | 保持连续   |
| **末尾补位法** | O(1)     | O(1) 追加                 | 移动一条 | 最后一条记录的 RID 变化 | 保持连续   |
| **空闲链表法** | O(1)     | O(1) 复用                 | 无       | RID 稳定                | 产生空洞   |
| **位图法**     | O(1)     | O(M) 扫描或 O(1) 硬件加速 | 无       | RID 稳定                | 产生空洞   |

**实际应用：**

- **位图法**是现代 DBMS 管理定长记录页面的**首选方案**（例如 PostgreSQL 的 Heap Page、MySQL InnoDB 的 Page Directory 虽用于变长但也借鉴了类似思想）。
- **末尾补位法**有时用于堆表（Heap Table）中，当没有二级索引时可以使用。
- **依次前移法**几乎仅存在于理论教学或极简单的嵌入式数据库中。
