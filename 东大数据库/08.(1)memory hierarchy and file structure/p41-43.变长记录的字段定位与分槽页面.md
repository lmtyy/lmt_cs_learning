### 第一页 PPT 详解：变长记录的字段定位与存储布局对比

#### 标题：变长记录 vs 定长记录

#### 知识点 1：变长属性通过偏移量和长度定位

**PPT 给出的表结构：**
- 学号，`char(10)`，定长 10 字节
- 姓名，`varchar`，变长
- 性别，`char(4)`，定长 4 字节
- 院系，`varchar`，变长
- 绩点，`decimal`，定长 4 字节

**核心问题：** 在定长记录中，每个字段的起始位置是**编译期固定**的。但在变长记录中，由于“姓名”和“院系”长度不确定，后续字段（如绩点）的物理位置会随之**漂移**，无法直接通过固定偏移访问。

**解决方案：** 在记录头中为每个变长字段存储 **`<起始偏移量，长度>`** 对。

#### 知识点 2：定长记录与变长记录的物理布局对比

**上半部分：定长记录布局**

| 记录头   | 学号       | 姓名 | 性别 | 院系     | 绩点 |
| :------- | :--------- | :--- | :--- | :------- | :--- |
| 定长记录 | 2021310001 | 张三 | 男   | 计算机系 | 3.9  |

- **特点：** 所有字段紧密排列，偏移量固定。
- **缺点：** 姓名字段如果定义为 `char(50)`，即使只存“张三”也要占用 50 字节，空间浪费严重。

**下半部分：变长记录布局**

| 记录头   | 学号       | 性别 | 绩点 | 姓名偏移/长度 | 院系偏移/长度 | 空值向量 |
| :------- | :--------- | :--- | :--- | :------------ | :------------ | :------- |
| 变长记录 | 2021310001 | 男   | 3.9  | 37,6          | 43,12         | 0        |

**字节偏移表：**

| 字段     | 学号 | 性别 | 绩点 | 姓名偏移/长度 | 院系偏移/长度 | 空值向量 |
| :------- | :--- | :--- | :--- | :------------ | :------------ | :------- |
| 起始字节 | 0    | 10   | 20   | 24            | 28            | 32       |

**变长数据实际存储位置（图示底部）：**
- 字节 36 处开始存储变长数据（具体取决于记录头大小和填充）。
- 姓名数据位于偏移 **37**，长度 **6** 字节。
- 院系数据位于偏移 **43**，长度 **12** 字节。
- 字节 **54** 为记录结束位置。

**布局的核心设计原则：**

1. **定长字段前置**：学号、性别、绩点这些定长字段被放在记录的**前半部分固定偏移处**。这样无论变长字段多长，要访问“绩点”字段，DBMS 始终从记录起始位置偏移 20 字节处读取即可。

2. **变长字段用偏移量表替代**：姓名字段本身不存数据，而是存一个指针对 `(37, 6)`，告诉 DBMS：“去偏移 37 字节处读取 6 个字节，那就是姓名”。

3. **变长数据统一后置**：真正的字符串数据放在记录尾部（或页面数据区的其他位置），这样记录头部分保持相对紧凑，且便于整条记录在页面内移动时只需重写头部的偏移量信息。

### 第二页 PPT 详解：分槽页面结构（Slotted Page Structure）

#### 标题：Variable-Length Records / Slotted Page Structure

这是管理变长记录的**页面级容器结构**，是现代数据库（PostgreSQL、MySQL InnoDB 等）的标准实现方式。

#### 知识点 1：分槽页面的整体布局

**PPT 图示：**

```
┌─────────────┬──────────────────────────────────┐
│ Block Header │            Records               │
├─────────────┤                                    │
│   Size      │                                    │
│ # Entries   │                                    │
│ Free Space  │                                    │
│  Location   │                                    │
├─────────────┤                                    │
│ End of Free │                                    │
│    Space    │                                    │
└─────────────┴──────────────────────────────────┘
```

**关键元素解析：**

| 元素                    | 说明                                                         |
| :---------------------- | :----------------------------------------------------------- |
| **Size**                | 页面总大小（如 8KB）                                         |
| **# Entries**           | 当前页面中存储的**记录条目数**（即槽位数量）                 |
| **Free Space Location** | 指向**空闲空间起始位置**的指针。由于槽数组向下增长、记录数据向上增长，这个指针标记了中间空闲区域的边界。 |
| **End of Free Space**   | 空闲空间的结束位置（即记录数据区的最顶端）。                 |

#### 知识点 2：分槽页面头的详细内容

**原文：**
> *Slotted page header contains:*
> - *number of record entries*
> - *end of free space in the block*
> - *location and size of each record*

**第三条是核心：**
- 页面头中**紧挨着一个数组**，数组的每个元素对应一条记录，存储该记录的 **`<起始偏移量，长度>`**。
- 这个数组就是**槽数组（Slot Array / Item Pointer Array）**。
- 槽数组**从页头之后开始，向页面底部方向增长**。

#### 知识点 3：记录可以自由移动而不影响外部引用

**原文：**
> *Records can be moved around within a page to keep them contiguous with no empty space between them; entry in the header must be updated.*

- 当页面内部产生碎片（删除记录留下的空洞），DBMS 可以选择**整理（Compact / Vacuum）**页面：
    1. 将存活记录**物理地**向页面底部**紧凑排列**，消除中间的空洞。
    2. **关键操作**：更新槽数组中**每条被移动记录的偏移量指针**，使其指向新的物理位置。
    3. **槽位索引（Slot Index）保持不变**。例如，原来在槽 3 的记录，物理位置从偏移 400 移到了偏移 200，槽 3 中存储的偏移量从 400 更新为 200。
- **结果**：外部索引通过 `(PageID, SlotIndex)` 引用记录。由于 **SlotIndex 从未改变**，索引**完全不需要更新**。

#### 知识点 4：指针的间接层设计

**原文：**
> *Pointers do not point directly to record — instead they point to the entry for the record in header.*

这是分槽页面最重要的抽象思想：

```
外部索引
    ↓
(PageID=5, SlotIndex=3)
    ↓
页面5 的 槽数组[3]  →  存储: (偏移量=200, 长度=78)
    ↓
页面内偏移200字节处  →  实际记录数据
```

**对比直接指针：**
- 如果索引存的是**记录在文件中的绝对字节偏移量**，记录一旦移动，**所有指向它的索引条目都必须更新**。这在 OLTP 高并发下是不可接受的。
- 通过引入**槽数组**这一间接层，记录在页面内部的物理移动被**完全封装在页面内部**，对外部模块透明。

### 第三页 PPT 详解：变长记录页面组织的中文图示

#### 标题：页面组织-变长记录

这一页用中文图文并茂地总结了分槽页面的核心设计。

#### 知识点 1：页面头的组成

**原文：**
> *一页面头：页类型、状态*
> *一槽位：记录起始地址、长度（索引会指向槽位）*

页面头包含：
- 页类型（数据页、索引页、Undo 页等）
- 状态标志（是否已刷盘、是否有校验错误等）
- 记录数量
- 剩余空间结束位置（End of Free Space）
- 上下页指针（用于堆表扫描或 B+ 树叶节点链表）
- Checksum 校验和

#### 知识点 2：双向增长（两头挤）设计

**原文：**
> *一两头挤：页面、槽位等页面内从前往后；记录从后往前*

**图示还原：**

```
页面起始
┌─────────────────────────────────────────────────────────────┐
│                        页面头                               │
│  (记录数量、剩余空间结束位置、checksum、上下页指针等)          │
├─────────────────────────────────────────────────────────────┤
│                    槽位数组（Slot Array）                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 记录0长度，偏移量 │ 记录1长度，偏移量 │ 记录2长度，偏移量 │   │
│  └─────────────────────────────────────────────────────┘   │
│                         ↓ 向下增长                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    剩  余  空  间                            │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                         ↑ 向上增长                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                     记录2                           │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │                     记录1                           │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │                     记录0                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                    记录数据区                               │
└─────────────────────────────────────────────────────────────┘
页面结束
```

**双向增长的优势：**

1. **空间利用最大化**：槽数组和记录数据共享同一块空闲空间，只要两者不碰头，就可以继续插入新记录。
2. **无需预先划分槽位数量**：槽数组可以根据实际插入的记录数动态增长，不像定长记录页面需要预先计算最大槽位数并预留位图空间。
3. **变长记录的天然支持**：记录从底部开始向上堆放，每条记录占用长度不同也完全没问题。

#### 知识点 3：操作流程示例

**插入新记录：**
1. 检查剩余空间是否足够容纳：新记录的**槽数组条目**（通常 4 字节）+ 新记录的**实际数据**。
2. 在**剩余空间**中分配：
    - 槽数组在顶部**向下**新增一个条目（记录新记录的偏移量和长度）。
    - 记录数据在底部**向上**存放实际字节。
3. 更新页面头中的：
    - 记录数量 +1
    - 剩余空间结束位置（下移）

**删除记录：**
1. 将对应槽数组条目标记为“未使用”（例如长度为 0，或偏移量为特殊值）。
2. 记录数据暂时保留在原地（形成空洞）。
3. 可选地，后续 Vacuum 进程会整理页面，将存活记录向上紧凑排列，更新槽数组偏移量，释放连续空闲空间。

### 三页 PPT 逻辑关系总结

| 页码       | 核心内容                               | 解决的问题                                                   |
| :--------- | :------------------------------------- | :----------------------------------------------------------- |
| **第一页** | 变长记录内部的字段定位（偏移量表）     | 如何在一条记录内部快速找到变长字段的值                       |
| **第二页** | 分槽页面结构（Slotted Page）的抽象原理 | 如何在页面级别管理多条变长记录，并实现记录的物理移动对索引透明 |

**核心结论：**
- **第一页**讲的是**微观**（一条记录内部的字节排布）。
- **第二、三页**讲的是**中观**（一个页面内多条记录的集合管理）。
- 两者结合，构成了 DBMS 存储变长数据的完整物理层方案。
