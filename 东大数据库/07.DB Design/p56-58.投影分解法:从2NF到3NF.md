## 第1页：问题的原因与解决方法

### 1. 原因回顾
- 在关系模式 **SL(Sno, Sdept, Sloc)** 中，存在函数依赖：
  - `Sno → Sdept`
  - `Sdept → Sloc`
- 由此导出传递依赖：**Sloc 传递函数依赖于 Sno**（因为 Sno → Sdept 且 Sdept → Sloc，而 Sdept 不是候选码）。
- 正是这个传递依赖导致了之前提到的插入异常、删除异常、数据冗余和修改复杂等问题。

### 2. 解决方法：投影分解法
- 为了消除传递依赖，将 SL 分解为两个关系模式：
  - **SD(Sno, Sdept)**：包含学号和系名。
  - **DL(Sdept, Sloc)**：包含系名和住处。
- 分解后：
  - **SD的码为 Sno**（每个学生属于一个系）。
  - **DL的码为 Sdept**（每个系只有一个住处）。
- 这样，原来的传递依赖 `Sno → Sdept → Sloc` 被拆分到两个独立的关系中，不再存在非主属性对码的传递函数依赖。

## 第2页：分解后的关系模式图示

- 这一页用更直观的方式展示了分解结果：
  - **SD**：两个属性 Sno（学号）和 Sdept（系名）。  
    - 函数依赖：`Sno → Sdept`，Sno 是主码。
  - **DL**：两个属性 Sdept（系名）和 Sloc（住处）。  
    - 函数依赖：`Sdept → Sloc`，Sdept 是主码。
- 两个关系通过 **Sdept** 属性可以自然连接，恢复原信息。
- 此时，每个关系都满足**第三范式（3NF）**：
  - SD：只有直接依赖，无传递依赖。
  - DL：Sdept 是码，Sloc 直接依赖于码，无传递依赖。

## 第3页：分解后解决的问题

这一页明确指出，分解为 SD 和 DL 后，**在一定程度上**（实际上是完全）解决了之前 SL 存在的四个问题：

### 1. 插入异常 —— 已解决
- **原问题**：无法插入没有在校学生的系的信息。
- **解决后**：可以在 **DL 关系**中直接插入一个系的 `Sdept` 和 `Sloc`，而不需要依赖 SD 中有该系的学生记录。  
  *例如：新增“网络空间安全系”，即使没有学生，也能在DL表中添加一行。*

### 2. 删除异常 —— 已解决
- **原问题**：某系学生全部毕业，删除学生信息时连同系的信息一起丢失。
- **解决后**：删除学生时，只需从 **SD 关系**中删除对应学生的元组。**DL 关系**中关于该系的信息仍然保留，不会丢失。

### 3. 数据冗余 —— 已解决
- **原问题**：同一个系的住处信息在学生记录中重复存储。
- **解决后**：**DL 关系**中每个系只存储一次住处信息。SD 关系中不存储住处，因此完全消除了冗余。

### 4. 修改复杂（更新异常） —— 已解决
- **原问题**：修改某系的住处需要更新该系所有学生的 Sloc 值。
- **解决后**：只需在 **DL 关系**中修改对应系的 Sloc 属性值（修改一行），SD 关系无需改动，避免了不一致性。

## 总结：从2NF到3NF的关键步骤

| 步骤            | 内容                               |
| --------------- | ---------------------------------- |
| 1. 识别传递依赖 | 找到非主属性对码的传递函数依赖     |
| 2. 投影分解     | 将传递依赖的两个部分拆分为两个关系 |
| 3. 独立存储     | 使每个非主属性只直接依赖于候选码   |
| 4. 获得3NF      | 分解后的每个关系均满足3NF          |

这组PPT完整演示了**一个2NF关系如何通过投影分解法转换为3NF关系，以及转换后如何消除异常和冗余**。这是数据库规范化理论中非常经典的案例。
