这四页PPT非常硬核，它们介绍的是数据库设计中一个非常经典且实用的算法——**3NF 分解算法 (3NF Decomposition Algorithm)**，也常被称为**保持依赖的 3NF 合成算法 (Synthesis Algorithm for 3NF)**。

与 BCNF 那种“发现违规就拆分”的破坏性算法不同，3NF 算法采用的是一种**“搭积木（合成）”**的建设性思维。下面我按照PPT顺序为你详细拆解。

------

### 第一页PPT：3NF 分解（合成）算法的伪代码

这页展示了算法的标准化执行步骤，我们一段一段来看：

**1. 准备工作：求正则覆盖 (Canonical Cover)**

- **代码：** `Let Fc be a canonical cover for F; i := 0;`
- **知识点：** 这是算法的**绝对第一步**。拿到原表的函数依赖集 $F$ 后，不能直接用，必须先“挤水分”。你要算出它的正则覆盖 $F_c$（也叫最小依赖集）。
  - 这意味着你要消除 $F$ 中所有多余的规则（比如能被其他规则推导出来的冗余依赖），并且消除每条规则左右两边多余的属性。这一步是为了确保我们后面搭的“积木”是最精简的。

**2. 核心合成：根据规则建表**

- **代码逻辑：** 遍历 $F_c$ 中的每一条精简后的规则 $\alpha \rightarrow \beta$。
  - `if none of the schemas... contains \alpha \beta`：检查一下，我们目前建好的表里面，有没有哪张表已经把 $\alpha$ 和 $\beta$ 这两个属性都包含进去了？
  - `then begin ... Ri := \alpha \beta`：如果没有，那我们就**单独为这条规则建一张新表**，这张表的属性就是 $\alpha$ 和 $\beta$ 的并集。
- **思想精髓：** 这就是“保持依赖”的秘诀！既然我们害怕拆表会把规则切断，那我就干脆**把每一条精简后的规则都变成一张独立的物理表**。这样规则自然就保留在单表内部了。

**3. 兜底保障：确保无损连接**

- **代码逻辑：** `if none of the schemas... contains a candidate key for R` ... `Ri := any candidate key for R;`
- **知识点：** 前面按规则建表，保证了“保持依赖”。但万一建完所有的表之后，发现**原表的主键（候选码）没有任何一张表能完整包含**怎么办？
  - 如果主键被打散了，以后 JOIN 的时候就会出现虚假数据，导致无法“无损连接”。
  - **兜底方案：** 算法强行干预，单独拿原表的一个完整候选码出来，为它专门建一张只有主键属性的“桥梁表”。这就死死保住了“无损连接”的底线。

### 第二页PPT：3NF 算法的神奇保证 (Algorithm Guarantees)

这页非常简短，但字字千金。它宣告了上一页那个算法的伟大之处：

- **Above algorithm ensures (上述算法保证了以下三点绝对成立)：**
  1. **each relation schema $R_i$ is in 3NF：** 生成出来的每一张小表，绝对都是满足 3NF 标准的。
  2. **decomposition is dependency preserving：** 这个分解过程绝对不会丢失任何一条原有的业务规则（因为我们就是按规则建表的）。
  3. **and lossless-join：** 凭借最后的兜底操作，这个分解绝对能完美拼凑回原表的数据。

这就是为什么业界如此推崇 3NF，因为有了这个算法，我们就能以一套机械化的流程，100% 获得一个既高效又安全的数据库设计方案。

### 第三页PPT：3NF 分解实例 - 引入案例

接下来两页是用一个具体的例子来“跑”一遍这个算法。这页给出了初始场景：

- **关系模式：** `cust_banker_branch` = (`customer_id`, `employee_id`, `branch_name`, `type`)
  - 代表：客户号，员工号（专属理财经理），分行名，账户类型。
- **给定的初始函数依赖 ($F$)：**
  1. `customer_id, employee_id` $\rightarrow$ `branch_name, type` (客户和他的理财经理，决定了他们所属的分行和账户类型)
  2. `employee_id` $\rightarrow$ `branch_name` (一个员工只属于一个确定的分行)
  3. `customer_id, branch_name` $\rightarrow$ `employee_id` (一个客户在一个特定分行里，只有唯一一个专属理财经理)

------

### 第四页PPT：3NF 分解实例 - 第一步：求正则覆盖

按照算法，动手第一步永远是**给初始的 $F$ 挤水分，求正则覆盖 $F_c$**。

**分析初始依赖 1 和 2：**

- 依赖 1：`customer_id, employee_id` $\rightarrow$ `branch_name, type`

- 依赖 2：`employee_id` $\rightarrow$ `branch_name`

- **发现冗余 (Extraneous attribute)：**

  你看依赖 1 的右边有 `branch_name`。但是根据依赖 2，我们知道单凭 `employee_id` 自己就能推导出 `branch_name` 了。

  在依赖 1 左边的组合里明明有 `employee_id`，那右边再写一个 `branch_name` 就是纯纯的废话（冗余）！

  *PPT 原文指出：branch_name is extraneous in the r.h.s. (right hand side) of the 1st dependency.*

**生成精简版规则集 ($F_c$)：**

我们要把依赖 1 右边的 `branch_name` 砍掉。除此之外，没有其他可以精简的地方了。所以我们得到了最终用于“搭积木”的正则覆盖 $F_c$：

1. `customer_id, employee_id` $\rightarrow$ `type` (砍掉冗余后的规则 1)
2. `employee_id` $\rightarrow$ `branch_name` (原规则 2)
3. `customer_id, branch_name` $\rightarrow$ `employee_id` (原规则 3)

*(注：后续的 PPT 应该会接着演示如何把这三条 $F_c$ 变成三张物理表，并检查候选码。但这四页到这里就结束了第一阶段的准备工作。)*
