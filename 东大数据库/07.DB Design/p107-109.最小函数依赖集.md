这三页 PPT 是数据库规范化理论中非常具有决定性的一环：**求 Canonical Cover（正交集，国内教材也常称为“最小函数依赖集”或“最小覆盖”）**。

如果说前面的课程教了你如何“判断等价”和“抓出多余属性”，那么这三页 PPT 就是把这些招式组合起来，教你如何从头到尾、彻彻底底地把一个臃肿的规则库，洗练成最纯粹、最干净的终极形态（即正交集 $F_c$）。

下面我按照 PPT 的顺序为你极其详细地拆解：

------

### 第一页：什么是 Canonical Cover (正交集)？—— 【定义篇】

这页 PPT 给出了正交集 $F_c$ 的严格数学定义。要让一个函数依赖集 $F_c$ 有资格被称为原集合 $F$ 的“正交集”，必须**同时满足以下三个严苛的条件**：

1. **$F$ logically implies all dependencies in $F_c$, and $F_c$ logically implies all dependencies in $F$**
   - **详细解释**：这就是上一节课学的“等价”。$F$ 能够推导出 $F_c$ 的所有规则，且 $F_c$ 也能推导出 $F$的所有规则（即 $F^+ = F_c^+$）。
   - **大白话**：不管你怎么精简，**绝对不能改变原有的业务逻辑**。精简后的规则库，其威力必须和原规则库一模一样。
2. **No functional dependency in $F_c$ contains an extraneous attribute**
   - **详细解释**：$F_c$ 中的**任何一条规则**，无论是左边（前提条件）还是右边（推导结果），都**绝对不能包含多余属性**。
   - **大白话**：里面没有任何一个“混子”或“废话”。（这就是上一节课“踢出左右两侧多余属性”算法的最终目的）。
3. **Each left side of functional dependency in $F_c$ is unique**
   - **详细解释**：在 $F_c$ 中，箭头左边的属性（或属性组合）必须是唯一的。
   - **大白话**：如果你有规则 $A \rightarrow B$ 和 $A \rightarrow C$，它们的左边都是 $A$，这是不允许的。你必须把它们合并成一条规则：$A \rightarrow BC$。这保证了规则库的形式最简，方便后续转化为数据库表结构。

------

### 第二页：如何计算 Canonical Cover？—— 【算法篇】

既然知道了终极目标长什么样，这页 PPT 就给出了达到目标的**具体执行步骤（伪代码算法）**。

这是一个 `repeat ... until`（重复执行直到没有变化）的死循环过程，主要包含两大核心动作：**合并**与**删减**。

- **动作 1：合并同类项 (Use the union rule...)**
  - **PPT 内容**：Use the union rule to replace any dependencies in $F$, $\alpha_1 \rightarrow \beta_1$ and $\alpha_1 \rightarrow \beta_2$ with $\alpha_1 \rightarrow \beta_1 \beta_2$. *(注：PPT 这里有个小笔误，第二个 $\beta_1$ 应该是 $\beta_2$)*。
  - **解释**：扫描整个规则库，只要发现有两条规则的**左边完全一样**，就用“合并律”把它们揉成一条。比如把 $A \rightarrow B$ 和 $A \rightarrow C$ 合并为 $A \rightarrow BC$。
- **动作 2：抓出并剔除混子 (Find... an extraneous attribute...)**
  - **PPT 内容**：Find a functional dependency $\alpha \rightarrow \beta$ with an extraneous attribute either in $\alpha$ or in $\beta$. If an extraneous attribute is found, delete it...
  - **解释**：利用上一节课学的测试算法，挨个检查每条规则的左边和右边，发现多余属性就无情删掉。比如 $AB \rightarrow C$ 测试出 $B$ 是多余的，就删掉 $B$ 变成 $A \rightarrow C$。
- **循环的奥秘 (Note 备注部分)**
  - **PPT 内容**：Note: Union rule may become applicable after some extraneous attributes have been deleted, so it has to be re-applied.
  - **核心灵魂**：为什么需要 `repeat` 循环？因为**删减操作可能会创造出新的合并机会**！
  - *举个生动的例子*：假设现在有两条规则：$A \rightarrow D$ 和 $AB \rightarrow C$。目前左边不同，无法合并。但是在“动作 2”中，你发现 $AB \rightarrow C$ 里面的 $B$ 是多余的，把它删了，规则变成了 $A \rightarrow C$。此时奇迹发生了，现在你有 $A \rightarrow D$ 和 $A \rightarrow C$ 了，它们的左边变得一样了！所以你必须再跑一圈“动作 1”，把它们合并成 $A \rightarrow CD$。
  - 一直重复这个“合并 $\rightarrow$ 删减 $\rightarrow$ 再合并 $\rightarrow$ 再删减”的过程，**直到整个规则库一个字都改不动了 (`until F does not change`)**，恭喜你，你得到的最终结果就是 Canonical Cover。

------

### 第三页：正交集实例分析 —— 【实战检验篇】

这页 PPT 用一个具体的数据库模式，给你展示了什么是“干净的正交集”，什么是“带有冗余的非正交集”。

- **背景设定：**
  - $U = \{SNO, SDEPT, MN, CNAME, G\}$ （学号，所在系，系主任，课程名，成绩）。
- **观察干净的集合 $F$：**
  - $F = \{SNO \rightarrow SDEPT, SDEPT \rightarrow MN, (SNO, CNAME) \rightarrow G\}$
  - **分析**：学号决定系，系决定系主任，(学号+课程名)联合决定成绩。这里面左边各不相同（满足唯一性），且逻辑非常紧凑，没有任何一条规则能被别人推导出来。**因此，$F$ 已经是一个完美 Canonical Cover。**
- **观察臃肿的集合 $F'$（找茬游戏）：**
  - $F' = \{SNO \rightarrow SDEPT,$ <span style="color:blue">$SNO \rightarrow MN$</span>$, SDEPT \rightarrow MN, (SNO, CNAME) \rightarrow G,$ <span style="color:blue">$(SNO, SDEPT) \rightarrow SDEPT$</span>$\}$
  - **分析**：为什么 $F'$ 不是正交集？PPT 底部给出了数学证明：
    1. **$F' - \{SNO \rightarrow MN\} = F'$**
       - 意思是：即便我把 $SNO \rightarrow MN$ 从库里删掉，剩下的规则依然能推导出它（因为 $SNO \rightarrow SDEPT$ 且 $SDEPT \rightarrow MN$，根据传递律，自然就能得出 $SNO \rightarrow MN$）。所以这条规则是**纯纯的全局冗余**，必须剔除。
    2. **$F' - \{(SNO, SDEPT) \rightarrow SDEPT\} = F'$**
       - 意思是：$(SNO, SDEPT) \rightarrow SDEPT$ 是一条**平凡函数依赖**（自己的一部分当然能决定自己的一部分，就像“知道学号和系，当然知道系”一样）。这种全是废话的规则在正交集中是绝对不能容忍的。

**总结：**

这三页 PPT 完整地闭环了规范化理论的底层逻辑。在现实的数据库系统（如 MySQL, Oracle）底层设计理论中，**把复杂的业务逻辑提炼成 Canonical Cover**，是进行数据库表拆分（消除数据冗余和更新异常，达到 3NF 或 BCNF）的绝对前提条件。
