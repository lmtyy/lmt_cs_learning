## 第1页：第一范式（First Normal Form）

### 核心定义
> 一个关系模式 R 满足**第一范式（1NF）**，当且仅当 R 的所有属性的**域（domain）都是原子的（atomic）**。

### 什么是“原子域”？
- 原子意味着域中的每个值被认为是**不可再分的基本单位**。
- 换句话说，一个属性不能包含：
  - **多个值**（如一组名字、列表、集合）
  - **复合结构**（如组合属性：地址可以拆分为省、市、街道）
  - **可拆分的有意义编码**（如课程编号 `CS101` 中 `CS` 代表专业，`101` 代表序号）

### 非原子属性的典型例子
| 非原子类型 | 示例                                           |
| ---------- | ---------------------------------------------- |
| 多值集合   | `{‘张三’, ‘李四’}` 作为“学生姓名”属性的值      |
| 复合属性   | `(北京, 海淀区, 中关村大街)` 作为一个“地址”值  |
| 可拆分编码 | `CS101` 作为“课程号”，实际包含专业和序号两部分 |

### 重要结论
- **1NF 是关系数据库的最低要求**。如果一个表不满足 1NF，它就不属于关系数据库（不能直接用关系模型表达）。
- 但满足 1NF 的关系模式**不一定就是好的设计**，仍然可能存在数据冗余、更新异常等问题。

## 第2页：第一范式（续）—— 局限性说明

> 第一范式是对关系模式的最起码的要求。不满足第一范式的数据库模式不能称为关系数据库。  
> 但是满足第一范式的关系模式并不一定是一个好的关系模式。

### 为什么 1NF 还不够好？
即使所有属性都是原子的，依然可能存在：
- **数据冗余**：相同信息在多个元组中重复出现。
- **插入异常**：有些信息无法单独插入，因为缺少主键的其他部分。
- **删除异常**：删除某些信息时会意外丢失其他相关信息。
- **更新异常**：修改一个值需要改动多处，容易导致不一致。

### 举例说明（1NF 但设计不佳）
考虑一个选课表 `SC(学号, 姓名, 系名, 课程号, 成绩)`，假设：
- 一个学生只能属于一个系。
- 但学生选修多门课，则“学号→姓名, 系名”这个函数依赖会导致：每个学生的系名在每一门选课记录中重复存储。

**问题**：
- 如果学生转系，需要修改所有该学生的选课记录中的系名，否则不一致。
- 如果学生没有选课，则无法在表中记录其姓名和系名（插入异常）。

## 第3页：目标 —— 建立理论来解决这些问题

> 我们的理论基于：  
> - 函数依赖（functional dependencies）  
> - 多值依赖（multivalued dependencies）

### 为什么需要依赖理论？
为了解决 1NF 中仍然存在的冗余和异常问题，我们需要**更精细的约束理论**来指导关系模式的设计。这些约束能够表达属性之间的依赖关系，从而帮助我们将关系分解成更合理的形式（如 2NF、3NF、BCNF、4NF）。

### 1. 函数依赖（FD）
- **定义**：对于关系 R 中的属性子集 X 和 Y，若任意两个元组在 X 上的值相同，则它们在 Y 上的值也必然相同，记作 X → Y。
- **示例**：`学号 → 姓名`（一个学号唯一确定一个学生姓名）。
- **作用**：检测冗余、判断主键、指导分解。

### 2. 多值依赖（MVD）
- **定义**：对于关系 R 中的属性子集 X、Y、Z（其中 Z = R - X - Y），若给定 X 的值，Y 的取值集合与 Z 的取值集合相互独立，则称 X 多值决定 Y，记作 X ↠ Y。
- **示例**：一个教师可以讲授多门课程，也可以参与多个研究项目，且课程和项目之间没有关系。则 `教师ID ↠ 课程` 且 `教师ID ↠ 项目`。
- **作用**：用于处理比函数依赖更复杂的冗余，解决 4NF 的分解问题。

### 理论的目标
通过分析这些依赖，可以：
- 形式化定义“好的关系模式”标准（范式：2NF、3NF、BCNF、4NF）。
- 提供规范化算法（无损分解、依赖保持）。
- 自动或半自动地将不理想的 1NF 模式分解成更优的范式。
