这几页PPT展示了同一个查询需求（查找在Perryridge分行同时拥有账户和贷款的客户）的三种不同SQL实现方式。下面我将逐一详细解析每一页的写法，指出其中的关键知识点和潜在问题。

---

## 第1页：问题描述与提示

**内容回顾**  
- 给出了四个表：`Loan`、`Account`、`Borrower`、`Depositor`。
- 需求：找出同时在Perryridge分行有账户和贷款的客户。
- 提示了两种实现思路：**连接**和**子查询**。

**详细讲解**  
- 这是一个典型的“交集”查询，需要同时满足两个条件：
  - 客户在Perryridge分行有贷款（通过 `Borrower` 和 `Loan` 关联）。
  - 客户在Perryridge分行有账户（通过 `Depositor` 和 `Account` 关联）。
- 同时要求客户是同一个人，因此还需要通过客户名将两边关联起来。
- 两种思路各有千秋，后面几页分别展示了具体写法。

---

## 第2页：多表连接写法

**原始SQL**（已修正拼写错误）：
```sql
SELECT DISTINCT customer_name
FROM borrower, loan, account, depositor
WHERE borrower.loan_number = loan.loan_number
  AND loan.branch_name = 'Perryridge'
  AND depositor.account_number = account.account_number
  AND account.branch_name = 'Perryridge'
  AND borrower.customer_name = depositor.customer_name;
```

### 逻辑解析
1. **隐式连接**：`FROM` 子句中用逗号列出四个表，产生笛卡尔积。然后通过 `WHERE` 子句中的条件筛选出有效行。
2. **连接条件**：
   - `borrower.loan_number = loan.loan_number`：将贷款人与贷款信息关联。
   - `depositor.account_number = account.account_number`：将存款人与账户信息关联。
   - `borrower.customer_name = depositor.customer_name`：确保贷款人和存款人是同一客户。
3. **过滤条件**：
   - `loan.branch_name = 'Perryridge'`：只考虑Perryridge分行的贷款。
   - `account.branch_name = 'Perryridge'`：只考虑Perryridge分行的账户。
4. **去重**：一个客户可能有多笔贷款或多个账户，连接后会产生多行，因此使用 `DISTINCT` 确保每个客户只出现一次。

### 优缺点
- **优点**：逻辑直接，符合关系代数思想，所有条件集中在一个查询中。
- **缺点**：书写繁琐，容易漏写连接条件；四个表笛卡尔积可能产生巨大中间结果，但现代优化器会将其优化为合适的连接顺序。

### 知识点
- 隐式连接（逗号连接）与显式 `JOIN` 的等价性。
- 多表连接时，必须通过等值条件将相关表关联起来，避免无意义的笛卡尔积。
- `DISTINCT` 用于消除因一对多关系产生的重复行。

## 第3页：混合写法（连接+子查询）——存在错误

**原始SQL**（有笔误）：
```sql
SELECT DISTINCT customer_name
FROM account, depositor
WHERE borrower.account_number = account.account_number   -- 错误！
  AND account.branch_name = 'Perryridge'
  AND customer_name IN (
      SELECT customer_name
      FROM loan, borrower
      WHERE borrower.loan_number = loan.loan_number
        AND loan.branch_name = 'Perryridge'
  );
```

### 错误分析
- 在 `FROM` 子句中只列出了 `account` 和 `depositor`，但 `WHERE` 条件中却出现了 `borrower.account_number`，而 `borrower` 表并没有被引入。这会导致语法错误或语义错误（如果数据库有默认表引用规则，但通常报错）。
- **正确写法**应该是使用 `depositor.account_number` 而不是 `borrower.account_number`，因为外层是从 `depositor` 和 `account` 连接。修正后的查询应为：
  ```sql
  SELECT DISTINCT customer_name
  FROM account, depositor
  WHERE depositor.account_number = account.account_number
    AND account.branch_name = 'Perryridge'
    AND customer_name IN (
        SELECT customer_name
        FROM loan, borrower
        WHERE borrower.loan_number = loan.loan_number
          AND loan.branch_name = 'Perryridge'
    );
  ```

### 修正后的逻辑
1. **外层查询**：通过 `account` 和 `depositor` 连接，找出所有在Perryridge分行有账户的客户及其客户名（注意 `DISTINCT` 去重）。
2. **子查询**：独立执行，找出所有在Perryridge分行有贷款的客户名集合。
3. **IN 测试**：外层客户名必须出现在子查询结果中，即该客户同时有贷款。

### 优缺点
- **优点**：逻辑分层清晰：先找有账户的，再从中挑出有贷款的。子查询只执行一次，结果集可能较小。
- **缺点**：若子查询结果集很大，`IN` 可能效率较低；且需注意 `IN` 子查询中的 `NULL` 陷阱（但此处客户名通常非空）。

### 知识点
- 子查询的独立性：这里子查询是非相关的，独立于外层。
- `IN` 操作符用于集合成员测试。
- 注意表别名的正确使用和表的存在性。

## 第4页：多列IN写法

**原始SQL**：
```sql
SELECT DISTINCT customer_name
FROM borrower, loan
WHERE borrower.loan_number = loan.loan_number
  AND loan.branch_name = 'Perryridge'
  AND (loan.branch_name, borrower.customer_name) IN (
      SELECT account.branch_name, depositor.customer_name
      FROM depositor, account
      WHERE depositor.account_number = account.account_number
  );
```

### 逻辑解析
1. **外层查询**：从 `borrower` 和 `loan` 连接，找出所有在Perryridge分行有贷款的客户及其对应的支行名（注意此处 `loan.branch_name` 固定为 'Perryridge'）。
2. **子查询**：从 `depositor` 和 `account` 连接，找出所有有账户的客户及其账户所在支行名，返回 `(branch_name, customer_name)` 对。
3. **多列IN**：外层使用 `(loan.branch_name, borrower.customer_name) IN (子查询)`，要求每一对值必须出现在子查询的结果中。由于外层 `branch_name` 已经是 'Perryridge'，这实际上等价于检查该客户是否在子查询中存在一个 'Perryridge' 支行的账户记录。
4. **去重**：同样需要 `DISTINCT`，避免同一客户因多笔贷款重复出现。

### 优缺点
- **优点**：利用行值比较，一次完成两个条件的匹配，代码简洁。
- **缺点**：多列IN的语法可能不被所有数据库完全支持（但主流数据库如MySQL、PostgreSQL、Oracle都支持），且可读性稍差，需要理解行构造器。

### 知识点
- **行构造器**：`(col1, col2)` 表示一个行值，可以与子查询返回的行进行比较。
- **多列子查询**：子查询必须返回与左边列数相同的列，且类型兼容。
- 这种写法本质上也是将两边分别计算后取交集，但通过行值避免了显式的客户名关联条件（因为子查询中已经包含了支行名和客户名，外层通过支行名和客户名对匹配）。

## 三种方法的对比总结

| 方法        | 优点                         | 缺点                                   | 关键知识点                   |
| ----------- | ---------------------------- | -------------------------------------- | ---------------------------- |
| 多表连接    | 逻辑直接，关系代数直观       | 书写繁琐，易漏条件，可能产生大中间结果 | 隐式连接，笛卡尔积，DISTINCT |
| 连接+子查询 | 分层清晰，子查询独立，易理解 | 子查询结果集大时可能性能问题           | 非相关子查询，IN 操作符      |
| 多列IN      | 利用行值比较，代码简洁       | 需要数据库支持行构造器，可读性稍差     | 行构造器，多列子查询         |

### 注意事项
- 所有方法都使用了隐式连接（逗号连接），但在实际开发中，推荐使用显式的 `JOIN` 语法（如 `INNER JOIN ... ON`），以提高可读性和可维护性。
- 第三页的原始写法中存在错误，需要修正才能正确执行。
- 对于“同时拥有”这类查询，还可以使用 `INTERSECT` 操作，但需要两个子查询分别返回客户名，然后取交集。例如：
  ```sql
  (SELECT customer_name FROM borrower, loan WHERE ...)
  INTERSECT
  (SELECT customer_name FROM depositor, account WHERE ...);
  ```
  但要注意 `INTERSECT` 默认去重，不需要 `DISTINCT`。
