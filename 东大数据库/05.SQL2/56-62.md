---

## 第1页：嵌套子查询概述

**内容回顾**  
- 嵌套查询是SQL最强大的特性之一。
- 嵌套查询是指一个查询内部嵌入了另一个查询。
- 被嵌入的查询称为**子查询**，它是一个完整的 `SELECT-FROM-WHERE` 表达式。

**详细讲解**  
- **子查询的定义**：子查询就是在一个外层查询（也称为主查询）内部再写一个查询。子查询的结果可以被主查询用于条件判断、值比较、集合成员测试等。
- **为什么强大**：子查询允许你将一个查询的结果作为另一个查询的输入，从而实现多步逻辑操作，使SQL能够表达非常复杂的查询需求。
- **子查询的位置**：子查询可以出现在多个地方，最常见的是在 `WHERE` 子句中，但也可以在 `FROM` 子句（作为派生表）、`HAVING` 子句、甚至 `SELECT` 子句（作为标量子查询）中。

---

## 第2页：子查询的常见位置和用途

**内容回顾**  
- 子查询通常出现在 `WHERE` 子句中。
- 有时也出现在 `FROM` 子句或 `HAVING` 子句中。
- 常见的用途包括：集合成员测试（如 `IN`）、集合比较（如 `> ANY`）、集合基数测试（如 `EXISTS`）等。

**详细讲解**  
- **WHERE 子句中的子查询**：这是最常见的用法，子查询返回一个值或一组值，主查询用这些值进行条件过滤。
- **FROM 子句中的子查询**：此时子查询充当一个临时表（派生表），主查询从这个临时表中查询数据。必须为派生表指定别名。
- **HAVING 子句中的子查询**：子查询用于对分组后的组进行条件过滤，类似于 `WHERE` 中的子查询，但作用于组。
- **集合成员测试**：使用 `IN` 或 `NOT IN` 检查某个值是否在子查询返回的集合中。
- **集合比较**：使用 `> ANY`、`> ALL` 等将某个值与子查询返回的集合进行比较。
- **存在性测试**：使用 `EXISTS` 检查子查询是否返回任何行。

## 第3页：子查询的简单示例对比

**内容回顾**  
- 给出了两个查询：
  1. 普通查询：`SELECT * FROM loan WHERE amount > 1200;`
  2. 有语法错误的子查询：`SELECT * FROM loan WHERE amount > SELECT AVG(amount) FROM loan;`

**详细讲解**  
- **第一个查询**：简单的条件查询，找出所有金额大于1200的贷款。
- **第二个查询**：意图是找出所有金额大于平均金额的贷款。但写法是错误的，因为比较运算符 `>` 后面跟了一个子查询，而子查询返回的是单个值（平均金额），但语法上需要将子查询用括号括起来。正确的写法是：
  ```sql
  SELECT * FROM loan
  WHERE amount > (SELECT AVG(amount) FROM loan);
  ```
- **错误原因**：子查询必须用括号包围，否则数据库无法正确解析。这是书写子查询时需要特别注意的语法点。

## 第4页：子查询的通用特性和与JOIN的关系

**内容回顾**  
- 子查询可以替换一个值或一组值。
- 强调“Must be illegal value”可能是指子查询必须返回合法值（即与主查询比较的类型兼容）。
- 子查询通常可以重写为等价的 `JOIN` 查询。

**详细讲解**  
- **子查询的返回值**：
  - 标量子查询：返回单个值（一行一列），可以用在 `=`、`>` 等比较操作中。
  - 行子查询：返回一行多列，可以用在行比较中（如 `(col1, col2) = (subquery)`）。
  - 表子查询：返回多行多列，可以用在 `IN`、`EXISTS` 或 `FROM` 子句中。
- **子查询与JOIN的互换**：很多包含子查询的查询可以改写为 `JOIN` 形式，而且通常 `JOIN` 性能更好，但子查询在某些情况下更直观。例如，上面的“比平均金额大”的例子只能用子查询（因为需要全局平均值），而很多存在性测试可以用 `JOIN` 或 `EXISTS` 互换。
- **注意**：并非所有子查询都能改写为 `JOIN`，例如包含聚合函数且无 `GROUP BY` 的标量子查询。

## 第5页：不使用嵌套查询的等价写法（自连接）

**内容回顾**  
- 表：`students(sid, sname, age)`
- 需求：查询比“张晓红”年龄大的学生姓名和年龄。
- 给出的不使用嵌套查询的写法（自连接）：
  ```sql
  SELECT s1.sname, s1.age
  FROM students s1, students s2
  WHERE s2.sname = '张晓红' AND s1.age > s2.age;
  ```

**详细讲解**  
- **自连接**：将同一张表用不同的别名视为两个不同的实例，然后进行连接。这里 `s2` 代表“张晓红”所在的行，`s1` 代表其他学生，通过 `s1.age > s2.age` 找出年龄更大的学生。
- **执行逻辑**：
  1. 从 `students` 表中选择 `sname = '张晓红'` 的行，得到其年龄。
  2. 将该行与表中的其他行（包括自己，但会被条件排除）进行笛卡尔积，然后筛选出 `s1.age > s2.age` 的行。
  3. 输出符合条件的 `s1.sname` 和 `s1.age`。
- **优点**：不使用子查询，在某些数据库优化器下可能性能更好。
- **缺点**：写法相对复杂，尤其当需要多步逻辑时，可读性不如子查询。

## 第6页：使用嵌套查询的等价写法

**内容回顾**  
- 同样的需求，使用子查询：
  ```sql
  SELECT sname, age
  FROM students
  WHERE age > (SELECT age FROM students WHERE sname = '张晓红');
  ```

**详细讲解**  
- **子查询部分**：`(SELECT age FROM students WHERE sname = '张晓红')` 是一个标量子查询，它返回张晓红的年龄（假设姓名唯一，否则可能出错）。
- **主查询**：对于 `students` 表中的每一行，将其年龄与子查询返回的值比较，如果大于则输出。
- **执行逻辑**：通常数据库会先执行子查询，得到张晓红的年龄（假设为20），然后将主查询变为 `WHERE age > 20` 来执行。这种方式逻辑清晰，易于理解。
- **注意**：如果 `sname` 不唯一，子查询可能返回多行，导致错误。此时应确保子查询返回单值，或使用 `IN` 等集合比较。

## 第7页：返回多列的子查询

**内容回顾**  
- 表：`Teacher(id, name, age, title)`
- 需求：查询和王强老师相同年龄和职称的教师姓名。
- SQL语句：
  ```sql
  SELECT name
  FROM teacher
  WHERE (age, title) = (SELECT age, title FROM teacher WHERE name = '王强');
  ```

**详细讲解**  
- **多列子查询**：子查询可以返回多列，主查询使用行构造器 `(col1, col2)` 与子查询的结果进行比较。这里 `(age, title)` 是一个行值，与子查询返回的一行（两列）进行比较。
- **行比较**：`(age, title) = (subquery)` 要求子查询返回且仅返回一行，且列数匹配。它会逐列比较，只有当所有列都相等时，行才相等。
- **执行逻辑**：
  1. 子查询找出王强老师的年龄和职称（假设返回 (35, '教授')）。
  2. 主查询遍历 `teacher` 表，对于每一行，检查其 `(age, title)` 是否等于 (35, '教授')。
  3. 返回所有满足条件的教师姓名（包括王强自己，如果不想包括，可以加 `AND name != '王强'`）。
- **注意事项**：
  - 子查询必须保证返回且仅返回一行，否则会出错。如果王强老师有多条记录（如姓名重复），则需要处理。
  - 行比较的语法在主流数据库中支持（如 PostgreSQL、MySQL、Oracle），但某些数据库可能不支持，需用 `AND` 连接多个条件替代。

## 补充知识：子查询的分类和注意事项

### 1. 根据返回结果分类
- **标量子查询**：返回单行单列，可用在 `=`、`>` 等比较中。
- **行子查询**：返回单行多列，可用在行构造器比较中。
- **表子查询**：返回多行多列，可用在 `IN`、`EXISTS`、`FROM` 子句中。

### 2. 相关子查询与非相关子查询
- **非相关子查询**：子查询独立于主查询，可单独执行，如上面的比年龄例子。
- **相关子查询**：子查询引用了主查询的列，需要为每一行执行一次，如 `EXISTS` 或 `IN` 配合外层列的情况。性能通常较差。

### 3. 使用 `IN` 与 `EXISTS`
- `IN` 适合子查询结果集较小的情况，`EXISTS` 适合外层表较大且子查询可快速判断的情况，但现代优化器 often 能做出合理选择。

### 4. 子查询与 NULL
- 如果子查询返回 `NULL`，使用 `=` 或 `>` 比较会得到 `UNKNOWN`，导致行被过滤。使用 `IN` 时，如果集合中包含 `NULL`，`IN` 的行为需要谨慎（可能不会出错，但逻辑需清晰）。`NOT IN` 遇到 `NULL` 会返回空结果，这是常见陷阱。

### 5. 性能建议
- 尽量将子查询改写为 `JOIN`，尤其是当子查询是非相关且返回大量数据时。
- 为子查询中涉及的列建立索引。
- 使用 `EXISTS` 代替 `IN` 有时能提升性能，但需测试。
