---

## 1. 字符串匹配操作简介

**内容回顾**  
- SQL 提供了一个用于字符串比较的运算符：`LIKE`。

**详细讲解**  
- `LIKE` 是 SQL 中实现**模式匹配**的关键字，它允许我们使用通配符来描述要查找的字符串模式，而不是进行精确的等值比较。这在需要模糊查询时非常有用，例如查找所有姓“张”的人、包含特定关键词的文本等。
- `LIKE` 通常用在 `WHERE` 子句中，语法为：
  ```sql
  WHERE 列名 LIKE '模式'
  ```
  其中模式是一个字符串，可以包含通配符。

---

## 2. 固定字符串匹配：LIKE 与 = 的等价性

**内容回顾**  
- 当模式串中不包含任何通配符时，`LIKE` 的行为与 `=` 完全相同。
- 示例：查询学号为 `200215121` 的学生：
  ```sql
  SELECT * FROM Student WHERE Sno LIKE '200215121';
  -- 等价于
  SELECT * FROM Student WHERE Sno = '200215121';
  ```

**详细讲解**  
- **为什么需要这样？** 虽然 `=` 就可以完成精确匹配，但有时为了保持代码的一致性（例如在动态生成的查询中统一使用 `LIKE`），或者某些数据库对 `LIKE` 的优化不同，可能会出现这种用法。但从可读性和性能角度，精确匹配推荐使用 `=`。
- **注意**：`LIKE` 的字符串比较可能受数据库的排序规则（collation）影响，例如大小写敏感性问题，而 `=` 同样受排序规则影响。因此两者在比较行为上是一致的。

## 3. 通配符 %（百分号）

**内容回顾**  
- `%` 代表**任意长度**（包括长度为0）的字符串。
- 例如：`a%b` 匹配以 `a` 开头、以 `b` 结尾的任意字符串，如 `acb`、`addgb`、`ab` 等。

**详细讲解**  
- `%` 是最常用的通配符，它可以匹配零个或多个字符。注意它匹配的是**任意字符**，包括空格、标点符号等。
- 常见模式示例：
  - `'刘%'`：所有以“刘”开头的字符串（“刘晨”、“刘洋”、“刘”等）。
  - `'%科技%'`：所有包含“科技”的字符串（“科学技术”、“高科技”、“科技”等）。
  - `'%有限公司'`：所有以“有限公司”结尾的字符串。
- **性能注意事项**：
  - 如果模式以 `%` 开头（如 `'%科技'`），则数据库无法使用该列上的索引进行快速查找，通常会进行全表扫描。这是因为索引是按列值从左到右排序的，当左边不确定时，无法定位。
  - 如果模式以固定字符串开头（如 `'刘%'`），则有可能利用索引进行范围扫描（取决于数据库优化器）。

## 4. 通配符 _（下划线）

虽然这一页在提供的文本中没有单独出现，但结合前面讨论过的内容以及后面的示例，我们知道 `_` 代表**任意单个字符**。

**详细讲解**  
- `_` 严格匹配一个字符，不能多也不能少。例如 `'a_b'` 匹配长度为3且以 `a` 开头、以 `b` 结尾的字符串，如 `acb`、`afb`，但不匹配 `ab`（太短）或 `acdb`（太长）。
- 常见模式示例：
  - `'张_'`：姓“张”且名字只有一个字的人（如“张飞”，但“张翼德”不符合）。
  - `'__'`：任意两个字符的字符串。
  - `'1_3'`：第一位是1，第三位是3，中间任意一个字符的三位字符串（如 `123`、`1a3`）。
- 中文环境下，一个汉字通常算一个字符，所以 `'欧阳__'` 表示以“欧阳”开头，后跟两个汉字的名字（如“欧阳明明”），总长度为4个汉字。

## 5. 字符匹配示例（一）

**内容回顾**  
- 查询所有姓刘学生的姓名、学号和性别：
  ```sql
  SELECT Sname, Sno, Ssex FROM Student WHERE Sname LIKE '刘%';
  ```
- 查询姓“欧阳”且全名为三个汉字的学生的姓名：
  ```sql
  SELECT Sname FROM Student WHERE Sname LIKE '欧阳_';
  ```

**详细讲解**  
- **姓刘的查询**：`'刘%'` 匹配所有以“刘”开头的名字，无论后面有多少个字。这是最常见的用法。
- **姓欧阳且全名三个汉字**：在SQL中，中文字符同样适用通配符。一个中文字符通常占一个字符位置（取决于数据库的字符集，如UTF-8中一个中文字符可能占多个字节，但在模式匹配时，`_` 通常匹配一个字符，而非一个字节）。因此 `'欧阳_'` 表示以“欧阳”开头，后面紧跟任意字符，总长度为3个字符（两个字“欧阳”加上一个字的名字）。
- 更安全的做法是使用长度函数结合条件，但简单查询下，`LIKE` 是最直观的。

## 6. 字符匹配示例（二）

**内容回顾**  
- 查询名字中第2个字为“阳”字的学生的姓名和学号：
  ```sql
  SELECT Sname, Sno FROM Student WHERE Sname LIKE '_阳%';
  ```
- 查询所有不姓刘的学生姓名：
  ```sql
  SELECT Sname, Sno, Sex FROM Student WHERE Sname NOT LIKE '刘%';
  ```

**详细讲解**  
- **第二个字为“阳”**：`'_阳%'` 中，第一个下划线匹配姓（任意一个汉字），接着是“阳”（第二个字），然后 `%` 匹配剩余任意字符（包括空）。这样就能找到例如“欧阳明”、“王阳阳”等名字中第二个字是“阳”的人。
- **不姓刘**：`NOT LIKE '刘%'` 找出所有不以“刘”开头的名字。但要注意，如果 `Sname` 列包含 `NULL`，那么 `NOT LIKE` 的结果是 `UNKNOWN`，该行不会被返回。因此如果需要包括所有非刘姓且非NULL的学生，这个查询是正确的。如果想包括 `NULL` 作为“不姓刘”的一部分，则需要额外处理。

## 7. 转义通配符：ESCAPE

**内容回顾**  
- 当需要匹配字面意义的 `%` 或 `_` 时，使用转义字符。
- 示例1：查询课程名为 `DB_Design` 的课程号和学分：
  ```sql
  SELECT Cno, Credit FROM Course WHERE Cname LIKE 'DB\_Design' ESCAPE '\';
  ```
- 示例2：查询以 `DB_` 开头，且倒数第3个字符为 `i` 的课程的详细情况：
  ```sql
  SELECT * FROM Course WHERE Cname LIKE 'DB\_%i__' ESCAPE '\';
  ```
- **`ESCAPE '\'`** 指定反斜杠为转义字符。

**详细讲解**  
- **为什么需要转义**：因为 `%` 和 `_` 在 `LIKE` 模式中有特殊含义，如果要匹配它们本身，就必须告诉数据库将它们视为普通字符。例如课程名 `DB_Design` 中包含下划线，如果直接写 `LIKE 'DB_Design'`，下划线会被解释为通配符，匹配 `DB` 后面任意一个字符，如 `DBADesign`、`DBBDesign` 等，这显然不是我们想要的。
- **转义方法**：在需要转义的通配符前加上转义字符，并用 `ESCAPE` 子句指定该字符为转义符。通常使用反斜杠 `\`，但也可以使用其他字符（如 `#`），只要用 `ESCAPE` 声明即可。
- **示例1解析**：`'DB\_Design'` 中的 `\_` 表示一个字面下划线，因此该模式精确匹配 `DB_Design`。
- **示例2解析**：模式 `'DB\_%i\__'` 的含义需要仔细分析：
  - `DB\_`：匹配字面字符串 `DB_`。
  - `%`：匹配任意长度的字符串（包括空）。
  - `i`：匹配字母 `i`。
- **转义字符本身**：如果需要匹配转义字符本身（如反斜杠），则需连续写两个转义字符。例如 `'\\'` 匹配一个字面反斜杠。
- **ESCAPE 是否必须？** 不，只有当模式中包含需要转义的通配符时才需要。如果模式中没有 `%` 或 `_` 的字面需求，就不必写 `ESCAPE`。但为了代码清晰和可移植性，建议在需要转义时显式指定，而不是依赖数据库的默认转义行为（有些数据库可能没有默认转义符）。

## 8. NULL 值处理

**内容回顾**  
- 判断空值必须使用 `IS NULL` 或 `IS NOT NULL`，不能使用 `=` 或 `<>`。
- 示例：查询缺少成绩的学生的学号和课程号：
  ```sql
  SELECT Sno, Cno FROM SC WHERE Grade IS NULL;
  ```
- 查询有成绩的学生学号和课程号：
  ```sql
  SELECT Sno, Cno FROM SC WHERE Grade IS NOT NULL;
  ```

**详细讲解**  
- **为什么不能用 `=` 判断 NULL？** 因为 SQL 使用三值逻辑：`TRUE`、`FALSE` 和 `UNKNOWN`。`NULL` 表示“未知”值，任何与 `NULL` 的直接比较（如 `Grade = NULL`）结果都是 `UNKNOWN`，而 `WHERE` 子句只接受 `TRUE` 的行，所以这样的条件永远不会返回任何行。同样，`Grade <> NULL` 也是 `UNKNOWN`。
- **正确的判断**：必须使用 `IS NULL` 或 `IS NOT NULL`，它们专门用于检测空值，返回 `TRUE` 或 `FALSE`。
- **示例场景**：在选课表（SC）中，如果学生选修了课程但没有参加考试，则成绩（Grade）列为 `NULL`。要找出这些学生，就需要用 `IS NULL`。
- **注意聚合函数对 NULL 的处理**：例如 `COUNT(Grade)` 会忽略 NULL 值，只计数非空的行；而 `COUNT(*)` 会计数所有行，包括 NULL 行。这在使用聚合时需特别注意。
- **NULL 在逻辑运算中**：当 `AND`、`OR`、`NOT` 与 `NULL` 结合时，结果遵循三值逻辑表。例如 `TRUE AND NULL` 结果为 `UNKNOWN`，`FALSE OR NULL` 为 `UNKNOWN`，`NOT NULL` 为 `UNKNOWN`。理解这一点有助于调试复杂查询。

## 总结与扩展

通过这几页PPT，我们学习了：

- **字符串匹配（LIKE）**：
  - 通配符 `%` 和 `_` 的用法。
  - 如何结合它们进行各种模糊查询。
  - 使用 `ESCAPE` 转义通配符，匹配字面特殊字符。
  - 注意性能问题：避免前导通配符，尽量利用索引。
- **空值处理（IS NULL / IS NOT NULL）**：
  - 不能使用 `=` 或 `<>` 判断 NULL。
  - 理解三值逻辑及其对查询结果的影响。
  - 在实际应用中，经常需要处理 NULL，例如统计缺失数据、排除空值等。

这些知识点是 SQL 查询的基础，掌握它们后，你可以更灵活地构建条件，满足各种业务需求。在后续的学习中，还会接触到更复杂的查询（如连接、子查询、聚合等），但 `WHERE` 子句始终是数据过滤的核心。
