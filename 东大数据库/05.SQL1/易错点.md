## 一、数据类型相关易错点

### 1. **`FLOAT` 的精度含义**
- **易错**：误以为 `FLOAT(p)` 中的 `p` 是十进制小数位数，或认为它能精确存储小数。
- **正解**：`p` 是**二进制精度**（尾数位数），决定了十进制有效数字的位数（约 `p * 0.3010`）。浮点数是**近似存储**，不能用于精确计算（如货币）。
- **示例**：`FLOAT(24)` 对应单精度，约7位十进制有效数字；`1234567.89` 可能存为 `1234567.875`。

### 2. **定点数与浮点数的选择**
- **易错**：用 `FLOAT` 存储金额、数量等需要精确等值比较的数据。
- **正解**：精确数值用 `DECIMAL`（如 `DECIMAL(10,2)`），浮点数用于科学计算等可接受误差的场景。

### 3. **字符类型长度与空格**
- **易错**：用 `CHAR(n)` 存变长数据，导致末尾空格影响比较（如 `'abc'` 与 `'abc '` 在 `CHAR` 中视为相等，但在 `VARCHAR` 中不等）。
- **正解**：变长字符串用 `VARCHAR(n)`，定长字符串（如固定编码）用 `CHAR(n)`。

### 4. **二进制类型的使用**
- **易错**：误将 `BLOB` 当作字符串处理，导致字符集转换或数据损坏。
- **正解**：`BLOB` 存原始字节，如图片、文件；插入/查询需用十六进制或参数化接口，避免字符集干扰。

### 5. **日期时间类型的时区处理**
- **易错**：混淆 `TIMESTAMP` 与 `TIMESTAMP WITH TIME ZONE`，导致时区转换错误。
- **正解**：
  - `TIMESTAMP`：无时区，存“墙上时间”。
  - `TIMESTAMP WITH TIME ZONE`：内部存 UTC，显示时根据会话时区转换。比较时基于 UTC。

## 二、完整性约束相关易错点

### 1. **主键与唯一约束对 NULL 的处理**
- **易错**：认为 `UNIQUE` 约束不允许 `NULL`，或不清楚不同数据库对多行 `NULL` 的处理差异。
- **正解**：
  - **主键**：不允许 `NULL`（实体完整性）。
  - **唯一约束**：标准允许 `NULL` 且认为 `NULL` 互不相等，但数据库实现有差异：
    - PostgreSQL/MySQL：允许多个 `NULL`。
    - SQL Server：只允许一个 `NULL`。
  - **最佳实践**：若需真正唯一且非空，用 `UNIQUE NOT NULL`。

### 2. **候选键与 NULL 的理论 vs 实践**
- **易错**：误以为候选键在 SQL 中自动禁止 `NULL`。
- **正解**：理论要求候选键非空，但 SQL 中通过 `UNIQUE` 定义的候选键允许 `NULL`（除非加 `NOT NULL`）。需根据业务决定是否加非空。

### 3. **`CHECK` 约束的行为误区**
- **易错**：
  - 认为 `CHECK` 一定会拒绝 `NULL`。
  - 在 MySQL 中误以为 `CHECK` 生效（旧版本忽略）。
  - 条件写错（如 `CHECK (age > 15 AND age < 50)` 误写成 `age > 15 OR age < 50`）。
- **正解**：
  - `CHECK` 只拒绝结果为 `FALSE` 的行，`UNKNOWN`（含 `NULL`）的行通过。
  - MySQL 8.0.16 起 InnoDB 支持 `CHECK`，之前版本解析但忽略。
  - 测试 `CHECK` 是否真正生效，可插入违规数据验证。

### 4. **外键约束的级联选项误用**
- **易错**：未指定 `ON DELETE/UPDATE` 选项，或选错导致数据不一致或误删。
- **正解**：
  - **CASCADE**：适用于“子随父走”（如订单明细）。
  - **SET NULL/SET DEFAULT**：适用于子记录仍需保留但解绑。
  - **NO ACTION/RESTRICT**：禁止删除被引用的父记录。
  - 必须明确业务逻辑，避免意外级联删除。

### 5. **复合主键与外键的 NULL 问题**
- **易错**：认为复合主键中某列可以为 `NULL`（如 `(sid, cid)` 中 `sid` 为 `NULL`）。
- **正解**：主键所有列必须 `NOT NULL`，否则无法唯一标识行。

## 三、表操作与模式修改相关易错点

### 1. **`DROP TABLE` 与 `DELETE` 混淆**
- **易错**：误用 `DROP` 想清空数据，结果删除了整个表。
- **正解**：
  - `DELETE FROM table;` 删数据，留结构。
  - `DROP TABLE table;` 删表结构和数据。

### 2. **`ALTER TABLE ADD COLUMN` 时默认值为 `NULL`**
- **易错**：添加列后忘记处理现有行的 `NULL`，导致应用逻辑报错。
- **正解**：若需非空，可先添加允许 `NULL`，填充数据后再改 `NOT NULL`；或直接加 `DEFAULT` 值并指定 `NOT NULL`（部分数据库支持）。

### 3. **修改数据类型时的数据转换风险**
- **易错**：直接修改类型，导致数据截断或转换失败。
- **正解**：
  - 先检查现有数据是否都能转换为新类型。
  - 备份数据。
  - 使用 `USING` 子句（如 PostgreSQL）自定义转换逻辑。

### 4. **添加约束前未验证现有数据**
- **易错**：添加 `UNIQUE` 或 `FOREIGN KEY` 时因现有数据不满足约束而失败。
- **正解**：先查询确认数据质量：
  ```sql
  -- 检查重复
  SELECT col, COUNT(*) FROM table GROUP BY col HAVING COUNT(*) > 1;
  -- 检查外键引用是否存在
  SELECT DISTINCT fk_col FROM child WHERE fk_col NOT IN (SELECT pk_col FROM parent);
  ```

### 5. **约束命名的重要性**
- **易错**：不命名约束，导致后续管理困难（如无法直观知道错误来自哪个约束）。
- **正解**：始终使用 `CONSTRAINT 约束名` 为约束命名，便于维护。

## 四、参照完整性执行策略易错点

### 1. **更新主键时的处理**
- **易错**：只考虑删除的级联，忽略更新主键时子表数据的同步。
- **正解**：用 `ON UPDATE CASCADE` 确保子表外键随父表主键更新。

### 2. **`SET NULL` 与主键列的冲突**
- **易错**：在外键是主键一部分时使用 `SET NULL`，违反主键非空约束。
- **正解**：若外键列是主键的一部分，只能用 `CASCADE` 或 `NO ACTION`，不能 `SET NULL`。

### 3. **不同数据库的默认行为差异**
- **易错**：以为所有数据库默认都是 `NO ACTION`，但某些数据库可能默认有其他行为。
- **正解**：显式指定所需选项，不依赖默认值。

---

## 五、SQL标准与实现差异

### 1. **`UNIQUE` 约束中 NULL 的处理**
- 如上所述，不同数据库差异大，务必查阅文档。

### 2. **`CHECK` 约束的支持度**
- MySQL 8.0.16 前不支持，SQLite 支持但有限，PostgreSQL/Oracle 完全支持。

### 3. **`ALTER TABLE` 语法差异**
- 修改列类型、重命名列等操作在不同数据库语法不同，不可移植。

### 4. **标识符大小写敏感**
- MySQL 在 Windows 上不敏感，Linux 上敏感；PostgreSQL 默认转小写；Oracle 默认大写。跨数据库迁移时需注意。

---

## 总结：避免易错点的最佳实践

1. **明确业务需求**：选择数据类型和约束前，先想清楚是否允许 `NULL`、是否需精确、唯一性范围等。
2. **查阅文档**：不同数据库对标准 SQL 的实现有差异，尤其是 `CHECK`、`UNIQUE NULL` 处理、`ALTER` 语法。
3. **先验证后执行**：添加约束或修改结构前，先检查现有数据是否满足新规则。
4. **命名约束**：为所有约束起有意义的名字，便于后期维护。
5. **备份**：任何 DDL 操作（尤其是 `DROP`、`ALTER`）前，备份数据。
6. **测试环境验证**：先在测试环境模拟操作，确认无误后再在生产环境执行。
