## 一、布尔类型（Boolean Type）

### 1. 概念
布尔类型用于存储逻辑真值，即 **`TRUE`** 或 **`FALSE`**。它也可以接受 `UNKNOWN`（即 `NULL`）表示不确定的状态。

### 2. 标准 SQL 语法
```sql
CREATE TABLE test (
    id DECIMAL PRIMARY KEY,
    col1 BOOLEAN
);
```
- `id` 列定义为 `DECIMAL` 类型（通常用于主键，但更常见的做法是使用整数类型）。
- `col1` 列定义为 `BOOLEAN` 类型，可存储 `TRUE` 或 `FALSE`。

### 3. 数据库实现差异
- **PostgreSQL**：原生支持 `BOOLEAN`，可直接使用 `TRUE`/`FALSE` 字面量。
- **MySQL**：支持 `BOOLEAN` 关键字，但实际上是 `TINYINT(1)` 的别名，存储时 0 表示 `FALSE`，非 0 表示 `TRUE`。
- **SQL Server**：没有直接的 `BOOLEAN` 类型，通常使用 `BIT` 类型（0/1）来模拟。
- **Oracle**：也没有 `BOOLEAN` 类型（PL/SQL 中有，但 SQL 层面没有），通常用 `NUMBER(1)` 或 `CHAR(1)` 加约束实现。

### 4. 使用建议
- 尽量使用标准布尔类型（如果数据库支持），否则选择最接近的模拟类型。
- 插入数据时可直接使用 `TRUE`/`FALSE` 或对应的数值（视数据库而定）。

## 二、二进制类型（Binary Type）

### 1. 概念
二进制类型用于存储**二进制数据**，如图片、音频、视频、压缩文件等。它们存储的是字节序列，不涉及字符集转换。

### 2. 主要类型
| 类型               | 说明                                                         |
| ------------------ | ------------------------------------------------------------ |
| **`BINARY(n)`**    | 固定长度为 `n` 字节的二进制数据。如果实际数据不足 `n` 字节，右侧用 `0x00` 填充（通常）。 |
| **`VARBINARY(n)`** | 可变长度二进制数据，最大长度为 `n` 字节，实际存储只占用实际数据长度（加额外字节记录长度）。 |
| **`BLOB`**         | 二进制大对象，用于存储超长二进制数据（如图片、音视频文件）。具体最大长度由 DBMS 定义（如 MySQL 的 `TINYBLOB`、`BLOB`、`MEDIUMBLOB`、`LONGBLOB`）。 |

### 3. 示例
```sql
CREATE TABLE test (
    col1 BINARY(8),      -- 固定8字节
    col2 VARBINARY(140), -- 最多140字节
    col3 BLOB            -- 大对象
);
```

### 4. 注意事项
- 二进制类型存储的是原始字节，因此不会进行字符集转换或空格处理（除 `BINARY` 填充外）。
- 检索和插入时通常需要处理为十六进制字符串或使用参数化查询（如 JDBC 的 `setBytes`）。
- 不同数据库的二进制类型名称略有差异：例如 SQL Server 中对应 `BINARY`、`VARBINARY`、`IMAGE`（已过时）；PostgreSQL 中对应 `BYTEA`。
- 对于非常大的文件，通常建议将文件存储在文件系统中，数据库中只存储文件路径，而不是直接存 `BLOB`，以减轻数据库负担。

## 三、日期时间类型（Datetime Types）

### 1. 概述
SQL 提供多种日期时间类型，用于存储时刻、日期、时间或带时区的时间。不同的类型有不同的存储需求和精度。

### 2. 标准 SQL 类型及格式

| 类型                           | 描述                               | 标准格式示例                |
| ------------------------------ | ---------------------------------- | --------------------------- |
| **`DATE`**                     | 仅存储日期（年-月-日）             | `2026-03-12`                |
| **`TIME`**                     | 仅存储时间（时:分:秒），可带小数秒 | `14:30:00.123`              |
| **`TIME WITH TIME ZONE`**      | 时间带时区信息                     | `14:30:00+08:00`            |
| **`TIMESTAMP`**                | 日期和时间组合，不带时区           | `2026-03-12 14:30:00.123`   |
| **`TIMESTAMP WITH TIME ZONE`** | 日期和时间组合，带时区             | `2026-03-12 14:30:00+08:00` |

### 3. 示例
```sql
CREATE TABLE test (
    col1 DATE,                        -- 日期
    col2 TIME,                         -- 时间
    col3 TIMESTAMP(9),                 -- 时间戳，小数秒精度为9位（纳秒级）
    col4 TIMESTAMP WITH TIME ZONE      -- 带时区的时间戳
);
```
- `TIMESTAMP(9)` 中的 `(9)` 表示小数秒部分保留 9 位数字（最大精度通常为 9，即纳秒）。不同数据库支持的精度范围不同。
- `WITH TIME ZONE` 类型在存储时会包含时区信息，通常以 UTC 存储，并在检索时转换为会话时区显示。

### 4. 时区处理
- **不带时区的类型**：存储的是“墙上时钟时间”，不包含时区信息。应用需要自己处理时区转换。
- **带时区的类型**：数据库内部通常存储为 UTC，并记录时区偏移。检索时可根据会话时区自动转换。
- 使用时需注意：`TIMESTAMP WITH TIME ZONE` 在比较时是基于 UTC 进行的，而 `TIMESTAMP` 是基于本地时间直接比较。

### 5. 常见数据库的日期时间类型
- **MySQL**：`DATE`、`TIME`、`DATETIME`（无时区）、`TIMESTAMP`（有时区，但实际存储为 UTC，受时区设置影响）。
- **PostgreSQL**：`DATE`、`TIME`、`TIMESTAMP`、`TIMESTAMPTZ`（带时区）。
- **SQL Server**：`DATE`、`TIME`、`DATETIME`、`DATETIME2`（更高精度）、`DATETIMEOFFSET`（带时区）。
- **Oracle**：`DATE`（包含日期和时间）、`TIMESTAMP`、`TIMESTAMP WITH TIME ZONE`、`TIMESTAMP WITH LOCAL TIME ZONE`。

## 四、日期时间类型参数表

PPT 最后给出了一张表格，列出了几种日期时间类型的格式、取值范围、精度和存储尺寸。这些类型多见于 **SQL Server**，但也可作为其他数据库的参考。

| 数据类型            | 格式                    | 取值范围                           | 精度                    | 存储尺寸 |
| ------------------- | ----------------------- | ---------------------------------- | ----------------------- | -------- |
| **`date`**          | yyyy-mm-dd              | 0001-01-01 到 9999-12-31           | 1天                     | 3字节    |
| **`time`**          | hh:mm:ss.nnnnnn         | 00:00:00.000000 到 23:59:59.999999 | 100纳秒                 | 3-5字节  |
| **`smalldatetime`** | yyyy-mm-dd hh:mm:ss     | 1900-01-01 到 2079-06-06           | 1分钟                   | 4字节    |
| **`datetime`**      | yyyy-mm-dd hh:mm:ss.nnn | 1753-01-01 到 9999-12-31           | 0.00333秒（约3.33毫秒） | 8字节    |

### 详细说明：
- **`date`**：仅日期，占用3字节，范围很广，适用于大多数业务场景。
- **`time`**：仅时间，精度可达100纳秒（6位小数秒），存储空间取决于精度设置（例如 SQL Server 中 `time` 的存储大小根据精度变化）。
- **`smalldatetime`**：SQL Server 特有，范围较小，精度为分钟，适合记录无需秒级精度的数据，占用空间小。
- **`datetime`**：SQL Server 传统类型，精度约3.33毫秒，范围从1753年开始（因为英国及殖民地1752年更改历法），占用8字节。现代开发更推荐使用 `datetime2`，精度更高且范围更广。

### 选择建议
- 如果只需要日期，使用 `DATE`。
- 如果需要时间但不需要日期，使用 `TIME`。
- 如果需要精确到秒或毫秒的日期时间，且范围要求广，使用 `TIMESTAMP`（或 `DATETIME2`）。
- 如果涉及多个时区，使用带时区的类型（如 `TIMESTAMP WITH TIME ZONE`）。
- 考虑存储空间和精度需求，选择最合适的类型。

## 总结

这几页 PPT 涵盖了 SQL 中重要的非数值数据类型：
1. **布尔类型**：存储逻辑真值，注意数据库间的实现差异。
2. **二进制类型**：存储图片、文件等原始字节，分为固定长度、可变长度和大对象。
3. **日期时间类型**：包括日期、时间、时间戳以及带时区版本，每种类型有特定的格式、精度和存储需求。

在设计数据库表时，合理选择数据类型不仅可以准确表达业务含义，还能优化存储空间和查询性能。务必参考所用数据库的官方文档，确认具体的类型名称、行为和支持的特性。
