这几页PPT主要讲解了关系数据库中的**参照完整性（Referential Integrity）**，包括其概念、相关术语以及如何在SQL的`CREATE TABLE`语句中定义主键、候选键和外键。下面我们将详细解析每一页的内容，并补充必要的知识点和注意事项。

---

## 第一页：参照完整性概述

**内容原文：**
> - Ensures that a value that appears in one relation for a given set of attributes also appears for a certain set of attributes in another relation
> - Primary and candidate keys and foreign keys can be specified as part of the SQL **create table** statement

### 1. 什么是参照完整性？
参照完整性是一种数据库约束，用于保证表与表之间数据的一致性。它规定：**如果一个表（称为“从表”或“引用表”）的某个列（或列组合）的值引用了另一个表（称为“主表”或“被引用表”）的某个列（或列组合）的值，那么该值必须确实存在于主表的相应列中**（或者为`NULL`，如果允许的话）。

简单来说，就是**不能引用不存在的数据**。例如，在学生选课表（SC）中，学号（Sno）必须存在于学生表（Student）中；课程号（Cno）必须存在于课程表（Course）中。

### 2. 涉及的关键对象
- **主键（Primary Key）**：唯一标识表中每一行的列或列组合，且值不能为`NULL`。
- **候选键（Candidate Key）**：与主键性质相同（唯一且非空），但一个表可以有多个候选键，其中被选为主键的称为主键，其余称为候选键（通常用`UNIQUE`约束定义）。
- **外键（Foreign Key）**：用于建立和加强两个表之间链接的列或列组合，它引用另一个表的主键或候选键。

### 3. 在`CREATE TABLE`中指定
SQL标准允许在创建表时同时定义这些键，通过`PRIMARY KEY`、`UNIQUE`和`FOREIGN KEY`子句实现。这保证了数据库系统能够自动维护参照完整性。

## 第二页：参照完整性子句详解

**内容原文：**
> - The **primary key** clause lists attributes that comprise the primary key.
> - The **unique** clause lists attributes that comprise a candidate key.
> - The **foreign key** clause lists the attributes that comprise the foreign key and the name of the relation referenced by the foreign key.
>   – By default, a foreign key references the primary key attributes of the referenced table.

### 1. `PRIMARY KEY` 子句
- 用于声明表的主键。可以是单个列或多个列的组合。
- 主键列隐含`NOT NULL`约束，且值必须唯一。
- 一个表只能有一个主键。
- 语法示例：
  ```sql
  PRIMARY KEY (Sno)                 -- 单列主键
  PRIMARY KEY (Sno, Cno)             -- 复合主键
  ```

### 2. `UNIQUE` 子句
- 用于声明候选键（即唯一键），保证列或列组合的值唯一。
- 与主键不同，`UNIQUE`约束允许`NULL`（通常最多一个`NULL`，具体依赖数据库实现）。
- 一个表可以有多个`UNIQUE`约束。
- 语法示例：
  ```sql
  UNIQUE (Sname)                     -- 单列唯一
  UNIQUE (Sno, Cno)                   -- 组合唯一
  ```

### 3. `FOREIGN KEY` 子句
- 用于声明外键，指定当前表中的哪些列引用另一个表中的列。
- 必须指定：
  - 当前表中的外键列（可多个）。
  - 被引用的表名。
  - 被引用的列（如果不指定，默认引用被引用表的主键）。
- 语法示例：
  ```sql
  FOREIGN KEY (Sno) REFERENCES Student(Sno)
  FOREIGN KEY (Cno) REFERENCES Course(Cno)
  ```
- **默认行为**：如果省略被引用的列名，则默认引用被引用表的**主键**列。因此，通常建议显式写出被引用列，以提高可读性。

## 第三页：示例——学生表和课程表

**内容原文：**
```sql
CREATE TABLE Student (
    Sno CHAR(9),
    Sname CHAR(20) UNIQUE,
    Ssex CHAR(2),
    Sage SMALLINT,
    Sdept CHAR(20),
    PRIMARY KEY(Sno)
);

CREATE TABLE Course (
    Cno CHAR(4) PRIMARY KEY,
    Cname CHAR(40),
    Ccredit SMALLINT
);
```

### 解析
- **Student表**：
  - `Sno`：学号，定义为主键（隐含非空且唯一）。
  - `Sname`：姓名，定义为`UNIQUE`，即姓名不能重复（但允许`NULL`，尽管实际中姓名通常不应为空，这里未加`NOT NULL`）。
  - 其他列：性别、年龄、系别，无特殊约束。
- **Course表**：
  - `Cno`：课程号，直接定义为`PRIMARY KEY`（列级主键）。
  - 课程名和学分。

这两个表是后续选课表（SC）的父表，为参照完整性提供了基础。

## 第四页：示例——选课表（SC）

**内容原文：**
```sql
CREATE TABLE SC (
    Sno CHAR(9),
    Cno CHAR(4),
    Grade SMALLINT,
    PRIMARY KEY (Sno, Cno),
    FOREIGN KEY (Sno) REFERENCES Student(Sno),
    FOREIGN KEY (Cno) REFERENCES Course(Cno)
);
```

### 解析
- **SC表**：记录学生选课及成绩。
- **主键**：`(Sno, Cno)`组合主键，确保同一个学生同一门课只能有一条成绩记录。
- **外键约束**：
  - `FOREIGN KEY (Sno) REFERENCES Student(Sno)`：保证`Sno`的值必须存在于`Student`表的`Sno`列中（即学生必须先存在）。
  - `FOREIGN KEY (Cno) REFERENCES Course(Cno)`：保证`Cno`的值必须存在于`Course`表的`Cno`列中（即课程必须先存在）。
- 这两个外键共同维护了选课数据的参照完整性：不能为不存在的学生或课程添加选课记录。

### 关于`Grade`
- `Grade`列没有约束，但可以添加`CHECK (Grade BETWEEN 0 AND 100)`来保证成绩范围。

## 补充知识点：参照完整性的扩展与注意事项

### 1. 外键约束的级联操作
当被引用表中的行被删除或更新时，可以定义外键的响应行为。常用的选项有：
- **`ON DELETE CASCADE`**：删除被引用行时，自动删除引用它的所有行。
- **`ON DELETE SET NULL`**：删除被引用行时，将外键列设为`NULL`（要求外键列允许`NULL`）。
- **`ON DELETE RESTRICT`**（或`NO ACTION`）：禁止删除被引用行，如果存在依赖记录。
- **`ON UPDATE CASCADE`**：更新被引用主键时，自动更新外键值。
例如：
```sql
FOREIGN KEY (Sno) REFERENCES Student(Sno) ON DELETE CASCADE
```

### 2. 外键列的数据类型匹配
外键列的数据类型必须与被引用列的数据类型**完全一致**（包括长度、精度、符号等）。例如，`Student(Sno)`是`CHAR(9)`，那么`SC(Sno)`也必须是`CHAR(9)`。

### 3. NULL值处理
- 外键列允许`NULL`值，除非显式添加了`NOT NULL`约束。如果外键列为`NULL`，则参照完整性**不检查**该行（即允许孤儿行）。
- 在实际业务中，应根据需求决定外键列是否允许`NULL`。

### 4. 自引用外键
外键也可以引用同一张表的其他列（例如员工表的经理ID引用员工ID）。语法相同，只需将`REFERENCES`指向自身。

### 5. 复合外键
外键可以由多个列组成，此时必须引用被引用表的复合主键或复合唯一键，且列数、顺序和数据类型要匹配。

### 6. 循环依赖
在多个表之间定义外键时，可能形成循环依赖（如A引用B，B引用A）。创建表时需要先创建其中一个表并暂时禁用外键检查，或者使用`ALTER TABLE`在表创建后添加外键。

### 7. 数据库对参照完整性的支持
- 主流关系数据库（如MySQL、PostgreSQL、Oracle、SQL Server）都支持外键约束，但默认存储引擎可能有所不同（例如MySQL的MyISAM不支持外键，而InnoDB支持）。
- 某些数据库允许临时禁用外键检查（如`SET FOREIGN_KEY_CHECKS=0`），用于批量数据导入。

### 8. 性能影响
外键约束会带来额外的检查开销，尤其是在插入、更新、删除操作时。但它们是保证数据一致性的重要手段，通常利大于弊。

---

## 总结
参照完整性是关系数据库设计的核心原则之一，通过外键约束将表之间的数据关联起来，防止孤立数据和不一致现象。PPT通过学生-课程-选课的经典示例，清晰地展示了如何在`CREATE TABLE`中使用`PRIMARY KEY`、`UNIQUE`和`FOREIGN KEY`来定义这些约束。理解并正确使用参照完整性，能够确保数据库中的数据始终保持逻辑上的正确性和可靠性。
