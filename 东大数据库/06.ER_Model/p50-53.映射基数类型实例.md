这4页PPT通过一个统一的业务场景——**客户（customer）与贷款（loan）之间的借贷关系（borrower）**，分别展示了四种映射基数类型在ER图中的具体应用。下面我将按顺序逐页详细讲解，并补充相关知识点。

---

## 第1页：一对一关系（One-to-One）

### 页面内容回顾
- **示例描述**：
  - 一个客户通过 `borrower` 关系最多关联一笔贷款。
  - 一笔贷款通过 `borrower` 关系最多关联一个客户。
- **ER图元素**：
  - 实体：`customer`（属性：customer-id, customer-name, customer-street, customer-city）
  - 实体：`loan`（属性：loan-number, amount）
  - 关系：`borrower`

### 详细讲解

这一页展示的是一对一关系。在现实世界中，一对一关系并不常见于客户与贷款之间（通常一个客户可以有多笔贷款，一笔贷款也可由多人共同借款），但此处作为示例说明概念。

#### 一对一关系的特征
- 客户与贷款之间是**双向唯一关联**。
- 在ER图中，通常通过在关系 `borrower` 的两侧分别画箭头（→）来表示“一”的约束，即每个实体最多参与一次。

#### 业务含义
假设业务规则为：
- 一笔贷款只能借给一个客户，且一个客户最多只能借一笔贷款。
- 这意味着 `customer` 和 `loan` 之间的关联数都是 `0..1`（最大为1）。

#### 在关系数据库中的实现
- 方法一：在 `loan` 表中添加 `customer_id` 外键，并设置该字段为 `UNIQUE`，确保每笔贷款只对应一个客户；同时，在应用层保证每个客户最多关联一笔贷款。
- 方法二：创建中间表 `borrower`，其中包含 `customer_id` 和 `loan_number`，并将两个字段分别设置 `UNIQUE` 约束，或者将其中一列设为主键，另一列设唯一。
- 通常更简洁的做法是**将关系合并到任一方实体表中**（比如在贷款表中添加客户外键），因为一对一关系往往可以归并为单表。

## 第2页：一对多关系（One-to-Many）

### 页面内容回顾
- **示例描述**：
  - 一笔贷款最多关联一个客户（通过 `borrower`）。
  - 一个客户可以关联多笔贷款（包括0笔）。
- **ER图**：同样的实体和关系，但映射基数不同。

### 详细讲解

这一页展示的是一对多关系，这是最常见的映射基数类型之一。

#### 一对多关系的特征
- 从 `loan` 到 `customer` 是“多对一”（即一笔贷款只对应一个客户），但从 `customer` 到 `loan` 是“一对多”（一个客户可以有多个贷款）。
- 在ER图中，通常在“一”方（这里是 `loan` 侧）画箭头，在“多”方（`customer` 侧）画横线或无向线。

#### 业务含义
典型场景：一个客户可以申请多笔贷款，但每笔贷款只属于一个客户（例如个人贷款）。这是银行系统常见的建模方式。

#### 在关系数据库中的实现
- 在 `loan` 表中添加 `customer_id` 外键，**不需要设置唯一约束**，因为一个客户可以有多个贷款。
- 这种方式直接体现了“多”在 `loan` 端（多个贷款记录可以指向同一个客户）。

## 第3页：多对一关系（Many-to-One）

### 页面内容回顾
- **示例描述**：
  - 一笔贷款可以关联多个客户（包括0个）。
  - 一个客户最多关联一笔贷款。

### 详细讲解

这一页展示了多对一关系，实际上与第2页是一对多关系的“反向视角”。

#### 多对一关系的特征
- 从 `customer` 到 `loan` 是“多对一”（多个客户可以共享同一笔贷款）。
- 从 `loan` 到 `customer` 是“一对多”（一笔贷款可以有多个客户）。
- 常见场景：**联名贷款**，即多个人共同借款，但每个人只能参与一笔贷款（业务规则限制）。

#### 业务含义
假设业务规则规定：
- 一笔贷款可以由多个客户共同承担（联合借款人）。
- 每个客户最多只能有一笔贷款（比如为了风险控制）。

#### 在关系数据库中的实现
- 需要在 `loan` 与 `customer` 之间建立关联表，因为一个贷款对应多个客户，且一个客户最多对应一个贷款。
- 创建中间表 `borrower`，包含 `customer_id` 和 `loan_number`。
  - 在 `customer_id` 上设置 `UNIQUE` 约束，确保每个客户只出现在一笔贷款中。
  - `loan_number` 不需要唯一，因为一笔贷款可以有多个客户。

#### 注意
多对一实际上是一对多的对称，在ER图中通常通过箭头的指向来区分哪一侧是“一”。在这个例子中，`customer` 侧为“一”，`loan` 侧为“多”。

## 第4页：多对多关系（Many-to-Many）

### 页面内容回顾
- **示例描述**：
  - 一个客户可以关联多笔贷款（可能0笔）。
  - 一笔贷款可以关联多个客户（可能0个）。

### 详细讲解

这一页展示的是最灵活的多对多关系，现实中客户与贷款的关系通常就是多对多（一个客户可以有多个贷款，一笔贷款可以有多个客户，如联名贷款且允许客户有多笔贷款）。

#### 多对多关系的特征
- 双向无数量限制（除参与约束可能规定最小基数外）。
- 在ER图中，关系两侧通常都用横线（无向线）表示“多”。

#### 业务含义
典型场景：银行中，一个客户可以办理多笔贷款，一笔贷款可以有多个共同借款人（如夫妻共同贷款）。这种模型能覆盖大多数实际情况。

#### 在关系数据库中的实现
- **必须使用关联表**（中间表）`borrower`，包含两个外键：`customer_id` 和 `loan_number`。
- 通常将两个字段的**组合**设为主键，以唯一标识每个借贷关系。
- 不需要在任一外键上设置唯一约束，因为多对多允许重复出现（一个客户可以有多个贷款，一个贷款可以有多个客户）。

## 补充知识点

### 1. 映射基数在ER图中的标准表示
- **箭头（→）**：表示“一”，即该实体在关系中最多出现一次。
- **横线（—）**：表示“多”，即该实体可以出现多次。
- 在实际绘图中，箭头通常画在关系菱形与实体矩形之间的连线上，靠近实体一侧。

### 2. 参与约束（Participation Constraint）
除了映射基数（最大基数），ER模型还可以表达**最小基数**，即一个实体是否必须参与关系。
- **总参与（Total Participation）**：用双线表示，表示该实体的每个实例都必须在关系中至少出现一次。
- **部分参与（Partial Participation）**：用单线表示，允许实体实例不参与关系。

例如，在“一对多”的贷款例子中，如果规定“每个客户至少有一笔贷款”，则需要在 `customer` 侧画双线（总参与）。但在示例中，PPT注明“包括0笔”，所以是部分参与。

### 3. 函数依赖视角
映射基数本质上描述了实体间的函数依赖关系：
- 一对一：`customer_id → loan_number` 且 `loan_number → customer_id`
- 一对多：`loan_number → customer_id`（但反之不成立）
- 多对一：`customer_id → loan_number`（但反之不成立）
- 多对多：无函数依赖

### 4. 选择映射基数的业务考量
- 实际设计时，应仔细分析业务规则，避免将多对多错误简化为其他类型，否则会造成信息丢失或数据冗余。
- 若出现“多对多”且关联本身有属性（如“借款日期”），则中间表还可以包含这些属性。

### 5. 转换为关系模式的通用规则
| 映射基数 | 转换方法                                                     |
| -------- | ------------------------------------------------------------ |
| 一对一   | 任选一方表添加对方主键作为外键，并设为 UNIQUE；或合并为一张表。 |
| 一对多   | 在“多”方表中添加“一”方的主键作为外键。                       |
| 多对一   | 同“一对多”，只需确定哪一侧是“多”。                           |
| 多对多   | 创建新表，包含双方主键作为外键，联合主键。                   |
