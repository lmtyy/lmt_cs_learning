二元联系（两个实体集之间的联系）是 E-R 模型中最常见、最基础的联系类型。将二元联系转化为关系模型（即实际的数据库表结构），核心逻辑取决于联系的**基数比例（一对一、一对多、多对多）**。

下面为你详细总结这三种情况的转换规则，并附上可以直接用于测试的完整 SQL 建表代码，方便你直接复制记笔记。

------

### 一、 一对一（1:1）联系的转换

**1. 核心规则**

在一对一联系中，通常**不需要**建立新的关系表。你可以采用以下两种方案之一：

- **方案 A（常用）：** 将任意一端实体的主键，作为外键加入到另一端实体的表中。同时，可以把联系本身的属性也加到这个表中。
- **方案 B（按参与度优化）：** 最佳实践是看哪一端是**完全参与（强制参与）**的，将外键加在完全参与的那一端，这样可以避免产生大量的 NULL 值。

**2. 具体场景与完整 SQL**

假设有“系（Department）”和“系主任（Manager）”两个实体，一个系只有一个系主任，一个教师最多只能当一个系的系主任（1:1）。我们将系主任（Manager）的 ID 作为外键放在系（Department）表中。

SQL

```mysql
create table manager
( 
    manager_id char(10) primary key,
    manager_name char(20) not null,
    title char(20)
);

create table department
( 
    dept_id char(10) primary key,
    dept_name char(30) not null,
    -- 将系主任的主键作为外键加进来
    manager_id char(10) unique, 
    -- 建立外键约束
    foreign key (manager_id) references manager(manager_id)
);
```

*注意：在 1:1 联系中，作为外键的列通常需要加上 `UNIQUE` 约束，以确保一个系主任不会被分配给两个不同的系。*

------

### 二、 一对多（1:N）联系的转换

**1. 核心规则**

一对多联系也**不需要**建立新的关系表。

- **铁律：** 必须将**“1”端实体的主键**，作为外键加入到**“N”端实体的表中**。如果这个联系本身有属性，也一并加入到“N”端的表中。
- **原因：** 如果加在“1”端，由于“1”对应多个“N”，会导致单个字段需要存储多个值，违反了关系数据库的第一范式（原子性）。

**2. 具体场景与完整 SQL**

假设有“部门（Department）”和“员工（Employee）”实体，一个部门有多名员工，但一名员工只能属于一个部门（1:N）。

SQL

```mysql
create table department
( 
    dept_id char(10) primary key,
    dept_name char(30) not null,
    location char(50)
);

create table employee
( 
    emp_id char(10) primary key,
    emp_name char(20) not null,
    salary decimal(10, 2),
    -- 将“1”端（部门）的主键放在“N”端（员工）作为外键
    dept_id char(10), 
    foreign key (dept_id) references department(dept_id)
);
```

------

### 三、 多对多（M:N）联系的转换

**1. 核心规则**

多对多联系**必须**建立一张独立的、新的关系表（通常称为中间表或交叉表）。

- **表结构：** 新表需要包含参与联系的两个实体的主键。
- **主键选取：** 这两个实体的主键组合在一起，构成这张新表的**复合主键（Primary Key）**。
- **外键约束：** 这两个字段同时也分别是**外键（Foreign Key）**，指向各自的实体表。
- **联系的属性：** 如果这个多对多联系本身带有属性（比如学生选课的“成绩”），必须放在这张新表中。

**2. 具体场景与完整 SQL**

假设有“学生（Student）”和“课程（Course）”实体，一个学生可以选修多门课程，一门课程也可以被多名学生选修（M:N）。他们之间有一个“选修（Enrollment）”联系，且包含“成绩（Grade）”属性。

SQL

```mysql
create table student
( 
    stu_id char(10) primary key,
    stu_name char(20) not null,
    age int
);

create table course
( 
    course_id char(10) primary key,
    course_name char(30) not null,
    credits int
);

-- 必须为 M:N 联系新建一张表
create table enrollment
( 
    stu_id char(10),
    course_id char(10),
    grade decimal(4, 1), -- 联系自身的属性
    -- 两个实体的主键组合成复合主键
    primary key (stu_id, course_id),
    -- 分别设置外键约束
    foreign key (stu_id) references student(stu_id),
    foreign key (course_id) references course(course_id)
);
```

### 核心总结速查表

为了方便你快速复习，这里提取了最核心的规律：

| **联系类型** | **是否需要新建表** | **外键放置位置**               | **新表/修改表的主键**          | **联系自身属性的去向** |
| ------------ | ------------------ | ------------------------------ | ------------------------------ | ---------------------- |
| **1:1**      | 不需要             | 放任意一端（推荐放强制参与端） | 原实体主键保持不变             | 跟随外键放在同一个表中 |
| **1:N**      | 不需要             | 必须放在 **N（多）** 的一端    | 原实体主键保持不变             | 放在 **N** 端的表中    |
| **M:N**      | **必须新建**       | 新表中包含两端的主键           | 两端主键的**组合（复合主键）** | 放在**新建的联系表**中 |
