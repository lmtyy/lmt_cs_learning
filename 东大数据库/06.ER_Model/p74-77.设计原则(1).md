## 第一页：设计原则总览

这一页列出了ER建模中应遵循的五条核心设计原则：

1. **忠实性**  
2. **简单性**  
3. **避免冗余**  
4. **选择正确的联系**  
5. **选择正确的元素种类**

这些原则的目标是使ER图**准确、清晰、无冗余、易于理解和实现**。

### 补充：ER设计原则的理论背景
在数据库设计中，ER图是概念模型的核心，其质量直接影响后续关系模式的正确性和可维护性。上述原则可对应到数据库设计中的**规范化理论**（避免冗余）、**清晰表达业务规则**（忠实性）、**最小化复杂性**（简单性）等。

## 第二页：忠实性

### 知识点
- **忠实性**：模型必须忠实地反映应用的具体要求，不能歪曲或遗漏业务规则。
- **举例**：教师教授课程的映射基数。  
  如果业务规则规定“每个教师最多教3门课，每门课至少由1名教师教”，那么ER图中就必须正确体现这些**基数约束**（如使用 `(min,max)` 或 1:N 等标注）。

### 补充知识点
1. **如何做到忠实**  
   - 充分理解需求，将业务陈述中的“必须”“至少”“最多”“恰好”等词汇转化为准确的参与约束和基数。  
   - 不随意简化或复杂化约束。  
   - 对模糊需求进行澄清，避免主观臆断。

2. **常见不忠实的情况**  
   - 把“可选”误画成“强制”（如将 `0..*` 画成 `1..*`）。  
   - 忽略时间维度的约束（如“每个员工在同一时间只能负责一个项目”），导致模型无法表达。  
   - 将三元关系错误拆分为多个二元关系，丢失了三方同时存在的语义。

3. **检验方法**  
   - 使用自然语言对ER图进行“走读”，检查每个实体和关系是否完全匹配业务描述。  
   - 用具体实例数据测试约束是否成立。

## 第三页：简单性

### 知识点
- **简单性**：在没有必要的情况下，不要在设计中添加更多的成分。即“**如无必要，勿增实体**”。
- **举例**：图中展示了一个不必要的中间联系。  
  原图可能想表达“电影”（movies）通过“represent”连接到“holdings”，再通过“owns”连接到“studios”。实际上，如果“represent”和“owns”可以合并或直接连接，就不应引入多余的中间关系。

### 补充知识点
1. **简单性的价值**  
   - 减少模型的复杂度和理解成本。  
   - 降低后续数据库实现的难度（表数量、连接操作）。  
   - 减少维护时出错的可能性。

2. **何时引入中间实体/关系**  
   - 当中间成分本身具有独立属性时（例如“holdings”如果是一个实体，有其自己的属性，则保留）。  
   - 当业务要求记录中间关系的详细信息时（如电影与工作室的“代理合同”包含合同日期、佣金等）。  
   - 当需要表达多对多关系的额外属性时，应使用关联实体，而不是简单的冗余关系。

3. **简单性不等于过度简化**  
   - 简单性是在满足忠实性的前提下追求简洁。如果删减必要元素导致丢失语义，则违背了忠实性，不可取。

4. **与其他原则的关系**  
   - 简单性常与“避免冗余”相辅相成，但更侧重于结构的精简，而非数据重复的消除。

## 第四页：选择正确的联系

### 知识点
- **选择正确的联系**：如果一个联系可以由其他联系推导出来，那么它就不需要显式地加入ER图。
- **举例**：图中展示了 `stars`（演员）、`star-in`（出演）、`movies`（电影）、`owns`（拥有）、`studios`（制片厂）等元素。如果存在 `stars → star-in → movies → owns → studios` 的路径，且业务上“演员与制片厂的关系”能够通过“演员出演的电影所属的制片厂”推导出来，那么就不需要单独画一个“演员–制片厂”的直接联系。

### 补充知识点
1. **可推导联系（冗余联系）**  
   - 在ER图中，如果存在一个关系集，其存在的所有实例都可以通过其他关系集的组合唯一确定，那么该关系集就是冗余的。  
   - 冗余联系会破坏数据库的**更新异常**风险：当基础关系变化时，冗余关系可能变得不一致。

2. **识别冗余联系的方法**  
   - 检查是否可以通过关系路径推导出另一个关系。例如，已知 `A –R1– B` 和 `B –R2– C`，如果 `R1` 和 `R2` 都是函数依赖关系（如1:N），那么 `A` 与 `C` 的关系就隐含了，不需要再画 `A –R3– C`。  
   - 对于多对多关系，情况更复杂，通常不能简单推导，需要具体分析。

3. **举例说明**  
   - 正确的：学生 –选修– 课程，课程 –开设– 系。如果业务规则是“每个学生选修的课程所属的系”可以通过“选修”和“开设”两个关系得到，就不需要再画“学生 –属于– 系”的联系（除非学生本身也有直接所属的系，那又是另一回事）。  
   - 但有时为了查询效率或表达特殊语义，也可以在物理设计中保留冗余，但概念模型一般应避免。

4. **与规范化的关系**  
   - 该原则类似于关系数据库中的**传递依赖**消除（第三范式）。在ER图中消除可推导的联系，可以防止在转换为关系模式后出现不必要的函数依赖和更新异常。

## 整体总结与补充

### 设计原则之间的内在联系
- **忠实性**是基础，确保模型正确反映业务。  
- **简单性**和**避免冗余**（第三页提到但未展开）旨在保持模型清晰高效。  
- **选择正确的联系**和**选择正确的元素种类**（如实体 vs 属性 vs 关系）进一步细化建模决策。

### 其他常见设计原则（补充）
1. **避免使用复合属性**：除非业务需要，否则应将复合属性拆分为原子属性，便于查询。  
2. **谨慎使用派生属性**：如果属性可以通过其他属性计算得出（如“年龄”可由“出生日期”计算），通常不存储在模型中，以免产生不一致。  
3. **区分实体与属性**：如果某个“属性”具有独立意义且可能被多个实体共享，应考虑提升为实体（如“城市”不是“地址”的一部分，而是一个独立实体）。  
4. **命名规范**：使用清晰、一致的命名，避免缩写歧义。

### 实践建议
- 在设计ER图时，先画出所有必要的实体和关系，然后根据上述原则进行审查和优化。  
- 与业务方反复确认每个约束的准确性，确保忠实性。  
- 使用工具（如ERwin、PowerDesigner）辅助检查冗余和一致性。
