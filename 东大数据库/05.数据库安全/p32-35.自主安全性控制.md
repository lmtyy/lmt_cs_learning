这四张幻灯片构成了一个完整的逻辑链条，详细讲解了数据库系统中**自主安全性控制 (Discretionary Access Control, DAC)** 的核心概念。它们从现实世界的需求出发，逐步深入到数据库底层是如何通过不同的机制来实现这些安全需求的。

为了让你更好地理解，我按照这四页 PPT 的顺序，为你拆解其中的知识点：

### 第一页：自主安全性控制示例（需求提出）

**核心知识点：业务场景中的权限需求分析**

这一页设定了一个具体的业务场景：一个员工管理数据库。有一张核心表 `Employee`，包含员工号(P#)、姓名(Pname)、年龄(Page)、性别(Psex)、工资(Psalary)、部门号(D#)和部门领导(HEAD)等字段。

PPT 列举了五种不同角色的现实需求，这其实反映了数据库安全控制的几个主要维度：

1. **员工管理人员**：需要所有权限（增、删、改、查）。这是最高权限的体现。
2. **收发人员**：只需要知道“谁在哪个部门”。**关键点**：他们不能看工资等敏感信息。这就要求数据库能够做到**列级（字段级）**的安全控制。
3. **每个员工**：只能看**自己的**记录，不能看别人的。**关键点**：这要求数据库能够识别当前用户身份，并做到**行级（记录级）**的安全控制。
4. **部门领导**：只能看**自己部门下属**的记录。同样是行级/记录级的控制。
5. **高层领导**：能看所有人、所有列，但**只能读**，不能修改。这是对**操作类型**的控制。

### 第二页：自主安全性控制示例（需求结构化映射）

**核心知识点：S-O-T-P 访问控制模型模型，以及“按名控制”与“按内容控制”的概念**

这一页将第一页的自然语言需求，抽象转化为了计算机可以理解的结构化规则表。它引入了四个核心概念：

- **S (Subject 主体)**：谁在访问？（如：收发员、员工）
- **O (Object 客体)**：访问什么数据？（如：Employee表）
- **T (Type 操作类型)**：能做什么操作？（如：读、改、插、删）
- **P (Predicate 谓词/条件)**：满足什么条件的数据才能被访问？

**结合表格，引出了两种安全控制的核心策略：**

1. **按名控制安全性 (Name-based Control)**：
   - **例子**：收发员那一行。客体 $O$ 不是整个表，而是 `Employee(Pname, D#)`。
   - **含义**：直接通过数据对象的“名字”（表名或具体的列名）来进行权限限制。收发员只能访问这两个“名字”的列。
2. **按内容控制安全性 (Content-based Control)**：
   - **例子**：员工和部门领导这两行。客体 $O$ 是完整的 `Employee` 表，但谓词 $P$ 规定了条件（`P#=Userid` 或 `Head=Userid`）。
   - **含义**：不仅看你要访问哪个表，还要看表里的“内容”是否符合条件。只有满足特定谓词条件（如“数据的负责人ID等于当前登录用户ID”）的“数据行”才允许被访问。

### 第三页：第一种：按名控制安全性（底层实现机制 1）

**核心知识点：访问控制矩阵 (Access Control Matrix)**

这一页讲的是，数据库系统在底层是如何存储和管理上面提到的“按名控制”规则的。

它介绍了一种最经典的存储结构：**存储矩阵**。

- **行**：代表所有的主体（Subject），即 $S_1, S_2 ... S_n$（用户或角色）。
- **列**：代表所有的数据对象（Object），即 $O_1, O_2 ... O_n$（整张表或者特定的列）。
- **交叉点 (单元格)**：里面填写的是权限类型（Type）。比如 All (全部权限)、Read (只读)、- (无权限)。

当用户发起访问请求时，数据库的安全子系统就会去查这个矩阵，匹配“你是谁”、“你要访问哪个对象”，然后看看对应的单元格里有没有相应的权限，从而决定是否放行。这解决了“按名控制”的落地问题。

------

### 第四页：第二种：按内容控制安全性（底层实现机制 2）

**核心知识点：利用“视图 (View)”实现行级/内容级安全控制**

第三页的矩阵很好用，但它只能精确到“列”（按名）。如果要实现“按内容”控制（比如员工只能看自己的数据行），在庞大的矩阵里为每一行数据都建一列是不现实的。这一页给出了完美的解决方案：**视图 (View)**。

- **视图的本质**：视图本身不存储数据，它是一条被保存下来的 SQL 查询语句，是一个“虚拟表”。
- **如何实现安全控制**：我们可以把第二页讲到的 **谓词条件 (P)** 直接写到视图的 `WHERE` 子句里。
  - 例如：`Create View EmpV3 as select * from Employee where P# = :UserId`。这里的 `:UserId` 是当前登录用户的系统变量。
- **巧妙的结合**：这个视图创建好之后，对于数据库来说，它就成了一个**新的“数据对象” (新的 Object)**。
  - 系统接下来只需要在第三页讲的**存储矩阵**中，不给员工赋予 `Employee` 原表的权限，而是直接给他赋予 `EmpV3` 这个视图的权限（这就回到了“按名控制”）。
  - 当员工查询视图时，视图底层的 `WHERE` 条件自动生效，过滤掉别人的数据，从而完美实现了“按内容控制”。

**总结：**

这四页 PPT 讲述了：从 **用户需求** $\rightarrow$ 抽象为 **S-O-T-P规则** $\rightarrow$ 使用 **访问控制矩阵** 实现表/列级控制（按名） $\rightarrow$ 借助 **视图+矩阵** 实现行级控制（按内容）。
