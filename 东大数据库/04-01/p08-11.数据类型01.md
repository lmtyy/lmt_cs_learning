作为一名软件工程专业的学生，理解这些类型不仅是为了写 SQL，更是为了在设计系统时优化存储空间和计算精度。以下是为你整理的核心笔记：

---

## 1. 层次结构：从 Database 到 Table
课件通过两个层级展示了 DDL 的作用：
* **Database 级：** `CREATE DATABASE soft2018;` 
    * 这是最外层的“容器”。在实际科研环境中，你可能会为不同的实验项目（如 `Exp_DiffPure`）建立独立的数据库。
* **Relation（表）级：** `CREATE TABLE Students (...);`
    * 定义了数据的**域（Domain）**，即每一列必须符合特定的类型。DBMS 会在数据输入时进行强制校验。

---

## 2. 数值类型（Numeric Types）：精度与空间的博弈
课件详细对比了整数和浮点数，这部分在处理科研数据指标时非常关键：

### 整数类（Integer）
根据字节数（Byte）不同，取值范围差异巨大：
* **TINYINT (1B):** 适合存性别、等级（0-255）。
* **INT (4B):** 最常用，范围约 $\pm 21$ 亿。
* **BIGINT (8B):** 适合存时间戳或超大规模 ID。

### 浮点类（Float/Fixed-point）
这里有一个非常容易混淆的考点：**精确值 vs 近似值**。
* **DECIMAL(M, D):** **精确数值**。例如 `DECIMAL(5, 2)` 表示总共 5 位，小数占 2 位（如 `236.95`）。它不存在精度丢失，适合存**金钱**或**实验精确度**。
* **FLOAT / REAL / DOUBLE:** **近似数值**。基于 IEEE 754 标准，运算速度快但会有微小的精度偏差。对于你研究的深度学习模型输出（如 Softmax 概率），通常使用 `FLOAT` 或 `REAL` 即可。

## 3. 语法实战演练
最后一张课件通过一个 `test` 表展示了综合运用：
```sql
CREATE TABLE test (
  name  VARCHAR(100),    -- 变长字符串，上限100字符
  col1  DECIMAL(5,2),    -- 精确到百分位
  col2  SMALLINT,        -- 节省空间的短整数
  col5  FLOAT(2)         -- 至少保留两位小数精度的近似值
);
```

---

### 🔍 同学，这里有两个细节建议你注意：

1.  **VARCHAR vs CHAR：** 课件里出现了 `VARCHAR(100)`。虽然它省空间，但在底层索引排序时，定长的 `CHAR` 有时会更快。如果你存的是固定 8 位的 `sid`（学号），用 `CHAR(8)` 是更好的选择。
2.  **MySQL 规范：** 看到你课件里提到了 MySQL 具体的字节对照表。要注意 `INT(11)` 这种写法，括号里的数字在很多 MySQL 版本里只代表“显示宽度”，并不改变存储的大小或取值范围（依然是 4 字节）。
