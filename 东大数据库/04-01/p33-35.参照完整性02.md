这三页课件深入探讨了**参照完整性（Referential Integrity）的维护策略**。

当两张表通过外键“锁”在一起时，如果父表（被参照表）的数据发生了变动（删除或修改），子表（参照表）该如何应对？课件给出了四种经典的解决方案。

---

## 1. 核心矛盾：当父表变动时
假设 `sid` 在 `Enrolled`（选课表）中是引用 `Students`（学生表）的外键：
* **插入冲突：** 如果往 `Enrolled` 插入一个不存在的 `sid`？**结论：直接拒绝（Reject it!）**。
* **删除/更新冲突：** 如果从 `Students` 中删掉一个学生，但他已经选了课，那些选课记录怎么办？

---

## 2. 四种处理策略（重点）
在 `SQL/92` 标准中，你可以通过 `ON DELETE` 或 `ON UPDATE` 后面跟以下子句来定义行为：

| 策略 | 英文名称 | 行为描述 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **拒绝** | **NO ACTION / RESTRICT** | **默认行为**。如果子表还有引用，父表的删除/更新操作直接报错。 | 保护数据安全，防止误删。 |
| **级联** | **CASCADE** | **连带责任**。父表删了，子表相关记录也跟着删；父表主键改了，子表外键跟着改。 | 彻底清理数据，如“删除账户时删除其所有动态”。 |
| **置空** | **SET NULL** | **脱离关系**。父表记录消失后，子表相关外键字段变为 `NULL`。 | 父表记录不重要时。*注意：如果外键是主键的一部分，不能设为 NULL。* |
| **设默认值**| **SET DEFAULT** | **寻找替身**。父表记录消失后，子表外键变为一个预设的默认值。 | 需要业务保底逻辑时。 |


---

## 3. 代码实战解析
最后一页课件展示了具体的 SQL 写法：
```sql
CREATE TABLE Enrolled (
    sid CHAR(20),
    cid CHAR(20),
    PRIMARY KEY (sid, cid),
    FOREIGN KEY (sid) REFERENCES Students(sid)
        ON DELETE CASCADE    -- 删掉学生，选课记录自动消失
        ON UPDATE CASCADE    -- 改了学号，选课记录自动同步
);
```

---

### 💡 针对你 CS 背景的深度解读

#### A. 级联（CASCADE）的威力与危险
作为大学生，你可能觉得 `CASCADE` 非常方便。但在实际大型系统开发中，`ON DELETE CASCADE` 是要慎用的：
* **性能问题：** 如果删掉一个父节点，可能会触发链式反应，导致数据库瞬间删除成千上万行数据，产生严重的 IO 压力。
* **数据不可恢复：** 一旦误删，关联数据全部消失。因此，很多公司更倾向于使用**逻辑删除**（如设置 `is_deleted = 1`）而非物理删除。

#### B. 你的科研场景 (DiffPure 实验管理)
在你管理 **Adversarial Attack** 实验数据时：
* 如果你有一个 `Attacks` 表和 `Results` 表。
* 建议使用 `ON UPDATE CASCADE`：因为如果你发现某个攻击方法的命名写错了，修改父表后，所有实验结果能自动同步。
* 建议使用 `ON DELETE RESTRICT`：防止你不小心删掉了一个攻击策略，导致相关的宝贵实验结果被数据库“顺手”也给清空了。
