## 一、完整性约束概述

### 1. 什么是完整性约束？
完整性约束是定义在数据库表上的一组规则，用于**保证数据的正确性、一致性和有效性**。它们防止了对数据库的意外破坏，确保即使是对数据库的合法修改（如插入、更新、删除）也不会导致数据不一致。

### 2. 示例说明
PPT中给出了三个业务上的完整性要求：
- **支票账户余额必须大于$10,000.00**：这可以通过`CHECK`约束实现，限制`balance`字段的值。
- **银行员工时薪至少$4.00**：同样可以用`CHECK`约束保证`hourly_wage >= 4.00`。
- **客户的电话号码不能为空**：用`NOT NULL`约束强制`phone`字段必须有值。

这些约束将业务规则直接嵌入数据库，由数据库系统自动强制执行，避免了应用程序层面的遗漏或错误。

## 二、单表上的约束类型

PPT列出了可以在单个表上定义的四种基本约束：

- **`NOT NULL`**：非空约束，确保列不允许存储`NULL`值。
- **`PRIMARY KEY`**：主键约束，唯一标识表中的每一行，隐含非空和唯一性。
- **`UNIQUE`**：唯一约束，保证列或列组合的值唯一，但允许`NULL`（通常）。
- **`CHECK (P)`**：检查约束，`P`是一个谓词（条件表达式），必须对每一行为真或未知（`NULL`不触发失败）。

下面分别详细讲解。

---

## 三、`NOT NULL` 约束

### 1. 作用
强制某列的值不能为`NULL`，即该列在每一行中都必须有具体值。

### 2. 语法
在列定义后直接添加`NOT NULL`关键字：
```sql
Ssex CHAR(2) NOT NULL
```

### 3. 注意事项
- `NOT NULL`只能作为列级约束，不能作为表级约束（因为它是针对单个列的）。
- 主键列隐含了`NOT NULL`，无需显式声明。
- 如果向`NOT NULL`列插入`NULL`或更新为`NULL`，数据库会拒绝操作并报错。

---

## 四、`UNIQUE` 约束

### 1. 作用
保证指定列或列组合的值在表中是唯一的，即不能有重复值。这样的列或列组合构成了**候选键**。

### 2. 语法
- 列级：`Sname CHAR(20) UNIQUE`
- 表级：`UNIQUE (col1, col2)`

### 3. 与主键的区别
- **`PRIMARY KEY`** 隐含`NOT NULL`，且一个表只能有一个主键。
- **`UNIQUE`** 允许`NULL`值（通常最多一个`NULL`，具体取决于数据库实现，如MySQL允许多个`NULL`，而SQL Server只允许一个`NULL`）。
- `UNIQUE`约束可以有多个，而主键只能有一个。

### 4. 候选键概念
候选键是可以唯一标识元组的属性集，主键是选定的一个候选键。因此，`UNIQUE`约束定义的正是候选键。

### 5. 注意事项
- 当对允许`NULL`的列建立唯一约束时，要了解数据库对重复`NULL`的处理规则，避免业务逻辑出错。
- 组合唯一约束中，所有列的组合值必须唯一，但单个列可以有重复。

---

## 五、`CHECK` 约束

### 1. 作用
通过一个布尔表达式（谓词）来限制列或列组合的取值，确保满足特定条件。

### 2. 语法
- 列级：`assets INTEGER CHECK (assets >= 0)`
- 表级：`CHECK (assets >= 0)`
- 也可以定义复杂的表达式，引用多列：
  ```sql
  CHECK (start_date < end_date)
  ```

### 3. 示例分析
PPT中的例子：
```sql
CREATE TABLE branch (
    branch_name CHAR(15),
    branch_city CHAR(30),
    assets INTEGER,
    PRIMARY KEY (branch_name),
    CHECK (assets >= 0)
);
```
这里`CHECK (assets >= 0)`确保`assets`列的值不能为负数，符合业务逻辑（资产不能为负）。

### 4. 注意事项
- **谓词中的NULL处理**：如果条件中的列值为`NULL`，则整个条件的结果为`UNKNOWN`，`CHECK`约束不会拒绝该行（因为`CHECK`只拒绝结果为`FALSE`的行）。因此，如果希望禁止`NULL`，需要同时加`NOT NULL`约束。
- **数据库支持差异**：某些数据库（如MySQL 5.7及之前版本）会解析`CHECK`但不会真正强制执行（InnoDB从8.0.16开始支持），使用时需查阅文档。PostgreSQL、Oracle等则完全支持。
- **表达式限制**：`CHECK`中不能使用子查询、用户函数或不确定函数（如当前时间），但可以引用同一行中的其他列。
- **性能考虑**：复杂的`CHECK`表达式可能会影响插入和更新的性能，但通常影响很小。

## 补充知识点

### 1. 约束的命名
为约束指定名称有助于后续管理和修改。例如：
```sql
CREATE TABLE student (
    Sno CHAR(9) CONSTRAINT pk_student PRIMARY KEY,
    Sname CHAR(20) CONSTRAINT uq_student_name UNIQUE,
    Sage SMALLINT CONSTRAINT ck_student_age CHECK (Sage > 15 AND Sage < 50)
);
```
如果没有命名，数据库会自动生成一个系统名，不易识别。

### 2. 组合约束
- **组合主键**：`PRIMARY KEY (col1, col2)`
- **组合唯一**：`UNIQUE (col1, col2)`
- **组合检查**：`CHECK (col1 + col2 > 100)`

### 3. 约束的修改与删除
可以使用`ALTER TABLE`语句添加或删除约束：
```sql
-- 添加NOT NULL（需先处理现有NULL值）
ALTER TABLE student MODIFY Ssex CHAR(2) NOT NULL;

-- 添加CHECK约束
ALTER TABLE student ADD CONSTRAINT ck_student_age CHECK (Sage > 15);

-- 删除约束
ALTER TABLE student DROP CONSTRAINT ck_student_age;
```

### 4. 约束与业务逻辑
完整性约束将数据规则集中在数据库层，具有以下优点：
- **集中控制**：所有应用程序共享相同的规则，避免重复编码。
- **保证一致性**：无论通过何种方式修改数据，约束都会强制执行。
- **提高可靠性**：减少因应用程序缺陷导致的数据错误。

但也要注意，复杂的业务规则可能需要结合触发器或应用层逻辑实现。

---

## 总结

完整性约束是关系数据库确保数据质量的重要机制。通过`NOT NULL`、`PRIMARY KEY`、`UNIQUE`和`CHECK`，我们可以：
- 防止关键字段为空；
- 确保数据唯一性；
- 限制取值范围的合法性；
- 定义表内的业务规则。

在设计数据库时，合理运用这些约束不仅能保证数据的正确性，还能减轻应用程序的负担，提升系统的整体健壮性。但需留意不同数据库对约束的支持细节，尤其是`CHECK`约束的实现差异，以及`NULL`在唯一约束中的处理方式。
