## 一、`CREATE TABLE` 基本语法

PPT中给出的通用格式如下：

```sql
CREATE TABLE r (
    A1 D1 integrity-constraint,
    A2 D2 integrity-constraint,
    ...,
    An Dn integrity-constraint,
    integrity-constraint1,
    integrity-constraint2,
    ...,
    integrity-constraintk
);
```

- **`r`**：要创建的关系（表）的名称。
- **`Ai`**：第 i 个属性的名称（列名）。
- **`Di`**：属性 `Ai` 的数据类型（如 `CHAR(9)`、`SMALLINT` 等）。
- **`integrity-constraint`**：完整性约束，可以定义在列级（紧随数据类型之后）或表级（在列定义列表之后单独列出）。

### 1. 列级约束与表级约束
- **列级约束**：只针对该列，写在列定义之后，例如 `Sno CHAR(9) PRIMARY KEY`。
- **表级约束**：可以涉及多列，写在所有列定义之后，例如 `PRIMARY KEY (Sno, Sname)` 或 `FOREIGN KEY (Sdept) REFERENCES Department(DeptID)`。

**注意**：某些约束（如 `NOT NULL`）只能作为列级约束；而主键、唯一、外键、检查约束既可以作为列级也可以作为表级。

## 二、示例详解：学生表 Student

PPT中给出了两个略有差异的建表语句，我们将其合并分析。

### 示例1（带列级主键和CHECK）

```sql
CREATE TABLE Student (
    Sno   CHAR(9) PRIMARY KEY,   -- 列级主键
    Sname CHAR(20) UNIQUE,       -- 列级唯一约束
    Ssex  CHAR(2) NOT NULL,      -- 非空约束
    Sage  SMALLINT,
    Sdept CHAR(20),
    CHECK (Sage > 15 AND Sage < 50)   -- 表级检查约束
);
```

### 示例2（表级主键）

```sql
CREATE TABLE Student (
    Sno   CHAR(9),
    Sname CHAR(20) UNIQUE,   -- 列级唯一
    Ssex  CHAR(2),
    Sage  SMALLINT,
    Sdept CHAR(20),
    PRIMARY KEY (Sno)        -- 表级主键
);
```

两个示例都定义了：
- **主键**：`Sno` 作为主码，值唯一且非空（主键隐式包含非空约束）。
- **唯一约束**：`Sname` 取值唯一，但允许 `NULL`（通常唯一约束允许一个 `NULL`，但不同数据库处理可能不同）。
- 数据类型：`CHAR(9)` 固定长度字符串，`SMALLINT` 小整数。

**区别**：
- 示例1在列级定义主键，同时添加了 `NOT NULL` 对 `Ssex` 和表级 `CHECK` 对年龄的限制。
- 示例2使用表级主键，没有 `NOT NULL` 和 `CHECK`。

## 三、完整性约束详解

### 1. 主键约束（`PRIMARY KEY`）
- 保证一列或一组列的值**唯一且非空**。
- 每个表只能有一个主键。
- 可以是单列（列级或表级）或多列组合（必须用表级）。
- **易错点**：
  - 误以为主键列可以不写 `NOT NULL`，但标准 SQL 中主键隐式包含非空，但某些数据库（如 MySQL）允许在列级定义 `PRIMARY KEY` 而不显式写 `NOT NULL`，实际会强制非空。
  - 组合主键时，所有列的组合值必须唯一，但单个列可以有重复。
  - 主键列的数据类型应尽量短小（如用 `INT` 而非 `CHAR(100)`），以提高索引效率。

### 2. 唯一约束（`UNIQUE`）
- 保证列或列组合的值**唯一**，但允许 `NULL` 值（通常最多一个 `NULL`，因为 `NULL` 视为未知，多个 `NULL` 是否违反唯一性取决于数据库实现，标准允许一个 `NULL`，但有些数据库如 MSSQL 只允许一个 `NULL`）。
- 一个表可以有多个唯一约束。
- **易错点**：
  - 混淆唯一约束与主键：主键不允许 `NULL`，唯一约束允许 `NULL`。
  - 当对允许 `NULL` 的列建唯一约束时，要清楚数据库对重复 `NULL` 的处理规则（例如 Oracle 允许多个 `NULL`，而 SQL Server 仅允许一个 `NULL`）。

### 3. 非空约束（`NOT NULL`）
- 确保列中不存储 `NULL` 值。
- 只能作为列级约束。
- **易错点**：
  - 忘记为业务上必填的字段添加 `NOT NULL`，导致数据不完整。
  - 修改现有表添加 `NOT NULL` 时需要先处理已有 `NULL` 值，否则会失败。

### 4. 检查约束（`CHECK`）
- 定义每行必须满足的条件（布尔表达式）。
- 可以引用同一行中的多个列，但不能引用其他表或不确定函数（如 `GETDATE()`）。
- **易错点**：
  - 不同数据库对 `CHECK` 的支持力度不同：MySQL 解析但忽略 `CHECK`（除非使用特定存储引擎或版本），PostgreSQL 和 Oracle 完全支持。
  - 条件表达式可能因数据类型隐式转换导致意外结果，例如 `Sage > 15` 若 `Sage` 为字符类型会比较字典序而非数值。
  - `CHECK` 中的表达式对 `NULL` 的处理：如果条件中涉及列且该列为 `NULL`，则 `CHECK` 通常被视为通过（因为 `NULL` 参与比较结果未知，而 `CHECK` 只拒绝明确为 `FALSE` 的行）。例如 `CHECK (Sage > 15)` 对于 `Sage IS NULL` 的行是允许的，除非加上 `Sage IS NOT NULL`。

### 5. 外键约束（`FOREIGN KEY`）（PPT未出现，但属于重要完整性约束）
- 保证当前表的列值与另一表的主键或唯一键列的值匹配。
- 可定义级联操作（`ON DELETE CASCADE` 等）。
- **易错点**：
  - 外键列的数据类型必须与引用列完全匹配（包括长度、符号等）。
  - 外键列允许 `NULL`，除非加了 `NOT NULL`。
  - 在删除或更新被引用表数据时，如果未正确处理级联选项，可能因外键冲突而失败。

## 四、数据类型选择要点

PPT中使用了 `CHAR(9)`、`CHAR(20)`、`SMALLINT`。这些是常见类型，但需要留意：

- **`CHAR(n)`**：固定长度字符串，如果存储的字符串长度不固定，会浪费空间，且比较时会在末尾补空格（可能影响查询结果）。对于变长字符串，优先用 `VARCHAR(n)`。
- **`SMALLINT`**：占用2字节，范围小，适合年龄（0~65535 无符号或 -32768~32767 有符号），但年龄通常不会超过150，用 `SMALLINT` 合适。
- **性别 `Ssex`**：用 `CHAR(2)` 可能存储 '男'/'女' 或 'M'/'F'，但如果用单字符，`CHAR(1)` 更省空间。

**易错点**：
- 不根据实际数据范围选择整数类型：例如年龄用 `INT` 浪费空间，用 `TINYINT`（如果支持）更合适。
- 字符类型长度定义过长或过短：如 `Sname CHAR(20)` 若姓名超过20字会截断；若长度远大于实际值，用 `CHAR` 浪费空间。
- 忽略字符集和排序规则对比较和存储的影响（如大小写敏感、重音敏感）。

## 五、表定义中的常见易错点总结

1. **主键选择不当**  
   - 使用业务字段（如身份证号）做主键，但业务字段可能变更或过大，影响性能。推荐使用自增整数或 UUID 作为代理主键。
   - 忘记定义主键，导致数据重复且无法建立有效索引。

2. **唯一约束与业务逻辑冲突**  
   - 对允许重复的业务字段误加唯一约束，导致插入失败。
   - 唯一约束与 `NULL` 的交互未考虑清楚，例如允许多个 `NULL` 还是只允许一个。

3. **非空约束遗漏**  
   - 业务上必填字段未加 `NOT NULL`，导致数据缺失。

4. **`CHECK` 约束不生效或条件错误**  
   - 在 MySQL 中误以为 `CHECK` 有效，实际需要应用层或触发器保证。
   - 条件写反或逻辑错误，如 `Sage < 15 AND Sage > 50` 永远为假。
   - 未处理 `NULL` 导致数据绕过检查。

5. **字符类型与比较问题**  
   - 用 `CHAR` 存变长数据导致末尾空格影响查询（例如 `WHERE Sname = '张三'` 可能不匹配存储为 `'张三   '` 的记录）。
   - 不同字符集导致乱码或比较异常。

6. **语法错误**  
   - 列定义最后多逗号或少逗号。
   - 表级约束漏掉括号，如 `PRIMARY KEY Sno` 错误，应为 `PRIMARY KEY (Sno)`。
   - 约束命名重复（可显式命名避免）。

7. **隐式提交与事务**  
   - 在事务中执行 `CREATE TABLE` 会导致隐式提交（某些数据库），打断事务。

8. **保留字作为标识符**  
   - 使用了 SQL 保留字（如 `DATE`、`USER`）作为列名或表名，需用引号括起，但不同数据库引号规则不同（MySQL 用反引号，SQL Server 用方括号，标准用双引号）。最好避免使用保留字。

9. **不同数据库的方言差异**  
   - 如 `AUTO_INCREMENT`（MySQL）、`IDENTITY`（SQL Server）、`SERIAL`（PostgreSQL）等自增语法不同。
   - 数据类型名称差异，如 `INTEGER` 与 `INT`，`DOUBLE` 与 `FLOAT` 等。

## 六、补充：约束命名与管理

- 可以为约束指定名称，便于后续修改或删除。例如：
  ```sql
  CONSTRAINT pk_student PRIMARY KEY (Sno),
  CONSTRAINT uq_student_name UNIQUE (Sname),
  CONSTRAINT ck_student_age CHECK (Sage > 15 AND Sage < 50)
  ```
- 未命名时，数据库会自动生成系统名，不易管理。

---

## 七、总结

`CREATE TABLE` 是 SQL 中最基础也最重要的 DDL 语句，它定义了表的结构和数据的完整性规则。正确使用完整性约束可以保证数据的准确性和一致性，减少应用层的校验负担。在实际开发中，需要：
- 根据业务需求合理选择数据类型和长度。
- 明确主键、唯一、非空、检查等约束的适用场景。
- 注意数据库之间的差异，确保建表语句在目标平台上正确执行。
- 提前规划好约束的命名和后期维护。

通过深入理解这些知识点并留意易错点，可以有效避免在数据库设计阶段埋下隐患。
