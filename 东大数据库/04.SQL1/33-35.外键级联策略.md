这三页PPT主要围绕**参照完整性（Referential Integrity）的强制执行**展开，讨论了当对父表（被引用表）进行删除或更新操作时，数据库系统应如何处理子表（引用表）中的相关数据。下面我们逐一详细讲解每页的内容，并补充相关知识点。

---

## 第一页：插入违规的处理

**内容回顾**  
- 以学生表（`Students`）和选课表（`Enrolled`）为例，`Enrolled` 表中的 `sid` 是外键，引用 `Students` 表的 `sid`。  
- 问题：如果试图向 `Enrolled` 表中插入一条记录，而该记录的 `sid` 在 `Students` 表中不存在，应该怎么办？  
- 答案：**拒绝插入（Reject it!）**。

### 解释
参照完整性的基本要求是：**外键列的值必须等于被引用表中某个主键或候选键的值，或者为 `NULL`（如果允许）**。当插入违反这一规则时，数据库必须拒绝该操作，并返回错误，以防止产生“孤立记录”。这是最直接、最严格的保护措施，确保数据的一致性。

---

## 第二页：删除父表记录时的处理选项

**内容回顾**  
当删除 `Students` 表中的一条记录时，该学生的 `sid` 可能被 `Enrolled` 表中的多条记录引用。数据库系统提供了多种处理策略：

1. **级联删除（Cascading Deletion）**：同时删除所有引用该学生的选课记录。
2. **拒绝删除（Disallow deletion）**：如果存在引用，则禁止删除该学生记录。
3. **设置默认值（Set to a default sid）**：将 `Enrolled` 表中相关记录的 `sid` 设为某个默认值。
4. **设为 NULL（Set to null）**：将 `sid` 设为 `NULL`，但本例中 `sid` 是 `Enrolled` 表主键的一部分，因此该选项不适用（主键列不允许为 `NULL`）。

此外，如果更新了 `Students` 表中的 `sid`（主键），也需要相应更新 `Enrolled` 表中引用该 `sid` 的记录，以保持一致性。

### 详细说明
#### 为什么需要多种选项？
不同的业务场景需要不同的行为：
- **级联删除**：适用于“子记录完全依赖于父记录”的场景，例如订单明细随订单一起删除。
- **拒绝删除**：适用于需要保护历史记录或防止误删，例如某学生已有成绩记录，不能直接删除学生信息。
- **设为默认值**：适用于将孤立记录转移到一个“默认”实体（如“未知学生”）。
- **设为 NULL**：适用于外键列允许为空的场景，表示该记录不再关联任何父记录。

#### 主键列的特殊性
在本例中，`Enrolled` 表的主键是 `(sid, cid)`，因此 `sid` 是主键的一部分，不能为 `NULL`。这就是为什么“设为 NULL”选项不适用。

#### 更新父表主键的处理
如果更新了父表的主键值（例如修改学号），通常也需要同步更新子表中对应的外键值，否则会破坏参照完整性。这可以通过 `ON UPDATE CASCADE` 实现。

## 第三页：SQL/92 中的参照完整性选项

**内容回顾**  
- SQL/92 标准支持在定义外键时指定删除和更新时的四种动作：
  - **NO ACTION**（默认）：拒绝删除/更新（与“拒绝删除”相同，但检查时机可能有细微差别）。
  - **CASCADE**：级联删除/更新。
  - **SET NULL**：将外键列设为 `NULL`。
  - **SET DEFAULT**：将外键列设为默认值。
- 示例建表语句：
  ```sql
  CREATE TABLE Enrolled (
      sid CHAR(20),
      cid CHAR(20),
      grade CHAR(2),
      PRIMARY KEY (sid, cid),
      FOREIGN KEY (sid) REFERENCES Students(sid)
          ON DELETE CASCADE
          ON UPDATE CASCADE
  );
  ```

### 详细解释四种动作
1. **NO ACTION**  
   - 当删除或更新父表记录时，如果存在依赖的子记录，则操作被拒绝。  
   - 这是默认行为（如果不指定任何选项）。  
   - 注意：在某些数据库中，`NO ACTION` 和 `RESTRICT` 的行为可能略有不同（`RESTRICT` 立即检查，`NO ACTION` 可能在事务结束时检查），但多数情况下含义相同。

2. **CASCADE**  
   - 删除父记录时，自动删除所有引用它的子记录。  
   - 更新父记录的主键时，自动将子表中的外键值也更新为新值。  
   - 示例中指定了 `ON DELETE CASCADE` 和 `ON UPDATE CASCADE`，意味着删除学生时其选课记录也被删除，修改学号时选课表中的学号也同步更新。

3. **SET NULL**  
   - 删除或更新父记录时，将子表中相关行的外键列设为 `NULL`。  
   - 要求该外键列允许 `NULL`。  
   - 例如，如果部门被删除，可以将员工的外键部门ID设为 `NULL`，表示员工暂时未分配部门。

4. **SET DEFAULT**  
   - 删除或更新父记录时，将子表中相关行的外键列设为预先定义的默认值。  
   - 要求该列已定义 `DEFAULT` 值，且默认值必须存在于被引用表中（或允许 `NULL`）。  
   - 例如，将孤立订单的客户ID设为默认的“未知客户”。

### 示例解析
PPT 中的建表语句创建了 `Enrolled` 表，其外键 `sid` 引用 `Students(sid)`，并指定了 `ON DELETE CASCADE` 和 `ON UPDATE CASCADE`。这意味着：
- 如果从 `Students` 表中删除某学生，该学生的所有选课记录会自动从 `Enrolled` 表中删除。
- 如果更新 `Students` 表中某学生的学号，`Enrolled` 表中对应记录的学号也会自动更新为新值。

这种设计保证了无论对学生表做什么操作，选课表都能自动保持一致性，无需应用程序额外处理。

## 补充知识点：参照完整性的实际应用

1. **选择合适动作**  
   - 如果业务要求子记录必须随父记录消失（如购物车与商品项），用 `CASCADE`。  
   - 如果父记录被删除后，子记录仍需保留但需置空或置默认值，用 `SET NULL` 或 `SET DEFAULT`。  
   - 如果禁止删除有依赖的父记录（如已经产生交易的用户），用 `NO ACTION`。

2. **复合外键的处理**  
   当外键由多列组成时，`SET NULL` 或 `SET DEFAULT` 会作用于所有外键列，必须保证这些列都允许 `NULL` 或有默认值。

3. **自引用外键**  
   例如员工表的经理ID引用同一表的员工ID，同样可以使用上述选项，但要避免循环依赖导致的级联操作无限递归（数据库通常能检测并处理）。

4. **数据库实现的细微差异**  
   - 某些数据库（如 MySQL InnoDB）完全支持这些选项。  
   - 某些数据库（如 SQLite）可能只支持部分选项，或行为略有不同。  
   - 在定义外键前，建议查阅所用数据库的官方文档。

5. **性能影响**  
   级联操作会触发额外的删除或更新，可能影响性能，尤其是在大数据量下。但它是保证数据一致性的可靠手段，通常利大于弊。

---

## 总结

这三页 PPT 系统地介绍了参照完整性的强制执行策略：
- **插入时**：直接拒绝非法外键值。
- **删除/更新时**：提供了四种动作供选择，以适应不同业务需求。
- SQL/92 标准明确了这些动作的语法和行为，使数据库能够自动维护跨表的数据一致性。

理解并正确使用这些选项，是设计健壮数据库应用的关键。在实际开发中，应根据业务逻辑仔细选择每个外键的删除和更新规则，并在必要时通过测试验证其行为是否符合预期。
