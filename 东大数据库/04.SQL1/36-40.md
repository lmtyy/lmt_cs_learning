这五页PPT主要讲解了SQL中用于修改数据库模式的两个重要DDL命令：`DROP TABLE` 和 `ALTER TABLE`。下面将逐一详细解释每一页的内容，并补充相关知识点和注意事项。

---

## 第一页：`DROP TABLE` 命令

**内容原文**：
- `drop table` 命令从数据库中删除被删除关系的所有信息。
- 语法：`drop table r`
- 它不仅删除 `r` 中的所有元组，还删除 `r` 的模式（定义）。

**详解**：
- **作用**：`DROP TABLE` 用于完全移除一个表及其所有相关对象（数据、索引、触发器、约束、权限等）。执行后，表的结构不再存在，任何对该表的后续引用都会失败。
- **与 `DELETE` 的区别**：
  - `DELETE FROM r;` 只删除表中的数据，但保留表结构（空表）。
  - `DROP TABLE r;` 删除整个表，包括其定义。
- **依赖处理**：如果其他数据库对象（如视图、存储过程、外键）依赖于该表，直接删除可能会失败。此时可以使用级联删除选项，例如在 PostgreSQL 中：`DROP TABLE r CASCADE;`，这会同时删除依赖于该表的对象。
- **注意事项**：
  - 该操作不可逆，执行前务必确认或备份。
  - 某些数据库支持 `IF EXISTS` 子句，避免表不存在时出错：`DROP TABLE IF EXISTS r;`。
  - 在大多数数据库中，`DROP TABLE` 会隐式提交当前事务，无法回滚。

---

## 第二页：`ALTER TABLE` —— 添加属性（列）

**内容原文**：
- `alter table` 命令用于向现有关系添加属性。
- 语法：`alter table r add A D`
  - 其中 `A` 是要添加的属性名，`D` 是属性的域（数据类型）。
- 示例：`alter table student add special varchar(20);`
- 新属性在所有现有元组中的值被赋为 `NULL`。

**详解**：
- **作用**：为已存在的表增加新列。
- **新列的默认值**：如果不指定默认值，所有现有行的新列值均为 `NULL`。若希望新列有非空默认值，可在定义中使用 `DEFAULT`，例如：
  ```sql
  ALTER TABLE student ADD special VARCHAR(20) DEFAULT '无';
  ```
- **添加 `NOT NULL` 列**：如果新列要求 `NOT NULL`，则不能直接添加（因为现有行无法提供非空值）。通常做法是先添加允许 `NULL` 的列，填充数据后再修改为 `NOT NULL`；或者使用 `DEFAULT` 值并同时加上 `NOT NULL`（某些数据库允许）。
- **注意事项**：
  - 添加列可能会影响应用程序，因为新列在旧记录中为 `NULL`，需要确保应用逻辑能处理。
  - 不同数据库对添加列的位置可能有限制（例如只能加到最后），但标准SQL未指定位置。

---

## 第三页：`ALTER TABLE` —— 删除属性（列）

**内容原文**：
- `alter table` 命令也可用于删除关系的属性。
- 语法：`alter table r drop A`
- 示例：`alter table student drop special;`

**详解**：
- **作用**：从表中永久移除一列，包括该列的所有数据。
- **依赖处理**：如果该列被其他对象（如约束、索引、视图）引用，删除可能会失败。许多数据库支持 `CASCADE` 选项来同时删除依赖项，例如 PostgreSQL：
  ```sql
  ALTER TABLE student DROP special CASCADE;
  ```
- **注意事项**：
  - 这是一个破坏性操作，数据无法恢复。
  - 删除列后，依赖于该列的视图、存储过程等将失效，需要相应更新。
  - 某些数据库（如 SQLite）对删除列的支持有限，可能需要重建表。

---

## 第四页：`ALTER TABLE` —— 修改数据类型

**内容原文**：
- 修改数据类型。
- 示例：`alter table student alter special varchar(30);`

**详解**：
- **作用**：改变现有列的数据类型。
- **语法差异**：不同数据库的语法差异很大，常见写法：
  - **MySQL**: `ALTER TABLE student MODIFY special VARCHAR(30);`
  - **PostgreSQL**: `ALTER TABLE student ALTER COLUMN special TYPE VARCHAR(30);`
  - **SQL Server**: `ALTER TABLE student ALTER COLUMN special VARCHAR(30);`
  - **Oracle**: `ALTER TABLE student MODIFY special VARCHAR2(30);`
- **数据转换**：修改类型时，数据库会尝试将现有数据转换为新类型。如果转换失败（如将包含非数字的字符串改为数字类型），操作会报错。
- **注意事项**：
  - 修改类型可能影响性能和存储，尤其是改变长度或精度时。
  - 如果列上有约束或索引，可能需要先删除约束，修改后再重建。
  - 建议在操作前备份数据，并确保新类型能容纳所有现有值。

## 第五页：`ALTER TABLE` —— 添加约束

**内容原文**：
- 给表添加唯一性约束：
  ```sql
  ALTER TABLE student ADD CONSTRAINT nameid UNIQUE (sname);
  ```
- 给表添加主键约束：
  ```sql
  ALTER TABLE student ADD PRIMARY KEY (sid);
  ```
- 给表添加外键约束：
  ```sql
  ALTER TABLE sc ADD CONSTRAINT sfk FOREIGN KEY (sid) REFERENCES student;
  ```
- 给表添加 `CHECK` 约束：
  ```sql
  ALTER TABLE sc ADD CONSTRAINT gradecheck CHECK (grade >= 0 AND grade <= 100);
  ```

**详解**：
- **作用**：在表创建后，通过 `ALTER TABLE` 添加各种约束，以增强数据完整性。
- **约束添加的检查**：添加约束时，数据库会验证现有数据是否满足该约束。如果不满足，操作将被拒绝。例如：
  - 添加 `UNIQUE` 约束前，需确保列（或列组合）中没有重复值。
  - 添加 `PRIMARY KEY` 前，需确保列非空且唯一。
  - 添加 `FOREIGN KEY` 前，需确保外键列的所有值在被引用表中存在（或为 `NULL`）。
  - 添加 `CHECK` 前，需确保所有行满足条件。
- **约束命名**：使用 `CONSTRAINT 约束名` 可以为约束指定一个有意义的名字，便于后续管理和删除。如果不命名，系统会自动生成一个名称。
- **注意事项**：
  - 添加约束可能会锁定表，影响并发，尤其是在大表上。
  - 添加主键会自动创建唯一索引，可能增加存储开销。
  - 对于外键约束，被引用的列必须是被引用表的主键或唯一键。
  - 某些数据库（如 MySQL 的某些存储引擎）对 `CHECK` 约束的支持有限，可能只解析而不强制执行。

## 补充知识点：`ALTER TABLE` 的其他常用操作

除了上述操作，`ALTER TABLE` 还可以用于：
- **重命名表**：`ALTER TABLE old_name RENAME TO new_name;`（标准SQL）
- **重命名列**：`ALTER TABLE r RENAME COLUMN old TO new;`（标准SQL，但各数据库实现有差异）
- **修改列的默认值**：
  ```sql
  ALTER TABLE student ALTER COLUMN special SET DEFAULT '未知';   -- PostgreSQL
  ALTER TABLE student ALTER special SET DEFAULT '未知';          -- MySQL?
  ```
- **删除列的默认值**：
  ```sql
  ALTER TABLE student ALTER COLUMN special DROP DEFAULT;
  ```
- **添加或删除 `NOT NULL` 约束**：
  ```sql
  ALTER TABLE student MODIFY sname VARCHAR(20) NOT NULL;   -- MySQL
  ALTER TABLE student ALTER COLUMN sname SET NOT NULL;     -- PostgreSQL
  ```

---

## 总结

这五页PPT涵盖了数据库模式修改的核心命令：
- **`DROP TABLE`**：彻底删除表，需谨慎使用。
- **`ALTER TABLE`**：用于动态调整表结构，包括添加/删除列、修改数据类型、添加各种约束。

在实际应用中，这些操作常用于数据库的维护和升级。由于不同数据库的语法和支持程度存在差异，执行前务必查阅对应数据库的官方文档，并考虑对现有数据和应用的影响。对于生产环境，建议在维护窗口进行，并提前备份数据。
