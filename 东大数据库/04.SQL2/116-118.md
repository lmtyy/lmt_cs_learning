## 第1页：自然连接（NATURAL JOIN）的风险与USING子句

### 1.1 自然连接的问题

**自然连接**是一种隐式连接，它会自动将两个表中所有同名的列作为连接条件。例如：
```sql
SELECT * FROM loan NATURAL JOIN borrower;
```
- 如果`loan`和`borrower`都有`loan_number`列，则自然连接会自动基于`loan_number`进行等值连接。
- 结果中同名列只出现一次。

**为什么说自然连接是危险的？**
- **隐式依赖表结构**：自然连接的行为完全取决于当前两个表的列名。如果将来在其中一个表中添加了新列，并且恰好与另一个表中的某列同名，那么这个新列就会**自动成为连接条件的一部分**，从而改变查询的语义。
- **例子**：假设`loan`表新增一列`branch_name`，而`borrower`表原本也有一列`branch_name`（例如表示借款人所在分行）。那么原本只基于`loan_number`的自然连接，会突然变成同时基于`loan_number`和`branch_name`的连接。这可能导致结果集变小甚至为空，而查询代码本身并没有任何修改。这种“静默改变”是维护中的大忌。

**结论**：在编写可维护、健壮的SQL时，**应避免使用NATURAL JOIN**。

### 1.2 更安全的替代：JOIN ... USING

`JOIN ... USING` 显式指定连接所基于的列，既保持了简洁性，又避免了隐式风险。

```sql
SELECT *
FROM loan JOIN borrower USING (loan_number);
```
- `USING(loan_number)` 明确告诉数据库：只使用`loan_number`这一列作为连接条件。
- 结果中`loan_number`列只出现一次，类似于自然连接，但连接列是显式指定的，不会因表结构变化而意外增加连接条件。

**对比**：
- `NATURAL JOIN`：隐式、不可控。
- `JOIN ... USING`：显式、可控，且代码自文档化。
- `JOIN ... ON`：最通用，可以指定任意复杂条件，但稍显冗长。`USING`是`ON`的一种简洁特例，适用于连接列同名且需要等值连接的情况。

## 第2页：全外连接（FULL OUTER JOIN）的应用示例

### 2.1 全外连接示例

```sql
SELECT *
FROM loan FULL OUTER JOIN borrower USING (loan_number);
```
**结果表**：
| loan_number | branch_name | amount | customer_name |
| ----------- | ----------- | ------ | ------------- |
| L-170       | Downtown    | 3000   | Jones         |
| L-230       | Redwood     | 4000   | Smith         |
| L-260       | Perryridge  | 1700   | null          |
| L-155       | null        | null   | Hayes         |

**解析**：
- 全外连接保留左表（`loan`）和右表（`borrower`）的所有行。
- 匹配的行（L-170, L-230）正常显示。
- 左表有但右表无（L-260）：右表列（`customer_name`）填`NULL`。
- 右表有但左表无（L-155）：左表列（`branch_name`, `amount`）填`NULL`。
- 由于使用了`USING(loan_number)`，同名列`loan_number`只出现一次，且包含了所有出现的贷款号。

### 2.2 业务问题：找出只有账户或只有贷款（但不同时拥有）的客户

原PPT中的查询：
```sql
SELECT customer_name
FROM (depositor NATURAL FULL OUTER JOIN borrower)
WHERE account_number IS NULL OR loan_number IS NULL;
```
**注意**：这里用了`NATURAL FULL OUTER JOIN`，但根据第一页的建议，更安全的写法应该是使用`USING`明确连接列，假设两表都有`customer_name`列（通常如此）：
```sql
SELECT customer_name
FROM depositor FULL OUTER JOIN borrower USING (customer_name)
WHERE account_number IS NULL OR loan_number IS NULL;
```
**逻辑解析**：
- 全外连接的结果包含所有客户（只要在任一表出现）。
- 对于既有账户又有贷款的客户，`account_number`（来自`depositor`）和`loan_number`（来自`borrower`）都不为`NULL`。
- 对于只有账户的客户，`loan_number`为`NULL`（因为`borrower`侧无匹配）。
- 对于只有贷款的客户，`account_number`为`NULL`（因为`depositor`侧无匹配）。
- 因此，条件 `account_number IS NULL OR loan_number IS NULL` 正好选出那些**不是两边都有的**客户，即要么只有账户，要么只有贷款。

**这个技巧的巧妙之处**：
- 用全外连接一次性获取所有客户及其两边的情况。
- 通过检查`NULL`来区分客户属于哪一类，避免了使用`NOT IN`或`EXISTS`的复杂逻辑，代码简洁。

## 第3页：多表连接的语法

### 3.1 连接三个表

```sql
FROM (sc JOIN student ON student.sid = sc.sid) JOIN course ON course.cid = sc.cid
```
或者更清晰的写法（去掉多余括号）：
```sql
FROM sc 
JOIN student ON student.sid = sc.sid
JOIN course ON course.cid = sc.cid
```
- 第一个`JOIN`将`sc`与`student`连接，得到中间结果（包含学生信息）。
- 然后第二个`JOIN`将这个中间结果与`course`连接，基于`sc.cid`与`course.cid`。
- 最终结果包含`sc`、`student`、`course`三张表的所有列（注意列名重复时需要处理）。

### 3.2 连接四个表

```sql
FROM ((表1 INNER JOIN 表2 ON 表1.字段 = 表2.字段)
      JOIN 表3 ON 表1.字段 = 表3.字段) 
      JOIN 表4 ON 表1.字段 = 表4.字段
```
- 这种写法通过嵌套括号明确了连接顺序，但通常数据库优化器会重新排序，所以括号更多是语法要求。
- 实际上，可以连续使用`JOIN`，数据库会按照从左到右的顺序处理（但逻辑上连接是可交换的）。更常见的写法是：
```sql
FROM 表1
JOIN 表2 ON 表1.字段 = 表2.字段
JOIN 表3 ON 表1.字段 = 表3.字段
JOIN 表4 ON 表1.字段 = 表4.字段
```
- 注意：连接条件不一定非要用同一个表的字段，可以根据业务需要指定任意连接条件。

### 3.3 多表连接的关键点
- 每个`JOIN`都需要一个`ON`子句（或`USING`）指定连接条件。
- 连接顺序会影响中间结果的大小，但数据库优化器通常会选择最优执行计划。
- 可以使用括号强制连接顺序，但通常不必要。
- 多表连接时，如果表中有同名列，需要通过表名前缀或别名来区分。

## 总结

| 知识点             | 核心内容                                             | 注意事项                           |
| ------------------ | ---------------------------------------------------- | ---------------------------------- |
| **自然连接的风险** | 隐式基于所有同名列连接，表结构变化会静默改变查询语义 | **避免使用**，改用 `USING` 或 `ON` |
| **USING子句**      | 显式指定连接列，结果中该列只出现一次                 | 比自然连接安全，比ON简洁           |
| **全外连接**       | 返回左右表所有行，无匹配侧填NULL                     | 用于需要保留所有记录的场合         |
| **利用NULL筛选**   | 通过检查某侧是否为NULL，可选出“只在一方存在”的记录   | 注意连接列的选择，确保业务语义正确 |
| **多表连接**       | 连续使用JOIN，每个JOIN后跟ON/USING子句               | 可连续连接多个表，注意连接条件     |

掌握这些连接的高级用法，能让你在处理复杂数据需求时更加得心应手，同时避免常见的陷阱。
