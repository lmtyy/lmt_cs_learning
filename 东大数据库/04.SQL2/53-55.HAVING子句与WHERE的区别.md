## 第1页：HAVING 子句的基本用法

**内容回顾**  
- 表：`account(branch-name, account-number, balance)`
- 需求：找出所有平均账户余额大于 700 的支行名称。
- SQL语句：
  ```sql
  SELECT branch_name, AVG(balance)
  FROM account
  GROUP BY branch_name
  HAVING AVG(balance) > 700;
  ```
- 说明：`HAVING` 子句中的条件在分组形成之后应用于每个组。

**详细讲解**  
- **`HAVING` 的作用**：用于对分组后的结果进行过滤，只保留满足条件的组。它通常与 `GROUP BY` 一起使用，但也可以单独使用（此时整个表被看作一个组）。
- **执行逻辑**：
  1. 先根据 `GROUP BY branch_name` 将表按支行分组。
  2. 对每个分组计算 `AVG(balance)`。
  3. 然后应用 `HAVING AVG(balance) > 700`，只保留平均余额大于 700 的分组。
  4. 最后输出这些组的 `branch_name` 和对应的平均余额。
- **为什么这里不能用 `WHERE` 代替 `HAVING`？**  
  因为 `WHERE` 是在分组前对单个行进行过滤，而 `AVG(balance)` 是一个聚合函数，需要对一组行进行计算后才能得到。在 `WHERE` 执行时，分组尚未形成，因此无法使用聚合函数。这正是 `HAVING` 存在的意义。

## 第2页：HAVING 与 WHERE 的区别

**内容回顾**  
- 作用对象不同：
  - `WHERE` 子句作用于基表或视图，从**行**中筛选满足条件的元组。
  - `HAVING` 短语作用于**组**，从分组中筛选满足条件的组。
- `WHERE` 子句中不能出现聚集函数，因为它是针对每一行进行过滤，而不是针对集合。

**详细讲解**  
### 区别对比表
| 特性                   | WHERE 子句                                 | HAVING 子句                                              |
| ---------------------- | ------------------------------------------ | -------------------------------------------------------- |
| **作用对象**           | 行（元组）                                 | 组（由 GROUP BY 形成）                                   |
| **执行时机**           | 在分组之前，数据源筛选阶段                 | 在分组之后，对组进行过滤                                 |
| **能否使用聚合函数**   | **不能**（因为聚合需要多行，此时尚未分组） | **可以**（直接在组上计算聚合）                           |
| **与 GROUP BY 的关系** | 可以独立使用，不依赖 GROUP BY              | 通常与 GROUP BY 配合，但也可单独使用（此时整个表为一组） |
| **示例**               | `WHERE balance > 500`                      | `HAVING AVG(balance) > 700`                              |

### 为什么 WHERE 不能使用聚合函数？
- 聚合函数（如 `AVG`、`SUM`）需要对一组值进行计算。在 `WHERE` 执行阶段，数据还没有被分组，因此无法确定“组”的范围。如果允许在 `WHERE` 中使用聚合函数，那么它是对整个表做聚合还是对某一部分？语义不清晰。所以SQL设计者将聚合过滤的任务交给了 `HAVING`。

### 一个容易混淆的例子
假设我们想查询“平均余额大于700的支行”，如果错误地写成：
```sql
SELECT branch_name, AVG(balance)
FROM account
WHERE AVG(balance) > 700   -- 错误！
GROUP BY branch_name;
```
数据库会报错，因为在执行 `WHERE` 时，`AVG(balance)` 根本还没有被计算出来。

## 第3页：SELECT 的执行流程与 HAVING 的位置

**内容回顾**  
- 标题为“SELECT的执行(续) HAVING”，并附有一些表格，可能试图展示分组聚合后的中间结果和最终结果，但由于排版混乱，不易阅读。不过核心意图是强调 `HAVING` 在分组后执行。

**详细讲解**  
### SQL 查询的逻辑执行顺序
为了正确理解 `HAVING` 的位置，必须掌握 SQL 语句的逻辑执行顺序（注意，这不是数据库实际的物理执行顺序，而是逻辑上的步骤）：
1. **FROM**：确定数据源（表、视图、子查询）。
2. **WHERE**：对数据源中的行进行过滤（不能使用聚合）。
3. **GROUP BY**：将过滤后的行分组。
4. **HAVING**：对分组后的组进行过滤（可以使用聚合）。
5. **SELECT**：选择需要输出的列，计算表达式，确定别名。
6. **ORDER BY**：对结果集排序。
7. **LIMIT/OFFSET**：限制输出行数。

所以 `HAVING` 是在分组之后、选择列之前执行的。

### 用 account 表的数据举例
假设 `account` 表数据如下：

| branch_name | account_number | balance |
| ----------- | -------------- | ------- |
| Perryridge  | a-102          | 400     |
| Perryridge  | a-201          | 900     |
| Brighton    | a-217          | 750     |
| Brighton    | a-215          | 750     |
| Redwood     | a-222          | 700     |

执行查询：
```sql
SELECT branch_name, AVG(balance)
FROM account
GROUP BY branch_name
HAVING AVG(balance) > 700;
```

**执行过程**：
1. **FROM**：读取 `account` 表。
2. **WHERE**：无，所以所有行进入下一步。
3. **GROUP BY branch_name**：分成三组：
   - Perryridge 组：balance = 400, 900
   - Brighton 组：balance = 750, 750
   - Redwood 组：balance = 700
4. **HAVING AVG(balance) > 700**：
   - Perryridge 组：AVG = (400+900)/2 = 650，不大于700 → 过滤掉。
   - Brighton 组：AVG = 750，大于700 → 保留。
   - Redwood 组：AVG = 700，不大于700 → 过滤掉。
5. **SELECT**：输出保留的组：Brighton 和它的平均余额 750。
6. **ORDER BY**（无），直接返回结果。

最终结果：
| branch_name | AVG(balance) |
| ----------- | ------------ |
| Brighton    | 750          |

### 如果 HAVING 单独使用（无 GROUP BY）
例如：
```sql
SELECT AVG(balance) FROM account HAVING AVG(balance) > 500;
```
此时整个表作为一个组，`HAVING` 判断平均余额是否大于500。如果满足，返回一行结果；否则返回空。

## 补充：HAVING 与 WHERE 可以同时存在
一个完整的查询可以同时包含 `WHERE` 和 `HAVING`，它们分工明确：
- `WHERE` 在分组前过滤行。
- `HAVING` 在分组后过滤组。

例如：查询“在 Perryridge 支行中，平均余额大于 700 的账户类型”（假设有账户类型列），可以先用 `WHERE` 选出 Perryridge 支行的账户，再分组并 `HAVING`。

---

## 总结
- **`HAVING`** 用于对分组后的结果进行过滤，常与 `GROUP BY` 搭配，可以使用聚合函数。
- **`WHERE`** 用于对分组前的行进行过滤，**不能**使用聚合函数。
- 牢记 SQL 逻辑执行顺序：`WHERE` 先于 `GROUP BY`，`HAVING` 后于 `GROUP BY`。
- 理解这两者的区别，能帮助你避免常见的语法错误，写出高效准确的查询。
