这几页PPT主要讲解了SQL中的嵌套子查询，重点区分了**非相关子查询**和**相关子查询**两种类型，并通过示例展示了它们的用法。下面我将逐一详细解析每一页的内容，并补充关键概念和注意事项，帮助初学者彻底理解。

## 第1页：非相关子查询（Non Correlated Nested Queries）

**定义**  
- 非相关子查询是指**内层子查询完全独立于外层查询**，即子查询不引用外层表的任何列，可以单独运行并返回结果。
- 执行顺序：**由里向外逐层处理**。先执行最内层的子查询，将得到的结果集（一个临时表或值集合）传递给外层查询，外层查询利用这个结果进行过滤或计算。
- 子查询的结果用于建立父查询的查找条件（例如用在 `WHERE` 子句的 `IN`、比较运算符、`EXISTS` 等中）。

### 特点
- 子查询只执行**一次**，结果被缓存，外层查询直接使用。
- 通常效率较高，因为不需要为外层每一行重复执行子查询。
- 常见于 `IN`、`> ANY`、`> ALL` 等操作中。

### 示例
假设我们想找出所有选修了课程的学生姓名（假设学生表为 `students`，选课表为 `sc`）。使用非相关子查询：
```sql
SELECT sname
FROM students
WHERE sid IN (SELECT sid FROM sc);
```
- 子查询 `(SELECT sid FROM sc)` 独立执行，返回所有有选课记录的学生ID列表。
- 外层查询再用这些ID去筛选学生表。

这个过程就是“由里向外”，子查询先求出结果集。

## 第2页：通过 `IN` 实现查询（但并非相关子查询）

**内容回顾**  
这一页给出了两个查询，目的都是“查询选修课程的学生姓名”：
1. 连接写法：
   ```sql
   SELECT DISTINCT sname
   FROM students, sc
   WHERE students.sid = sc.sid;
   ```
2. 子查询写法：
   ```sql
   SELECT sname
   FROM students
   WHERE sid IN (SELECT sid FROM sc);
   ```

**详细讲解**  
- 这两个查询在逻辑上等价，都找出有选课记录的学生。
- 注意：这里的子查询 `(SELECT sid FROM sc)` 是**非相关**的，因为它不依赖于外层 `students` 表的任何列，可以独立运行。
- 连接写法可能会因为一对多关系产生重复行，因此需要 `DISTINCT`；而 `IN` 子查询自动去重（因为 `IN` 只关心是否存在，不会产生重复）。
- 这一页的标题可能是想引出相关子查询，但这两个例子都是非相关的，需要区分。

## 第3页：相关子查询（Correlated Nested Queries）示例

**内容回顾**  
使用 `EXISTS` 实现相同需求：
```sql
SELECT sname
FROM students
WHERE EXISTS (SELECT * FROM sc WHERE sid = students.sid);
```
并附带了示例数据表格：

| students表 |        |      | sc表  |        |      |
| ---------- | ------ | ---- | ----- | ------ | ---- |
| 学号       | 姓名   | 性别 | 学号  | 课程号 | 成绩 |
| 08001      | 张小红 | 女   | 08001 | 001    | 90   |
| 08002      | 王伟   | 男   | 08001 | 002    | 80   |
| 08003      | 小刚   | 男   | 08003 | 002    | 60   |

**详细讲解**  

### 什么是相关子查询？
- 相关子查询是指**内层子查询引用了外层查询的列**，因此子查询的执行依赖于外层当前行的值。
- 对于外层表的每一行，子查询都会执行一次，用当前行的值代入子查询进行判断。

### 执行过程
1. 外层查询从 `students` 表取第一行（例如学号 08001，张小红）。
2. 执行子查询：`SELECT * FROM sc WHERE sid = 08001`，检查 `sc` 表中是否有该学生的记录。如果有，则子查询返回至少一行，`EXISTS` 为真，该学生被选中。
3. 然后外层取下一行（08002，王伟），子查询再次执行：`WHERE sid = 08002`，此时 `sc` 表中没有学号 08002 的记录，子查询返回空，`EXISTS` 为假，王伟不被选中。
4. 继续处理下一行（08003，小刚），子查询执行并找到记录，小刚被选中。
5. 最终结果返回张小红和小刚。

### 与 `IN` 的对比
- `IN` 子查询是非相关的，只执行一次；而 `EXISTS` 是相关的，执行多次（外层表行数）。
- 但现代数据库优化器可能会将 `IN` 也转换为相关或半连接，实际性能需看执行计划。
- 关键区别：`EXISTS` 可以处理更复杂的逻辑，例如需要引用外层多列的情况；而 `IN` 通常用于简单成员测试。

### 关于 `EXISTS` 的 `SELECT *`
- 在 `EXISTS` 子查询中，`SELECT *` 或 `SELECT 1` 都一样，因为 `EXISTS` 只关心是否有行返回，不关心具体列。通常写 `SELECT 1` 更清晰。

## 总结：非相关 vs 相关子查询

| 特性       | 非相关子查询                   | 相关子查询                                      |
| ---------- | ------------------------------ | ----------------------------------------------- |
| 独立性     | 子查询独立于外层，可单独运行   | 子查询依赖外层当前行的值                        |
| 执行次数   | 一次，结果缓存                 | 外层每一行执行一次                              |
| 性能       | 通常较快                       | 可能较慢，但优化器可优化                        |
| 常见运算符 | `IN`、`> ANY`、`> ALL`、比较符 | `EXISTS`、`NOT EXISTS`、`IN` 有时也可优化为相关 |
| 典型场景   | 集合成员测试，值比较           | 存在性检查，复杂条件过滤                        |

### 如何选择？
- 如果子查询结果集较小且不需要外层列参与，优先用非相关子查询（如 `IN`）。
- 如果需要外层列参与判断（例如“找出每个部门中工资高于部门平均的员工”），则必须用相关子查询或连接。
- 当子查询可能包含 `NULL` 时，`EXISTS` 比 `IN` 更安全（因为 `IN` 遇到 `NULL` 可能导致意外结果）。

理解这两种子查询的区别，是掌握复杂SQL查询的关键一步。
