### 第一页：Referencing Constraint —— 参照约束的基本规则

这一页用集合包含的直观方式，展示了参照完整性约束的核心规则。

**原文内容：**
> - Only values occurring in the primary key attribute of the **referenced relation** may occur in the foreign key attribute of the **referencing relation**
>
> （配有表格图示，展示 `depositor` 表的外键值集合是 `account` 表主键值集合的子集）

**详细知识点讲解：**

**1. 参照完整性规则**
- **原文翻译**：只有**出现在被参照关系主键中的值**，才能出现在**参照关系的外键**中。
- **通俗表述**：外键的取值空间，是被参照表主键取值空间的**子集**。
- **图示含义**：PPT 中画出了 `depositor` 表的外键列与 `account` 表的主键列，并用 "Is subset of..." 标注。这表明：
    \[
    \pi_{\text{account\_number}}(\text{depositor}) \subseteq \pi_{\text{account\_number}}(\text{account})
    \]
    其中 \( \pi \) 表示投影操作，即提取该列的所有值。

**2. 表格数据验证**
**被参照表 `account` 的主键值集合：**
```
{A-101, A-102, A-201, A-215, A-217, A-222, A-305}
```

**参照表 `depositor` 的外键值集合（示例中出现的）：**
```
{A-102, A-101, A-215, A-222, A-302}
```

**对比分析：**
| 值        | 是否在 account 主键中？ | 合法性       |
| :-------- | :---------------------- | :----------- |
| A-102     | ✅ 存在                  | 合法         |
| A-101     | ✅ 存在                  | 合法         |
| A-215     | ✅ 存在                  | 合法         |
| A-222     | ✅ 存在                  | 合法         |
| **A-302** | ❌ **不存在**            | **违反约束** |

- `A-302` 不在 `account` 表的主键集合中，因此它作为外键值出现在 `depositor` 表中是**非法**的。数据库系统会拒绝这样的插入或更新操作。

**3. 集合视角的深刻意义**
- 将外键关系视为**子集关系**，是关系代数中处理参照完整性的数学基础。
- 这一定义也隐含了：被参照表的主键值可以**不被任何外键引用**（即 `account` 中的 `A-201`、`A-217`、`A-305` 没有出现在 `depositor` 中，这是完全合法的，表示这些账户当前没有关联的存款人）。
- 但反过来，外键值**绝不能**超出主键值的范围。

**4. 与 SQL 中 FOREIGN KEY 约束的对应**
这一页描述的规则，在 SQL 中通过如下语句实现：
```sql
CREATE TABLE depositor (
    customer_name VARCHAR(50),
    account_number VARCHAR(10),
    FOREIGN KEY (account_number) REFERENCES account(account_number)
);
```
任何试图在 `account_number` 列中插入一个 `account` 表中不存在的值的操作，都会被数据库拒绝。

### 第二页：Referencing Constraint —— 外键值的两种合法情况

这一页进一步明确了外键可以取值的两种具体情况，并特别指出了 `NULL` 的使用限制。

**原文内容：**
> The value in the foreign key column (or columns) FK of the referencing relation \( R_1 \) can be either:
>
> (1) a value of an existing primary key value of the corresponding primary key PK in the referenced relation \( R_2 \), or..
>
> (2) a null.
>
> In case (2), the FK in \( R_1 \) should not be a part of its own primary key.

**详细知识点讲解：**

**1. 外键值的两种情况**
根据参照完整性约束，外键的取值只有两种**合法可能**：

- **情况一：取被参照表主键中已存在的值**
    - 这是最常见的情况，表示**确切的引用关系**。
    - 例如：`depositor` 表中的 `account_number = 'A-101'`，说明该存款记录确实关联到了一个真实的账户。
    - 这种值确保了引用的**有效性**和**一致性**。

- **情况二：取空值 `NULL`**
    - 表示**暂时没有引用**或**引用未知**。
    - 例如：一个新客户刚刚登记，但还没有开设任何账户。此时如果强行要求外键非空，就无法录入该客户。
    - `NULL` 在这里的语义是“不存在关联”或“关联暂缺”。

**2. 为什么允许 NULL？**
- 允许外键为 `NULL` 是对现实世界**部分参与**语义的支持。
- 在 E-R 模型中，实体参与联系可以是**全部参与**（每个实体都必须参与联系）或**部分参与**（某些实体可以不参与）。
- 允许外键为 `NULL` 正是**部分参与**在关系模型中的实现方式。

**3. 外键为 NULL 的重要限制**
- **原文**：*In case (2), the FK in \( R_1 \) should not be a part of its own primary key.*
- **解释**：如果外键列同时也是**该表主键的一部分**，则它**不允许**为 `NULL`。
- **原因**：主键的定义要求其值必须**唯一且非空**。这是实体完整性的要求。如果外键是主键的组成部分，赋予它 `NULL` 将直接违反主键约束。
- **例子分析**：
    - `depositor` 表的主键通常是 `(customer_name, account_number)` 的组合。
    - 如果 `account_number` 是主键的一部分，那么它**不能**为 `NULL`。这意味着 `depositor` 表中的每一条记录都**必须**同时指定客户和账户，两者缺一不可。
    - 这正好对应了现实语义：一条存款记录必须同时明确“谁”存了“哪个账户”，缺少任何一个信息，这条记录就没有意义。

**4. 两种情况的对照表**

| 情况       | 外键值                     | 语义                 | 前提条件                   |
| :--------- | :------------------------- | :------------------- | :------------------------- |
| **情况一** | 被参照表主键中的某个具体值 | 存在有效的引用       | 无特殊限制                 |
| **情况二** | `NULL`                     | 不存在引用或引用未知 | 外键不能是自身主键的一部分 |

### 第三页：Referencing Constraint —— 中文示例

这一页用一个贴近中文教学环境的例子，具体展示了外键为 `NULL` 的实际应用场景。

**原文内容：**
> - 学生（学号，姓名，性别，专业号）
> - 专业（专业号，专业名）
> - 学生的专业号是外码
>   - 空值。表示尚未给学生分配专业

**详细知识点讲解：**

**1. 关系模式定义**
- **学生表**：`学生(学号, 姓名, 性别, 专业号)`
    - 主键：`学号`
    - 外键：`专业号`，参照 `专业` 表的主键。
- **专业表**：`专业(专业号, 专业名)`
    - 主键：`专业号`

**2. 外键 `专业号` 的取值分析**
根据第二页的规则，`专业号` 作为外键，其合法取值有两种：

- **情况一：取某个具体的专业号**
    - 例如 `'CS01'`（计算机科学与技术）。
    - 语义：该学生已被分配到这个专业。

- **情况二：取空值 `NULL`**
    - 语义：**尚未给学生分配专业**。
    - 这在高校管理系统中非常常见：
        - 大一新生刚入学时，尚未进行专业分流，此时 `专业号` 就是 `NULL`。
        - 或者某些院校允许学生在前两年不分专业，此时外键自然为空。

**3. 为什么这里外键可以取 NULL？**
- 在 `学生` 表中，外键 `专业号` **不是**主键的一部分。主键只有一个属性 `学号`。
- 因此，`专业号` 为 `NULL` 不会违反主键的非空约束。
- 这是符合第二页规则 (2) 的标准应用场景。

**4. 反面对比：如果外键是主键的一部分**
假设有另一个关系模式：
- **选课表**：`选课(学号, 课程号, 成绩)`
    - 主键：`(学号, 课程号)`
    - 外键：`学号` 参照 `学生` 表，`课程号` 参照 `课程` 表。
- 在这个设计中，`学号` 和 `课程号` 都**不能**为 `NULL`，因为它们共同构成了主键。
- 你不可能插入一条“不知道是谁选了某门课”或“某学生选了一门不知道是什么的课”的记录。这符合选课场景的业务逻辑：选课记录必须明确学生和课程。

**5. 中文示例的教学价值**
- 这一页用简洁的中文关系模式，完整复现了外键与 NULL 的理论规则。
- 它强调了数据库设计必须紧密贴合**业务语义**：是否允许外键为空，取决于现实世界中该实体是否必须参与该联系。
