这是一份为您整理的数据库关系（Cardinality）系统性笔记。这部分内容是数据库设计的灵魂，决定了表与表之间如何正确关联。

我们将涵盖三种核心关系：**一对一 (1:1)**、**一对多 (1:N)**、**多对多 (N:M)**。

------

### 一、 一对一关系 (One-to-One, 1:1)

#### 1. 定义

表 A 中的一条记录，**至多**对应表 B 中的一条记录；反之亦然。这种关系在实际业务中不如“一对多”常见，但在特定场景下非常重要。

#### 2. 典型场景

- **垂直分表（性能优化/安全）**：将一张大表的“常用字段”和“隐私/大字段”拆开。
  - *例子*：`用户基础信息表` (User) 和 `用户详细档案表` (User_Profile)。
- **专有扩展**：
  - *例子*：`员工表` (Employee) 和 `登录账户表` (Account)。不是所有员工都有登录系统的权限，但一个账号只能属于一个员工。

#### 3. 实现方式 (两种核心方案)

**方案 A：外键 + 唯一约束 (Foreign Key with UNIQUE)**

这是最灵活的方式。在**任意一方**（通常是“从表”）添加外键，并给这个外键加上 `UNIQUE` 约束。

- **表结构设计**：
  - **主表 (Employee)**: `ID (PK)`, `Name`
  - **从表 (Account)**: `ID (PK)`, `Password`, **`Emp_ID (FK, UNIQUE)`**
- **关键点**：
  - `Emp_ID` 是外键，保证了参照完整性。
  - **`UNIQUE` 约束** 保证了 `Emp_ID` 不能重复，从而强制了 1:1 关系（否则就变成 1:N 了）。

**方案 B：共享主键 (Shared Primary Key)**

这是最严格的方式（我们刚讨论过）。从表的主键，同时也是指向主表的外键。

- **表结构设计**：
  - **主表 (Employee)**: `Emp_ID (PK)`, `Name`
  - **从表 (Account)**: **`Emp_ID (PK & FK)`**, `Password`
- **关键点**：
  - 两个表的主键值完全一致。
  - 这种方式表示两者是**强绑定**的（同生共死）。

------

### 二、 一对多关系 (One-to-Many, 1:N)

#### 1. 定义

表 A（一端）的一条记录，可以对应表 B（多端）的多条记录；但表 B 的一条记录，只能对应表 A 的一条记录。

**这是数据库中最常见的关系，占比 90% 以上。**

#### 2. 典型场景

- **班级与学生**：一个班级有多个学生，一个学生只属于一个班级。
- **用户与订单**：一个用户可以下多个订单，一个订单只属于一个用户。

#### 3. 实现方式

**黄金法则：在“多”的一方加外键。**

- **表结构设计**：
  - **一端 (Class)**: `Class_ID (PK)`, `Class_Name`
  - **多端 (Student)**: `Student_ID (PK)`, `Name`, **`Class_ID (FK)`**
- **关键点**：
  - 千万不要在“一端”存“多端”的 ID（比如在班级表里存一个学生 ID 列表），这违反了第一范式。
  - 外键必须建在“多”的那张表里，指向“一”的那张表的主键。
  - 这里的 `Class_ID` **不需要** `UNIQUE` 约束（因为多个学生可以在同一个班）。

------

### 三、 多对多关系 (Many-to-Many, N:M)

#### 1. 定义

表 A 的一条记录对应表 B 的多条记录；表 B 的一条记录也对应表 A 的多条记录。

**注意：关系数据库无法直接在两个表之间实现 N:M，必须引入第三张表。**

#### 2. 典型场景

- **学生与课程**：一个学生选多门课，一门课被多个学生选。
- **文章与标签**：一篇文章有多个标签，一个标签下有多篇文章。
- **角色与权限**：一个角色（如管理员）有多个权限，一个权限可赋予多个角色。

#### 3. 实现方式

**引入“中间表” (Junction Table / Association Table)**

这个中间表专门用来存储两者的关联关系。

- **表结构设计**：
  - **表 A (Student)**: `Student_ID (PK)`, `Name`
  - **表 B (Course)**: `Course_ID (PK)`, `Title`
  - **中间表 (Enrollment)**:
    - **`Student_ID (FK)`**：指向学生表。
    - **`Course_ID (FK)`**：指向课程表。
    - **联合主键 (PK)**：**`(Student_ID, Course_ID)`**
- **关键点**：
  - 中间表把一个 N:M 关系拆成了**两个 1:N 关系**。
    - Student -> Enrollment 是 1:N
    - Course -> Enrollment 是 1:N
  - **联合主键**：中间表通常使用这两个外键的组合作为主键，以防止重复关联（同一个学生不能重复选同一门课）。
  - **业务属性**：中间表不仅可以存 ID，还可以存关系的属性（例如：这门课的“考试成绩”、选课的“时间”）。

------

### 四、 自关联 (Self-Referencing Relationship)

这是上述三种关系的特殊变体，参与关系的只有**一张表**。

#### 1. 一对多自关联 (Hierarchy / Tree)

- **场景**：员工与直属上司（上司也是员工）。
- **设计**：
  - `Employee` 表：`Emp_ID (PK)`, `Name`, **`Manager_ID (FK)`**
  - `Manager_ID` 是外键，指向**本表**的 `Emp_ID`。

#### 2. 多对多自关联 (Network / Graph)

- **场景**：社交网络中的“关注”关系（A 关注 B，B 也可以关注 A，或者 C）。
- **设计**：
  - 需要引入**中间表** `Follows`。
  - `Follower_ID (FK)` 指向 User 表。
  - `Followee_ID (FK)` 指向 User 表。

------

### 总结笔记图表

| **关系类型** | **符号** | **核心口诀**       | **典型设计方案**                    | **约束特点**                     |
| ------------ | -------- | ------------------ | ----------------------------------- | -------------------------------- |
| **一对一**   | **1:1**  | 夫唱妇随，独一无二 | **外键 + 唯一索引** 或 **共享主键** | FK 必须有 `UNIQUE` 约束          |
| **一对多**   | **1:N**  | 一个领导，多个下属 | **外键建在“多”端**                  | FK 允许重复                      |
| **多对多**   | **N:M**  | 双向选择，必须拆分 | **引入第三张“中间表”**              | 中间表有两个 FK，通常组成联合 PK |

记笔记时，请重点关注**“外键放在哪里”**以及**“是否需要唯一约束”**，这是考试和面试最常考察的细节。
