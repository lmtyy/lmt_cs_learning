### 第一页：Schema Diagram —— 模式图

这一页展示了一个完整的银行数据库**模式图**，用图形化的方式直观呈现了多张表之间的**主键 - 外键引用关系**。

**原文内容（图结构解读）：**

PPT 中展示了六张相互关联的表，每张表用矩形框表示，框内上方是表名，下方是该表的属性列表。

**表结构整理：**

| 表名        | 属性列表                                            | 主键（按惯例推断）                |
| :---------- | :-------------------------------------------------- | :-------------------------------- |
| `branch`    | `branch_name`, `branch_city`, `assets`              | `branch_name`                     |
| `account`   | `account_number`, `branch_name`, `balance`          | `account_number`                  |
| `depositor` | `customer_name`, `account_number`                   | `(customer_name, account_number)` |
| `customer`  | `customer_name`, `customer_street`, `customer_city` | `customer_name`                   |
| `loan`      | `loan_number`, `branch_name`, `amount`              | `loan_number`                     |
| `borrower`  | `customer_name`, `loan_number`                      | `(customer_name, loan_number)`    |

**详细知识点讲解：**

**1. 什么是模式图？**
- **定义**：模式图是一种**可视化工具**，用于描述关系数据库的**逻辑结构**。
- **组成部分**：
    - 每个**矩形框**代表一个**关系**。
    - 矩形框上方是**关系名**。
    - 矩形框内部列出该关系的**属性名**。
    - 通常用**下划线**标注主键（本页未显式标注，但可根据语义推断）。
    - 用**箭头**或**连线**表示外键与主键之间的参照关系。

**2. 六张表的语义拆解**

**实体表（存储实体自身信息）：**
- **`branch`**：银行分行信息。主键是 `branch_name`（分行名）。
- **`customer`**：客户信息。主键是 `customer_name`（客户姓名，假设唯一）。
- **`account`**：账户信息。主键是 `account_number`（账号）。包含外键 `branch_name`，参照 `branch` 表。
- **`loan`**：贷款信息。主键是 `loan_number`（贷款编号）。包含外键 `branch_name`，参照 `branch` 表。

**联系表（存储实体之间的关联）：**
- **`depositor`**：存款联系。连接 `customer` 和 `account`。
    - 外键 `customer_name` 参照 `customer` 表。
    - 外键 `account_number` 参照 `account` 表。
    - 主键通常是两者的组合：`(customer_name, account_number)`。
- **`borrower`**：借款联系。连接 `customer` 和 `loan`。
    - 外键 `customer_name` 参照 `customer` 表。
    - 外键 `loan_number` 参照 `loan` 表。
    - 主键通常是两者的组合：`(customer_name, loan_number)`。

**3. 外键参照关系汇总**

| 参照关系（子表） | 外键属性         | 被参照关系（父表） | 被参照主键       |
| :--------------- | :--------------- | :----------------- | :--------------- |
| `account`        | `branch_name`    | `branch`           | `branch_name`    |
| `loan`           | `branch_name`    | `branch`           | `branch_name`    |
| `depositor`      | `customer_name`  | `customer`         | `customer_name`  |
| `depositor`      | `account_number` | `account`          | `account_number` |
| `borrower`       | `customer_name`  | `customer`         | `customer_name`  |
| `borrower`       | `loan_number`    | `loan`             | `loan_number`    |

**4. 模式图的价值**
- **全局视图**：一眼看清整个数据库由哪些表组成，以及它们之间的**关联路径**。
- **参照完整性可视化**：箭头明确指出了数据依赖的方向。
- **规范化检验**：通过观察外键分布，可以初步判断设计是否符合规范化要求。
- **开发文档**：是数据库设计文档的重要组成部分，供开发人员和 DBA 参考。

**5. 模式的逻辑解读示例**
- 如果想查询“客户 Hayes 在 Downtown 分行的所有账户余额”：
    - 从 `customer` 找到 Hayes。
    - 通过 `depositor` 找到 Hayes 拥有的所有 `account_number`。
    - 在 `account` 表中筛选 `branch_name = 'Downtown'` 的账户。
    - 取出 `balance`。
- 模式图清晰地展示了这条查询路径：`customer` → `depositor` → `account` → `branch`。

### 第二页：Integrity Constraint —— 完整性约束

这一页是对关系模型中三类**完整性约束**的高度概括总结，是整个关系模型理论中约束体系的“总纲”。

**原文内容：**
> - **Entity Integrity**
>   - The values of primary key are not null
>
> - **Referential Integrity**
>
> - **User-defined Integrity**

**详细知识点讲解：**

完整性约束是关系数据库为了保证**数据正确性**和**一致性**而设立的规则体系。关系模型定义了三个层次的完整性约束。

---

**1. 实体完整性**

- **定义**：主键的值**不能为空**。
- **原文**：*The values of primary key are not null.*
- **解释**：
    - 主键用于唯一标识关系中的每个元组。
    - 如果主键为 `NULL`，则意味着“不知道这个实体是谁”，那么它就无法履行“唯一标识”的功能。
    - 空值表示“未知”，而标识一个实体不能建立在“未知”的基础上。
- **约束范围**：针对**主键属性**。
- **例子**：
    - `customer` 表中的 `customer_name` 不能为 `NULL`。每个客户必须有姓名（哪怕是系统生成的匿名 ID）。
    - `account` 表中的 `account_number` 不能为 `NULL`。每个账户必须有账号。
    - 如果主键是复合主键（如 `depositor` 的 `(customer_name, account_number)`），则这两个属性**均不能**为 `NULL`。

---

**2. 参照完整性**

- **定义**：外键的值要么是被参照表主键中**已存在的值**，要么是**空值**。
- **原文**：仅列出标题，因为前几页已详细展开讲解。
- **回顾要点**：
    - 外键必须引用**真实存在**的实体。
    - 如果外键是自身主键的一部分，则不能为 `NULL`。
- **约束范围**：针对**外键属性**。
- **例子**：
    - `account` 表中的 `branch_name` 必须存在于 `branch` 表的 `branch_name` 中。
    - 不能将账户分配到一个不存在的分行。

---

**3. 用户定义完整性**

- **定义**：由用户根据**具体业务需求**自定义的数据约束规则。
- **特点**：这不是关系模型强制要求的，而是特定应用场景的**业务规则**。
- **常见形式**：
    - **域约束**：限制属性的取值范围。例如：`balance >= 0`（余额不能为负），`age BETWEEN 0 AND 150`。
    - **唯一性约束**：非主键属性也要求唯一。例如：`email` 字段虽然不一定是主键，但业务要求每个客户的邮箱必须唯一。
    - **CHECK 约束**：定义更复杂的布尔表达式。例如：`loan.amount <= branch.assets`（某分行的贷款总额不能超过该分行的资产总额）。
    - **触发器**：更复杂的、由事件驱动的业务规则。
- **例子**：
    - 银行规定：活期账户余额不能低于 10 元。
    - 学校规定：学生年龄必须在 16 到 60 岁之间。
    - 商场规定：会员积分不能为负数。

---

**4. 三类完整性约束的对比总结**

| 完整性类型         | 约束对象 | 约束内容                          | 来源               |
| :----------------- | :------- | :-------------------------------- | :----------------- |
| **实体完整性**     | 主键     | 主键值不能为 `NULL`               | 关系模型本身的要求 |
| **参照完整性**     | 外键     | 外键值必须引用有效主键或为 `NULL` | 关系模型本身的要求 |
| **用户定义完整性** | 任意属性 | 满足特定业务规则                  | 应用场景的需求     |

---

**5. 三类完整性之间的关系**

这三者共同构成了关系数据库的**数据质量防线**：

- **实体完整性**确保了每个实体**可被识别**。
- **参照完整性**确保了实体之间的**关联有效**。
- **用户定义完整性**确保了数据符合**业务逻辑**。

违反任何一种完整性约束的操作（插入、删除、更新）都会被数据库系统**拒绝执行**，从而保证数据库中存储的数据始终处于**一致、正确、有意义**的状态。

### 两页 PPT 的逻辑关系

| 页数       | 主题       | 作用                                                   |
| :--------- | :--------- | :----------------------------------------------------- |
| **第一页** | 模式图     | 展示数据库的**结构全景**，明确各表的属性和表间关联。   |
| **第二页** | 完整性约束 | 定义施加在这些表结构之上的**规则体系**，保证数据质量。 |

**联系**：模式图定义了“**骨架**”（有哪些表、怎么连），完整性约束定义了“**纪律**”（什么数据能存、什么不能存）。两者结合，才是一个完整、健壮的关系数据库设计。
