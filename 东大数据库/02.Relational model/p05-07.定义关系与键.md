### 第一页：Relation Schema（关系模式）

这一页 PPT 定义了**关系模式**的概念，并将它与具体的**关系实例**进行了区分。这是理解数据库设计（DDL）与数据内容（DML）分离的关键。

**原文内容（含图示结构）：**

```
Relation Schema

• A relation schema consists of a relation name and a set of attributes.
  Relation: Account

Account-schema = (account_number, branch_name, balance)

• In general, a relation schema R, denoted by R(A1, A2, ..., An),
  is made up of a relation name R and a list of attributes A1, A2, ..., An.

• We use schema to describe the structure of a relation at the logical level.

• A relation instance r is a set of tuples corresponding to the relation schema at a particular moment.
```

**详细知识点讲解：**

**1. 模式（Schema）与实例（Instance）的严格区分**
这是关系数据库中最重要的**元数据**概念。

- **Relation Schema（关系模式）**：
    - **是什么**：它是对表**结构**的**定义**、**蓝图**、**框架**。
    - **包含内容**：关系名 + 属性列表（以及属性对应的域，稍后讲解）。
    - **类比**：相当于编程语言中的 **`class`（类）定义**，或者 Excel 表格的**表头行**。
    - **记法**：`R(A1, A2, ..., An)`。例如 `Account(account_number, branch_name, balance)`。
    - **特点**：**相对稳定，不常变化**。数据库建表时确定。

- **Relation Instance（关系实例）**：
    - **是什么**：在**某一特定时刻**，按照关系模式填入的具体**数据行集合**。
    - **类比**：相当于编程语言中 `new` 出来的 **`object`（对象）**，或者 Excel 表格里填了数据的**数据行**。
    - **特点**：**随时间频繁变化**（插入、删除、修改）。

**2. 逻辑层的描述（Logical Level）**
PPT 强调 "We use schema to describe the structure ... at the logical level"。这意味着关系模式隐藏了物理存储细节。用户只需要知道有 `account_number` 这一列，而不需要关心它在硬盘的第几个扇区。

**3. 形式化记法（Formal Notation）**
- `R(A1, A2, ..., An)`
- **R**：Relation 的首字母，代表关系名。
- **A1..An**：Attribute 的首字母，代表属性名。
- **n**：关系的**度**。

> **小结**：这一页在教你**画表格的框框**，而不是填内容。

### 第二页：Attributes（属性与域）

这一页紧接上一页，深入到**属性**的细节，特别是属性的**数据类型与域**。

**原文内容（含图表）：**

```
Attributes

• The set of allowed values for each attribute is called the domain of the attribute

• Attribute values are required to be atomic; that is, indivisible

• Domain - legal type and range of values of an attribute denoted by dom(A_i)

• The special value null is a member of every domain, which signifies that the value is unknown or not available.
```

**详细知识点讲解：**

**1. 域的回顾与深化（Domain）**
- 这是对上文知识的巩固。每个属性都绑定一个**域**。
- 记法 `dom(A_i)` 在查询优化和完整性检查中至关重要。例如：尝试比较 `account_number` 和 `balance` 的大小，如果它们的域不同（一个是字符串，一个是数字），数据库系统可能会报错或进行隐式转换。

**2. 原子性（Atomicity）—— 第一范式（1NF）的基石**
- **原文**：*Attribute values are required to be **atomic**; that is, indivisible.*
- **核心解释**：属性的值必须是**不可再分的**。
- **反例（非原子性）**：
    - **集合值**：一个人有多个电话号码，全塞在一个 `phone_numbers` 列里，用逗号分隔：`"123456, 789012"`。这在关系模型中是不允许的（尽管现实中很多人这么做）。
    - **复合值**：一个地址列存了 `"北京市海淀区中关村大街1号"`，无法直接按"市"或"区"查询。
- **正例（原子性）**：
    - 电话号码单独拆成另一张表（`person_phone`），一行一个号码。
    - 地址拆分为：`province`, `city`, `street` 多列。
- **为什么要原子性？** 为了能利用关系代数（特别是选择、投影）进行精确的查询操作。如果值是复合的，SQL 的 `WHERE` 子句效率会极低。

**3. 空值（Null Value）**
- **原文**：*The special value **null** is a member of every domain...*
- **定义**：NULL 不是一个具体的值（不是空字符串 `''`，也不是数字 `0`），它是一种**标记**。
- **语义**：
    - **Unknown（未知）**：我知道这个人有生日，但我现在不知道。
    - **Not Available（不可用/缺失）**：这个人根本没留电话。
- **重要后果**：
    - 任何涉及 NULL 的算术运算（如 `Salary + NULL`）结果都是 **NULL**。
    - 任何涉及 NULL 的逻辑比较（如 `Age = NULL`）结果既不是 TRUE 也不是 FALSE，而是 **UNKNOWN**（三值逻辑）。
    - 这是 SQL 中 `IS NULL` 和 `IS NOT NULL` 语法存在的根本原因。

### 第三页：Database Schema and Keys（数据库模式与键）

这一页将视角从单张表扩展到整个数据库，并引入了关系模型中最核心的约束机制——**键**。

**原文内容（含图示）：**

```
Database Schema

• Database schema: is the logical design of the database.
• Database instance: is a snapshot of the data in the database at a given instant in time.

Keys

• Let K ⊆ R (K is a subset of R's attributes)
• K is a superkey of R if values for K are sufficient to identify a unique tuple of each possible relation r(R)
  - Example: {ID} and {ID,name} are both superkeys of instructor.
• K is a candidate key if K is minimal superkey.
  - Example: {ID} is a candidate key for Instructor.
• One of the candidate keys is selected to be the primary key.
• Foreign key constraint: Value in one relation must appear in another relation (Referencing and Referenced relation).
```

**详细知识点讲解：**

**1. 数据库模式 vs 数据库实例**
- 这是上一页概念的放大版。
- **Database Schema（数据库模式）**：库中所有**关系模式**的集合 + 所有的**约束**。
- **Database Instance（数据库实例）**：某一时刻库中所有**关系实例**的快照。

**2. 超键（Superkey）**
- **定义**：设 \( K \subseteq R \)，若关系 \( r(R) \) 中任意两个不同的元组在属性集 \( K \) 上的取值都**不同**，则称 \( K \) 为超键。
- **核心功能**：**唯一标识**一个元组。
- **例子分析**：学生表有 `ID`（学号）、`Name`（姓名）。
    - `{ID}` 是超键：因为全校学号唯一。
    - `{ID, Name}` **也是**超键：因为学号唯一，再加上姓名，组合起来当然还是唯一的。
    - **特点**：超键可能包含**冗余**属性。

**3. 候选键（Candidate Key）与 主键（Primary Key）**
- **候选键定义**：**极小超键**（Minimal Superkey）。即去掉了任何一个属性，它就不再是超键了。
    - 继续上面的例子：`{ID, Name}` 不是候选键，因为去掉 `Name` 后，`{ID}` 仍然能唯一标识一行。所以 `{ID}` 才是候选键。
- **主键定义**：由数据库设计者从多个候选键中**人为挑选**的一个，作为该表最主要的唯一标识方式。
    - **PPT 惯例**：通常在关系模式记法中，用**下划线**标出主键。
    - 例如：`Account(account_number, branch_name, balance)`。

**4. 外键约束（Foreign Key Constraint）**
这是将多个独立的关系（表）**连接**成一个整体的关键约束。

- **定义**：关系 `r1` 中的某（些）属性 \( A \)，其取值必须**引用自**关系 `r2` 中的主键值。
- **术语**：
    - **Referencing Relation（参照关系）**：含有外键的表（子表）。
    - **Referenced Relation（被参照关系）**：被引用的主键所在的表（父表）。
- **例子（结合第一页的 Account 表）**：
    - 假设有 `Branch(branch_name, city, assets)` 表，主键是 `branch_name`。
    - 在 `Account` 表中，`branch_name` 就是一个**外键**。
    - **约束逻辑**：你不能在 `Account` 表中插入一个 `branch_name = 'MoonBase'` 的账户，除非 `'MoonBase'` 已经存在于 `Branch` 表的主键中。这确保了数据引用的**完整性**。

### 本轮三页 PPT 的逻辑总结

1. **第一页（Schema）**：定义了表的**框架**（模式）与**内容**（实例）的区别。
2. **第二页（Attributes）**：定义了框架里每一根柱子的**规格**（原子性、域、空值处理）。
3. **第三页（Keys）**：定义了框架内部的**规则**（如何唯一找人、如何表间连线）。
