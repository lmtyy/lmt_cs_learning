这是一个非常关键的进阶概念。很多初学者以为外键只能是一列对应一列，但实际上，**如果父表的主键是“联合主键”（Composite Key），那么子表的外键也必须是“联合外键”**。

这就像**“双孔插座”**：如果墙上的插座（父表主键）有两个孔，你的插头（子表外键）也必须有两个脚才能插得进去，缺一不可。

我们用一个**“航班”与“机票”**的例子来详细讲解。

------

### 1. 场景设定：为什么需要联合主键？

想象一下航空公司的**航班时刻表（Flight_Schedule）**。

- **航班号（Flight_No）**：比如 `CA123`。光有这个是不够的，因为这趟航班**每天**都有。
- **日期（Flight_Date）**：比如 `2023-10-01`。
- **结论**：必须是 **(航班号 + 日期)** 结合在一起，才能唯一确定那一架具体的飞机。这就是父表的**联合主键**。

------

### 2. 联动机制：子表如何引用？

现在有一张**机票表（Ticket）**，记录了张三买了一张票。

- 如果你只存 `CA123`，不知道他是哪天的。
- 如果你只存 `2023-10-01`，不知道他是哪个航班的。
- **必须同时存两个！**

#### 父表：航班表 (Schedule)

主键是：**(Flight_No + Date)**

| **Flight_No (PK)** | **Date (PK)**  | **机型**    |
| ------------------ | -------------- | ----------- |
| **CA123**          | **2023-10-01** | Boeing 737  |
| **CA123**          | **2023-10-02** | Boeing 737  |
| **MU456**          | **2023-10-01** | Airbus A320 |

#### 子表：机票表 (Ticket)

为了关联到父表，子表必须**完整复制**父表的那两列作为外键。

| **Ticket_ID (PK)** | **Passenger** | **Ref_Flight (FK)** | **Ref_Date (FK)** |
| ------------------ | ------------- | ------------------- | ----------------- |
| T001               | 张三          | **CA123**           | **2023-10-01**    |
| T002               | 李四          | **CA123**           | **2023-10-01**    |
| T003               | 王五          | **CA123**           | **2023-10-02**    |

> **注意：** 在这里，`Ref_Flight` 和 `Ref_Date` 这两列**共同组成了一个外键约束**。它们必须作为一个整体去父表里找对应关系。

------

### 3. 联动的“三大铁律”

联合主键和外键的联动非常严格，必须遵守以下规则：

#### 规则一：数量必须对等（2对2，3对3）

- 父表主键有 2 列，子表的外键就必须有 2 列。
- 你不能只引用父表联合主键中的其中一列。这就好比你只有一把钥匙，却想开两道锁，门是打不开的。

#### 规则二：顺序与类型必须匹配

- 如果父表是 `(String, Date)`，子表的外键定义也必须是 `(String, Date)`。
- 数据库在校验时，是按顺序比对的：子表的第一列找父表的第一列，子表的第二列找父表的第二列。

#### 规则三：整体存在性

- 子表插入数据 `(CA123, 2023-10-05)` 时，数据库会去父表检查：
  - 有没有 `Flight_No = CA123` **且** `Date = 2023-10-05` 的记录？
  - 如果父表里只有 10月1号 和 10月2号 的航班，这条数据就会被**拒绝写入**。

------

### 4. 常见的误区

很多同学会问：**“子表里这两个外键列，必须也是子表的主键吗？”**

**答案：不一定。**

- **情况 A（普通外键）：** 就像上面的“机票表”。`Ticket_ID` 是主键，那两个列只是普通的外键列（允许重复，因为很多人可以买同一趟航班）。

- **情况 B（多对多关系的中间表）：**

  比如“学生选课表”。

  - 父表1：学生 `(Student_ID)`
  - 父表2：课程 `(Course_ID)`
  - 子表（选课表）：`(Student_ID, Course_ID)`。
  - 在这种情况下，这两个外键列**同时也组成了子表的联合主键**（因为一个学生不能重复选同一门课）。

### 总结

联合主键与外键的联动，就是**“打包捆绑”**的关系：

1. 父表说：“我的身份证是两张卡拼起来的。”
2. 子表说：“那我也得留两个卡槽，才能把你存下来。”
3. 查询时：“拿着两张卡同时去比对，必须两张都对得上，才算找到了亲爹。”
