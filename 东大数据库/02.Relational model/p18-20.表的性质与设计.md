### 第一页：Properties of Tables（表的性质）

这一页总结了关系模型中**表**必须满足的五条核心性质，是对前面所学概念的集中归纳。

**原文内容：**
> - A table is **finite**.
> - **Atomic data values**: the data type of a column must be a single value (1NF)
> - There are **no duplicate rows** in a table.
> - The **order of the rows** in a table is not important.
> - The **order of the columns** in a row is not important.

**详细知识点讲解：**

**1. 表的有限性（Finite）**
- **定义**：一个关系（表）中元组的数量是**有限的**。
- **理论背景**：虽然在数学定义中，关系是笛卡尔积的子集，理论上可以有无穷多个元组（例如所有正整数的集合），但在实际数据库系统中，表只能存储**有限数量的行**，受制于物理存储空间。
- **意义**：这确保了数据库查询在有限时间内能够完成。

**2. 数据值的原子性（Atomic Data Values）**
- **定义**：每一列的数据类型必须是**单一值**，即不可再分的原子。
- **与第一范式的关系**：这是**第一范式**的直接要求。违反原子性意味着表中嵌套了表，或者一个字段里存了用逗号分隔的多值。
- **反例**：某列的值为 `"A-101, A-102"`（两个账号）—— 违反原子性。
- **正例**：每个账号单独占一行，或拆分为另一张关联表。

**3. 不允许重复行（No Duplicate Rows）**
- **定义**：同一个关系中，不能存在两个**完全一样**的元组。
- **数学基础**：关系被定义为**元组的集合**，而数学中的集合不允许有重复元素。
- **实践中的主键**：为了强制这一性质，每个表通常都会定义一个**主键**。即使你没有显式定义主键，数据库系统在逻辑上仍然视两行完全相同为非法（某些 SQL 实现允许物理上存储全同的行，但那是依赖于隐藏的物理行 ID，不符合纯关系模型）。
- **例子**：在 `account` 表中，不可能存在两行 `(A-101, Downtown, 500)`。

**4. 行的顺序不重要（Row Order Unimportant）**
- 前文已详述，此处归纳为一条性质：表是一个集合，行没有内在的先后次序。`ORDER BY` 是输出时的呈现要求，而非数据本身的属性。

**5. 列的顺序不重要（Column Order Unimportant）**
- 类似地，属性在关系模式中的排列顺序也没有数学意义。我们总是通过**列名**访问列，而不是通过列的位置（尽管 SQL 允许 `SELECT *` 按定义顺序输出，但这是实现约定，而非逻辑必需）。

> **小结**：这一页是关系模型的“五条军规”，每一条都源于集合论的定义，共同保证了关系数据库的逻辑简洁性和操作一致性。

### 第二页：Relational Database（关系数据库 —— 多关系的协作）

这一页将视角从单个表扩展到整个数据库，解释了为什么需要将信息**拆分**到多个关系（表）中。

**原文内容：**
> - A database consists of **multiple relations**
> - Information about an enterprise is broken up into parts, with each relation storing one part of the information
>
> **Example**
> - `account`: stores information about accounts
> - `depositor`: stores information about which customer owns which account
> - `customer`: stores information about customers

**详细知识点讲解：**

**1. 数据库是多关系的集合**
- 一个实际的关系数据库**极少**只包含一张表。相反，它是由**多张相互关联的表**组成的。
- 这符合数据库设计的**分而治之**原则。

**2. 信息拆分的原因**
- **避免冗余**：如果将所有信息塞进一张大表，会导致大量重复数据（下一页会展开）。
- **语义清晰**：每张表对应现实世界中的一个**实体集**或**联系集**。
- **维护方便**：修改某个实体的属性时，只需要在一张表中更新。

**3. PPT 中的银行示例拆解**
这个例子展示了一个最简化的银行数据库，包含三张表：

| 表名        | 类型       | 存储内容                           |
| :---------- | :--------- | :--------------------------------- |
| `account`   | 实体集     | 账户的固有信息：账号、开户行、余额 |
| `customer`  | 实体集     | 客户的固有信息：姓名、住址、电话等 |
| `depositor` | **联系集** | 连接客户与账户：谁拥有哪个账号     |

- **联系集的特殊性**：`depositor` 表的存在价值是表达“多对多”关系——一个客户可以有多个账户，一个账户也可以被多个客户联合持有。如果这种关系只用一张大表存储，就会产生严重的数据冗余问题。

**4. 表之间的关联方式**
- `account` 表的主键是 `account_number`。
- `customer` 表的主键是 `customer_name`（实际系统中通常用 `customer_id`）。
- `depositor` 表包含两列：`customer_name` 和 `account_number`，它们分别是**外键**，引用自 `customer` 和 `account` 的主键。

通过外键的**参照完整性约束**，这三张表被逻辑地“拼接”成一个有机整体，既保持了数据的独立性，又能通过 **JOIN 操作**恢复出完整信息。

### 第三页：Relational Database（单表存储的弊端与规范化）

这一页以反面教材的方式，展示了如果将全部信息塞进一张大表会引发什么问题，并由此引出了**规范化理论**的重要性。

**原文内容：**
> - Storing all information as a single relation
>   `bank(account_number, balance, customer_name, ...)`
>
> Result in
>   - **repetition of information**
>     - if two customers own an account (What gets repeated?)
>   - the need for **null values**
>     - to represent a customer without an account
>
> - **Normalization theory** deals with how to design relational schemas

**详细知识点讲解：**

**1. 反面案例：单一全量表 `bank`**
假设我们不拆表，而是设计一张包含了所有信息的 `bank` 表，结构如下：
```
bank(account_number, balance, customer_name, customer_city, ...)
```
这张表试图同时存储账户信息、客户信息以及它们之间的关联。

**2. 问题一：信息重复（Repetition of Information）**
- **场景**：一个账户由两个客户共同拥有。
- **实际数据**：
    - 账户 `A-101`，余额 `500`，持有者：`张三` 和 `李四`。
- **单一表存储方式**：
    | account_number | balance | customer_name |
    | -------------- | ------- | ------------- |
    | A-101          | 500     | 张三          |
    | A-101          | 500     | 李四          |
- **重复了什么？**
    - `A-101` 的余额 `500` 被存储了**两次**。
    - 如果账户还有 `branch_name` 等其他属性，同样会被重复。
- **后果**：
    - **存储浪费**：磁盘空间被冗余数据占据。
    - **更新异常**：如果需要修改 `A-101` 的余额（比如从 500 变成 600），必须同时更新**所有**包含该账户的行。如果漏掉一行，就会造成**数据不一致**（同一个账户出现两个不同余额）。

**3. 问题二：空值的滥用（Need for Null Values）**
- **场景**：银行有一位新客户刚刚注册，但还没有开设任何账户。
- **在单一 `bank` 表中如何存储？**
    - 表中必须包含 `account_number` 和 `balance` 列，但这位客户没有账户。
    - 只能在这些列中填入 **`NULL`**。
    | account_number | balance | customer_name |
    | -------------- | ------- | ------------- |
    | NULL           | NULL    | 王五          |
- **后果**：
    - `NULL` 本该表示“未知”或“缺失”，这里却用来表示“不存在”或“不适用”，**语义被混淆**。
    - 如果 `account_number` 是主键，它不能为 `NULL`，所以这种设计根本**无法实现**（主键约束失败）。

**4. 解决方案：规范化理论**
- **原文**：*Normalization theory deals with how to design relational schemas.*
- **定义**：**规范化**是一套形式化的方法，用于评判关系模式的优劣，并将“不好”的模式**分解**为“好”的模式。
- **目标**：消除冗余、避免更新异常、减少空值滥用。
- **在本例中**：规范化的做法正是将 `bank` 分解为第二页展示的 `account`、`customer` 和 `depositor` 三张表。
    - `account`：存储账户自身信息，余额只存一次。
    - `customer`：存储客户自身信息，即使没有账户也能存在（因为 `customer` 表不依赖 `account` 表）。
    - `depositor`：仅存储**连接关系**，不重复存储余额或客户城市。
- **后续课程内容**：规范化理论通常包含**第一范式、第二范式、第三范式、BCNF**等递进层次，是关系数据库设计的核心内容。
