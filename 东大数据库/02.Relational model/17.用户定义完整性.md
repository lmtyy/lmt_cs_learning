这是为您整理的**用户定义完整性（User-defined Integrity）**详细笔记。

如果有前两个（实体完整性、参照完整性）是数据库的“宪法”和“基本法”，那么**用户定义完整性**就是针对具体业务场景定制的**“地方法规”**或**“公司规章”**。

它弥补了前两者的不足，确保数据不仅符合“结构要求”，还要符合**“现实逻辑”**。

------

### 一、 核心定义

**用户定义完整性**（也称为域完整性或语义完整性）是指：

针对某一具体应用的数据必须满足的**语义要求**。它由数据库设计者根据具体业务规则来定义。

- **目的**：防止不符合业务逻辑的数据进入数据库。
- **灵活性**：它不是数据库系统强制规定的（不像主键必须非空），而是**你想怎么定就怎么定**。

------

### 二、 为什么要有个“用户定义”？

前两个完整性只能解决基础问题：

- **实体完整性**：保证了“人是唯一的”。
- **参照完整性**：保证了“班级是存在的”。

**但是，它们解决不了以下问题：**

- 人的年龄能不能填 **-5岁**？（主键和外键都不管这个）
- 人的性别能不能填 **“外星人”**？
- 考试成绩能不能填 **120分**（满分100）？
- 入职日期能不能早于出生日期？

这时候，必须依靠**用户定义完整性**来拦截这些荒谬的数据。

------

### 三、 常见的实现方式（SQL 关键字）

在数据库设计中，我们通常通过以下几种约束（Constraints）来实现用户定义完整性：

#### 1. CHECK 约束（最核心）

这是最直接的逻辑检查。它规定了一个列的取值范围。

- **场景**：年龄必须大于0且小于150。

- **SQL 示例**：

  SQL

  ```
  CREATE TABLE Student (
      Name VARCHAR(20),
      Age INT,
      CHECK (Age > 0 AND Age < 150) -- 限制年龄范围
  );
  ```

- **效果**：如果你插入 `Age = 200`，数据库直接报错。

#### 2. NOT NULL 约束（非空）

虽然主键天然非空，但对于非主键字段，我们需要手动规定。

- **场景**：用户的“手机号”虽然不是主键，但在业务上规定**必须填写**，不能留白。

- **SQL 示例**：

  SQL

  ```
  Phone_Number VARCHAR(20) NOT NULL
  ```

#### 3. UNIQUE 约束（唯一）

这也是针对非主键列的。

- **场景**：用户的“身份证号”或“邮箱”。它们不是主键（主键可能是个自增ID），但业务要求**不能重复**。

- **SQL 示例**：

  SQL

  ```
  Email VARCHAR(50) UNIQUE
  ```

#### 4. DEFAULT 约束（默认值）

当用户不填写时，自动填入一个合理的值。

- **场景**：注册时间。如果用户没填，默认填“当前系统时间”。

- **SQL 示例**：

  SQL

  ```
  Register_Date DATETIME DEFAULT CURRENT_TIMESTAMP
  ```

#### 5. ENUM / 域约束（枚举）

限制值只能在几个固定的选项里选。

- **场景**：性别。只能是“男”或“女”。

- **SQL 示例**：

  SQL

  ```
  Gender CHAR(1) CHECK (Gender IN ('M', 'F'))
  ```

------

### 四、 进阶：触发器（Triggers）

有时候，业务逻辑太复杂，简单的 `CHECK` 搞不定。

比如：**“一个客户的信用额度，不能超过他所在城市的人均收入水平。”**

这涉及到了跨表查询和计算，这时候就需要用到**触发器（Trigger）**。

- **定义**：一段预先写好的代码，当插入或修改数据时自动执行。
- **作用**：可以实现极其复杂的、动态的用户定义完整性。

------

### 五、 总结对比（三大完整性复习）

这是最容易考的知识点，建议直接背诵这个对比表：

| **完整性类型**     | **对象**            | **核心规则**     | **例子**                   | **强制性**   |
| ------------------ | ------------------- | ---------------- | -------------------------- | ------------ |
| **实体完整性**     | **主键** (Row)      | 唯一且非空       | 学号不能重复，不能为空     | 必须遵守     |
| **参照完整性**     | **外键** (Table)    | 要么空，要么存在 | 不能引用不存在的班级       | 必须遵守     |
| **用户定义完整性** | **属性值** (Column) | **符合业务逻辑** | 年龄不能为负，性别只能男女 | **按需设定** |

**一句话总结：**

用户定义完整性就是**“业务逻辑的守门员”**，它防止了那些符合语法但**不符合常理**的数据进入数据库。
