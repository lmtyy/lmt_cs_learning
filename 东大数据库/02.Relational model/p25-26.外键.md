### 第一页：Foreign Keys —— 外键的定义与术语

这一页给出了外键的正式定义，并引入了**参照关系**和**被参照关系**两个关键术语。

**原文内容：**
> - A relation \( r_1 \) may have an attribute that corresponds to the primary key of another relation \( r_2 \). The attribute is called a **foreign key**.
>   - \( r_1 \) is **referencing relation**
>   - \( r_2 \) is **referenced relation**
>
> **Example**
> - `customer_name` and `account_number` attributes of `depositor` are foreign keys to `customer` and `account` respectively. `depositor` is referencing relation, `customer` and `account` are referenced relation.

**详细知识点讲解：**

**1. 外键的定义**
- **核心描述**：如果关系 \( r_1 \) 中的某个（或某组）属性，它的取值必须**对应**到另一个关系 \( r_2 \) 的**主键**值，则称该属性为**外键**。
- **关键要素**：
    - 外键是**一个属性或一组属性的集合**。
    - 外键的值必须在被参照表的主键中存在（或为空，视约束而定）。
    - 外键建立了表与表之间的**逻辑联系**。

**2. 两个角色的定义**
- **Referencing Relation（参照关系）**：包含外键的表，即 \( r_1 \)。
    - 形象理解：**“子表”** 或 **“从表”**，它依赖另一张表的数据。
    - 在例子中：`depositor` 是参照关系。
- **Referenced Relation（被参照关系）**：外键所指向的主键所在的表，即 \( r_2 \)。
    - 形象理解：**“父表”** 或 **“主表”**，它的数据被其他表引用。
    - 在例子中：`customer` 和 `account` 是被参照关系。

**3. PPT 示例的三表结构拆解**
基于前面课程中反复出现的银行数据库，这里的三张表结构如下：

| 表名        | 主键                                            | 角色               |
| :---------- | :---------------------------------------------- | :----------------- |
| `customer`  | `customer_name`                                 | 被参照关系（父表） |
| `account`   | `account_number`                                | 被参照关系（父表） |
| `depositor` | 通常是 `(customer_name, account_number)` 的组合 | 参照关系（子表）   |

- `depositor` 表中的 `customer_name` 列，其值必须来自 `customer` 表的主键 `customer_name`。因此，`customer_name` 是 `depositor` 表的一个**外键**，参照 `customer` 表。
- `depositor` 表中的 `account_number` 列，其值必须来自 `account` 表的主键 `account_number`。因此，`account_number` 是 `depositor` 表的另一个**外键**，参照 `account` 表。

**4. 外键约束的语义**
- **存在依赖**：你不能在 `depositor` 表中插入一条记录，说“王五拥有账户 A-999”，除非：
    - “王五”这个客户**已经存在**于 `customer` 表中。
    - “A-999”这个账户**已经存在**于 `account` 表中。
- **删除/更新规则**：如果尝试删除 `customer` 表中的“王五”，而 `depositor` 表中仍有引用他的记录，数据库会如何反应？
    - 这取决于定义外键时指定的**参照动作**（如 `ON DELETE CASCADE` 级联删除、`ON DELETE RESTRICT` 禁止删除等）。这是后续高级 SQL 的内容。

**5. 外键与主键的对应关系**
- 外键在参照表中**不一定**是主键（在 `depositor` 中，`customer_name` 单独出现多次是允许的，因为一个客户可以有多个账户）。
- 外键在被参照表中**必须**对应**主键**（或至少是候选键，即具有唯一性约束的列）。这是保证引用有效性的基础。

### 第二页：Example —— 外键实例的图形化展示

这一页通过一个具体的表格数据示例，直观展示了外键是如何工作的，以及违反外键约束的情况。

**原文内容（含图表解读）：**

PPT 中展示了两个表格片段：

**表一：account（被参照关系）**
| account_number                                               | branch_name | balance |
| ------------------------------------------------------------ | ----------- | ------- |
| A-101                                                        | Downtown    | 500     |
| A-102                                                        | Perryridge  | 400     |
| A-201                                                        | Brighton    | 900     |
| A-215                                                        | Mianus      | 700     |
| A-217                                                        | Brighton    | 750     |
| *(注意：原文中还混入了一行 "Round Hill 350"，这应该是排版问题，实际属于另一行记录的一部分)* |             |         |

**表二：depositor（参照关系，包含外键）**
| customer_name | account_number |
| ------------- | -------------- |
| Hayes         | A-102          |
| Johnson       | A-101*         |
| Johnson       | A-201          |
| Jones         | A-217          |
| Lindsay       | A-222          |
| Smith         | A-215          |
|               | A-305*         |

图中用箭头标注了：
- `account_number` 列是**外键**，对应于 `Account` 关系的主键。
- `customer_name` 列是**外键**，对应于 `Customer` 关系的主键。

**详细知识点讲解：**

**1. 被参照表 `account` 的数据分析**
- 主键是 `account_number`。
- 表中存在五个有效账户：`A-101`, `A-102`, `A-201`, `A-215`, `A-217`。
- **重要**：表中**不存在**账户 `A-222`，也**不存在**账户 `A-305`。

**2. 参照表 `depositor` 的数据与外键约束验证**
我们逐行检查 `depositor` 表中的外键值：

| customer_name | account_number | 验证结果                                        |
| :------------ | :------------- | :---------------------------------------------- |
| Hayes         | A-102          | ✅ 合法：`A-102` 在 `account` 表中存在           |
| Johnson       | A-101          | ✅ 合法：`A-101` 在 `account` 表中存在           |
| Johnson       | A-201          | ✅ 合法：`A-201` 在 `account` 表中存在           |
| Jones         | A-217          | ✅ 合法：`A-217` 在 `account` 表中存在           |
| Lindsay       | **A-222**      | ❌ **违反外键约束**：`A-222` 不在 `account` 表中 |
| Smith         | A-215          | ✅ 合法：`A-215` 在 `account` 表中存在           |
| (空)          | **A-305**      | ❌ **违反外键约束**：`A-305` 不在 `account` 表中 |

**3. 标注 \* 的含义**
- 原文中 `Johnson A-101*` 和单独的 `A-305*` 标有星号，通常表示**异常数据**或**重点标注**。
- 在这个上下文中，可能意在指出：
    - `A-101*`：合法引用（存疑，可能想强调它是正常引用）。
    - `A-305*`：**非法引用**，因为 `A-305` 在 `account` 表中不存在。
- 实际教学中，这张 PPT 很可能被用来演示**参照完整性违规**的场景。

**4. 另一个外键：`customer_name`**
- 虽然本页 PPT 未完整展示 `customer` 表的数据，但同样的逻辑适用于 `customer_name` 列。
- 例如：如果 `depositor` 表中出现了一个 `customer_name = 'Turner'`，而 `customer` 表中并没有 `'Turner'` 这个人，则同样违反外键约束。
- 只有 `customer_name` 既在 `customer` 表中存在，`account_number` 又在 `account` 表中存在，这一行数据才是完全合法的。

**5. 外键的实际作用体现**
- **防止孤儿记录**：外键约束阻止了 `depositor` 表中出现“引用了一个不存在的账户”或“引用了一个不存在的客户”的情况。这些没有父记录的行被称为**孤儿记录**，外键正是用来杜绝它们的。
- **保证数据一致性**：外键确保了数据库中跨表引用的**正确性**和**完整性**。任何破坏这种引用的插入或更新操作都会被数据库系统**直接拒绝**。
