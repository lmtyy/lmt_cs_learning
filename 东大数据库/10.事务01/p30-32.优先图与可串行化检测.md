这三页PPT开始从"可串行化理论"过渡到"如何在实际系统中实现和检测可串行化"。

### 第1页：隔离性的实现——核心权衡

这一页点明了并发控制方案的本质。

**原文**：
> - Schedules must be serializable.
> - A policy in which only one transaction can execute at a time generates serial schedules, but provides a poor degree of concurrency.
> - Concurrency-control schemes tradeoff between the amount of concurrency they allow and the amount of overhead that they incur.

**详细解读**：

- **目标明确**：调度必须是**可串行化的**。这是正确性的底线，系统必须确保所有被允许并发执行的调度，都等价于某个串行调度。

- **极端方案**：最简单的实现方法就是"**一次只允许一个事务运行**"。这确实天然产生串行调度，绝对正确，但并发度为零，系统吞吐量极差——这正是之前一直批评的"串行执行"。

- **核心权衡（Tradeoff）**：
  > 并发控制方案本质上是做权衡：在"允许多高的并发度"和"带来多少额外开销"之间寻找平衡点。
  
  - **高并发度**：允许事务大量并行执行 → 系统吞吐量高，但检测和保证正确性需要复杂的机制，开销大。
  - **低开销**：简单的串行执行 → 没有并发控制开销，但并发度为零。
  - 实际系统（如锁协议、时间戳、MVCC等）都是在这两个极端之间的折中方案。

这一页是承上启下的转折点：既然需要权衡，接下来就要介绍具体的工具和手段。

---

### 第2页：可串行化检测——优先图的定义

这一页引入了判断一个调度是否可串行化的图形化工具——**优先图**。

**定义**：
> **优先图（Precedence Graph）**是一个有向图，其中：
> - **顶点**：参与调度的事务（`T₁, T₂, ..., Tₙ`）。
> - **有向边**：从 `Tᵢ` 指向 `Tⱼ`，当且仅当满足两个条件：
>   1. `Tᵢ` 和 `Tⱼ` **存在冲突**（访问同一数据项，且至少有一个是写）；
>   2. **`Tᵢ` 先执行**了引发冲突的那个操作。

**添加边的规则（三种冲突情况）**：

1. **写读冲突**：`Tᵢ` 先 `write(Q)`，然后 `Tⱼ` `read(Q)`。`Tⱼ` 读到了 `Tᵢ` 写入的值。→ 边：`Tᵢ → Tⱼ`

2. **读写冲突**：`Tᵢ` 先 `read(Q)`，然后 `Tⱼ` `write(Q)`。`Tⱼ` 覆盖了 `Tᵢ` 当时读到的值。→ 边：`Tᵢ → Tⱼ`

3. **写写冲突**：`Tᵢ` 先 `write(Q)`，然后 `Tⱼ` `write(Q)`。`Tⱼ` 覆盖了 `Tᵢ` 的值。→ 边：`Tᵢ → Tⱼ`

**判断准则**：
> 如果优先图中**没有环**，则该调度是**冲突可串行化的**。  
> 如果优先图中**存在环**，则该调度**不是**冲突可串行化的。

优先图将"是否可串行化"的语义判断转化为一个纯粹的图论问题——检测有向图是否存在环。

PPT中给出一个示意图，包含 `T₁` 和 `T₂` 两个节点，以及连接它们的有向弧。这示意了当事务间产生冲突时，它们之间就会建立起一种"先后依赖"关系，箭头方向表示谁必须在谁前面。

### 第3页：用优先图测试调度——案例

这一页是**第2页方法的实际应用**，展示如何根据一个具体调度画出优先图，并判断其是否可串行化。

PPT上的表格看起来是两栏并排，但实际上它**不是在展示交错执行的调度顺序**，而是分别列出了 `T₁` 和 `T₂` 两个事务的完整定义（它们各自的所有操作），这与前文银行转账的例子完全一致：

- **T₁**：从A转50到B
  ```
  read(A) → A:=A-50 → write(A) → read(B) → B:=B+50 → write(B)
  ```
- **T₂**：从A转10%到B
  ```
  read(A) → temp:=A*0.1 → A:=A-temp → write(A) → read(B) → B:=B+temp → write(B)
  ```

**结合前文的两个调度案例，用优先图分析**：

**案例一：正确的并行调度（等价于 T₁ → T₂）**
交错顺序为：
```
T₁: read(A) → write(A)
T₂:                read(A) → write(A)
T₁: read(B) → write(B)
T₂:                          read(B) → write(B)
```
冲突分析：
- `T₁` 先 `write(A)`，`T₂` 后 `read(A)`：T₁ → T₂
- `T₁` 先 `write(B)`，`T₂` 后 `read(B)`：T₁ → T₂

唯一边是 `T₁ → T₂`，图中无环，**是冲突可串行化的**。

**案例二：错误的并行调度（结果不一致）**
交错顺序为：
```
T₁: read(A)
T₂:       read(A) → write(A) → read(B)
T₁: write(A)          ← 这里有问题
T₂:                          write(B)
T₁: read(B) → write(B)
```
冲突分析：
- `T₂` 先 `read(A)`，`T₁` 后 `write(A)`：T₂ → T₁（读写冲突）
- `T₁` 先 `read(B)`，`T₂` 后 `write(B)`：T₁ → T₂（读写冲突）

图中既有 `T₁ → T₂`，又有 `T₂ → T₁`，构成一个环。存在环，**不是冲突可串行化的**。

### 总结：这三页的逻辑

1.  **提出权衡问题**（第1页）：系统必须保证可串行化，但又不能牺牲太多并发度，因此需要高效的检测/保证机制。
2.  **给出检测工具**（第2页）：优先级图是判断冲突可串行化的形式化工具，冲突产生有向边，无环则正确。
3.  **实例应用**（第3页）：结合之前的转账事务，展示优先图如何在实际调度中工作——正确调度产生无环图，错误调度产生带环图。

优先图是后续并发控制协议（如基于锁的协议、时间戳排序）正确性证明的理论基础。除了在事后分析调度，它更重要的作用是**在线检测或强制预防**——比如两段锁协议就是保证产生的调度对应的优先图一定是无环的。
