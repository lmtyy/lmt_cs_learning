### 第1页：隔离级别总览与三种异常现象

这一页是事务隔离级别的“全景图”，定义了用来区分隔离级别的三种数据异常现象，并给出了对应的标准表格。

**三种异常现象的定义：**

- **脏读**：一个事务读到了另一个事务**尚未提交**的修改数据。这些数据随时可能被回滚，所以是“脏”的。
- **不可重复读**：同一个事务内，两次读取**同一条记录**，却读到了不同的内容。因为在这两次读之间，另一个事务**修改**了这条记录并提交了。
- **幻读**：同一个事务内，两次执行**相同条件的查询**，结果集的行数不同。因为另一个事务在此期间**插入**或**删除**了符合条件的记录并提交了。

> **不可重复读与幻读的关键区别**：前者针对同一行数据的值被修改，后者针对结果集多了或少了几行。

**隔离级别与允许异常对应表：**

| 隔离级别     | 脏读     | 不可重复读 | 幻读     |
| ------------ | -------- | ---------- | -------- |
| **读未提交** | 允许     | 允许       | 允许     |
| **读已提交** | ❌ 不允许 | 允许       | 允许     |
| **可重复读** | ❌ 不允许 | ❌ 不允许   | 允许     |
| **可串行化** | ❌ 不允许 | ❌ 不允许   | ❌ 不允许 |

- 级别从低到高，一致性保证越来越强，但并发度通常越来越低。
- 这实际上是 **ANSI SQL 标准** 的定义，工业界实现（如MySQL的MVCC）在此基础上有所延伸。

### 第2页：读未提交——以脏读为例

这一页通过一个银行转账的实例，展示了最低隔离级别 **读未提交** 下，脏读是如何发生的。

**设定**：事务A要增加小明的存款，事务B要查询小明的存款。

**执行序列：**

| 时刻 | 事务A               | 事务B                            |
| ---- | ------------------- | -------------------------------- |
| t₁   | 读取小明存款，为100 |                                  |
| t₂   | 增加100 → 200       |                                  |
| t₃   | 更新存款为200       |                                  |
| t₄   |                     | **读取小明存款，为200** ← 脏读！ |
| t₅   | **回滚**            |                                  |

- 事务B在 t₄ 读到了200，这个值是事务A尚未提交的中间结果。
- 事务A随后在 t₅ 回滚，真实的、已提交的值仍是100。事务B读到的200就成了一笔从未真实存在过的“脏数据”。

**特点总结**：
- 读未提交完全不考虑事务之间的影响，不加任何保护，并发度最高，一致性最差。
- 适用场景：各事务之间没有相互影响，且偶尔读到未提交数据影响不大的业务。

### 第3页：读已提交——解决脏读

这一页针对上一页的问题，引入 **读已提交** 隔离级别，并说明它如何防止脏读。

**核心定义**：
> 事务只能读取到**其他事务已经提交**的数据值，忽略那些尚未提交的修改。

**用同样的场景对比**：

| 时刻 | 事务A               | 事务B                     |
| ---- | ------------------- | ------------------------- |
| t₁   | 读取小明存款（100） |                           |
| t₂   | 增加100 → 200       |                           |
| t₃   | 更新存款为200       |                           |
| t₄   | 回滚                | 读取小明存款，为**100** ✓ |

- 事务B在 t₄ 读取时，因为事务A尚未提交，数据库系统直接忽略A的修改版本，返回的是上一个已提交值100。
- 脏读被防止了。

**实现方法**：PPT提到了通过 MVCC（多版本并发控制）、锁、时间戳等技术实现，比如MVCC可以让读操作直接访问行的旧版本，从而无需等待写事务提交。

**遗留问题**：读已提交仍然允许 **不可重复读** 和 **幻读**。

### 第4页：读已提交——以不可重复读为例

这一页继续分析 **读已提交** 级别，展示它无法防止 **不可重复读** 的具体案例。

**设定已调整**：事务A本次会正常提交，不再是回滚。

**执行序列：**

| 时刻 | 事务A                                       | 事务B                                    |
| ---- | ------------------------------------------- | ---------------------------------------- |
| t₁   |                                             | 读取小明存款，为**100**                  |
| t₂   | 增加100 → 200                               |                                          |
| t₃   | 更新存款为200                               |                                          |
| t₄   | 读取小明存款，仍为**100** （因为A尚未提交） |                                          |
| t₅   | **提交**                                    |                                          |
| t₆   |                                             | **再次读取**小明存款，为**200** ← 冲突！ |

**问题分析**：
- 事务B在 t₁ 和 t₆ 两次读取同一条“小明”的记录。
- t₁ 读到100，t₆ 读到200。值在同一个事务内发生了无法重现的变化。
- 读已提交只保证了每次读到的值都是已提交的，但不保证一个事务内两次读取这个值会相同——这正是“不可重复读”的定义。

**总结此时的问题残留**：
> 读已提交解决了脏读，但不可重复读仍然存在；要解决不可重复读，需要把隔离级别提升到 **可重复读**。

---

这四页为后续介绍可重复读和可串行化做了完整铺垫，它们建立了一个清晰的分析框架：**每种隔离级别就是通过禁止一部分并发异常，来换取更强的一致性保证**。
