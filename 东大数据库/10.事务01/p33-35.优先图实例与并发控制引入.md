### 第1页：调度实例——提出问题

PPT给了一个包含三个事务 `T1、T2、T3` 的并发调度，问题是：**这个调度是冲突可串行化的吗？**

表格按时间从上到下排列：

| T1   | T2   | T3   |
| ---- | ---- | ---- |
| R(X) |      |      |
|      | R(Y) |      |
|      |      | W(Z) |
| W(X) |      |      |
|      | R(X) |      |
|      |      | W(X) |
| W(Z) |      |      |
|      | R(Y) |      |
|      | W(Y) |      |
|      |      | W(Z) |

你需要用优先图的方法来回答这个问题。

---

### 第2页：调度分析——画出优先图，发现环路

这一页给出了分析过程和结论。

**第一步：找出所有冲突对，画出有向边。**

逐个数据项分析：

- **关于数据项 X 的冲突**：
  - T1 先执行 `R(X)`，T2 后执行 `W(X)` → T2 的写会改变 T1 读到的值，冲突。边：**T1 → T2**
  - T2 先执行 `W(X)`？注意，T2 对 X 的操作是 `R(X)` 然后没有写，T3 有 `W(X)`。再看顺序：T1 先 `W(X)`，T3 后 `W(X)` → 写写冲突，T1 的写先于 T3。边：**T1 → T3**
  - T2 先 `R(X)`，T3 后 `W(X)` → T3 的写改变了 T2 曾经读到的 X 值，冲突。边：**T2 → T3**

- **关于数据项 Y 的冲突**：
  - T2 先 `R(Y)`（第一次），后来 T2 又 `W(Y)`。T2 内部顺序自然满足。
  - T3 没有对 Y 的操作，T1 也没有。
  - 但 T2 和后面对 Y 有写操作的事务有冲突。这里只有 T2 自己读写 Y，暂不产生跨事务的 Y 依赖边。

- **关于数据项 Z 的冲突**：
  - T3 先 `W(Z)`（第一次），T1 后 `W(Z)` → 写写冲突。边：**T3 → T1**
  - T1 先 `W(Z)`，T3 后 `W(Z)`（第二次） → 写写冲突。边：**T1 → T3**（这条前面已经有了）
  - 实际上要看完整的顺序：T3 第一次 `W(Z)` 在 T1 的 `W(Z)` 之前，所以 **T3 → T1** 是关键边。

**第二步：汇总有向边。**

从上面分析，我们得到三条关键边：
- T1 → T2 （来自 X 的读写冲突）
- T2 → T3 （来自 X 的读写冲突）
- T3 → T1 （来自 Z 的写写冲突）

**第三步：检查环路。**

这三条边形成了：**T1 → T2 → T3 → T1**，这是一个完整的环。

PPT 右侧画出的图清晰地展示了这个环，并给出了结论：

> The schedule is not serializable because there is a circle T1, T2, T3, T1. Therefore the schedule will fail under any protocol that aims at conflict serializability.

**结论：该调度不是冲突可串行化的。** 因为它存在环 T1→T2→T3→T1，意味着这三个事务相互依赖、谁也等不了谁，任何等价串行执行顺序都找不到。任何以冲突可串行化为正确性标准的并发控制协议，都必须拒绝这种调度。

### 第3页：并发控制 vs. 可串行化测试——工程上的关键取舍

这一页点出了一个非常重要的工程实践问题，把前面所有理论点串联起来。

**原文要点**：
> - Testing a schedule for serializability after it has executed is too late!
> - Goal – to develop concurrency control protocols that will assure serializability.
> - They will generally not examine the precedence graph as it is being created; instead a protocol will impose a discipline that avoids nonserializable schedules.

**详细解读**：

- **事后检测来不及**：
  等调度全部执行完，再画优先图、检测环，这种事后的分析在现实数据库系统中毫无意义。因为如果检测出不可串行化，说明数据库中已经产生了不一致的数据，事务可能已经基于脏数据提交了，错误已经发生且很难挽回。

- **真正的目标**：
  不是在调度执行完后判断它是否正确，而是要设计一套**并发控制协议**，这套协议能**保证**系统在任何情况下**只产生可串行化的调度**。相当于在源头就堵死了错误调度的可能性。

- **协议怎么做**：
  这些协议（比如两段锁协议、时间戳排序）一般**不会实时地画优先图、检测环**——那样的开销太大。它们采取的策略是**施加一种纪律**：给每个事务的操作加上严格的规则（比如访问数据前必须加锁，释放锁后不能再加锁），只要所有事务都遵守这个纪律，系统产生的并发调度**自然就是可串行化的**，根本不需要事后检查。

**打个比方**：
- **事后检测**：像开车到了悬崖边才发现走错了路，已经掉下去了。
- **并发控制协议**：像在悬崖很远的地方就设了护栏和路标，让所有车根本开不到危险地带。

---

### 总结：这三页的逻辑

1. **给考题**（第1页）：一个三事务的并发调度，问你能不能判定它正确。
2. **做分析**（第2页）：用优先图工具一步步分析冲突、画边，发现环路，判定它**不是冲突可串行化的**。
3. **论方法**（第3页）：指出这种事后分析没意义，数据库系统需要的是**事前预防**——即下一阶段要学习的各种并发控制协议，它们通过规则约束，从根本上杜绝非可串行化调度的产生。
