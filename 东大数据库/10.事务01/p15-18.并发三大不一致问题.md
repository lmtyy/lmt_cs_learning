这四页PPT围绕 **“事务并发执行带来的数据不一致问题”** 展开，前三页分别列举了三种经典的不一致场景，第四页引出解决这些问题的方案——**并发控制**。

---

### 第1页：丢失修改（Lost Update）——写写冲突

这一页通过飞机订票系统展示了 **“丢失修改”** 问题。

**场景表格：**

| 事务 T₁     | 事务 T₂   |
| ----------- | --------- |
| ① 读 A=16   |           |
|             | 读 A=16   |
| ② A ← A-1   |           |
|             | A ← A-3   |
| ③ 写回 A=15 |           |
|             | 写回 A=13 |

**执行过程分析：**
- T₁ 和 T₂ 先后读到了 `A=16`（初始库存票数）。
- T₁ 要减去1张票，将 `15` 写回数据库。
- T₂ 也要减去3张票，但因为它在 T₁ 写回之前已经读了旧值 `16`，于是计算 `16-3=13`，并将 `13` 写回。
- T₂ 的写回 **直接覆盖了** T₁ 刚写回的 `15`。

**结果：** 数据库最终是 `13` 票，而正确的并发结果应该是 `16 - 1 - 3 = 12`。T₁ 的那一次减票操作就像根本没发生过一样，被 T₂ 覆盖掉了。

- **本质**：两个事务 **同时写同一个数据项**，后提交的事务把前一个事务的更新覆盖了。
- **别名**：写-写冲突（Write-Write Conflict），PPT中英文标注为 `lost update (write write conflict)`。
- **危害**：数据不一致，比如票多卖了，但系统记录的库存却没真实减少。

### 第2页：不可重复读（Non-repeatable Read）——读写冲突

这一页展示了 **“不可重复读”** 问题。

**场景表格：**

| T₁                                       | T₂                              |
| ---------------------------------------- | ------------------------------- |
| ① 读 A=50，读 B=100，求和=150            |                                 |
|                                          | ② 读 B=100，B ← B*2，写回 B=200 |
| ③ 读 A=50，读 B=200，求和=250 (验算不对) |                                 |

**执行过程分析：**
- T₁ 先读取 A 和 B，此时 `A=50, B=100`，计算出和为 `150`。
- 这时 T₂ 对 B 进行了更新，将 B 翻倍为 `200` 并写回。
- T₁ 在稍后再次读取 A 和 B（比如为了做验算），发现 `A=50, B=200`，和变成了 `250`。

**结果：** T₁ 在同一个事务内，两次读取 **同一数据项**（或相关联的多个数据项）得到的结果不同。

- **本质**：事务 T₁ 两次读之间，T₂ 执行了 **更新并提交**，导致 T₁ 无法重现前一次读到的结果。
- **别名**：读-写冲突的一种表现，PPT中英文标注为 `non-repeatable read`。
- **危害**：如果 T₁ 是基于第一次读到的 `150` 做后续业务决策（比如判断总资产），第二次再读到的 `250` 会让它的逻辑出现混乱，验算不通过，数据可信度丧失。

### 第3页：脏读（Dirty Read）

这一页展示了 **“脏读”** 问题。

**场景表格：**

| T₁                              | T₂         |
| ------------------------------- | ---------- |
| ① 读 C=100，C ← C*2，写回 C=200 |            |
|                                 | ② 读 C=200 |
| ③ ROLLBACK，C 恢复为 100        |            |

**执行过程分析：**
- T₁ 将 C 从 `100` 修改为 `200` 并写回数据库，但此时 **尚未提交**。
- T₂ 此时读取 C，读到了 `200`。
- 随后 T₁ 因为某种原因执行了 **ROLLBACK**（回滚），C 的值被恢复为原来的 `100`。
- T₂ 手中握着的 `200` 就成了一个 **从未真正存在过** 的值。

**结果：** T₂ 读到了 T₁ 未提交的中间结果，这个数据随后被撤销了。

- **本质**：一个事务读到了另一个事务 **未提交的更新数据**。这种“不干净”的数据称为 **脏数据**。
- **别名**：脏读（Dirty Read）。
- **危害**：T₂ 如果基于这个 `200` 进行后续操作（比如转账、扣库存），那么它所依据的数据根本就是错误的、失效的，会导致严重的业务逻辑错误。

PPT中特别阐明了脏读的产生条件：
> 事务1修改某一数据，并将其写回磁盘  
> 事务2读取同一数据后，事务1由于某种原因被撤消，  
> 这时事务1已修改过的数据恢复原值，  
> 事务2读到的数据就与数据库中的数据不一致，是不正确的数据，又称为“脏”数据。

### 第4页：并发控制的引入

前三页用实例揭示了并发执行如果不加控制会引发的三类严重问题：丢失修改、不可重复读、脏读。第四页自然引出解决问题的关键——**并发控制**。

原文：
> Concurrency control schemes – mechanisms to achieve isolation, i.e., to control the interaction among the concurrent transactions in order to prevent them from destroying the consistency of the database  
> (problems occur when concurrent transactions access the same items).

**详细解读：**

- **并发控制方案**就是一套 **机制**，它的目标是 **实现事务的隔离性（Isolation）**。
- 隔离性是 ACID 特性之一，它要求并发执行的多事务相互隔离，使它们看起来像是 **串行执行** 一样，互不干扰。
- 并发控制的核心工作就是 **控制并发事务之间的交互**，从根本上防止它们访问相同数据项时产生那些破坏数据库一致性的问题（写写冲突、读写冲突等）。
- 常见并发控制技术包括：基于锁的协议、时间戳排序、多版本并发控制（MVCC）等，这些方法都是为了在允许多事务并发带来的高吞吐量的同时，又能避免前三页所示的种种不一致。

**总体逻辑线索：**
1. 事务并发执行是为了高性能（之前几页）。
2. 但随意并发会引发丢失修改、不可重复读、脏读（这三页）。
3. 所以必须引入并发控制机制来保证事务隔离，进而维护数据库一致性（这一页）。

后续课程通常会围绕 **封锁（Lock）、两段锁协议、隔离级别** 等内容展开，这四页PPT就是引出这些技术的前置铺垫。
