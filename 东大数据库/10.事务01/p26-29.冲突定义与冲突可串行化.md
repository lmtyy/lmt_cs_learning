这四页PPT从生动的比喻出发，逐步给出了**冲突可串行化**的严格定义和判定方法。这是判断一个并发调度是否正确（即是否等价于某个串行调度）的核心技术之一。

---

### 第1页：什么是冲突？——一个直觉比喻

PPT用了一个非常形象的例子，来说明并发操作中“顺序”什么时候重要、什么时候不重要。

- **比喻**：
  - 一幅画，**东坡题字**（写操作），**乾隆御览**（读操作），**顾童涂鸦**（写操作）。
  - 顺序1：东坡题字 → 乾隆御览 → 顾童涂鸦。结果：东坡的真迹被小孩涂鸦毁了，**成为废物**。
  - 顺序2：东坡题字 → 顾童涂鸦 → 乾隆御览。结果：小孩的涂鸦覆盖在东坡字上，乾隆看到的是涂鸦，但因为是古物，**最终成了文物**。
  - 相同的三个操作，**仅仅因为顺序不同，最终结果天差地别**。

- **直观解释**：
  - 如果两个操作都是**读**（欣赏画），比如两个人先后欣赏，谁先谁后，看到的东西都一样，**不发生冲突**。
  - 但只要其中一个操作是**写**（改画），先后次序就变得**至关重要**。写的时机和顺序，会直接改变其他操作看到的内容，进而影响最终结果。

这就引出了数据库并发控制中对“冲突”的严谨定义：

> 多个事务访问同一数据项时，如果操作全是读，无所谓顺序；如果至少有一个写，操作的先后顺序就会影响数据库状态和事务看到的返回值，这种操作对就叫**冲突**。

---

### 第2页：冲突的形式化定义

这一页给出了判断**两条指令是否冲突**的严格标准。

设指令 `l_i` 属于事务 `T_i`，指令 `l_j` 属于事务 `T_j`。

**冲突的充要条件**：
1. 两条指令访问的是**同一个数据项 Q**；
2. 且**至少有一条指令是 write(Q)**。

据此可列出四种情况：

| `l_i`    | `l_j`    | 是否冲突   | 原因                                                         |
| -------- | -------- | ---------- | ------------------------------------------------------------ |
| read(Q)  | read(Q)  | **不冲突** | 都是读，不会改变数据，无论谁先执行，读到的值都一样。         |
| read(Q)  | write(Q) | **冲突**   | 写操作会改变数据值，读操作的结果取决于它在写之前还是之后执行。 |
| write(Q) | read(Q)  | **冲突**   | 同上，读到的值是写之前还是之后的值，完全由顺序决定。         |
| write(Q) | write(Q) | **冲突**   | 两次写会先后覆盖，最终数据库里保留的是后执行的那个结果，顺序直接影响数据的最终状态。 |

**重要推论**：
> 如果在一个调度中，两条相邻的指令是**不冲突**的，那么交换它们的执行顺序，**不会改变调度的最终效果**。

这个推论是判断冲突可串行化的操作基础——我们只能通过交换**不冲突**的相邻指令，把一个并发调度“变形”成串行调度，而不会改变它的语义。

### 第3页：冲突可串行化的定义与反例

这一页给出了**冲突可串行化**的判断标准，并举了一个不满足条件的例子。

**定义**：
> 如果一个调度 S **可以通过一系列交换相邻非冲突指令的操作**，转换为一个**串行调度**，那么我们说调度 S 是**冲突可串行化**的。

这个定义非常操作化：拿到一个并发调度，你只需要反复寻找相邻的、不冲突的两条指令，交换它们的位置。如果能最终让同一个事务的所有指令都连在一起，形成“一个事务执行完再执行下一个”的串行形式，那么原调度就是冲突可串行化的，也就是**正确的**。

**非冲突可串行化的反例**（PPT中给出的例子）：
PPT的表格里只画出了事务 T₃ 的三条指令：
```
T₃ : read(Q)
     write(Q)
     write(Q)
```
而文字说明指出：“我们无法通过交换指令，从上面的调度得到串行调度 `< T₃, T₄ >` 或 `< T₄, T₃ >`。” 这里显然是省略了 T₄ 的操作。实际上，如果一个调度只包含一个事务，它本身就已经是串行调度，不存在可串行化问题。这个例子的本意是展示一个包含冲突操作的调度，例如 T₃ 和 T₄ 对 Q 都有读写，且顺序已被冲突锁死，导致无论如何交换相邻非冲突指令，都无法让 T₃ 和 T₄ 的指令各自汇合在一起，因此**不是冲突可串行化的**。

### 第4页：冲突可串行化实例

这一页展示了一个**正例**，说明如何通过交换非冲突指令来证明一个调度是冲突可串行化的。

PPT中的表格将两个事务的操作对齐列出：
```
T₁: read(A)   T₂: read(A)
    write(A)       write(A)
    read(B)        read(B)
    write(B)       write(B)
```
文字说明：“下面的调度通过一系列交换非冲突指令，等价于 T₂ 跟随 T₁ 的串行调度（即先完整执行 T₁，再完整执行 T₂）。”

这里需要理解的是，表格所展示的**并不是原始的并发交错调度**，而是**已经通过交换非冲突指令之后得到的最终形态**，也就是目标串行调度。例如，原始的并发调度可能是：
```
r₁(A)  r₂(A)  w₁(A)  w₂(A)  r₁(B)  r₂(B)  w₁(B)  w₂(B)
```
这是一个典型的完全交错调度。但仔细看，若要把它变成“先 T₁ 全部，后 T₂ 全部”的串行调度，需要交换很多指令，但 `w₁(A)` 和 `r₂(A)` 冲突，`r₁(B)` 和 `w₂(A)` 呢？它们操作不同数据项（一个 B 一个 A），不冲突，可以交换。通过一系列可以交换的操作，某些交错调度确实可以变换为 T₁ 全在 T₂ 之前的顺序。PPT上直接呈现出这个最终的串行顺序，就是在说：**瞧，通过若干次合法交换，我们得到了这个串行形式，因此原调度是冲突可串行化的。**

### 总结：从直觉到方法

这四页的逻辑线非常清晰：
1. **直觉建立**：用东坡题字的故事说明，读读不冲突，只要有一个写，顺序就决定一切。
2. **严格定义**：给出 read/write 冲突的判定表，并建立“不冲突可以交换”这个核心工具。
3. **冲突可串行化定义**：如果一个调度能通过合法交换变成串行的，它就是冲突可串行化的；并给出反例说明存在无法变换的调度。
4. **实例证明**：用一个具体调度展示，只要存在一条连续交换不冲突指令的路径到达串行调度，它就是正确的，系统可以放行。

冲突可串行化是**保证并发调度正确性**的最实用标准。后续的两段锁协议（2PL）等并发控制技术，本质上就是在保证系统只产生冲突可串行化的调度。
