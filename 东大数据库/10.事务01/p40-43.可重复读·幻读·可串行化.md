### 第1页：可重复读——解决不可重复读

这一页针对上一组PPT中“读已提交”无法解决的不可重复读问题，引入了**可重复读**隔离级别。

**核心定义**：
> 在同一个事务内，**先后两次读到的同一条记录的内容必须保持不变**。

换言之，一个事务一旦读取了某行数据，其他事务在该事务结束前，就不能修改这行数据。

**实现方式**（PPT提到两种典型思路）：

- **基于锁**：事务在读取数据时，就对所读行加**共享锁**并持有到事务结束。其他事务要修改这些行，必须获取排他锁，因此会被阻塞，直到读事务提交释放锁。这就杜绝了两次读之间数据被篡改的可能。
- **基于 MVCC（多版本并发控制）**：与读已提交的MVCC不同，可重复读级别下，一个事务在整个生命周期内都使用**同一个快照版本**，而不是每次读都去找最新已提交版本。因此，即使其他事务修改并提交了新版本，当前事务第二次读时，看到的仍是事务开始时的那个旧快照，值自然不变。

PPT中的表格示意：
- 事务A读取了`小明`的存款（可能加了读锁或有快照）。
- 事务B试图更新`小明`的存款，但**等待锁释放**——直观展示了锁机制的阻塞效果。

**遗留问题**：可重复读通过锁定“已有行”防止了不可重复读，但**仍然允许幻读**。

### 第2页：幻读——可重复读无法解决的问题

这一页正式定义并举例说明了**幻读**异常。

**定义**：
> 同一个事务内，先后两次执行**相同条件的查询**，却发现**结果集的行数发生了变化**（多了或少了若干行）。

这与不可重复读有本质区别：
- **不可重复读**：同一行数据的**值**被改了。
- **幻读**：符合查询条件的**行集合本身**变了（有插入或删除）。

**具体案例（银行系统）**：

| 时刻 | 事务A                              | 事务B                  |
| ---- | ---------------------------------- | ---------------------- |
| t₁   | 统计银行总存款 → 100               |                        |
| t₂   |                                    | 加入新储户 `小张，100` |
| t₃   |                                    | 提交                   |
| t₄   | 再次统计总存款 → **200，不一致！** |                        |

- 事务A两次统计总存款，条件都是“所有人”。
- 第一次统计时只有旧储户，总和100。
- 事务B插入并提交了新储户小张。
- 第二次统计时，小张这条新行也被纳入计算，总和变成了200。
- 事务A读到的并不是某一行值变了，而是**凭空多出了一行**——像出现了“幻觉”，因此称为幻读。

**为什么可重复读挡不住幻读**：可重复读只锁住了已存在的行（或使用快照），但无法阻止其他事务**插入新行**。新行没有锁可以提前加，快照也不能预知新行的出现。

### 第3页：可串行化——解决幻读

这一页引入了最高隔离级别**可串行化**，并说明它如何杜绝幻读。

**核心定义**：
> 可串行化是**最严格的隔离级别**，它要求并发事务的调度必须是**可串行化调度**，执行效果与这些事务按某种顺序**一个接一个串行执行**完全相同。

这已经不是单纯禁止某几类异常，而是从调度等价性的根本上保证完全隔离。

**解决幻读的方法**（PPT以锁为例）：

- 仅仅锁住已存在的行不够，还需要使用更复杂的锁策略，比如**间隙锁（Gap Lock）**或**谓词锁（Predicate Lock）**。
- 事务A在第一次统计时，不仅锁住所有现有行，还会锁定这些行之间的“间隙”或整个查询条件范围。这可以阻止其他事务在满足同样条件的范围内插入新行。

**案例分析**：

| 时刻 | 事务A                | 事务B                      |
| ---- | -------------------- | -------------------------- |
| t₁   | 统计银行总存款 → 100 |                            |
| t₂   |                      | 加入新储户小张（**阻塞**） |
| t₃   | 再次统计总存款 → 100 |                            |
| t₄   | 提交                 |                            |
| t₅   |                      | （加入新储户小张成功）     |
| t₆   |                      | 提交                       |

- 事务A在 t₁ 执行范围查询时，锁住了整个表或相关范围。
- 事务B在 t₂ 试图插入新行，因为与A的间隙锁冲突，被**强制阻塞**。
- 事务A在 t₃ 再次统计，因为阻塞存在，没有任何新行插入，读到仍是100。
- 等到事务A提交释放锁，事务B才得以继续执行。
- 这样，事务A的两次统计结果完全一致，幻读被彻底消除。

**可串行化=完全隔离=等价于某种串行执行。**

### 第4页：隔离级别的权衡与总结

这一页用一个**斜坡图**对四个隔离级别做了最终总结。

**核心权衡维度**：

| 隔离级别 | 数据安全性（一致性） | 并发程度 |
| -------- | -------------------- | -------- |
| 读未提交 | 最弱                 | 最高     |
| 读已提交 | ↑ 渐强               | ↓ 渐低   |
| 可重复读 | ↑ 渐强               | ↓ 渐低   |
| 可串行化 | 最强                 | 最低     |

- **数据安全性与并发程度是一对根本矛盾**：隔离级别越高，系统为阻止各种异常而加的锁或检查越多，事务等待时间越长，能同时并发执行的事务数量就越少。
- 反过来，隔离级别越低，并发吞吐量越大，但数据异常的风险也越高。

**实践指导**：
> **根据具体业务场景选择合适的隔离级别。**

- 银行核心转账、订单扣库存等**对一致性要求极高**的场景，通常需要可重复读甚至可串行化。
- 报表统计、日志分析等**对并发和性能要求高、能容忍少量不一致**的场景，可能只用读已提交就足够。
- **没有银弹**：理解这四层的区别和代价，根据业务需要做权衡，是数据库应用设计的重要一环。
