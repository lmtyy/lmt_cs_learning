### 第1页：带锁转换的两段锁协议——正式总结

这一页是对上一轮锁转换讨论的正式收束，用清晰的阶段划分，定义了带锁转换后2PL允许的操作。

**两阶段划分**：

-   **Phase 1: 扩展阶段（Growing Phase）**
    -   可以申请共享锁（lock-S）
    -   可以申请排他锁（lock-X）
    -   可以将共享锁升级为排他锁（upgrade）
    
    > 这一阶段锁的控制力只增不减。
    
-   **Phase 2: 收缩阶段（Shrinking Phase）**
    -   可以释放共享锁（lock-S）
    -   可以释放排他锁（lock-X）
    -   可以将排他锁降级为共享锁（downgrade）
    
    > 这一阶段锁的控制力只减不增。

**要点**：升级只能发生在扩展阶段，降级只能发生在收缩阶段。这等价地保持了2PL的核心纪律——在扩展阶段绝不释放任何锁（降级视为部分释放），在收缩阶段绝不获取任何更强的锁（升级视为获取更强的锁）。

### 第2页：级联回滚问题

这一页用一个具体的并发调度实例，揭示了基本2PL协议无法避免的**级联回滚（Cascading Rollback）**现象。

**场景调度**：

| T₉                 | T₁₀                | T₁₁                |
| ------------------ | ------------------ | ------------------ |
| lock-X(A), read(A) |                    |                    |
| lock-S(B), read(B) |                    |                    |
| write(A)           |                    |                    |
| **unlock(A)**      |                    |                    |
|                    | lock-X(A), read(A) |                    |
|                    | write(A)           |                    |
|                    | **unlock(A)**      |                    |
|                    |                    | lock-S(A), read(A) |

PPT上的文字说明直指问题核心：
> 如果在事务 T₁₁ 的 read() 指令执行之后事务 T₉ 发生故障，从而导致事务 T₁₀ 与 T₁₁ 的级联回滚。

**逐步解析**：

1.  **T₉**：修改A 并写入后，**在事务提交前就释放了A的锁**。
2.  **T₁₀**：在 T₉ 尚未提交时，基于 T₉ 写入的 A 值进行了读写，并同样在提交前释放了锁。
3.  **T₁₁**：又在 T₁₀ 未提交时，读取了 A 的值。

**故障时的连锁反应**：

-   假设 T₉ 在稍后某个时刻发生故障，必须回滚。
-   它写入的 A 值被撤销，恢复为旧值。
-   而 **T₁₀ 读到的、基于它写入的 A 值就成了脏数据**，因此 T₁₀ 也必须回滚。
-   同样，**T₁₁ 又从 T₁₀ 那里读取了脏数据**，因此 T₁₁ 也必须回滚。
-   这种“一人出错，连累所有读过它未提交数据的事务”的现象，就是**级联回滚**。

**根本原因**：基本2PL协议只规定了锁的申请和释放顺序，但**并没有强制规定排他锁必须保留到事务提交之后**。事务可以在提交前就释放锁，导致其他事务读到了未提交的“脏数据”。

**引出后续概念**：级联回滚会导致大量事务工作白费，系统开销巨大。为了解决它，需要更严格的2PL变体——比如**严格两段锁**要求排他锁必须持有到事务提交；**强两段锁**要求所有锁都持有到事务提交。这样其他事务根本看不到未提交的数据，彻底杜绝级联回滚。
