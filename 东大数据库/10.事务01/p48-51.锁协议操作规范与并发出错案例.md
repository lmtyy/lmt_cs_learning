这四页PPT围绕**基于锁的并发控制协议**，用银行转账的具体案例，展示了简单加锁操作在实际并发调度中可能遇到的问题。

---

### 第1页：锁协议的基本操作规范

这一页是对之前锁概念的进一步细化，明确事务使用锁时的标准操作和规则。

**三种基本操作：**

- **LOCK-S(Q)**：事务申请数据项 `Q` 上的**共享锁**，表示打算读 `Q`。
- **LOCK-X(Q)**：事务申请数据项 `Q` 上的**排他锁**，表示打算写 `Q`。
- **UNLOCK(Q)**：事务释放数据项 `Q` 上的锁。

**三条核心规则：**

1. **要存取必须先加锁**：事务 `T` 要对数据项 `Q` 执行任何操作，必须先成功申请到相应的锁。
2. **遇冲突锁必须等待**：如果 `Q` 已被其他事务加了与当前请求不相容的锁，`T` 必须等待，直到那些锁全部释放。
3. **事务可自行释放锁**：事务可以在任何时候释放它所持有的某个数据项上的锁。

这页定义的是一套“自由加锁/解锁”的规则，即只要求冲突时等待，但没有强制规定锁必须在什么时候释放。后面的例子会证明，仅有这些规则是不够的。

### 第2页：银行转账例子

这一页给出两个事务的完整定义，用锁操作和读写操作的形式写出来。

**业务场景：**
- 账户 `A=100`，`B=200`
- `T₇`：从 B 转 50 到 A（两个写操作）
- `T₈`：显示 A 和 B 的总金额（两个读操作）

**T₇ 的操作序列：**
```
LOCK-X(B)   → 申请B的排他锁
READ(B)     → 读B
B := B - 50
WRITE(B)    → 写回B
UNLOCK(B)   → 释放B的锁

LOCK-X(A)   → 申请A的排他锁
READ(A)     → 读A
A := A + 50
WRITE(A)    → 写回A
UNLOCK(A)   → 释放A的锁
```

**T₈ 的操作序列：**
```
LOCK-S(A)   → 申请A的共享锁
READ(A)     → 读A
UNLOCK(A)   → 释放A的锁

LOCK-S(B)   → 申请B的共享锁
READ(B)     → 读B
UNLOCK(B)   → 释放B的锁
DISPLAY(A+B) → 显示总和
```

注意两个事务的锁释放方式：
- `T₇` 是**用完一个数据项就立刻释放锁**（写B完后立刻解B，写A完后立刻解A）。
- `T₈` 也是同样的方式，读完A立刻解A锁，再去加B锁。

### 第3页：调度1——串行执行

这一页指出：**如果这两个事务是串行执行的，结果永远正确。**

- 串行顺序 `T₇, T₈`（先转账再查询）：`T₈` 会看到 A 和 B 都是 `T₇` 更新后的值，总和是 `150+150=300`。
- 串行顺序 `T₈, T₇`（先查询再转账）：`T₈` 会看到 A 和 B 的初始值，总和也是 `100+200=300`。

> **关键结论：不管串行顺序如何，T₈ 永远显示 300 元，正确。**

这一页为下一面出现的错误并发调度做了铺垫。

### 第4页：调度2——并发执行出错

这一页是重点，展示了一个**违反可串行化、结果错误**的并发执行。

PPT没有用传统的表格，而是用时间线的方式描述。整理成表格如下：

| T₇                                                   | T₈                                            |
| ---------------------------------------------------- | --------------------------------------------- |
| LOCK-X(B), READ(B), B:=B-50, WRITE(B), **UNLOCK(B)** |                                               |
|                                                      | LOCK-S(A), READ(A) → **读到A=100**, UNLOCK(A) |
|                                                      | LOCK-S(B), READ(B) → **读到B=150**, UNLOCK(B) |
|                                                      | DISPLAY(A+B) → **显示250**                    |
| LOCK-X(A), READ(A), A:=A+50, WRITE(A), UNLOCK(A)     |                                               |

**分步解析发生了什么：**

1. `T₇` 先锁 B，更新 B 从 200 到 150，然后释放了 B 的锁。
2. `T₈` 此时可以申请 A 的共享锁，读到 `A=100`，然后释放 A 的锁。
3. `T₈` 接着申请 B 的共享锁，读到 `B=150`（这是 `T₇` 刚刚更新的值），然后释放 B 的锁。
4. `T₈` 显示总和 `100 + 150 = 250`。
5. `T₇` 此时还没有更新 A，后面才继续执行更新 A 的操作。

**为什么会出错？**

- `T₈` 读到了不一致的数据：A 是 `T₇` 更新**之前**的值（100），B 却是 `T₇` 更新**之后**的值（150）。
- `T₇` 的转账操作本身应该是原子的——A减50和B加50应当被看作一个整体，要么全看到，要么全看不到。但 `T₈` 却只看到了一半的转账。
- 根源在于 `T₇` **过早地释放了 B 的锁**，使得 `T₈` 可以在这个间隙里“偷窥”到不一致的中间状态。

**最后的显示结果是 250 元，而非正确的 300 元。**

### 总结：引出两段锁协议

这四页的逻辑递进：

1. **第1页**：建立了“存取前加锁、冲突时等待”的基本规则。
2. **第2页**：定义一个转账事务和一个查询事务，两者都使用“用完即解锁”的策略。
3. **第3页**：串行执行没问题。
4. **第4页**：并发执行下，由于事务过早释放锁，导致其他事务读到了不一致的中间状态，最终结果错误。

这直接引出一个关键问题：**简单的加锁/解锁操作，如果不在释放锁的时机上加任何限制，仍然会产生不可串行化的调度。**

因此，后面就需要引入**两段锁协议**：它强制要求每个事务在释放**任何**锁之后，不能再申请**任何新锁**。这样就能避免像 `T₇` 那样“先放B锁再拿A锁”的情况，从而保证并发调度一定是可串行化的。
