这三页PPT通过正反实例对比，验证了两段锁协议的判定规则，并最终揭示了2PL与冲突可串行化之间的理论关系。

---

### 第1页：反面案例——不满足2PL

这一页把之前出错的 `T₇` 和 `T₈` 重新拿出来，用2PL的规则来审视。

**T₇ 的操作序列**：
```
LOCK-X(B) → READ(B) → B:=B-50 → WRITE(B) → UNLOCK(B)
→ LOCK-X(A) → READ(A) → A:=A+50 → WRITE(A) → UNLOCK(A)
```

**T₈ 的操作序列**：
```
LOCK-S(A) → READ(A) → UNLOCK(A)
→ LOCK-S(B) → READ(B) → UNLOCK(B)
→ DISPLAY(A+B)
```

**违反2PL的判断依据**：

- **T₇**：在释放了B的锁（`UNLOCK(B)`）之后，又去申请了A的锁（`LOCK-X(A)`）。这明显违反了2PL的第二条规则——“在释放一个封锁之后，事务不再获得任何其他封锁”。T₇出现了**释放→再加锁**的模式。

- **T₈**：同样，先释放了A的共享锁（`UNLOCK(A)`），然后又去申请B的共享锁（`LOCK-S(B)`）。也是典型的**释放→再加锁**，同样违反2PL。

**结论**：`T₇` 和 `T₈` 都不满足两段锁协议。与此对应，前面我们已经看到它们产生的并发调度导致了读到250元的错误，是不可串行化的。

### 第2页：正面案例——满足2PL

这一页展示了改进后的 `T₉` 和 `T₁₀`，并明确标注 **Yes**，表示它们通过了2PL的检查。

**T₉ 的操作序列**：
```
LOCK-X(B) → READ(B) → B:=B-50 → WRITE(B)
→ LOCK-X(A) → READ(A) → A:=A+50 → WRITE(A)
→ UNLOCK(B) → UNLOCK(A)
```

**为什么满足？**
- 所有加锁操作（`LOCK-X(B)`、`LOCK-X(A)`）都在前面。
- 所有解锁操作（`UNLOCK(B)`、`UNLOCK(A)`）都集中在最后。
- 加锁阶段一气呵成，没有“释放后再加锁”的情况。严格按照“先扩展、后收缩”的顺序。

**T₁₀ 的操作序列**：
```
LOCK-S(A) → READ(A)
→ LOCK-S(B) → READ(B)
→ DISPLAY(A+B)
→ UNLOCK(A) → UNLOCK(B)
```

**为什么满足？**
- 同样是先申请了所有需要的锁（`LOCK-S(A)`、`LOCK-S(B)`）。
- 等所有读操作和计算都完成后，才进行解锁。
- 也完全符合“生长阶段 → 收缩阶段”的模式。

**结论**：`T₉` 和 `T₁₀` 都严格遵循了两段锁协议。与之对应，它们能够保证并发调度不会出现读到不一致中间状态的问题，其结果等价于某种串行执行。

### 第3页：2PL与冲突可串行化的理论关系

这一页是2PL理论的点睛之笔，阐明了2PL协议和冲突可串行化调度之间的精确关系。

**命题一：2PL产生的调度必然是冲突可串行化的。**

> If a schedule is executed by 2PL it must be conflict serializable.

- 这是2PL的**正确性保证**。如果系统中所有事务都遵守2PL纪律，那么无论这些事务的操作如何交叉执行，形成的并发调度都一定是冲突可串行化的。这是可以严格证明的定理。

**命题二：可串行化顺序由“锁点”决定。**

> It can be proved that the transactions can be serialized in the order of their lock points (i.e. the point where a transaction acquired its final lock).

- **锁点**：事务获得**最后一把锁**的那个瞬间。
- 举例：`T₉` 在 `LOCK-X(A)` 成功的那一刻获得最后一把锁，在此之前它已经持有B的锁。这个瞬间就是 `T₉` 的锁点。
- 意义：如果我们把所有并发事务按照它们锁点的时间先后排个序，那么这个顺序就是它们等价的那个串行执行顺序。锁点机制提供了一个非常实用的方法来确定“到底等价于哪个串行顺序”。

**命题三：2PL是充分不必要条件。**

> If a schedule is conflict serializable it may or may not be executed by 2PL.

- 这句话揭示了2PL在理论上的局限性：
  - 所有2PL能产生的调度，都是冲突可串行化的（**充分性**成立）。
  - 但是，**并不是所有**冲突可串行化调度都能通过2PL产生（**必要性不成立**）。
- 直观理解：2PL是一种**限制性纪律**，它为了确保正确性，把一些“虽然正确、但不按2PL出牌”的调度也挡在了门外。2PL比“冲突可串行化”这个标准更加严格、更加保守。

**用集合关系来理解**：
- 所有可能的调度是一个大圈。
- 冲突可串行化调度是里面的一个圈。
- 2PL能产生的调度是更里面的一个小圈。
- 2PL ⊂ 冲突可串行化调度 ⊂ 所有调度

### 总结

这三页的逻辑清晰有力：
1. 用旧案例（`T₇`、`T₈`）证实它违反2PL且结果错误。
2. 用新案例（`T₉`、`T₁₀`）证实它遵守2PL且结果正确。
3. 从理论上精确界定2PL的地位：**2PL是保证可串行化的充分但不必要条件**。实践中我们愿意接受这种“多余的限制”，因为它简单、可操作、易于实现，足以保证数据库的一致性。
