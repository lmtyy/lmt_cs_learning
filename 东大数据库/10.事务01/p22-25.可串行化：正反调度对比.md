这四页PPT是一个整体，通过**正反对比**的方式，引出了数据库并发控制理论中最重要的概念——**可串行化**。

---

### 第1页：不是串行，但是正确的调度

这一页展示了：**不按串行顺序执行，但结果完全正确**的调度。

**场景与之前一样**：
- T1：从A转账50到B
- T2：从A转账10%到B
- 初始：A=100, B=200

**调度执行过程**：

| 顺序 | T1          | T2                    |
| ---- | ----------- | --------------------- |
| 1    | read(A)     |                       |
| 2    | A := A - 50 |                       |
| 3    | write(A)    |                       |
| 4    |             | read(A) → 读到A=50    |
| 5    |             | temp := A * 0.1 = 5   |
| 6    |             | A := A - temp → A=45  |
| 7    |             | write(A)              |
| 8    | read(B)     |                       |
| 9    | B := B + 50 |                       |
| 10   | write(B)    |                       |
| 11   |             | read(B) → 读到B=250   |
| 12   |             | B := B + temp → B=255 |
| 13   |             | write(B)              |

**最终结果**：A=45, B=255，总和保持300。

**为什么说它“不是串行”但“正确”？**
- 它不是串行调度：T1并没有全部执行完再执行T2，两个事务的指令是交叉的。
- 它等价于前文讲过的串行调度：这个调度中，T2读到的A是50（已经是T1更新过的），T2读到的B也是250（也是T1更新过的）。从效果上看，**等价于先完整执行T1，再完整执行T2**。
- PPT原文特别指出：`equivalent to the previous serial schedule`，这里的"previous serial schedule"指的正是上一组PPT中 T1→T2 的串行执行。

这一页的关键启示是：**并发调度不一定非要和某个串行调度长得一模一样，只要最终效果等价，它就是正确的。**

---

### 第2页：不是串行，并且不正确的调度

这一页展示了一个**同样是并行调度，但结果错误**的例子。

**同样场景，但交错方式不同**：

| 顺序 | T1          | T2                    |
| ---- | ----------- | --------------------- |
| 1    | read(A)     |                       |
| 2    | A := A - 50 |                       |
| 3    |             | read(A) → 读到A=100   |
| 4    |             | temp := A * 0.1 = 10  |
| 5    |             | A := A - temp → A=90  |
| 6    |             | write(A)              |
| 7    |             | read(B) → 读到B=200   |
| 8    | write(A)    |                       |
| 9    | read(B)     |                       |
| 10   | B := B + 50 |                       |
| 11   | write(B)    |                       |
| 12   |             | B := B + temp → B=210 |
| 13   |             | write(B)              |

**最终结果**：A=50, B=210，总和为260，不等于300。

**为什么这个并行调度是错误的？**
- 问题出在**第3步**：T2在T1写回A之前就读取了A，读到了旧值100。
- T2基于旧值`A=100`算出`temp=10`，并把A更新为90写回。
- 随后T1在第8步把自己的更新`A=50`写回，**覆盖了T2的A=90**——这正是典型的“丢失修改”问题。
- T2基于旧值算出的`temp=10`本身也是错的，导致B的更新也跟着出错。
- 最终总和260与初始总和300不一致，破坏了数据库的一致性。

PPT明确指出：`does not preserve the value of the sum A + B - The schedule is wrong and should not be allowed`。这个调度是**错误的，数据库系统必须阻止这种调度发生**。

---

### 第3页：事务调度与可串行性——问题的提出

这一页是**承上启下的关键总结**，提出了核心矛盾和目标。

- **回顾已知事实**：
  - 每个事务单独运行时没有问题（这是事务的基本假设）。
  - **串行调度一定产生正确结果**。前面已证明，不管串行顺序是T1→T2还是T2→T1，总和都保持一致。
  - 但串行调度**限制了并发性**，系统吞吐量低。

- **揭示核心矛盾**：
  - **并行调度可能导致不正确的结果**！第2页的调度就是铁证。

- **提出核心目标与研究问题**：
  - 我们既想要并行调度的高效率，又想要串行调度的正确性。
  - 所以需要一种系统方法来判断：**一个并行调度是“正确”还是“错误”？**
  - 这个判断标准的理论基础就是 **可串行性理论**——如果某个并行调度的效果“等价于”某个串行调度，那它就是正确的，数据库就可以放心执行它。

---

### 第4页：可串行化的正式定义

这一页给出了“可串行化”的形式化定义和基本假设。

- **基本假设**：每个事务单独运行时，能够保持数据库的一致性。这是ACID特性中C（一致性）和I（隔离性）共同作用的前提。

- **推理**：既然单个事务保持一致性，那么**一组事务的串行执行也必然保持一致性**。因为串行执行就是把一个事务的一致性保持结果，作为下一个事务的初始状态，链式传递，自然不会出错。

- **核心定义**：
  > A (possibly concurrent) schedule is **serializable** if it is **equivalent** to a serial schedule.
  
  **如果一个（可能是并行的）调度等价于某个串行调度，那么这个调度就是“可串行化”的。**
  
  这就是判断并发调度是否正确的黄金标准。第1页的并行调度“可串行化”于 T1→T2 的串行调度，所以正确。第2页的并行调度找不到任何一个效果等价的串行调度，所以错误。

- **简化模型**：为了集中研究调度的顺序问题，我们简化事务模型：
  - **只关注 `read` 和 `write` 两种操作**，忽略事务内部在本地缓冲区做的复杂计算。
  - 这是因为数据不一致问题，根源只在于多个事务对**同一数据项**的读写操作的**顺序冲突**。内部计算不出缓冲区，不会直接影响数据库状态。

---

### 总结：这四页的完整逻辑线

1.  **举正例**（第1页）：一个并行调度，因为操作交叉顺序合理，效果等价于某个串行调度，结果是正确的。
2.  **举反例**（第2页）：另一个并行调度，因为交叉顺序错误，导致丢失修改等并发问题，结果是错误的。
3.  **提出理论需求**（第3页）：必须找到一种方法，区分上面两种调度，让系统能“放行”正确的、阻止错误的。这就是可串行性理论。
4.  **给出核心定义**（第4页）：正式定义“可串行化”（等价于某个串行调度），并说明我们只需要研究`read/write`操作序列就能判定。

后续课程通常就会进入具体的实现层面，比如**冲突可串行化**、**冲突图检测**、**两段锁协议**等，都是围绕“如何保证调度是可串行化的”这一目标来展开的。
