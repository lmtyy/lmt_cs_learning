这三页PPT正式进入了数据库并发控制的具体实现技术——**基于锁的协议（Lock-Based Protocols）**。它从锁的基本概念讲起，然后定义锁的两种基本模式，最后给出锁相容矩阵来判断请求能否被满足。下面按顺序详细讲解。

---

### 第1页：封锁的基本概念

这一页给出了**锁**在数据库并发控制中的基本定义和作用。

**原文核心：**
> - A lock is a mechanism to control concurrent access to a data item
> - 封锁就是事务T在对某个数据对象操作之前，先向系统发出请求，对其加锁
> - 加锁后事务T就对该数据对象有了一定的控制，在事务T释放它的锁之前，其它的事务不能更新此数据对象。
> - 封锁是实现并发控制的一个非常重要的技术

**分点详解：**

- **锁是什么**：锁是一种**控制并发访问的机制**。可以把它理解为数据项上的“通行令牌”——只有拿到正确令牌的事务，才能对该数据项执行相应操作。

- **加锁的过程**：事务在对某个数据对象（表、记录、页等）进行操作之前，必须先向系统的**并发控制管理器**发出加锁请求。这是一个“申请-授权”的过程，事务不能绕过。

- **锁的效果**：事务T一旦成功加锁，就对该数据对象**有了一定的控制权**。在它释放锁之前，其他事务**不能更新**这个数据对象。这就从根本上杜绝了“丢失修改”这类写写冲突。

- **重要性强调**：PPt明确指出封锁是实现并发控制的**非常重要**的技术。后续讲的两段锁协议、各种隔离级别的实现，底层都依赖于锁机制或其变体。

---

### 第2页：锁的两种模式

这一页介绍了锁的两种基本类型——**共享锁和排他锁**。

**原文核心：**
> - Data items can be locked in two modes:
>   1. exclusive (X) mode. Data item can be both read as well as written.
>   2. shared (S) mode. Data item can only be read.
> - Lock requests are made to concurrency-control manager.
> - Should only allow conflict-serializable schedules.

**分点详解：**

- **排他锁（X锁）**：
  - 事务可以对数据项**读也可以写**。
  - 使用 `lock-X` 指令申请。
  - 如果一个事务持有排他锁，其他事务就**不能再加任何类型的锁**（无论是S锁还是X锁）。

- **共享锁（S锁）**：
  - 事务只能对数据项进行**读操作**。
  - 使用 `lock-S` 指令申请。
  - 多个事务可以**同时持有同一个数据项**的共享锁，因为它们都只是读，不会互相破坏数据。

- **并发控制管理器**：所有加锁请求都发送给**并发控制管理器**（它是DBMS的一部分）。事务只有在**请求被批准**之后才能继续执行。如果请求不能立即批准，事务就会被阻塞等待。

- **目标**：PPt最后特别强调，基于锁的协议应该**只允许产生冲突可串行化的调度**。也就是说，锁协议的设计目标，与我们之前用优先图检测出来的“正确调度”标准是一致的——锁是实现可串行化的手段。

---

### 第3页：锁相容矩阵

这一页是锁协议的核心规则——**什么锁可以共存，什么锁不能共存**。

**锁相容矩阵：**

|       | **S**    | **X** |
| ----- | -------- | ----- |
| **S** | **true** | false |
| **X** | false    | false |

读法：矩阵的行表示“某个事务请求的锁类型”，列表示“另一个事务已经在该数据项上持有的锁类型”。单元格为 `true` 表示请求可以被批准（相容），`false` 表示不能被批准（不相容）。

**三条核心规则：**

1. **S 与 S 相容（true）**：
   - 多个事务**可以同时持有同一数据项**的共享锁。
   - 原因：大家都是只读，谁也不会改数据，互相不影响。

2. **S 与 X 不相容（false）**：
   - 某数据项上已有共享锁，任何事务**不能再申请排他锁**。
   - 某数据项上已有排他锁，任何事务**也不能再申请共享锁**。
   - 原因：读写冲突——如果有人在读的时候有人写，读到的值可能是不一致的；如果有人写的时候有人读，读到的可能是未提交的脏数据。

3. **X 与 X 不相容（false）**：
   - 一个数据项上**最多只能有一个排他锁**。
   - 原因：写写冲突——两个事务同时写会导致丢失修改。

**请求处理规则：**
> - 如果请求的锁与已有锁相容 → 锁可以被授予。
> - 如果请求的锁与已有锁不相容 → 发起请求的事务必须**等待**，直到所有不相容的锁被持有者释放，锁才能被授予。

这意味着，当一个事务因为锁冲突而阻塞时，它就进入等待队列，等到时机成熟再继续往下执行。

---

### 总结：这三页的递进逻辑

1. **什么是锁**（第1页）：事务操作数据前必须加锁，加锁后就有了对该数据的控制权，其他事务不能随意修改。这是并发控制的基本手段。

2. **有哪几种锁**（第2页）：锁分为**S锁（共享，只读）**和**X锁（排他，可读可写）**。并发控制管理器负责接收和批准加锁请求，目标是保证只产生可串行化调度。

3. **锁之间怎么配合**（第3页）：通过**锁相容矩阵**定义规则——S与S共存，X与一切不相容。如果请求的锁与已有锁不相容，事务就必须**等待**。这套相容性规则直接对应了冲突可串行化理论中“读读不冲突，读写、写写冲突”的判断依据。

后续PPT通常会在此基础上，进一步讲解**两段锁协议（2PL）**——即事务必须按照“先集中加锁，再集中解锁”的纪律来管理锁的申请和释放，才能保证产生的调度一定是冲突可串行化的。
