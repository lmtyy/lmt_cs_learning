这三页PPT提出了对基本两段锁协议的一项重要优化——**锁转换**，它通过允许事务在持有锁期间动态调整锁的级别，在不破坏可串行化的前提下，显著提升了系统的并发度。

---

### 第1页：基本2PL的并发度缺陷

这一页用一个具体场景，揭示了严格遵守基本2PL可能导致的**过度串行化**问题。

**场景设定**：

-   **事务 T₁₁**：一个“先大量读取，最后少量写入”的事务。
    -   它会依次读取 `a₁`、`a₂` ... `aₙ`，最后才将计算结果写入 `a₁`。
-   **事务 T₁₂**：一个纯读事务，读取 `a₁` 和 `a₂` 并显示总和。

**基本2PL的局限性分析**：

-   按照基本2PL，T₁₁ 因为最终要写 `a₁`，为了不违反“释放锁后不能再加锁”的规则，它必须在**一开始访问 `a₁` 时就申请排他锁**。
-   这就导致在 T₁₁ 的整个漫长执行期间（读完 a₁ 直到最后写完 a₁），它一直霸占着 a₁ 的排他锁。
-   而 T₁₂ 需要读 a₁ 和 a₂，由于 a₁ 上的排他锁与任何锁都不相容，T₁₂ 只能在 T₁₁ **完全结束后**才能开始读取。
-   **结果**：T₁₁ 和 T₁₂ 虽然逻辑上很多操作不冲突，但实际只能串行执行，并发度被严重扼杀。

**改进直觉**：

> T₁₁ 在大部分时间只是读 `a₁`，只在最后才写。如果它一开始只加共享锁，就能允许 T₁₂ 同时读取 `a₁` 和 `a₂`。等 T₁₁ 完成所有读操作、准备写 `a₁` 时，再把共享锁“升级”成排他锁。这样在大部分时间里，两个事务就能真正并发运行了。

### 第2页：锁转换操作及其规则

这一页正式引入了**锁转换**机制，并把它纳入2PL的框架内。

**两个新操作**：

-   **UPGRADE**（升级）：将已持有的**共享锁转换为排他锁**。
-   **DOWNGRADE**（降级）：将已持有的**排他锁转换为共享锁**。

**关键规则（锁转换不可乱用）**：

> - **UPGRADE 只能在加锁阶段（Growing Phase）使用**。
> - **DOWNGRADE 只能在解锁阶段（Shrinking Phase）使用**。

**为什么这么规定？**

-   它保证锁的“严格程度”变化方向与2PL的两个阶段完全一致：
    -   在**扩展阶段**，锁的控制力只能越来越强（加锁、升级），绝不能削弱。
    -   在**收缩阶段**，锁的控制力只能越来越弱（解锁、降级），绝不能增强。
-   如果把 UPGRADE 看作“加了一把更强的锁”，把 DOWNGRADE 看作“释放了一把更强的锁，但还留着一把弱锁”，那么这套规则就完全等价于原来的2PL：“扩展阶段只许获得资源不许释放，收缩阶段只许释放资源不许获得”。

**T₁₁ 在改善协议下的执行示例**：

PPT中展示了T₁₁的加锁与转换流程：
```
LOCK-S(a₁)，LOCK-S(a₂)，... LOCK-S(aₙ)   // 大量加共享锁
→ UPGRADE(a₁)                            // 在最后一步，升级a₁为排他锁
→ (执行写操作)
→ UNLOCK(a₁), ... UNLOCK(aₙ)            // 进入收缩阶段，释放所有锁
```
在这个流程里，T₁₂ 可以全程并发读取数据，直到 T₁₁ 需要写 `a₁` 并执行升级时，才需要短暂等待。并发度得到极大提升。

### 第3页：改善后的2PL协议操作细则

这一页把升级锁的逻辑，通过具体的系统实现视角做了总结。

**系统面对读/写请求时采取的动作：**

-   **当事务提交 `READ(Q)` 操作时**：
    -   系统先执行 `LOCK-S(Q)`。
    -   然后执行 `READ(Q)`。
    -   这与原来一致，读操作前必须至少获得共享锁。

-   **当事务提交 `WRITE(Q)` 操作时**，系统需要判断当前锁状态，分两种情况：
    1.  **事务已经持有 Q 的共享锁**：
        -   系统执行 `UPGRADE(Q)`，尝试将共享锁升级为排他锁。
        -   升级成功后，执行 `WRITE(Q)`。
    2.  **事务没有持有 Q 的共享锁**（或之前根本没锁 Q）：
        -   系统执行 `LOCK-X(Q)`，直接申请排他锁。
        -   申请成功后，执行 `WRITE(Q)`。

**核心价值**：
这套规则让锁协议更加智能。事务不需要一开始就“预判”自己最终要写而霸道地加排他锁，它可以先谦逊地用共享锁，把真正需要排他访问的时机推迟到最后一刻。这在很大程度上缓解了基本2PL“过度保守”的问题，是实践中的一项重要性能优化。

---

**总结**：锁转换的核心思想是在2PL的纪律框架内，允许事务动态调整锁的强度，以此在保证可串行化的前提下，尽可能缩小排他锁的持有时间，从而提升并发度。
