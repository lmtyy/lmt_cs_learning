这四页PPT通过一个前后对比和改进的案例，揭示了简单加锁策略的致命缺陷，并由此引出了**两段锁协议**的核心思想。

---

### 第1页：对上例教训的总结

这一页对上一个例子（`T₇` 和 `T₈` 导致读到250元）进行了深刻的反思。

**原文结论：**
> - 一个事务只要存取一个数据项，它就必须持有该数据项上的一个锁。
> - 如果一个事务完成了对一个数据项的最后一次存取之后就立即放弃它的锁，则不能确保调度的可串行性。

**解读：**

-   **第一点**：重申了锁的基本规则——没锁就不能操作。这是并发控制的基础底线。
-   **第二点指向了问题的根源——“过早释放锁”**。`T₇` 在更新完B之后立刻释放了B的排他锁，然后再去申请A的锁。这个间隙让 `T₈` 趁虚而入，读到了转账进行到一半的数据库状态（B已扣款但A未加钱），导致总和错误。
-   **核心教训**：仅仅有加锁/解锁操作是不够的。如果事务在完成其全部工作前，就过早地释放了已获得的部分锁，就会打破原子性，让其他事务看到不一致的中间状态。**不控制释放锁的时机，就保证不了可串行化。**

---

### 第2页：改进方案——推迟释放锁

这一页给出了对 `T₇` 和 `T₈` 的“补丁”方案。

**核心修改：把所有释放锁的 `UNLOCK` 操作全部挪到事务的最后。**

-   **修改 `T₇` 得到 `T₉`**：
    ```
    LOCK-X(B);   // 1. 锁B
    READ(B);
    B := B - 50;
    WRITE(B);
    // 注意：这里不再释放B的锁
    
    LOCK-X(A);   // 2. 锁A（此时同时持有B锁和A锁）
    READ(A);
    A := A + 50;
    WRITE(A);
    
    UNLOCK(B);   // 3. 全部工作完成后，最后才释放所有锁
    UNLOCK(A);
    ```

-   **修改 `T₈` 得到 `T₁₀`**：
    ```
    LOCK-S(A);   // 1. 锁A
    READ(A);
    LOCK-S(B);   // 2. 锁B（此时同时持有A和B的读锁）
    READ(B);
    DISPLAY(A+B);
    
    UNLOCK(A);   // 3. 全部工作完成后，最后才释放所有锁
    UNLOCK(B);
    ```

**为什么要这样修改？**
`T₉` 在整个转账过程中，始终持有B和A的锁，直到账全部划完才松手。`T₁₀` 查询时，也会同时持有A和B的读锁。这样一来，当 `T₁₀` 读取时，要么 `T₉` 还没拿到锁，它读到转账前的旧状态；要么 `T₂` 已经执行完释放了锁，它读到转账后的新状态。**它再也看不到转账到一半的中间状态了。** 这样，并发调度的可串行性就得到了保证。

### 第3页：新方案引发的问题——死锁

这一页指出，虽然 `T₉` 和 `T₁₀` 解决了数据不一致，却引入了新的灾难：**死锁**。

**死锁发生的时间示意图：**

| 时刻 | `T₉`                                     | `T₁₀`                                   |
| ---- | ---------------------------------------- | --------------------------------------- |
| 1    | LOCK-X(B) → 成功                         |                                         |
| 2    | READ(B), B := B-50, WRITE(B)             |                                         |
| 3    |                                          | LOCK-S(A) → 成功                        |
| 4    |                                          | READ(A)                                 |
| 5    |                                          | LOCK-S(B) → **阻塞！** 等待T₉释放B的X锁 |
| 6    | LOCK-X(A) → **阻塞！** 等待T₁₀释放A的S锁 |                                         |

**死锁的直接原因：**
-   `T₉` 持有B的排他锁，等待获取A的排他锁。
-   `T₁₀` 持有A的共享锁（S锁与X锁不相容），等待获取B的共享锁。
-   两个事务彼此持有对方需要的资源，形成了一个**循环等待**，谁都动不了。

**系统的处理办法**：
PPT指出，此时系统必须强制**放弃至少一个**处于死锁状态的事务，也就是进行**死锁检测与恢复**。通常做法是选择一个事务作为“受害者”，将它回滚并释放它持有的所有锁，让另一个事务可以继续运行。

### 第4页：锁协议的权衡与总结

这一页是对整个案例分析的提炼，点出了并发控制方案的根本矛盾。

**原文核心：**
> - 如果为了获得最大的并发性而尽早地释放数据项的锁，则可能导致不一致的数据库状态。
> - 如果我们在申请其他锁之前不尽量地释放已拥有的锁，则可能导致死锁。

**两大问题的权衡：**

| 策略                     | 核心做法                       | 优点                   | 致命缺陷                                                     |
| :----------------------- | :----------------------------- | :--------------------- | :----------------------------------------------------------- |
| **早释放锁**（如T₇/T₈）  | 访问完一个数据项后立即释放锁   | 死锁风险低             | **导致不一致**，可能读到事务未提交的中间状态，破坏可串行性。 |
| **晚释放锁**（如T₉/T₁₀） | 将所有锁保留到事务结束时才释放 | 保证了一致性和可串行性 | **容易死锁**，降低了系统并发度。                             |

**解决方案——锁协议**：
正是由于这种自由加锁/解锁的方式存在无法调和的矛盾，数据库系统必须强制每个事务遵守一组关于何时加锁、何时解锁的严格规定，这组规定就被称为**锁协议**。

这四页PPT完美地铺垫了接下来将要学习的核心：
**最著名、最实用的锁协议就是“两段锁协议”——它要求在事务执行过程中，所有加锁操作都必须排在所有解锁操作之前，从而在避免不一致状态的同时，将死锁控制在可检测、可恢复的范围内。**
