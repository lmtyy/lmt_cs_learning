### 第1页：IS锁——意向共享锁

这一页给出了IS锁的精确定义和使用规则。

**定义**：
> 如果对一个数据对象加IS锁，表示它的后裔结点拟（意向）加S锁。

**含义**：IS锁本身不锁定当前节点的所有数据，它只是一个“声明”——告诉系统：“我下面有后代节点要被加共享锁了，请上面别把我整个树都锁死。”

**加锁规则**：如果要在一个节点上加S锁，必须先对其所有祖先节点加IS锁，自顶向下，逐层申请。

**例子**：
> 事务T1要对R1中某个元组加S锁，则要首先对关系R1和数据库加IS锁。

- **步骤**：`数据库加IS锁 → 关系R1加IS锁 → 目标元组加S锁`
- **原理**：这叫“意向锁的传播”。如果省去这一步，其他事务想对数据库或R1加X锁时，就没法知道下层有元组正在被读，可能导致锁冲突检查出错。

---

### 第2页：IX锁——意向排他锁

这一页定义IX锁。

**定义**：
> 如果对一个数据对象加IX锁，表示它的后裔结点拟（意向）加X锁。

**含义**：它表明事务打算在这个子树的某个下层节点上进行写操作。IX锁本身不是排他锁，它允许与其他意向锁兼容，但它表明下层会有排他性的修改。

**例子**：
> 事务T1要对R1中某个元组加X锁，则要首先对关系R1和数据库加IX锁。

- **步骤**：`数据库加IX锁 → 关系R1加IX锁 → 目标元组加X锁`
- **典型场景**：`INSERT`、`DELETE` 或 `UPDATE` 某一行时，需要对表加IX锁。多个事务可以对同一张表同时加IX锁，因为它们要修改的是表中不同的行，这些IX锁之间不冲突。

---

### 第3页：SIX锁——共享意向排他锁

这一页定义SIX锁，它是S锁和IX锁的组合。

**定义**：
> 如果对一个数据对象加SIX锁，表示对它加S锁，再加IX锁，即 SIX = S + IX。

这个锁可以看作“做两件事”的合并申明：我既想读整个节点（S锁），同时也要修改它的某些后代（IX锁）。

**例子**：
> 对某个表加SIX锁，则表示该事务要读整个表（所以要对该表加S锁），同时会更新个别元组（所以要对该表加IX锁）。

**SIX与S、IX的对比**：
- **S锁**：全读，不写。不阻塞别的只读事务，但阻塞所有写操作。
- **IX锁**：偶尔写，不扫描全表。允许多个IX共存，允许其他只读事务（S锁），但不允许全表扫描的S锁。
- **SIX锁**：同时全读 + 偶尔写。相比对表直接加X锁，它允许其他只读事务（IS、S）访问表中别的元组，并发度更高。

### 第4页：多粒度锁的完整相容矩阵

这一页是本节内容的汇总和核心，它给出了包含所有5种锁模式（IS、IX、S、SIX、X）的相容矩阵。

|         | **IS** | **IX** | **S** | **SIX** | **X** |
| ------- | ------ | ------ | ----- | ------- | ----- |
| **IS**  | ✓      | ✓      | ✓     | ✓       | ×     |
| **IX**  | ✓      | ✓      | ×     | ×       | ×     |
| **S**   | ✓      | ×      | ✓     | ×       | ×     |
| **SIX** | ✓      | ×      | ×     | ×       | ×     |
| **X**   | ×      | ×      | ×     | ×       | ×     |

**如何读懂这个矩阵**：
- **行**：代表事务T想申请的新锁。
- **列**：代表另一个事务已经在该节点上持有的锁。
- **✓（相容）**：可以同时持有，不冲突。
- **×（不相容）**：T必须等待。

**行解读分析**：

- **申请IS锁**：与全部类型相容（除了X锁）。只要没有别人独占整个节点，我声明“下面有读”总是可以的。
- **申请IX锁**：与IS和IX相容，但与S和SIX不相容。（有了S锁正全表读时，不能突然有行被写，会破坏可重复读。）
- **申请S锁**：与IS和S相容，与IX和SIX不相容。（有人在下面写时，我不能再要求一个稳定的全局快照来全读整个节点。）
- **申请SIX锁**：只与IS相容。只要下面有只读事务（IS），可以同时进行“全读+少量写”。
- **申请X锁**：与任何类型都不相容。排他锁一锁，其他人全部等。

**这个矩阵的威力**：
正是有了这个相容矩阵，系统才能高效检查锁冲突。任何事务要在某个节点加锁，只需查看这个节点上已有的锁模式，然后用这个矩阵完成 O(1) 的判断，无需再向下遍历检查所有子孙节点的锁状态。
