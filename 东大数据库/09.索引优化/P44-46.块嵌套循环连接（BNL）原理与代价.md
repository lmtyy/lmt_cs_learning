### 第1页：Block Nested-Loop Join 算法伪代码与示例表

**知识点1：BNL 的基本思想与伪代码**
```
for each block B_r of r do begin
  for each block B_s of s do begin
    for each tuple t_r in B_r do begin
      for each tuple t_s in B_s do begin
        Check if (t_r, t_s) satisfy the join condition
        if they do, add t_r · t_s to the result.
      end
    end
  end
end
```
- **外层循环**：遍历外表 R 的**每一个块（页）**，而不是每一个元组。
- **内层循环**：对于 R 的每一个块，遍历内表 S 的**每一个块**。
- **最内层**：对当前块 B_r 中的所有元组与当前块 B_s 中的所有元组逐一匹配。
- 与之前的“stupid loop join”对比：
  - 简单嵌套循环：`for each tuple in R → for each tuple in S`（内表对每个元组扫描一次）。
  - 块嵌套循环：`for each block in R → for each block in S`（内表对每个块扫描一次），这让内表 S 被读入内存的次数从 **m 次（元组数）降到 M 次（页数）**。

**知识点2：示例表 R 和 S 的作用**
- R(id, name)：7 行，如 (600, MethodMan) 等。
- S(id, value, cdate)：5 行，如 (100,2222,10/4/2021) 等。
- 这两张表在后续代价分析中虽不直接代入数字，但能让读者形象理解“块”的遍历：假设每页可存多条记录，块与块之间做笛卡尔积筛选，而不是行与页之间反复读取。

### 第2页：BNL 的磁盘访问改进、代价公式与外表选择

**知识点3：减少磁盘访问的根本原因**
- **对 R 的每一个块，S 只需要被扫描一次**，而不是对 R 的每个元组扫描一次。
- 因为一次性读入 R 的一个块（包含多个元组）后，将内表 S 的所有页顺序读入，就可以完成该 R 块中所有元组与 S 所有元组的匹配，无需反复重新读取 S。
- 这大幅减少 I/O 次数，正如后续计算所示。

**知识点4：外表选择原则——基于页数，而非元组数**
- “The smaller table should be the outer table.” 这里“smaller”指的是**占用的磁盘页数少**，而不是元组数量少。  
- 原因：代价公式中外表因数是 `b_r`（外表页数），内表扫描次数等于外表页数。如果外表页数少，内表被全表扫描的总次数就少。
- 这一原则比简单循环连接要更精确：在嵌套循环连接里，小外表指元组数少；而在 BNL 中，小外表指**页数少**。

**知识点5：最坏情况 I/O 代价公式**
\[
\text{最坏情况 I/O 次数} = b_r \times b_s + b_r
\]
- 这里用 `b_r`、`b_s` 分别表示外表 R、内表 S 的块（页）数，即之前示例中的 M 和 N。
- 公式解释：
  - `b_r`：读一遍外表 R 的全部页（需一次全表扫描）。
  - `b_r × b_s`：对于外表 R 的每一页，都需要把内表 S 的 `b_s` 页全部读一遍。所以总的内表扫描代价是 `b_r × b_s`。
  - 因此总 I/O = 外表读取次数 + 内表总读取次数 = `b_r + b_r × b_s`。
- 这一假设建立在**内存只能容纳两个块**（一个外表块、一个内表块）的最坏情况下，内表不能常驻内存。

**知识点6：最好情况 I/O 代价**
\[
\text{最好情况 I/O 次数} = b_r + b_s
\]
- 如果**内表 S 能够完全放在内存中**，那么只需要：
  - 把 S 的全部 `b_s` 页读入内存一次。
  - 把外表 R 的 `b_r` 页顺序读入，每读一页直接与内存中的整个 S 进行连接，无需重复读取 S。
- 总 I/O = 读 S 的所有页 + 读 R 的所有页 = `b_r + b_s`。
- 这体现了数据库缓存（buffer pool）大小对连接性能的巨大影响。当内存足够容纳内表时，连接代价降到理论下限（两张表各读一次）。

**知识点7：合适的外表选择问题**
- 幻灯片中提出 “Question: which relation is suitable for outer relation?”  
- 答案：占用页数少的作外表。因为外表页数 `b_outer` 越小，内表被扫描次数 `b_outer` 就越小，`b_outer × b_inner` 也就越小。如果内表能被缓存，外表选择对开销影响变小。

### 第3页：BNL 代价计算示例及与简单循环连接的对比

**知识点8：代入具体数据计算**
- 沿用之前的数据：表 R：M = 1000 页，m = 100,000 元组；表 S：N = 500 页，n = 40,000 元组。
- 假设 R 作外表，按照最坏情况公式（内存仅够两个块）：
  \[
  \text{IO 次数} = M + (M \times N) = 1000 + (1000 \times 500) = 501,000 \ \text{IOs}
  \]
- 如果一次 I/O 平均耗时 0.1 ms，总时间 ≈ 50,100 ms ≈ **50 秒**。

**知识点9：与简单循环连接的对比**
- 之前在简单循环连接（stupid loop join）中：
  - 同样 R 作外表：`M + m × N = 1000 + 100,000×500 = 50,001,000 IOs`，约 1.3 小时。
  - S 作外表优化后：`N + n × M = 500 + 40,000×1000 = 40,000,500 IOs`，约 1.1 小时。
- **BNL 的改进幅度**：从 **5000 万次** I/O 降到 **50 万次**，减少了约 100 倍，时间从小时级降到分钟级（50 秒）。本质原因是用外表**页数 M（1000）**代替了外表**元组数 m（100,000）**来乘以内表页数 N。

**知识点10：外表优化在 BNL 中的体现**
- 如果按照 BNL 原则，页数少的表应该作外表。S 的页数 N = 500，R 的页数 M = 1000，因此**用 S 作外表能得到更低代价**：
  \[
  N + (N \times M) = 500 + (500 \times 1000) = 500,500 \ \text{IOs}
  \]
  约 50.05 秒，比 501,000 IOs 更少（虽然差异不大，但原则正确）。当两个表页数相差悬殊时，影响会更显著。
- 幻灯片此处直接用 M + M×N 计算，是为了与前面 stup loop join 的 R 作外表对比，突显改进幅度，同时引出“选择页数少的表作外表”的结论。

**知识点11：从 BNL 到更优算法的过渡**
- 即使 BNL 把时间缩短到 50 秒，对于大表或高并发查询仍可能成为瓶颈。内存能否容纳内表、块大小如何利用等，又衍生出进一步优化：
  - **带缓冲的 BNL**：一次读入外表多个块，增加内表扫描的共享效益。
  - **索引嵌套循环**：在内表连接键上有索引时，每次只读取内表相关块。
  - **归并连接**、**哈希连接**等（在后续 PPT 会介绍）。
- 因此，这三页在继承简单循环连接代价模型的基础上，展示了**基于块的处理**如何极大降低 I/O，并为学习者建立“块 I/O 是主要开销，外表页数决定了内表扫描次数”这一核心概念。

---

### 总结
这三页 PPT 系统介绍了**块嵌套循环连接**的原理和代价分析：
1. 算法通过**以块为单位**扫描外表、内表，将内表全表扫描次数从“外表元组数”降至“外表页数”，从而大幅减少 I/O。
2. 代价公式 `b_r × b_s + b_r` 反映了最坏情况，`b_r + b_s` 则展示了内存充裕时的理想状况。
3. 外表选择应**基于页数最小**，以进一步降低 `外表页数 × 内表页数` 的乘积。
4. 具体计算（501,000 IOs，50 秒）相比简单循环（千万级 IOs，小时级）有质的飞跃，但仍可继续优化，为后续讲解索引连接、哈希连接等打下基础。
