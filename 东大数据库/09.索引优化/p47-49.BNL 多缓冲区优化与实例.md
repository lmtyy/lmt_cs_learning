这三页 PPT 在基本块嵌套循环连接（BNL）的基础上，进一步探讨了**利用更多内存缓冲区来优化 I/O 代价**，以及**针对等值连接和键属性的微优化**，最后用例子巩固。我们按顺序细讲。

---

### 第1页：引入缓冲区概念与对应的算法逻辑

**知识点1：当有 B 个缓冲区时的内存分配策略**
- 假设内存中可供连接操作使用的缓冲区总数为 **B** 页。
- 将这些缓冲区划分为三个用途：
  - **B - 2 个页面用于扫描外表**：一次性将外表的 B-2 页读入内存，作为一批处理单元。
  - **1 个页面用于缓冲内表**：内表每次仍是一页一页地读入。
  - **1 个页面用于存储结果输出**：匹配上的元组组合先暂存在输出缓冲区，满了再写回磁盘。
- 这个分配方式比基本 BNL 只能容纳一个外表页和一个内表页更充分地利用了内存，能进一步减少对内表的全表扫描次数。

**知识点2：对应算法的伪代码**
```
foreach B-2 blocks b_R ∈ R:          // 每次读入外表 R 的 B-2 个块
    foreach block b_s ∈ S:           // 对 S 的每一个块
        foreach tuple r ∈ B-2 blocks: // 对前述 B-2 块中的所有元组
            foreach tuple s ∈ b_s:
                if r 和 s 匹配 then 输出
```
- 外层循环的单位从“1 个外表块”变成了“一批 B-2 个外表块”。
- 对于每一批外表页，内表 S 依然被完整扫描一次，但**外表的分批次数大幅减少**：外表 R 总共 `b_r` 页，分批次数为 `⌈b_r / (B-2)⌉`，而不是原来的 `b_r` 次。

### 第2页：改进后的代价公式与进一步微优化

**知识点3：改进块嵌套循环的代价公式**
- 用 **M** 表示内存可用于连接的总页数（同前文的 B），则公式为：
\[
\text{Cost} = b_r + \lceil b_r / (M-2) \rceil \times b_s
\]
- 式中：
  - `b_r`：外表 R 的总页数，需要从磁盘读一次。
  - `⌈b_r / (M-2)⌉`：外表 R 被分成多少批（每批 M-2 页）。
  - `b_s`：内表 S 的总页数。每一批外表页会引发一次对 S 的完整扫描，因此内表扫描总次数 = 批次数。
- 当 `M = 3` 时，`M-2 = 1`，每批只有 1 页，批次数 = `b_r`，代价退化为 `b_r + b_r × b_s`，即最坏 BNL。
- 当 `M` 大到足以容纳整个内表或更多时，代价可以逼近 `b_r + b_s`。

**知识点4：等值连接与键属性的提前终止优化**
- “If equi-join attribute forms a key or inner relation, stop inner loop on first match.”
- 如果连接条件是等值连接，且连接属性在内表 S 中是键（key，唯一），那么对于外表 R 中的某个元组，在内层循环中**找到第一条匹配后就可以停止**该元组的内层扫描。
- 这是因为键唯一性保证了内表最多只有一行匹配，继续扫描后续页也不会再有匹配结果，从而省掉一部分无意义的块读取与比较。
- 这特别适用于外键-主键连接（例如 depositor 的 account_number 参照 customer 的主键），内表是主键所在表，可提前终止。

**知识点5：交替正向/反向扫描内表以利用缓冲区残留页**
- “Scan inner loop forward and backward alternately, to make use of the pages remaining in buffer (with LRU replacement)”
- 当一批外表页处理完后，内表刚被完整扫描过，假设缓冲区使用 LRU 替换策略，内表的最后一些页可能还留在内存中。
- 下一批外表页处理时，如果反向扫描内表（从最后一页开始向前读），可以直接命中缓冲区内残留的这些页，减少实际磁盘读取。
- 这种交替扫描方向利用了顺序扫描的预读特性和缓冲区复用，是一种工程层面的微优化。

### 第3页：具体例子——用 depositor 和 customer 表量化分析

**知识点6：示例参数与连接场景**
- 来自最早的统计数据：`depositor` 100 页，`customer` 400 页。
- 决定以 `depositor` 作为外表（因为它页数少），`customer` 作为内表。
- 这个选择符合“页数少的作外表”原则，以最小化 `外表批次 × 内表页数`。

**知识点7：最坏情况块嵌套循环代价**
- 公式：`b_outer * b_inner + b_outer = 100 * 400 + 100 = 40,100` 次页面访问。
- 对应前面最坏公式 `b_r + b_r × b_s`（即每批只有一页外表，M=3）。
- **问题“需要多少主存页面？”**  
  答：只要 **3 页** 就能运行：
  - 1 页装外表的一批（每次 1 页），1 页装内表当前页，1 页输出。
  - 内存再大也不会改变这个公式下的代价（除非大到能缓存更多），所以最坏情况所需最小内存是 3 页。

**知识点8：最好情况块嵌套循环代价**
- 公式：`b_outer + b_inner = 100 + 400 = 500` 次页面访问。
- 这需要内表 `customer`（400 页）**完全常驻内存**，外表顺序读入即可。
- **问题“需要多少主存页面才能达到这个代价？”**  
  答：至少需要 **400 + 2 = 402 页**（内表全部 400 页，加上1页外表读入、1页输出）。  
  实际上 401 页也可以勉强（1 页输出，400 页装内表，外表逐页读入用内表页面替换，但为了不淘汰内表，通常要留出至少 1 页给外表，加输出页即 402 页）。PPT 的意图是强调内存足够容纳整张内表是关键。

**知识点9：给定 52 个主存页面时的实际代价**
- 内存 M = 52 页。
- 外表：`depositor` 100 页。
- 每批外表大小 = `M - 2 = 50` 页。
- 批次数 = `⌈100 / 50⌉ = 2`。
- 代价 = `b_outer + 批次数 × b_inner = 100 + 2 × 400 = 900` 次页面访问。
- 这就是 PPT 上写的 “2 * 400 + 100 = 900 page accesses”。
- 相比最坏情况的 40,100 次，52 页内存将 I/O 降低了约 **44 倍**；相比最好情况的 500 次，只多了 400 次读取（因为内表被额外完整扫描了一次）。这个计算直观展示了**内存大小对于 BNL 代价的平滑影响**：内存越多，批次数越少，代价向最好情况靠拢。

---

### 总结这三页的进展
1. 引入多缓冲区后，**外表分批扫描**的概念使得 I/O 与可用内存直接挂钩，代价从 `b_r * b_s` 向 `b_r + b_s` 过渡。
2. 具体公式 `b_r + ⌈b_r/(M-2)⌉ * b_s` 统一了最坏、一般和最好情况，成为 BNL 代价分析的通用模型。
3. 结合实际参数的计算，展示了内存从 3 页增加到 52 页，乃至到容纳整个内表，代价的显著变化，强化了“内存是影响连接性能的关键资源”这一核心思想，为后续讲解哈希连接、归并连接等需要较大内存的算法打下基础。
