### 第一页 PPT：内容提纲

这一页列出了查询处理章节将要深入讲解的几个核心模块。理解这个提纲，就抓住了整个章节的脉络。

**1. Overview**

- **知识点**：引言部分，回答“什么是查询处理”和“为什么要学它”。
- **详细解释**：当用户输入一条 SQL 语句（如 `SELECT * FROM student WHERE age > 20`），数据库内部需要做一系列工作才能把结果取出来。这个过程就是**查询处理**。由于一条 SQL 往往有数种甚至上万种等效的执行方式（例如先过滤还是先连接），不同方式的执行速度可能相差百万倍。Overview 强调的就是从 SQL 文本到结果数据的转换过程的重要性。

**2. Measures of Query Cost**

- **知识点**：查询代价的度量标准。
- **详细解释**：优化器如何判断方案 A 比方案 B 好？不能靠猜，需要量化指标。
    - **时间单位**：通常不直接用“秒”，因为秒依赖硬件。
    - **核心指标**：**磁盘 I/O 次数**（读/写磁盘块数）。因为内存操作是纳秒级，磁盘操作是毫秒级，I/O 是主要瓶颈。
    - **次要指标**：CPU 时间、网络传输成本（分布式数据库）。

**3. Selection Operation**

- **知识点**：选择操作的实现算法。
- **详细解释**：对应 SQL 中的 `WHERE` 子句（非连接部分）。如何从表中找出满足 `age = 20` 的行？
    - **线性扫描**：一行一行看，适合小表或无索引。
    - **索引扫描**：利用 B+ 树索引快速定位，适合返回少量数据的查询。
    - **主键等值查询** vs **范围查询** vs **多条件合取/析取** 的不同处理策略。

**4. Sorting**

- **知识点**：排序算法。
- **详细解释**：对应 `ORDER BY` 或 `GROUP BY` 前的预处理。数据库的排序与操作系统课程中的快排不太一样，因为数据量可能远超内存大小。
    - **外排序**：核心算法是**外部归并排序**。先分块在内存排序，写回磁盘，再通过多路归并合并成最终有序文件。

**5. Join Operation**

- **知识点**：连接操作的实现算法。
- **详细解释**：查询处理的**重中之重**。连接是关系数据库的精华，也是最慢的操作之一。
    - **Nested-Loop Join**：两层 for 循环，暴力解法。
    - **Block Nested-Loop Join**：改进版，一次读一块数据。
    - **Index Nested-Loop Join**：利用内表索引。
    - **Sort-Merge Join**：先排序，后归并。
    - **Hash Join**：用哈希表做匹配，常用于等值连接。

**6. Transformation of Relational Expressions**

- **知识点**：关系代数表达式的等价变换。
- **详细解释**：这是**查询优化器**的理论基础。在生成执行计划前，优化器会把 SQL 对应的关系代数树进行改写，让计算量变小。
    - **核心规则**：**选择下推** —— 把 `WHERE` 条件尽可能推到叶子节点（最底层），**先过滤再连接**，数据量会指数级减少。

### 第二页 PPT：查询处理的基本步骤

这一页展示了从 SQL 文本到最终结果输出的流水线。结合图中的箭头关系来理解：

#### 第一步：解析与翻译

- **输入**：SQL 文本字符串（比如 `SELECT name FROM student WHERE id = 123`）。
- **过程**：
    1.  **词法分析与语法分析**：检查关键字拼写、括号匹配、表名列名是否存在。
    2.  **生成解析树**。
    3.  **翻译成关系代数表达式**。
- **输出**：一棵**关系代数树**。
    - *图中对应*：`query` -> `parser and translator` -> `relational-algebra expression`。

#### 第二步：优化

- **输入**：初始的关系代数表达式。
- **过程**：这是大脑部分。优化器发现上述代数树执行效率很低（比如先做笛卡尔积再筛选，计算量巨大）。它会利用**等价变换规则**（第一页的第 6 点）生成逻辑上等价但执行代价更低的代数树。
    - **图中注意细节**：优化器不仅接收代数表达式，还从下方接收 **`statistics about data`**。这说明优化是基于**代价估算**的。优化器需要知道表有多少行、索引分布情况，才能算出哪个方案 I/O 次数最少。
- **输出**：**执行计划**。
    - *图中对应*：`relational-algebra expression` -> `optimizer` -> `execution plan`。

#### 第三步：评估执行

- **输入**：执行计划（具体指示每一步用什么算法、顺序如何）。
- **过程**：执行引擎像流水线工人一样，一步步执行计划中的算子（如先做 Index Scan，再做 Filter，再做 Projection）。
- **输出**：查询结果数据。
- *图中对应*：`evaluation engine` 结合 `data`，最终返回结果。

### 总结：两页内容的衔接逻辑

1.  第二页讲了**宏观流程**：SQL -> 翻译 -> 优化 -> 执行。
2.  第一页的提纲正是对**宏观流程中核心模块的展开**：
    - `Measures of Query Cost` 是 **第二步（优化）** 的数学依据。
    - `Selection`, `Sorting`, `Join` 是 **第三步（评估执行）** 引擎里具体可用的算法武器库。
    - `Transformation` 是 **第二步（优化）** 的逻辑手段。
