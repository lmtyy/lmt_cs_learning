这三页 PPT 聚焦于 **索引嵌套循环连接（Indexed Nested-Loop Join）的算法细节、代价公式与实际例子**，把前面两页的概念具象化。我们按顺序深入讲解。

---

### 第1页：伪代码、示例表与索引示意

**知识点1：算法伪代码**
```
foreach tuple r ∈ R:
    foreach tuple s ∈ Index(rᵢ = sⱼ):
        emit, if r and s match
```
- 外层：顺序扫描外表 R 中的**每个元组** `r`。
- 内层：利用内表 S 在连接属性上的索引，直接查找所有满足 `r_i = s_j` 的元组 `s`（而不是扫描整张 S 表）。
- 这里的 `Index(condition)` 表示一次索引查找操作，它返回指向匹配记录的指针（或直接返回记录），然后回表获取完整元组。
- 与之前的简单循环对比：内层循环不再是“扫描 S 的每一页/每一行”，而是仅遍历索引找到的匹配行。

**知识点2：示例表 R、S 与索引**
- 表 R（id, name）：7 行，如 (600, MethodMan) 等，外表。
- 表 S（id, value, cdate）：5 行，如 (100, 2222, …) 等，内表。
- 在 S 的 `id` 列上建有索引 `Index(S.id)`。图中特意标注“N pages, n tuples”，强调 S 有自己的页数和元组数统计。
- 实际执行过程（示例）：  
  对于 R 的元组 (100, Andy) → 在 Index(S.id) 上查找 100 → 得到两条匹配记录（2222 和 9999） → 组合输出。  
  对于 R 的元组 (200, GZA) → 查找 200 → 得到 (200, 8888) → 输出。

**知识点3：索引查找代替全表扫描的直观效果**
- 如果没有索引，处理 R 每个元组都要把 S 的 N 个页面全读一遍（像 stupid loop join）。但有了索引，只需几次 I/O 就能定位到 S 中匹配的数据页，绝大部分与连接无关的页面不会被访问。这正是索引嵌套循环连接的最大优势。

### 第2页：代价公式与外表选择策略

**知识点4：代价公式的构成**
\[
\text{Cost} = b_r + n_r \times c
\]
- **`b_r`**：外表 R 的页数。需要把外表全部页读入内存，这部分代价与简单循环一样。
- **`n_r`**：外表 R 的元组总数。
- **`c`**：对于外表 R 中的**一个元组**，利用索引在 S 中查找所有匹配元组并获取这些元组的**平均磁盘 I/O 次数**。

**知识点5：`c` 的具体含义与估算**
- `c` 可以看作：在表 S 上执行一个等值选择操作（`SELECT * FROM S WHERE join_attr = value`）所花费的磁盘访问次数。
- `c` 的构成通常包括：
  - **索引遍历代价**：从根节点到叶子节点。对于 B+-树，通常等于索引的高度（层级数）。例如高度为 4，遍历需要 4 次 I/O。
  - **回表代价**：通过索引叶子节点中存储的指针，读取数据页以获取完整元组。
    - 如果索引是 **聚簇索引**（主索引），匹配的所有行可能在同一个或相邻的数据页中，回表只需很少 I/O（可能 1 次）。
    - 如果是 **非聚簇索引**且匹配行较多且分散，每个匹配行可能都要一次独立的 I/O，`c` 会很大。
  - 如果连接条件是内表的**键（主键或唯一键）**，则外表每个元组至多匹配一条内表记录，此时 `c` 通常 = 索引高度 + 1（回表一次）。
- 因此 `c` 取决于索引类型、数据聚簇性以及连接属性的唯一性。

**知识点6：两个表都有索引时的外表选择**
- “If indices are available on join attributes of both r and s, use the relation with fewer tuples as the outer relation.”
- 如果外表和内表的连接列上都有索引，优化器可以选择任意一个表做外表。
- 原则：选择**元组数更少的表**作为外表。因为外表元组数 `n_r` 直接与代价中的 `n_r × c` 成正比。外表元组越少，索引探针次数越少，总代价越低。
- 这里与块嵌套循环不同（块嵌套是按页数少选外表），索引嵌套循环是按**元组数少**选外表，因为索引查找是以元组为单位驱动，外表的每一条元组都会触发一次内表索引查找。

### 第3页：具体示例——depositor JOIN customer 代价计算

**知识点7：示例参数设定**
- 连接：`depositor JOIN customer`
- 外表：depositor
  - 页数 `b_depositor = 100`
  - 元组数 `n_depositor = 5000`
- 内表：customer
  - 在连接属性 `customer-name` 上建有 **主索引（B+-树）**，且该属性是 `customer` 的**主键（primary key）**。
  - 索引级别：4 层（height = 4）。
  - 因为是主键索引，每个外表元组查找内表时**最多匹配一行**。

**知识点8：计算 `c` 的值**
- 由于是等值连接且内表连接属性为主键，外表每个 depositor 元组在 customer 中最多匹配一条记录。
- 索引查找过程：
  - 遍历 B+-树，从根到叶：4 次 I/O（因为索引 4 层）。
  - 通过叶子节点指针回表读取 customer 的数据页：1 次 I/O。
- 因此 `c = 4 + 1 = 5` 次磁盘访问。

**知识点9：代入公式计算总代价**
\[
\text{Cost} = b_{\text{depositor}} + n_{\text{depositor}} \times c
 = 100 + 5000 \times 5
 = 100 + 25,000
 = 25,100 \ \text{disk accesses}
\]
- 对比之前块嵌套循环在 52 页内存下的 900 次 I/O，索引嵌套循环的 I/O 次数反而更高（25,100）。这很正常，因为：
  - 示例中外表 depositor 有 5000 个元组，每一条都引发 5 次 I/O 去访问内表，即便每次很快，总量相乘后也不低。
  - 块嵌套循环利用内存批量处理外表块，内表顺序扫描，内表相对较小（400 页），顺序扫描比 5000 次随机访问可能更经济。
- 这说明索引嵌套循环连接并非总是最优，它在外表元组数**非常少**或内表**非常庞大**且索引能精准定位时才显优势。

**知识点10：最佳适用场景——外表带选择性条件**
- “Indexed Nested-Loop is the best algorithm if there are **selective conditions** on the outer relation.”
- 如果外表在参与连接之前经过**高选择性的过滤条件**（例如 `WHERE depositor.balance > 100000`），使得外表实际参与连接的元组数急剧减少（比如从 5000 降到 50），那么代价就变成 `100 + 50 × 5 = 350` 次 I/O，将远低于块嵌套循环。
- 典型场景：
  - 外表是经过 WHERE 过滤后的小结果集。
  - 外表是小表，内表是大表（例如客户表上千万行，而订单外表只有几百行）。
- 这也是为什么许多数据库优化器在“外表小且内表有索引”时倾向于选择索引嵌套循环连接。

**知识点11：与其他算法的对比启示**
- 幻灯片最后这句强调适用条件，也为后续的归并连接（merge join）和哈希连接（hash join）做铺垫：那些算法对两个大表连接更有效，而索引嵌套循环连接在小表驱动大表且内表有索引时极佳。
- 反过来，如果外表元组很多，而且内表索引非聚簇，回表 I/O 巨大，索引嵌套循环就可能成为灾难（大量随机 I/O）。

---

### 总结
这三页 PPT 完成了对索引嵌套循环连接的完整阐述：
1. **算法**：外表逐元组驱动，内表用索引查找匹配，彻底消除内表全表扫描。
2. **代价模型**：`Cost = b_r + n_r × c`，核心在于 `c` 的大小，`c` 由索引遍历高度及回表次数决定。
3. **外表选择**：元组数少的作外表，以减少探针次数。
4. **实例**：depositor (100页,5000行) JOIN customer (带4层主键索引) → 25,100 I/O，体现公式计算。
5. **适用判断**：当外表有高选择性过滤条件，外表元组数锐减时，该算法通常是最佳选择。
