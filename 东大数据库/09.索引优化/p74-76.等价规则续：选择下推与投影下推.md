好的，这三页 PPT 继续讲等价规则，分别是规则 7（选择对连接的分配律）、规则 5/6a/7a 的图示、以及规则 8（投影对连接的分配律）。这几条规则是查询优化中"选择下推"和"投影下推"的直接理论依据，非常重要。

## 第一页：规则 7 —— 选择对 θ 连接的分配律（Selection Distribution over Join）

这条规则是查询优化中最核心的规则之一，它是"选择下推"（selection push-down）优化的理论基础。

### 7a：选择条件只涉及一个关系的属性

> "When all the attributes in θ₀ involve only the attributes of one of the expressions (E₁) being joined."

$$\sigma_{\theta_0}(E_1 \bowtie_\theta E_2) = (\sigma_{\theta_0}(E_1)) \bowtie_\theta E_2$$

条件：θ₀ 中涉及的所有属性都只来自 E₁。

含义：如果选择条件只和连接的左输入 E₁ 有关，那么可以把选择操作"下推"到 E₁ 上，先过滤 E₁，再做连接。

举例：
```
σ_{branch_city='Brooklyn'}(branch ⋈ account)
= (σ_{branch_city='Brooklyn'}(branch)) ⋈ account
```
因为 branch_city 只是 branch 表的属性，所以选择可以下推到 branch 上。

为什么这样更好？
- 原来：先连接 branch 和 account（可能产生巨大的中间结果），再过滤
- 优化后：先过滤 branch（假设只有 5 个 Brooklyn 的支行），再用这 5 行去连接 account
- 中间结果大幅缩小，I/O 代价大幅降低

⚠️ 易考点：规则 7a 的前提条件——θ₀ 的属性必须全部来自 E₁（或全部来自 E₂）。如果 θ₀ 同时涉及 E₁ 和 E₂ 的属性（比如 E₁.A > E₂.B），就不能用 7a，因为在连接之前 E₁ 还没有 E₂ 的数据，无法评估这个条件。

⚠️ 易错点：注意下推的方向。θ₀ 涉及 E₁ 的属性，就下推到 E₁ 上；涉及 E₂ 的属性，就下推到 E₂ 上。不要推错方向。

### 7b：选择条件可以拆分，分别涉及两个关系

> "When θ₁ involves only the attributes of E₁ and θ₂ involves only the attributes of E₂."

$$\sigma_{\theta_1 \wedge \theta_2}(E_1 \bowtie_\theta E_2) = (\sigma_{\theta_1}(E_1)) \bowtie_\theta (\sigma_{\theta_2}(E_2))$$

条件：选择条件是 θ₁ ∧ θ₂ 的形式，其中 θ₁ 只涉及 E₁ 的属性，θ₂ 只涉及 E₂ 的属性。

含义：如果一个 AND 选择条件可以拆成两部分，分别只涉及连接的左右输入，那么可以把两部分分别下推到各自的关系上。

举例：
```
σ_{branch_city='Brooklyn' ∧ balance>1000}(branch ⋈ account)
= (σ_{branch_city='Brooklyn'}(branch)) ⋈ (σ_{balance>1000}(account))
```
因为 branch_city 只属于 branch，balance 只属于 account，所以两个条件可以分别下推。

这实际上是规则 1（选择串接律）+ 规则 7a 的组合应用：
1. 先用规则 1 把 σ_{θ₁∧θ₂} 拆成 σ_{θ₁}(σ_{θ₂}(...))
2. 再用规则 7a 分别把 σ_{θ₁} 下推到 E₁，σ_{θ₂} 下推到 E₂

⚠️ 易考点：考试经常给一个复杂的选择条件，让你判断哪些部分可以下推、下推到哪个关系。解题步骤：
1. 先用规则 1 把 AND 条件拆分成多个独立的选择
2. 分析每个选择条件涉及哪个关系的属性
3. 只涉及 E₁ 的 → 下推到 E₁
4. 只涉及 E₂ 的 → 下推到 E₂
5. 同时涉及 E₁ 和 E₂ 的 → 不能下推，留在连接之上

⚠️ 易错点：如果选择条件是 θ₁ ∨ θ₂（OR），即使 θ₁ 只涉及 E₁、θ₂ 只涉及 E₂，也不能分别下推！因为 OR 的语义是"满足任一即可"，下推后会丢失满足 θ₂ 但不满足 θ₁ 的 E₁ 元组。这是一个非常经典的陷阱。

举例说明 OR 不能下推：
```
σ_{branch_city='Brooklyn' ∨ balance>1000}(branch ⋈ account)
≠ (σ_{branch_city='Brooklyn'}(branch)) ⋈ (σ_{balance>1000}(account))
```
左边：只要支行在 Brooklyn 或者余额大于 1000，就保留
右边：支行必须在 Brooklyn 且余额必须大于 1000，才能出现在结果中
显然右边丢失了"支行不在 Brooklyn 但余额大于 1000"的记录。

## 第二页：Pictorial Depiction of Equivalence Rules —— 等价规则的查询树图示

这一页用查询树的形式直观展示了三条规则的变换效果。

### 上方：Rule 5 —— 连接的交换律

```
  ⋈_θ              ⋈_θ
 / \      ↔       / \
E1  E2           E2  E1
```

左右两个输入交换位置，连接条件 θ 不变。

图示很直观：就是把查询树中连接节点的左右子树互换。

实际应用场景：
- 在 Hash Join 中，较小的关系应该做 build input（通常是左输入），所以如果 E₂ 比 E₁ 小，就用交换律把 E₂ 换到左边
- 在 Nested Loop Join 中，较小的关系放外层（左输入）更好

### 中间：Rule 6a —— 自然连接的结合律

```
    ⋈                    ⋈
   / \                  / \
  ⋈   E3     ↔       E1   ⋈
 / \                      / \
E1  E2                   E2  E3
```

左边：先连接 E₁ 和 E₂，再和 E₃ 连接（左深树 / left-deep tree）
右边：先连接 E₂ 和 E₃，再和 E₁ 连接（右深树 / right-deep tree）

这条规则改变了连接的执行顺序。注意图中不仅改变了结合方式，还隐含了交换律的应用（E₁ 从左下移到了左上）。

连接顺序的选择对性能影响巨大。假设：
- E₁ 有 10 行，E₂ 有 10000 行，E₃ 有 100 行
- 左边方案：E₁ ⋈ E₂ 可能产生很大的中间结果，再和 E₃ 连接
- 右边方案：E₂ ⋈ E₃ 如果选择性好，中间结果可能很小，再和 E₁ 连接

⚠️ 易考点：查询优化器在选择连接顺序时，通常偏好"左深树"（left-deep tree），即所有连接的右输入都是基本关系（叶子节点）。原因是左深树更容易实现流水线（pipeline），因为每次连接的右输入可以直接从磁盘读取，不需要物化中间结果。

### 下方：Rule 7a —— 选择对连接的分配律（选择下推）

```
  σ_θ                  ⋈
   |                  / \
  ⋈        →       σ_θ   E2
 / \                |
E1  E2              E1
```

条件：θ 只涉及 E₁ 的属性。

左边：先连接 E₁ 和 E₂，再对结果做选择
右边：先对 E₁ 做选择，再和 E₂ 连接

图中的箭头是单向的（→），但实际上等价规则是双向的。不过在优化中，我们几乎总是从左到右应用（把选择下推），因为先过滤再连接总是更好的。

⚠️ 易考点：这三条规则的图示经常出现在选择题中，让你判断变换是否正确、应用了哪条规则。要能快速识别：
- 左右子树互换 → Rule 5（交换律）
- 连接树的形状改变（左深变右深或反之）→ Rule 6a（结合律）
- 选择节点从连接上方移到连接下方的某个子树上 → Rule 7a（选择下推）

## 第三页：规则 8 —— 投影对 θ 连接的分配律（Projection Distribution over Join）

这条规则是"投影下推"（projection push-down）优化的理论基础。投影下推的目的是尽早去掉不需要的列，减少中间结果的宽度（每行的字节数），从而减少 I/O 和内存消耗。

### 8a：简单情况 —— 连接条件只涉及投影保留的属性

> "if θ involves only attributes from L₁ ∪ L₂"

$$\Pi_{L_1 \cup L_2}(E_1 \bowtie_\theta E_2) = (\Pi_{L_1}(E_1)) \bowtie_\theta (\Pi_{L_2}(E_2))$$

条件：
- L₁ 是最终需要的来自 E₁ 的属性集
- L₂ 是最终需要的来自 E₂ 的属性集
- 连接条件 θ 只涉及 L₁ ∪ L₂ 中的属性（即投影保留的属性已经包含了连接需要的属性）

含义：如果连接条件用到的属性都在最终投影中保留了，那么可以直接把投影分别下推到两个关系上，先去掉不需要的列，再做连接。

举例：
```
Π_{customer_name, branch_name}(depositor ⋈ account)
```
假设连接条件是 account-number（自然连接的公共属性），但 customer_name 来自 depositor，branch_name 来自 account。

这里连接需要 account-number，但最终投影不包含 account-number。所以 8a 不适用，需要用 8b。

### 8b：复杂情况 —— 连接条件涉及了投影中不包含的属性

这是更常见的情况。连接条件需要某些属性来做匹配，但这些属性在最终结果中不需要。

定义：
- L₁：最终需要的来自 E₁ 的属性集
- L₂：最终需要的来自 E₂ 的属性集
- L₃：E₁ 中参与连接条件 θ 但不在 L₁ ∪ L₂ 中的属性
- L₄：E₂ 中参与连接条件 θ 但不在 L₁ ∪ L₂ 中的属性

$$\Pi_{L_1 \cup L_2}(E_1 \bowtie_\theta E_2) = \Pi_{L_1 \cup L_2}((\Pi_{L_1 \cup L_3}(E_1)) \bowtie_\theta (\Pi_{L_2 \cup L_4}(E_2)))$$

含义：
1. 对 E₁ 做投影，保留 L₁（最终需要的）+ L₃（连接需要但最终不需要的）
2. 对 E₂ 做投影，保留 L₂（最终需要的）+ L₄（连接需要但最终不需要的）
3. 做连接（此时连接条件需要的属性 L₃、L₄ 都还在）
4. 最后再做一次投影 Π_{L₁∪L₂}，去掉 L₃ 和 L₄

注意：最外层还有一个 Π_{L₁∪L₂}，这是因为 L₃ 和 L₄ 是为了连接临时保留的，连接完成后需要去掉。

举例：
```
Π_{customer_name, branch_name}(depositor ⋈_{depositor.account-number = account.account-number} account)
```

分析：
- L₁ = {customer_name}（来自 depositor，最终需要）
- L₂ = {branch_name}（来自 account，最终需要）
- 连接条件涉及 account-number，它同时来自两个表
- L₃ = {account-number}（depositor 中参与连接但不在 L₁∪L₂ 中）
- L₄ = {account-number}（account 中参与连接但不在 L₁∪L₂ 中）

变换后：
```
Π_{customer_name, branch_name}(
    (Π_{customer_name, account-number}(depositor))
    ⋈_{account-number}
    (Π_{branch_name, account-number}(account))
)
```

效果：depositor 原来可能有很多列（customer_name, account-number, access-date, ...），投影后只保留 2 列；account 同理。中间结果的每行字节数大幅减少。

⚠️ 易考点：8b 的关键在于理解 L₃ 和 L₄ 的定义——它们是"连接需要但最终不需要"的属性。考试可能给你一个具体的查询，让你：
1. 确定 L₁、L₂、L₃、L₄ 分别是什么
2. 写出投影下推后的表达式

⚠️ 易错点 1：忘记最外层的 Π_{L₁∪L₂}。如果连接条件涉及了最终不需要的属性（L₃、L₄ 非空），下推投影后连接结果中会多出这些属性，必须在最外层再做一次投影去掉它们。只有当 L₃ = L₄ = ∅ 时（即 8a 的情况），才不需要外层投影。

⚠️ 易错点 2：下推投影时漏掉连接需要的属性。比如连接条件是 R.A = S.A，但最终投影不包含 A，如果你在下推时把 A 也去掉了，连接就无法执行了。必须在下推的投影中保留 L₃ 和 L₄。

⚠️ 易错点 3：投影下推不一定总是有益的。如果关系本身列数就很少，或者投影操作本身的代价（排序去重等）很高，下推可能反而增加开销。优化器需要基于代价估算来决定是否下推。

## 三条规则的综合对比与联系

| 规则 | 操作               | 下推对象             | 核心条件                   | 优化效果                 |
| ---- | ------------------ | -------------------- | -------------------------- | ------------------------ |
| 7a   | 选择下推           | σ 下推到 ⋈ 的子树    | θ₀ 只涉及一个关系的属性    | 减少参与连接的行数       |
| 7b   | 选择分别下推       | σ 拆分后分别下推     | θ₁ 只涉及 E₁，θ₂ 只涉及 E₂ | 两边都减少行数           |
| 8a   | 投影下推           | Π 下推到 ⋈ 的子树    | θ 只涉及 L₁∪L₂ 中的属性    | 减少每行的列数/宽度      |
| 8b   | 投影下推（带补充） | Π 下推但保留连接属性 | θ 涉及 L₁∪L₂ 之外的属性    | 减少列数，但需外层再投影 |

选择下推减少的是行数（纵向缩小），投影下推减少的是列数（横向缩小）。两者结合使用，可以最大限度地缩小中间结果的大小。

在实际的查询优化器中，通常的优化策略顺序是：
1. 先做选择下推（效果最显著，因为减少行数对后续所有操作都有好处）
2. 再做投影下推（减少每行宽度）
3. 然后选择连接顺序（用规则 5、6）
4. 最后为每个操作选择物理算法
