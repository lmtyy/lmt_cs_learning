### 第一页：代价估计

**标题**：代价估计  
**核心任务**：在决定用哪种物理执行方式之前，先估算不同方案的资源消耗。

#### 1. 为什么需要代价估计？

- **背景回顾**：经过逻辑优化，我们得到了一棵高效的**逻辑查询树**（比如条件被下推了）。
- **新的问题**：这棵树上的每个逻辑算子（如 `⋈`、`σ`）都有多种**物理实现算法**。
    - 连接可以用：Nested-Loop Join、Hash Join、Sort-Merge Join。
    - 选择可以用：全表扫描、索引扫描。
- **决策依据**：究竟选哪种算法？这取决于**数据量**。例如，如果 `SC` 表里只有 10 行数据，用复杂的哈希连接反而开销更大；如果有 1000 万行，嵌套循环连接会慢到无法忍受。
- **结论**：必须有一个数学模型来**量化**每种方案的代价，这个模型就是**代价估计模型**。

#### 2. 代价估计依赖什么？—— 统计信息

- **知识点**：优化器不是凭空猜测，它需要**元数据**。
- **PPT 中的表格**：展示了 `SC` 表的片段（学号、课程号、成绩）。但优化器实际存储的是关于这张表的**统计摘要**，例如：
    - **表的元组数**：`SC` 表共有多少行？
    - **不同值的个数**：`Cno` 列有多少种不同的课程号？
    - **数据分布**：直方图，记录每个取值区间的频数。

#### 3. 基数估计示例

- **知识点**：**基数**指的是某个操作产生的结果集的行数。代价估计的核心就是**估算中间结果的大小**。
- **PPT 底部示例**：
    - 查询条件：`2 < Cno < 4`。
    - 图上标注了 `30`。
- **详细解释**：优化器会根据统计信息计算。假设 `SC` 表总共有 100 行，`Cno` 的取值范围是 1~5，且均匀分布。那么 `2 < Cno < 4` 对应 `Cno=3` 这一个值（如果离散），占比约 1/5，估算基数 ≈ 20。这里 PPT 标注 `30`，可能是根据更精确的直方图得出的结果。
- **意义**：知道了结果是 30 行，优化器就知道这个中间结果**很小**，可以放在内存里，后续连接优先选择 **Hash Join**；如果是 300 万行，可能就要选 **Sort-Merge Join** 以减少内存溢出。

### 第二页：物理优化

**标题**：物理优化  
**核心任务**：基于代价估计，从众多可能的物理执行计划中选出代价最小的一个。

#### 1. 定义

- **知识点**：物理优化的输入是**逻辑查询树**，输出是**物理执行计划**。
- **物理执行计划**与逻辑查询树的区别：逻辑查询树只说了“做什么”（Join），物理执行计划明确了“**怎么做**”（Hash Join）以及“**按什么顺序做**”（先 Join A 和 B，再 Join C）。

#### 2. 枚举的两个维度

根据 PPT 内容，物理优化主要枚举两方面的选择：

- **第一方面：逻辑算子的物理实现方式**
    - 示例：逻辑树上的 `σ`（选择）算子。
        - 物理实现 1：**全表顺序扫描**（适合无索引或高选择性）。
        - 物理实现 2：**索引扫描**（适合有 B+ 树索引且条件过滤率高）。
    - 示例：逻辑树上的 `⋈`（连接）算子。
        - 物理实现 1：**Nested-Loop Join**。
        - 物理实现 2：**Hash Join**。

- **第二方面：多表连接的顺序**
    - 示例：逻辑计划中有 `A ⋈ B ⋈ C`。
    - 枚举顺序 1：`(A ⋈ B) ⋈ C`。
    - 枚举顺序 2：`(A ⋈ C) ⋈ B`。
    - **重要性**：不同的连接顺序导致中间结果大小差异巨大，直接影响总代价。

#### 3. 过程与结果

- **过程**：枚举所有可能的组合（或使用动态规划、启发式规则剪枝搜索），利用第一页讲的**代价估计模型**计算每种组合的总 I/O 代价。
- **输出**：**代价最小的物理执行计划**。

#### 4. PPT 图例解析

- **左侧：逻辑计划树**
    ```
    Π (Sno, Sname, Grade)
         |
         ⋈ (Sno=Student.Sno)
        /  \
       σ (Cno='1')  Student
        |
       SC
    ```
    这是纯逻辑描述，没有指定算法。

- **右侧：物理计划树**
    ```
    Π (Sno, Sname, Grade)        (物理算法：物化执行)
         |
    Hash Join (Sno=Student.Sno)  (物理算法：哈希连接，内表为 Student)
       /          \
    Index Scan    Table Scan
    (Cno='1')     (Student)
    on SC
    ```
    - **改变 1**：`σ` 被具体化为 **Index Scan**（如果 `SC.Cno` 上有索引）。
    - **改变 2**：`⋈` 被具体化为 **Hash Join**。
    - **改变 3**：内表 `Student` 使用 **Table Scan**（全表扫描，因为可能很小）。
    - **注意 PPT 笔误**：右侧树里写的是 `σSno="1"`，根据上下文应该是 `σCno='1'`，这是 PPT 制作时的文字错误，知识点逻辑本身是清晰的。

### 第三页：磁盘 I/O 代价度量

**标题**：磁盘 I/O 代价度量  
**核心任务**：明确衡量代价的**单位**是什么。

#### 1. 代价的构成

- **知识点**：总代价 = 磁盘 I/O + CPU 计算 + 内存占用 + 网络通信（分布式场景）。

#### 2. 主导因素：磁盘 I/O

- **关键结论**：在大型数据库系统中，**磁盘 I/O 是最主要的瓶颈**。
- **原因**：CPU 和内存的操作是纳秒/微秒级，而机械硬盘寻道是毫秒级（相差数百万倍）。即便现在 SSD 普及，I/O 延迟依然是最大的开销之一。

#### 3. 度量指标：磁盘 I/O 的块数

- **知识点**：代价单位通常不用“秒”，而是用**磁盘块传输次数**。
- **定义**：数据在磁盘和内存之间以**块**为单位传输。
- **估算公式示例**：
    - 顺序扫描一张表：代价 ≈ 表占用的总块数。
    - 索引查找一条记录：代价 ≈ B+ 树的高度 + 读取数据块的次数。
- **意义**：有了统一单位，优化器才能比较“全表扫描”和“索引扫描”哪个更便宜。例如：
    - 表很小（只有 10 个块），全表扫描代价 = 10。
    - 索引查找虽然单次快，但如果要回表查 1000 次，I/O 次数可能远超 10，此时优化器会聪明地选择全表扫描。

### 第四页：查询执行

**标题**：查询执行  
**核心任务**：把纸面上的**物理计划**变成真正运行的程序，拿到数据。

#### 1. 输入与输出

- **输入**：物理执行计划（前一页的输出）。
- **输出**：查询结果（返回给客户端的数据行）。

#### 2. 计划编译

- **知识点**：现代数据库（如 PostgreSQL、SQL Server）会将物理计划**编译**成更底层的可执行代码。
    - 传统方式：**解释执行**（Volcano 模型迭代器），算子间通过 `next()` 函数调用拉取数据。灵活但函数调用开销大。
    - 现代方式：**JIT 编译**，将执行计划编译成一段紧凑的机器码或 LLVM IR，消除虚函数调用开销，大幅提升 CPU 密集型查询速度。

#### 3. 数据访问接口

- **知识点**：执行引擎不直接操作磁盘文件，而是调用**存储引擎**的 API。
- **PPT 图示**：`存储引擎 -> 数据访问接口 -> 编译执行`。
- **作用**：存储引擎负责缓冲池管理、页的加载与刷盘、索引的查找。执行引擎只关心“给我下一行数据”，底层是走索引还是走堆表由接口屏蔽。

#### 4. 计划执行与优化技术

- **执行模型**：如火山模型、向量化执行模型。
- **并行执行**：将一个查询的不同部分分发到多个 CPU 核心或分布式节点同时运行。
    - 示例：并行扫描一张大表的各个分区。

#### 5. 最终产物

- **PPT 底部**：箭头指向 **查询结果**。
- **总结**：经过解析、重写、代价估计、物理选择、编译执行，用户终于看到了 `Sname` 和 `Grade` 的数据。
