下面按照顺序详细讲解这两页 PPT 的知识点，重点围绕“为什么需要索引嵌套循环连接”以及“它的原理和使用条件”。

---

### 第1页：为什么基础嵌套循环连接很差，以及索引如何解决

**知识点1：基础嵌套循环连接的根本问题**
- 回顾之前的简单循环连接（stupid loop join）：对于外表的**每一个元组**，内表都要做一次**完整的顺序扫描**（sequential scan）。
- 如果外表有 `m` 个元组，内表有 `N` 个页面，那么内表就要被扫描 `m` 次，总磁盘 I/O 为 `M + m × N`，非常昂贵。
- 顺序扫描意味着即使内表只有极少数元组能满足连接条件，也要把整个内表的所有页面全部读一遍，大量无用 I/O。

**知识点2：用索引查找代替顺序扫描**
- 避免顺序扫描的直观方法：如果内表在连接属性上建有索引，就不需要扫描整个内表，而可以直接通过索引快速定位到匹配的元组所在的数据页面。
- “We can avoid sequential scans by using an index to find inner table matches.” 这句话是核心。
- 索引查找是一种**基于键的快速定位**，I/O 次数通常与内表大小无关，只与索引高度和匹配元组数相关。

**知识点3：利用已有索引**
- “Use an existing index for the join.” 数据库系统可能已经在内表的连接列上创建了索引（例如主键索引、外键索引，或用户显式创建的二级索引）。
- 连接执行时可以直接使用这些索引，而不需要在查询时临时构建，节省额外开销。这也意味着数据库物理设计时，可以为常见连接条件预先建好索引，以提高连接性能。

---

### 第2页：索引查找代替文件扫描的条件与算法流程

**知识点4：索引查找可代替顺序扫描的三个条件**
1. **连接类型必须是等值连接或自然连接**  
   - 索引通常基于等值查找（B+树索引），自然连接本质也是基于相等条件。对于非等值连接（如 `<`, `>`），一般索引无法高效支持查找，因此索引嵌套循环连接主要适用于等值连接。
2. **内表的连接属性上必须有可用的索引**  
   - 索引必须构建在内表（inner relation）的连接列上。外表元组每个值作为查找键，去内表索引中匹配。没有索引就退化为普通嵌套循环扫描。
3. **可以为了计算连接而临时构建索引**  
   - 即使没有预先存在的索引，如果连接操作代价很高且重复执行，系统可能临时在内表连接列上构建一个索引（如散列索引或 B+树），用完后丢弃。这可以看作一种优化：一次构建索引的成本，可以被后续大量索引查找的节省所抵消。
   - 注意：这通常适用于查询执行计划优化器决策。

**知识点5：算法流程**
- “For each tuple t_r in the outer relation r, use the index to look up tuples in s that satisfy the join condition with tuple t_r.”
- 具体步骤：
  1. 外表 r 按顺序逐元组读取（也可能是逐页读，但索引查找是基于元组值）
  2. 对于外表当前元组 t_r，提取其连接属性值 v。
  3. 在内表 s 的连接属性索引上进行查找，找出所有索引项等于 v 的记录指针。
  4. 通过指针（rowid/page）获取内表匹配的元组。
  5. 组合 t_r 与匹配的 t_s 输出。

**知识点6：与简单嵌套循环的对比**
- 外表依然是驱动表，顺序扫描外表的代价无法省略（需要读取外表的全部页面）。
- 但内表的访问由“全表顺序扫描”变为了“索引查找+回表”，内表代价与外表元组数成正比，但与内表页面数无直接比例关系。
- 因此，如果外表元组数很少，而内表很大，索引嵌套循环连接会极其高效（典型场景：外表是结果小表或经过过滤的小表）。

**知识点7：代价模型（隐含知识）**
- 虽然 PPT 未列出公式，但通常索引嵌套循环连接的 I/O 代价可表示为：
  \[
  \text{Cost} = M + m \times c
  \]
  其中：
  - `M`：外表 r 的页数（顺序扫描外表）。
  - `m`：外表 r 的元组数。
  - `c`：对于外表每个元组，使用索引查找内表匹配元组的平均代价。
    - `c` 通常由索引类型决定：例如 B+树索引，`c` 约为索引高度（2-4）+ 回表读取的数据页数（如果索引是聚簇的，回表可能只需 1 页；如果是非聚簇索引，且匹配多行，则可能需要多次 I/O）。
  - 如果连接条件是内表的 **键（key）**，则每个外表元组最多匹配一行，`c` 更小；如果是非键属性，可能匹配多行，代价增大。
- 对比块嵌套循环：BNL 代价为 `M + ⌈M/(B-2)⌉ × N`，当内表很大时 `N` 很大，BNL 依然很高。而索引 Nested Loop 的代价与 `N` 解耦，只要索引能快速定位，即使内表有 1 亿页，探针代价也维持在几页 I/O，优势巨大。

**知识点8：外表选择策略与前面一致**
- 索引嵌套循环中，外表仍然是驱动表，因此外表元组数 `m` 越小越好。这与简单嵌套循环中外表选小表的原则一致。
- 同样，如果优化器可以选择，通常是让查询过滤后剩余元组更少的表做外表，或者让连接完成后能够立即被外层过滤的关系作为外表。

**知识点9：临时构建索引的现实场景**
- 如果查询是一个重要报表，需要反复执行，且内表没有合适的索引，数据库可能选择**用散列连接**（Hash Join）代替构建索引，因为哈希连接是专为连接设计的临时索引（哈希表）。这是下一类算法的内容。
- 这里提到“Can construct an index just to compute a join.” 是为后续讲 Hash Join 埋下伏笔，因为哈希表本质上就是一种临时索引。

---

### 总结
这两页 PPT 清晰地给出了**索引嵌套循环连接**的动机与原理：
1. 基础嵌套循环连接在内表上反复做顺序扫描，代价太高。
2. 如果内表连接列上有索引（或可临时创建索引），就能将顺序扫描替换为索引查找，大幅度降低 I/O。
3. 适用条件：等值连接/自然连接 + 内表有索引。
4. 性能上，代价主要取决于外表元组数和每次索引探针的开销，而非内表大小，特别适合外表小、内表大的连接场景。

这一算法承接前文的嵌套循环，开启了“利用索引加速连接”的优化方向，也为后面哈希连接、归并连接等更高级算法提供了对比基础。
