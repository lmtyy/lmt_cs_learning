## 第一页：Introduction —— 查询优化的定义与示例查询

### 1. 什么是查询优化？

> "Query optimization is the process of selecting the most efficient query-evaluation plan from among the many strategies usually possible for processing a given query."

查询优化是指：对于一条给定的 SQL 查询，从众多可能的执行策略中，选出代价最小（最高效）的那个查询执行计划（query-evaluation plan）。

这里有几个关键概念需要理解：

- 一条 SQL 查询是声明式的（declarative）：用户只说"我要什么数据"，不说"怎么去拿"。
- 数据库系统需要自己决定"怎么拿"，这就是查询处理（query processing）。
- 同一条查询可能有很多种不同的执行方式（不同的关系代数表达式、不同的物理算法），它们的结果相同但代价可能天差地别。
- 查询优化器（query optimizer）的任务就是在这些方案中找到（或近似找到）代价最低的那个。

查询优化是数据库系统中最核心、最复杂的模块之一。一个好的优化器可以让查询快几个数量级，一个差的优化器可能让简单查询跑几天。

### 2. 示例查询

PPT 给了一个具体的查询例子：

> "Find the names of all customers who have an account at any branch located in Brooklyn."

翻译：找出所有在 Brooklyn 的支行有账户的客户姓名。

对应的关系代数表达式：

$$\Pi_{customer\text{-}name}(\sigma_{branch\text{-}city = \text{"Brooklyn"}}(branch \bowtie (account \bowtie depositor)))$$

这个表达式从内到外的执行顺序是：
1. 先做 account ⋈ depositor（账户表和存款人表的自然连接）
2. 再把结果和 branch 做自然连接
3. 对连接结果做选择 σ_{branch-city="Brooklyn"}（只保留 Brooklyn 的行）
4. 最后做投影 Π_{customer-name}（只取客户姓名列）

这个表达式是正确的，但不一定是最高效的。为什么？因为：
- 第一步 account ⋈ depositor 可能产生一个非常大的中间结果
- 然后再和 branch 连接，又产生一个大的中间结果
- 最后才做选择过滤，但其实大部分数据在过滤后都被丢弃了

如果能把选择操作提前（下推），先过滤 branch 表只保留 Brooklyn 的支行，再做连接，中间结果会小很多，效率会高很多。这就是查询优化要做的事情。

### 3. 涉及的三个关系

- branch(branch-name, branch-city, assets)：支行表
- account(account-number, branch-name, balance)：账户表
- depositor(customer-name, account-number)：存款人表

它们之间的连接关系：
- branch 和 account 通过 branch-name 连接
- account 和 depositor 通过 account-number 连接

## 第二页：Introduction —— 等价表达式与不同的查询树

### 1. 同一查询的不同执行方式

PPT 指出，对于同一个查询，存在多种不同的执行方式（alternative ways），主要来自两个维度：

#### 维度一：等价表达式（Equivalent Expressions）

同一个查询可以用不同的关系代数表达式来表示，只要它们对任何合法的数据库实例都产生相同的结果，就称为等价表达式。

比如上面的查询，至少有以下两种等价的关系代数表达式（对应图中的两棵查询树）：

左边的查询树（原始表达式）：
```
Π_customer_name
    |
σ_branch_city=Brooklyn
    |
   ⋈
  / \
branch  ⋈
       / \
   account depositor
```

执行顺序：
1. 先连接 account ⋈ depositor
2. 再连接 branch ⋈ (上一步结果)
3. 然后选择 branch_city = Brooklyn
4. 最后投影 customer_name

右边的查询树（优化后的表达式）：
```
Π_customer_name
    |
   ⋈
  / \
 ⋈   ⋈
 |   / \
σ_branch_city=Brooklyn  account  depositor
 |
branch
```

执行顺序：
1. 先对 branch 做选择 σ_{branch_city=Brooklyn}，只保留 Brooklyn 的支行（结果很小）
2. 同时连接 account ⋈ depositor
3. 再把两个结果连接
4. 最后投影 customer_name

但更仔细看右边的树，它实际上是：
1. 先对 branch 做选择过滤（选出 Brooklyn 的支行）
2. 把过滤后的 branch 和 account ⋈ depositor 的结果做连接
3. 投影

关键区别在于：选择操作 σ 的位置不同。

左边的树：选择在最上面，连接完所有表之后才过滤 → 中间结果巨大
右边的树：选择被下推到 branch 表上，先过滤再连接 → 中间结果小很多

这就是查询优化中最经典的优化技术之一：选择下推（selection push-down）。

#### 维度二：不同的物理算法（Different Algorithms for Each Operation）

即使关系代数表达式确定了，每个操作符（⋈、σ、Π）还可以选择不同的物理算法来实现：

- 连接 ⋈ 可以用：Nested Loop Join、Block Nested Loop Join、Index Nested Loop Join、Sort-Merge Join、Hash Join
- 选择 σ 可以用：全表扫描（linear scan）、索引扫描（index scan，B+树或哈希索引）
- 投影 Π 如果需要去重，可以用：排序去重、哈希去重

所以"等价表达式"决定了操作的逻辑顺序，"物理算法选择"决定了每个操作的具体实现方式。两者的组合构成了巨大的搜索空间，查询优化器需要在这个空间中找到最优（或近似最优）的方案。

### 2. 为什么右边的树更好？

假设 branch 表有 1000 个支行，但只有 5 个在 Brooklyn：

左边的树：
- account ⋈ depositor 可能产生几十万行
- 再和 1000 个支行连接，中间结果更大
- 最后过滤只留 Brooklyn 的，大部分工作白做了

右边的树：
- 先过滤 branch，只剩 5 行
- 5 行和 account ⋈ depositor 连接，中间结果小很多
- 效率可能提升几个数量级

## 第三页：Introduction (Cont.) —— 查询执行计划（Evaluation Plan）

### 1. 什么是执行计划？

> "An evaluation plan defines exactly what algorithm is used for each operation, and how the execution of the operations is coordinated."

执行计划（evaluation plan）= 查询树 + 每个节点的具体算法 + 操作之间的协调方式

它比查询树更具体，不仅规定了"做什么操作、按什么顺序"，还规定了"每个操作用什么算法实现"以及"操作之间如何传递数据"。

### 2. 图中的执行计划详解

PPT 给出了一个完整的执行计划示例，我从底部往上逐层分析：

#### 最底层：两个叶子节点

左边：branch 表
- 对 branch 做选择 σ_{branch_city=Brooklyn}
- 使用的算法：use index 1（使用索引 1，说明 branch_city 上有一个索引）
- 这意味着不需要全表扫描 branch，直接通过索引定位到 Brooklyn 的支行，非常快

右边：account 表
- 对 account 做选择 σ_{balance < 1000}
- 使用的算法：use linear scan（线性扫描，即全表扫描）
- 这说明 balance 上没有合适的索引，只能逐行扫描

注意：这里多了一个原始查询中没有的条件 balance < 1000，这可能是为了演示不同选择算法的选择，或者是一个扩展的查询。

#### 中间层：连接操作

左下角的连接：σ(branch) ⋈ σ(account)
- 使用的算法：merge join（排序归并连接）
- 两个输入都标注了 pipeline（流水线），意思是选择操作的结果不需要物化（写到磁盘），而是直接以流水线的方式传递给连接操作

上面的连接：(上一步结果) ⋈ depositor
- 使用的算法：hash join（哈希连接）

#### 最顶层：投影

Π_{customer_name}
- 标注了 sort to remove duplicates（通过排序来去除重复值）
- 因为投影可能产生重复的 customer_name，需要去重

### 3. Pipeline（流水线）的概念

图中标注了两个 pipeline，这是一个非常重要的概念：

物化（Materialization）：一个操作的结果完整地写到磁盘上的临时文件中，然后下一个操作再从磁盘读取。代价高，因为涉及额外的磁盘 I/O。

流水线（Pipelining）：一个操作产生一个元组后，立即传递给下一个操作处理，不需要等所有结果都产生完，也不需要写磁盘。代价低，因为避免了中间结果的磁盘 I/O。

在这个执行计划中：
- σ_{branch_city=Brooklyn}(branch) 的结果通过 pipeline 直接传给 merge join，不写磁盘
- σ_{balance<1000}(account) 的结果也通过 pipeline 直接传给 merge join
- 但 merge join 的结果可能需要物化后再传给 hash join（因为 hash join 需要分区，不太适合纯流水线）

流水线能显著减少 I/O 代价，是查询优化器在制定执行计划时的重要考量。

### 4. 为什么不同操作选择不同算法？

- σ_{branch_city=Brooklyn} 用索引：因为 branch_city 上有索引，且 Brooklyn 的支行很少（高选择性），索引扫描比全表扫描快得多
- σ_{balance<1000} 用线性扫描：可能 balance 上没有索引，或者满足条件的元组比例很高（低选择性），全表扫描反而比索引扫描更划算
- 第一个连接用 merge join：可能两个输入已经按连接属性排序（或者排序代价不大）
- 第二个连接用 hash join：可能数据量大且是等值连接，hash join 效率最高
- 投影用排序去重：需要消除重复的 customer_name

这些选择都是查询优化器根据统计信息和代价估算做出的决策。

## 第四页：Introduction (Cont.) —— 基于代价的查询优化步骤

### 1. 不同执行计划的代价差异可以非常巨大

> "Cost difference between evaluation plans for a query can be enormous. E.g. seconds vs. days in some cases."

这不是夸张。举个例子：
- 一个涉及三个大表连接的查询
- 最差的计划：三层嵌套循环，没有索引 → 可能需要几天
- 最优的计划：选择下推 + 索引连接 + 流水线 → 可能只需要几秒

差距可以达到 5~6 个数量级。这就是为什么查询优化如此重要——它直接决定了数据库的性能。

### 2. 基于代价的查询优化（Cost-Based Query Optimization）的三个步骤

这是查询优化器的核心工作流程：

#### 步骤 1：使用等价规则生成逻辑上等价的表达式

> "Generate logically equivalent expressions using equivalence rules"

等价规则（equivalence rules）是一组关系代数的变换规则，保证变换前后的表达式对任何数据库实例都产生相同的结果。常见的等价规则包括：

- 选择下推（Push Selection Down）：σ_θ(R ⋈ S) = σ_θ(R) ⋈ S（如果 θ 只涉及 R 的属性）
  - 这是最重要的优化规则之一，把选择操作尽量推到查询树的底部，尽早过滤数据
- 连接交换律（Join Commutativity）：R ⋈ S = S ⋈ R
  - 交换连接的左右输入，可能影响算法选择（比如哪个做 build input）
- 连接结合律（Join Associativity）：(R ⋈ S) ⋈ T = R ⋈ (S ⋈ T)
  - 改变连接的顺序，对代价影响巨大
- 投影下推（Push Projection Down）：尽早去掉不需要的列，减少中间结果的大小
- 选择的合并与拆分：σ_{θ1 ∧ θ2}(R) = σ_{θ1}(σ_{θ2}(R))

通过这些规则，优化器可以从原始表达式出发，生成大量等价的候选表达式。

对于 n 个表的连接，仅连接顺序就有 (2(n-1))! / (n-1)! 种（Catalan 数），当 n 较大时搜索空间爆炸式增长。所以实际的优化器会用动态规划或启发式方法来剪枝。

#### 步骤 2：对等价表达式标注具体算法，得到候选执行计划

> "Annotate resultant expressions to get alternative query plans"

对每个等价表达式，为每个操作节点选择具体的物理算法（如前面第三页所示），形成完整的执行计划。

一个逻辑表达式可以产生多个执行计划，因为每个操作都有多种算法可选。

#### 步骤 3：基于估算代价选择最优计划

> "Choose the cheapest plan based on estimated cost"

对每个候选执行计划，用代价模型估算其总代价（主要是磁盘 I/O 次数），然后选择代价最小的那个。

注意是"estimated cost"（估算代价），不是精确代价。因为在查询执行之前，我们不知道实际的数据分布和中间结果大小，只能基于统计信息来估算。

### 3. 代价估算依赖的三类信息

#### (1) 关系的统计信息（Statistical Information about Relations）

数据库系统会维护每个关系的统计信息，包括：
- nᵣ：关系 r 的元组数（number of tuples）
- bᵣ：关系 r 占的磁盘块数（number of blocks）
- V(A, r)：属性 A 在关系 r 中的不同值个数（number of distinct values）
- lᵣ：每个元组的字节数
- fᵣ：每个块能容纳的元组数（blocking factor）

这些统计信息通常存储在系统目录（system catalog）中，通过 ANALYZE 或 UPDATE STATISTICS 命令更新。

#### (2) 中间结果的统计估算（Statistics Estimation for Intermediate Results）

一个复杂查询有多个操作，每个操作的输出是下一个操作的输入。要估算整个计划的代价，就需要估算每个中间结果的大小。

比如：σ_{branch_city=Brooklyn}(branch) 的结果有多少行？
- 如果 branch 有 1000 行，branch_city 有 50 个不同的值
- 假设均匀分布，估算结果 = 1000 / 50 = 20 行

再比如：R ⋈ S 的结果有多少行？
- 这取决于连接属性的值分布，估算公式更复杂

中间结果的估算是否准确，直接影响优化器选择的计划是否真的最优。如果统计信息过时或分布假设不成立，优化器可能选出一个很差的计划。

#### (3) 算法的代价公式（Cost Formulae for Algorithms）

每种物理算法都有对应的代价公式，比如前面学过的：
- Hash Join：3(bᵣ + bₛ)
- Block Nested Loop Join：bᵣ + ⌈bᵣ/(M-2)⌉ × bₛ
- Sort-Merge Join：bᵣ + bₛ + 排序代价

优化器把统计信息代入这些公式，就能估算出每个候选计划的总代价，从而选出最优的。

### 4. 整体工作流程总结

```
SQL 查询
    ↓ 解析
关系代数表达式（初始查询树）
    ↓ 步骤1：等价规则变换
多个等价的关系代数表达式
    ↓ 步骤2：标注物理算法
多个候选执行计划
    ↓ 步骤3：代价估算 + 选择
最优执行计划
    ↓ 执行
查询结果
```
