## 第一页：Hash-Join 的基本概念与分区机制

### 1. 什么是 Hash-Join？

Hash-Join 是数据库系统中实现连接操作（Join）的一种物理算法。它和 Nested-Loop Join、Sort-Merge Join 并列，是三大经典连接算法之一。

它的核心思想很朴素：如果两个元组能连接，那它们在连接属性上的值一定相等 → 相等的值经过同一个哈希函数一定得到相同的哈希值 → 所以我们可以用哈希把"可能匹配的元组"提前归到一起，避免无意义的比较。

### 2. 适用条件

PPT 明确说了：只适用于等值连接（equi-join）和自然连接（natural join）。

为什么？因为哈希函数的性质是：相同的输入 → 相同的输出。这只能保证"值相等"的元组落入同一个桶。如果连接条件是 `r.A > s.B` 这种范围条件，哈希值相同并不能说明什么，所以 Hash-Join 就不适用了。

对比一下：
- Nested-Loop Join：什么连接条件都能用，但效率最低
- Sort-Merge Join：适合等值连接，也能处理某些范围连接
- Hash-Join：只能等值连接，但在等值连接场景下通常效率最高

### 3. 哈希函数 h 的作用

PPT 说用一个哈希函数 *h* 把两个关系的元组分到 *n* 个桶里。这里有几个细节：

- *h* 的输入是元组在连接属性（JoinAttrs）上的值。比如 `r(A, B)` 和 `s(B, C)` 做自然连接，JoinAttrs 就是 B，那 h 的输入就是每个元组的 B 属性值。
- *h* 的输出范围是 {0, 1, ..., n-1}，也就是桶的编号。
- *n* 的选择很有讲究：通常选择使得较小关系的每个分区能放进内存。如果内存有 M 个块（block），一般取 n ≈ ⌈b_s / M⌉ + 1（b_s 是较小关系的块数），留一些余量。

### 4. 分区的形式化定义

对关系 *r*：
- 分成 r₀, r₁, ..., r_{n-1}
- 对于 *r* 中的任意元组 tᵣ，计算 i = h(tᵣ[JoinAttrs])，然后把 tᵣ 放入分区 rᵢ

对关系 *s*：
- 分成 s₀, s₁, ..., s_{n-1}
- 对于 *s* 中的任意元组 tₛ，计算 i = h(tₛ[JoinAttrs])，然后把 tₛ 放入分区 sᵢ

举个具体例子：假设 r 有一个元组 tᵣ，它的 JoinAttrs 值是 "张三"，h("张三") = 2，那这个元组就被放入 r₂。如果 s 中有个元组 tₛ 的 JoinAttrs 值也是 "张三"，h("张三") = 2，它就被放入 s₂。后续只需要在 r₂ 和 s₂ 之间做匹配就能找到它们。

## 第二页：Hash-Join 的分区过程图示

### 图的结构

这张图分为四个部分：

- 最左边：关系 *r* 的原始数据（一个竖条表示整个关系存储在磁盘上）
- 中间左侧：*r* 经过哈希分区后的 5 个桶（r₀ ~ r₄）
- 中间右侧：*s* 经过哈希分区后的 5 个桶（s₀ ~ s₄）
- 最右边：关系 *s* 的原始数据

### 箭头的含义

从 *r* 到 r 的分区的箭头：表示 *r* 中的元组根据哈希值被分配到不同的桶。注意箭头是交叉的，说明原始数据中相邻的元组不一定落入同一个桶——这取决于它们连接属性的哈希值。

从 *s* 到 s 的分区的箭头：同理。

中间 rᵢ 和 sᵢ 之间的双向箭头：这是最关键的部分，表示只有同编号的分区之间才需要做连接匹配：
- r₀ ↔ s₀
- r₁ ↔ s₁
- r₂ ↔ s₂
- r₃ ↔ s₃
- r₄ ↔ s₄

### 这个过程在实际执行中是怎样的？

Hash-Join 实际上分两个阶段：

阶段一：分区阶段（Partition Phase / Build Phase）
1. 顺序扫描关系 *r*，对每个元组计算 h(tᵣ[JoinAttrs])，写入对应的分区文件
2. 顺序扫描关系 *s*，对每个元组计算 h(tₛ[JoinAttrs])，写入对应的分区文件
3. 这一步的 I/O 代价：读 r 和 s 各一次，写分区各一次 → 2(bᵣ + bₛ) 次块传输

阶段二：探测阶段（Probe Phase / Match Phase）
1. 对于每一对 (rᵢ, sᵢ)：
   - 把较小的那个分区（比如 sᵢ）读入内存，在内存中建一个哈希表（注意：这里用的是另一个哈希函数 h₂，和分区用的 h 不同）
   - 然后逐个读取另一个分区（rᵢ）的元组，用 h₂ 在内存哈希表中查找匹配
   - 输出所有匹配的结果
2. 这一步的 I/O 代价：读所有分区各一次 → bᵣ + bₛ 次块传输

总 I/O 代价：3(bᵣ + bₛ) 次块传输（不考虑溢出的情况下）。

### 为什么图中是 5 个桶？

这只是一个示例，实际的 n 取决于内存大小和关系大小。图中用 n=5 只是为了方便展示。

## 第三页：Hash-Join 正确性的证明——为什么只比较同编号分区就够了

### 核心论点

rᵢ 中的元组只需要和 sᵢ 中的元组比较，不需要和任何 sⱼ（j ≠ i）中的元组比较。

### 详细推理过程

这是一个反证法的思路：

假设 *r* 中的元组 tᵣ 和 *s* 中的元组 tₛ 能够满足连接条件（即它们在 JoinAttrs 上的值相等）。

第一步：因为满足连接条件，所以 tᵣ[JoinAttrs] = tₛ[JoinAttrs]。

第二步：因为哈希函数是确定性的（相同输入一定得到相同输出），所以 h(tᵣ[JoinAttrs]) = h(tₛ[JoinAttrs])。假设这个值等于 i。

第三步：根据分区规则，tᵣ 一定在 rᵢ 中，tₛ 一定在 sᵢ 中。

结论：任何能匹配的元组对，一定在同编号的分区中。所以我们只比较同编号分区，不会遗漏任何结果。

反过来说：如果 tᵣ 在 rᵢ 中，tₛ 在 sⱼ 中（i ≠ j），那说明 h(tᵣ[JoinAttrs]) ≠ h(tₛ[JoinAttrs])，进而说明 tᵣ[JoinAttrs] ≠ tₛ[JoinAttrs]（注意：哈希冲突的情况除外，但反过来值不同哈希一定可能不同），所以它们一定不满足连接条件，比较了也是白比较。

### 注意：哈希冲突不影响正确性

你可能会想：哈希函数会有冲突啊，不同的值可能映射到同一个桶。没错，但这只会导致同一个桶里有一些"不匹配"的元组对（false positive），在探测阶段做精确比较时会被过滤掉。关键是：不会有"匹配的元组对被分到不同桶"的情况（no false negative）。

所以哈希冲突只影响效率（桶越均匀效率越高），不影响正确性。

### 这带来了多大的效率提升？

假设两个关系各有 N 个元组，分成 n 个桶，每个桶平均 N/n 个元组：
- 不用哈希：需要比较 N × N = N² 次
- 用哈希分区后：每对桶比较 (N/n) × (N/n) 次，共 n 对桶，总比较次数 = n × (N/n)² = N²/n

比较次数直接降为原来的 1/n，这就是 Hash-Join 的威力。
