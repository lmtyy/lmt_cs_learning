## 第一页：典型的启发式规则

这一页总结了四条经典的启发式优化规则。所谓"启发式"（heuristic），就是不需要精确计算代价，凭经验就知道"这样做通常更好"的规则。

### 规则 1：选择运算应尽可能先做

> "在优化策略中这是最重要、最基本的一条"

这是查询优化的第一原则。原因很简单：选择操作会过滤掉大量不满足条件的元组，减少后续所有操作（连接、投影等）需要处理的数据量。

越早做选择，中间结果越小，后续操作的 I/O 和计算代价越低。

理论依据：等价规则 7（选择对连接的分配律），允许我们把选择从连接上方移到连接下方。

举例：如果 Student 有 10000 行，SC 有 50000 行，但满足 Cno='2' 的 SC 只有 500 行。
- 不下推：先做 Student × SC = 5亿行，再过滤
- 下推后：先过滤 SC 得到 500 行，再做 Student × σ(SC) = 500万行
- 差距：1000 倍

### 规则 2：把投影运算和选择运算同时进行

在执行选择操作的同时，顺便把不需要的列也去掉（投影）。这样可以避免单独做一次投影操作的开销，同时减少中间结果每行的宽度。

实际实现中，选择和投影经常合并成一个"扫描 + 过滤 + 投影"的操作，一次扫描完成两件事。

### 规则 3：把投影同其前或其后的双目运算结合起来

双目运算指的是连接（⋈）、笛卡尔积（×）、并（∪）、差（-）等需要两个输入的操作。

含义：不要单独做一个投影操作，而是把投影"嵌入"到连接操作中。比如在做连接的时候，输出结果时就只输出需要的列，不输出多余的列。

这样可以减少一次额外的扫描和写入。

### 规则 4：把某些选择同在它前面要执行的笛卡尔积结合起来成为一个连接运算

这就是等价规则 4a 的直接应用：σ_θ(E₁ × E₂) = E₁ ⋈_θ E₂

笛卡尔积的代价是灾难性的（两个关系的行数相乘），而连接操作可以用高效的算法（Hash Join、Merge Join 等）实现，代价远小于笛卡尔积。

所以只要笛卡尔积后面跟着一个涉及两个关系属性的选择，就应该把它们合并成连接。

⚠️ 这四条规则的优先级：规则 1 > 规则 4 > 规则 2 ≈ 规则 3。选择下推是最重要的，其次是消除笛卡尔积。

## 第二页：一个例子 —— 初始查询树

### 查询

> 求选修了 2 号课程的学生姓名。

```sql
SELECT Student.Sname
FROM Student, SC
WHERE Student.Sno = SC.Sno AND SC.Cno = '2';
```

涉及两个关系：
- Student(Sno, Sname, ...)：学生表
- SC(Sno, Cno, Grade)：选课表

### 初始的关系代数语法树

SQL 查询直接翻译成关系代数后，得到的是一个"未优化"的查询树：

```
    Π_Sname
       |
  σ_{SC.Cno='2'}
       |
  σ_{Student.Sno=SC.Sno}
       |
      ×
     / \
Student  SC
```

从下往上读：
1. 先做 Student × SC（笛卡尔积）
2. 再做 σ_{Student.Sno=SC.Sno}（选择出学号匹配的行）
3. 再做 σ_{SC.Cno='2'}（选择课程号为 2 的行）
4. 最后做 Π_{Sname}（只保留学生姓名）

这个查询树是最朴素的翻译结果，效率很差：
- 第一步笛卡尔积就产生了巨大的中间结果（Student 行数 × SC 行数）
- 然后才开始过滤，大部分工作都浪费了

注意：WHERE 子句中的两个条件（Student.Sno=SC.Sno 和 SC.Cno='2'）被翻译成了两个独立的选择操作（用了规则 1 的串接律拆分）。

## 第三页：选择操作下移

这一页展示了优化的第一步：把选择操作尽可能下推到查询树的底部。

### 左边：优化前的查询树（和第二页一样）

```
    Π_Sname
       |
  σ_{SC.Cno='2'}
       |
  σ_{Student.Sno=SC.Sno}
       |
      ×
     / \
Student  SC
```

### 右边：选择下推后的查询树

```
    Π_Sname
       |
  σ_{Student.Sno=SC.Sno}
       |
      ×
     / \
Student  σ_{Cno='2'}
            |
           SC
```

### 变换过程详解

原来有两个选择操作在笛卡尔积上方：
- σ_{SC.Cno='2'}：只涉及 SC 的属性 Cno
- σ_{Student.Sno=SC.Sno}：同时涉及 Student 和 SC 的属性

分析哪些可以下推：

σ_{SC.Cno='2'} 只涉及 SC 的属性 → 可以下推到 SC 上（应用规则 7a）

σ_{Student.Sno=SC.Sno} 同时涉及 Student 和 SC 的属性 → 不能下推到任何一个关系上，必须留在笛卡尔积上方

所以变换后：
1. σ_{Cno='2'} 被下推到 SC 上，先过滤 SC 只保留课程号为 2 的选课记录
2. σ_{Student.Sno=SC.Sno} 留在笛卡尔积上方

效果：假设 SC 有 50000 行，其中 Cno='2' 的只有 500 行。
- 优化前：Student(10000) × SC(50000) = 5 亿行中间结果
- 优化后：Student(10000) × σ(SC)(500) = 500 万行中间结果
- 中间结果缩小了 100 倍

进一步优化：σ_{Student.Sno=SC.Sno} + × 可以合并成连接操作（应用规则 4a），变成 Student ⋈_{Sno} σ_{Cno='2'}(SC)，这样就可以用高效的连接算法了。

## 第四页：投影操作下移

这一页讲的是优化的第二步：在选择下推完成后，进一步把投影操作下推。

### 投影下移的总体原则

> "将投影操作尽量朝叶端移动"

目的：尽早去掉不需要的列，减少中间结果每行的宽度，从而减少 I/O 量和内存占用。

### 投影下移过程中遇到选择操作的处理

投影在向下移动的过程中，可能会"路过"一个选择操作。这时分两种情况：

情况 (1)：选择条件涉及的属性在投影属性范围内

比如投影是 Π_{Sname, Sno}，选择是 σ_{Sno='001'}。Sno 在投影保留的属性中，所以投影可以直接穿过选择继续下移。

情况 (2)：选择条件涉及的属性不在投影属性范围内

比如投影是 Π_{Sname}，选择是 σ_{Sno=SC.Sno}。Sno 不在 Π_{Sname} 中，但选择需要 Sno 来做比较。

这时不能直接把 Π_{Sname} 下推到选择下面（否则 Sno 被丢掉了，选择就无法执行）。

解决方法：在选择操作下面增加一个新的投影操作，这个新投影包含原投影属性 + 选择条件涉及的属性。然后继续下移这个新投影。

即：Π_{Sname} → σ_{Sno=SC.Sno} → 变成 → Π_{Sname} → σ_{Sno=SC.Sno} → Π_{Sname, Sno}

### 投影下移遇到其他操作的处理

- 遇到投影操作：合并（规则 3，投影级联律）
- 遇到双目运算（连接、笛卡尔积等）：按规则 8 分配到相关分支上

## 第五页：投影下移的完整示例

这一页展示了对前面选择下推后的查询树，进一步做投影下推的结果。

### 左边：选择下推后的查询树（第三页的结果）

```
    Π_Sname
       |
  σ_{Student.Sno=SC.Sno}
       |
      ×
     / \
Student  σ_{Cno='2'}
            |
           SC
```

### 右边：投影下推后的查询树

```
  σ_{Student.Sno=SC.Sno}
       |
      ×
     / \
Π_{Sname,Sno}   Π_{Sno}
    |              |
Student      σ_{Cno='2'}
                   |
                  SC
```

### 变换过程详解

原来最顶层有 Π_{Sname}，我们要把它尽量下推。

第一步：Π_{Sname} 遇到 σ_{Student.Sno=SC.Sno}

Sname 在投影中，但选择条件需要 Student.Sno 和 SC.Sno，这两个属性不在 Π_{Sname} 中。

按照情况 (2) 的处理方法：需要保留 Sno 给选择用。所以投影暂时变成 Π_{Sname, Student.Sno, SC.Sno}。

但注意：最终结果中 Π_{Sname} 的效果可以在最后输出时实现（或者和 σ 合并执行），所以图中最顶层变成了 σ 而不是 Π。实际上 Π_{Sname} 可以和最终输出合并。

第二步：投影继续下移，遇到笛卡尔积 ×

笛卡尔积是双目运算，投影需要分配到两个分支（规则 8）：

左分支（Student）：最终需要 Sname，连接需要 Student.Sno → Π_{Sname, Sno}(Student)
右分支（SC）：最终不需要 SC 的任何属性（Sname 来自 Student），但连接需要 SC.Sno → Π_{Sno}(σ_{Cno='2'}(SC))

所以：
- Student 上加了 Π_{Sname, Sno}：去掉了 Student 中除 Sname 和 Sno 之外的所有列（如 Sage、Sdept 等）
- SC 上加了 Π_{Sno}：先过滤 Cno='2'，再只保留 Sno 列（去掉了 Cno、Grade 等）

### 优化效果分析

假设 Student 有 5 列，SC 有 3 列：

优化前笛卡尔积的每行：5 + 3 = 8 列
优化后笛卡尔积的每行：2 + 1 = 3 列

每行的宽度减少了 62.5%，意味着同样的内存可以容纳更多的元组，磁盘 I/O 也相应减少。

### 完整的优化链路总结

```
原始查询树                    选择下推                     投影下推
Π_Sname                     Π_Sname                    σ_{Sno=Sno}
  |                           |                           |
σ_{Cno='2'}                σ_{Sno=Sno}                   ×
  |                           |                         /   \
σ_{Sno=Sno}         →        ×            →      Π_{Sname,Sno}  Π_{Sno}
  |                         /   \                    |           |
  ×                    Student  σ_{Cno='2'}       Student    σ_{Cno='2'}
 / \                              |                            |
Student SC                       SC                           SC
```

最后还可以进一步优化：把 σ_{Student.Sno=SC.Sno} + × 合并成 ⋈_{Sno}（规则 4a），得到最终的优化查询树：

```
Π_{Sname}(Π_{Sname,Sno}(Student) ⋈_{Sno} Π_{Sno}(σ_{Cno='2'}(SC)))
```
