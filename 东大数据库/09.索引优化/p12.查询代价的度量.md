以下是针对这一页的详细知识点拆解：

### 1. 核心概念：Measures of Query Cost

**知识点**：查询代价用什么来衡量？

- **定性理解**：数据库优化器是一个“会计”，它必须能算出方案 A 花多少钱，方案 B 花多少钱，才能做出选择。
- **核心度量指标**：**磁盘 I/O 次数**（传输的磁盘块数）。
    - **原因**：CPU 运算和内存复制是纳秒/微秒级的，而磁盘读取是毫秒级的（慢 100 万倍以上）。因此，**减少磁盘访问是优化的第一目标**。
- **辅助指标**：
    - **CPU 代价**：虽然占比小，但在纯内存计算或大规模哈希运算时不能忽略。
    - **网络代价**：在分布式数据库中尤为重要（数据跨节点 Shuffle）。

### 2. 这一页 PPT 的结构含义

这一页提纲中的三个项目 **Selection**、**Sort**、**Join** 并不是随意罗列的，它们代表了查询处理中**最难、最耗时、最需要被度量**的三个基本操作。

为什么是这三个？

- **Selection**：对应 SQL 中的 `WHERE` 过滤。
    - 它的代价决定了**基数估计**的起点：如果全表扫描代价是 1000 块，索引扫描代价是 3 块，选错就会导致性能灾难。
- **Sort**：对应 SQL 中的 `ORDER BY` 和 `GROUP BY`（分组通常依赖排序或哈希）。
    - 排序是**连接操作的前置步骤**（Sort-Merge Join）。
    - 排序也是**去重**、**分析函数**的基石。
    - 特殊点：当数据放不进内存时，必须使用**外排序**，这涉及大量磁盘 I/O 的来回读写，代价计算极其复杂。
- **Join**：对应 SQL 中的 `JOIN ... ON` 或隐式连接。
    - 这是关系数据库**最昂贵**的操作。两张百万级表的连接如果不加选择地做笛卡尔积，计算量是天文数字。
    - 连接算法的选择（Nested-Loop, Hash, Sort-Merge）完全依赖于代价度量的结果。

### 3. 隐含的流程逻辑

这一页暗示了接下来课程讲述的具体顺序：

1.  **首先建立代价模型**：确定 1 次顺序 I/O 等于多少单位，1 次随机 I/O 等于多少单位。
2.  **然后逐个分析单个算子的代价公式**：
    - **Selection 代价公式**：全表扫描代价 vs 索引扫描代价（需要结合索引高度和聚簇因子）。
    - **Sort 代价公式**：外归并排序的趟数 * 数据块数 * 2（一读一写）。
    - **Join 代价公式**：嵌套循环 = 外表块数 + 外表元组数 * 内表块数；哈希连接 = 建表 I/O + 探查 I/O。
3.  **最后组合代价**：一个复杂的执行计划树，其总代价 = 所有叶子节点操作（扫描） + 所有中间节点操作（排序、连接）的累加。

### 4. 深入理解：为什么度量单位必须是“块数”而不是“秒”？

这里有一个重要的知识点补充：**代价模型的独立性**。

- **问题**：如果代价模型输出的是“预计耗时 0.03 秒”，那么这个模型就与特定的硬盘转速（7200转 vs 15000转 vs SSD）绑定了。
- **数据库的做法**：代价模型输出的是**无量纲的数值**（例如 PostgreSQL 的 `total_cost` 是一个像 `1245.67` 这样的数字），这个数字是由 `磁盘块读取次数 + CPU 操作数折算成块读取次数的当量` 计算出来的。
- **意义**：这个数值只反映**逻辑资源消耗的相对大小**。无论是快硬盘还是慢硬盘，计划 A 的数值是计划 B 的十分之一，那 A 大概率比 B 快。这种设计让优化器具有跨硬件的通用性。
