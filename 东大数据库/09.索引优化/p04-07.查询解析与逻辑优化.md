好的，我们按照这四页 PPT 的顺序，从**查询解析**到**逻辑优化**，详细拆解每个阶段的知识点。这四页内容构成了查询处理的前半部分：**把 SQL 文本变成高效的代数表达式**。

### 第一页：查询解析

**标题**：查询解析  
**核心任务**：输入 SQL 语句，输出查询树（关系代数表达式树）。

#### 1. 词法分析
- **知识点**：扫描 SQL 字符串，识别并切分出有意义的**单词单元**。
- **详细解释**：
    - **关键词**：`SELECT`、`FROM`、`WHERE`、`AND` 等。
    - **标识符**：表名 `SC`、`Student`，列名 `Sname`、`Grade`、`Sno`、`Cno`。
    - **常量**：字符串 `'1'`。
    - **运算符**：等号 `=`。
- **PPT 中示例**：右侧的树状标签结构（如 `<select>`, `<from>`, `<where>` 等）是词法分析后构建的**解析树**雏形，反映了 SQL 语句的结构层次。

#### 2. 语法分析
- **知识点**：检查词法单元序列是否符合 SQL 语法规则，并构建**语法分析树**。
- **详细解释**：语法分析器利用上下文无关文法，将扁平的 Token 流组装成结构化的树。例如，确保 `SELECT` 后面跟着的是列名列表，`FROM` 后面是表名列表。PPT 中的标签树就是这一步的产物。

#### 3. 语义分析
- **知识点**：检查语句的**逻辑合法性**，并将语法树转换为**关系代数查询树**。
- **详细解释**：语法正确不等于语义正确。语义分析主要做：
    - **名字解析**：检查 `SC.Sno`、`Student.Sno`、`Cno`、`Sname`、`Grade` 是否真实存在于对应表中。
    - **类型检查**：检查 `Cno = '1'` 中列类型与常量类型是否兼容。
    - **作用域检查**：列名是否歧义（如两个表都有 `Sno`，必须加表名前缀）。
- **输出**：一棵**查询树**。PPT 底部展示了这棵树的结构：
    - **叶子节点**：关系表 `SC` 和 `Student`。
    - **中间节点 `×`**：笛卡尔积（对应 `FROM SC, Student`，隐式交叉连接）。
    - **上层节点 `σ`**：选择操作，条件为 `Cno="1" ∧ SC.Sno=Student.Sno`。
    - **最上层 `Π`**：投影操作，只保留 `Sname, Grade` 列。

> **注意**：这棵初始查询树是**朴素翻译**的结果，执行效率极低（先做笛卡尔积生成巨大中间表再过滤），这正是后续优化存在的理由。

### 第二页：查询优化概述（动机篇）

**标题**：查询优化概述  
**核心问题**：为什么要优化？

#### 1. 多个逻辑等价的执行计划
- **知识点**：同一条 SQL 可以有多种代数等价变换。
- **示例**：上面的查询可以先做笛卡尔积再过滤，也可以先对 `SC` 表过滤 `Cno='1'` 再连接，结果相同但效率天壤之别。

#### 2. 同一逻辑算子的多种物理实现
- **知识点**：即便逻辑计划确定了（如“做连接”），具体怎么实现也有不同算法。
- **示例**：连接算子 `⋈` 可以用 **Nested-Loop Join**、**Hash Join**、**Sort-Merge Join** 实现。在数据量小的时候 Nested-Loop 够用，在数据量大且无索引时 Hash Join 更优。

#### 3. 查询优化的定位
- **知识点**：优化发生在**解析之后、执行之前**。
- **PPT 关键句**：*“为了保证查询能以较低的资源消耗执行，数据库在查询解析和查询执行这两个过程之间存在查询优化的过程。”*
- **流程图对应**：之前第二页总流程图中，`relational-algebra expression` → `optimizer` → `execution plan`。

### 第三页：查询优化概述（框架篇）

**标题**：查询优化概述  
**核心流程**：逻辑优化 → 代价估计 → 物理优化

#### 1. 输入与输出
- **输入**：查询树（由解析器生成的初始关系代数表达式）。
- **输出**：物理执行计划（包含每个算子具体算法的可执行指令）。

#### 2. 三个核心子过程

| 阶段         | 目标                               | 操作对象                           |
| :----------- | :--------------------------------- | :--------------------------------- |
| **逻辑优化** | 生成更高效的**逻辑等价**查询树     | 关系代数表达式                     |
| **代价估计** | 为物理优化提供统计依据             | 数据的元信息（统计直方图、行数等） |
| **物理优化** | 为每个逻辑算子选择**最优物理算法** | 逻辑算子到物理算子的映射           |

- **逻辑优化**是纯代数层面的改写，不涉及具体数据量。
- **物理优化**必须依赖**代价估计**，因为选择哈希连接还是嵌套循环连接，取决于表的大小和内存大小。

### 第四页：逻辑优化

**标题**：逻辑优化  
**核心思想**：对查询树进行**逻辑等价变换**，以减少中间结果的数据量。

#### 1. 定义与目标
- **输入/输出**：都是查询树，只是结构变了。
- **目标**：找出**执行效率更高**的关系代数表达式。

#### 2. 实现方法：查询重写
- **知识点**：基于**关系代数等价变换规则**对树进行变形。
- **经典规则**：
    1.  **选择下推**：将 `σ` 条件尽可能向叶子节点移动。
    2.  **投影下推**：尽早裁掉不需要的列。
    3.  **连接消除**：去掉无用的连接。
    4.  **子查询展开**：将嵌套子查询改写为连接。

#### 3. PPT 中的示例逻辑优化

PPT 底部展示了三棵查询树的演变（虽然图中第三棵树文字有重复，但意图清晰）：

- **初始树**：
    - `Π` → `σ` → `×` → (`SC`, `Student`)
    - 先做笛卡尔积，再过滤，最后投影。
- **第一步优化（选择下推）**：
    - 将条件 `Cno='1'` **下推**到 `SC` 表上。
    - 将条件 `SC.Sno=Student.Sno` 保留（或转化为连接条件）。
    - 这样在 `SC` 表上先过滤掉大量不满足 `Cno='1'` 的行，笛卡尔积退化为**有效连接**，中间结果急剧缩小。
- **第二步优化（转化为连接）**：
    - 将 `×` 与 `σ` 合并，明确表示为 **Join 算子** `⋈`（条件为 `SC.Sno=Student.Sno`）。

**优化后的查询树逻辑结构**（隐含在 PPT 图示意图中）：
```
Π (Sname, Grade)
    |
    ⋈ (SC.Sno = Student.Sno)
   /   \
 σ(Cno='1')  Student
   |
  SC
```
这种结构下，数据库先扫描 `SC` 表找出课程号为 '1' 的记录（可能只有几十行），再与 `Student` 表做连接，效率远高于全表笛卡尔积。
