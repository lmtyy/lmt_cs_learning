## 第一页：Hash-Join Algorithm（哈希连接算法的完整步骤）

这一页给出了 Hash-Join 的完整执行流程，分为两大阶段：分区阶段（步骤1、2）和探测阶段（步骤3）。

### 步骤 1：对关系 r 进行分区

> "Partition the relation r using hashing function h. When partitioning a relation, one page of memory is reserved as the output buffer for each partition."

用哈希函数 *h* 对关系 *r* 的所有元组进行分区。具体过程：

1. 从磁盘顺序读入关系 *r* 的每一页（page/block）。
2. 对于读入的每个元组 tᵣ，计算 h(tᵣ[JoinAttrs])，得到一个桶编号 i（范围 0 到 n-1）。
3. 把这个元组写入编号为 i 的输出缓冲区。

关于内存缓冲区的管理，这里有一个非常重要的细节：

在分区过程中，内存需要为每个分区保留一页（one page）作为输出缓冲区（output buffer）。也就是说：
- 如果你要分成 n 个桶，就需要 n 个输出缓冲页。
- 再加上 1 个输入缓冲页（用来读入 r 的数据），总共需要 n + 1 页内存。
- 所以如果内存有 M 页，那最多能分成 n = M - 1 个桶。

为什么需要输出缓冲区？因为磁盘 I/O 是以页为单位的，不能每来一个元组就写一次磁盘。所以每个分区在内存中维护一个缓冲页，元组先写入缓冲页，缓冲页满了再一次性刷到磁盘上对应的分区文件中。这样可以大幅减少磁盘 I/O 次数。

### 步骤 2：对关系 s 进行同样的分区

> "Partition s similarly."

用完全相同的哈希函数 *h*，对关系 *s* 做同样的分区操作。这一步和步骤 1 完全对称：
- 顺序读入 *s* 的每一页
- 对每个元组计算哈希值
- 写入对应编号的输出缓冲区

分区完成后，磁盘上就有了 n 对分区文件：(r₀, s₀), (r₁, s₁), ..., (r_{n-1}, s_{n-1})。

这里必须强调：r 和 s 必须使用同一个哈希函数 h，否则相同的连接属性值可能被分到不同编号的桶里，导致结果错误。

### 步骤 3：逐对分区进行探测匹配

> "For each i: Load bucket rᵢ into memory and build an in-memory hash index on it using the join attribute. This hash index uses a different hash function than the earlier one h."

对于每一对 (rᵢ, sᵢ)，执行以下操作：

第一步——构建阶段（Build）：
- 把分区 rᵢ 的所有元组读入内存
- 在内存中对这些元组建立一个哈希索引（hash index），索引的键是连接属性 JoinAttrs
- 这里用的哈希函数是 h₂，和分区阶段用的 h 不同

为什么要用不同的哈希函数？因为分区阶段的 h 已经把"哈希值相同"的元组放到了同一个桶里，如果探测阶段还用 h，那桶内所有元组的 h 值都一样，哈希索引就退化成了一个链表，完全失去了加速查找的作用。用一个不同的 h₂ 可以在桶内进一步区分元组，实现 O(1) 的查找效率。

关系 *r* 被称为 build input（构造用输入），因为它是被用来"构建"内存哈希表的那个关系。

第二步——探测阶段（Probe）：
- 从磁盘逐页读入分区 sᵢ 的元组
- 对于 sᵢ 中的每个元组 tₛ，用 h₂(tₛ[JoinAttrs]) 在内存哈希索引中查找
- 如果找到了哈希值匹配的 rᵢ 中的元组 tᵣ，再精确比较 tᵣ[JoinAttrs] 是否真的等于 tₛ[JoinAttrs]（因为 h₂ 也可能有冲突）
- 如果确实相等，就输出连接结果 tᵣ ⋈ tₛ

关系 *s* 被称为 probe input（探查用输入），因为它是被用来"探查"内存哈希表的那个关系。

### 为什么选 r 做 build input？

通常选择较小的关系作为 build input，因为 build input 的每个分区需要完整放入内存。较小的关系分区后每个桶更小，更容易放进内存。如果 r 比 s 小，就选 r 做 build input；反之选 s。

### 两个哈希函数的总结

| 哈希函数     | 用途           | 阶段     | 输入           | 输出范围                   |
| ------------ | -------------- | -------- | -------------- | -------------------------- |
| h（第一个）  | 分区           | 分区阶段 | JoinAttrs 的值 | {0, 1, ..., n-1}（桶编号） |
| h₂（第二个） | 内存中快速查找 | 探测阶段 | JoinAttrs 的值 | 内存哈希表的槽位           |

## 第二页：Hash-Join 的具体计算例子

这一页用 customer 和 depositor 两个关系的连接来演示完整的代价计算。

### 给定条件

- 内存大小 M = 25 页
- b_depositor = 100（depositor 关系占 100 个磁盘页）
- b_customer = 400（customer 关系占 400 个磁盘页）

### 角色分配

- depositor 是 build input（构造用输入）—— 因为它更小（100 页 < 400 页）
- customer 是 probe input（探查用输入）

为什么选 depositor 做 build input？因为 build input 的分区需要能放进内存。depositor 更小，分区后每个桶也更小，更容易满足内存约束。

### 分区数量的确定：为什么是 5 个桶？

PPT 说把 depositor 分成 5 个桶，每个桶 20 页。我们来验证这个选择是否合理：

内存有 25 页。在探测阶段，需要把 build input 的一个分区完整读入内存并建哈希索引。如果分成 5 个桶，每个桶 100/5 = 20 页。20 页 < 25 页，可以放进内存，还剩 5 页可以用作输入缓冲区读取 probe input。完全可行。

如果分成 4 个桶呢？每个桶 100/4 = 25 页，刚好占满内存，没有空间留给输入缓冲区和哈希索引的额外开销了。所以 5 个桶是一个合理的选择。

一般的经验公式：n ≈ ⌈b_s / M⌉，其中 b_s 是 build input 的页数。这里 ⌈100/25⌉ = 4，但考虑到哈希索引本身也需要一些额外空间，以及需要留缓冲页，所以取 n = 5 更安全。实际中还会乘一个 fudge factor（通常 1.2 左右）。

### 分区阶段的详细过程

对 depositor 分区：
- 内存分配：1 页输入缓冲 + 5 页输出缓冲（每个桶一页）= 6 页，远小于 25 页，绰绰有余
- 顺序读入 depositor 的 100 页，每个元组计算哈希值后写入对应桶的输出缓冲
- 缓冲满了就刷到磁盘
- PPT 说 "This partitioning can be done in one pass"，意思是只需要扫描一遍 depositor 就能完成分区
- I/O 代价：读 100 页 + 写 100 页 = 200 次页传输

对 customer 分区：
- 同样的过程，分成 5 个桶，每个桶 400/5 = 80 页
- 同样 one pass 完成
- I/O 代价：读 400 页 + 写 400 页 = 800 次页传输

分区阶段总 I/O：200 + 800 = 1000 次页传输 = 2 × (100 + 400) = 2 × (b_r + b_s)

### 探测阶段的详细过程

对于每一对 (rᵢ, sᵢ)，即 (depositor_i, customer_i)：

1. 把 depositor_i（20 页）读入内存，建立内存哈希索引
2. 逐页读入 customer_i（80 页），对每个元组在哈希索引中查找匹配
3. 输出匹配结果

每一对的 I/O：读 20 页（build）+ 读 80 页（probe）= 100 页
5 对总共：5 × 100 = 500 次页传输 = b_r + b_s = 100 + 400

探测阶段总 I/O：500 次页传输

### 总代价计算

总 I/O = 分区阶段 + 探测阶段 = 2(b_r + b_s) + (b_r + b_s) = 3(b_r + b_s)

代入数值：3 × (100 + 400) = 3 × 500 = 1500 次页传输

PPT 还提到 "ignores cost of writing partially filled pages"，这是什么意思？

在分区阶段，每个桶的最后一页可能没有写满就要刷到磁盘（因为没有更多元组了）。这些"半满页"的写入也算 I/O，但在理论分析中通常忽略不计，因为最多多出 n 次额外写入（每个桶最多一个半满页），相对于总 I/O 来说很小。

### 和其他算法的对比

如果用 Simple Nested Loop Join：代价约为 100 + 100 × 400 × (每页元组数) ，远远大于 1500。
如果用 Block Nested Loop Join（假设内存 25 页）：代价约为 100 + ⌈100/24⌉ × 400 = 100 + 5 × 400 = 2100，也比 1500 大。

所以 Hash-Join 在这个场景下效率很高。

## 第三页：Conclusion —— 各种 Join 算法的 IO 代价对比

这一页是一个总结表格，把所有学过的 Join 算法的 IO 代价放在一起对比。这里的 M 和 N 分别代表两个关系的磁盘页数（即 b_r 和 b_s），m 代表外层关系的元组数，C 代表索引查找的代价。

### 1. Simple Nested Loop Join（简单嵌套循环连接）

IO Cost = M + (m · N)

- M：读外层关系一次
- m · N：外层关系的每个元组都要扫描一遍内层关系
- m 是外层关系的元组数（不是页数），所以代价非常大
- 这是最朴素的算法，两层 for 循环，逐元组比较

举例：如果外层有 5000 个元组，内层 400 页，那就是 100 + 5000 × 400 = 2,000,100 次页传输。非常恐怖。

### 2. Block Nested Loop Join（块嵌套循环连接）

IO Cost = M + (M · N)

- 改进点：不再逐元组扫描内层，而是逐块（block）扫描
- 外层关系每读入一块（或者尽可能多的块到内存），就扫描一遍内层关系
- 如果内存有 Mem 页，外层关系占 M 页，那外层需要 ⌈M/(Mem-2)⌉ 轮，每轮扫描内层 N 页
- 简化写法就是 M + M · N（这里的 M 是页数级别的乘法，比 m · N 小很多）

注意：这里表格中的 M · N 是简化表示。更精确的公式是 M + ⌈M/(Mem-2)⌉ × N，其中 Mem 是可用内存页数。

### 3. Index Nested Loop Join（索引嵌套循环连接）

IO Cost = M + (M · C)

- 前提：内层关系的连接属性上有索引（B+树索引或哈希索引）
- 外层关系的每个元组，不再全表扫描内层，而是通过索引查找匹配的元组
- C 是单次索引查找的代价，通常很小：
  - B+树索引：C ≈ 树的高度（通常 2~4）+ 匹配的叶子页数
  - 哈希索引：C ≈ 1~2（一次哈希查找）
- 所以总代价从 M · N 降到了 M · C，当 C 远小于 N 时效率提升巨大

举例：如果 M = 100，C = 3（B+树高度），那代价 = 100 + 100 × 元组数 × 3，远小于全表扫描。

### 4. Sort-Merge Join（排序归并连接）

IO Cost = M + N + (sort cost)

- 思路：先把两个关系按连接属性排序，然后用归并的方式扫描匹配
- M + N：排序完成后，两个关系各扫描一遍做归并匹配
- sort cost：排序的代价，取决于外部排序的趟数
  - 外部排序的代价通常是 2 × 页数 × ⌈log_{Mem-1}(⌈页数/Mem⌉)⌉
  - 如果数据已经有序或者有索引，sort cost 可以为 0
- 优点：如果数据本身就有序（比如有聚簇索引），Sort-Merge Join 非常高效
- 缺点：如果数据无序，排序本身的代价不小

### 5. Hash Join（哈希连接）

IO Cost = 3(M + N)

- 这就是前两页详细讲的算法
- 3 的来源：
  - 第 1 次：分区阶段读入两个关系 → M + N
  - 第 2 次：分区阶段写出所有分区 → M + N
  - 第 3 次：探测阶段读入所有分区 → M + N
  - 总计：3(M + N)
- 注意：这个公式假设分区足够均匀，且每个 build input 分区能放进内存
- 如果某个分区太大放不进内存（称为 partition overflow），需要递归分区（recursive partitioning），代价会增加

### 各算法的横向对比分析

| 算法               | 代价量级                | 适用条件      | 特点             |
| ------------------ | ----------------------- | ------------- | ---------------- |
| Simple Nested Loop | O(m·N) ≈ O(元组数×页数) | 任何连接条件  | 最慢，但最通用   |
| Block Nested Loop  | O(M·N) ≈ O(页数×页数)   | 任何连接条件  | 比 Simple 好很多 |
| Index Nested Loop  | O(M·C)                  | 内层有索引    | 索引好时很快     |
| Sort-Merge         | O(M+N+排序)             | 等值/范围连接 | 数据有序时极快   |
| Hash Join          | O(M+N) 即 3(M+N)        | 仅等值连接    | 等值连接通常最快 |

从代价量级看：
- Simple Nested Loop 是乘法级别（元组数 × 页数），最差
- Block Nested Loop 是乘法级别（页数 × 页数），好一些
- Index Nested Loop 取决于索引效率
- Sort-Merge 和 Hash Join 都是加法级别（M + N），是最优的量级

Hash Join 的 3(M+N) 是线性的，而 Nested Loop 类的算法是二次方的。当数据量大时，Hash Join 的优势非常明显。

### 数据库优化器如何选择？

实际的数据库查询优化器会根据以下因素选择算法：
- 关系的大小（页数、元组数）
- 可用内存大小
- 是否有索引
- 数据是否已排序
- 连接条件的类型（等值 vs 范围）
- 数据的分布（是否倾斜）

一般规律：
- 小表连大表且有索引 → Index Nested Loop
- 数据已排序或需要排序输出 → Sort-Merge Join
- 等值连接且数据量大 → Hash Join
- 什么条件都没有 → Block Nested Loop（兜底方案）
