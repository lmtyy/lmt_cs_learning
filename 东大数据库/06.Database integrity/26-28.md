## 第一页：Triggers – 触发器的定义与设计要素

### 1. 触发器的定义
> **A trigger is a statement that is executed automatically by the system as a side effect of a modification to the database.**

- **自动执行**：触发器不需要用户或应用程序显式调用，而是由数据库系统在特定事件发生时自动触发。
- **作为副作用**：触发器的执行是对数据修改操作（如插入、更新、删除）的附带反应，通常用于确保数据一致性或执行附加业务逻辑。

### 2. 设计触发器必须明确的两个要素
> To design a trigger mechanism, we must:
> - Specify the conditions under which the trigger is to be executed.
> - Specify the actions to be taken when the trigger executes.

- **条件（触发条件）**：指明触发器在什么情况下激活。通常包括：
  - **触发事件**：`INSERT`、`UPDATE` 或 `DELETE`。
  - **触发时机**：事件之前（`BEFORE`）或之后（`AFTER`）。
  - **触发粒度**：行级（`FOR EACH ROW`）或语句级（`FOR EACH STATEMENT`），行级触发器对受影响的每一行执行一次，语句级对整个操作执行一次。
  - **可选的附加条件**：例如只有当某列的值满足一定条件时才触发（如 `WHEN` 子句）。

- **动作（触发动作）**：指定触发器被激活后要执行的SQL语句块。可以是简单的单条语句，也可以是复杂的多语句块，甚至包含过程控制逻辑（如条件判断、循环）。在行级触发器中，可以通过 `OLD` 和 `NEW` 引用修改前后的行数据。

**示例**：假设有一个订单表 `orders`，当更新订单状态为“已发货”时，自动记录发货时间到日志表。触发条件为 `AFTER UPDATE ON orders`，动作是插入一条日志记录。

## 第二页：Triggers – SQL标准与数据库实现

### 1. 触发器进入SQL标准的时间
> Triggers introduced to SQL standard in SQL:1999, but supported even earlier using non-standard syntax by most databases.

- **SQL:1999**（也称为SQL3）首次将触发器纳入国际标准。这意味着从1999年起，有了一个统一的规范指导各数据库实现触发器。
- 然而，早在SQL:1999之前，许多主流数据库（如Oracle、Sybase、DB2等）就已经以**非标准语法**提供了触发器功能，以满足用户对自动数据处理的需求。这些早期实现语法各异，但核心思想一致。

### 2. 语法差异与注意事项
> Syntax illustrated here may not work exactly on your database system; check the system manuals.

- 由于各数据库厂商在标准出台前已有自己的实现，且标准本身也提供了多种可选特性，导致不同数据库的触发器语法存在差异。例如：
  - MySQL 使用 `BEFORE/AFTER`、`FOR EACH ROW`，触发器体内用 `BEGIN...END`，且允许使用 `OLD` 和 `NEW`。
  - Oracle 的触发器语法更丰富，支持 `INSTEAD OF` 触发器、语句级触发器，以及复杂的PL/SQL块。
  - SQL Server 使用 `AFTER` 和 `INSTEAD OF`，并可访问 `inserted` 和 `deleted` 虚拟表。
- **重要提示**：在学习或实际开发中，一定要查阅所用数据库系统的官方手册，确认具体的语法规则和限制。PPT中给出的示例可能只是通用概念，直接复制可能无法运行。

## 第三页：触发器（中文概述）

### 1. 触发器的本质
> 触发器是一种在基本表被修改时自动执行的内嵌过程，它主要是通过事件进行触发而被执行。

- **内嵌过程**：触发器是存储在数据库中的程序单元，类似于存储过程，但不由用户主动调用。
- **事件驱动**：触发器的执行依赖于特定事件的发生。这里的“事件”专指对基本表的数据修改操作（`INSERT`、`DELETE`、`UPDATE`）。

### 2. 触发器的执行方式
> 它的执行不是由程序调用，也不是手工启动，而是由事件来触发，比如当对一个表进行操作（insert, delete, update）时就会激活它执行。

- 强调触发器的**被动性**：它永远在幕后等待，一旦关联表发生指定的数据变更，数据库系统会自动唤醒并执行它。
- 这种机制使得触发器非常适合实现**自动化的数据维护**，而无需在应用程序中反复编写相同的逻辑。

### 3. 触发器的典型用途
> 触发器经常用于加强数据的完整性约束和业务规则等。

- **加强数据完整性**：可以补充表约束无法实现的动态完整性规则。例如：
  - 确保库存数量不低于0（在 `UPDATE` 前检查）。
  - 级联删除或更新相关表中的记录（如删除部门时自动将员工部门置空）。
  - 防止非工作时间修改数据。
- **实现复杂业务规则**：例如银行系统中，每次取款后自动检查透支限额并记录日志；订单系统中，当订单状态变化时自动更新库存和财务流水。

**结合前两页**：这一页用更通俗的语言重申了触发器的定义（第一页）和用途（补充完整性约束），并强调了它的事件驱动特性，为后续学习如何创建触发器打下基础。
