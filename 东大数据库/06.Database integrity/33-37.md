## 第一页：BEFORE 触发器示例——转换空成绩为 NULL

### 代码回顾
```sql
create trigger setnull_trigger 
before update of sc 
referencing new row as nrow 
for each row 
when (nrow.grade = ' ') 
begin atomic 
    set nrow.grade = null; 
end;
```

### 知识点详解

#### 1. 触发时机：BEFORE UPDATE
- 在 `UPDATE` 操作**之前**触发，此时数据尚未写入表中。
- 允许在数据最终存储前对其进行**修改或验证**，甚至可以抛出异常阻止操作。

#### 2. 触发事件：UPDATE OF sc
- 这里存在语法不严谨之处，标准写法应为 `BEFORE UPDATE ON sc` 或 `BEFORE UPDATE OF grade ON sc`。该触发器意图在更新 `sc` 表时激活，但未限定特定列。实际应用中，若仅针对 `grade` 列的更新，应使用 `OF grade` 来减少不必要的触发。

#### 3. 引用新行：REFERENCING NEW ROW AS nrow
- 为即将更新的行定义一个别名 `nrow`，在触发器体内通过 `nrow.grade` 访问该行 `grade` 列的新值。
- 在 `BEFORE` 触发器中，可以修改 `nrow` 的字段值，修改后的值将最终写入数据库。

#### 4. 行级触发器：FOR EACH ROW
- 表示对受 `UPDATE` 影响的**每一行**都执行一次触发器动作。如果一条 `UPDATE` 语句更新了多行，触发器会逐行调用。

#### 5. 触发条件：WHEN (nrow.grade = ' ')
- 仅当 `grade` 的新值为一个空格（可能表示空字符串或空白）时，才执行动作。
- 条件中使用的是即将更新的新值，不是旧值。

#### 6. 原子操作块：BEGIN ATOMIC ... END
- `BEGIN ATOMIC` 表示块内的语句作为一个原子操作执行。此处只有一条 `SET` 语句，但即使单条语句，也使用了块包裹（某些数据库要求必须用块，如 Oracle）。
- 动作：将 `nrow.grade` 设置为 `NULL`，从而将原本要存入的空字符串转换为 `NULL`。

#### 7. 业务意义
- 实现数据清洗：防止用户误输入空白字符串，保证成绩字段存储的都是有效数值或 `NULL`，避免后续查询出现空字符串导致的逻辑错误。

## 第三页：AFTER UPDATE 触发器示例——记录成绩大幅提升

### 代码回顾
```sql
CREATE TRIGGER SC_T
AFTER UPDATE OF Grade ON SC
REFERENCING
    OLD ROW AS OldTuple,
    NEW ROW AS NewTuple
FOR EACH ROW
WHEN (NewTuple.Grade >= 1.1 * OldTuple.Grade)
INSERT INTO SC_u VALUES (OldTuple.sno, OldTuple.cno, OldTuple.grade, NewTuple.grade);
```

### 知识点详解

#### 1. 触发时机：AFTER UPDATE
- 在 `UPDATE` 操作**之后**触发，此时数据已经更新到表中。
- 适用于记录日志、发送通知、更新汇总表等不需要干涉原始操作的任务。

#### 2. 限定列更新：UPDATE OF Grade ON SC
- 指定只有当 `Grade` 列被更新时触发器才激活。如果更新语句只修改了其他列（如 `sno`），触发器不会执行，提高了效率。

#### 3. 引用新旧行：REFERENCING OLD ROW AS OldTuple, NEW ROW AS NewTuple
- `OldTuple` 引用更新前的行，`NewTuple` 引用更新后的行。
- 通过这两个别名可以同时访问旧值和新值，用于比较或记录。

#### 4. 行级触发器：FOR EACH ROW
- 每行满足条件的更新都会触发一次，可能多次执行 `INSERT`。

#### 5. 触发条件：WHEN (NewTuple.Grade >= 1.1 * OldTuple.Grade)
- 仅当新成绩达到旧成绩的1.1倍（即提升幅度≥10%）时才执行动作。
- 条件中使用算术表达式，注意乘法优先级。

#### 6. 动作：INSERT INTO SC_u ...
- 向审计表 `SC_u` 插入一条记录，包含学号、课程号、旧成绩、新成绩。
- 由于是 `AFTER` 触发器，插入操作与原始更新在同一个事务中，若插入失败，整个事务回滚。

#### 7. 业务意义
- 自动监控成绩异常提升，便于后续教学检查或防止舞弊。例如，成绩从50分提升到90分（提升80%）会被记录，而从80到82则不会记录。

## 第四页：语句级触发器概念介绍

### 原文内容
> Instead of executing a separate action for each affected row, a single action can be executed for all rows affected by a transaction  
> - Use **for each statement** instead of **for each row**  
> - Use **referencing old table** or **referencing new table** to refer to temporary tables (called **transition tables**) containing the affected rows  
> - Can be more efficient when dealing with SQL statements that update a large number of rows

### 知识点详解

#### 1. 语句级触发器的核心思想
- **行级触发器**：对受影响的每一行执行一次动作。如果一条SQL语句更新了1000行，触发器就会执行1000次。
- **语句级触发器**：无论影响多少行，触发器动作只执行**一次**。这可以显著降低开销，尤其在批量操作时。

#### 2. 语法标志：FOR EACH STATEMENT
- 在创建触发器时使用 `FOR EACH STATEMENT` 替代 `FOR EACH ROW`，声明为语句级触发器。

#### 3. 过渡表（Transition Tables）
- 由于语句级触发器没有“当前行”的概念，无法直接使用 `OLD ROW` 和 `NEW ROW`。
- 取而代之的是**过渡表**（临时表），它们包含了所有受影响的行的集合：
  - `REFERENCING OLD TABLE AS old_table`：所有被删除或被更新前的行组成的临时表。
  - `REFERENCING NEW TABLE AS new_table`：所有被插入或被更新后的行组成的临时表。
- 这些过渡表只在触发器执行期间存在，可以像普通表一样查询（如 `SELECT COUNT(*) FROM old_table`）。

#### 4. 效率优势
- 当处理大量数据时，语句级触发器可以避免重复执行相同逻辑，尤其适合需要基于整体结果做聚合操作的场景（如统计影响行数、计算总和等）。

#### 5. 注意事项
- 不是所有数据库都支持语句级触发器。例如，MySQL只支持行级触发器，而Oracle、PostgreSQL、SQL Server等支持。
- 过渡表中的数据是只读的，不能修改。

## 第五页：语句级触发器示例——统计每次插入的学生人数

### 代码回顾
```sql
CREATE TRIGGER Student_Count
AFTER INSERT ON Student
REFERENCING NEW TABLE AS delta
FOR EACH STATEMENT
INSERT INTO StudentInsertLog(Numbers)
SELECT COUNT(*) FROM delta;
```

### 知识点详解

#### 1. 触发目标
- 每次向 `Student` 表插入数据后，将**本次插入的总行数**记录到日志表 `StudentInsertLog` 中。

#### 2. 触发时机与事件
- `AFTER INSERT ON Student`：在插入操作完成后触发，确保可以获取所有新插入的行。

#### 3. 语句级声明
- `FOR EACH STATEMENT`：无论插入一行还是多行，触发器只执行一次。

#### 4. 使用过渡表
- `REFERENCING NEW TABLE AS delta`：将新插入的所有行作为一个临时表，命名为 `delta`。
- 在 `INSERT` 语句中，通过 `SELECT COUNT(*) FROM delta` 计算本次插入的总行数。

#### 5. 动作
- `INSERT INTO StudentInsertLog(Numbers) SELECT COUNT(*) FROM delta;`
- 向日志表插入一条记录，`Numbers` 列存储本次插入的行数。

#### 6. 与行级触发器的对比
- 如果使用行级触发器实现相同需求，可能会在每插入一行时插入一条 `Numbers=1` 的记录，导致大量冗余数据，且无法直接统计批量插入的总数。
- 语句级触发器只需一条日志记录，即可反映一次插入操作的整体情况，非常适合监控批量数据加载。

#### 7. 业务意义
- 可以用于监控数据增长速度，例如每小时统计新增学生数，或者作为数据导入的审计日志。
