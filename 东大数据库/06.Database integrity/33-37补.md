### 关于 `BEGIN...END` 是否可省略的总结笔记

#### 一、通用原则
- **单条语句**：在标准 SQL 中，如果触发器、存储过程或函数的主体**只有一条 SQL 语句**，可以省略 `BEGIN...END`，直接书写该语句。
- **多条语句**：只要包含两条或以上语句（包括变量声明、流程控制等），**必须**使用 `BEGIN...END` 包裹成一个复合语句块。

#### 二、主流数据库的具体实现

| 数据库         | 触发器 (Trigger)                                             | 存储过程/函数 (Procedure/Function)                           |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **MySQL**      | 单条语句可省略 `BEGIN...END`；多条语句必须用 `BEGIN...END`，且需配合 `DELIMITER` 修改结束符。 | 同触发器：单条语句可省略，多条必须用 `BEGIN...END`。若需声明局部变量，必须使用 `BEGIN...END` 块。 |
| **Oracle**     | 无论语句条数，**必须**使用 `BEGIN...END` 包裹（PL/SQL 块强制）。 | 同触发器：任何情况下都必须用 `BEGIN...END`。                 |
| **PostgreSQL** | 若使用 `LANGUAGE plpgsql`，必须用 `BEGIN...END`；若使用 `LANGUAGE sql`，单条语句可直接写。 | 与触发器类似：SQL 语言函数单条可省略，PL/pgSQL 必须用块。    |
| **SQL Server** | 单条语句可省略 `BEGIN...END`（但常建议加上）；多条必须用块。 | 同触发器：单条可省略，多条必须用 `BEGIN...END`。             |

#### 三、特殊注意事项
- **变量声明**：任何需要在过程中声明的局部变量（如 `DECLARE`），必须位于 `BEGIN...END` 块的开头，因此即使只有一条语句，如果有变量声明，也**必须写块**。
- **流程控制**：`IF`、`LOOP`、`WHILE` 等控制结构必须出现在 `BEGIN...END` 块内。
- **可读性与维护性**：即使当前只有一条语句，提前加上 `BEGIN...END` 可以为未来扩展留下空间，且使代码结构更清晰。

#### 四、建议
- 开发前**务必查阅所用数据库的官方文档**，确认语法规则。
- 在不确定或跨数据库迁移时，**统一使用 `BEGIN...END` 包裹**主体，以确保兼容性和可扩展性。
