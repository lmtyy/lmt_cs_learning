## 第一页：触发器的创建语法（CREATE TRIGGER）

这一页展示了SQL标准中创建触发器的通用语法框架（虽然不同数据库有差异，但核心要素一致）。我们逐段解析：

```sql
CREATE TRIGGER trigger_name 
BEFORE | AFTER 
{INSERT | DELETE | UPDATE [OF colname {, colname ...}]}
ON tablename
[REFERENCING NEW | OLD ROW AS <correlation_name>]
[FOR EACH ROW | FOR EACH STATEMENT]
[WHEN (search_condition)]
{statement | BEGIN ATOMIC statement; {statement; ...} END}
```

### 1. 触发器名称
- `trigger_name`：给触发器起的名字，用于标识。

### 2. 触发时机（BEFORE / AFTER）
- `BEFORE`：在触发事件（INSERT/UPDATE/DELETE）**之前**执行触发器中的动作。
- `AFTER`：在触发事件**之后**执行触发器中的动作。

### 3. 触发事件
- 可以是 `INSERT`、`DELETE` 或 `UPDATE`。
- 对于 `UPDATE`，可以进一步细化到具体的列：`UPDATE OF colname1, colname2, ...`，表示只有当这些列的值被修改时才触发触发器。如果没有指定列，则任何列的更新都会触发。

### 4. 作用表
- `ON tablename`：指定触发器建立在哪个表上，该表的数据变化会激活触发器。

### 5. 引用新旧行（REFERENCING）
- 可选项，用于给被修改的行起一个别名，方便在触发器体内引用。
- `REFERENCING OLD ROW AS old_name`：在 `DELETE` 或 `UPDATE` 中，可以使用该别名访问修改前的行数据。
- `REFERENCING NEW ROW AS new_name`：在 `INSERT` 或 `UPDATE` 中，可以使用该别名访问修改后的行数据。
- 有些数据库（如MySQL）固定使用 `OLD` 和 `NEW` 关键字，无需用 `REFERENCING` 子句；此处的语法更接近标准SQL。

### 6. 触发粒度
- `FOR EACH ROW`：行级触发器，每影响一行就执行一次触发器动作。这是最常用的。
- `FOR EACH STATEMENT`：语句级触发器，整个SQL语句执行后只触发一次，无论影响多少行。某些数据库支持，但MySQL只支持行级。

### 7. 触发条件（WHEN）
- 可选项，`WHEN (search_condition)` 指定一个条件，只有当条件为真时才执行触发器动作。这可以避免不必要的触发。

### 8. 动作体
- 可以是单条语句，也可以是一个语句块（`BEGIN ATOMIC ... END`）。`ATOMIC` 表示块内的语句作为一个原子单元执行（要么全部成功，要么全部回滚）。
- 在动作体中，可以使用前面定义的别名（OLD/NEW）访问数据，也可以包含复杂的SQL逻辑。

## 第二页：触发事件与引用新旧行

这一页补充说明了触发事件的具体细节和如何引用修改前后的行值。

### 1. 触发事件可以是 INSERT、DELETE 或 UPDATE
- 这是触发器的基本分类，每个事件对应不同的数据变化类型。

### 2. 可以为 UPDATE 指定特定列
- 语法：`UPDATE OF 列名1, 列名2, ...`
- 意义：只有当指定的列被更新时，触发器才被激活。例如：`AFTER UPDATE OF grade ON takes` 表示只有当 `grade` 列的值发生变化时，触发器才执行。如果只修改了其他列（如 `semester`），则不会触发。
- 这样可以避免不必要的触发器执行，提高效率。

### 3. 可以引用修改前后的属性值
- 在触发器动作中，经常需要比较修改前和修改后的值，或者用旧值做某些操作。
- **对于 DELETE 和 UPDATE**：可以使用 `OLD` 引用删除前或更新前的行。例如 `OLD.grade` 表示更新前的成绩。
- **对于 INSERT 和 UPDATE**：可以使用 `NEW` 引用插入的新行或更新后的行。例如 `NEW.grade` 表示更新后的成绩。
- 通过引用 OLD 和 NEW，可以实现诸如“禁止降低工资”、“记录变更日志”等业务逻辑。

> 在PPT中，用 `referencing old row as :` 和 `referencing new row as :` 表示需要定义别名，实际使用中在 `REFERENCING` 子句中指定。

## 第三页：AFTER 触发器

这一页解释了 `AFTER` 触发器的行为特点。

### AFTER 触发器的定义
- **在记录操纵之后触发**：即先完成数据的插入、删除或更新操作，然后再执行触发器中的语句。
- 触发的语句晚于监视的增删改操作，因此**无法影响前面的增删改动作**——因为数据修改已经发生，触发器只能“事后处理”，不能撤销或修改已经执行的操作（除非在事务中通过回滚，但那是事务层面，触发器本身不能直接干预）。

### 适用场景
- 用于记录日志、审计、更新统计信息、发送通知等不需要干涉原始操作的任务。
- 例如：当员工表删除记录后，自动将删除的员工信息插入到退休员工表（如PPT之前提到的例子）。因为删除操作已经完成，触发器只需把已删除的数据（通过OLD）归档。

### 注意
- 在 `AFTER` 触发器中，如果触发器内执行了会导致数据变化的操作（例如插入日志），这些操作和原始操作在同一个事务中，如果触发器失败，整个事务会回滚，原始操作也会被撤销。
- 但这并不等同于触发器可以修改原始操作的结果，而是通过事务整体回滚来保持一致性。

## 第四页：BEFORE 触发器

这一页重点介绍了 `BEFORE` 触发器的行为特点，并与 `AFTER` 形成对比。

### BEFORE 触发器的定义
- **在记录操纵之前触发**：先执行触发器中的语句，然后再进行数据的插入、删除或更新操作。
- 触发的语句先于监视的增删改操作，因此**我们有机会判断、修改即将发生的操作**，甚至阻止操作的发生。

### 主要用途
1. **验证和修改数据**：可以在数据写入表之前，检查 `NEW` 值是否满足业务规则。如果不满足，可以修改 `NEW` 的值，或者直接抛出错误阻止操作。
   - 例如：在学生选课成绩表中，如果更新的成绩超过100分，可以自动将其修正为100分（通过 `SET NEW.grade = 100`），或者报错不让更新。
2. **实现复杂的完整性约束**：比如检查外键关系。如果要在删除主表记录前，确保从表没有相关记录，可以在 `BEFORE DELETE` 触发器中检查从表，如果有则先删除或阻止删除。
   - PPT中举例：“在删除主键时，必须要先删除外键表，这时就有先后之分，这里before相当于设置了断点，我们可以处理删除外键。” 即在 `BEFORE DELETE` 触发器中，可以先删除（或检查）外键表中的相关行，然后再删除主表记录，从而避免违反外键约束。

### 特点总结
- `BEFORE` 触发器可以**干预**即将发生的操作：可以修改 `NEW` 的值，也可以抛出异常终止操作。
- 它相当于在数据修改前插入了一道自定义的验证/处理逻辑。

## 总结与对比

| 特性            | BEFORE 触发器                                | AFTER 触发器                       |
| --------------- | -------------------------------------------- | ---------------------------------- |
| 触发时机        | 数据操纵之前                                 | 数据操纵之后                       |
| 能否影响操纵    | 能（修改NEW值、阻止操作）                    | 不能（操纵已发生，只能做后续处理） |
| 典型应用        | 数据验证、补充默认值、级联处理（如先删外键） | 日志记录、审计、归档、更新统计信息 |
| 能否引用OLD/NEW | 可以，NEW可修改                              | 可以，但NEW只读（因为操作已完成）  |

通过这四页PPT，你可以系统掌握触发器的基本语法、触发事件、新旧行引用以及 `BEFORE` 和 `AFTER` 的核心区别。在实际开发中，根据需求选择合适的触发器类型，可以有效实现复杂的业务规则和数据完整性保障。
