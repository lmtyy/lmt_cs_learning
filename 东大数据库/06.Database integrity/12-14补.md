您提出的问题非常关键，涉及SQL中**子查询返回空集**时的行为，以及**三值逻辑**的影响。我们需要区分两种情况：

1. **标量子查询**（返回单个值的子查询）  
2. **作为集合的子查询**（用于 `ALL`、`ANY`、`IN` 等比较）

下面分别解释，并结合PPT中的例子说明。

---

### 1. 标量子查询为空时返回 `NULL`
- 如果一个子查询期望返回**单个值**（例如放在 `SELECT` 列表、`WHERE` 条件中作为表达式），但实际查询结果为空行，则SQL将其结果视为 `NULL`。
- 例如：`SELECT (SELECT age FROM student WHERE sno='不存在') FROM dual;` 返回 `NULL`。
- 当这个 `NULL` 参与比较时（如 `150 >= NULL`），根据三值逻辑，结果为 `UNKNOWN`，导致整个条件不成立（因为 `WHERE` 只接受 `TRUE`）。

**但是**，PPT中的两个例子**都不是标量子查询**，而是作为集合使用的子查询。

---

### 2. 集合子查询为空时的 `ALL`/`ANY` 规则
在第二个例子中：
```sql
150 >= ALL ( SELECT count(*) FROM sc GROUP BY cid )
```
- 子查询返回的是**一个结果集**（可能有多行，也可能为空集），然后使用 `ALL` 与 `150` 比较。
- SQL标准对 `ALL` 的定义是：  
  - 如果子查询结果集为空，则 `ALL` 比较**自动返回 TRUE**。  
  - 原因：逻辑上“所有行都满足条件”对于空集是成立的（空真，vacuous truth）。
- 同样，对于 `ANY`（或 `SOME`），如果子查询结果集为空，则返回 **FALSE**。

因此，当没有选课记录时，`GROUP BY` 返回空集，`150 >= ALL (空集)` 的结果是 `TRUE`，断言成立，符合直觉（因为没有课程违反人数限制）。

---

### 3. 第一个例子中的子查询
第一个例子：
```sql
150 >= ( SELECT count(*) FROM course, sc WHERE ... )
```
这里子查询是**标量子查询**吗？  
- `SELECT count(*)` 在没有匹配行时会返回 `0`，因为 `COUNT` 聚合函数在没有行时返回 `0`，而不是 `NULL`。  
- 注意：`COUNT(*)` 永远不会返回 `NULL`，即使 `WHERE` 条件不匹配任何行，聚合结果也是 `0`。  
- 因此该子查询总是返回一个数值，不会出现 `NULL`。所以比较 `150 >= 0` 总是 `TRUE`，没问题。

如果子查询用的是某个列（比如 `SELECT score FROM ...`），且无匹配行，则返回 `NULL`，这时比较会产生 `UNKNOWN`。但这里用了 `COUNT`，所以避免了 `NULL`。

---

### 4. 为什么第一个例子用了 `COUNT` 而不是直接选列？
- 因为需要统计人数，人数为0是合法情况，用 `COUNT` 自然返回0。
- 若改用 `SELECT cname` 这样的非聚合，无匹配行会返回 `NULL`，导致断言在无此课程时变成 `150 >= NULL` → `UNKNOWN` → 断言失败，这显然不合理。所以设计者明智地用了聚合。

---

### 总结回答您的问题
- **子查询为空时返回什么？**  
  - 如果是非聚合的标量子查询，返回 `NULL`。  
  - 如果是聚合函数（如 `COUNT`、`SUM`），即使无行也返回一个值（`COUNT` 返回0，其他聚合返回 `NULL`，但 `COUNT` 特殊）。  
  - 如果是集合子查询（多行结果），则空集是一个空的结果集，不是 `NULL`。
- **比较时的结果？**  
  - 标量子查询返回 `NULL` 时，比较产生 `UNKNOWN`，导致条件不成立。  
  - 集合子查询用于 `ALL` 时，空集返回 `TRUE`；用于 `ANY` 时返回 `FALSE`。这是SQL标准的逻辑。

在PPT的例子中，两个断言都正确处理了空集情况，没有出现 `NULL` 比较问题。
