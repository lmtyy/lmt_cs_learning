这几页PPT主要介绍数据库中的**断言（Assertion）**机制，它是SQL标准中用于定义复杂完整性约束的一种方式，能够表达涉及多表或聚合操作的全局条件。下面我们逐页详细讲解，并补充易错点与常考点。

---

## 第一页：断言（Assertion）概述

### PPT原文
- 一个断言就是一个谓词表达式，希望数据库总能满足的条件  
  - 表和列约束是一种特殊的断言  
- 创建断言的语法：  
  ```sql
  CREATE ASSERTION <assertion-name> CHECK (predicate)
  ```
- 断言会加重数据库系统负担，小心使用复杂的断言

### 详细讲解

#### 1. 什么是断言？
- **断言**是一个命名的、全局的完整性约束，它定义了一个**谓词**（即返回真值的条件），并要求数据库在任何时候都必须满足该谓词。
- 与列级约束（如 `NOT NULL`）和表级约束（如 `CHECK`、`FOREIGN KEY`）不同，断言可以跨越多个表，并且可以包含聚合函数（如 `COUNT`、`SUM`）和子查询，因此能够表达更复杂的业务规则。

#### 2. 断言与普通约束的关系
- 可以说，列约束和表约束是断言的**特例**，因为它们也只定义了一个必须满足的条件，但作用范围仅限于单列或单表。而断言的作用范围是整个数据库（或指定的表集合）。
- 例如，`CHECK (age >= 0)` 可以看作是作用于`age`列的一个断言片段，但它在语法上是列约束。

#### 3. 断言语法
- `CREATE ASSERTION 断言名 CHECK (谓词);`
- 谓词是一个布尔表达式，可以包含任何合法的SQL条件，包括子查询、`EXISTS`、`IN`、比较运算符、逻辑运算符等。
- 当对数据库进行任何可能改变断言结果的操作（INSERT、UPDATE、DELETE）时，DBMS会自动检查断言。如果谓词为假，操作被拒绝。

#### 4. 断言的代价
- 断言需要**全局检查**：每次数据更新都可能触发断言评估，尤其当断言涉及聚合和多表时，开销可能非常大。
- 因此，PPT警告“小心使用复杂的断言”，以免严重影响数据库性能。在实际系统中，断言往往被触发器或应用程序逻辑替代。

### 易错点与常考点

- **易错点**：
  - 误以为断言可以只在需要时检查，实际上断言是**立即检查**（除非支持延迟检查，但标准默认立即）。
  - 混淆断言与触发器：断言是声明式约束，自动检查；触发器是过程式代码，可以执行更复杂动作，但也能实现断言功能。
  - 忽略断言对性能的影响：在考试中可能要求权衡使用断言还是触发器。
- **常考点**：
  - 断言的定义语法（记住 `CREATE ASSERTION ... CHECK (...)`）。
  - 断言能解决哪类问题？——涉及多表、聚合、全局条件的约束。
  - 举例说明哪些约束必须用断言而不能用列/表约束（如“每个部门的平均工资不超过5000”）。

## 第二页：例子1——限制“数据库”课程的选修人数

### PPT原文
- 限制数据库课程的选修人数不能超过150人

```sql
Create assertion dbcourse_num check  
(150>= (select count(*)  
    from course, sc  
    where course.c#=sc.c# and cname='数据库') )
```

### 详细讲解

#### 1. 业务规则
- 不允许选修“数据库”这门课程的学生人数超过150人。
- 这里假设课程名称 `cname` 是唯一的，或者至少我们限制的是特定名称的课程。但实际中，可能存在同名课程（不同院系），更好的做法是用课程号，但示例中用课程名是为了简化。

#### 2. SQL分析
- 子查询：连接 `course` 和 `sc` 表，筛选出课程名为“数据库”的选课记录，然后统计记录数（即选课人数）。
- 断言条件：150 >= (子查询结果)。如果当前选课人数 ≤150，条件为真；如果 >150，条件为假，则任何导致人数超过150的操作都会被拒绝。
- 注意：当没有“数据库”课程或没有学生选课时，子查询返回0，条件成立。

#### 3. 检查时机
- 任何可能改变选课人数或课程信息的操作都会触发断言：
  - 向 `sc` 表插入一条选课记录（如果该课程是“数据库”）。
  - 更新 `sc` 表的课程号（可能改为或改出“数据库”）。
  - 删除 `sc` 表的记录（可能减少人数，不会违反，但断言也会检查，即使不会失败）。
  - 修改 `course` 表的课程名（可能将某课程改名为“数据库”，或者将“数据库”改名，影响连接结果）。
- 因此，断言需要检查很多操作，开销较大。

#### 4. 可能存在的问题
- **课程名重复**：如果存在多门同名“数据库”课程，子查询会统计所有同名课程的选课人数总和，可能不符合“每门课程单独限制”的初衷。
- **硬编码课程名**：约束写死了课程名，不够灵活。如果课程改名，约束可能失效。
- **性能**：每次检查都要做连接和计数，如果 `sc` 表很大，代价高。

### 易错点与常考点

- **易错点**：
  - 忘记考虑子查询结果可能为 `NULL`？这里子查询是标量子查询，`COUNT(*)` 永远不会返回 `NULL`，即使没有匹配行也返回0，所以比较安全。
  - 错误理解断言的作用范围：以为只检查插入 `sc`，其实所有可能影响子查询结果的操作（如修改 `course`）都会触发。
  - 把断言写成 `CHECK` 表约束，但 `CHECK` 不能跨表（除非某些DBMS支持子查询），所以必须用断言。
- **常考点**：
  - 给出业务规则，要求写出断言（如本题）。
  - 分析断言可能带来的性能问题。
  - 修改断言以处理课程号而不是课程名。

## 第三页：例子2——限制每门课程的选修人数

### PPT原文
- 限制每门课程的选修人数不能超过150人

```sql
Create assertion course_num check  
(150>=all(select count(*)  
    from sc  
    group by cid)  
);
```

### 详细讲解

#### 1. 业务规则
- 对**每一门**课程，其选修人数都不能超过150人。

#### 2. SQL分析
- 子查询：`select count(*) from sc group by cid` 按课程号分组，统计每门课的选课人数，返回一个结果集，包含所有课程的选课人数。
- 断言条件：`150 >= ALL (子查询结果集)`。`ALL` 要求150必须大于等于子查询返回的**每一个**值，即所有课程的选课人数都不超过150。
- 如果没有任何选课记录，子查询返回空集。对于空集，`ALL` 条件的结果是**真**（因为“所有值都满足”对空集是自然成立的），所以断言成立。

#### 3. 与第一个例子的区别
- 第一个例子只针对一门特定课程，第二个例子针对所有课程。
- 第二个例子使用 `ALL` 和分组聚合，展示了断言处理多行结果的能力。

#### 4. 注意点
- 该断言假设 `sc` 表中的 `cid` 是外键，且每个 `cid` 对应一门课程。如果存在某些课程没有选课记录，它们不会出现在分组结果中，因此不会受150限制。这符合业务：没学生选课当然人数为0，不违反。
- 断言会在插入、更新、删除 `sc` 记录时触发，重新计算所有课程的人数并比较。如果课程数很多，开销会很大。

### 易错点与常考点

- **易错点**：
  - 忘记使用 `ALL`，而用 `> 150` 会变成要求子查询返回单值，但分组查询返回多行，导致语法错误。
  - 误以为子查询必须返回单列，`ALL` 可以正确处理多行单列的情况。
  - 对空集的处理：`ALL` 在子查询为空时返回真，这符合直观（因为不存在违反的课程），但需注意其他情况如 `SOME` 在空集时返回假。
  - 可能忽略 `cid` 为空值的情况：如果 `sc` 表中允许 `cid` 为空，`group by cid` 会把空值分为一组，统计空值课程的人数，但空值不代表实际课程，可能导致错误限制。因此，应确保 `cid` 非空或使用 `WHERE cid IS NOT NULL`。
- **常考点**：
  - 写出类似的断言（如限制每个学生的选课门数）。
  - 解释 `ALL` 和 `SOME` 在断言中的作用。
  - 分析断言的性能代价，并提出优化或替代方案（如使用触发器，只在相关表变动时检查）。

## 总结：断言的优缺点与适用场景

### 优点
- **声明式**：只需定义条件，DBMS自动维护。
- **表达能力强**：可以处理跨表、聚合、多行比较的复杂规则。
- **集中管理**：所有完整性规则在数据字典中统一存储。

### 缺点
- **性能开销大**：每次更新都可能触发全局检查，尤其复杂断言会严重影响并发性能。
- **实际支持有限**：主流数据库（如MySQL、PostgreSQL、Oracle、SQL Server）对断言的支持程度不同，很多数据库**没有实现**断言，或建议用触发器替代。
- **调试困难**：当断言失败时，很难定位是哪个操作引起的。

### 替代方案
- **触发器**：可以实现同样的逻辑，并且可以更精细地控制检查时机和响应动作，但需要手动编写过程代码。
- **物化视图**：某些数据库允许在物化视图上定义约束，间接实现类似功能。

### 易错点与常考点汇总

1. **断言语法**：`CREATE ASSERTION 名称 CHECK (条件)`，条件可以是任何返回布尔值的SQL表达式。
2. **与表约束的区别**：表约束只能引用本表列，不能包含聚合（除非是单行聚合？实际上标准允许表约束中使用子查询，但支持有限），而断言可以跨表、聚合。
3. **聚合与分组的使用**：注意 `ALL`、`ANY`（`SOME`）与子查询的搭配。
4. **空值处理**：子查询结果为空时，`ALL` 返回真，`ANY` 返回假。需要特别小心。
5. **性能考虑**：复杂断言应避免，考试中可能要求评估。
6. **实际数据库支持**：虽然SQL标准有断言，但多数数据库未实现，考试中通常按标准SQL回答。

通过这几页PPT，我们学习了断言的基本概念、语法和两个典型例子，掌握了如何用断言实现复杂的完整性约束。在实际应用中，需权衡性能与便利性，必要时选择触发器。
