这页PPT主要介绍了数据库完整性的核心概念以及SQL中实现完整性的几种机制。下面我们逐一详细讲解。

---

## 1. 数据库完整性的概念

**数据库完整性**是指数据库中数据的**正确性**、**一致性**和**相容性**。它确保数据库中的数据符合现实世界的语义，避免无效或错误的数据进入数据库。完整性通常通过一系列**完整性约束**来维护，主要分为三类：

- **实体完整性（Entity Integrity）**  
  要求每个表的主键字段值必须唯一且非空（如 `PRIMARY KEY`）。这保证了表中每一行的唯一标识。

- **参照完整性（Referential Integrity）**  
  要求外键字段的值要么为空，要么等于被引用表中某行的主键值（如 `FOREIGN KEY`）。这保证了表之间数据关系的正确性。

- **用户定义完整性（User-defined Integrity）**  
  针对具体业务规则定义的约束，如字段取值范围、格式等（如 `CHECK`、`NOT NULL`、`UNIQUE`）。

数据库管理系统（DBMS）会自动检查这些约束，违反时拒绝操作，从而保护数据的完整性。

---

## 2. SQL-DDL中关于完整性的命令

在SQL的数据定义语言（DDL）中，可以在创建或修改表时定义各种完整性约束。主要命令是 `CREATE TABLE` 和 `ALTER TABLE`。

### 常用约束类型
- **NOT NULL**：字段值不能为空。
- **UNIQUE**：字段值必须唯一（可包含一个空值）。
- **PRIMARY KEY**：主键，相当于 `NOT NULL` + `UNIQUE`，一个表只能有一个主键。
- **FOREIGN KEY**：外键，引用另一表的主键或唯一键。
- **CHECK**：指定字段必须满足的条件（布尔表达式）。
- **DEFAULT**：为字段指定默认值（虽不是完整性约束，但有助于维护数据完整性）。

### 定义方式
可以在列级别直接定义，也可以在表级别单独定义（尤其适合复合约束）。

#### 示例：
```sql
CREATE TABLE student (
    sno    CHAR(10)      PRIMARY KEY,                -- 列级主键
    sname  VARCHAR(20)   NOT NULL,
    age    INT           CHECK (age >= 0 AND age <= 150),
    sex    CHAR(2)       DEFAULT '男',
    deptno CHAR(5),
    CONSTRAINT fk_dept FOREIGN KEY (deptno) REFERENCES department(deptno)
);
```

使用 `ALTER TABLE` 添加或删除约束：
```sql
ALTER TABLE student ADD CONSTRAINT unique_sname UNIQUE(sname);
ALTER TABLE student DROP CONSTRAINT fk_dept;
```

这些命令确保了数据库结构层面的完整性规则。

---

## 3. 断言（Assertion）

**断言**是SQL标准中提供的一种更强大的完整性约束，它可以定义涉及多个表或包含聚合操作的复杂条件。与列级或表级约束不同，断言独立于表，对整个数据库状态进行约束。

### 语法
```sql
CREATE ASSERTION <断言名> CHECK (<条件>);
```
条件可以是任何返回布尔值的SQL表达式，包括子查询、聚合等。如果条件为假，则任何导致该条件为假的数据修改操作都会被拒绝。

### 示例
保证学生的年龄不能大于所在系的成立年限（假设系表有成立年份字段）：
```sql
CREATE ASSERTION age_check
CHECK (NOT EXISTS (
    SELECT * FROM student s JOIN department d ON s.deptno = d.deptno
    WHERE s.age > (YEAR(CURRENT_DATE) - d.est_year)
));
```

**注意**：虽然断言在SQL标准中定义，但大多数主流DBMS（如MySQL、PostgreSQL、Oracle）并未完全实现或推荐使用断言，而是建议用**触发器**或**物化视图**等方式替代。因此实际开发中较少直接使用断言。

---

## 4. 触发器（Trigger）

**触发器**是一种特殊的存储过程，它会在指定的数据库事件（INSERT、UPDATE、DELETE）发生时自动执行。触发器常用于实现复杂的业务规则、数据审计、级联更新等，是维护数据完整性的重要工具。

### 触发器的组成
- **触发事件**：INSERT、UPDATE 或 DELETE（可指定具体列）。
- **触发时机**：BEFORE 或 AFTER 事件发生。
- **触发级别**：行级（FOR EACH ROW）或语句级（FOR EACH STATEMENT）。
- **触发条件**（可选）：WHEN 子句指定触发条件。
- **触发动作**：要执行的SQL语句或过程体。

### 触发器的引用
- **OLD**：引用更新或删除前的旧行（用于UPDATE、DELETE）。
- **NEW**：引用插入或更新后的新行（用于INSERT、UPDATE）。

### 示例
1. **保证学生年龄不能小于0**（BEFORE INSERT 行级触发器）：
```sql
CREATE TRIGGER check_age_before_insert
BEFORE INSERT ON student
FOR EACH ROW
BEGIN
    IF NEW.age < 0 THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Age cannot be negative';
    END IF;
END;
```

2. **记录学生表修改日志**（AFTER UPDATE 行级触发器）：
```sql
CREATE TRIGGER log_student_update
AFTER UPDATE ON student
FOR EACH ROW
BEGIN
    INSERT INTO student_log(sno, old_name, new_name, change_time)
    VALUES (OLD.sno, OLD.sname, NEW.sname, NOW());
END;
```

3. **级联删除**（当删除部门时，自动将该部门学生的部门号置空）：
```sql
CREATE TRIGGER cascade_dept_delete
BEFORE DELETE ON department
FOR EACH ROW
BEGIN
    UPDATE student SET deptno = NULL WHERE deptno = OLD.deptno;
END;
```

### 优缺点
- **优点**：灵活，可实现任意复杂的逻辑；自动化，对应用透明。
- **缺点**：可能降低性能；调试和维护较困难；过度使用可能导致逻辑分散。

不同DBMS的触发器语法略有差异，上述示例以MySQL风格示意。

---

## 总结

数据库完整性是保证数据质量的核心，SQL提供了从简单约束到复杂触发器的多层次机制：
- **DDL约束**（主键、外键、CHECK等）用于声明式的基本完整性规则。
- **断言**（理论上）用于跨表复杂约束，但实际支持有限。
- **触发器**则提供了过程式的灵活性，能够实现动态、复杂的完整性控制。

在设计数据库时，应根据业务需求合理选择完整性实现方式，平衡数据安全与系统性能。
