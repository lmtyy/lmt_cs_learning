## 第一页：索引评估指标（Index Evaluation Metrics）

这一页 PPT 从**工程角度**告诉我们，衡量一个索引好坏的五个核心维度。设计或选择索引时，不能只看查询快不快，要综合权衡。

### 1. 支持的访问类型（Access types supported efficiently）

这里重点区分了两种最基础的查询模式，并指出了有序索引和哈希索引在此处的根本差异。

| 查询类型     | PPT原文解释                                                  | 支持情况                            |
| :----------- | :----------------------------------------------------------- | :---------------------------------- |
| **等值查询** | records with a specified value in the attribute              | **有序索引** 和 **哈希索引** 都支持 |
| **范围查询** | records with an attribute value falling in a specified range | **只有有序索引** 支持               |

- **深入理解等值查询：**
    - SQL 示例：`SELECT * FROM student WHERE id = '2024001';`
    - 有序索引通过**二分查找**找到 `2024001`。
    - 哈希索引通过**哈希函数**直接算出 `2024001` 所在的桶。
    - **结论：** 两者都擅长。

- **深入理解范围查询：**
    - SQL 示例：`SELECT * FROM student WHERE score BETWEEN 80 AND 90;`
    - 有序索引：找到 `80` 的位置后，因为数据是**物理相邻/逻辑相邻**排列的，直接顺着叶子节点向右扫描即可，效率极高。
    - 哈希索引：哈希函数会把 `80`、`81`、`85` 随机打散在不同的桶里，**完全无法利用索引**，只能被迫扫描全表。
    - **结论：** 这是哈希索引的致命短板。

### 2. 访问时间（Access Time）

- **定义：** 使用该索引找到特定数据项所花费的时间。
- **衡量标准：** 通常是磁盘 I/O 次数或 CPU 时间。
- **注意：** 访问时间不仅取决于索引结构的高度（如B+树的层数），也取决于数据是否在缓存中。

### 3. 插入时间（Insertion Time）

- **PPT原文：** 包括找到插入这个新数据项的正确位置所需的时间 **+** 更新索引结构所需要的时间。
- **详细拆解：**
    1.  **定位开销：** 找到新数据应该插在索引的哪个位置（对于B+树是找到叶子节点）。
    2.  **结构调整开销：**
        - 如果是**有序索引**：插入可能导致叶子节点**分裂**，甚至引发父节点向上层分裂，这是比较重的操作。
        - 如果是**哈希索引**：如果桶满了，可能触发**桶扩展**或**溢出链**处理。

### 4. 删除时间（Deletion Time）

- **PPT原文：** 包括找到删除项所需的时间 **+** 更新索引结构所需要的时间。
- **详细拆解：**
    - 同样分为查找开销和结构调整开销。
    - 在B+树中，删除可能导致节点中元素过少，进而发生**节点合并**。

### 5. 空间开销（Space Overhead）

- **定义：** 索引结构本身占用的额外磁盘/内存空间。
- **对比：**
    - **主索引（聚集索引）**：通常空间开销较小，因为叶子节点就是数据本身（或者数据按该顺序物理存放），不需要额外存指针指向别处。
    - **辅助索引（非聚集索引）**：空间开销较大，因为它存了键值，还要存指向主键或物理地址的指针。

## 第二页：有序索引详解（Ordered Indices）

这一页开始具体展开**有序索引**这一大类，并引入了非常容易混淆的概念：**主索引 / 聚集索引**。

### 1. 有序索引的定义

- **PPT原文：** In an **ordered index**, index entries are stored sorted on the search key value.
- **通俗解释：** 索引条目按照搜索键的**字母顺序**或**数字大小**排列。
- **PPT类比：** 图书馆的**作者目录**。所有的索引卡片是按照作者名字的 A-B-C 顺序排列的。

### 2. 核心概念：主索引 / 聚集索引

这是这一页乃至这一章的重难点，PPT 给出了精准的学术定义。

- **PPT原文定义（英文）：**
    > **Primary index** (also called **clustering index**): in a sequentially ordered file, the index whose search key specifies the sequential order of the file.

- **中文翻译与核心要点提炼：**
    1.  **前提条件：** 数据文件本身必须是**按照某个字段顺序排列**的（Sequentially ordered file）。
    2.  **索引作用：** 该索引的搜索键，恰好规定了文件的这种排列顺序。
    3.  **关键误区澄清：** The search key of a primary index is **usually but not necessarily the primary key.**
        - 意思是：这个顺序字段**通常**是主键，但**不强制**是主键。只要文件物理上是按某列顺序存的，这列上的索引就是主索引。

### 3. 概念辨析：主索引 vs. 主键

这是最容易犯错的地方，结合PPT最后一句话来看：

| 概念       | 定义                                                    | 关系                                                         |
| :--------- | :------------------------------------------------------ | :----------------------------------------------------------- |
| **主键**   | 逻辑概念，用来唯一标识一行记录的约束。                  | 主键上通常建有索引（可能是主索引也可能是辅助索引）。         |
| **主索引** | 物理/存储概念，指数据的**物理排列顺序**与索引顺序一致。 | **一个表只能有一个主索引**（因为数据只能有一种物理排序方式）。 |

**举例：** 一个员工表，数据在磁盘上是按 `id` 顺序存的，那么 `id` 上的索引就是主索引。但我们可以把 `工号` 设为主键，它们通常是重叠的。如果某天数据物理上按 `入职日期` 排序了，那 `入职日期` 就成了主索引，即便 `id` 还是主键，`id` 上的索引此时也只能是辅助索引。

## 第三页：聚集索引图示（Clustering Index Illustration）

这一页的图表是为了直观展示第二页提到的“**文件顺序与索引顺序一致**”这一物理特性。

### 图表结构分析

PPT 左边是一个**索引文件**（稀疏索引），右边是一个**数据文件**。

**左边索引文件：**
| Index Block (Search Key) | Pointer |
| :----------------------- | :------ |
| **Brighton**             | →       |
| **Downtown**             | →       |
| **Mianus**               | →       |
| **Perryridge**           | →       |
| **Redwood**              | →       |
| **Round Hill**           | →       |

- 注意：这里的搜索键是 **Branch Name（分行名）** ，不是 ID。
- 这些分行名是按照 **字母顺序排列** 的：B, D, M, P, R, R。

**右边数据文件：**
| Branch Name | Account No. | ...  | Balance |
| :---------- | :---------- | :--- | :------ |
| Brighton    | A-217       | ...  | 750     |
| Downtown    | A-101       | ...  | 500     |
| Downtown    | A-110       | ...  | 600     |
| Mianus      | A-215       | ...  | 700     |
| Perryridge  | A-102       | ...  | 400     |
| Perryridge  | A-201       | ...  | 900     |
| Perryridge  | A-218       | ...  | 700     |
| Redwood     | A-222       | ...  | 700     |
| Round Hill  | A-305       | ...  | 350     |

### 关键观察：索引顺序 = 物理存储顺序

1.  **观察索引顺序：** Brighton → Downtown → Mianus → Perryridge → Redwood → Round Hill。
2.  **观察数据文件顺序：** 数据记录**确确实实是按照上述分行名的顺序连续存放的**。所有 Brighton 的记录在一起，接着是所有 Downtown 的记录，接着是 Mianus...
3.  **结论：** 因为数据的**物理存储顺序**和**索引的键值顺序**完全一致，所以这个建立在 `Branch Name` 字段上的索引，就是一个典型的 **Primary Index / Clustering Index（主索引/聚集索引）**。

### 为什么叫“聚集”？
因为相同键值的记录（比如所有 Perryridge 分行的账户）在磁盘上是**聚集在一起**、物理相邻存储的。当你查询 `WHERE Branch_Name = 'Perryridge'` 时，磁头只需要移动到一个位置，就可以连续读出所有相关数据，**I/O 效率极高**。
