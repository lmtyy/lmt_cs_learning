### 第 1 页：B⁺ 树实例——以 \( n = 5 \) 为例

这一页将前面抽象的扇出与阶的公式代入具体数值，帮助直观理解 B⁺ 树的节点到底能装多少数据。

**1. 给定参数**

- **最大指针数（扇出）\( n = 5 \)**。
- 这意味着每个节点最多拥有 5 个指针，即最多 5 个子节点（对非叶节点）或最多 5 个数据指针（对叶节点，但其中一个用作兄弟指针，因此最多存 4 个值）。

**2. 叶子节点的容量约束**

- 公式回顾：叶子节点最少存储 \(\lceil (n-1)/2 \rceil\) 个值，最多存储 \( n-1 \) 个值。
- 代入 \( n = 5 \)：
    - 最少：\(\lceil (5-1)/2 \rceil = \lceil 4/2 \rceil = 2\) 个值。
    - 最多：\(5-1 = 4\) 个值。
- **结论**：每个叶子节点必须存储 **2 到 4 个搜索键值**。若少于 2 个（除根外），会触发合并或重分配；若达到 5 个，则必须分裂。

**3. 非叶子节点（除根外）的容量约束**

- 公式回顾：非叶子节点最少拥有 \(\lceil n/2 \rceil\) 个子节点（即指针数），最多拥有 \( n \) 个子节点。
- 代入 \( n = 5 \)：
    - 最少：\(\lceil 5/2 \rceil = \lceil 2.5 \rceil = 3\) 个子节点。
    - 最多：\(5\) 个子节点。
- **注意区分**：指针数比键值数多 1。因此该节点内部存储的键值数量范围为 **2 到 4 个**（与叶子节点恰好一致，但这是巧合；若 \( n \) 为偶数则不同）。

**4. 根节点的特殊规则**

- 若根不是叶子（即树有多层），则根至少要有 **2 个子节点**。即使 \( n=5 \)，根可以只有 2 个子节点，无需满足 3 个的最低要求。
- 若根是叶子，则值数量可以为 0 到 4。

**5. 参数实例化的意义**

- 通过代入具体数字，可以直观感受到 B⁺ 树的填充率要求（50%~100%），以及为何分裂与合并操作总是围绕这些阈值展开。
- 实际数据库中 \( n \) 通常为几百甚至上千，但约束逻辑完全相同。

### 第 2 页：B⁺ 树的宏观特性总结

这一页将前面所有技术细节提炼为三条核心观察，回答了“为什么 B⁺ 树是数据库索引的最佳选择”。

**1. 观察一：非叶子层级形成稀疏索引的层次结构**

- **原文**：*The non-leaf levels of the B⁺-tree form a hierarchy of sparse indices.*

- **深度解读**：
    - **底层（叶子层）**：可以是稠密索引（每条记录一个条目），也可以直接存储数据行。
    - **上层（非叶子层）**：每一层都是对下一层的**稀疏索引**——每个内节点条目对应下一层的一整个节点（页面），而非对应每一条记录。
    - **层次压缩效应**：每一层都将下一层的数据规模压缩约 \( n \) 倍。这种“层层收口”的结构使得顶层体积极小，极易常驻内存。

**2. 观察二：层级数量极少，查询效率极高**

- **原文**：*The B⁺-tree contains a relatively small number of levels (logarithmic in the size of the main file), thus searches can be conducted efficiently.*

- **数量级估算**：
    - 假设 \( n = 100 \)（实际数据库常为几百），数据记录数为 \( 10^6 \)。
    - 叶子节点数 ≈ \( 10^6 / 100 = 10,000 \)。
    - 第二层节点数 ≈ \( 10,000 / 100 = 100 \)。
    - 第三层节点数 ≈ \( 100 / 100 = 1 \)（根节点）。
    - **树高仅为 3 层**。即使数据量达到 \( 10^9 \)（十亿），树高也仅约 4~5 层。

- **查询代价**：树高直接对应最坏情况下的磁盘 I/O 次数。百万级数据仅需 2~3 次 I/O，这正是 B⁺ 树能支撑高并发 OLTP 系统的数学基础。

**3. 观察三：插入与删除可在对数时间内高效完成**

- **原文**：*Insertions and deletions to the main file can be handled efficiently, as the index can be restructured in logarithmic time.*

- **什么是“对数时间”**：
    - 插入或删除一条记录，索引结构调整的代价与树的**高度**成正比，即 \( O(\log N) \)。
    - 具体操作包括：定位目标叶子节点（树高次 I/O），执行插入/删除，若触发分裂或合并则沿路径**向上回溯**调整祖先节点（最多回溯到根，即树高次操作）。

- **与索引顺序文件的对比**：
    - 索引顺序文件在插入时可能产生溢出链，导致性能逐渐退化，最终需要 \( O(N) \) 的全文件重组。
    - B⁺ 树通过局部调整维持全局平衡，**无需停机重组**，这是其压倒性优势。
