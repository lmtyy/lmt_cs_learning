### 第 1 页：SQL 索引基本语法与 SQL Server 默认行为

这一页介绍了 SQL 标准中创建和删除索引的基本命令，并补充了 SQL Server 的一个特定默认规则。

**1. 创建索引的标准语法**

```sql
CREATE INDEX <index-name> ON <relation-name> (<attribute-list>);
```

- **`index-name`**：索引的名称，在同一数据库中应具有唯一性（针对该表）。
- **`relation-name`**：要创建索引的表名。
- **`attribute-list`**：一个或多个列名的列表，构成索引的**搜索键**。多列索引的顺序会影响查询效率。
- **示例**：`CREATE INDEX myindex ON student(sname);`
    - 在 `student` 表的 `sname` 列上创建名为 `myindex` 的索引。
    - 默认情况下（未指定类型），创建的通常是**非聚集索引**，即索引结构与数据行分开存储，索引叶子节点存的是指向数据行的指针。

**2. 删除索引的标准语法**

```sql
DROP INDEX <index-name>;
```

- 删除不再需要的索引，以释放存储空间并减少数据修改时的索引维护开销。

**3. SQL Server 的特殊默认行为（PPT 中强调）**

> SQL Server 默认会将主键设置为聚集索引，不管是单列主键还是复合主键。

- **知识点**：这是 SQL Server 的**具体实现特性**，并非所有 DBMS 都如此（例如 MySQL 也是默认主键为聚集索引，但 Oracle 默认不是）。
- **聚集索引**决定了表中数据的**物理存储顺序**。因此，将主键设为聚集索引意味着数据行在磁盘上就是按主键值的顺序排列的，这对基于主键的范围查询非常有利。
- **注意**：如果你在 SQL Server 中创建表时定义了主键，系统会自动生成一个同名的聚集索引；如果你希望主键是非聚集的，需要在建表语句中显式指定 `NONCLUSTERED`。

### 第 2 页：聚集索引与唯一索引

这一页深入介绍了两种重要的索引类型在 SQL 中的显式创建语法。

**1. 聚集索引（Clustered Index）**

- **语法纠正**：PPT 中的 `create dustered index ...` 应为拼写错误，正确是 `CREATE CLUSTERED INDEX ...`。
- **示例**：`CREATE CLUSTERED INDEX b-index ON branch(branch-name);`
- **核心概念**：
    - **一个表只能有一个聚集索引**。因为数据行只能以一种物理顺序存储。
    - 索引的叶子节点**直接包含整行数据**（而不是指针）。因此，通过聚集索引找到键值后，无需额外回表即可获取全部列数据。
    - 适用于经常进行范围查询（如 `BETWEEN`、`>`、`<`）的列。

**2. 唯一索引（Unique Index）与候选键约束**

- **语法**：`CREATE UNIQUE INDEX b-index ON branch(branch-name);`
- **知识点**：`UNIQUE` 关键字有两个作用：
    1. **性能优化**：告知优化器该列的值是唯一的，可能产生更优的执行计划。
    2. **数据完整性约束**：**间接指定并强制**该搜索键是一个**候选键**。
- **候选键含义**：可以唯一标识表中每一行的列或列组合。如果试图插入重复值，数据库会报错并拒绝操作。
- **与主键的区别**：主键自动隐含 `UNIQUE` 和 `NOT NULL`，且默认通常创建聚集索引；而 `UNIQUE INDEX` 允许一个 `NULL` 值（不同 DBMS 实现略有差异），且默认创建的是非聚集索引。

### 第 3 页：MySQL 索引组织表与 InnoDB 主键机制

这一页从通用 SQL 语法转向具体数据库 MySQL 的 InnoDB 存储引擎实现原理。

**1. 索引组织表**

- **定义**：表的数据本身就是按照某个索引的 B+ 树结构来存储的，这种表称为**索引组织表**。
- **对比**：**堆表**的数据存储是无序的，索引叶子节点只存指针；而**索引组织表**中，主键索引（聚集索引）的叶子节点直接存放**整行数据**。

**2. InnoDB 的存储特性**

> InnoDB 中，表中的记录是按照主键的顺序存放的。

- **结论**：InnoDB 表**必然**是一个索引组织表。如果没有显式定义主键，InnoDB 会隐式生成一个。

**3. InnoDB 主键生成的优先级规则（非常关键的面试考点）**

PPT 中列出了 InnoDB 在没有人为指定主键时的自动选择逻辑：

| 优先级 | 条件                                     | 行为                                                         |
| :----- | :--------------------------------------- | :----------------------------------------------------------- |
| **1**  | 表中定义了 **PRIMARY KEY**               | 直接使用用户定义的主键作为聚集索引。                         |
| **2**  | 表中**没有主键**，但有**唯一的非空索引** | 选择**第一个**遇到的符合条件（所有列均为 `NOT NULL`）的唯一索引作为聚集索引。 |
| **3**  | 既无主键，也无符合条件的唯一索引         | InnoDB 自动在后台创建一个**6 字节**的隐藏列 `ROW_ID`，并以此构建聚集索引。 |

**4. 深度解读：为何如此设计？**

- **因为 InnoDB 数据必须挂在 B+ 树下**：InnoDB 的表数据文件（`.ibd`）本身就是按主键 B+ 树组织的。没有主键，数据就无法有序存放。
- **隐藏主键的缺陷**：隐藏的 6 字节 `ROW_ID` 对用户不可见，无法用于查询优化。且在复制或分布式环境中，多台机器生成的隐藏 `ROW_ID` 可能不同，导致主从数据逻辑顺序不一致。**强烈建议显式定义主键**。
