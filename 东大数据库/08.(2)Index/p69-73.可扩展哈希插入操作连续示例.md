### 第 1 页：初始状态与插入 18、32

这一页展示了可扩展哈希的初始结构，以及插入键值 `18` 和 `32` 后的结果。

**1. 初始哈希结构**

- **目录**：大小为 4，索引为 `00`、`01`、`10`、`11`（全局深度 GD = 2，使用最后 2 位）。
- **桶与局部深度**：
    - 目录项 `00` 指向桶 A，局部深度 LD = 2，存储键值：`2`、`6`、`10`。
    - 目录项 `01` 指向桶 B，局部深度 LD = 1，存储键值：`1`、`5`、`9`、`13`。
    - 目录项 `10` 指向桶 C，局部深度 LD = 2，存储键值：`8`、`16`、`64`。
    - 目录项 `11` 指向同一个桶 B（因为 LD=1，两个目录项 `01` 和 `11` 共享同一桶）。

**2. 哈希函数说明**

- 使用键值的**最低有效位**作为哈希值。
- 全局深度 GD = 2，因此取最后 2 位确定目录项。

**3. 插入 18（二进制 010010）**

- 二进制 `18` = `010010`，最后 2 位为 `10`。
- 定位到目录项 `10`，对应桶 C（存储 `8`、`16`、`64`）。
- 桶 C 当前有 3 个值，假设桶容量为 3（已满），插入 `18` 需分裂。
- **分裂过程**：
    - 桶 C 局部深度 LD = 2 = GD，分裂需**翻倍目录**。
    - GD 从 2 增至 3，目录大小从 4 翻倍为 8。
    - 桶 C 中的记录按最后 3 位重新分配：
        - `8` = `001000`，最后 3 位 `000` → 新目录项 `000`
        - `16` = `010000`，最后 3 位 `000` → 新目录项 `000`
        - `64` = `1000000`（假设取低 3 位 `000`）→ `000`
        - `18` = `010010`，最后 3 位 `010` → `010`
    - 分裂后形成两个桶：
        - 桶 C1（LD=3）：`8`、`16`、`64`（对应目录项 `000`）
        - 桶 C2（LD=3）：`18`（对应目录项 `010`）
    - 其他目录项根据原映射复制指针。

**4. 插入 32（二进制 100000）**

- `32` = `100000`，最后 3 位 `000`。
- 定位到目录项 `000`，桶 C1 已有 3 个值，再次满，需分裂。
- 桶 C1 的 LD = 3 = GD，分裂再次翻倍目录：
    - GD 从 3 增至 4，目录大小翻倍为 16。
    - 桶 C1 中的记录按最后 4 位重新分配：
        - `8` = `1000`（低 4 位）→ `1000`
        - `16` = `0000` → `0000`
        - `64` = `0000` → `0000`
        - `32` = `0000` → `0000`
    - 分裂后两个桶：
        - 一个存 `16`、`64`、`32`（对应低 4 位 `0000`）
        - 另一个存 `8`（对应低 4 位 `1000`）

**5. 第 1 页图示状态解读**

- PPT 中的表格可能展示的是插入后的最终状态，但由于排版较乱，核心是说明随着插入操作，目录不断翻倍，桶逐渐分裂，全局深度增加。

**2. 插入 4（二进制 100）**

**1. 插入 19（二进制 10011）**

- 定位到对应桶，若满则分裂。

**2. 插入 17（二进制 10001）**

### 第 4 页：插入 24 后的状态

这一页在插入 `19` 和 `17` 之后，又插入了 `24`。

**1. 插入 24（二进制 11000）**

- `24` = `11000`，取最后 GD 位（当前 GD=3）：最后 3 位 `000`。
- 定位到 `000` 桶，该桶已有 `2`、`6`、`10`、`18`，若桶容量为 4 则已满，需分裂。
- 分裂过程：
    - 桶 LD=3 = GD，分裂需翻倍目录，GD 从 3 变为 4。
    - 目录大小从 8 翻倍为 16。
    - 原 `000` 桶中的记录按最后 4 位重新分配：
        - `2` = `0010` → 低 4 位 `0010`
        - `6` = `0110` → `0110`
        - `10` = `1010` → `1010`
        - `18` = `0010` → `0010`（与 `2` 同）
        - `24` = `1000` → `1000`
    - 分裂后可能形成多个桶，例如：
        - 一个桶存 `2`、`18`（低 4 位 `0010`）
        - 一个桶存 `6`、`10`（低 4 位 `0110` 和 `1010`，可能再分）
        - 一个桶存 `24`（低 4 位 `1000`）

**2. 图示内容**

- 目录项从 `000` 到 `111`（8 项），但局部深度显示有些桶 LD=3，有些 LD=2。
- 具体桶内容：
    - `3` 桶：`1`、`9`、`17`（LD=3）
    - `2` 桶：`6`、`10`、`18`（LD=2）
    - `7` 桶：`3`、`19`（LD=2）
    - `4` 桶：`4`（LD=3）
    - `5` 桶：`13`（LD=3）
    - `8` 桶：`24`（LD=？）
- 注意此时 `8` 和 `16`、`64`、`32` 似乎未显示，可能因目录翻倍后它们被映射到其他目录项。

### 第 5 页：插入 24 之后的最终状态

这一页专门展示插入 `24` 之后稳定下来的哈希结构。

**1. 目录与全局深度**

- 目录项从 `0000` 到 `1111`，共 16 项，GD=4。
- 各目录项指向的桶及局部深度：
    - 桶存 `1`、`9`、`17`：LD=4
    - 桶存 `2`、`6`、`10`、`18`：LD=2（或 3？）
    - 桶存 `7`、`3`、`19`：LD=2
    - 桶存 `4`：LD=3
    - 桶存 `8`、`24`：LD=4
    - 桶存 `5`、`13`：LD=3
    - 桶存 `16`、`32`：LD=？
    - 桶存 `64`：LD=？

**2. 状态分析**

- 经过多次插入和分裂，目录已扩展到 16 项，GD=4。
- 不同桶的局部深度不同，反映了它们被创建时的目录层级。
- 空目录项可能指向同一个桶（因为局部深度小于全局深度时，多个目录项共享一个桶）。

**3. 核心知识回顾**

通过这五页的连续示例，我们实践了可扩展哈希的以下要点：

- **查找**：取哈希值的最后 GD 位作为目录索引。
- **插入**：定位桶，若空间足够直接插入；若满则分裂。
- **分裂**：
    - 若桶的 LD < GD，仅拆分桶，分配新页，重新分配记录，更新受影响的目录指针，LD 加 1，目录不翻倍。
    - 若桶的 LD = GD，则需先翻倍目录（GD 加 1），再拆分桶。
- **目录翻倍**：仅增加目录项，数据页不移动，通过调整指针完成重新映射。
