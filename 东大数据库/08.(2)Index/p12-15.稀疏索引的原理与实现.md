## 第一页：稀疏索引的定义与图示

### PPT原文

> **Sparse Index** — contains index records for only **some** search-key values.

### 核心定义解读

- **中文名称：** 稀疏索引
- **关键特征：** 索引文件中**只包含部分搜索键值**的索引记录，而非每一个键值都有。
- **对比稠密索引：** 稠密索引是“每个键值都有条目”，稀疏索引是“隔几个键值才有一个条目”。

### 图表结构分析

**左侧：稀疏索引文件（建立在 `branch_name` 上）**

| 搜索键 (branch_name) | 指针 |
| :------------------- | :--- |
| Brighton             | →    |
| Mianus               | →    |
| Redwood              | →    |

**右侧：主数据文件（按 `branch_name` 排序存储）**

| Branch     | Account | Balance |
| :--------- | :------ | :------ |
| Brighton   | A-217   | 750     |
| Downtown   | A-101   | 500     |
| Downtown   | A-110   | 600     |
| Mianus     | A-215   | 700     |
| Perryridge | A-102   | 400     |
| Perryridge | A-201   | 900     |
| Perryridge | A-218   | 700     |
| Redwood    | A-222   | 700     |
| Round Hill | A-305   | 350     |

### 关键观察：索引条目数远少于数据键值数

- **数据文件中出现的所有键值：** Brighton、Downtown、Mianus、Perryridge、Redwood、Round Hill（共 6 个）。
- **索引文件中出现的键值：** Brighton、Mianus、Redwood（**只有 3 个**）。
- **缺失的键值：** Downtown、Perryridge、Round Hill 在索引中**没有**独立条目。

### 为什么可以“缺失”一些键值？

这是稀疏索引能够工作的**前提条件**，也是理解稀疏索引的关键：

> **数据文件必须按照搜索键排序存储。**

因为数据已经按 `branch_name` 排序了，所以：
- 所有 Downtown 记录**物理上连续存放**在 Brighton 和 Mianus 之间。
- 要找 Downtown，可以先跳到 Brighton，然后**顺序向后扫描**，自然就能找到紧跟在后面的 Downtown 记录。

如果数据没有排序，稀疏索引**无法工作**——因为扫描时不知道目标数据在哪个方向、何时结束。

## 第二页：两个关键问题

这一页 PPT 以提问的形式，引导我们思考稀疏索引的适用边界和查找方法。

### Question 1: Apply to clustered index or secondary index?

**问题：** 稀疏索引适用于聚集索引还是辅助索引？

**答案：** **只适用于聚集索引（主索引）。**

**原因分析：**

| 条件                     | 聚集索引 | 辅助索引           |
| :----------------------- | :------- | :----------------- |
| 数据是否按索引字段排序   | **是**   | **否**             |
| 相同键值记录是否连续存放 | **是**   | **否**（散落各处） |
| 能否用稀疏索引           | **能**   | **不能**           |

- **聚集索引中：** 数据按索引字段排序。索引条目指向某个键值的起点，从此处顺序扫描，可以找到后续所有同键值或介于两个索引条目之间的记录。
- **辅助索引中：** 数据**不**按索引字段排序。如果索引中缺失了某个键值，我们无法知道该键值的记录散落在数据文件的哪些位置，扫描无从下手。

**结论：** 稀疏索引**必须**建立在已排序的数据文件上，因此只能是**聚集索引**。

## 第三页：稀疏索引的查找算法

这一页给出了稀疏索引查找的**完整规则**和**前提条件**。

### PPT原文

> **Sparse Index**
> - clustering indexes（文件必须排序）
>
> To locate a record with search-key value \( K \) we:
> 1. Find index record with largest search-key value **<= K**
> 2. Search file **sequentially** starting at the record to which the index record points

### 前提条件重申

> clustering indexes（文件必须排序）

这是稀疏索引**不可动摇的前提**。如果文件未排序，稀疏索引无法工作。

### 两步查找算法详解

假设我们要查找 `branch_name = 'Perryridge'`（即 K = 'Perryridge'）。

**第一步：找到最大的、小于等于 K 的索引条目**

索引文件内容：
| 搜索键   | 指针指向 |
| :------- | :------- |
| Brighton | A-217    |
| Mianus   | A-215    |
| Redwood  | A-222    |

- K = 'Perryridge'
- 遍历索引条目：
  - Brighton ≤ Perryridge？是。
  - Mianus ≤ Perryridge？是（M 在 P 之前）。
  - Redwood ≤ Perryridge？**否**（R 在 P 之后）。
- **选择的索引条目：** Mianus（最大的 ≤ K 的键值）。
- **获得的指针：** 指向 A-215（Mianus 的第一条记录）。

**第二步：从指针位置开始顺序扫描**

- 从 A-215（Mianus）开始，在数据文件中**逐条向后扫描**。
- 扫描过程：
  - A-215：Mianus（不是 Perryridge，继续）
  - A-102：Perryridge ✓ **命中第一条**
  - A-201：Perryridge ✓ **命中第二条**
  - A-218：Perryridge ✓ **命中第三条**
  - A-222：Redwood（超出范围，停止扫描）

**查找完成**，返回三条 Perryridge 记录。

### 图示化的查找路径

```
稀疏索引:
┌──────────┬─────────┐
│ Brighton │ → A-217 │
│ Mianus   │ → A-215 │  ← 第一步：找到这个（Mianus ≤ Perryridge）
│ Redwood  │ → A-222 │
└──────────┴─────────┘
              │
              ▼
数据文件（顺序扫描方向 →）:
A-217: Brighton
A-101: Downtown
A-110: Downtown
A-215: Mianus      ← 第二步：从这里开始顺序扫描
A-102: Perryridge  ← 命中
A-201: Perryridge  ← 命中
A-218: Perryridge  ← 命中
A-222: Redwood     ← 遇到大于 K 的值，停止
A-305: Round Hill
```

### 查找不存在的键值

假设查找 `branch_name = 'Downtown'`：

1. 找最大 ≤ K 的索引条目：Brighton ≤ Downtown？是。Mianus ≤ Downtown？**否**。选 Brighton。
2. 从 A-217（Brighton）开始顺序扫描。
3. 扫描 A-101（Downtown）→ 命中，A-110（Downtown）→ 命中。
4. 遇到 A-215（Mianus）时，发现已经大于 Downtown，停止。

即使 Downtown 不在索引中，依然能找到它。

假设查找 `branch_name = 'Lexington'`（不存在的值）：

1. 找最大 ≤ K 的索引条目：Brighton ≤ Lexington？是。Mianus ≤ Lexington？是。Redwood ≤ Lexington？**否**。选 Mianus。
2. 从 A-215（Mianus）开始扫描，经过 Perryridge、Redwood... 直到文件末尾，都没有找到 Lexington。
3. 当遇到大于 Lexington 的键值（或文件结束）时，确认不存在。

## 第四页：稀疏索引的优势与设计权衡

这一页讨论了稀疏索引的**工程价值**和**典型实现方式**。

### 优势一：更少的空间开销

> Less space and less maintenance overhead for insertions and deletions.

- **空间节省：** 稀疏索引的索引条目数远少于稠密索引，索引文件更小，更容易全部加载到内存中。
- **维护开销低：**
  - **插入新记录时：** 只要新记录落在已有索引条目覆盖的范围内，索引本身**不需要更新**。只有当插入的记录成为某个数据块的**新的最小值**时，才可能需要更新索引。
  - **删除记录时：** 同样，只要不涉及索引条目指向的那条记录，索引无需修改。

### 优势二：良好的权衡设计

> Good tradeoff: sparse index with an index entry for **every block in file**, corresponding to **least search-key value in the block**.

这是稀疏索引在数据库系统中的**经典实现方式**：

- **每个数据块建一个索引条目。**
- **索引条目存储该块中的最小搜索键值。**

### 块级稀疏索引的图示

```
数据文件（每个 Block 存多条记录）:
┌─────────────────────────────────────────────────────────────┐
│ Block 1                                                     │
│ ┌──────────┬─────────┬────────┐                            │
│ │ Brighton │ A-217   │ 750    │  ← 块内最小键: Brighton    │
│ │ Downtown │ A-101   │ 500    │                            │
│ │ Downtown │ A-110   │ 600    │                            │
│ └──────────┴─────────┴────────┘                            │
├─────────────────────────────────────────────────────────────┤
│ Block 2                                                     │
│ ┌──────────┬─────────┬────────┐                            │
│ │ Mianus   │ A-215   │ 700    │  ← 块内最小键: Mianus      │
│ │ Perryridge│ A-102  │ 400    │                            │
│ │ Perryridge│ A-201  │ 900    │                            │
│ │ Perryridge│ A-218  │ 700    │                            │
│ └──────────┴─────────┴────────┘                            │
├─────────────────────────────────────────────────────────────┤
│ Block 3                                                     │
│ ┌──────────┬─────────┬────────┐                            │
│ │ Redwood  │ A-222   │ 700    │  ← 块内最小键: Redwood     │
│ │ Round Hill│ A-305  │ 350    │                            │
│ └──────────┴─────────┴────────┘                            │
└─────────────────────────────────────────────────────────────┘

稀疏索引（每个 Block 一条索引记录）:
┌──────────┬──────────────┐
│ Brighton │ 指向 Block 1  │
│ Mianus   │ 指向 Block 2  │
│ Redwood  │ 指向 Block 3  │
└──────────┴──────────────┘
```

### 这种设计的精妙之处

1. **索引极小：** 如果每个数据块存 100 条记录，那么索引大小只有数据文件的大约 1/100。
2. **查找高效：** 定位到目标块后，只需在**一个块内**进行顺序扫描，扫描范围被严格控制。
3. **维护简单：** 插入新记录通常只影响一个数据块。只有当块分裂或合并时，才需要更新索引。
4. **与 B+ 树的关联：** 这种设计思想正是 B+ 树索引的**叶子层**的基础——上层是稀疏索引，叶子层是稠密数据。

### 稠密索引 vs. 稀疏索引 全面对比

| 对比项            | 稠密索引                     | 稀疏索引              |
| :---------------- | :--------------------------- | :-------------------- |
| **索引条目数**    | 等于不同搜索键值的数量       | 远少于数据记录数      |
| **适用索引类型**  | 聚集索引和辅助索引都可用     | **只能用于聚集索引**  |
| **数据排序要求**  | 聚集时要求排序，辅助时不要求 | **必须排序**          |
| **索引大小**      | 较大                         | 较小                  |
| **查找方式**      | 直接定位到第一条记录         | 定位到起点 → 顺序扫描 |
| **查找精确度**    | 精确跳转                     | 需要额外扫描少量记录  |
| **插入/删除开销** | 每次都可能需更新索引         | 多数情况无需更新索引  |
| **典型应用**      | 辅助索引、哈希索引           | 主索引、B+树上层节点  |
