### 第 1 页：InnoDB 的聚簇索引与辅助索引

这一页是理解 MySQL InnoDB 索引机制的核心，明确区分了两种索引的存储结构和查询行为。

**1. 聚簇索引**

- **定义**：根据表的**主键**建立的一个 B⁺ 树索引。

- **核心特征：叶子节点直接存放行数据**
    - 这与经典 B⁺ 树叶子节点存放“指针”不同。
    - 在 InnoDB 的聚簇索引中，B⁺ 树的叶子节点页里存储的是**完整的行记录**（包括所有列的值）。
    - **结论**：表的数据文件本身就是聚簇索引的叶子节点。因此，InnoDB 的表被称为**索引组织表**。

- **数量限制**：一个表**只能有一个**聚簇索引，因为数据行只能以一种物理顺序存储。

- **图示对应**：PPT 第二页的图例中，Level 0 Leaf 的页（Page 6~9）存放的就是完整的行数据。

**2. 辅助索引**

- **定义**：在非主键列上建立的其他所有索引。

- **数量**：一个表**可以有多个**辅助索引。

- **核心特征**：
    - **叶子节点不包括行数据**：辅助索引的叶子节点只存储索引列的值和对应的**主键值**。
    - 结构为：`<索引列值, 主键值>`。

- **查找过程**：
    1. 在辅助索引的 B⁺ 树中定位目标索引列值，获取对应的**主键值**。
    2. 如果查询所需的所有列都包含在辅助索引中（覆盖索引），则直接返回，查询结束。
    3. 如果还需要其他列的数据，则拿着获取到的**主键值**，再到**聚簇索引**的 B⁺ 树中进行第二次查找，以获取完整的行数据。

**3. “回表”的深刻含义**

- **定义**：*如果辅助索引里不包含全部要查找的字段，则需要根据索引叶子节点存的主键值，再到聚集索引里查找需要的字段，这个过程叫做回表。*

- **回表的代价**：
    - 一次辅助索引查找可能变成**两次 B⁺ 树搜索**（先搜辅助索引，再搜聚簇索引）。
    - 如果范围查询通过辅助索引返回多条记录，每条记录都可能需要一次**随机 I/O** 来回表读取完整行数据，代价极高。

- **优化策略——覆盖索引**：
    - 如果查询的列**全部**包含在辅助索引中（即索引列 + 主键列覆盖了 SELECT 子句），则无需回表。
    - 例如：表有索引 `INDEX idx_name_age (name, age)`，查询 `SELECT name, age FROM t WHERE name = 'Tom'` 就只访问辅助索引即可。

**4. 两种索引对比总结**

| 对比维度         | 聚簇索引             | 辅助索引                     |
| :--------------- | :------------------- | :--------------------------- |
| **数量**         | 最多 1 个            | 可以有多个                   |
| **叶子节点内容** | 完整行数据           | 索引列值 + 主键值            |
| **是否回表**     | 无需（直接拿到数据） | 可能需要（若未覆盖）         |
| **范围查询效率** | 极高（数据物理连续） | 一般（可能需要大量随机回表） |

### 第 2 页：InnoDB 页结构与记录组织

这一页通过一个图例展示了 InnoDB B⁺ 树的**页内结构**和**页间链接**关系，是理解 InnoDB 数据存储物理细节的关键。

**1. 图中的层级标识**

- **Level 2 Root**：根节点页（Page 3），位于 B⁺ 树的最顶层。
- **Level 0 Leaf**：叶子节点页（Page 6、7、8、9），位于 B⁺ 树的最底层，存放实际数据。

- **层级意义**：InnoDB 的 B⁺ 树通常只有 2~3 层。图中展示的是一个高度为 3 的 B⁺ 树（Level 2 → Level 1 → Level 0）。

**2. 页内关键元素**

- **页号**：每个页有一个唯一编号（如 Page 3、Page 4）。
- **Infimum 和 Supremum**：
    - 每个 InnoDB 数据页在逻辑上包含两个**虚拟记录**。
    - **Infimum**：比页内任何实际记录都**小**的虚拟最小记录。
    - **Supremum**：比页内任何实际记录都**大**的虚拟最大记录。
    - 作用：作为页内记录链表的**头和尾**，简化边界条件的处理（如插入到页首或页尾时无需特殊判断）。

**3. 页内记录的逻辑顺序——Next Record 指针**

- 图中用 `→` 表示记录之间的单向链表。
- 每个记录头部包含一个指向**下一条记录**的指针（Next Record），记录在页内按主键顺序**升序排列**。
- 例如：记录 A → 记录 B → 记录 C → ... → Supremum。
- **意义**：即使在同一个数据页内部，记录也不是完全连续存放的（可能存在碎片），而是通过链表维护逻辑顺序。这使得插入和删除可以在页内局部调整指针，而无需移动大量数据。

**4. 页间链接——Next Page 与 Previous Page**

- **Next Page**：指向**下一个**兄弟叶子页的指针（对应 B⁺ 树叶子节点的 \( P_n \) 右兄弟指针）。
- **Previous Page**：指向**上一个**兄弟叶子页的指针。
- **双向链表**：InnoDB 的叶子节点页之间构成了**双向链表**，支持正向和反向的范围扫描（如 `ORDER BY ASC` 和 `ORDER BY DESC`）。

**5. 图中路由关系解读**

- 根节点（Page 3）中存储导航键值和子页指针。
    - `≥0 → 4`：键值 ≥0 的记录路由到 Page 4。
    - `≥4 → 5`：键值 ≥4 的记录路由到 Page 5。
- Page 4 和 Page 5 是内节点（Level 1），继续向下路由：
    - Page 4 将 `≥0` 路由到 Page 6，`≥2` 路由到 Page 7。
    - Page 5 将 `≥4` 路由到 Page 8，`≥6` 路由到 Page 9。
- Page 6~9 是叶子节点（Level 0），存储实际数据记录，并彼此通过双向指针连接。

**6. 关键知识点总结**

| 概念                          | 作用                                     |
| :---------------------------- | :--------------------------------------- |
| **Infimum / Supremum**        | 虚拟边界记录，简化页内链表操作           |
| **Next Record**               | 页内记录的单向链表，维护逻辑有序性       |
| **Next Page / Previous Page** | 叶子页之间的双向链表，支持高效范围扫描   |
| **层级路由**                  | 根节点→内节点→叶子节点，逐层缩小查找范围 |
