### 第 1 页：可扩展哈希的补充评论

这一页讨论了可扩展哈希在实际应用中的性能表现、潜在问题以及删除操作的处理机制。

**1. 查询性能与目录内存驻留**

- **原文**：*If directory fits in memory, equality search answered with one disk access; else two.*

- **等值查询的 I/O 次数**：
    - **目录在内存中**：查找过程为：计算哈希值 → 在内存目录中定位桶地址 → 读取目标桶页（**1 次磁盘 I/O**）。
    - **目录不在内存中**：需先将目录页读入内存，再读取桶页（**2 次磁盘 I/O**）。

- **目录大小的实际估算**：
    - 假设数据文件 100MB，每条记录 100 字节，页面大小 4KB。
    - 记录总数 = 100MB / 100B = **1,000,000 条**。
    - 每页可存放的记录数 = 4KB / 100B ≈ **40 条**。
    - 所需桶（数据页）数量 = 1,000,000 / 40 = **25,000 个**。
    - 目录需要 25,000 个指针元素，每个指针约 4~8 字节，目录总大小约 **100KB ~ 200KB**。
    - **结论**：即使数据文件达到 100MB，目录大小也仅约几百 KB，**极大概率可以完全常驻内存**。因此，等值查询通常只需 **1 次磁盘 I/O**。

**2. 目录增长的两个潜在问题**

| 问题               | 原因                                                         | 后果                                                 |
| :----------------- | :----------------------------------------------------------- | :--------------------------------------------------- |
| **目录阶段性激增** | 当 LD = GD 的桶分裂时，目录翻倍。若连续多次触发，目录可能快速膨胀。 | 短期内内存占用增加，但通常仍可控。                   |
| **哈希值分布倾斜** | 若大量键值的哈希值集中在少数几个桶中，这些桶频繁分裂，导致目录不断翻倍，而大部分目录项指向空桶或稀疏桶。 | 目录利用率低，空间浪费，极端情况下目录可能变得很大。 |

- **相同哈希值问题**：
    - 若多个搜索键具有**完全相同的哈希值**（哈希冲突），它们必然落入同一个桶。
    - 当该桶满时，由于所有键的哈希值相同，**无法通过增加区分位来拆分**（无论取多少位，这些键的哈希值仍相同）。
    - 此时必须使用**溢出页**来处理，这是可扩展哈希的一个理论缺陷。

**3. 删除操作的处理机制**

- **原文**：*If removal of data entry makes bucket empty, can be merged with `split image`. If each directory element points to same bucket as its split image, can halve directory.*

- **删除后的桶合并**：
    - 当删除记录导致某个桶变为**空桶**时，可将其与它的 **split image** 合并。
    - **split image**：指与该桶在分裂时共同组成原桶的兄弟桶（即哈希值仅差分裂位的那一对桶）。
    - 合并后，局部深度减 1，释放一个桶页。

- **目录减半的条件**：
    - 检查目录中**每一对** split image 目录项（即仅最高有效位不同的两个目录项）。
    - 如果**所有这样的配对**都指向**同一个桶**，说明目录中一半的指针是冗余的。
    - 此时可将**全局深度减 1，目录大小减半**，释放目录空间。

- **收缩的意义**：
    - 可扩展哈希不仅支持动态**增长**，也支持动态**收缩**。
    - 当数据量下降时，通过合并桶和收缩目录，可以回收空间，维持较高的空间利用率。

**4. 可扩展哈希要点总结**

| 特性         | 说明                                     |
| :----------- | :--------------------------------------- |
| **查询性能** | 目录常驻内存时仅 1 次 I/O                |
| **目录大小** | 通常很小，易常驻内存                     |
| **增长方式** | 目录阶段性翻倍，桶按需分裂               |
| **潜在风险** | 哈希倾斜导致目录膨胀，相同哈希值需溢出页 |
| **收缩机制** | 空桶合并，目录可减半                     |

### 第 2 页：有序索引与哈希索引的对比

这一页从查询类型适用性和主流数据库实践两个角度，对 B⁺ 树（有序索引）和哈希索引进行了横向对比。

**1. 基于查询类型的选择原则**

| 查询类型     | 推荐索引 | 原因                                                         |
| :----------- | :------- | :----------------------------------------------------------- |
| **等值查询** | 哈希索引 | O(1) 定位，通常 1 次 I/O，优于 B⁺ 树的 O(log N)              |
| **范围查询** | 有序索引 | B⁺ 树叶子节点形成链表，范围扫描效率极高；哈希索引完全无法支持 |

- **补充说明**：
    - 哈希索引对于 `=`、`IN` 等精确匹配查询表现优异。
    - 哈希索引**不支持**：`>`、`<`、`BETWEEN`、`ORDER BY`、`GROUP BY`（非等值分组）、前缀模糊匹配（如 `LIKE 'abc%'`）。
    - 因此，在通用关系数据库中，B⁺ 树是默认且主要的选择。

**2. 主流数据库的实践态度**

| 数据库         | 对哈希索引的支持                           | 备注                                                         |
| :------------- | :----------------------------------------- | :----------------------------------------------------------- |
| **PostgreSQL** | 支持，但**不建议使用**                     | 官方文档明确指出哈希索引性能不如 B⁺ 树，且不记录 WAL 日志（早期版本），恢复时需重建。PG 10 后虽有改进，但 B⁺ 树仍是绝对主流。 |
| **Oracle**     | 支持**静态哈希文件组织**，但不支持哈希索引 | 可创建哈希分区表或哈希簇，但没有独立的哈希索引结构。索引默认均为 B⁺ 树。 |
| **SQL Server** | **仅支持 B⁺ 树**                           | 完全不提供哈希索引选项，所有索引均为 B⁺ 树或其变种。         |

- **MySQL 的补充说明**（PPT 未提及但值得了解）：
    - **Memory 引擎**：支持哈希索引（默认），也支持 B⁺ 树。
    - **InnoDB 引擎**：支持**自适应哈希索引**——系统自动在内存中为热点数据页构建哈希索引，用户无法手动创建，但对等值查询有隐形加速效果。

**3. 为什么哈希索引未成为主流？**

- **功能局限**：无法支持范围查询和排序，而这是关系数据库的核心需求。
- **持久化问题**：哈希索引在崩溃恢复时可能需要重建，而 B⁺ 树可通过日志完整恢复。
- **空间利用率波动**：动态哈希虽有改善，但在极端哈希倾斜下仍可能退化为长溢出链。
- **并发控制复杂**：目录翻倍等操作在并发环境下锁粒度较难控制。

**4. 实际应用场景中的选择策略**

| 场景                         | 推荐方案                             |
| :--------------------------- | :----------------------------------- |
| **OLTP 通用表**              | B⁺ 树主键索引 + 若干 B⁺ 树二级索引   |
| **等值查询极多、无范围查询** | 可考虑哈希索引（如在 Memory 引擎中） |
| **缓存层（如 Redis）**       | 哈希表是核心数据结构                 |
| **分布式 KV 存储**           | 一致性哈希等哈希变种广泛应用         |
| **数据仓库**                 | B⁺ 树或列存索引，范围聚合查询为主    |
