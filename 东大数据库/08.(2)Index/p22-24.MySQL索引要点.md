### 第 1 页：主键选择案例分析

这一页通过一个具体的建表语句和 `DESC` 命令的输出，演示了 MySQL InnoDB 在**没有显式定义 PRIMARY KEY** 时，如何按照内部规则自动选择主键。

**1. 建表语句分析**

```sql
CREATE TABLE Z
(a INT NOT NULL,
 b INT NULL,
 c INT NOT NULL,
 d INT NOT NULL,
 UNIQUE KEY (b),
 UNIQUE KEY (d),
 UNIQUE KEY (c)
);
```

- **表结构特点**：
    - 列 `a`：`NOT NULL`，没有任何索引约束。
    - 列 `b`：允许 `NULL`，定义了 `UNIQUE KEY`。
    - 列 `c`：`NOT NULL`，定义了 `UNIQUE KEY`。
    - 列 `d`：`NOT NULL`，定义了 `UNIQUE KEY`。
- **关键点**：表中**没有**显式定义 `PRIMARY KEY`。

**2. `DESC Z` 输出分析**

```
+-------+---------+------+-----+---------+-------+
| Field | Type    | Null | Key | Default | Extra |
+-------+---------+------+-----+---------+-------+
| a     | int(11) | NO   |     | NULL    |       |
| b     | int(11) | YES  | UNI | NULL    |       |
| c     | int(11) | NO   | UNI | NULL    |       |
| d     | int(11) | NO   | PRI | NULL    |       |
+-------+---------+------+-----+---------+-------+
```

- **`Key` 列的含义**：
    - `PRI`：表示该列是**主键**的一部分。
    - `UNI`：表示该列是**唯一索引**的一部分。
    - `MUL`：表示该列是**非唯一索引**的第一列（本表中未出现）。
    - 空白：表示该列不是任何索引的成员，或者只是多列索引中的非第一列。

**3. 为什么 `d` 成为了主键（`PRI`）？**

- **回顾 InnoDB 主键选择规则**（上一组 PPT 的知识点）：
    1. 优先使用显式定义的 `PRIMARY KEY`。
    2. 若没有，则选择**第一个**定义为 `NOT NULL` 的 `UNIQUE KEY`。
    3. 若还没有，则自动创建 6 字节的隐藏主键。

- **规则应用过程**：
    - 表中没有显式 `PRIMARY KEY` → 进入规则 2。
    - 候选唯一索引有：`b`（允许 NULL，不符合）、`c`（NOT NULL，符合）、`d`（NOT NULL，符合）。
    - InnoDB 按照索引定义的**先后顺序**遍历：
        - 第一个唯一索引是 `b`，但它允许 `NULL`，**不满足** `NOT NULL` 条件，跳过。
        - 第二个唯一索引是 `d`，它是 `NOT NULL`，**满足条件** → **选择 `d` 作为聚集索引（主键）**。
    - 注意：虽然 `c` 也是 `NOT NULL` 的唯一索引，但因为 `d` 先被定义，所以 `c` 落选了，只作为普通唯一索引存在。

**4. 总结要点**

- **`UNIQUE KEY` 允许有多个 `NULL` 值**（在 MySQL 中），因此含有 `NULL` 的唯一索引不能作为主键，因为主键隐含 `NOT NULL` 约束。
- 定义索引的顺序会影响 InnoDB 在没有主键时的**自动选择结果**。
- 使用 `DESC` 命令可以快速查看 MySQL 自动分配的主键情况。

### 第 2 页：MySQL 中 KEY 的双重含义

这一页从概念层面辨析了 MySQL 中 **KEY** 这个关键字的特殊地位。

**1. KEY 的两层意义**

> Key 是数据库的物理结构，它包含两层意义和作用：一是约束，二是索引。

- **知识点**：在标准 SQL 中，**Constraint（约束）** 和 **Index（索引）** 是两种不同的数据库对象。约束保证数据完整性，索引提升查询性能。
- **MySQL 的特色**：MySQL 将这两者**融合**在 `KEY` 语法中。定义一个 `KEY`，既创建了一个约束，也**自动**创建了一个同名的底层索引结构来支持该约束。

**2. 三种具体 KEY 的解析**

| KEY 类型        | 约束作用                                     | 索引作用                                         |
| :-------------- | :------------------------------------------- | :----------------------------------------------- |
| **PRIMARY KEY** | 保证实体完整性：非空且唯一标识一行。         | 自动创建**主键索引**（在 InnoDB 中即聚集索引）。 |
| **UNIQUE KEY**  | 保证候选键的唯一性（允许一个 NULL）。        | 自动创建**唯一索引**（B+ 树结构）。              |
| **FOREIGN KEY** | 保证引用完整性：子表值必须在父表主键中存在。 | **自动创建普通索引**（在 MySQL 中）。            |

**3. 关于 FOREIGN KEY 索引的特别说明**

- **自动建索引**：在 MySQL 中，当你定义一个外键约束时，MySQL 会**自动**在外键列上创建一个**非唯一索引**（索引名通常与外键约束名相同）。
- **性能原因**：当父表记录被删除或更新时，数据库需要快速检查子表中是否有匹配的记录。如果没有这个索引，每次级联操作都需要**全表扫描**子表，性能极差。
- **与其他数据库的区别**：Oracle 等数据库定义外键时**不会**自动创建索引，需要 DBA 手动创建，否则容易引发死锁和性能瓶颈。这是 MySQL 在易用性上的一种设计选择。

**4. 总结要点**

> MySQL 的 KEY 是同时具有 constraint 和 index 的意义。

- 理解这一点，有助于区分 `ALTER TABLE ADD CONSTRAINT` 和 `CREATE INDEX` 在某些场景下的等效性，以及理解为什么删除主键可能会导致表重建。

### 第 3 页：`SHOW INDEX` 命令详解

这一页介绍了如何查看表中的索引元数据，并对 `SHOW INDEX` 命令输出的各个字段进行了详细注释。

**1. 命令格式**

```sql
SHOW INDEX FROM student;
```

**2. 示例输出**

| Table   | Non_unique | Key_name | Seq_in_index | Column_name | Collation | Cardinality | Sub_part | Packed | Null | Index_type | Comment |
| :------ | :--------- | :------- | :----------- | :---------- | :-------- | :---------- | :------- | :----- | :--- | :--------- | :------ |
| student | 0          | PRIMARY  | 1            | Sno         | A         | 0           | NULL     | NULL   |      | BTREE      |         |
| student | 0          | Sname    | 1            | Sname       | A         | 0           | NULL     | NULL   | YES  | BTREE      |         |

**3. 关键字段详解（按 PPT 顺序补充深度解读）**

- **`Table`**：索引所属的表名。
- **`Non_unique`**：
    - `0` = 唯一索引（Unique Index）。
    - `1` = 非唯一索引。
- **`Key_name`**：索引的名称。
    - **重要识别技巧**：如果多行数据的 `Key_name` **相同**，说明这是一个**联合索引**。
- **`Seq_in_index`**：索引中的列序列号，从 1 开始。
    - **应用**：对于联合索引 `(A, B, C)`，`A` 的序列号为 1，`B` 为 2，`C` 为 3。这个顺序直接决定了**最左前缀匹配原则**的生效范围。
- **`Column_name`**：列名。
- **`Collation`**：排序方式。`A` 表示升序（Ascending），`NULL` 表示未排序（通常见于哈希索引）。
- **`Cardinality`**：基数估计值。
    - **含义**：索引中**不重复值**的数量的**估算值**。
    - **用途**：MySQL 优化器根据这个值判断索引的**选择性**。值越大（越接近总行数），说明索引区分度越好，越倾向于使用该索引；值很小（如性别列只有 2），优化器可能放弃索引直接全表扫描。
    - **注意**：这个值是近似值，可通过 `ANALYZE TABLE` 命令更新。
- **`Sub_part`**：前缀索引长度。
    - 如果只对字符串的前 N 个字符建索引，这里显示 N。如果对整个列建索引，则为 `NULL`。
    - 常用于长字符串字段（如 `TEXT`、长 `VARCHAR`）以节省索引空间。
- **`Packed`**：关键字压缩方式，`NULL` 表示未压缩。
- **`Null`**：该列是否允许 `NULL` 值。`YES` 表示允许。
- **`Index_type`**：索引的底层数据结构。
    - `BTREE`：B+ 树索引（InnoDB、MyISAM 默认）。
    - `HASH`：哈希索引（仅 Memory/Heap 引擎支持）。
    - `FULLTEXT`：全文索引。
    - `RTREE`：空间索引（用于地理数据）。
