### 第 1 页：多条件查询的索引困境与解决思路

这一页通过一个具体的 SQL 查询，揭示了单列索引在处理多条件组合查询时的局限性，并给出了两种可能的优化思路。

**1. 查询示例**

```sql
SELECT stu_id FROM scores WHERE maths IN [80,90] AND chem IN [80,90];
```

- 查询目标：找出数学成绩在 80~90 分**且**化学成绩也在 80~90 分的学生。
- 这是一个典型的**多属性等值/范围组合查询**。

**2. 仅有 `maths` 列索引的执行过程与问题**

- **执行计划**：
    1. 使用 `maths` 索引检索所有满足 `maths IN [80,90]` 的元组（记录指针）。
    2. 对每一个检索到的元组，**回表**读取完整记录，检查 `chem` 字段是否也在 80~90 之间。
- **问题所在**：
    - 满足数学条件的学生中，可能只有**很少一部分**同时满足化学条件。
    - 但数据库必须**逐一检查**所有满足数学条件的记录，产生大量**无效回表**，I/O 代价高昂。

**3. 复合索引 `(maths, chem)` 的局限**

- 如果建立一个 B⁺ 树复合索引，索引键为 `<maths, chem>`。
- **适用查询**：
    - `WHERE maths = 85 AND chem = 85`（等值查询）→ 高效。
    - `WHERE maths = 85 AND chem IN [80,90]`（第一列等值，第二列范围）→ 较高效，先定位 maths=85，再在 chem 范围扫描。
- **本例查询的困难**：
    - `WHERE maths IN [80,90] AND chem IN [80,90]`。
    - 第一列就是**范围查询**，第二列也是范围查询。
    - 复合索引 `<maths, chem>` 在定位 `maths` 范围后，对于每个 maths 值，`chem` 在索引中是有序的，但**不同 maths 值对应的 chem 区间在物理上不连续**，仍需扫描大量索引页。
    - **结论**：复合索引也无法高效处理**多列均为范围查询**的情况。

**4. 双独立索引 + 指针交集**

- 如果在 `maths` 和 `chem` 上分别建立独立索引。
- **优化策略**：
    1. 分别利用两个索引检索：
        - 索引 `maths` 返回满足 `maths IN [80,90]` 的**记录指针集合** \( P_m \)。
        - 索引 `chem` 返回满足 `chem IN [80,90]` 的**记录指针集合** \( P_c \)。
    2. 计算两个指针集合的**交集** \( P = P_m \cap P_c \)。
    3. 仅对交集 \( P \) 中的指针进行**回表**操作，读取完整记录。
- **优势**：
    - 大幅减少了无效回表的数量。
    - 交集操作可在内存中高效完成（如位图索引、哈希交集或排序归并）。
- **本质**：这已经是**多维索引**思想的雏形——通过组合多个单维索引的结果来回答多维查询。

### 第 2 页：网格文件的基本概念

这一页正式引入了一种专门为多维查询设计的索引结构——**网格文件**。

**1. 网格文件的定义与目的**

- **原文**：*Structure used to speed the processing of general multiple search-key queries involving one or more comparison operators.*

- **核心功能**：高效处理涉及**多个搜索键属性**、且可能包含**比较运算符**（`=`、`>`、`<`、`BETWEEN` 等）的查询。

**2. 网格文件的核心组件**

| 组件         | 说明                                                         |
| :----------- | :----------------------------------------------------------- |
| **网格数组** | 一个多维数组，维度数 = 搜索键属性个数。每个单元格存储一个指向**桶**的指针。 |
| **线性刻度** | 每个搜索键属性对应一个**线性刻度**，将该属性的值域划分为若干区间。 |

- **工作原理**：
    1. 对于给定的搜索键值，分别在各个属性的线性刻度上定位其所属区间。
    2. 根据各维度的区间索引，确定网格数组中的**单元格**。
    3. 通过该单元格的指针，找到对应的**桶**。
    4. 桶中存储了所有落入该单元格的记录指针（或直接存储记录）。

**3. 多单元格指向同一桶**

- **原文**：*Multiple cells of grid array can point to same bucket.*

- **含义**：多个相邻的网格单元格可能共享同一个桶。
- **原因**：如果某些区域数据稀疏，为每个单元格单独分配一个桶会造成空间浪费。允许多个单元格指向同一桶，可以提高空间利用率。
- **查询时的处理**：即使多个单元格指向同一桶，查询时仍会通过网格定位到该桶，然后可能需要在桶内进行少量筛选。

**4. 查询步骤总结**

1. 在各维度线性刻度上确定键值的区间索引。
2. 用这些索引定位网格数组中的单元格。
3. 沿单元格指针找到桶。
4. 在桶中检索符合条件的记录。

### 第 4 页：网格文件图示详解

这一页用图形化的方式展示了基于上述数据构建的网格文件。

**1. 维度划分（线性刻度）**

- **年龄维度**划分为 3 个区间：
    - `0-40`
    - `40-55`
    - `55+`
- **工资维度**划分为 3 个区间：
    - `0-90K`
    - `90K-225K`
    - `225K+`

- 这样形成了一个 **3 × 3 的网格数组**，共 9 个单元格。

**2. 网格单元格与桶的映射**

| 年龄 \ 工资 | 0-90K            | 90K-225K           | 225K+              |
| :---------- | :--------------- | :----------------- | :----------------- |
| **0-40**    | (25,60)          | 空                 | (30,260), (25,400) |
| **40-55**   | (45,60), (50,75) | (50,100), (50,120) | (45,350), (50,275) |
| **55+**     | 空               | (70,110), (85,140) | (60,260)           |

- 每个非空单元格指向一个**桶**，桶内存放该区域内的所有记录指针（或记录本身）。
- 注意：图中 `0-90K` 和 `40-55` 交叉的单元格有两条记录，它们被存放在同一个桶中。

**3. 查询示例**

- **查询一**：`WHERE age = 50 AND salary = 100`
    1. age=50 落入 `40-55` 区间。
    2. salary=100 落入 `90K-225K` 区间。
    3. 定位到网格单元格 (40-55, 90K-225K)。
    4. 进入对应桶，精确查找。

- **查询二**：`WHERE age BETWEEN 30 AND 50 AND salary BETWEEN 100 AND 300`
    1. age 区间覆盖 `0-40` 和 `40-55`。
    2. salary 区间覆盖 `90K-225K` 和 `225K+`。
    3. 需要访问 2×2 = 4 个网格单元格对应的桶。
    4. 在桶内进一步筛选满足精确条件的记录。

**4. 网格文件的优势与问题**

| 优势                                 | 问题/挑战                                                |
| :----------------------------------- | :------------------------------------------------------- |
| 多维范围查询效率高，直接定位到相关桶 | 维度划分（线性刻度）的设计直接影响性能和数据分布均衡性   |
| 支持任意维度的等值和范围组合查询     | 数据倾斜时，某些桶可能过满，需要溢出处理或动态调整网格   |
| 结构直观，易于实现                   | 维度增加时网格数组呈指数增长（维数灾难），适用于低维场景 |

**5. 网格文件与 B⁺ 树、哈希的对比**

| 对比维度     | B⁺ 树                    | 哈希       | 网格文件         |
| :----------- | :----------------------- | :--------- | :--------------- |
| **单维等值** | 良好                     | 极佳       | 良好             |
| **单维范围** | 极佳                     | 不支持     | 良好             |
| **多维等值** | 需复合索引或索引交集     | 需多维哈希 | 极佳             |
| **多维范围** | 低效（需扫描大量索引页） | 不支持     | 极佳             |
| **空间占用** | 中等                     | 较小       | 较大（网格数组） |
