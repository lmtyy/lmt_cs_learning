### 第 1 页：WHERE 查询慢怎么办

这一页简洁地指出了普通查询变慢的核心症结以及对应的优化手段。

**1. 问题根源：回表导致效率低下**

- 在使用二级索引进行查询时，如果 `SELECT` 子句中包含了索引未能覆盖的列，数据库需要先扫描二级索引叶子节点获取主键值，再拿着主键值回到聚簇索引中查找完整的行记录。这个过程就是**回表**。
- 回表操作通常触发**随机 I/O**，如果查询涉及大量数据行，回表的代价会急剧上升，成为查询变慢的主要原因。

**2. 核心优化手段：覆盖索引**

- **定义**：查询所需的所有列（包括 SELECT 列和 WHERE 条件列）都存在于同一个索引中，数据库仅扫描该索引的 B⁺ 树即可返回结果，无需回表。
- **优势**：
    - 消除了回表带来的随机 I/O。
    - 索引体积通常小于数据文件，扫描的页数更少，缓存命中率更高。
    - 在 `EXPLAIN` 输出的 `Extra` 字段中会显示 `Using index`。

**3. 优化方法一：优化 SQL 语句**

- **减少 SELECT 列数**：避免 `SELECT *`，只查询业务真正需要的列。查询的列越少，越容易命中覆盖索引。
- **改写查询逻辑**：将复杂的子查询改写为 `JOIN`，或将 `OR` 条件改写为 `UNION`，使优化器更容易选择正确的索引。
- **避免索引失效写法**：不在索引列上使用函数或表达式；避免前置模糊查询 `LIKE '%xxx'`；注意隐式类型转换。

**4. 优化方法二：优化联合索引**

- **合理设计复合索引的列顺序**：
    - 将**等值查询**的列放在最前面。
    - 将**范围查询**的列放在后面。
    - 将**需要排序或分组**的列依次排列，以消除 filesort。
- **构建“宽”索引以覆盖高频查询**：
    - 对于执行频率极高、且返回列数较少的查询，可以将 `SELECT` 子句涉及的列也加入到联合索引的末尾，人为构造覆盖索引。
    - **注意权衡**：索引列过多会占用更多磁盘空间，并增加写入（INSERT/UPDATE）时的索引维护开销。

### 第 2 页：COUNT 查询的性能差异

这一页深入讲解了 `COUNT` 函数的执行逻辑，以及 `COUNT(列)`、`COUNT(*)`、`COUNT(1)` 之间的性能区别。

**1. COUNT 函数的基本定义与执行流程**

- **定义**：`COUNT(expression)` 统计的是**表达式结果不为 NULL 的行数**。
- **执行过程**：
    1. 存储引擎根据查询条件检索出结果集（可能使用索引或全表扫描）。
    2. 服务器层（Server Layer）逐行判断 `expression` 是否为 `NULL`，不为 `NULL` 则计数器加 1。
- **关键点**：无论使用何种 COUNT 写法，存储引擎都必须返回足够的信息让服务器层判断 `NULL`。不同写法的差异在于**存储引擎返回的数据量**和**是否能够利用索引**。

**2. COUNT(非索引字段) —— 最慢**

- **原理**：非索引字段无法利用索引进行快速扫描，存储引擎必须执行**全表扫描**，读取所有数据页。
- **代价**：需要将完整的数据行（或至少包含该字段的部分）从磁盘读入内存，I/O 开销最大。
- **示例**：`SELECT COUNT(last_update) FROM inventory;`

**3. COUNT(索引字段) —— 可以使用索引，但仍需判断 NULL**

- **原理**：优化器会选择占用空间最小的**二级索引**进行全索引扫描（而不是全表扫描），因为二级索引的叶子节点体积远小于聚簇索引的数据页。
- **代价**：
    - 虽然扫描索引页的 I/O 较小，但服务器层仍需取出该索引列的值，**逐行判断是否为 NULL**。
    - 即使该列定义为 `NOT NULL`，MySQL 服务器层的代码路径仍包含 NULL 检查（除非优化器特别优化，但一般仍有一定开销）。

**4. COUNT(*) —— 理论最快**

- **原理**：`COUNT(*)` 是 SQL 标准中的特殊语法，表示“统计行的数量”，**不考虑字段值是否为 NULL**。
- **MySQL 的特殊优化**：
    - MySQL 优化器遇到 `COUNT(*)` 时，会优先选择**最小的二级索引**来扫描（如果没有二级索引，则扫描聚簇索引）。
    - 关键在于：服务器层**不需要读取任何具体的列值**来判断 NULL，存储引擎直接返回行的计数信息。在某些引擎（如 MyISAM）中，`COUNT(*)` 甚至可以从表元数据中直接读取，复杂度 O(1)。
    - 在 InnoDB 中，`COUNT(*)` 也避免了逐行判断 NULL 的逻辑，性能优于 `COUNT(列)`。
- **补充说明**：`COUNT(1)` 与 `COUNT(*)` 在 MySQL 中性能完全等价，优化器会将其转换为 `COUNT(*)` 处理。但从语义清晰角度，推荐使用 `COUNT(*)`。

**5. COUNT 性能对比总结**

| 写法                    | 扫描方式             | 是否判断 NULL      | 性能等级 |
| :---------------------- | :------------------- | :----------------- | :------- |
| `COUNT(非索引字段)`     | 全表扫描（聚簇索引） | 是                 | ⭐ 最慢   |
| `COUNT(索引字段)`       | 全二级索引扫描       | 是                 | ⭐⭐ 较快  |
| `COUNT(*)` / `COUNT(1)` | 全最小二级索引扫描   | 否（或有特殊优化） | ⭐⭐⭐ 最快 |

**6. 实战建议**

- 进行表行数统计时，**始终使用 `COUNT(*)`** 或 `COUNT(1)`，不要使用 `COUNT(列名)`，除非你确实需要统计该列非空的行数。
- 如果对 `COUNT(*)` 的实时性能仍有极高要求（例如大屏看板），可考虑使用 **汇总表** 或 **缓存** 来维护近似或精确的行数。
