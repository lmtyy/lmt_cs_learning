### 第 1 页：索引与 NULL 值、短索引（前缀索引）

**1. 索引不会存储 Null 值的列**

- **原话**：*索引不会存储 Null 值的列。只要列中包含 null 值，就不会被包含到索引中。*
- **深度解析**：
    - 在 MySQL 中，`NULL` 值会被视为“未知”，但**它仍然可以被索引**。
    - 澄清：并不是包含 `NULL` 值的记录完全不在索引中，而是**索引条目会存储 `NULL` 值**，只是在查询 `WHERE column IS NULL` 时，**可以使用索引**。
    - **真正的问题在于**：对于 `WHERE column NOT IN (SELECT ...)` 或 `WHERE column <> 'value'`，由于 `NULL` 的存在，会导致**三值逻辑**，优化器可能放弃使用索引，或者产生不符合预期的结果。
    - **核心建议**：尽量将列设置为 `NOT NULL`，并给予默认值。这不仅能避免索引歧义，还能节省存储空间（`NULL` 标志位占用 1 字节），简化查询逻辑。

**2. 使用短索引（前缀索引）**

- **场景**：当需要对 `VARCHAR`、`TEXT`、`BLOB` 等长字符串列建立索引时。
- **问题**：全列索引体积大，导致索引页存放的条目数变少（扇出变小），树的高度增加，且 I/O 代价变高。
- **解决方案**：只截取列值的前几个字符建立索引。
    ```sql
    CREATE INDEX idx_name ON user(name(10));  -- 只索引前10个字符
    ```
- **原理**：
    - 减少索引条目大小，增加单个索引页能容纳的条目数，降低树高。
    - 节省磁盘空间和内存占用（Buffer Pool）。
- **计算公式**（选择前缀长度）：
    \[
    \text{前缀基数} / \text{总行数} \approx \text{接近 1}
    \]
    需要保证前缀的**选择性**仍然足够高，避免大量重复前缀导致索引失效。
- **举例**：身份证号，前 6 位是地区码，重复率高；但如果取前 10-12 位，通常选择性已经很好。

### 第 2 页：排序与索引、LIKE 陷阱、函数表达式失效

**3. 索引列排序**

- **原话**：*通常 where 条件使用索引的话，那么 order by 就无法使用索引。*
- **准确理解**：这并非绝对，而是指**索引的列顺序**与 `ORDER BY` 的顺序不匹配时，会导致额外的文件排序（filesort）。
- **解决之道——覆盖索引与复合索引**：
    - 如果 `WHERE` 和 `ORDER BY` 涉及的列都在同一个复合索引中，且顺序正确，则既能过滤又能排序，性能达到最优。
    - **示例**：
        ```sql
        INDEX idx_a_b (a, b)
        SELECT * FROM t WHERE a = 1 ORDER BY b;  -- 完美使用索引避免排序
        ```
    - **反例**：
        ```sql
        SELECT * FROM t WHERE a > 1 ORDER BY b;  -- a 是范围查询，b 无法利用索引排序（范围查询后的列索引失效）
        ```

**4. LIKE 语句的索引失效问题**

- **规则**：
    - `LIKE 'xxx%'` ✅ **可以使用索引**（范围扫描）。引擎定位到前缀后，向后顺序扫描。
    - `LIKE '%xxx'` ❌ **索引失效**。因为 B+ 树只能根据前缀顺序查找，不知道后缀是什么。
    - `LIKE '%xxx%'` ❌ **索引失效**。全表扫描或全索引扫描。
- **优化手段**：
    - 如果业务必须模糊匹配中间部分，考虑使用**全文索引** 或 **Elasticsearch** 等专用搜索引擎。
    - 或者使用**覆盖索引**，即使 `LIKE '%xxx'` 无法走树搜索，但如果查询列都在索引中，至少能避免回表（Using index）。

**5. 列上使用函数或表达式导致索引失效**

- **示例**：`WHERE age + 1 = 5`
- **原因**：索引存储的是 `age` 列的**原始值**。在列上做运算后，优化器无法直接使用索引树进行二分查找，因为它不知道 `age+1` 的索引值是多少。
- **修正**：`WHERE age = 4`。
- **扩展——隐式类型转换**：
    - `WHERE phone = 13800138000` （phone 是字符串列）
    - 这属于在列上使用了隐式函数 `CAST(phone AS INT)`，同样会导致索引失效！务必保持类型一致：`WHERE phone = '13800138000'`。

### 第 3 页：负向操作符与复合索引最左原则

**6. 不使用 NOT IN、<> 等操作符**

- **原因**：`NOT IN`、`!=`、`<>` 这类**负向查询**，通常无法有效利用 B+ 树的有序结构。因为 B+ 树擅长定位特定值或范围，而“不等于某个值”意味着要扫描几乎所有叶子节点。
- **替代方案**：
    - **`NOT IN` → `NOT EXISTS`**：
        在 MySQL 中，`NOT EXISTS` 在处理子查询时，如果关联列有索引，可能会采用更优的**物化**或**反连接**策略，且能正确处理 `NULL` 值问题。
    - **`a <> 0` → `a < 0 OR a > 0`**：
        这种改写**不一定**能提高性能，优化器通常已经能自动处理。但在极老版本的数据库中，`OR` 展开有时能让优化器分别使用两个索引范围扫描然后合并（Index Merge）。不过目前 MySQL 8.0 对 `<>` 有时也能使用 **Range Scan**（当数据分布极不均匀时）。

**7. 复合索引与最左匹配原则**

- **定义**：对于复合索引 `(A, B, C)`，B+ 树首先按 `A` 排序，`A` 相同时按 `B` 排序，以此类推。
- **生效条件**：
    - 查询条件必须包含**索引最左列** `A`。
    - 如果跳过 `A` 直接查 `B`，无法利用该索引（除非是覆盖索引扫描）。
- **示例表**：
    ```sql
    INDEX idx_abc (a, b, c)
    
    SELECT * FROM t WHERE a = 1 AND b = 2;        -- ✅ 全匹配
    SELECT * FROM t WHERE a = 1 AND b > 2;        -- ✅ a 和 b 用上索引，c 无法用于排序/过滤
    SELECT * FROM t WHERE a = 1 ORDER BY b;       -- ✅ 避免排序
    SELECT * FROM t WHERE b = 2;                  -- ❌ 无 a，索引失效（除非优化器选择全索引扫描）
    ```
- **口诀**：*带头大哥不能死，中间兄弟不能断*。
    - 条件中如果包含 `A` 的等值查询，然后是 `B` 的范围查询，`C` 就无法利用索引进行过滤或排序了。
