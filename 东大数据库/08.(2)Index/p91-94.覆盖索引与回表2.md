好的，这四页 PPT 通过对比 `inventory_1`（无二级索引）和 `inventory`（有复合索引）两张表的实际查询执行计划，直观地展示了**覆盖索引**的工作原理、回表的发生条件以及 InnoDB 二级索引中隐含主键的设计细节。我将按照 PPT 顺序逐页详细讲解。

### 第 1 页：无索引表的全表扫描

这一页展示了在**没有索引**的 `inventory_1` 表上执行查询时的执行计划。

**1. 查询语句**

```sql
SELECT store_id, film_id FROM sakila.inventory_1 WHERE store_id = 1;
```

**2. 执行计划输出**

| id   | select_type | table       | type    | possible_keys | key  | key_len | ref  | rows | Extra           |
| :--- | :---------- | :---------- | :------ | :------------ | :--- | :------ | :--- | :--- | :-------------- |
| 1    | SIMPLE      | inventory_1 | **ALL** | NULL          | NULL | NULL    | NULL | 4581 | **Using where** |

**3. 关键字段解读**

| 字段              | 值            | 含义                                                         |
| :---------------- | :------------ | :----------------------------------------------------------- |
| **type**          | `ALL`         | **全表扫描**，性能最差的一种访问类型。数据库会扫描整张表的每一行数据。 |
| **possible_keys** | `NULL`        | 优化器认为**没有可用的索引**来加速查询。                     |
| **key**           | `NULL`        | 实际未使用任何索引。                                         |
| **rows**          | `4581`        | 预估需要扫描的行数，这里等于表的总行数。                     |
| **Extra**         | `Using where` | 存储引擎返回所有行后，由服务器层通过 `WHERE` 条件进行**过滤**。这意味着即便是不满足条件的行，也会被读取上来。 |

**4. 知识点总结**

- **没有索引的后果**：数据库必须从磁盘读取表的**所有数据页**，然后在内存中逐行检查 `store_id` 是否等于 1。
- **代价**：无论符合条件的行有多少（哪怕只有 1 行），都需要扫描全表 4581 行对应的数据块。
- **适用场景**：仅当表极小，或者需要返回表中大部分行时，全表扫描才可能是合理的。

### 第 2 页：有索引表的覆盖索引扫描

这一页展示了在**有复合索引**的 `inventory` 表上执行相同查询时的执行计划。

**1. 查询语句**

```sql
SELECT store_id, film_id FROM sakila.inventory WHERE store_id = 1;
```

**2. 执行计划输出**

| id   | select_type | table     | type    | possible_keys        | key                  | key_len | ref   | rows | Extra           |
| :--- | :---------- | :-------- | :------ | :------------------- | :------------------- | :------ | :---- | :--- | :-------------- |
| 1    | SIMPLE      | inventory | **ref** | idx_store_id_film_id | idx_store_id_film_id | 1       | const | 2269 | **Using index** |

**3. 关键字段解读**

| 字段              | 值                     | 含义                                                         |
| :---------------- | :--------------------- | :----------------------------------------------------------- |
| **type**          | `ref`                  | 使用**非唯一索引**进行等值匹配，性能远优于 `ALL`。           |
| **possible_keys** | `idx_store_id_film_id` | 优化器识别到该索引可用于查询。                               |
| **key**           | `idx_store_id_film_id` | 实际使用了这个复合索引。                                     |
| **key_len**       | `1`                    | 使用的索引长度为 1 字节（`store_id` 类型为 `TINYINT`）。     |
| **Extra**         | **`Using index`**      | **覆盖索引**的标志。表示查询所需的列全部包含在该索引中，**无需回表**。 |

**4. 为什么能实现覆盖索引？**

- 复合索引 `idx_store_id_film_id (store_id, film_id)` 的 B⁺ 树叶子节点中，存储了 **`(store_id, film_id, inventory_id)`**。
- 本查询的 `SELECT` 子句恰好只需要 `store_id` 和 `film_id`，这些数据直接从索引叶子节点即可获得。
- 数据库只需扫描该二级索引的 B⁺ 树中 `store_id = 1` 的部分，**完全不需要触碰聚簇索引（主键索引）的数据页**。

**5. 性能对比**

| 对比项   | inventory_1（无索引） | inventory（有索引，覆盖）    |
| :------- | :-------------------- | :--------------------------- |
| 扫描行数 | 4581（全表）          | 2269（仅符合条件的索引条目） |
| I/O 类型 | 读取所有数据页        | 仅读取索引页                 |
| 是否回表 | 不适用                | 否（`Using index`）          |

### 第 3 页：查询包含主键列，仍为覆盖索引

这一页展示了当 `SELECT` 子句中**增加主键列** `inventory_id` 时，为什么仍然不需要回表。

**1. 查询语句**

```sql
SELECT inventory_id, store_id, film_id FROM inventory WHERE store_id = 1;
```

**2. 执行计划输出**

| id   | select_type | table     | type | possible_keys        | key                  |
| :--- | :---------- | :-------- | :--- | :------------------- | :------------------- |
| 1    | SIMPLE      | inventory | ref  | idx_store_id_film_id | idx_store_id_film_id |

**3. 核心知识点：二级索引叶子节点隐含主键**

- **原文**：*`idx_store_id_film_id` 索引不仅包含 `store_id` 和 `film_id` 两个联合索引字段，还包含主键。*

- **原理**：
    - 在 InnoDB 存储引擎中，**所有二级索引的叶子节点，除了存储索引列的值外，都会自动附加主键值**。
    - 这是因为二级索引需要通过主键值来定位聚簇索引中的完整行记录（即回表的依据）。
    - 因此，复合索引 `(store_id, film_id)` 的叶子节点实际存储的是 **`(store_id, film_id, inventory_id)`**。

- **结论**：因为主键 `inventory_id` 天然存在于该二级索引的叶子节点中，所以即使 `SELECT` 了主键列，仍然可以**直接从索引中获取**，不需要回表。

**4. Extra 字段的细节**

- 这个执行计划中 `Extra` 列同样会显示 `Using index`（PPT 中限于篇幅未完全列出，但原理一致）。
- 这进一步证明了：**在 InnoDB 中，任何二级索引都是“索引列 + 主键列”的复合索引**。

### 第 4 页：查询非索引列导致回表

这一页展示了当 `SELECT` 子句中包含了**不在任何索引中**的列 `last_update` 时，执行计划的变化。

**1. 查询语句**

```sql
SELECT inventory_id, store_id, film_id, last_update FROM inventory WHERE store_id = 1;
```

**2. 执行计划输出**

| id   | select_type | table     | type | possible_keys        | key                  |
| :--- | :---------- | :-------- | :--- | :------------------- | :------------------- |
| 1    | SIMPLE      | inventory | ref  | idx_store_id_film_id | idx_store_id_film_id |

**3. 关键变化：Extra 字段**

- 此时 `Extra` 列**不再**显示 `Using index`，而是可能显示 `NULL` 或 `Using where`。
- **含义**：虽然查询仍然使用了 `idx_store_id_film_id` 索引来快速定位符合条件的行，但因为 `last_update` 列**不在该索引的叶子节点中**，数据库必须执行**回表**操作。

**4. 回表过程**

1. 在 `idx_store_id_film_id` 索引中扫描 `store_id = 1` 的所有索引条目。
2. 从每个索引条目中获取对应的主键值 `inventory_id`。
3. 拿着主键值，到**聚簇索引**中查找完整的行记录。
4. 从完整行记录中提取 `last_update` 列的值。
5. 组合结果返回。

**5. 性能影响**

- 回表通常意味着**随机 I/O**：因为二级索引的顺序与聚簇索引中数据的物理顺序无关，每一条符合条件的记录都可能位于不同的数据页上。
- 如果 `store_id = 1` 的记录有 2269 行，最坏情况下可能产生 **2269 次额外的随机磁盘读取**，性能显著下降。

**6. 如何优化？**

- **方案一（覆盖索引）**：如果业务频繁查询 `last_update`，可考虑将 `last_update` 也加入复合索引，例如 `(store_id, film_id, last_update)`。
- **方案二（延迟关联）**：先通过覆盖索引获取主键列表，再批量回表，减少随机 I/O 次数（如使用 `IN` 子查询）。
- **方案三**：接受回表，但在内存足够大、数据页已缓存的情况下，回表代价会降低。

| PPT 页码    | 核心主题                       | 索引使用情况           | Extra 标志       | 是否回表         |
| :---------- | :----------------------------- | :--------------------- | :--------------- | :--------------- |
| **第 1 页** | 无索引表的全表扫描             | 无                     | `Using where`    | 不适用（全表扫） |
| **第 2 页** | 覆盖索引：所需列全在索引中     | `idx_store_id_film_id` | `Using index`    | **否**           |
| **第 3 页** | 覆盖索引：主键隐含在二级索引中 | `idx_store_id_film_id` | `Using index`    | **否**           |
| **第 4 页** | 非覆盖索引：需要额外的列       | `idx_store_id_film_id` | 无 `Using index` | **是**           |

这四页 PPT 通过真实的 `EXPLAIN` 输出，清晰对比了覆盖索引与回表查询的差异。核心结论是：

- 查看 `Extra` 列是否出现 **`Using index`**，是判断覆盖索引是否生效的直接依据。
- InnoDB 二级索引自动包含主键列，这一设计使得很多看似需要回表的查询（如 `SELECT id FROM t WHERE indexed_col = ?`）实际仍是覆盖索引。
- 回表操作是二级索引查询的主要性能瓶颈，在设计索引时应尽量让高频查询实现覆盖索引。
