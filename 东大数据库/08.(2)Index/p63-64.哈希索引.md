### 第 1 页：哈希索引概述

这一页介绍了哈希索引的基本定义及其与哈希文件组织的区别。

**1. 哈希技术的双重用途**

- **原文**：*Hashing can be used not only for file organization, but also for index-structure creation.*

- **解读**：
    - **文件组织**：指数据文件本身按照哈希值进行物理存储（如哈希文件组织）。在这种组织中，记录的实际存放位置由其键值的哈希函数决定。
    - **索引结构创建**：指在现有数据文件（可能是堆文件或按其他键聚集的文件）之上，额外建立一个哈希索引结构来加速查找。

- **两者的区别**：
    - 哈希文件组织：数据本身就是按哈希分布的。
    - 哈希索引：数据可以按任意方式存储，哈希结构只是一个独立的索引，存储 `<搜索键, 记录指针>` 的映射。

**2. 哈希索引的定义**

- **原文**：*A hash index organizes the search keys, with their associated record pointers, into a hash file structure.*

- **核心组件**：
    - **搜索键**：要建立索引的列值。
    - **记录指针**：指向实际数据记录的物理地址。
    - **哈希文件结构**：将上述 `<键, 指针>` 对组织成桶（Bucket）的数组。

- **工作原理**：
    1. 给定一个搜索键值 \( K \)。
    2. 计算哈希函数 \( h(K) \)，得到桶编号。
    3. 在对应的桶中查找匹配的键值条目。
    4. 获取记录指针，访问数据文件中的实际记录。

**3. 图示说明**

- PPT 说明接下来的图展示了一个基于 `account` 文件的**辅助哈希索引**，搜索键为 `account-number`。
- **辅助索引**意味着数据文件本身不是按账户号哈希组织的，哈希索引是独立建立的。

### 第 2 页：哈希索引实例详解

这一页通过一个具体的 `account` 表实例，展示了哈希索引的物理结构。

**1. 哈希索引的整体结构**

PPT 中的图示分为左右两部分：

- **左侧：哈希桶目录**
    - 包含 4 个桶：bucket 0、bucket 1、bucket 2、bucket 3。
    - 每个桶中存放若干**索引条目**，每个条目包含 `账户号` 和指向数据记录的**指针**。

- **右侧：数据文件**
    - 存储 `account` 表的实际记录，包含字段：`account-number`、`branch-name`、`balance`。
    - 记录物理上**不按账户号排序**，是典型的堆文件或按其他键聚集的文件。

**2. 哈希桶内容分析**

| 桶编号       | 包含的索引条目（账户号） | 对应的数据记录                                               |
| :----------- | :----------------------- | :----------------------------------------------------------- |
| **bucket 0** | A-215, A-305             | A-215 (Mianus, 700), A-305 (Round Hill, 350)                 |
| **bucket 1** | A-217, A-101, A-110      | A-217 (Brighton, 750), A-101 (Downtown, 500), A-110 (Downtown, 600) |
| **bucket 2** | A-215, A-102, A-201      | A-215 (Mianus, 700), A-102 (Perryridge, 400), A-201 (Perryridge, 900) |
| **bucket 3** | A-218, A-202, A-305      | A-218 (Perryridge, 700), A-202 (Redwood, 700), A-305 (Round Hill, 350) |

**3. 关键观察点**

- **桶内条目是键值和指针的组合**：例如 bucket 1 中的 `A-217` 条目包含一个指针，指向右侧数据文件中 `(A-217, Brighton, 750)` 这条记录。
- **不同桶可能包含指向同一记录的指针吗？** 不，但不同桶可能包含相同账户号吗？在标准哈希索引中，一个键值只应出现在一个桶中。图示中 `A-215` 同时出现在 bucket 0 和 bucket 2，这可能是：
    - 图示的简化表示（实际应只有一个）。
    - 或者是由于哈希冲突，同一个桶中链接着多个条目。

- **数据文件中的孤立标识**：PPT 右侧底部列出了 `A-201`、`A-202`、`A-218`、`A-210`、`A-205`、`A-220`，这些是数据记录的主键标识，示意它们物理上分散存储，通过哈希索引中的指针被引用。

**4. 哈希索引的查找过程示例**

假设要查找账户 `A-217` 的信息：

1. **计算哈希值**：对 `A-217` 应用哈希函数，假设计算结果为 1。
2. **定位桶**：访问 bucket 1。
3. **桶内搜索**：在 bucket 1 的条目列表中查找键值 `A-217`。
4. **获取指针**：找到匹配条目后，获取其存储的记录指针。
5. **访问数据**：根据指针读取数据文件中 `(A-217, Brighton, 750)` 这条记录。

**5. 哈希索引的性能特征**

| 操作类型     | 性能                                       |
| :----------- | :----------------------------------------- |
| **等值查询** | 极高，通常 1~2 次 I/O（桶可能跨页）        |
| **范围查询** | 极差，哈希索引不支持顺序扫描，必须全表扫描 |
| **插入**     | 快，计算哈希值后插入对应桶                 |
| **删除**     | 快，从桶中移除条目即可                     |

**6. 哈希索引与 B⁺ 树索引的对比**

| 对比维度         | 哈希索引                             | B⁺ 树索引                    |
| :--------------- | :----------------------------------- | :--------------------------- |
| **等值查询**     | O(1)，极快                           | O(log N)，稍慢               |
| **范围查询**     | 不支持                               | 极快（叶子链表）             |
| **排序**         | 不支持                               | 支持                         |
| **最左前缀匹配** | 不支持                               | 支持                         |
| **空间占用**     | 较小                                 | 较大（需维护树结构）         |
| **动态增长**     | 需处理桶溢出和重哈希                 | 自动分裂合并，平衡性好       |
| **典型应用**     | 等值查询为主的场景（如缓存、内存表） | 通用场景，关系数据库默认索引 |

**7. 为什么关系数据库默认不用哈希索引？**

- 关系数据库需要支持丰富查询：`>`、`<`、`BETWEEN`、`ORDER BY`、`GROUP BY`，这些哈希索引都无法高效支持。
- 哈希索引在面临大量插入删除时，可能因桶溢出链过长或频繁重哈希导致性能波动。
- **例外**：MySQL 的 Memory 引擎支持哈希索引；InnoDB 也支持**自适应哈希索引**，在特定条件下自动为热点数据页构建内存哈希索引以加速等值查询。
