## 第一页：辅助索引的定义

### PPT原文

> **Secondary index** (also called **non-clustering index**): an index whose search key specifies an order **different from** the sequential order of the file.

### 核心定义解读

- **中文名称：** 辅助索引 / 非聚集索引
- **关键特征：** 索引的排序顺序 ≠ 数据文件的实际物理存储顺序

### 与主索引（聚集索引）的对比

| 对比维度               | 主索引 / 聚集索引                       | 辅助索引 / 非聚集索引                  |
| :--------------------- | :-------------------------------------- | :------------------------------------- |
| **索引顺序与数据顺序** | **一致**                                | **不同**                               |
| **一个表可以有**       | **最多 1 个**（数据只能有一种物理排序） | **多个**                               |
| **类比**               | 字典正文本身（按拼音排序）              | 字典后面的**部首检字表**或**笔画索引** |
| **指针指向**           | 直接指向数据页中的具体记录              | 指向数据记录的物理地址或主键值         |

### 类比理解

- **主索引**就像书的**目录**：目录的章节顺序和书里内容的实际排列顺序完全一样。
- **辅助索引**就像书后面的**关键词索引**：关键词按 ABC 排序，但书里这些关键词出现的页码是**散落各处**的，第 50 页有、第 200 页也有。你查到关键词后，要跳到完全不同的页码去读内容。

## 第二页：辅助索引的结构图示

这一页 PPT 通过一个具体的例子，展示了辅助索引在物理上是如何组织的。

### 图表解读

**左侧：辅助索引文件（建立在 `balance` 字段上）**

PPT 显示了一个按 `balance`（余额）排序的索引结构：

| 搜索键 (balance) | 指针 (指向桶) |
| :--------------- | :------------ |
| 350              | →             |
| 400              | →             |
| 500              | →             |
| 600              | →             |
| 700              | →             |
| 750              | →             |
| 900              | →             |

**右侧：主数据文件（按 `branch_name` 聚集存储）**

| Account | Branch     | Balance |
| :------ | :--------- | :------ |
| A-101   | Downtown   | 500     |
| A-217   | Brighton   | 750     |
| A-110   | Downtown   | 600     |
| A-215   | Mianus     | 700     |
| A-102   | Perryridge | 400     |
| A-201   | Perryridge | 900     |
| A-218   | Perryridge | 700     |
| A-222   | Redwood    | 700     |
| A-305   | Round Hill | 350     |

### 关键观察：索引顺序 ≠ 数据顺序

1. **索引顺序（按 balance）：** 350 → 400 → 500 → 600 → 700 → 750 → 900（**递增排列**）
2. **数据顺序（按 branch_name 聚集）：** Brighton → Downtown → Downtown → Mianus → Perryridge → Perryridge → Perryridge → Redwood → Round Hill
3. **观察 balance 在数据文件中的分布：**
   - `balance = 700` 的记录**散落在三处**：Mianus 分行（A-215）、Perryridge 分行（A-218）、Redwood 分行（A-222）。
   - 它们在数据文件中**不连续**，中间夹杂着其他余额的记录。

### 核心机制：“桶”结构

PPT 下方的注释是关键：

> We can have a secondary index with an index record for each search-key value; index record points to a **bucket** that contains pointers to all the actual records with that particular search-key value.

翻译与解读：

- 辅助索引中，**每个搜索键值**对应一个索引条目。
- 这个索引条目**不直接指向数据记录**，而是指向一个**桶**。
- 这个**桶**里面存储了**多个指针**，每个指针指向一条具有该搜索键值的实际数据记录。

### 图示化的桶结构（以 `balance = 700` 为例）

```
辅助索引文件:
┌──────┬───────────┐
│ 700  │ 指针 ────┐│
└──────┴───────────┘│
                    ▼
             ┌─────────────┐
             │   桶 (Bucket) │
             │ ┌───────────┐ │
             │ │ 指向 A-215 │ │  ← Mianus 分行的 700
             │ │ 指向 A-218 │ │  ← Perryridge 分行的 700
             │ │ 指向 A-222 │ │  ← Redwood 分行的 700
             │ └───────────┘ │
             └─────────────┘
                    │
                    ▼
            数据文件中散落的三条记录
```

### 为什么需要“桶”这种间接结构？

因为辅助索引的搜索键值与数据的物理顺序不一致。多个具有相同键值的记录可能分布在数据文件的不同位置。如果不用桶来**集中管理这些散落的指针**，索引本身就需要为**每一行数据**都建一个索引条目，会导致：

1. 索引文件急剧膨胀
2. 查找时需要在索引中扫描大量重复键值

通过桶结构，索引文件只存储**唯一的键值**，将“一对多”的映射关系放到桶中处理，结构更紧凑、查找更高效。

### 查询执行过程示例

假设执行 SQL：`SELECT * FROM account WHERE balance = 700;`

1. **访问辅助索引：** 在左侧索引文件中二分查找 `balance = 700`。
2. **找到桶地址：** 索引条目给出桶的指针。
3. **遍历桶：** 桶中有三个指针，分别指向 A-215、A-218、A-222。
4. **回表读取数据：** 根据这三个指针，分别跳转到主数据文件的三个不同位置，读取完整记录。
5. **返回结果：** 三条记录。

### 主索引 vs. 辅助索引 总结对比

| 对比项                 | 主索引（聚集索引）             | 辅助索引（非聚集索引）       |
| :--------------------- | :----------------------------- | :--------------------------- |
| **索引顺序与数据顺序** | 一致                           | 不一致                       |
| **每表数量限制**       | 1 个                           | 多个                         |
| **叶子节点存储内容**   | 完整的行数据                   | 指向数据行的指针（或主键值） |
| **查询效率**           | 极高（顺序读）                 | 较高（需回表/随机读）        |
| **适用场景**           | 范围查询、排序、频繁按此列扫描 | 等值查询、非主键列的快速查找 |
| **空间开销**           | 较小                           | 较大（需额外存储指针结构）   |
