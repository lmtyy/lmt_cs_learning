## 第一页：稠密索引的定义

### PPT原文

> **Dense index** — Index record appears for **every search-key value** in the file.

### 核心定义解读

- **中文名称：** 稠密索引
- **关键特征：** 文件中出现的**每一个**搜索键值，在索引文件中都**有一条对应的索引记录**。

### 通俗理解

- **稠密索引**就像一本书**末尾的术语索引**：书中出现的**每一个术语**，无论出现多少次，索引里至少有一条条目指向它出现的位置。
- **对比稀疏索引**：稀疏索引只给部分键值建索引，就像字典侧面的**字母标签**，只标了 A、B、C 章节的起始页，中间的条目需要顺序扫描。

### PPT注释解读

> (in this example we have clustering index)

这句话说明：第一页所展示的例子中，不仅索引是稠密的，而且由于数据文件**按照搜索键排序**，这个索引同时也是**主索引（聚集索引）**。

## 第二页：稠密索引示例一（数据按索引字段排序）

这一页展示了一个**理想情况**下的稠密索引结构。

### 图表结构分析

**左侧：稠密索引文件（建立在 `branch_name` 上）**

| 搜索键 (branch_name) | 指针 |
| :------------------- | :--- |
| Brighton             | →    |
| Downtown             | →    |
| Mianus               | →    |
| Perryridge           | →    |
| Redwood              | →    |
| Round Hill           | →    |

**右侧：主数据文件（按 `branch_name` 排序存储）**

| Branch     | Account | Balance |
| :--------- | :------ | :------ |
| Brighton   | A-217   | 750     |
| Downtown   | A-101   | 500     |
| Downtown   | A-110   | 600     |
| Mianus     | A-215   | 700     |
| Perryridge | A-102   | 400     |
| Perryridge | A-201   | 900     |
| Perryridge | A-218   | 700     |
| Redwood    | A-222   | 700     |
| Round Hill | A-305   | 350     |

### 关键观察

1. **索引条目数 = 数据文件中不同搜索键的数量**
   - 数据中有 9 条记录，但只有 6 个不同的分行名。
   - 索引文件有 **6 条记录**，每个分行名一条。
   - 这符合“每个搜索键值都有索引记录”的定义。

2. **数据按索引字段排序**
   - 数据文件严格按照 `branch_name` 的字母顺序存储（Brighton → Downtown → ... → Round Hill）。
   - 因此，这个索引同时也是**主索引（聚集索引）**。

3. **指针指向每个键值的第一条记录**
   - 例如 `Downtown` 的指针指向 A-101（第一条 Downtown 记录）。
   - 要找到所有 Downtown 记录，从 A-101 开始顺序扫描，直到遇到下一个不同的键值（Mianus）。

### 稠密 + 主索引的优势

- 等值查询 `WHERE branch_name = 'Perryridge'`：直接在索引中找到 Perryridge 条目，指针跳转到 A-102，然后顺序读取后续两条记录即可。
- 范围查询 `WHERE branch_name BETWEEN 'Mianus' AND 'Redwood'`：索引中找到 Mianus 的起点，顺序扫描数据文件，直到超出 Redwood 的范围。
- 效率极高，因为数据和索引顺序一致，磁盘顺序读性能最佳。

## 第三页：稠密索引示例二（数据未排序 + 索引字段不唯一）

这一页的场景与第二页**完全不同**，展示了一个更复杂、更实际的情况。

### 图表文字重点

PPT 上的注释明确指出了两个关键变化：

> - **引入指针桶处理非候选索引的多记录情况**
> - **主文件未按索引字段排序且索引字段不是候选键**

### 图表结构分析

**左侧：稠密索引文件（建立在 `branch_name` 上）**

| 搜索键 (branch_name) | 指针 (指向桶) |
| :------------------- | :------------ |
| Brighton             | →             |
| Downtown             | →             |
| Mianus               | →             |
| Perryridge           | →             |
| Redwood              | →             |
| Round Hill           | →             |

**右侧：主数据文件（未按 `branch_name` 排序！）**

| Branch     | Account | Balance |
| :--------- | :------ | :------ |
| Brighton   | A-217   | 750     |
| Downtown   | A-101   | 500     |
| Mianus     | A-215   | 700     |
| Perryridge | A-102   | 400     |
| Redwood    | A-222   | 700     |
| Round Hill | A-218   | 700     |
| （空）     | A-305   | 350     |
| （空）     | A-110   | 600     |
| （空）     | A-201   | 900     |

### 关键差异对比

| 对比维度                   | 第二页示例             | 第三页示例                 |
| :------------------------- | :--------------------- | :------------------------- |
| **数据是否按索引字段排序** | **是**（聚集索引）     | **否**（非聚集索引）       |
| **索引类型**               | 主索引（聚集）         | 辅助索引（非聚集）         |
| **相同键值记录的存储**     | 物理连续存放           | **散落在数据文件各处**     |
| **索引指针指向**           | 直接指向第一条数据记录 | 指向**桶**，桶中存多个指针 |

### 桶结构的作用

因为数据**没有**按 `branch_name` 排序，同一个分行的记录可能出现在数据文件的**任意位置**。例如：

- Downtown 的两条记录分别在位置 2（A-101）和位置 8（A-110）。
- Perryridge 的两条记录分别在位置 4（A-102）和位置 9（A-201）。

如果稠密索引的每个条目只存一个指针，那么：
- 方案一：只为每个键值存第一条记录的指针 → **会丢失其他同键值记录**。
- 方案二：为每一条数据记录都建一个索引条目 → 索引文件会膨胀，且搜索键大量重复。

**引入桶结构**完美解决了这个问题：

```
辅助索引文件:
┌────────────┬───────────┐
│ Downtown   │ 指针 ────┐│
└────────────┴───────────┘│
                          ▼
                   ┌─────────────┐
                   │ 桶 (Bucket)  │
                   │ ┌───────────┐ │
                   │ │ 指向 A-101 │ │
                   │ │ 指向 A-110 │ │
                   │ └───────────┘ │
                   └─────────────┘
```

- **索引层面**：仍然只有 6 个条目（每个不同的分行名一条）→ 保持稠密索引的简洁性。
- **桶层面**：处理了“一对多”的映射关系。

### 查询执行过程示例（第三页场景）

执行 `SELECT * FROM account WHERE branch_name = 'Downtown';`

1. 在左侧稠密索引中二分查找 `Downtown`。
2. 索引条目指向一个**桶**。
3. 桶中包含两个指针：A-101 和 A-110。
4. 根据指针分别跳转到数据文件的位置 2 和位置 8，读取完整记录。
5. 返回两条结果。

### 稠密索引 vs. 稀疏索引总结

| 对比项           | 稠密索引                         | 稀疏索引                             |
| :--------------- | :------------------------------- | :----------------------------------- |
| **索引条目数**   | 等于文件中**不同搜索键值的数量** | 少于数据记录数（仅为部分键值建索引） |
| **索引大小**     | 较大                             | 较小                                 |
| **查找速度**     | 更快（直接定位到具体记录）       | 较慢（定位后可能需要扫描数据块内部） |
| **适用场景**     | 辅助索引通常必须是稠密的         | 主索引（数据已排序）可以用稀疏索引   |
| **数据文件要求** | 无强制排序要求                   | 数据必须按搜索键**排序存储**         |

### 三页 PPT 知识脉络总结

1. **第一页**：给出稠密索引的**定义**——每个搜索键值都有索引记录。
2. **第二页**：展示稠密索引的**理想场景**——数据恰好按索引字段排序，稠密索引同时也是主索引，无需桶结构。
3. **第三页**：展示稠密索引的**一般场景**——数据未排序，索引字段不唯一，必须引入**桶结构**来处理同一键值的多条散落记录。

这三页通过对比，完整展示了稠密索引在不同数据组织方式下的实现机制。
