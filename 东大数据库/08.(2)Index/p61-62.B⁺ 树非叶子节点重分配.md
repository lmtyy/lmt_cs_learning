### 第 1 页：非叶子节点重分配示例——初始状态

这一页提出了一个假设的初始树结构，为后续的重分配操作搭建舞台。

**1. PPT 中的引导性问题**

> *Tree is shown below during deletion of 24\*. (What could be a possible initial tree?)*

- 含义：在删除 `24*` 的过程中，树呈现如下结构。请思考：什么样的初始树在删除 `24*` 后会变成这个样子？

**2. 当前展示的树结构**

```
                         [ 22 ]                          ← 根节点
                           |
           [ 5 | 14 | 17 | 20 ]                          ← 唯一的子内节点
           /    |    |    \      \
         /      |    |     \      \
    [2,3] [5,7,8] [14,16] [17,18] [20,21] [22,27,29] [33,34,38,39] ...
```

- **根节点**：仅包含一个键值 `22`，两个子节点指针（分别指向左内节点和右内节点）。
- **左内节点**：`[5, 14, 17, 20]`，包含 4 个键值，5 个子指针，指向 5 个叶子节点。
- **右内节点**：未完整画出，但从上下文可知它管理右侧的叶子节点（如包含 `24*` 的那个）。
- **目标叶子**：假设 `24*` 位于右侧某个叶子节点中（图中未直接显示该叶子，但根据后续操作可推断）。

**3. 可能的初始树推测**

- 在删除 `24*` 之前，初始树可能是一个高度为 3 的 B⁺ 树。
- 删除 `24*` 导致右侧某个叶子节点发生合并，进而导致其父节点（右内节点）丢失了一个索引条目。
- 丢失索引条目后，右内节点变得**过瘦**（低于最低填充要求），但它有一个**富余的左兄弟**（即图中的左内节点 `[5, 14, 17, 20]`，拥有 4 个键值，远超最低要求）。
- 这为重分配创造了条件：**可以从左兄弟借一个条目给右兄弟**。

**4. 与之前删除示例的对比**

| 对比维度       | 之前的删除示例         | 本页示例                    |
| :------------- | :--------------------- | :-------------------------- |

### 第 2 页：重分配之后的结果——Push Through 机制

这一页展示了非叶子节点重分配完成后的树结构，并提出了 **Push Through** 这一关键概念。

**1. 重分配后的树结构**

```
                         [ 20 ]                          ← 根节点
                        /      \
          [ 5 | 14 | 17 ]        [ 22 | 33 ]             ← 两个子内节点
          /    |    \   \         /    |    \
    [2,3] [5,7,8] [14,16] [17,18] [20,21] [22,27,29] [33,34,38,39]
```

**2. 结构变化分析**

- **根节点变化**：键值从 `22` 变为 `20`。
- **左内节点变化**：从 `[5, 14, 17, 20]` 缩减为 `[5, 14, 17]`（丢失了 `20`，且少了一个子指针）。
- **右内节点变化**：从原本过瘦的状态（图中未完整显示）扩充为 `[22, 33]`（获得了新的键值和指针）。

**3. Push Through 机制详解**

> *Intuitively, entries are re-distributed by `pushing through` the splitting entry in the parent node.*

- **Push Through** 描述的是**非叶子节点之间重分配**时，键值在父节点和子节点之间的流动方式。
- **操作分解**：
    1. 父节点（根节点）中的分隔键 `22` 被**下推**到右内节点中，成为右内节点的新键值。
    2. 左内节点中的一个富余键值 `20` 被**上推**到父节点，替换原分隔键 `22`，成为新的根节点键值。
    3. 同时，与键值 `20` 关联的子指针（原本指向 `[20, 21]` 叶子节点）被**转移**给右内节点。

- **直观理解**：
    - 键值 `22` 从父节点“穿过”到达右子节点。
    - 键值 `20` 从左子节点“穿过”父节点（替换了 `22`），并带着它的子树一起转移到右侧。
    - 整个过程中，键值像**穿针引线**一样在父子节点间流动，故称 **Push Through**。

**4. Push Through 与叶子节点重分配的区别**

| 对比维度           | 叶子节点重分配                | 非叶子节点重分配                     |
| :----------------- | :---------------------------- | :----------------------------------- |
| **键值处理**       | Copy-up（复制中间键到父节点） | Push Through（键值在父子间流动替换） |
| **父节点键值变化** | 可能更新为新的最小边界值      | 原有键值被下推，新键值上推替换       |
| **涉及结构**       | 叶子节点 + 直接父节点         | 两个子内节点 + 父节点                |
| **子节点内容**     | 兄弟间移动数据条目            | 兄弟间移动子指针和对应键值           |

**5. 为什么采用 Push Through？**

- 非叶子节点存储的是**导航键值和子指针**，不存储实际数据。
- 重分配的本质是在两个子内节点间重新平衡**子指针的数量**。
- 父节点中的分隔键正好位于两个子节点的逻辑边界上，将其下推可以自然地成为接收方的新边界键；同时，捐赠方将多余的边界键上推，替换父节点中原分隔键。
- 这种设计保证了 B⁺ 树的**区间完整性**：任何键值的路由规则在重分配后仍然正确。

**6. 重分配 vs 合并的权衡**

| 策略       | 何时使用           | 树高变化 | I/O 代价                   |
| :--------- | :----------------- | :------- | :------------------------- |
| **重分配** | 有富余兄弟可借     | 不变     | 局部调整（修改父子节点）   |
| **合并**   | 所有兄弟都刚好半满 | 可能降低 | 可能递归向上，涉及更多节点 |

- 实际数据库系统中，**优先尝试重分配**，只有在无法重分配时才执行合并，以尽量减少树结构调整的范围。
