### 第 1 页：B⁺ 树删除算法框架

这一页给出了 B⁺ 树删除操作的**通用算法步骤**，是理解后续实例的理论基础。

**1. 步骤一：定位目标叶子节点 L**

- 从根节点开始，按照搜索键值的大小，逐层向下，找到**包含待删除条目的叶子节点**。

**2. 步骤二：删除条目**

- 从叶子节点 L 中移除目标数据条目。
- 检查 L 删除后的填充情况：
    - 若 L 仍保持**至少半满**（即条目数 ≥ \(\lceil (n-1)/2 \rceil\)），则删除操作**完成**，无需进一步调整。
    - 若 L 删除后条目数不足（仅 **d-1** 个，其中 d 为最低填充数），则需要采取补救措施。

**3. 步骤三：补救措施——重分配与合并**

- **尝试重分配**：
    - 检查 L 的**相邻兄弟节点**（与 L 拥有相同父节点的左兄弟或右兄弟）。
    - 若某个兄弟节点有**多余**的条目（即条目数 > 最低要求），则从该兄弟**借**一个条目到 L。
    - 重分配后，需要**更新父节点**中对应两个子节点的分隔键值（Copy-up 中间键）。

- **若重分配失败**（兄弟节点均无多余条目），则执行**合并**：
    - 将 L 与某个兄弟节点**合并**为一个节点，将两个节点的所有条目放入其中。
    - 合并后，原父节点中指向被合并节点的索引条目将**不再需要**，必须从父节点中删除。

**4. 步骤四：递归向上调整**

- 由于父节点中删除了一个索引条目，父节点也可能变得过小而需要调整。
- 调整操作（重分配或合并）可能沿路径**递归向上传播**，直至根节点。

**5. 步骤五：树高降低**

- 如果合并传播到根节点，且根节点最终只剩下一个子节点指针（无键值），则该根节点被删除，其唯一的子节点成为**新的根**。
- 这是 B⁺ 树**降低高度**的唯一方式（自下而上收缩）。

### 第 2 页：初始 B⁺ 树状态与删除 19\*

这一页展示了插入 `8*` 后形成的 B⁺ 树（即上一组 PPT 的最终结果），并开始删除操作。

**1. 初始树结构**

```
                    [ 17 ]                     ← 根节点
                   /      \
          [ 5 | 14 ]        [ 24 | 33 ]        ← 内节点
          /    |    \        /    |    \
    [2,3] [5,7,8] [14,16] [19,20,22] [24,27,29] [33,34,38,39]
```

- 根节点：`17`
- 左内节点：`5, 14`，指向三个叶子节点
- 右内节点：`24, 33`，指向三个叶子节点
- 叶子节点存储实际数据（用 `*` 表示）

**2. 删除 19\***

- `19*` 位于第四个叶子节点 `[19*, 20*, 22*]` 中。
- 该叶子节点原有 3 个条目。删除 `19*` 后剩余 2 个条目（`20*, 22*`）。
- 假设最低填充要求为 2（对应 \( n=5 \) 时，\(\lceil (n-1)/2 \rceil = 2\)），删除后**仍满足半满要求**。
- **结论**：*Deleting 19\* is easy.* 无需任何额外调整，操作完成。

### 第 3 页：删除 20\* 与重分配

这一页继续删除 `20*`，触发了**兄弟节点重分配**操作。

**1. 删除 20\* 前状态**

- 在删除 `19*` 后，第四个叶子节点内容为 `[20*, 22*]`（2 个条目）。
- 若再删除 `20*`，该叶子节点将只剩 `[22*]`（1 个条目），**低于最低填充要求**。

**2. 尝试重分配**

- 检查该叶子节点的兄弟节点：
    - 左兄弟：`[14*, 16*]`，只有 2 个条目（刚刚满足最低要求），**无多余可借**。
    - 右兄弟：`[24*, 27*, 29*]`，有 3 个条目（超过最低要求），**可以借出一个**。
- **重分配过程**：
    1. 从右兄弟移动最左侧条目 `24*` 到当前叶子节点。
    2. 当前叶子节点变为 `[22*, 24*]`（2 个条目）。
    3. 右兄弟变为 `[27*, 29*]`（2 个条目）。
    4. 由于右兄弟的最小键值从 `24` 变为 `27`，必须**更新父节点**中的分隔键值。

**3. 父节点更新——Copy-up 的体现**

- 原父节点（右内节点）的键值为 `[24, 33]`。
- 重分配后，指向第四个叶子节点的子指针对应的分隔键应更新为新的最小键值 `27`。
- 因此，父节点中的 `24` 被替换为 `27`（或将 `27` Copy-up 到父节点）。
- **PPT 强调**：*Notice how middle key is copied up.* 这正是重分配时键值调整的特征——兄弟节点间移动条目后，父节点中对应分隔键需同步更新为新的最小边界值。

**4. 删除 20\* 后树结构**

```
                    [ 17 ]
                   /      \
          [ 5 | 14 ]        [ 27 | 33 ]        ← 父节点中 24 变为 27
          /    |    \        /    |    \
    [2,3] [5,7,8] [14,16] [22,24] [27,29] [33,34,38,39]
```

- 第四个叶子节点 `[22*, 24*]` 删除 `24*` 后，将只剩 `[22*]`（1 个条目），**低于最低要求**。
- 需要尝试与兄弟节点进行重分配或合并。

### 第 5 页：删除 24\* 触发合并与树高降低

这一页展示了删除 `24*` 后，因重分配失败而触发**节点合并**，并最终导致**树高降低**的完整过程。

**1. 删除 24\* 后的局部状态**

- 第四个叶子节点变为 `[22*]`，仅 1 个条目，低于最低要求。

**2. 尝试重分配——失败**

- 检查兄弟节点：
    - 左兄弟：`[14*, 16*]`，只有 2 个条目（刚好半满），**无法借出**。
    - 右兄弟：`[27*, 29*]`，只有 2 个条目（刚好半满），**也无法借出**。
- 重分配失败，必须执行**合并**。

**3. 叶子节点合并**

- 将 `[22*]` 与其右兄弟 `[27*, 29*]` 合并（也可与左兄弟合并，通常选择与右兄弟合并）。
- 合并后的新叶子节点：`[22*, 27*, 29*]`（3 个条目）。
- 此时，原父节点（右内节点）中指向被合并右兄弟的索引条目（键值 `27` 及对应指针）**不再需要**，必须从父节点中删除。

**4. 内节点调整——Toss 与 Pull Down**

- **Toss（丢弃）**：父节点中指向被合并叶子节点的索引条目被**直接丢弃**。
    - 在 PPT 图示中，右侧内节点原本为 `[27, 33]`，删除指向第二个叶子的指针后，该内节点变为 `[33]`（仅剩一个键值和一个指针）。
- 内节点 `[33]` 的条目数低于最低要求（内节点最低应有 3 个指针 / 2 个键值，假设 \( n=5 \) 时内节点最少 3 指针即 2 键值）。
- 尝试与左兄弟（`[5, 14]`）重分配？但左兄弟也仅有 2 个键值（刚好半满），无法借出。
- 因此，必须进一步**合并内节点**。

**5. 内节点合并与 Pull Down**

- 将右侧内节点 `[33]` 与其左兄弟 `[5, 14]` 合并。
- 合并时，父节点（根节点 `[17]`）中的分隔键 `17` 被**下拉**到合并后的新内节点中。
- **Pull Down（下拉）**：根节点中的键值 `17` 被移动下来，成为新内节点的一部分。
- 合并后的新内节点内容：`[5, 14, 17, 33]`（4 个键值，5 个指针）。
- 根节点中原有的唯一键值被拉下后，根节点变为**空**（无键值，仅剩一个指向合并后内节点的指针）。

**6. 树高降低**

- 空的根节点已无存在必要，被删除。
- 其唯一的子节点（合并后的内节点 `[5, 14, 17, 33]`）成为**新的根节点**。
- **树的高度从 3 层降低为 2 层**。

**7. 最终树结构**

```
        [ 5 | 14 | 17 | 33 ]           ← 新根节点（原内节点提升）
        /    |    |    \     \
      /     |    |     \     \
[2,3] [5,7,8] [14,16] [22,27,29] [33,34,38,39]
```

- 叶子节点共 5 个，由新根节点直接管理。

**8. Toss 与 Pull Down 总结**

| 术语                  | 含义                                                     | 发生场景                                                 |
| :-------------------- | :------------------------------------------------------- | :------------------------------------------------------- |
| **Toss（丢弃）**      | 父节点中指向被合并子节点的索引条目被直接删除，无需保留。 | 子节点合并后，原指针失效，对应键值随之丢弃。             |
| **Pull Down（下拉）** | 父节点中的分隔键被移动到合并后的新节点中。               | 内节点合并时，父节点中的分隔键需要下拉以维持区间完整性。 |
