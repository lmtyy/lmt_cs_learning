### 第 1 页：覆盖索引的概念与优势

**1. 什么是覆盖索引？**

- **定义**：*查询语句从执行到返回结果均使用同一个索引*。
- **更精确的理解**：一个查询需要返回的**所有列**（SELECT 子句中的列）和**查询条件涉及的列**（WHERE、ORDER BY、GROUP BY 等），都**包含在同一个索引**中。此时，数据库引擎只需要扫描这个索引的 B⁺ 树结构，就能获取全部所需数据，**无需再回到主键索引（聚簇索引）中去读取完整的行记录**。

**2. 覆盖索引如何有效减少回表？**

- **回表**：在使用二级索引时，如果查询需要索引中不包含的列，数据库必须拿着二级索引叶子节点中存储的**主键值**，再到聚簇索引中查找完整的行数据，这个过程称为回表。回表通常意味着**随机 I/O**，代价较高。
- **覆盖索引的作用**：如果索引的叶子节点已经包含了查询所需的所有数据，则直接返回，**省去了回表这一步**，将随机 I/O 转化为顺序扫描索引页，大幅提升查询效率。

**3. 判断标准（通过 EXPLAIN）**

- 在 `EXPLAIN` 输出的 `Extra` 字段中，如果出现 **`Using index`**，就表示该查询使用了覆盖索引，没有发生回表。

### 第 2 页：Sakila 数据库中的 inventory 表结构分析

这一页给出了 `inventory` 表的 DDL 语句和 `SHOW INDEX` 的输出结果。

**1. 表结构解读**

```sql
CREATE TABLE inventory (
    inventory_id MEDIUMINT UNSIGNED NOT NULL AUTO_INCREMENT,
    film_id SMALLINT UNSIGNED NOT NULL,
    store_id TINYINT UNSIGNED NOT NULL,
    last_update TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (inventory_id),
    KEY idx_fk_film_id (film_id),
    KEY idx_store_id_film_id (store_id, film_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

- **`PRIMARY KEY (inventory_id)`**：聚簇索引，叶子节点存放完整行数据。
- **`KEY idx_fk_film_id (film_id)`**：单列二级索引，叶子节点存放 `(film_id, inventory_id)`。
- **`KEY idx_store_id_film_id (store_id, film_id)`**：复合二级索引，叶子节点存放 `(store_id, film_id, inventory_id)`。

**2. `SHOW INDEX` 输出分析**

| Key_name             | Seq_in_index | Column_name  |
| :------------------- | :----------- | :----------- |
| PRIMARY              | 1            | inventory_id |
| idx_fk_film_id       | 1            | film_id      |
| idx_store_id_film_id | 1            | store_id     |
| idx_store_id_film_id | 2            | film_id      |

- 这验证了上述索引结构：复合索引 `idx_store_id_film_id` 包含两列，顺序为 `store_id` 在前，`film_id` 在后。

**3. 覆盖索引的应用示例**

基于这张表的结构，我们可以构造覆盖索引的典型场景：

- **查询 1**：
  ```sql
  SELECT film_id FROM inventory WHERE film_id = 5;
  ```
  - 使用的索引：`idx_fk_film_id`。
  - 所需列 `film_id` 就在索引中，无需回表 → **覆盖索引**。

- **查询 2**：
  ```sql
  SELECT store_id, film_id FROM inventory WHERE store_id = 1;
  ```
  - 使用的索引：`idx_store_id_film_id`。
  - 所需列 `store_id` 和 `film_id` 均在复合索引中，无需回表 → **覆盖索引**。

- **查询 3（反例）**：
  ```sql
  SELECT film_id, last_update FROM inventory WHERE film_id = 5;
  ```
  - 使用的索引：`idx_fk_film_id`。
  - `last_update` 不在该索引中，必须根据 `inventory_id` 回表到聚簇索引获取 → **非覆盖索引**，`Extra` 列显示 `NULL` 或 `Using where`。

### 第 3 页：简化版 inventory1 表（无外键与联合索引）

这一页展示了一个**去掉外键约束和复合索引**的简化版本。

```sql
CREATE TABLE inventory1 (
    inventory_id MEDIUMINT UNSIGNED NOT NULL AUTO_INCREMENT,
    film_id SMALLINT UNSIGNED NOT NULL,
    store_id TINYINT UNSIGNED NOT NULL,
    last_update TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

**1. 与原始表的差异**

| 对比项   | 原表 `inventory`                           | 简化表 `inventory1`                                        |
| :------- | :----------------------------------------- | :--------------------------------------------------------- |
| 主键     | `PRIMARY KEY (inventory_id)`               | 仅定义了列，未显式声明主键（但 InnoDB 会自动创建隐藏主键） |
| 外键约束 | 有（`film_id` 引用 `film` 表）             | 无                                                         |
| 二级索引 | `idx_fk_film_id` 和 `idx_store_id_film_id` | **无**                                                     |

**2. 知识点：缺少索引对覆盖索引的影响**

- 在 `inventory1` 表上，除了自动生成的主键索引外，**没有任何用户定义的二级索引**。
- 这意味着任何以 `film_id` 或 `store_id` 为条件的查询，都将无法利用索引完成覆盖。
- 例如：
  ```sql
  SELECT film_id FROM inventory1 WHERE film_id = 5;
  ```
  - 因为没有 `film_id` 索引，数据库必须执行**全表扫描**（或全聚簇索引扫描），无法使用覆盖索引优化。
  - 这恰好从反面说明了**创建合适的索引**对于实现覆盖索引、提升查询性能的重要性。

**3. 设计启示**

- 覆盖索引并非一种特殊的索引类型，而是**对现有索引的一种高效利用方式**。
- 在设计复合索引时，如果能把**经常作为查询条件的列**和**经常作为返回结果的列**都纳入索引中，就能让很多高频查询实现覆盖索引，从而显著降低 I/O 开销。
- 但也要注意权衡：索引列越多，索引体积越大，写入维护代价越高。应当优先覆盖**高频、关键**的查询。
