### 第 1 页：静态哈希的缺陷

这一页指出了传统静态哈希方案在数据库动态变化场景下的根本性不足。

**1. 静态哈希的定义与特征**

- **定义**：哈希函数 \( h \) 将搜索键值映射到**固定数量** \( B \) 个桶地址上。
- **核心问题**：桶的数量 \( B \) 在创建时确定，此后**不再改变**，而数据库的数据量会随时间**增长或收缩**。

**2. 三种典型困境**

| 困境               | 原因                   | 后果                                         |
| :----------------- | :--------------------- | :------------------------------------------- |
| **初始桶数太小**   | 低估数据增长速度       | 大量溢出块产生，溢出链变长，查询性能急剧下降 |
| **预分配过大空间** | 为避免溢出而预留过多桶 | 初期大量桶近乎空置，空间浪费严重             |
| **数据库收缩**     | 数据被大量删除         | 桶依然占用空间，利用率极低                   |

**3. 传统方案：周期性重组**

- **方法**：定期使用新的哈希函数对整个文件重新组织，调整桶数以适应当前数据量。
- **缺点**：
    - **代价高昂**：需读取并重写所有数据页。
    - **影响正常运行**：重组期间数据库性能严重下降，甚至需停机。

**4. 更优方案的方向**

- **原文**：*Better solution: allow the number of buckets to be modified dynamically.*
- **目标**：桶数量应能**动态调整**，且仅在必要时对局部数据进行重组，避免全局重构。

### 第 2 页：可扩展哈希的核心思想

这一页介绍了可扩展哈希如何通过**目录**机制实现低代价的动态扩容。

**1. 问题情境与直观但低效的方案**

- **情境**：某个桶（主数据页）已满，无法继续插入新记录。
- **直观想法**：为什么不直接**将桶总数翻倍**来缓解冲突？
    - **代价**：翻倍桶数量意味着必须**读取所有数据页并重新哈希分配**，I/O 成本极高，不可接受。

**2. 可扩展哈希的创新思路**

- **引入指针目录**：
    - 建立一个**目录**结构，目录中存储指向各个桶的指针。
    - 哈希函数计算出的值不再直接定位桶，而是定位**目录项**，再通过目录项找到桶。

- **扩容策略**：
    - 当某个桶溢出时，**只拆分那个溢出的桶**，而不是全部桶。
    - 如果需要更多目录项来指向新桶，则**翻倍目录大小**（即增加目录项数量）。
    - 目录远小于数据文件本身，因此翻倍目录的代价远低于翻倍所有数据页。

- **消除溢出页**：
    - 通过动态拆分桶，可扩展哈希**不需要**维护溢出页链表，所有数据均存放于主桶中。

**3. 关键技巧：哈希函数的动态调整**

- 如何在不重新哈希全表的情况下实现桶的动态拆分？
- 答案在于**只使用哈希值的一部分位**来决定目录项。
- 随着目录翻倍，**使用的位数逐位增加**，自然将原有桶的数据重新分布。

### 第 3 页：可扩展哈希实例与插入机制

这一页通过具体图示和数值，详细解释了可扩展哈希的结构、核心概念及插入分裂流程。

**1. 基本结构图示**

- **目录**：一个大小为 4 的指针数组，索引分别为 `00`、`01`、`10`、`11`（使用 2 位二进制）。
- **桶**：Bucket A、B、C、D，每个桶存储实际数据记录（如 `15*`、`7*`、`19*`）。

**2. 核心概念：全局深度与局部深度**

| 术语         | 符号 | 含义                                                         |
| :----------- | :--- | :----------------------------------------------------------- |
| **全局深度** | GD   | 当前目录使用的哈希值**位数**。目录项数 = \( 2^{GD} \)。图中 GD = 2，目录大小为 4。 |
| **局部深度** | LD   | 每个桶**实际使用**的哈希值位数。LD ≤ GD。记录该桶被创建时对应的目录层级。 |

**3. 查找规则**

- 给定记录 \( r \)，取其哈希值 \( h(r) \) 的**最后 GD 位**作为目录索引。
- 示例：若 \( h(r) = 5 \)（二进制 `101`），取最后 2 位 `01`，则定位到目录项 `01`，进而访问对应的桶。

**4. 插入与分裂流程**

- **步骤一**：计算哈希值，取最后 GD 位，定位目标桶。
- **步骤二**：若桶未满，直接插入，结束。
- **步骤三**：若桶已满，则**拆分该桶**：
    1. 为该桶分配一个新页（新桶）。
    2. 将原桶中的记录连同新记录，根据**多使用 1 位**哈希值（即考虑第 \( LD+1 \) 位）重新分配到两个桶中。
    3. 更新目录中受影响的指针，使其指向新桶。
    4. 两个桶的**局部深度均加 1**。
- **步骤四**：检查是否需要翻倍目录。
    - 若拆分前 **LD = GD**，则拆分意味着需要多 1 位来区分新旧桶，此时**GD 加 1，目录翻倍**。
    - 若拆分前 **LD < GD**，则目录大小已足够容纳更多桶指针，**无需翻倍目录**，仅需调整部分目录项指针。

**5. 目录翻倍判断表**

| 条件    | 是否翻倍目录 | 操作                                      |
| :------ | :----------- | :---------------------------------------- |
| LD < GD | 否           | 目录足够大，仅需修改部分指针指向新桶      |
| LD = GD | 是           | 目录已无多余位区分，GD 加 1，目录项数翻倍 |

### 第 4 页：动态哈希的通用框架

这一页将可扩展哈希提升到**动态哈希**的通用理论高度，阐述了动态哈希的共性特征和参数化定义。

**1. 动态哈希的适用场景与核心能力**

- **原文**：*Good for database that grows and shrinks in size.*
- **原文**：*Allows the hash function to be modified dynamically.*
- **核心价值**：能够随数据量动态增减而自动调整哈希结构，无需人工干预或周期性重组。

**2. 可扩展哈希作为动态哈希的一种实现**

- 可扩展哈希是动态哈希家族中最经典的成员之一。
- 其他动态哈希方案还包括**线性哈希**等。

**3. 动态哈希的关键机制：哈希前缀的使用**

- **哈希值范围**：哈希函数生成的值通常是一个**大范围**的整数，如 32 位（\( b = 32 \)），可表示约 43 亿个不同的哈希值。
- **前缀索引**：在任何时刻，**仅使用哈希值的前缀**（即前 \( i \) 位）来索引桶地址表。
- **前缀长度 \( i \)**：
    - \( i \) 是一个可变的整数，\( 0 \leq i \leq 32 \)。
    - 桶地址表的大小 = \( 2^i \)。
    - **初始状态**：\( i = 0 \)，桶地址表大小为 1（只有一个桶）。
- **\( i \) 的动态变化**：
    - 随着数据库**增长**，\( i \) 逐渐**增大**，目录翻倍，桶数量增加。
    - 随着数据库**收缩**，\( i \) 可以**减小**（通过合并桶和收缩目录），释放空间。

**4. 桶数量的动态变化**

- 除了目录大小随 \( i \) 变化外，**桶的数量**也通过**拆分**和**合并**操作动态改变。
- 拆分：当桶溢出时，将其一分为二。
- 合并：当多个桶的总数据量足够小时，可将它们合并为一个桶，以提升空间利用率。

**5. 动态哈希与静态哈希的对比总结**

| 对比维度       | 静态哈希                 | 动态哈希                   |
| :------------- | :----------------------- | :------------------------- |
| **桶数量**     | 固定                     | 随数据量动态变化           |
| **溢出处理**   | 依赖溢出链               | 通过拆分消除溢出           |
| **空间利用率** | 难以预测，可能浪费或不足 | 始终保持较高利用率         |
| **重组代价**   | 需周期性全局重组         | 仅局部拆分或合并，代价小   |
| **哈希函数**   | 固定映射                 | 使用前缀，动态调整有效位数 |
