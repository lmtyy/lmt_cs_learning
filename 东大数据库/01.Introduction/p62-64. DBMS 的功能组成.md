这三页 PPT 从**用户视角**和**系统内部视角**两个维度，全面剖析了 **DBMS 的功能组成**。最后一页更是给出了 DBMS 内部模块结构的经典图示，是理解“数据库是如何工作”的关键一页。

### 第一页：从用户视角看 DBMS 功能

> **PPT 内容提炼：**
> - 从**用户**角度看，DBMS 提供四大功能：
>     1. **数据定义**
>     2. **数据操纵**
>     3. **数据控制**
>     4. **数据维护**（备份、恢复等）

**知识点深度剖析：**

#### 1. 用户视角的四大功能

| 功能类别     | 英文              | 对应用户能做什么                       | 对应的数据库语言/工具                                  |
| :----------- | :---------------- | :------------------------------------- | :----------------------------------------------------- |
| **数据定义** | Data Definition   | 创建表、修改表结构、删除表             | **DDL**                                                |
| **数据操纵** | Data Manipulation | 查询数据、插入数据、更新数据、删除数据 | **DML**                                                |
| **数据控制** | Data Control      | 授予/收回权限、保证数据安全            | **DCL**（Data Control Language，如 `GRANT`, `REVOKE`） |
| **数据维护** | Data Maintenance  | 备份数据库、恢复数据库、数据导入导出   | 备份工具（如 `mysqldump`）、日志恢复机制               |

#### 2. 为什么要有“数据控制”和“数据维护”？

- **数据控制**：在多用户环境下，不是所有人都能看所有数据。财务部的数据不能让教务处的人随便改。DCL 提供了细粒度的权限管理。
- **数据维护**：硬件会坏、人会误操作、系统会崩溃。没有备份和恢复功能，数据库就是“一次性”的。

#### 3. 这四大功能与之前讲的概念对应

| 用户视角功能 | 之前讲过的对应概念                   |
| :----------- | :----------------------------------- |
| 数据定义     | DDL、数据字典、模式定义              |
| 数据操纵     | DML、SQL 查询、过程式 vs 声明式      |
| 数据控制     | 安全性、授权、数据字典中的授权信息   |
| 数据维护     | 备份与恢复、事务日志、原子性与持久性 |

### 第二页：从系统视角看 DBMS 功能

> **PPT 内容提炼：**
> - 从**系统**角度看，DBMS 包含：
>     1. **SQL 语言编译器**
>     2. **查询引擎**
>     3. **数据组织、存储与管理**（索引、顺序）
>     4. **事务管理**

**知识点深度剖析：**

这一页将视角从“用户能做什么”切换到“DBMS 内部怎么做”。

#### 1. 系统视角的四大组件

| 组件                     | 作用                                                         | 用户视角的对应功能                       |
| :----------------------- | :----------------------------------------------------------- | :--------------------------------------- |
| **SQL 语言编译器**       | 将用户输入的 SQL 文本**解析**成 DBMS 内部能理解的语法树和执行计划 | 数据定义 + 数据操纵（的第一步）          |
| **查询引擎**             | **执行**查询计划，实际去存取数据，完成增删改查操作           | 数据操纵（的核心执行层）                 |
| **数据组织、存储与管理** | 负责数据在磁盘上的**物理存放**（堆文件、B+树索引、哈希索引） | 数据定义（物理层）+ 数据维护（底层支持） |
| **事务管理**             | 保证**ACID**特性（原子性、一致性、隔离性、持久性）           | 数据控制（并发控制）+ 数据维护（恢复）   |

#### 2. SQL 编译器 vs 查询引擎

- **SQL 编译器**：是**前端的解析和优化器**。它拿到 `SELECT * FROM ... WHERE ...`，先检查语法，再生成**多个候选执行计划**，然后评估成本，选出**最优计划**。
- **查询引擎**：是**后端的执行器**。它拿着最优计划，调用存储管理器去读数据、做连接、做过滤，最后返回结果。

#### 3. 事务管理的核心地位

事务管理是 DBMS 区别于文件系统的**灵魂组件**。

| 子组件         | 职责                                               |
| :------------- | :------------------------------------------------- |
| **锁管理器**   | 实现并发控制，保证隔离性                           |
| **日志管理器** | 记录所有更新操作，用于崩溃恢复，保证原子性和持久性 |

### 第三页：DBMS 内部模块结构图

> **PPT 内容提炼：**
> 一幅完整的 DBMS 模块层次图，从上到下包括：
> - **顶部接口层**：应用程序接口、应用程序、查询工具、管理工具
> - **编译与执行层**：编译器与链接器、DML 查询、DDL 解释器、应用程序目标代码、DML 编译器与组织器、查询评估引擎
> - **存储与事务管理层**：缓冲管理器、文件管理器、授权与完整性管理器、事务管理器
> - **存储管理器**
> - **底层存储**：磁盘存储、索引、数据字典、统计数据

**知识点深度剖析：**

这张图是**理解 DBMS 内部工作原理的蓝图**，需要分层解读。

#### 1. 自顶向下的层次结构

```
┌─────────────────────────────────────────────────────────────┐
│ 第一层：应用接口层（用户/程序与 DBMS 的边界）                      │
│ - 应用程序接口（JDBC/ODBC）、应用程序、查询工具、管理工具          │
├─────────────────────────────────────────────────────────────┤
│ 第二层：查询处理层（将 SQL 转化为执行计划）                        │
│ - DDL 解释器、DML 编译器、查询评估引擎                            │
├─────────────────────────────────────────────────────────────┤
│ 第三层：事务与存储管理层（保证 ACID 和管理物理存取）                 │
│ - 事务管理器、授权与完整性管理器、缓冲管理器、文件管理器              │
├─────────────────────────────────────────────────────────────┤
│ 第四层：物理存储层（磁盘上的实际文件）                             │
│ - 数据文件、索引文件、数据字典、统计数据                            │
└─────────────────────────────────────────────────────────────┘
```

#### 2. 逐模块详解

##### 第一层：应用接口层

| 组件             | 说明                                     |
| :--------------- | :--------------------------------------- |
| **应用程序接口** | 如 JDBC、ODBC，供外部程序调用            |
| **应用程序**     | 编译好的业务程序（如银行转账系统）       |
| **查询工具**     | 如 MySQL Workbench、SQL*Plus，供人机交互 |
| **管理工具**     | 如备份工具、性能监控工具                 |

##### 第二层：查询处理层

这是 **SQL 语句的流水线**。

| 组件             | 职责                                         |
| :--------------- | :------------------------------------------- |
| **DDL 解释器**   | 解析 `CREATE TABLE` 等语句，更新**数据字典** |
| **DML 编译器**   | 解析 `SELECT` 等语句，生成**逻辑查询计划**   |
| **查询评估引擎** | 执行查询计划，调用存储管理器读写数据         |

**一个 SELECT 语句的执行路径**：
```
SQL 文本 → DML 编译器（语法分析、语义检查）→ 查询优化器（生成最优计划）→ 查询评估引擎（执行计划）→ 返回结果
```

##### 第三层：事务与存储管理层

这是 DBMS 的**核心保障层**。

| 组件                   | 职责                                        | 解决的问题               |
| :--------------------- | :------------------------------------------ | :----------------------- |
| **授权与完整性管理器** | 检查用户权限，检查约束（主键、外键、CHECK） | 安全性、完整性           |
| **事务管理器**         | 管理事务的开始、提交、回滚；协调锁和日志    | 并发控制、原子性、持久性 |
| **缓冲管理器**         | 在内存中缓存磁盘页，减少 I/O                | 性能                     |
| **文件管理器**         | 管理磁盘上的数据文件（分配空间、读写页）    | 物理存储访问             |

**缓冲管理器的重要性**：
- 数据库的数据在**磁盘**上，但操作必须在**内存**中进行。
- 缓冲管理器维护一个**内存缓冲池**，当需要读取某页数据时，先看缓冲池里有没有，没有再读磁盘。
- 修改数据时，先在缓冲池修改，标记为“脏页”，由后台进程异步写回磁盘。

##### 第四层：物理存储层

| 组件         | 内容                                      | 作用                   |
| :----------- | :---------------------------------------- | :--------------------- |
| **数据文件** | 实际存储表数据的文件（如 `.ibd`, `.mdf`） | 存放用户数据           |
| **索引文件** | 存储 B+树或哈希索引的文件                 | 加速查询               |
| **数据字典** | 元数据的存储区域                          | 描述表结构、约束、权限 |
| **统计数据** | 表行数、列值分布等统计信息                | 供查询优化器估算成本   |

#### 3. 模块协作实例：执行一条 UPDATE 的全流程

以 `UPDATE account SET balance = balance - 100 WHERE account_number = 'A-101';` 为例，穿行于各模块：

1. **应用接口层**：应用程序通过 JDBC 发送 SQL 字符串。
2. **DML 编译器**：解析 SQL，检查 `account` 表是否存在、`balance` 列是否存在。
3. **查询优化器**：决定用 `account_number` 上的主键索引定位记录。
4. **授权与完整性管理器**：检查用户是否有 `UPDATE` 权限；检查更新后 `balance` 是否违反 `CHECK (balance >= 0)`。
5. **事务管理器**：记录事务开始，写日志（UNDO/REDO 日志）。
6. **查询评估引擎**：调用**缓冲管理器**读取包含该记录的磁盘页到内存。
7. **缓冲管理器**：修改内存页中的 `balance` 值，标记为脏页。
8. **事务管理器**：事务提交时，保证日志写入磁盘。
9. **缓冲管理器**：在检查点或缓冲区满时，将脏页刷回磁盘。
10. **返回结果**：告知应用程序更新成功。
