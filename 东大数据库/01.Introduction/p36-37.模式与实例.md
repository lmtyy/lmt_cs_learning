### 第一页：三级抽象层次的图示

> **PPT 内容提炼：**
> - 顶部三个框：`ARR Sub schema`、`SW Dept Sub schema`、`Financial Office Sub schema`
> - 中间层：`conceptual schema`
> - 底层：`Physical schema` → `NEU database` → `Files on disks`

**知识点深度剖析：**

这张图是**三级模式架构**的标准教学图示，它把上一轮我们讲的“物理层、逻辑层、视图层”用数据库术语重新命名并建立了映射关系。

#### 1. 三层结构的术语对应

| 图示中的名称          | 标准中文术语 | 对应上一讲的层次 | 含义解释                               |
| :-------------------- | :----------- | :--------------- | :------------------------------------- |
| **Sub schema**        | **外模式**   | 视图层           | 子模式，即**用户看到的局部数据视图**   |
| **Conceptual schema** | **模式**     | 逻辑层           | 概念模式，即**数据库全局逻辑结构**     |
| **Physical schema**   | **内模式**   | 物理层           | 物理模式，即**数据在磁盘上的存储方式** |

#### 2. 图中三个 Sub schema 的具体含义

- **ARR Sub schema**：教务处看到的视图（可能只关心学生选课、成绩）。
- **SW Dept Sub schema**：软件学院看到的视图（可能只关心本院教师和学生）。
- **Financial Office Sub schema**：财务处看到的视图（可能只关心教职工工资信息）。

**关键点**：
- 这三个部门面对的是**同一个物理数据库**（NEU database）。
- 但他们各自看到的**数据范围和结构不同**。
- 财务处**不应该看到学生成绩**，软件学院**不应该看到教职工工资**。

#### 3. 箭头方向与数据独立性

图中的箭头方向表示**映射关系**：

- **向下箭头**：`Sub schema` → `Conceptual schema`
    - 表示 **外模式/模式映像**。
    - 作用：当概念模式改变（如全局表增加一列），只需修改这个映像，外模式可以保持不变 → **保证逻辑独立性**。

- **继续向下**：`Conceptual schema` → `Physical schema` → `Files on disks`
    - 表示 **模式/内模式映像**。
    - 作用：当物理存储改变（如换了硬盘、改了索引结构），只需修改这个映像，概念模式保持不变 → **保证物理独立性**。

#### 4. 图解的核心思想总结

```
用户A (ARR)          用户B (SW)          用户C (Finance)
    |                   |                    |
    v                   v                    v
外模式A              外模式B              外模式C
    \                  |                   /
     \                 |                  /
      \                v                 /
       ---------> 概念模式 (全局逻辑) <---
                       |
                       v
                  内模式 (物理存储)
                       |
                       v
                   磁盘文件
```

**一句话总结**：**同一个数据库，不同的人看到不同的视图，底层的物理存储对所有人透明。**

### 第二页：三级模式架构 + 模式与实例

> **PPT 内容提炼：**
> - 标题：**Three-schema architecture**（三级模式架构）
> - **Schema and Instance**（模式与实例）
> - 类比：类似于编程语言中的**类型**与**变量**
> - Schema：数据库的整体设计，类比变量的类型信息

**知识点深度剖析：**

这一页在做概念铺垫。在深入讲解三级模式之前，必须先区分两个极易混淆的术语：**模式**和**实例**。

#### 1. 模式 vs 实例 —— 核心概念辨析

| 对比维度       | **模式**                                                    | **实例**                                                     |
| :------------- | :---------------------------------------------------------- | :----------------------------------------------------------- |
| **英文**       | Schema                                                      | Instance                                                     |
| **定义**       | 数据库的**结构描述**，是骨架、是蓝图                        | 数据库中**某一时刻实际存储的数据**，是血肉、是具体内容       |
| **稳定性**     | **相对稳定**，一旦定义好，不常变动                          | **频繁变动**，每时每刻都在增删改                             |
| **编程类比**   | **类型定义**（如 `class Student { int id; string name; }`） | **对象/变量值**（如 `Student s1 = new Student(20, "常红")`） |
| **数据库举例** | `Student` 表有：学号(INT)、姓名(VARCHAR)、年龄(INT)         | 表中当前的三行记录：(20, 常红, 20)、(10, 都薇, 21)           |

#### 2. 为什么这个区分如此重要？

- **很多初学者会混淆**：他们指着 `Student` 表里的具体数据说“这是我的数据库模式”——**这是错的**。具体数据叫**实例**。
- **模式是设计阶段定的**：数据库设计者画出 E-R 图，转化成 `CREATE TABLE` 语句，这是在定义**模式**。
- **实例是运行阶段累积的**：用户每天 `INSERT` 数据，数据库**实例**在不断增长变化。

#### 3. 将“模式”概念套入“三级架构”

这一页标题虽然是 **Three-schema architecture**，但内容先讲了 Schema 的定义。这暗示了：**三个层级各自都有自己的 Schema**。

| 层级       | 对应的模式名称    | 内容举例                                                     |
| :--------- | :---------------- | :----------------------------------------------------------- |
| **外模式** | 子模式            | 视图定义：`CREATE VIEW SW_Student AS SELECT 姓名, 年级 FROM Student WHERE 系='软件学院';` |
| **模式**   | 概念模式          | 基本表定义：`CREATE TABLE Student (...); CREATE TABLE Course (...);` |
| **内模式** | 物理模式/存储模式 | 存储定义：`Student 表使用 B+树索引，数据文件存放在 /data/neu/student.dat` |

#### 4. 延伸理解：数据库的“状态”

- **数据库状态**：就是某一时刻数据库**实例**的总和。
- **初始状态**：刚建完表、还没插入任何数据时的空数据库。
- **当前状态**：每次增删改操作都会使数据库从一个状态迁移到另一个状态。
- DBMS 的职责之一就是**保证每次状态迁移都是合法、一致、完整的**（通过事务 ACID 保证）。
