### 第一页：DBMS 四大特性总览

> **PPT 内容提炼：**
> 1. 数据结构化（由数据模型描述，不依赖程序定义）
> 2. 数据共享性高、冗余度低、易于扩充
> 3. 数据独立性高
> 4. 数据由 DBMS 统一管理和控制

**知识点深度剖析：**

#### 特性一：数据结构化

- **文件系统的结构观**：在文件系统中，数据的“结构”是写在**应用程序的代码里**的。
    - 比如 C 语言程序里写 `fscanf(fp, "%10s %2d", name, &age);`。
    - 这意味着**只有这个程序**知道文件第1-10字节是姓名，第11-12字节是年龄。换一个程序来读，如果它不知道这个格式，读出来的就是乱码。
    - 这叫 **程序定义结构**。

- **数据库的结构观**：数据的结构被抽取出来，存储在**数据字典**中。
    - 你在建表时写的 `CREATE TABLE Student (Sname CHAR(10), Sage INT);`，这句定义被 DBMS 永久保存。
    - **任何应用程序**（无论是 Java 写的还是 Python 写的）连接数据库后，都可以通过查询数据字典得知 `Student` 表长什么样。
    - 这叫 **数据模型定义结构**，结构是**数据的一部分**，与程序解耦。

#### 特性二：高共享、低冗余、易扩充

- **高共享**：由于结构是公共定义的，财务系统、人事系统、门禁系统都可以访问同一个 `Employee` 表（当然是通过视图/权限控制）。
- **低冗余**：理想情况下，一个员工的基本信息只存一份。更新工龄时，只需修改一处。
- **易扩充**：要增加一个“员工邮箱”字段，只需执行 `ALTER TABLE Employee ADD Email VARCHAR(50);`。所有应用**无需修改代码**即可访问新字段（如果应用本身没写死列名的话）。而文件系统下，加一个字段意味着所有读写该文件的程序都要改代码重新编译。

### 第二页：DBMS 统一管理功能详解

> **PPT 内容提炼：**
> - 安全性：按指定方式使用数据。
> - 完整性：控制数据在有效范围内。
> - 并发控制。
> - 备份与恢复。

**知识点深度剖析：**

这四点正是针对上一轮我们讲的文件系统 **六大弊端** 的精准解药。

| 功能           | 文件系统的痛点（回顾）                                       | DBMS 的解决方案与机制                                        |
| :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **安全性**     | 要么能读整个文件，要么不能。无法做到“财务人员只能看工资表姓名列，不能看金额列”。 | **自主存取控制**：通过 `GRANT` 和 `REVOKE` 语句，精确授予用户对**表、视图、甚至列的** `SELECT, INSERT, UPDATE, DELETE` 权限。也支持**强制存取控制**（基于安全标签）。 |
| **完整性**     | 程序里写 `if (age < 0) error...`，容易被绕过，且每个程序都要写一遍。 | **声明式约束**：建表时定义 `CHECK (age >= 0 AND age <= 150)`。无论哪个程序试图插入负数年龄，DBMS 会**直接拒绝执行**，数据在入口处就被保护了。还包括**实体完整性**（主键非空唯一）、**参照完整性**（外键约束，不出现孤儿记录）。 |
| **并发控制**   | 两人同时取钱，可能只扣一次款。                               | **事务隔离性** + **封锁协议** / **MVCC**。保证多个用户同时操作时，感觉就像排着队一个一个来一样（串行化），逻辑正确。 |
| **备份与恢复** | 转账中断可能丢钱。                                           | **事务日志** + **检查点**。系统崩溃重启后，DBMS 自动读取日志，**撤销**未完成的事务，**重做**已提交但未写入磁盘的事务，保证数据恢复到一致状态。 |

### 第三页：程序与数据的关系转变

> **PPT 内容提炼：**
> - 图示：应用程序 1、应用程序 2 -> **DBMS** -> **数据库**。
> - 点明这是“数据库系统，数据，程序的对应关系”。

**知识点深度剖析：**

这张图虽然简单，但蕴含了数据库系统的**架构革命**。

- **文件系统架构**：
    - `App1` ——> `File1.dat`
    - `App2` ——> `File2.dat`
    - 每个应用自带一套数据访问逻辑，数据和程序是**紧耦合**的。

- **数据库系统架构**（PPT 图示）：
    - `App1` \
    - `App2` —-> **[ DBMS ]** —-> **[ Database ]**
    - `...` /

- **核心转变：中间层的引入**
    - **DBMS 作为一个中间层/代理层**，接管了所有对数据的访问请求。
    - **好处**：
        1. **统一入口**：监控、审计、优化都可以在这个入口做。
        2. **逻辑解耦**：应用程序不再需要知道数据具体存在哪个磁盘扇区，只需要知道表名和列名。
        3. **语言的统一**：所有应用都讲 **SQL**，而不是各自用不同的 C/Python/Java 文件解析库。

### 第四页：数据独立性问题引入

> **PPT 内容提炼：**
> - 开发中的大问题：应用程序与数据的分离。
> - **灵魂拷问**：发生以下物理或逻辑变化时，我**必须**修改程序吗？
>     - [ ] 换硬盘？
>     - [ ] 存储结构从哈希文件变成 B 树？
>     - [ ] 物理文件拆分或合并？
>     - [ ] 薪资数据类型从整数变浮点数？
>     - [ ] 增加新字段支持其他应用？
> - **答案/解决方案**：引入**抽象层次**。

**知识点深度剖析：**

这一页是**数据独立性**概念的导入页，非常经典。

1.  **问题拆解：如果我们在用文件系统**
    - **换硬盘**？ 程序不用改（因为操作系统抽象了盘符，路径不变即可）。
    - **哈希变 B 树**？ 程序**必须大改**！因为你原来调用的是 `hash_search()` 函数，现在要改成 `btree_search()`。
    - **文件拆分**？ 程序**必须改**！原来 `fopen("data.dat")`，现在要 `fopen("part1.dat")` 和 `fopen("part2.dat")` 自己合并逻辑。
    - **加字段**？ 程序**必须改**！原来按 10 字节读姓名，现在结构变了，程序读出来的字节偏移全乱套了。

2.  **数据库的承诺：高数据独立性**
    - 上面那一串复选框，在理想的数据系统里，**全部不应该修改应用程序**。
    - **物理独立性**：换了硬盘、索引从哈希变 B 树、文件分了区 —— 这些是**物理层**的事，应用程序不知道也不关心。
    - **逻辑独立性**：表里加了一列、改了一个列的数据类型（兼容范围内）—— 只要应用不访问改动部分，原有程序**照常运行**。

3.  **如何做到的？ —— 抽象层次**
    - 答案就在**抽象**二字。让用户/程序面对一个**逻辑上的简单视图**，把复杂的物理实现细节**隐藏**在底层。

### 第五页：数据抽象的三个层次

> **PPT 内容提炼：**
> - 数据库系统提供**数据的抽象视图**。
> - 通过**三个抽象层次**实现：
>     1. **物理层**。
>     2. **逻辑层**。
>     3. **视图层**。

**知识点深度剖析：**

这是数据库系统最核心的架构图——**三级模式结构**的简化版/等价版。理解这个，数据库系统才算入门。

| 层次       | 英文名         | 别名（三级模式对应） | **这一层描述什么？**                                         | **例子**                                                     | **谁关心？**                  |
| :--------- | :------------- | :------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :---------------------------- |
| **视图层** | View Level     | **外模式**           | 描述**部分数据**。隐藏了数据类型细节，甚至隐藏了部分列，只为特定应用提供定制画面。 | 人事专员看到的 `员工通讯录视图`（只有姓名、电话，**没有工资**）。 | **应用程序员、最终用户**      |
| **逻辑层** | Logical Level  | **模式**             | 描述数据库中存储**什么数据**以及数据间的**关系**。这是全局性的完整描述。 | `Student` 表包含：学号(INT)、姓名(VARCHAR)、年龄(INT)、系名(VARCHAR)。表之间通过外键关联。 | **数据库设计者、DBA**         |
| **物理层** | Physical Level | **内模式**           | 描述数据**实际上怎么存的**。不考虑逻辑关系，只考虑物理字节。 | 记录在磁盘上是顺序存储、B+树存储还是哈希存储？索引存在第几个文件？数据块大小是多少？ | **DBMS 内核开发者、资深 DBA** |

**数据独立性的实现机制（重点理解）：**

- **物理独立性**：
    - 架构：逻辑层 **独立于** 物理层。
    - 机制：当物理层改变（如把 B+树索引换成哈希索引），只需修改 **模式/内模式映像**（逻辑层到物理层的映射规则）。
    - 结果：逻辑层（表结构）看起来**没有任何变化**，应用程序自然不用改。

- **逻辑独立性**：
    - 架构：视图层 **独立于** 逻辑层。
    - 机制：当逻辑层改变（如在 `Student` 表增加一列 `Email`），只需修改 **外模式/模式映像**（视图层到逻辑层的映射规则），原有视图的定义可以保持不变。
    - 结果：使用原有视图的程序（比如只看姓名和年龄的旧应用），**感觉不到表结构变了**，不用改代码。

**总结这五页的逻辑链：**
1. **第一页** 列口号（四大特性）。
2. **第二页** 解释最复杂的那个口号（DBMS 统一控制）。
3. **第三页** 画架构图，表明 DBMS 在中间当管家。
4. **第四页** 抛出问题：为什么当管家就不用改程序了？
5. **第五页** 给出答案：因为管家提供了**三个层次的抽象视图**，屏蔽了底层的变动。
