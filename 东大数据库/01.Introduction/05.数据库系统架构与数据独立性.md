# 数据库系统架构与数据独立性

## 1. 核心挑战：数据独立性 (Data Independence)

在应用程序开发中，一个核心难题是如何实现**应用程序与数据存储的解耦（Separation）**。如果缺乏这种分离，底层数据的任何微小变动都会导致上层程序必须重写。

### 为什么需要数据独立性？

如果我们不引入抽象层，当发生以下底层变更时，程序员将被迫修改代码：

- **硬件变更**：例如更换硬盘驱动器。
- **存储结构变更**：例如将底层数据存储结构从哈希文件（Hash file）更改为B+树（B-tree）。
- **物理文件调整**：例如将数据拆分到两个物理文件中，或者将两个文件合并为一个。
- **数据类型变更**：例如将薪资字段的存储类型从整数（Integer）改为浮点数（Floating point）。
- **扩展需求**：例如为了支持新应用而添加新的数据字段。

**解决方案**：引入**抽象（Abstraction）**。通过分层架构，隐藏底层的复杂性和细节，对外只提供稳定的接口。

------

## 2. 核心概念：模式与实例 (Schema and Instance)

为了理解数据库的抽象，首先需要区分“静态结构”和“动态内容”。

- **模式 (Schema)**：
  - **定义**：指数据库的整体逻辑设计（Overall Design）。
  - **性质**：它是相对稳定的，不经常改变。
  - **类比**：类似于编程语言中的**类型（Type）**定义（例如 `class Student { ... }`）。它规定了数据的结构、约束和关系。
- **实例 (Instance)**：
  - **定义**：指特定时刻数据库中实际存储的数据集合。
  - **性质**：它是动态的，随着数据的增删改查时刻在变化。
  - **类比**：类似于编程语言中的**变量（Variable）**的值（例如 `Student s1 = new Student();`）。

------

## 3. 三级模式架构 (Three-Schema Architecture)

数据库系统通过三个层次的抽象来实现“数据视图（View of Data）”，这被称为三级模式架构。

### 3.1 物理层 / 内模式 (Physical Level / Internal Schema)

这是最低层次的抽象，最接近硬件。

- **功能**：描述数据在存储介质上**实际上是如何存储的**。
- **关注点**：复杂的底层数据结构和存取路径。
  - **存储布局**：例如，“将客户记录分为三个分区，分别存储在磁盘1、2、3上”。
  - **存取方法**：使用顺序访问（Sequential access）、哈希访问（Hash access）还是索引（Index）。
  - **优化技术**：数据压缩（Compression）、数据加密（Encryption）。
  - **记录详情**：字节偏移量、数据块大小等。
- **数量**：一个数据库只有一个物理模式。

### 3.2 逻辑层 / 概念模式 (Logical Level / Conceptual Schema)

这是中间层次的抽象，是数据库管理员（DBA）主要关注的层面。

- **功能**：描述数据库中**存储了什么数据（What data）以及数据之间的逻辑关系**。

- **抽象程度**：它对物理存储细节进行了隐藏，只关注逻辑结构。

- **描述内容**：

  - 所有数据的逻辑结构和特征。

  - 类似于高级编程语言（如Pascal或C）中的结构体（struct）或记录（record）定义。

  - **示例**：

    Plaintext

    ```
    Type customer = record
        name: string;
        street: string;
        city: integer;
    end;
    ```

- **数量**：一个数据库只有一个概念模式（它代表了全局的逻辑视图）。

### 3.3 视图层 / 外模式 (View Level / Sub Schema / External Schema)

这是最高层次的抽象，直接面向终端用户或应用程序。

- **功能**：仅描述整个数据库的**一部分**，即用户能看到和使用的那部分数据的逻辑结构。
- **目的**：
  - **简化交互**：系统非常复杂，大多数用户不需要（也不应该）关心整个逻辑模型，只需关注与其应用相关的子集。
  - **安全性 (Security)**：通过视图机制隐藏敏感数据。例如，财务部门可以看到员工薪资字段，但普通员工查询系统时，该字段对其实际上是不可见的。
- **数量**：一个数据库可以有**多个**视图模式，对应不同的用户组或应用场景。

------

## 4. 两种数据独立性

基于上述三层架构，通过在层与层之间建立**映射（Mapping）**，数据库系统实现了两种关键的独立性。这保证了当下一层发生变化时，上一层可以保持不变。

### 4.1 物理数据独立性 (Physical Data Independence)

- **定义**：应用程序（基于逻辑模式编写）不依赖于物理模式。当**物理模式（Physical Schema）**改变时，不需要重写应用程序。
- **机制**：
  - 应用程序依赖于逻辑模式。
  - 物理存储细节（如索引方式、磁盘分区、哈希算法）被隐藏在逻辑模式之下。
  - 当物理层改变时，只需调整“物理-逻辑”之间的映射，保持逻辑层不变。
- **现实意义**：极大地提高了系统的可维护性和性能调优的灵活性（DBA可以随意优化底层存储而不影响业务代码）。

### 4.2 逻辑数据独立性 (Logical Data Independence)

- **定义**：用户（基于视图模式）不依赖于逻辑模式。当**逻辑模式（Logical Schema）**改变时，用户视图和应用程序不需要改变。
- **机制**：
  - 用户通过视图层与数据交互。
  - 当逻辑层发生变化（例如表拆分、增加非空列但有默认值）时，只需调整“逻辑-视图”之间的映射（视图定义）。
- **难度**：逻辑数据独立性通常比物理数据独立性更难完全实现，因为逻辑结构的变化往往涉及业务语义的变更。

------

## 5. 架构总结：层级映射关系

整个架构形成了一个从上到下的闭环流转：

1. **Application（应用层）**：用户发出请求。
2. **View Level（视图层）**：系统根据**视图定义（View definitions）**解析请求，提供逻辑数据独立性。
3. **Logical Level（逻辑层）**：系统根据**逻辑模式（Logical schema）**理解全局数据结构，提供物理数据独立性。
4. **Physical Level（物理层）**：系统根据**物理模式（Physical schema）**在磁盘上执行实际的I/O操作。

这种分层设计是现代数据库管理系统（DBMS）的基石，它使得复杂的数据库系统变得可管理、可扩展且安全。
