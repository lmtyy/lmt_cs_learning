# 为什么要使用数据库系统：文件处理系统的六大缺陷

# (Drawbacks of File Processing Systems & Purpose of Database Systems)

在数据库系统普及之前，组织机构主要依赖文件系统来存储数据。随着应用需求的复杂化，这种“应用程序直接管理文件”的模式暴露出了六大严重的弊端，这也正是数据库系统（DBMS）诞生的根本原因。

## 1. 数据冗余与不一致 (Data Redundancy and Inconsistency)

### 核心痛点

由于不同部门独立开发应用，数据往往被重复存储在不同的文件中。

- **冗余 (Redundancy)**：同一份信息存储多份。
  - *场景*：**人事处**的文件中存储了员工的“姓名、年龄、单位、职称”。**财务处**的文件中也存储了员工的“姓名、单位、职称、工资”。
- **不一致 (Inconsistency)**：冗余存储导致数据同步困难。
  - *后果*：如果某员工职称晋升，人事处修改了记录，但财务处忘记修改，系统中的数据就会发生冲突，导致发错工资。

### 扩展知识

- **存储成本**：冗余数据浪费了大量的存储空间。
- **更新异常 (Update Anomaly)**：这是文件系统的典型特征，无法通过系统机制自动保证多副本数据的一致性。

## 2. 数据隔离 (Data Isolation)

### 核心痛点

数据分散在不同文件、不同格式中，导致跨业务整合极其困难。

- **格式壁垒**：文件格式不统一（文本、二进制、Excel等），且可能由不同语言（Java, Fortran, C++）生成。
- **集成噩梦**：
  - *场景*：银行系统中，**储蓄账户 (Saving)**、**支票账户 (Checking)** 和 **VIP客户名单** 存储在三个独立的文件中。
  - *问题*：如果要查询“所有拥有支票账户的VIP客户”，程序员必须编写复杂的代码来解析三种不同的文件格式并进行匹配，无法简单通过一条指令完成。

### 扩展知识

- **缺乏标准接口**：文件系统缺乏像 SQL 这样标准化的查询语言，导致数据像孤岛一样无法互通。

## 3. 数据访问困难 (Difficulty in Accessing Data)

### 核心痛点

缺乏灵活的、即席的（Ad-hoc）查询能力。

- **硬编码限制**：所有的查询逻辑都必须写死在程序代码中。
  - *场景*：你现有的程序只能“按邮编查询居民”。如果不修改代码，你就无法“查询年龄大于50岁的居民”。
- **响应滞后**：每当有一个新的查询需求（哪怕很简单），都需要程序员编写新程序、调试、编译。这无法满足现代企业对数据查询的实时性要求。

### 扩展知识

- **逻辑依赖性**：应用程序与数据结构紧密耦合，数据结构稍有变动，所有程序都得重写。

## 4. 原子性问题 (Atomicity of Updates)

### 核心痛点

系统无法保证复杂操作的完整性，容易导致数据处于“半完成”的错误状态。

- **原子性定义**：一个事务（Transaction）中的所有操作，要么全部成功，要么全部失败，不能停在中间。
- **故障场景**：银行转账 A 转给 B $100。
  1. 从 A 账户扣除 $100（A: 200 -> 100）。
  2. **[系统崩溃/断电]**。
  3. B 账户增加 $100 的操作未执行。
- **后果**：A 的钱少了，B 的钱没多，这 $100 凭空消失了。文件系统缺乏自动回滚（Rollback）机制来修复这种错误。

### 扩展知识

- **ACID 特性**：数据库通过事务日志（Log）和恢复机制（Recovery）来严格保证原子性。

## 5. 并发访问异常 (Concurrent Access Anomalies)

### 核心痛点

在多用户同时操作数据时，缺乏控制机制会导致数据错误。

- **性能需求**：大型系统必须允许并发访问以提高性能。
- **丢失修改 (Lost Update)**：
  - *场景*：账户余额 $2000。用户 A 取款 $500，用户 B 取款 $300。
  - *过程*：两人同时读取到 $2000。A 算出 $1500 并写入；B 算出 $1700 并写入（覆盖了 A 的结果）。
  - *结果*：余额变成了 $1700（错误），实际应为 $1200。

### 扩展知识

- **锁机制 (Locking)**：数据库通过行锁、表锁和隔离级别（Isolation Level）来解决并发冲突，而文件系统通常只能锁定整个文件，效率极低。

## 6. 安全性问题 (Security Problems) [新增点]

### 核心痛点

难以实现精细化的访问控制。

- **权限粒度粗**：文件系统通常只能控制“谁能读写整个文件”。
- **部分访问困难**：
  - *场景*：在一个包含“员工姓名、工资、家庭住址”的文件中。
  - *需求*：你想让普通员工能看到“姓名”和“住址”，但**绝不能**看到“工资”。
  - *困境*：在文件系统中，你要么给他整个文件的权限（泄露工资），要么完全不让他看（无法工作）。很难做到“只开放文件中的某几列数据”。

### 扩展知识

- **视图 (Views) 与 授权 (Grant/Revoke)**：数据库可以轻松创建“视图”来隐藏敏感字段，并对不同用户授予不同级别的增删改查权限。

### 总结归纳表

| **维度**       | **文件系统的表现 (缺陷)** | **数据库系统的解决方案**     |
| -------------- | ------------------------- | ---------------------------- |
| **数据一致性** | 冗余存储，易导致不一致    | 集中存储，减少冗余，约束机制 |
| **数据集成**   | 格式隔离，跨文件难以整合  | 统一模型，标准 SQL 接口      |
| **查询灵活性** | 需编写新程序，响应慢      | 灵活的 SQL 查询，即问即答    |
| **事务可靠性** | 故障可能导致数据损坏      | 事务机制 (ACID) 保证原子性   |
| **并发能力**   | 易出现“丢失修改”          | 并发控制，锁机制             |
| **安全性**     | 只能控制文件级权限        | 字段/行级精细权限控制        |
