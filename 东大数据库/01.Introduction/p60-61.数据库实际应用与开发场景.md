### 第一页：SQL 查询实例

> **PPT 内容提炼：**
> - **任务**：查询 `customer_id` 为 `192-83-7465` 的客户的**姓名**。
> - **SQL 语句**：
>   ```sql
>   SELECT customer.customer_name
>   FROM customer
>   WHERE customer.customer_id = '192-83-7465';
>   ```
> - **执行结果展示**：`customer` 表的完整数据，其中第一行（`192-83-7465, Johnson, ...`）被高亮标记。

**知识点深度剖析：**

#### 1. SQL 语句的逐子句解析

这是 SQL 查询的**最基本骨架**，由三个核心子句构成：

| 子句       | 关键词   | 作用                           | 本例中的具体含义                              |
| :--------- | :------- | :----------------------------- | :-------------------------------------------- |
| **SELECT** | `SELECT` | **投影**：指定要返回哪些列     | 只要 `customer_name` 这一列                   |
| **FROM**   | `FROM`   | **指定数据来源**：从哪个表里查 | 从 `customer` 表里查                          |
| **WHERE**  | `WHERE`  | **选择**：筛选符合条件的行     | 只保留 `customer_id = '192-83-7465'` 的那一行 |

**对应关系代数**：
- 这条 SQL 等价于关系代数中的 **π~customer_name~(σ~customer_id='192-83-7465'~(customer))**。
- 先做**选择**（`WHERE` 筛选行），再做**投影**（`SELECT` 筛选列）。

#### 2. 声明式语言的典型特征

回顾上一轮讲的**声明式 DML** 特点：
- **你只说“要什么”**：你要 `customer_id` 为某值的客户的姓名。
- **你不管“怎么找”**：你没有写任何关于“用索引还是全表扫描”、“先查哪一列”的指令。

**DBMS 在幕后做了什么？**
1. **解析 SQL**，检查语法和语义（`customer` 表存在吗？`customer_id` 列存在吗？）。
2. **查询优化器**生成执行计划：
    - 如果 `customer_id` 上有**主键索引**，则直接用索引定位到唯一一行，**1 次 I/O** 即可完成。
    - 如果没有索引，则执行**全表扫描**，逐行比对 `customer_id`，需要读取表中所有数据页。
3. **执行引擎**按计划读取数据，返回结果。

**关键启示**：同样的 SQL，**底层执行方式可能天差地别**，但对用户完全透明。

#### 3. 执行结果的数据类型

- 查询返回的是一个**单行单列**的结果（标量值 `Johnson`）。
- 在关系模型中，任何查询结果**本身也是一个关系**（表）。即使是单行单列，也是一个 1×1 的表。

#### 4. 图示的作用

PPT 下方展示了完整的 `customer` 表数据，并将 `Johnson` 那一行高亮。这直观地说明了：
- `WHERE` 子句是如何**过滤行**的。
- `SELECT` 子句是如何**过滤列**的（只显示了姓名，但图中为了方便理解展示了整行）。

### 第二页：应用程序如何访问数据库

> **PPT 内容提炼：**
> - 应用程序通常通过以下**两种方式**访问数据库：
>     1. **语言扩展**：允许在编程语言中**嵌入 SQL**（嵌入式 SQL）。
>     2. **应用程序接口**：如 **ODBC / JDBC**，允许将 SQL 查询发送到数据库。
> - 图示标注了 **ODBC**（Open Database Connectivity）和 **JDBC**（Java Database Connectivity）。

**知识点深度剖析：**

#### 1. 为什么需要专门的“访问方式”？

- **SQL 不是通用编程语言**。SQL 长于数据操作，但弱于复杂业务逻辑（如循环、条件判断、图形界面交互）。
- 因此，实际应用是由 **Java / Python / C# / PHP** 等**宿主语言**编写，它们需要一种机制来**调用 SQL 并获取结果**。

#### 2. 方式一：嵌入式 SQL —— 将 SQL 写在代码里

| 维度               | 说明                                                         |
| :----------------- | :----------------------------------------------------------- |
| **核心思想**       | 将 SQL 语句**直接嵌入**到 C、COBOL、Java 等宿主语言的源代码中。 |
| **前置处理**       | 需要一个**预编译器**，将嵌入的 SQL 语句转换为宿主语言可识别的函数调用。 |
| **典型语法**       | 以 `EXEC SQL` 开头。                                         |
| **示例（C 语言）** | `EXEC SQL SELECT customer_name INTO :cust_name FROM customer WHERE customer_id = :cust_id;` |
| **数据传递**       | 通过**宿主变量**（如 `:cust_name`）在 SQL 和程序变量之间传递数据。 |
| **优缺点**         | 优点：代码紧凑，SQL 与业务逻辑写在一起。<br>缺点：与具体 DBMS 耦合较紧，移植性差。 |

**嵌入式 SQL 的完整工作流程**：
```
源代码（含 EXEC SQL） → [预编译器] → 纯宿主语言代码 → [宿主语言编译器] → 可执行程序
```

#### 3. 方式二：API 接口 —— 通过函数调用发送 SQL

这是**当前最主流**的方式。应用程序不直接嵌入 SQL，而是调用数据库厂商或标准组织提供的**函数库**。

##### 3.1 ODBC

| 维度         | 说明                                                         |
| :----------- | :----------------------------------------------------------- |
| **全称**     | Open Database Connectivity                                   |
| **定位**     | **C 语言级别的标准 API**，由微软提出，后成为工业标准。       |
| **核心思想** | 提供一套**统一的函数接口**，应用程序调用 ODBC 函数，ODBC **驱动程序**负责翻译成特定 DBMS 的方言。 |
| **工作方式** | 应用程序 → ODBC 驱动管理器 → 特定数据库的 ODBC 驱动 → 数据库。 |
| **优点**     | **数据库无关性**。更换底层数据库（从 MySQL 换到 Oracle），只需更换驱动，应用程序代码**无需修改**。 |

##### 3.2 JDBC

| 维度             | 说明                                                         |
| :--------------- | :----------------------------------------------------------- |
| **全称**         | Java Database Connectivity                                   |
| **定位**         | **Java 语言的标准数据库访问 API**。                          |
| **核心思想**     | 与 ODBC 类似，但专为 Java 设计。通过 `java.sql` 包提供接口，各数据库厂商提供对应的 **JDBC 驱动**（JAR 包）。 |
| **典型代码示例** | `Connection conn = DriverManager.getConnection(url);`<br>`Statement stmt = conn.createStatement();`<br>`ResultSet rs = stmt.executeQuery("SELECT ...");` |
| **优点**         | 跨平台、跨数据库，与 Java 语言无缝集成。                     |

#### 4. ODBC 与 JDBC 的关系

| 对比维度 | **ODBC**                                             | **JDBC**                          |
| :------- | :--------------------------------------------------- | :-------------------------------- |
| 目标语言 | **C / C++**                                          | **Java**                          |
| 跨平台性 | 主要针对 Windows，但也有 Unix/Linux 实现             | **天然跨平台**（Java 虚拟机保证） |
| 桥接     | 可以通过 **JDBC-ODBC 桥**（已淘汰）让 Java 调用 ODBC | 纯 Java 实现，不依赖本地代码      |
| 当前地位 | 在非 Java 的 Windows 应用中仍广泛使用                | **Java 企业应用的事实标准**       |

#### 5. 当前主流开发中的实际做法

今天大多数开发者不会直接写 JDBC 底层代码，而是使用**更高级的持久层框架**，但这些框架底层**都基于 JDBC**：

| 框架/技术                             | 底层依赖                                       | 说明                                       |
| :------------------------------------ | :--------------------------------------------- | :----------------------------------------- |
| **JPA / Hibernate**                   | JDBC                                           | 对象关系映射，将 Java 对象自动同步到数据库 |
| **MyBatis**                           | JDBC                                           | SQL 映射框架，将 SQL 写在 XML 中           |
| **Python 的 SQLAlchemy / Django ORM** | 各数据库的 Python 驱动（如 psycopg2, PyMySQL） | 原理类似 JDBC                              |
| **Node.js 的 Sequelize / TypeORM**    | 各数据库的 Node.js 驱动                        | 原理类似 JDBC                              |

**完整图景**：
1. 数据库设计者用 **DDL** 建表（`CREATE TABLE customer...`）。
2. 应用程序员用 **宿主语言（Java/Python）** 编写业务逻辑。
3. 在需要访问数据时，通过 **JDBC/ODBC** 发送 **DML（SQL）** 语句。
4. **DBMS** 执行 SQL，将结果返回给应用程序。
5. 应用程序将结果展示给用户（如网页表格）。

**一句话总结**：
> **SQL 是人与数据库对话的语言；JDBC/ODBC 是程序与数据库握手的协议。**
