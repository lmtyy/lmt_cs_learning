### **第一页：建立视图——行列子集视图**

**内容原文：**
- 从单个基本表导出
- 只是去掉了基本表的某些行和某些列
- 保留了主码

**详细讲解：**

1. **行列子集视图的定义**  
   “行列子集视图”是一种最简单的视图形式。它直接从**单个基本表**中导出，通过选择（行筛选）和投影（列筛选）操作，生成一个原表的子集。具体来说：
   - **去掉了某些行**：通过 `WHERE` 条件过滤掉不满足条件的行。
   - **去掉了某些列**：通过 `SELECT` 子句只保留需要的列。
   - **保留了主码**：视图仍然包含原表的主码列（或至少主码列在视图中），这样视图中的每一行都能唯一对应到基表中的一行。

2. **示例**  
   之前定义的 `sf_stu` 视图就是一个行列子集视图：
   ```sql
   CREATE VIEW sf_stu AS
   SELECT Sid, Sname, Sage
   FROM Student
   WHERE Sdept = '软件学院';
   ```
   - 从单个表 `Student` 导出。
   - 去掉了非软件学院的行（行筛选）。
   - 去掉了 `Sdept` 等其他列（列投影）。
   - 保留了主码 `Sid`，因此视图中每一行仍能唯一对应基表的一行。

3. **行列子集视图的特点**  
   - 结构简单，与基表保持紧密的对应关系。
   - 通常**支持更新操作**（如 `INSERT`、`UPDATE`、`DELETE`），因为每行都能唯一地映射回基表的一行，更新语义明确。
   - 是数据库可更新视图的典型代表。

4. **为什么保留主码很重要？**  
   主码的存在确保了视图中每一行的唯一性，使得数据库能够准确地将对视图的更新转换为对基表对应行的更新。如果视图不包含主码，那么某些行可能无法唯一确定基表中的原始行，从而导致更新操作被禁止。

### **第二页：更新视图的限制（以 DB2 为例）**

**内容原文：**
- 允许对行列子集视图进行更新
- 对其他类型视图的更新不同系统有不同限制

DB2对视图更新的限制：
(1) 若视图是由两个以上基本表导出的，则此视图不允许更新。
(2) 若视图的字段来自字段表达式或常数，则不允许对此视图执行INSERT和UPDATE操作，但允许执行DELETE操作。
(3) 若视图的字段来自集函数，则此视图不允许更新。
(4) 若视图定义中含有GROUP BY子句，则此视图不允许更新。
(5) 若视图定义中含有DISTINCT短语，则此视图不允许更新。

**详细讲解：**

1. **前言**  
   - 行列子集视图是可更新的，但其他类型的视图（如多表连接视图、分组视图、包含表达式的视图等）在不同数据库系统中的处理方式不同。DB2 是一个典型的关系数据库管理系统，它对视图更新的限制非常严格，PPT 列举了 DB2 的 7 条限制（本页包含前 5 条，第三页包含后 2 条）。
   - 这些限制的核心原因是：当视图的定义复杂到一定程度时，无法将视图的更新操作唯一地、有意义地转换为对基表的更新。

2. **限制 (1)：由两个以上基本表导出的视图不允许更新**  
   - 若视图是通过连接两个或多个基本表得到的（例如 `FROM table1, table2 WHERE ...`），那么视图中的一行可能对应多个基表中的多行组合。当用户试图更新视图时，数据库无法确定应该修改哪个基表的哪一行，或者修改后的值如何分配到各个基表，因此禁止更新。
   - **例外**：某些数据库（如 Oracle、PostgreSQL）允许对某些可更新的连接视图进行更新，但必须满足严格的条件（例如保留键表、只影响一个基表等）。DB2 则直接禁止所有多表视图的更新。

3. **限制 (2)：视图字段来自字段表达式或常数，不允许 INSERT 和 UPDATE，但允许 DELETE**  
   - 如果视图的某一列是通过表达式计算得到的（例如 `salary*1.1` 或 `'常量'`），那么这一列在基表中并不直接存在。当用户通过视图执行 `INSERT` 或 `UPDATE` 时，数据库无法确定应该将新值存储到基表的哪个字段（因为表达式列是虚拟的）。因此，禁止对这类视图进行插入和修改。
   - 但是，**允许 DELETE** 操作，因为删除操作只涉及行的存在性，不涉及具体列值的计算。删除时，数据库只需要确定要删除的基表行（通过视图条件定位），而不需要关心表达式列的值。

4. **限制 (3)：视图字段来自集函数（聚合函数），不允许更新**  
   - 如果视图包含聚合函数（如 `AVG`、`SUM`、`COUNT` 等），则视图中的数据是经过汇总计算的，与基表的行没有一一对应关系。例如，一个显示每个部门平均工资的视图，其每一行对应一个部门的多行员工记录。对这样的视图执行更新（如修改平均工资）无法唯一地转换为对基表的具体行的修改，因此禁止所有更新操作。

5. **限制 (4)：视图定义中含有 GROUP BY 子句，不允许更新**  
   - `GROUP BY` 将基表行分组，每组生成一行视图数据。与包含聚合函数的视图类似，分组视图中的行不直接对应基表的单一行，而是对应一组行。因此，对分组视图的更新无法确定如何映射到基表，故禁止更新。

6. **限制 (5)：视图定义中含有 DISTINCT 短语，不允许更新**  
   - `DISTINCT` 会去除重复行，导致视图中的行可能与基表中的多行对应（例如，投影某些列后出现重复，只保留一行）。这种视图的行与基表行之间不是一一对应关系，因此无法唯一确定更新目标，禁止更新。

### **第三页：更新视图的限制（续）**

**内容原文：**
(6) 若视图定义中有嵌套查询，并且内层查询的FROM子句中涉及的表也是导出该视图的基本表，则此视图不允许更新。

例：将SC中成绩在平均成绩之上的元组定义成一个视图  
**GOOD_SC**  
CREATE VIEW GOOD_SC AS  
SELECT Sno, Cno, Grade  
FROM SC  
WHERE Grade > (SELECT AVG(Grade) FROM SC);

(7) 一个不允许更新的视图上定义的视图也不允许更新。

**详细讲解：**

1. **限制 (6)：嵌套查询涉及同一基表，不允许更新**  
   - 这种视图的定义中包含了子查询，并且子查询的 `FROM` 子句引用了与外部查询相同的基表（即导出该视图的基本表）。这样的视图通常称为**递归依赖视图**或**自引用视图**。
   - 以例子中的 `GOOD_SC` 视图为例：
     ```sql
     CREATE VIEW GOOD_SC AS
     SELECT Sno, Cno, Grade
     FROM SC
     WHERE Grade > (SELECT AVG(Grade) FROM SC);
     ```
     - 该视图筛选出成绩高于全部成绩平均值的选课记录。
     - 子查询 `(SELECT AVG(Grade) FROM SC)` 与外部查询使用了同一个表 `SC`。
   - **为什么不允许更新？**  
     因为当用户通过该视图更新数据时，视图的定义依赖于整个表的当前状态（平均值）。如果允许更新，那么更新操作会改变基表的数据，进而改变平均值，从而导致视图本身的内容发生动态变化，造成循环依赖和不确定性。例如，如果通过视图将某个成绩降低，可能会使一些原本在视图中的行不再满足条件，而原本不在视图中的行可能进入视图，这种自反性使得更新语义无法稳定定义。因此，DB2 禁止对此类视图进行更新。

2. **限制 (7)：不允许更新的视图上定义的视图也不允许更新**  
   - 如果视图 `V2` 是基于另一个视图 `V1` 定义的，而 `V1` 本身是不可更新的，那么 `V2` 也是不可更新的。这是因为 `V2` 的更新最终也要转换为对基表的操作，但由于底层视图 `V1` 已经不可更新，`V2` 的更新也就无法实现。
   - 换句话说，**不可更新性会沿着视图依赖链传递**。这个规则符合逻辑：如果一个视图的底层数据源已经无法支持更新，那么基于它的任何上层视图自然也无法支持更新。

3. **总结**  
   DB2 的这 7 条限制覆盖了绝大多数复杂视图的情况，确保了只有最简单、最直接的行列子集视图才是可更新的。其他数据库系统（如 Oracle、SQL Server、MySQL）可能对某些情况放宽限制，但基本原理相似：只有那些能够唯一且无歧义地转换为基表更新的视图才被认为是可更新的。

通过这三页 PPT，我们了解了：
- 什么是行列子集视图及其可更新性。
- DB2 中视图更新的 7 条具体限制及其背后的逻辑（无法唯一映射到基表、存在歧义或循环依赖）。
- 实际应用中，设计视图时应考虑其可更新需求，避免创建不可更新的复杂视图，或明确将其作为只读视图使用。
