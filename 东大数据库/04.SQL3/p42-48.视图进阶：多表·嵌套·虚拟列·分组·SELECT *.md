这几页PPT深入讲解了视图创建的各种高级用法和注意事项，包括列名指定、多表连接视图、视图的嵌套、虚拟列、分组视图以及使用 `SELECT *` 的潜在问题。下面逐页详细解析。

---

### 第一页：组成视图的属性列名

这一页说明了在创建视图时，何时需要**明确指定视图的列名**，以及何时可以省略。

- **全部省略**：
  - 如果视图的列直接来源于子查询中 `SELECT` 的目标列，且这些列名是简单字段名（无重复、无表达式），则可以省略视图的列名列表，视图会自动采用子查询中的列名。
  - 例如：`CREATE VIEW sf_stu AS SELECT sid, sname, sage FROM student WHERE ...`，视图列名就是 `sid`, `sname`, `sage`。

- **必须明确指定视图列名的三种情况**：
  1. **某个目标列是聚集函数或列表表达式**：例如 `AVG(grade)` 或 `2012 - sage`，这类表达式本身没有列名，必须为视图的该列指定一个名称。
  2. **多表连接时选出了几个同名列作为视图的字段**：例如从 `student` 和 `sc` 表中都选了 `sno`，但这两个 `sno` 实际含义相同，合并为一列时需指定名称，或者如果保留了多个同名列（虽不常见），也需要区分命名。
  3. **需要在视图中为某个列启用新的更合适的名字**：比如将英文列名改为中文，或使列名更具业务含义。

---

### 第二页：包含视图属性名的示例

- **示例**：
  ```sql
  CREATE VIEW sf_stu (学号, 姓名, 年龄) AS
  SELECT sid, sname, sage
  FROM student
  WHERE sdept = '软件学院'
  WITH CHECK OPTION;
  ```
- **解析**：
  - 这里明确指定了视图的列名为中文：`学号, 姓名, 年龄`，对应子查询中的 `sid, sname, sage`。
  - 这样做的好处是让视图更易读，尤其当基表列名是英文缩写时，视图可以提供更直观的列名。
  - 同时使用了 `WITH CHECK OPTION`，确保通过视图的更新操作不会破坏“软件学院”这一条件。

---

### 第三页：基于多个基表的视图

- **示例**：建立软件学院选修了1号课程的学生视图。
  ```sql
  CREATE VIEW sf_S1 (Sno, Sname, Grade) AS
  SELECT Student.Sno, Sname, Grade
  FROM Student, SC
  WHERE Student.Sno = SC.Sno
    AND Sdept = '软件学院'
    AND SC.Cno = '1';
  ```
- **解析**：
  - 视图基于两个基表：`Student` 和 `SC`，通过连接获取数据。
  - 视图中包含学号、姓名和成绩，隐藏了其他字段（如院系、课程号等）。
  - 这种视图将复杂的连接查询封装起来，用户可以直接查询 `sf_S1` 获得所需数据，无需关心表连接细节。
  - **注意**：多表视图的更新通常受限（一般不允许 `INSERT`、`DELETE`、`UPDATE`），因为可能无法唯一确定如何映射到基表。

---

### 第四页：基于视图的视图

- **示例**：在上一视图 `sf_S1` 的基础上，建立成绩在90分以上的学生视图。
  ```sql
  CREATE VIEW sf_S2 AS
  SELECT Sno, Sname, Grade
  FROM sf_S1
  WHERE Grade >= 90;
  ```
- **解析**：
  - 视图可以建立在其他视图之上，形成视图的层次结构。
  - `sf_S2` 的数据来源于视图 `sf_S1`，而 `sf_S1` 又来源于基表。
  - 这种嵌套使用进一步细化了数据展示，但也会增加查询的解析开销（视图解析可能多层嵌套）。
  - 同样，多层视图的更新限制更加严格。

---

### 第五页：带表达式（虚拟列）的视图

- **示例**：定义一个反映学生出生年份的视图。
  ```sql
  CREATE VIEW BT_S (Sno, Sname, Sibirth) AS
  SELECT Sno, Sname, 2012 - Sage
  FROM Student;
  ```
- **重要概念——虚拟列**：
  - `Sibirth` 列是通过表达式 `2012 - Sage` 计算得到的，在基本表 `Student` 中并不存在此列。
  - 这种列称为“虚拟列”或“派生列”，它只在视图的逻辑结构中存在，数据是动态计算出来的，不占用物理存储。
  - **优点**：可以减少数据冗余，避免在基表中存储可以由其他列计算出的信息。
  - **注意**：包含虚拟列的视图通常是**只读**的，因为数据库无法确定如何将对该列的更新映射回基表（例如，修改出生年份应该对应修改年龄还是其他字段？）。

---

### 第六页：分组视图

- **示例**：将学生的学号及他的平均成绩定义为一个视图。
  ```sql
  CREATE VIEW S_G (Sno, Gavg) AS
  SELECT Sno, AVG(Grade)
  FROM SC
  GROUP BY Sno;
  ```
- **解析**：
  - 视图使用了分组和聚合函数 `AVG(Grade)`，为每个学生生成一条平均成绩记录。
  - 这种视图通常也是**只读**的，因为分组后的数据与基表行没有一一对应关系，无法执行更新操作。
  - 分组视图对于统计分析非常有用，可以简化复杂的分组查询。

---

### 第七页：以 `SELECT *` 方式创建视图的缺点

- **示例**：将Student表中所有女生记录定义为一个视图。
  ```sql
  CREATE VIEW F_Student (F_Sno, name, sex, age, dept) AS
  SELECT *
  FROM Student
  WHERE Sex = '女';
  ```
- **缺点分析**：
  - **可扩充性差**：视图使用了 `SELECT *`，意味着视图的列与基表的列完全绑定，且顺序也一致。
  - **修改基表结构后的风险**：
    - 如果在 `Student` 表中增加新列，视图并不会自动包含新列，因为视图的定义在创建时已经固定（`*` 会被解析为当时的所有列）。
    - 如果删除或重命名基表中的某列，视图可能会失效，因为视图的列与基表的对应关系被破坏。
    - 如果改变基表的列顺序，视图的列顺序也会相应改变，可能影响依赖该视图的应用程序。
  - **建议**：尽量避免在视图中使用 `SELECT *`，而应显式列出需要的列名，以增强视图的稳定性和可维护性。

### 总结

这几页PPT涵盖了视图创建的多种高级场景：

1. **列名指定**：掌握何时必须指定列名。
2. **多表视图**：封装连接查询。
3. **视图嵌套**：在视图上构建新视图。
4. **虚拟列**：通过表达式生成计算列。
5. **分组视图**：使用聚合和分组。
6. **避免 `SELECT *`**：保证视图的稳定性。

这些知识点帮助你在实际应用中灵活运用视图，同时注意视图的更新限制和可维护性问题。
