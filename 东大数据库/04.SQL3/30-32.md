## 第一页：使用两条UPDATE语句（顺序很重要）

### 业务需求
- 对于账户余额大于$10,000的账户，增加6%（即乘以1.06）。
- 对于余额小于等于$10,000的账户，增加5%（即乘以1.05）。
- 表结构：`account(account_number, branch_name, balance)`

### 实现方式
```sql
-- 第一条：处理高余额账户
update account
set balance = balance * 1.06
where balance > 10000;

-- 第二条：处理低余额账户
update account
set balance = balance * 1.05
where balance ≤ 10000;
```

### 知识点与问题分析
1. **顺序的重要性**  
   这两条语句的执行顺序至关重要。如果先执行第二条（低余额更新），再执行第一条（高余额更新），会出现什么问题？
   - 假设有一个账户余额恰好为$10,000，它本应属于低余额（≤10000），享受5%增幅。
   - 若先执行第二条，该账户被乘以1.05，余额变为$10,500。
   - 接着执行第一条时，`balance > 10000`条件现在为真（因为$10,500 > 10000），于是它又被乘以1.06，最终余额变为$10,500 * 1.06 = $11,130，相当于被增加了两次，违背了业务规则。
   - 反之，如果先执行第一条（高余额更新），再执行第二条（低余额更新），则高余额账户只被更新一次（因为更新后仍大于10000，但第二条条件`≤10000`不满足），低余额账户也只被更新一次。**因此，正确的顺序必须是先高后低**。

2. **边界值处理**  
   需要注意`>`和`≤`的边界。如果条件写成`balance >= 10000`和`balance < 10000`，则余额等于10000的账户会被分到高余额组，这取决于业务定义。示例中明确用了`>10000`和`≤10000`，所以10000属于低余额。

3. **潜在风险**  
   用两条语句实现存在逻辑上的脆弱性：必须确保顺序正确，且不能有并发干扰。如果有人在两条语句之间插入其他操作，也可能导致数据不一致。因此，这种方式虽然直观，但不够健壮。

## 第二页：使用CASE语句实现条件更新（推荐做法）

### 实现方式
```sql
update account
set balance = case
                 when balance <= 10000 then balance * 1.05
                 else balance * 1.06
              end;
```

### 知识点与优势
1. **CASE语句在UPDATE中的用法**  
   - `CASE`在SQL中是一个条件表达式，可以在`SET`子句中直接使用。它根据条件返回不同的值。
   - 语法结构：
     ```sql
     CASE
         WHEN condition1 THEN result1
         WHEN condition2 THEN result2
         ...
         ELSE resultN
     END
     ```
   - 在这里，`balance`的值根据原值是否≤10000被分别计算为乘以1.05或1.06。

2. **原子性保证**  
   - 整个更新操作在一个语句中完成，数据库会对每一行独立判断条件，然后一次性更新。**不存在多条语句顺序问题**，因为每一行的更新都基于其原始值独立计算，并且整个操作是原子的（要么全部成功，要么全部失败）。
   - 避免了先更新某行后导致条件变化而影响其他行的风险。

3. **性能与可读性**  
   - 单条语句通常比两条语句执行效率更高（减少了一次表扫描或索引查找）。
   - 逻辑更清晰，易于维护。当条件复杂时，CASE语句的可读性优于多个分散的UPDATE。

4. **注意事项**  
   - `CASE`语句中的条件是按顺序评估的，一旦找到满足的条件，后续条件不再检查。示例中只有一个`WHEN`和一个`ELSE`，所以等价于二分支。
   - 如果分支较多，要注意条件的互斥性和顺序，避免逻辑重叠。

### 对比两条UPDATE语句
| 方面       | 两条UPDATE                    | 单条UPDATE + CASE |
| ---------- | ----------------------------- | ----------------- |
| 逻辑复杂度 | 需人工保证顺序                | 无需考虑顺序      |
| 原子性     | 非原子，中间可能被干扰        | 原子操作          |
| 性能       | 两次扫描/更新（可能更大开销） | 一次扫描/更新     |
| 可维护性   | 条件变化时需修改多条语句      | 只需修改一条语句  |
| 边界值处理 | 容易出错，需严格定义条件      | 在CASE中自然处理  |

因此，**推荐使用CASE语句实现条件更新**。

## 第三页：数据修改操作总结

### 内容回顾
- **INSERT**：插入数据。
  - 单行插入：`insert into r (A1, A2, ...) values (v1, v2, ...);`
  - 批量插入：`insert into r (A1, A2, ...) select ...` 或插入多行值列表。
- **DELETE**：删除数据。
  - `delete from r where P;` 省略WHERE则删除所有行。
- **UPDATE**：更新数据。
  - `update r set A1 = expr1, A2 = expr2 where P;`

### 补充说明
1. **INSERT**  
   - 可以指定列列表，若不指定则需为所有列按顺序提供值。
   - 批量插入常使用子查询从其他表获取数据，实现表间数据复制。

2. **DELETE**  
   - 删除操作通常不可逆，务必先确认WHERE条件。
   - 删除全部数据可使用`TRUNCATE TABLE`（快速但不可回滚），但PPT未涉及。

3. **UPDATE**  
   - 可同时更新多列。
   - 表达式可以包含算术运算、函数、甚至标量子查询。
   - 使用CASE语句实现复杂条件分支更新是高级技巧。

### 总结
这三类操作构成了DML（数据操作语言）的核心，掌握它们的语法和注意事项是数据库应用的基础。特别是条件更新时，合理运用CASE语句可以写出更健壮、高效的SQL。
