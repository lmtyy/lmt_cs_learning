这四页PPT系统地介绍了数据库中的**视图（View）**概念，从为什么需要视图、视图的定义、本质特点到具体语法，层层递进。下面为你详细讲解每页的核心知识点。

---

### 第一页：为什么需要视图？

-   **核心思想**：并非所有用户都需要看到数据库中的全部数据（即所有基本表）。出于**安全性**、**简化操作**或**数据抽象**的考虑，我们往往希望只向特定用户展示部分数据。
-   **举例**：
    -   某个用户需要查询客户的姓名、贷款编号和支行名称，但他**不应该看到贷款的金额**（敏感信息）。
    -   如果直接让他访问底层的 `borrower` 和 `loan` 表，他就能看到所有字段（包括贷款金额），这不符合权限控制的要求。
-   **解决方案**：创建一个只包含所需字段的“虚拟表”，即**视图**。该视图由以下查询定义：
    ```sql
    SELECT customer_name, borrower.loan_number, branch_name
    FROM borrower, loan
    WHERE borrower.loan_number = loan.loan_number;
    ```
    -   这个查询连接了 `borrower` 和 `loan` 表，只选出需要的三列，隐藏了 `loan` 表中的 `amount` 列。
-   **知识点**：视图的第一大作用就是**提供安全性和数据屏蔽**，让不同用户只能看到他们被允许看到的数据。

---

### 第二页：视图的定义与本质

-   **正式定义**：视图是一个**保存了的 SQL SELECT 语句的文本**，这个文本可以被当作数据源在查询中引用。
-   **虚表（Virtual Tables）**：视图常被称为“虚表”。为什么叫虚表？
    -   因为它在**逻辑上**像一张表（有行有列，可以对其进行查询操作），但**物理上**并不像基本表那样在磁盘上存储实际的数据。
    -   它只是数据库里保存的一条**查询语句**的定义。
-   **子模式（Sub-schema）**：在数据库的三级模式结构中，视图属于**外模式**（用户模式），是用户看到的数据视图，是全局逻辑模式（基本表）的一个子集或重组。因此视图也称为“子模式”。

---

### 第三页：创建视图的语法

-   **SQL 语法**：
    ```sql
    CREATE VIEW view_name AS
    <query expression>;
    ```
    -   `view_name`：你给视图起的名字。
    -   `<query expression>`：任何合法的 SQL 查询语句（SELECT 语句）。这个查询定义了视图的结构（列名、数据类型）和数据来源。
-   **示例**：结合第一页的需求，创建视图：
    ```sql
    CREATE VIEW customer_loan_info AS
    SELECT customer_name, borrower.loan_number, branch_name
    FROM borrower, loan
    WHERE borrower.loan_number = loan.loan_number;
    ```
    -   之后，用户就可以像查询普通表一样查询这个视图：
        ```sql
        SELECT * FROM customer_loan_info;
        ```
    -   用户完全不知道底层的 `borrower` 和 `loan` 表，也接触不到贷款金额字段。

---

### 第四页：视图的特点（重点）

1.  **虚表**：视图是从一个或几个基本表（或视图）导出的表。它本身不独立存在于物理存储中。
2.  **只存储定义，不存储数据**：数据库中只保存视图的定义（即 SELECT 语句的文本）。视图对应的数据仍然存储在原始基本表中。
3.  **数据动态更新**：当基本表中的数据发生变化时（插入、删除、修改），从视图中查询出的结果也会**自动随之改变**。因为每次查询视图时，数据库都会重新执行视图定义中的 SELECT 语句，从基表中获取最新数据。
    -   例如，在 `loan` 表中修改了一笔贷款的金额，虽然视图并不包含金额列，但如果视图包含了其他字段，查询结果也会反映出基表的最新记录（如新增或删除的贷款记录）。如果视图包含金额，那金额也会实时更新。
    -   **结论**：视图始终与基表数据保持同步，这是虚表的核心特性。

---

### 补充：视图的其他优点

-   **简化复杂查询**：可以将复杂的多表连接查询、嵌套查询定义为视图，用户每次只需简单查询视图即可，无需重复编写复杂 SQL。
-   **逻辑数据独立性**：当基表结构发生变化时（如拆分表、重命名列），可以通过修改视图的定义来保持用户查询不变，从而屏蔽变化对应用程序的影响。
-   **权限控制更精细**：可以授予用户对视图的访问权限，而不授予对基表的权限，实现列级或行级的安全控制。

### 注意事项

-   **更新限制**：并非所有视图都支持更新操作（INSERT、UPDATE、DELETE）。通常，如果视图来自单个表且不包含聚合、DISTINCT、GROUP BY 等，可能可以更新。但一般建议将视图主要用于查询，更新操作直接在基表上进行或通过触发器处理。

通过这四页PPT，你应该已经掌握了视图的基本概念、创建方法、本质特点以及其主要应用场景。视图是数据库管理中实现安全、简化、抽象的重要工具。
