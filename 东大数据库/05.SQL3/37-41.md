### 第一页：视图的功能框架

这一页是目录性质的提纲，列出了视图相关的四个核心操作层面：

1.  **Define view（定义视图）**：如何创建视图。
2.  **Query view（查询视图）**：创建后如何像查表一样使用视图。
3.  **Update view（更新视图）**：对视图进行插入、修改、删除操作时的规则与限制。
4.  **The function of view（视图的功能）**：为什么要用视图（安全性、简化查询、逻辑独立性等）。

后续几页主要围绕 **定义视图** 和 **更新视图的约束** 展开。

### 第二页：创建视图的语句格式与执行原理

-   **语句格式**：
    ```sql
    CREATE VIEW <视图名> [(<列名> [, <列名>]...)]
    AS <子查询>
    [WITH CHECK OPTION];
    ```
    -   **视图名**：给视图起的名称。
    -   **列名列表**（可选项）：为视图的每一列指定名称。如果省略，则直接使用子查询中的列名或字段名。
    -   **子查询**：任何合法的 `SELECT` 语句，定义了视图的数据来源和结构。
    -   **WITH CHECK OPTION**：可选项，对视图更新操作的约束（后面详细讲）。

-   **RDBMS 的执行原理**：
    -   **执行 `CREATE VIEW` 时**：数据库管理系统（RDBMS）**并不执行** `AS` 后面的 `SELECT` 语句。
    -   **实际动作**：只是将视图的定义（即这个 `SELECT` 语句的文本）**存入数据字典**（系统表）中。
    -   **查询视图时**：当用户对视图发起查询（`SELECT * FROM 视图名`）时，RDBMS 才会从数据字典中取出视图定义，将其与用户的查询结合，转化为对基本表的查询，然后执行并返回结果。

-   **知识点**：这进一步印证了视图是“虚表”的本质——**只存定义，不存数据**。

---

### 第三页：创建视图的简单示例

-   **示例 1**：建立软件学院学生的视图。
    ```sql
    CREATE VIEW sf_stu AS
    SELECT sid, sname, sage
    FROM student
    WHERE sdept = '软件学院';
    ```
-   **说明**：
    -   这里省略了视图的列名列表，因此视图 `sf_stu` 的列名自动采用子查询中的列名：`sid`、`sname`、`sage`。
    -   这个视图对用户隐藏了学生的其他信息（如性别、生日等），只显示软件学院学生的部分字段。
-   **作用**：实现了**行级**（只显示软件学院学生）和**列级**（只显示部分列）的数据安全。

---

### 第四页：`WITH CHECK OPTION` 的作用

-   **示例 2**：同样建立软件学院学生的视图，但增加了 `WITH CHECK OPTION`。
    ```sql
    CREATE VIEW sf_stu AS
    SELECT sid, sname, sage
    FROM student
    WHERE sdept = '软件学院'
    WITH CHECK OPTION;
    ```
-   **`WITH CHECK OPTION` 的核心定义**：
    -   对视图执行 `UPDATE`、`INSERT`、`DELETE` 操作时，RDBMS 会自动检查这些操作，保证被修改、插入或删除的行**满足视图定义中的谓词条件**（即子查询中的 `WHERE` 条件）。
-   **为什么需要它？**
    -   如果没有这个选项，可能会通过视图修改数据，导致修改后的数据**从视图中“消失”**（即不再符合视图的定义），造成数据不一致或逻辑混乱。
    -   `WITH CHECK OPTION` 就是为了保证**透过视图操作的数据，在操作后仍然属于该视图**。

---

### 第五页：`WITH CHECK OPTION` 的具体行为

这一页通过 `sf_stu` 视图（带 `WITH CHECK OPTION`）的例子，详细说明了 RDBMS 如何自动保证视图的一致性。

-   **修改操作（UPDATE）**：
    -   **自动加条件**：当通过视图修改某学生的院系时，RDBMS 会自动在更新语句中加入 `Sdept = '软件学院'` 的条件检查。
    -   **举例**：如果执行 `UPDATE sf_stu SET sdept = '数学学院' WHERE sid = '001';`，RDBMS 会拒绝执行，因为修改后该行不再满足视图的 `sdept='软件学院'` 条件，违反了 `WITH CHECK OPTION` 的约束。

-   **删除操作（DELETE）**：
    -   **自动加条件**：删除时也会自动检查，确保被删除的行原本就属于软件学院。但删除本身不会改变院系值，所以通常没有问题。

-   **插入操作（INSERT）**：
    -   **自动检查属性值**：当通过视图插入新学生时，RDBMS 会检查插入行的 `Sdept` 属性值是否为 `'软件学院'`。
    -   **如果不提供 `Sdept` 值**：示例中提到“如果没有提供 `Sdept` 属性值，则自动定义 `Sdept` 为‘软件学院’”。这是部分数据库的实现行为，即自动补全视图定义中的常量条件。
    -   **如果提供了错误的值**：如果插入时显式指定 `Sdept = '数学学院'`，RDBMS 会拒绝该插入操作。

-   **知识点总结**：
    -   `WITH CHECK OPTION` 相当于为视图的更新操作加了一道**防火墙**。
    -   它确保了通过视图所做的修改不会使数据“逃离”该视图的视角，从而维护了数据在逻辑上的完整性和一致性。
    -   如果没有这个选项，可能会出现通过视图插入了一条不属于该视图的记录，或者将记录改成了不属于该视图的状态，导致该记录在视图中消失，但仍在基表中存在的情况。这往往不是用户期望的行为。

### 补充说明

-   **WITH CHECK OPTION 的级联**：如果一个视图建立在另一个视图之上，并且都使用了 `WITH CHECK OPTION`，那么检查会级联进行。
-   **视图的可更新性**：并非所有视图都可以更新。一般来说，如果视图定义中包含分组、聚合、DISTINCT、多表连接等，通常不允许更新（或只有某些数据库允许有限更新）。`WITH CHECK OPTION` 只对**可更新视图**有效。

通过这几页，你应该全面掌握了创建视图的语法，以及 `WITH CHECK OPTION` 是如何精细地控制视图更新行为的。
