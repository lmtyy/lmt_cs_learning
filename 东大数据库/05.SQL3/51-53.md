## 第一页：视图的四个操作层面

这一页是目录式的，列出了视图相关的四个核心操作：

1. **定义视图**：使用 `CREATE VIEW` 语句创建视图，将查询语句保存为数据库对象。
2. **查询视图**：像查询普通表一样使用 `SELECT` 语句从视图中检索数据。
3. **更新视图**：对视图执行 `INSERT`、`UPDATE`、`DELETE` 操作，但受限于视图的可更新性规则（如单表、无聚合等）。
4. **视图的作用**：包括简化查询、安全性（隐藏敏感数据）、逻辑数据独立性等。

接下来的两页重点讲解 **查询视图** 的具体实现。

---

## 第二页：查询视图的示例

### 示例需求
在软件学院学生的视图中找出年龄小于20岁的学生。

### 视图定义（已创建）
```sql
CREATE VIEW sf_stu AS
SELECT Sid, Sname, Sage
FROM Student
WHERE Sdept = '软件学院';
```
该视图 `sf_stu` 包含了软件学院学生的学号、姓名和年龄。

### 查询语句
```sql
SELECT Sno, Sage
FROM sf_stu
WHERE Sage < 20;
```
- **用户角度**：这条语句看起来就像是在查询一张名为 `sf_stu` 的真实表。用户无需关心视图背后的基表结构、连接条件或筛选逻辑，只需关注自己需要的数据。
- **输出**：结果将显示所有年龄小于20岁的软件学院学生的学号和年龄。

## 第三页：视图查询的实现原理——视图消解法

### 用户感知 vs 实际执行
- **从用户角度**：查询视图与查询基本表在语法上完全相同，用户完全意识不到视图的存在。
- **RDBMS实现角度**：数据库管理系统必须将视图查询转换为对基表的查询才能执行。这一转换过程称为 **视图消解法（View Resolution）**。

### 视图消解的过程
以本页的查询为例，用户提交：
```sql
SELECT Sno, Sage
FROM sf_stu
WHERE Sage < 20;
```
数据库内部执行以下步骤：
1. **检查视图定义**：从数据字典中取出 `sf_stu` 的定义语句：
   ```sql
   CREATE VIEW sf_stu AS
   SELECT Sid, Sname, Sage
   FROM Student
   WHERE Sdept = '软件学院';
   ```
2. **将视图引用替换为其定义的子查询**：将查询中的视图名 `sf_stu` 替换为视图定义的查询体（通常经过语法树合并）。
3. **合并条件**：将用户查询中的 `WHERE Sage < 20` 与视图定义中的 `WHERE Sdept = '软件学院'` 通过 `AND` 连接，形成对基表的最终查询：
   ```sql
   SELECT Sid, Sage
   FROM Student
   WHERE Sdept = '软件学院' AND Sage < 20;
   ```
4. **优化与执行**：优化器对该查询进行优化（如选择索引、确定连接顺序等），然后执行并返回结果。

### 视图消解法的要点
- **透明性**：整个过程对用户完全透明，用户只看到最终结果。
- **动态性**：每次查询视图时，都会重新执行消解过程，因此基表数据的变化能实时反映在视图中。
- **潜在问题**：如果视图定义复杂（如多层嵌套、聚合、连接等），消解后生成的查询可能变得庞大，但现代优化器通常能有效处理。

### 补充说明
- **不是所有视图都能被消解**：对于某些更新操作，消解法可能不适用，需要借助触发器或规则（如可更新视图的条件）。
- **物化视图**：某些数据库支持物化视图（存储实际数据），其查询不经过消解，直接读取存储的数据。但标准视图（虚表）都采用消解法。
