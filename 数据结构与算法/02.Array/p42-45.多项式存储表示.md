这四页 PPT 是数据结构中**“一元多项式（Polynomial）的抽象数据类型（ADT）与存储实现”**的经典案例。

这几页 PPT 的核心目的，是教你**“如何针对同一个问题（多项式相加），设计不同的数据存储结构，并分析它们的优缺点”**。

为了便于你理解，我将这四页分为 **“数学逻辑”**、**“方法一（稠密表示）”**、**“方法二（稀疏表示）”** 三个部分来详细拆解。

---

### 第一部分：多项式的数学逻辑（图1）

*   **数学模型**：
    *   \(P(x)\) 和 \(Q(x)\) 是两项多项式。最高次项分别是 \(n\) 和 \(m\)。
    *   **多项式加法**：假设 \(m < n\)（即 \(Q\) 的最高次小于 \(P\)）。相加的结果 \(R(x)\) 是：**相同次数的项系数相加，其余项保持不变**。
*   **数组表示法（底部黄框）**：
    *   这是数学上的“数组/向量”视角。多项式可以直接用其**系数组成的一维数组**来表示：
        *   \(P = (p_0, p_1, p_2, ..., p_n)\)
        *   \(Q = (q_0, q_1, ..., q_m)\)
        *   **加法运算法则**：\(R\) 数组的前 \(m+1\) 项是 \(P\) 和 \(Q\) 对应位置系数相加，剩下的项直接等于 \(P\) 的剩余项。
*   **启示**：从这个数学逻辑来看，用**数组（顺序表）**来存储多项式是非常自然的想法。

---

### 第二部分：实现方法一 —— 稠密表示 / 顺序表（图2、图3）

这是最直接、最贴近数学公式的做法。

**1. 结构体定义（图2）**
*   `#define MAX_DEGREE 101`：假设多项式最高次数不超过 100。
*   `typedef struct { int degree; float coef[MAX_DEGREE]; } polynomial;`
    *   这里使用了**静态数组**。数组的**下标代表指数（次数）**，数组中存放的**值代表系数**。
    *   例如：\(3x^{50}\)，对应数组 `coef[50] = 3`。
*   **优点（Advantage）**：算法极其简单。做加法时，只需要一个 `for` 循环从 0 到最高次，把两个数组的元素直接相加即可，不需要复杂的逻辑判断。

**2. 致命缺陷与内存浪费（图3 示例）**
PPT 给出了一个非常直观的**“稀疏多项式”**例子：\(S(x) = 1 + 3x^{50} + 2x^{100}\)。
*   **用方法一存储**：数组长这个样子：`(1, 0, 0, ..., 0, 3, 0, ..., 0, 2)`。需要长度为 101 的数组。
*   **空间占用（Space size）**：\(101 \times 4\text{ bytes} = 404\text{ bytes}\)。
*   **问题所在**：这个多项式**只有 3 个非零项**，其余 98 个格子全都是无用的 `0`。
    *   如果你要存 \(x^{1000}\)，就要开 1001 个格子的数组，内存浪费极其严重。
    *   这被称为**“稀疏多项式（Sparse Polynomial）”**问题。**如果多项式非常稀疏（非零项极少），方法一非常不划算。**

### 第三部分：实现方法二 —— 稀疏表示 / 结构体数组（图4）

为了解决方法一中“大量系数为0占用空间”的问题，数据结构引入了**“只存非零项”**的策略。

**1. 结构体定义（图4）**
不再用数组下标代表次数了，而是将“每一项”打包成一个结构体：
*   **单项结构体 `termType`**：
    *   `float coef;`：存储系数。
    *   `int expn;`：存储指数（次数）。
*   **多项式结构体 `polynomial`**：
    *   `int noelem;`：记录当前多项式有多少个**非零项**（这替代了原来的 `degree`）。
    *   `termType elem[MAX_TERMS];`：一个结构体数组，用来存所有的非零项。

**2. 方法二如何存储图3的示例？**
以 \(S(x) = 1 + 3x^{50} + 2x^{100}\) 为例：
*   **存储结果**：`((1, 0), (3, 50), (2, 100))`
*   可以看到，**中间那些系数为 0 的项（如 \(x^1, x^2... x^{49}\)）全都被省略掉了**，只存了 3 个有效项。
*   **空间占用**：\(3 \times (\text{系数占用字节} + \text{指数占用字节})\)。
    *   例如系数 `float` 占 4 字节，指数 `int` 占 4 字节。总共需要 \(3 \times (4 + 4) = 24\text{ bytes}\)。
*   **对比**：方法一要 404 字节，方法二只要 24 字节。**在稀疏多项式上，方法二的空间效率完胜！**

---

### 💡 核心总结与深度延伸

这三页 PPT 向我们揭示了数据结构中一个极其重要的核心思想：**“空间换时间，还是时间换空间”的权衡**。

1.  **什么时候选方法一（稠密顺序表）？**
    *   **适用场景**：计算**稠密多项式**（比如 \(1+x+x^2+...+x^{100}\)，几乎没有 0）。
    *   **优势**：加法算法极其简单，就是一个 `for` 循环，时间复杂度非常低 \(O(n)\)。

2.  **什么时候选方法二（稀疏结构体数组）？**
    *   **适用场景**：计算**稀疏多项式**。
    *   **优势**：极大地节省了内存空间。
    *   **代价（算法变复杂了）**：因为忽略了中间的 0，我们无法再简单地用“下标”来定位相同的指数了。
    *   **思考题（后续课程会讲）**：使用方法二时，如果要计算两个多项式的和（比如 \(P = 3x^{50} + 2\)，\(Q = 5x^{50} - 4x^{20}\)），你要如何找到指数相同的项（比如找到都有 \(x^{50}\) 的那两个项）并相加？
    *   **解法**：如果数组是无序的，需要双重嵌套循环查找（\(O(n^2)\)）；如果数组**按指数从小到大排序**，就可以用上一组 PPT 讲的**“归并算法（双指针）”**，将加法的时间复杂度优化到 **\(O(m+n)\)**。

**总结一句话**：这一节只是介绍了**“表示法”**。实际使用时，对于方法二（稀疏多项式的加法），只要把它的结构体数组**按指数排序**，就可以完美套用之前讲过的**“有序表合并算法”**来实现高效计算。
