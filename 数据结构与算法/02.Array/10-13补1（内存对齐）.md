内存对齐（Memory Alignment）是 C/C++ 语言中一个非常经典且重要的底层概念，也是大厂面试中“必考”的知识点。

它的核心逻辑可以概括为：**以空间换时间**。

为了让你彻底理解，我将从“为什么要对齐”讲起，然后详细拆解“三大对齐规则”，最后通过案例带你手动计算。

------

### 一、 为什么要内存对齐？

你可能会认为内存是一个挨着一个的字节序列（1, 2, 3...），CPU 可以随意读取任意位置。但实际上，CPU 不是一个字节一个字节读取的，它是一块一块读取的（通常是 **4字节** 或 **8字节** 为一个“存取粒度”，Chunk）。

**举个栗子：**

假设你有一个 4 字节的 `int` 整数。

- **对齐时：** 它的地址在 0x00。CPU 读取一次（读取 0x00-0x03），直接拿到完整整数。
- **不对齐时：** 它的地址在 0x01。
  1. CPU 第一次读取 0x00-0x03，拿到了这个整数的前 3 个字节。
  2. CPU 第二次读取 0x04-0x07，拿到了这个整数的最后 1 个字节。
  3. CPU 需要把这两部分拼接起来，才能得到你要的数据。

**结论：** 如果内存不对齐，CPU 的读取效率会降低一半，甚至在某些硬件架构（如某些 ARM 或 RISC 架构）上直接导致程序崩溃（Bus Error）。为了保证速度和稳定性，编译器会自动给数据中间塞入“填充物”（Padding），让数据“对齐”。

------

### 二、 核心的三大对齐规则

在默认情况下（没有使用 `#pragma pack` 指令），编译器遵循以下三个核心规则。请务必记住这三条“铁律”：

#### 规则 1：自身对齐 (Natural Alignment)

每个数据成员的起始地址（Offset），必须是该成员**自身大小**的整数倍。

- `char` (1字节)：可以在任意地址开始（因为 $N \times 1$ 总是整数）。
- `short` (2字节)：起始地址必须是 2 的倍数（0, 2, 4...）。
- `int` / `float` (4字节)：起始地址必须是 4 的倍数（0, 4, 8...）。
- `double` / `long long` (8字节)：起始地址必须是 8 的倍数（0, 8, 16...）。

> **直观理解：** 如果你是个胖子（占 8 字节），你只能坐在 8 号、16 号这种宽敞的专座上，不能挤在 3 号这种位置。

#### 规则 2：结构体尾部填充 (Structure Padding)

结构体的**总大小**，必须是其**最宽基本成员**（Largest Member）大小的整数倍。

- **为什么？** 这是为了保证当你定义一个**结构体数组**时，第二个结构体也能自然对齐。如果第一个结构体长度是 13，第二个结构体就会从 13 开始，这会瞬间打破所有内部成员的对齐。

#### 规则 3：嵌套结构体对齐

如果结构体 A 里面包含结构体 B，那么 B 的起始地址必须是 **B 中最宽基本成员**的整数倍。

### 三、 实战演练：手算结构体大小

让我们通过两个例子，看看同样的成员，只是**顺序不同**，内存占用会有多大差别。

#### 案例 A：乱序排列（浪费空间）

C

```
struct A {
    char c;   // 1 byte
    int i;    // 4 bytes
    short s;  // 2 bytes
};
```

**计算过程：**

1. **`c` (offset 0):** `char` 占 1 字节。当前位置走到了 1。
2. **`i` (offset 1?):** `int` 要求起始地址是 4 的倍数。当前是 1，不是 4 的倍数。
   - **填充：** 编译器在 `c` 后面塞入 **3 个字节** 的 Padding（空字节）。
   - **存放：** `i` 从 offset 4 开始存放，占 4 字节。当前位置走到了 8。
3. **`s` (offset 8):** `short` 要求起始地址是 2 的倍数。8 是 2 的倍数，直接存放。
   - **存放：** `s` 从 offset 8 开始，占 2 字节。当前位置走到了 10。
4. **整体对齐（规则 2）：** 目前总大小是 10。
   - 结构体中最宽的是 `int` (4字节)。
   - 10 不是 4 的倍数。
   - **尾部填充：** 补 **2 个字节**，使总大小变为 12。

**最终大小：** `sizeof(struct A) = 12` 字节。（有效数据 7 字节，浪费 5 字节）

------

#### 案例 B：优化排列（节省空间）

我们把占用小的放在一起。

C

```
struct B {
    char c;   // 1 byte
    short s;  // 2 bytes
    int i;    // 4 bytes
};
```

**计算过程：**

1. **`c` (offset 0):** `char` 占 1 字节。当前位置 1。
2. **`s` (offset 1?):** `short` 要求 2 的倍数。
   - **填充：** 补 **1 个字节** Padding。
   - **存放：** `s` 从 offset 2 开始，占 2 字节。当前位置 4。
3. **`i` (offset 4):** `int` 要求 4 的倍数。4 是 4 的倍数。
   - **存放：** `i` 从 offset 4 开始，占 4 字节。当前位置 8。
4. **整体对齐（规则 2）：** 目前总大小 8。
   - 最宽成员是 `int` (4)。
   - 8 是 4 的倍数，不需要尾部填充。

**最终大小：** `sizeof(struct B) = 8` 字节。（有效数据 7 字节，只浪费 1 字节）

### 四、 高级考点：`#pragma pack`

有时候（比如网络传输协议包、二进制文件读写），我们不希望编译器乱塞 Padding，我们需要数据紧凑。这时可以使用预编译指令：

C

```
#pragma pack(1)  // 告诉编译器：按照 1 字节对齐（也就是不对齐，紧挨着存）
struct C {
    char c;
    int i;
};
#pragma pack()   // 恢复默认对齐
```

- **效果：** `sizeof(struct C)` 变成了 **5** (1+4)。
- **代价：** 节省了内存，但牺牲了 CPU 读取效率。

### 总结

当你计算 `sizeof` 时，脑子里要画出那张内存条：

1. **看成员**：把它放到它该在的位置（它是几字节，起始地址就要能被几整除）。
2. **看整体**：最后检查总长度，必须能被最大成员整除，不够就补齐。
