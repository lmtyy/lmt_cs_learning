### 第一页（Slide 92）：补全第二次称重“<”分支的终极判定
**核心知识点：当第二次称重左边更轻时（\(a+d < b+e\)），如何一步步锁定真凶**

其实这一页的文字和下方的树图一起，解释了根节点第一次称重是 **`<`（左边轻）** 的情况下，最后两种分支的走向。

*   **情况一：第二次称重为 `=` 平衡**
    *   PPT 说明：`If a+d was equal to b+e, then c or f would be the counterfeit coin.`
    *   **逻辑**：在第一次左轻的前提下，如果 `a+d` 和 `b+e` 平衡，说明 `b` 和 `d` 是真币。那么假币必定在剩下的 `c` 或 `f` 中。
*   **情况二：第二次称重为 `<` 左边轻（重点）**
    *   PPT 后半部分说：`Knowing at this point that either a or e is the counterfeit...`（此时我们知道假币不是 `b`、`c`、`d`、`f`，只能是 **`a` 偏轻** 或 **`e` 偏重**）。
    *   **终极判定（只需要最后一次称重）**：拿一枚“确定是真币”的硬币（比如已经排除嫌疑的 `b`），去跟 **`a`** 比一比。
        *   如果 `a = b`：说明 `a` 是真币，那假币一定是 **`e-H`**（e 偏重）。
        *   如果 `a != b`：说明 `a` 就是假币，那它一定是 **`a-L`**（a 偏轻）。

### 第二页（Slide 93）：推导“第一次称重完全平衡”的情况
**核心知识点：在树图的正中央分支，路径最短。**

*   **场景**：`If a+b+c was equal to d+e+f`（如果第一把天平左、右各3枚硬币，居然**绝对平衡**）。
*   **立刻得出结论**：6枚硬币 `a` 到 `f` 全部都是真币。**假币必然藏在剩下的 `g` 和 `h` 中。**
*   **后续逻辑（看树中间的 `g ? h` 分支）**：
    *   因为 `g` 和 `h` 中必有一个假币，你只要把 `g` 和 `h` 放天平两边就能立刻知道谁重谁轻。
    *   为了进一步确认（并且在代码逻辑中统一判断），PPT 对应的树图里，天平倾斜后，会用其中一枚和确定的真币（比如 `a`）做验证，最终得出 `g-H`、`h-L`、`h-H` 或 `g-L`。
    *   *注：这是理论上最快的一条路，只需要 **2次** 称重即可破案。*

### 第三页（Slide 94）：决策树的数学完备性与程序映射
**核心知识点：为什么这棵树设计得如此完美？**

这一页是对整个八硬币问题最精辟的数学统计：
1.  **完美覆盖（16个叶子节点）**：因为 8 枚硬币，每一枚硬币都有“**偏重（Heavier）**”或“**偏轻（Lighter）**”两种可能，所以**全世界总共只存在 16 种最终状态**。你数一数这棵决策树的底层叶子节点，**刚好 16 个**。说明这棵树覆盖了**100%** 的所有可能情况。
2.  **绝对最优（最多 3 次称重）**：PPT 强调 `Every path requires exactly 3 comparisons`。每次天平称重有 3 种结果（`>`, `=`, `<`）。称 2 次最多区分 `3²=9` 种情况，完全不够区分 16 种。3 次理论上能区分 27 种，**所以 3 次是该问题的理论最短路径，这个决策树达到了最优解。**
3.  **代码怎么做？**：PPT 提议 `we must write a series of tests which mirror the structure of the tree`。这意味着你要写程序，**核心就是模拟这棵树的分支结构**。

### 第四页（Slide 95）：从理论到代码——if-else 的痛点与 switch-case 的魅力
**核心知识点（您刚补充的）：如何优雅地写这段代码**

*   **错误写法：庞大的 if-else 嵌套**。
    *   如果使用 `if (left > right) { ... } else if (left == right) { ... } else { ... }` 这种写法，由于各种称重和逻辑排除像迷宫一样，代码会写得像**“一盘意大利面（Spaghetti）”**，层层嵌套，极其臃肿，**你根本看不懂程序的流向**，这显然是不好的编程风格。
*   **正确写法：switch-case 语句**。
    *   因为天平的结果永远只有三种情况（大于、等于、小于）。
    *   **推荐的做法**：使用 `switch(天平结果)`。
        ```c
        switch(天平结果) {
            case '>': // 处理左边重的情况
                // 调用针对 > 的子树逻辑
                break;
            case '=': // 处理平衡的情况
                // 调用针对 = 的子树逻辑
                break;
            case '<': // 处理右边重的情况
                // 调用针对 < 的子树逻辑
                break;
        }
        ```
    *   这种写法完美对应了 PPT 第88页那棵树的分支结构，**代码结构与树形逻辑实现了一对一的优雅映射**，一目了然。
