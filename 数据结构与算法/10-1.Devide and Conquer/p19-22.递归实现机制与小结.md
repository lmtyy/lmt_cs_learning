# 递归实现机制与小结（第19-22页）

本段讲解递归在系统层面如何通过栈来实现，以及递归算法的优缺点总结。

---

## 第19页：调用算法前系统做的三件事

当被调用算法返回调用算法时，系统需要完成：

| # | 操作 | 说明 |
|---|------|------|
| 1 | 将所有实参指针，返回地址等信息传递给被调用算法| 初始信息和“收件地址“传给被调用者 |
| 2 |为被调用算法的局部变量分配存储区 | 给予局部变量空间 |
| 3 |将控制转移到被调用算法的入口 | 跳转到被调用函数继续运行 |

> 类比：打电话问人帮忙查东西 → 要告诉对方在哪查，给对方查询的工具，查完了结果放哪→ 你给对方找一片工作区，好让对方展开手脚→对方利用你给定信息，执行自己的查找操作，并把操作结果放在你指定的地方

---

## 第20页：递归返回时系统做的三件事

当被调用算法返回调用算法时，系统需要完成：

| # | 操作 | 说明 |
|---|------|------|
| 1 | **保存**被调用算法的计算结果 | 返回值传回给调用者 |
| 2 | **释放**分配给被调用算法的数据区 | 栈帧弹出，回收局部变量空间 |
| 3 | 依照被调用算法保存的**返回地址**将控制转移到调用算法 | 回到调用点继续执行 |

> 类比：打电话问人帮忙查东西 → 对方查完告诉你结果（保存结果）→ 对方挂电话释放通话资源（释放数据区）→ 你继续做之前被打断的事（控制转移回来）

---

## 第21页：递归调用栈结构

```
┌─────────────────────────────────────┐
│           主算法栈块                  │  ← 最先入栈
├┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┤
│   主算法调用递归算法A的栈块            │
├─────────────────────────────────────┤
│   算法A的第一层递归调用工作记录        │
├─────────────────────────────────────┤
│   算法A的第二层递归调用工作记录        │
├─────────────────────────────────────┤
│              TOP                     │  ← 栈顶（最后入栈）
└─────────────────────────────────────┘
```

### 栈帧中保存的内容（工作记录）

每一层递归调用的"工作记录"包含：

| 内容 | 作用 |
|------|------|
| 局部变量 | 该层递归自己的数据 |
| 形式参数 | 本次调用传入的参数值 |
| 返回地址 | 调用结束后回到哪一行继续执行 |
| 返回值 | 本层计算的结果 |

### 运行过程

```
递归调用时：入栈（Push）
  → 分配新栈帧，保存参数、局部变量、返回地址

递归返回时：出栈（Pop）
  → 恢复上一层的状态，继续执行
```

> ✅ 这就是为什么递归深度太大会导致 **Stack Overflow**（栈溢出）——栈空间有限！

## 第22页：递归小结

### 优点

> 结构清晰，可读性强，而且容易用**数学归纳法**来证明算法的正确性，因此它为设计算法、调试程序带来很大方便。

### 缺点

> 递归算法的运行效率较低，无论是耗费的**计算时间**还是占用的**存储空间**都比非递归算法要多。

### 对比表

| 维度 | 递归 | 非递归（迭代） |
|------|------|--------------|
| 代码简洁性 | ✅ 简洁、自然 | ❌ 可能复杂 |
| 可读性 | ✅ 结构清晰 | 一般 |
| 正确性证明 | ✅ 易用数学归纳法 | 需要循环不变式 |
| 时间效率 | ❌ 函数调用开销 | ✅ 无额外开销 |
| 空间效率 | ❌ 栈帧累积 | ✅ 通常 O(1) |
| 栈溢出风险 | ❌ 深度大时危险 | ✅ 无此问题 |

---

## 总结

```
递归的系统实现 = 栈

每次调用：Push 一个栈帧（参数 + 局部变量 + 返回地址）
每次返回：Pop 栈帧，恢复上一层状态

递归的取舍：
  ┌── 选递归：问题天然递归结构，规模不大，代码清晰优先
  └── 选迭代：性能敏感，递归深度大，空间受限

至此递归部分完结，下面进入本章核心——分治法。
```
