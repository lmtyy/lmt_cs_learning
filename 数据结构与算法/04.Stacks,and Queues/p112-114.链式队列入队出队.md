### 第一页 (第112页)：入队操作 `EnQueue`
**核心目标：** 实现将新元素加到链表末尾的操作。

*   **代码逻辑拆解：**
    1.  **申请新节点内存：** `p = (QueuePtr)malloc(sizeof(QNode));`
        *   不论队列原来有多大，每次入队都要动态申请一个新的节点内存。
    2.  **内存分配失败处理：** `if (!p) exit(OVERFLOW);`
    3.  **填充新节点数据：** `p->data = e; p->next = NULL;`
        *   将数据存入节点，并将新节点的 `next` 指针置为 `NULL`（因为它将变成新的队尾）。
    4.  **挂接新节点（核心指针操作）：** `Q.rear->next = p;`
        *   将**当前队尾节点**的 `next` 指针，指向刚刚创建的新节点 `p`。新节点被挂在链表末尾。
    5.  **更新队尾指针：** `Q.rear = p;`
        *   将队列的 `rear` 指针更新为指向新节点 `p`。现在 `p` 是新的队尾。
*   **💡 核心理念：**
    *   链式队列入队**不需要判满**！因为 `malloc` 只要成功，就能无限扩容（直到内存耗尽）。这极大地简化了入队的逻辑。

### 第二页 (第113页)：出队操作 `DeQueue`
**核心目标：** 将队头元素移除，并返回其值。**（这页代码包含一个极易被忽视的关键边界处理！）**

*   **代码逻辑拆解：**
    1.  **判空：** `if (Q.front == Q.rear) return ERROR;`
        *   如果 `front` 和 `rear` 都指向同一个节点（哑结点），说明队列里没有有效数据。
    2.  **定位要删除的节点：** `p = Q.front->next;`
        *   **⚠️ 核心差异：** 这里**不是**删除 `front` 指向的哑结点，而是删除哑结点后面的**第一个有效数据节点**。这个节点才是真正的队头。
    3.  **取出数据：** `e = p->data;`
    4.  **特殊边界判断（绿色代码，极度重要！）：** `if (Q.rear == p) Q.rear = Q.front;`
        *   **这个 `if` 是为了解决什么问题的？**
        *   如果你出队的是队列中的**最后一个有效节点**（即 `p` 既是队头，也是队尾）。删除 `p` 后，队列就会变回只有哑结点的“空状态”。
        *   此时，如果**只**执行 `Q.front->next = p->next;`，那么 `Q.rear` 依然指向已经被 `free` 掉的 `p`，这就变成了一个**悬空指针（野指针）**！
        *   **解决办法：** 如果发现 `rear == p`，说明删掉了最后一个有效数据，必须把 `rear` 指针**重新指回哑结点**（`Q.front`），以恢复“空队列”的正确状态。
    5.  **断开链表并释放内存：** `Q.front->next = p->next; free(p);`
        *   让哑结点的 `next` 跳过 `p`，指向下一个节点（如果 `p` 是最后一个节点，则指向 `NULL`）。然后释放 `p` 的内存。
*   **💡 极高频考题：** 面试官经常会问：“链式队列删除最后一个元素后，`rear` 指针怎么办？” 答案就是这行绿色代码：`Q.rear = Q.front;`。

### 第三页 (第114页)：实战结果与核心结论
**核心目标：** 用一个超大规模的数据（32767个元素）验证链式队列的稳定性，并给出终极结论。

*   **实战测试代码：**
    *   `for(i=0; i<32767; i++) EnQueue(new_q, i);`
        *   一次性将 32767 个数字（0 到 32766）全部塞进队列。如果是循环数组，容量稍微小一点就崩了。
    *   `while(!IsEmpty(new_q)) { DeQueue(...); 打印; }`
        *   依次取出并打印，并设置了每 5 个数字换行的格式。
*   **控制台运行结果截图（右下）：**
    *   完美输出了 `32760, 32761, 32762, 32763, 32764` 和 `32765, 32766`。
    *   证实了在如此大量的数据下，链表队列依然没有溢出，且保持了 FIFO 顺序。
*   **💡 核心结论（顶部红字）：**
    *   `"Queue implemented using linked list will be never full."`
    *   **（使用链表实现的队列永远不会满。）**

### 💡 工程结论：链式队列 VS 循环队列

这两页代码（入队和出队）完成了链式队列的完整拼图。至此，你可以清晰地对比出两者的优劣：

| 特性         | 链式队列（链表实现）                                         | 循环队列（数组实现）                                         |
| :----------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **内存**     | 按需申请，**永不溢出**（除非物理内存爆了）                   | 有固定 `MAXQSIZE`，可能溢出（假溢出已解决，但有容量上限）    |
| **判满逻辑** | 不需要判满（无 `IsFull`）                                    | 必须写 `(rear+1)%MAXQSIZE == front` 这种复杂数学公式         |
| **关键风险** | 指针操作复杂。特别要注意**出队最后一个元素时，`rear` 指针的回落**（PPT 113页绿字）。 | 只有整数的加减取模运算，没有复杂的指针动态分配；**不需要担心内存碎片**。 |
| **工程选择** | 适用于**数据量未知、可能极大**的场景（如消息队列、网络数据缓冲）。 | 适用于**数据量上限固定，且追求极致性能**的场景（如 CPU 任务调度、嵌入式开发）。 |
