# 邻接矩阵的代码实现 Graph ADT（p025-028）

从数学定义落地到C语言代码——如何用结构体把"图"这个抽象概念装进内存，以及为什么这种方案注定"浪费"。

---

## 第25页：数据结构定义（上）——常量与边的单元格

### 完整代码

```c
#define INFINITY  INT_MAX      // 最大值，表示"不可达"
#define MAX_VERTEX_NUM  20     // 最多的顶点个数

typedef enum {DG, DN, UDG, UDN} GraphKind;
        // {有向图, 有向网, 无向图, 无向网}

typedef struct ArcCell
{
    VRType adj;        // 顶点关系类型：图：0/1  网：权值
    InfoType *info;    // 该弧相关信息指针
} ArcCell, AdjMaTrix[MAX_VERTEX_NUM][MAX_VERTEX_NUM];
```

### 逐段精讲

**1. 宏定义部分**

```c
#define INFINITY  INT_MAX
#define MAX_VERTEX_NUM  20
```

- **动作**：定义两个编译期常量。
- **目的**：`INFINITY` 用于带权图中表示"两点之间没有直接边"（不可达）；`MAX_VERTEX_NUM` 是顶点数量的上限，决定了数组开多大。
- **为什么用 `INT_MAX`？** 因为在最短路等算法中经常做 `if (dist[u] + w < dist[v])` 这种比较，用 `INT_MAX` 能保证"不可达"的距离总是比任何实际距离大。

> ⚠️ **陷阱**：如果用 `INT_MAX` 做 ∞，两个 ∞ 相加会**整数溢出**！实际工程中常用 `0x3f3f3f3f`（约10⁹），因为 `0x3f3f3f3f + 0x3f3f3f3f` 不会溢出 `int`。

**2. 图的种类枚举**

```c
typedef enum {DG, DN, UDG, UDN} GraphKind;
```

| 枚举值 | 全称 | 含义 |
|--------|------|------|
| DG | Directed Graph | 有向图（边无权） |
| DN | Directed Network | 有向网（边有权） |
| UDG | Undirected Graph | 无向图（边无权） |
| UDN | Undirected Network | 无向网（边有权） |

- **目的**：用一个字段标记当前图的类型，后续创建/遍历时可以做分支处理。
- **为什么区分"图"和"网"？** 因为无权图的矩阵存 0/1，带权图存实际权值和 ∞，初始化逻辑不同。

**3. 边的单元格结构 `ArcCell`**

```c
typedef struct ArcCell
{
    VRType adj;        // 核心：存关系值
    InfoType *info;    // 附加信息指针（通常不用）
} ArcCell, AdjMaTrix[MAX_VERTEX_NUM][MAX_VERTEX_NUM];
```

- **动作**：定义邻接矩阵中**每个格子**的数据类型。
- **`adj` 字段**：对于无权图，存 0 或 1；对于带权图，存权值（有边）或 `INFINITY`（无边）。
- **`info` 指针**：为边附加额外信息预留的扩展口，大多数时候是 `NULL`。
- **`AdjMaTrix` 类型别名**：一行 `typedef` 同时定义了结构体名 `ArcCell` 和一个 20×20 的二维数组类型 `AdjMaTrix`。

> 💡 **C语言技巧**：`AdjMaTrix` 不是变量，而是一个**类型名**，等价于 `ArcCell[20][20]`。后面声明 `AdjMaTrix arcs;` 就是开了一个 20×20 的矩阵。

## 第26页：数据结构定义（下）——图的完整结构体

### 完整代码

```c
typedef struct
{
    VertexType vexs[MAX_VERTEX_NUM];  // 顶点向量
    AdjMaTrix arcs;                    // 邻接矩阵
    int vexnum, arcnum;                // 顶点数和弧或边数
    GraphKind kind;                    // 图的种类标志
} Mgraph;
```

### 逐段精讲

| 字段 | 类型 | 作用 | 大小 |
|------|------|------|------|
| `vexs` | `VertexType[20]` | 存每个顶点的数据（名字/编号） | 20个单元 |
| `arcs` | `ArcCell[20][20]` | 邻接矩阵本体 | 400个格子 |
| `vexnum` | `int` | 实际使用的顶点数 | 4字节 |
| `arcnum` | `int` | 实际使用的边/弧数 | 4字节 |
| `kind` | `GraphKind` | 标记是哪种图 | 4字节 |

> 🎯 **内存布局类比**：把 `Mgraph` 想象成一个"图的档案袋"——
> - `vexs` 是名单（"有哪些顶点"）
> - `arcs` 是关系表（"谁和谁有联系"）
> - `vexnum/arcnum` 是统计信息（"一共几个人、几条关系"）
> - `kind` 是档案类型标签

> ⚠️ **注意**：`MAX_VERTEX_NUM=20` 是编译时写死的。如果实际只有4个顶点，数组还是开了20×20=400格，其中只用 4×4=16 格。这就是下面第28页说"浪费"的根源。

## 第27页：构造无向网的完整代码 `CreateUDN`

### 完整代码

```c
Status CreateUDN(Mgraph &G)
{
    // 采用邻接矩阵表示法构造无向网
    scanf(&G.vexnum, &G.arcnum, &IncInfo);  // 读入顶点和弧的个数
    
    for(i=0; i<G.vexnum; ++i) 
        scanf(&G.vexs[i]);                   // 逐个读入顶点信息
    
    for(i=0; i<G.vexnum; ++i)
        for(j=0; j<G.vexnum; ++j)
            G.arcs[i][j] = {Infinity, NULL};  // 初始化邻接矩阵
    
    for(k=0; k<G.arcnum; ++k)
    {
        scanf(&v1, &v2, &w);                  // 读入一条弧信息
        i = LocateVex(G, v1); 
        j = LocateVex(G, v2);                 // 找到顶点在数组中的下标
        G.arcs[i][j] = w;                     // 存储权值
        if(IncInfo) Input(*G.arcs[i][j].info);// 输入弧信息（如果有）
        G.arcs[j][i] = w;                     // 无向图：对称赋值！
    }
    return OK;
}
```

### 逐段精讲

**第1段：读入基本参数**

```c
scanf(&G.vexnum, &G.arcnum, &IncInfo);
```

- **动作**：从输入读取顶点数、边数、是否有附加信息。
- **目的**：确定后续循环的边界。`IncInfo` 是个标志位，决定是否需要读入弧的额外信息。

**第2段：读入顶点**

```c
for(i=0; i<G.vexnum; ++i) 
    scanf(&G.vexs[i]);
```

- **动作**：把每个顶点的标识（名字/编号）存入 `vexs` 数组。
- **目的**：建立"顶点名 → 数组下标"的映射关系。后面 `LocateVex` 就是通过遍历这个数组来做反向查找的。

**第3段：初始化矩阵为全 ∞（关键！）**

```c
for(i=0; i<G.vexnum; ++i)
    for(j=0; j<G.vexnum; ++j)
        G.arcs[i][j] = {Infinity, NULL};
```

- **动作**：把整个矩阵每个格子初始化为 `{∞, NULL}`。
- **目的**：在带权图中，"没有边"用 ∞ 表示（不是0！）。先全部设为"不可达"，后面读到哪条边就把对应格子改成实际权值。
- **为什么不初始化为0？** 因为权值可能是0（比如距离为0），如果用0表示"无边"就会产生歧义。用 ∞ 才能明确区分"有边权为w"和"根本没边"。

> 💡 **时间复杂度**：这个双重循环是 O(N²)。即使图只有2条边，初始化也要扫满 N×N。这是邻接矩阵"天生的固定开销"。

**第4段：逐条读入边并填入矩阵**

```c
for(k=0; k<G.arcnum; ++k)
{
    scanf(&v1, &v2, &w);
    i = LocateVex(G, v1); 
    j = LocateVex(G, v2);
    G.arcs[i][j] = w;
    if(IncInfo) Input(*G.arcs[i][j].info);
    G.arcs[j][i] = w;    // ← 关键：对称赋值
}
```

- **动作**：读入一条边的两个端点和权值，找到它们在数组中的下标，填入矩阵。
- **`LocateVex(G, v1)`**：在 `G.vexs` 数组中查找顶点 `v1` 的下标位置。类似于"在花名册里找人的座号"。
- **`G.arcs[j][i] = w`**：因为是**无向网**，边 (i,j) = 边 (j,i)，所以要**对称赋值**。

> ⚠️ **如果忘了对称赋值**：那么从 j 查 i 时会看到 ∞（以为不可达），但实际上有边。这会导致遍历、最短路等算法**漏掉一半的边**。

> 💥 **反面教材（有向图不能对称赋值）**：如果你把这段代码原封不动地用来建有向图，`G.arcs[j][i] = w` 这行就是 bug——它会凭空多出一条反向边。**有向图只写 `G.arcs[i][j] = w`，不写 `[j][i]`**。

### 整体时间复杂度

| 阶段 | 操作 | 复杂度 |
|------|------|--------|
| 读顶点 | 循环 N 次 | O(N) |
| 初始化矩阵 | 双重循环 N×N | **O(N²)** |
| 读边 | 循环 E 次，每次 LocateVex O(N) | O(E·N) |
| **总计** | | **O(N² + E·N)** |

> 💡 如果用哈希表实现 `LocateVex`，可以把读边阶段优化到 O(E)，但矩阵初始化的 O(N²) 是逃不掉的。

## 第28页：邻接矩阵的空间代价分析

### 核心结论

> **邻接矩阵直接存为普通数组，通常是非常浪费（very wasteful）的。**

### 原因分析

| 事实 | 说明 |
|------|------|
| 矩阵大小永远是 **N²** | 不管实际有几条边，都要开 N×N 的空间 |
| 边数最多 N²（有自环有向图） | 只有"每个点连所有点（包括自己）"才能填满，这种图几乎没意义 |
| 实际图通常是**稀疏的** | 如社交网络：10⁶用户，每人平均100好友 → 边数10⁸，但矩阵要10¹² 格 |

### 何时可以接受邻接矩阵？

| 场景 | 边数 E 约等于 | 矩阵利用率 | 结论 |
|------|-------------|-----------|------|
| 稠密图（完全图） | N² | ~100% | ✅ 适合 |
| 中等密度 | N·log(N) | ~log(N)/N | ⚠️ 勉强 |
| 稀疏图 | O(N) | ~1/N | ❌ 巨大浪费 |

> ✅ **做题判断标准**：如果题目给出 E ≈ N²（稠密图），用邻接矩阵；如果 E << N²（稀疏图），用邻接表。

> 🎯 **引出下一节**：正因为邻接矩阵浪费空间，所以需要**邻接表（Adjacency List）**——只存实际存在的边，空间 O(N+E)。

## 总结

```
邻接矩阵代码实现路线：

定义阶段：
  ArcCell{adj, *info} → AdjMaTrix[N][N] → Mgraph{vexs, arcs, vexnum, arcnum, kind}

构造阶段（以无向网 UDN 为例）：
  读参数 → 读顶点 → 矩阵全部初始化为∞ → 逐条读边并对称填入

代价：
  空间 O(N²) 固定 | 初始化 O(N²) | 建图 O(N²+E·N)
  稀疏图严重浪费 → 引出邻接表
```

> 💡 **箭头流程线**：读入 N、E → 开 N×N 矩阵置 ∞ → 逐边 LocateVex 找下标 → 填 arcs[i][j] （无向再填 [j][i]）→ 完成
