# 邻接表 Adjacency List（p029-034）

邻接矩阵空间浪费的解药——邻接表。只记录"实际存在的边"，稀疏图的救星。本段从概念引入、节点结构设计，到有向图/无向图的具体示例，讲清邻接表的全貌。

---

## 第29页：邻接表引入——为什么需要它？（"按需分配"哲学）

### 核心定义

**邻接表（Adjacency List）**：对每个顶点，维护一个链表，链表中存储所有与它**直接相邻**的顶点。

### 关键数据

| 属性 | 邻接矩阵 | 邻接表 |
|------|---------|--------|
| 空间 | O(N²)，与边数无关 | **O(N + E)**，按需分配 |
| 适用场景 | 稠密图 | **稀疏图** |
| 查某条边是否存在 | O(1) | O(degree) 需遍历链表 |

> 🎯 **生活类比**：邻接矩阵像一本"所有人×所有人"的通讯录方阵，大部分格子是空的。邻接表像"每人一个通讯录本"，里面只记自己认识的人——认识3个人就3条记录，不浪费。

### PPT示例（5顶点有向图）

```
顶点 | 邻接链表（出边指向谁）
  1  → [3] → [5]
  2  → [1]
  3  → [4]
  4  → [2]
  5  → (空)
```

对应边：1→3, 1→5, 2→1, 3→4, 4→2。顶点5出度为0（链表空）。

> 💡 **空间计算**：5个顶点的头 + 5条边的节点 = O(5+5) = O(N+E)。如果用邻接矩阵，是 5×5=25 格，其中只有5格是1。

## 第30页：节点结构设计——两种角色

### Head 节点（顶点表中的一项）

```
┌──────────┬──────────┐
│   data   │   adj    │
└──────────┴──────────┘
```

| 字段 | 含义 |
|------|------|
| `data` | 顶点的数据（名字/编号） |
| `adj` | 指向该顶点**邻接链表第一个边节点**的指针 |

### Edge 节点（链表中的一个边）

```
┌──────────┬──────────┬──────────┐
│   dest   │   cost   │   Link   │
└──────────┴──────────┴──────────┘
```

| 字段 | 含义 |
|------|------|
| `dest` | 这条边指向的**目标顶点编号** |
| `cost` | 边的**权值**（无权图可省略） |
| `Link` | 指向**同一顶点的下一条边**的指针 |

> 🎯 **类比**：Head 节点像"人名卡片"，Edge 节点像"联系人条目"。每张人名卡片后面挂着一串联系人条目，形成链表。

### 整体结构

```
HEAD[0] → edge → edge → NULL
HEAD[1] → edge → NULL
HEAD[2] → edge → edge → edge → NULL
...
```

> **`HEAD` 数组**大小为 N（顶点数），每个元素是一个链表头。这就是"数组 + 链表"的混合结构。

## 第31页：无向图的邻接表（G8 示例）

### 图：A-B, A-D（4顶点无向图）

```
图 G8:    A --- C 
          |
    D --- B
```

PPT中 G8 的边：A-C(即0-2), B-A(即1-0), C-A(即2-0), D-B(即3-1)

实际邻接表：

```
下标 | data | 链表
  0  |  A   | → [2] → [1] → ∧     (A连C和B)
  1  |  B   | → [0] → [3] → ∧     (B连A和D)
  2  |  C   | → [0] → ∧           (C连A)
  3  |  D   | → [1] → ∧           (D连B)
```

### 无向图的关键特性

> ✅ **每条无向边在邻接表中出现两次！**

如果 (i, j) ∈ E，则：
- 顶点 i 的链表中有一个 dest=j 的节点
- 顶点 j 的链表中有一个 dest=i 的节点

> **为什么？** 因为无向边是双向的——A认识C，C也认识A，两边都要记。

> 💡 **空间代价**：无向图的邻接表，边节点总数 = **2E**（每条边两份）。总空间 = O(N + 2E)。

> ⚠️ **度的计算**：无向图中，顶点 i 的度 = 它的邻接链表长度。直接数链表里有几个节点即可——**不用像邻接矩阵那样扫整行**。

## 第32页：无向图邻接表示例（5顶点，更完整）

### 图：V0-V1, V0-V3, V1-V2, V1-V4, V2-V3, V2-V4

邻接表：

```
下标 | data |  链表（dest 序列）
  0  |  V₀  | → [1] → [3] → ∧           度=2
  1  |  V₁  | → [0] → [2] → [4] → ∧     度=3
  2  |  V₂  | → [1] → [3] → [4] → ∧     度=3
  3  |  V₃  | → [0] → [2] → ∧           度=2
  4  |  V₄  | → [1] → [2] → ∧           度=2
```

> **规则 `If <Vi,Vj> ∈ E then <Vj,Vi> ∈ E`**：这就是无向图"对称性"在邻接表中的体现。

> 💡 **对比邻接矩阵**：这个图有6条边，邻接矩阵需要5×5=25格（利用率6×2/25=48%还行）。但如果是1000个顶点3000条边的图，矩阵要10⁶格而邻接表只需1000+6000=7000个节点——差距140倍。

## 第33页：无向图邻接表（8顶点树形图）

### 图结构：V0为根的树

```
         V0
       /    \
     V1      V2
    /  \     /  \
  V3   V4   V5--V6
   \    /
     V7
```

边（无向）：V0-V1, V0-V2, V1-V3, V1-V4, V1-V7, V2-V5, V2-V6, V2-V0（重复，V0-V2已有）

邻接表：

```
0  V₀ → [1] → [2] → ∧
1  V₁ → [3] → [4] → [0] → ∧
2  V₂ → [5] → [6] → [0] → ∧
3  V₃ → [7] → [1] → ∧
4  V₄ → [7] → [1] → ∧
5  V₅ → [6] → [2] → ∧
6  V₆ → [2] → [5] → ∧
7  V₇ → [3] → [4] → ∧
```

> 💡 **观察**：这是一棵树（8顶点7条边），邻接表边节点总数 = 2×7 = 14。邻接矩阵需要 8×8=64 格（利用率 14/64 ≈ 22%）。越稀疏，邻接表优势越大。

## 第34页：有向图的邻接表（出边表）

### 有向图 vs 无向图的关键区别

有向图中，邻接表存的是**出边**（从该顶点出发的边）。每条有向边**只出现一次**。

### 示例 G7（5顶点有向图）

图：A→D, A→E, B→D, C→D, D→E, E→A

```
下标 | data | 出边链表
  0  |  A   | → [3] → [1] → ∧     (A→D, A→E，即dest=3,dest=1... 
                                    等等PPT给的是dest=3,1)
  1  |  B   | → [2] → ∧           (B→D)
  2  |  C   | → [3] → ∧           (C→D)
  3  |  D   | → [4] → ∧           (D→E)
  4  |  E   | → [0] → ∧           (E→A)
```

### 有向图邻接表的属性

| 属性 | 说明 |
|------|------|
| 空间 | O(N + E)，每条弧只存一次 |
| 求顶点 i 的**出度** | 数 HEAD[i] 链表长度，O(OD(i)) |
| 求顶点 i 的**入度** | ⚠️ 需要遍历**所有**链表，看谁的 dest == i，O(N+E) |

> 💥 **重大缺陷**：求入度非常慢！要扫描全部链表才能知道"谁指向我"。

> 💡 **解决方案（逆邻接表）**：如果经常需要查入度，可以额外维护一个**逆邻接表（Inverse Adjacency List）**——每个顶点存"所有指向它的边"。或者用"十字链表"（后续内容）同时支持出度和入度查询。

## 总结

```
邻接表设计思路：
  数组（N个头节点）+ 链表（实际存在的边）

                 无向图                    有向图
边存储方式      每条边出现2次             每条边出现1次（出边表）
总边节点数      2E                        E
空间            O(N + 2E)                O(N + E)
求度            链表长度即度              链表长度 = 出度
                                         入度需扫全表 → O(N+E)

适用场景：稀疏图（E << N²）
核心优势：空间按需分配，不浪费
核心代价：查"某两点间有没有边"需 O(degree)，不如邻接矩阵 O(1)
```

> 💡 **决策总结**：
> - 稀疏图 + 遍历为主 → **邻接表**
> - 稠密图 + 频繁查边 → **邻接矩阵**
> - 有向图需要同时快速查出度和入度 → **十字链表**或同时维护正/逆邻接表
