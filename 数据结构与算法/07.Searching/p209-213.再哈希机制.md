1. 这5页PPT（209-213）讲的是哈希表工程实现中非常关键的一个机制：**再哈希（Rehashing）**，也就是**哈希表的“动态扩容”**。

   我之前讲过，对于**封闭哈希（开放寻址，如线性探测、平方探测）**，如果表太满，性能会呈指数级恶化，甚至会陷入死循环导致插入失败。再哈希就是为了解决这个死穴而存在的。

   我为你按照“**为什么触发、怎么解决、实战走查**”的逻辑详细拆解：

   ---

   ### 🟨 第1步：为什么要再哈希？（Slide 209）
   *   **性能雪崩**：PPT明确指出，一旦哈希表**变满（too full）**，搜索时间将急剧增加，并且在使用平方探测时，插入操作可能会直接**失败**（因为找不到空位）。
   *   **隐藏的陷阱**：文中提到 `This can happen if there are too many deletions intermixed with insertions.`（当插入和删除交替频繁发生时，也会触发这个问题）。
   *   *深度解释*：因为封闭哈希的删除必须使用**“懒删除（只打标记，不置空）”**，如果表里有一半都是“已删除”的墓碑标记，即使实际有效的元素不多，但空位被逻辑占满，新元素依然很难找到真正的空位，同样会触发再哈希。

   ### 🟨 第2步：再哈希的“搬家”策略（Slide 210）
   PPT给出了再哈希的标准工程做法，类似搬家：
   1.  **建新房**：创建一个**新表**，大小大约是旧表的**两倍（twice as big）**。
   2.  **换新锁**：因为表的大小变了，所以必须配合使用**全新的哈希函数**（因为取模的基数变了）。
   3.  **逐个搬运**：扫描旧表里的每一个元素，**跳过那些标记为“已删除”的元素**，把活着的元素用新哈希函数重新计算位置，重新插入到新表中。

   *(💡 注：再哈希是一个非常耗时的 `O(n)` 操作，但因为它发生的频率很低（只有当表快满时才会触发），所以平摊下来的时间复杂度依然维持在 `O(1)`。)*

   ### 🟨 第3步：实战推演——从小表扩容到大表（Slide 211 - 213）
   这3张PPT一步步演示了具体发生了什么。

   *   **👀 Slide 211：初始小表状态**
       *   表大小：**7**（这是一个质数，非常规范）。
       *   哈希函数：`h(x) = x mod 7`。采用**线性探测**。
       *   插入序列：`13, 15, 24, 6`。
       *   **冲突发生**：`13` 放在下标6；插入 `6` 时，算出来也是下标6（冲突），通过线性探测往后找，放入下标0。
       *   最终状态：插入了4个元素，表内还有3个空位。

   *   **👀 Slide 212：插入新元素，触发扩容警报**
       *   接下来插入 `23`。`23 % 7 = 2`，下标2是空的，放入成功。
       *   **触发扩容**：此时表里有了 **5** 个元素。表的总大小只有 7。装填因子达到了 `5/7 ≈ 71.4%`，**超过了70%的安全警戒线**。封闭哈希表绝对不能这么满！所以，系统决定立刻执行**再哈希**。

   *   **👀 Slide 213：执行真正的“再哈希”搬家**
       *   选择新大小：旧表大小是7，两倍是14。PPT明确提到：`The size of this table is 17, because this is the first prime which is twice as large as the old table size.`（新表大小选为 **17**，因为17是大于14的第一个**质数**。再次验证了平方探测/双重哈希要求表大小必须为质数的铁律）。
       *   新哈希函数：`h(x) = x mod 17`。
       *   搬家过程：依次拿旧表里的 `6, 15, 23, 24, 13` 这5个元素，用 `% 17` 重新算位置插进去。
       *   *结果*：虽然6、23、24依然有轻微冲突，但因为表从 7 扩张到了 17，空间变得极其充裕，所有元素都能非常轻松地找到自己的坑，哈希表的性能瞬间恢复了巅峰。

   **💡 学完这5页的核心启示：**
   在实际工程中（比如Java的HashMap，或者C++的unordered_map），**“再哈希”是自动触发的**，不需要程序员手动操作。但作为一名合格的程序员，你必须清楚它背后的数学原理（为什么选质数、为什么要在负载因子约0.75时触发），否则当你的大数据存入哈希表时，可能会因为扩容而引发瞬间的CPU卡顿（称为“STW - Stop The World”现象）。
