### 🟨 Slide 225：哈希表线性探测与 ASL 计算（经典大题）
这题是数据结构考试中**高频的“必考画图题”**。
*   **注意题目的陷阱**：哈希函数是 `key % 13`，但**表的大小指定为 19**！这意味着你要把数据散列到大小为 19 的数组里（下标 0~18），13 仅仅是哈希计算用的模数，不是表的大小。
*   **解题步骤指导**：
    1.  **画表**：画一个长度为 19 的表格。依次计算每个键的哈希值。
        *   *举例*：`27 % 13 = 1`，但位置1被`1`占了（冲突）。用**线性探测**往后找空位（位置2被`14`占了，继续往后找，直到找到空位为止，插入）。
    2.  **成功查找的平均查找长度（ASL_success）**：
        *   公式：`(所有元素在表中的探测次数之和) / 元素总数(12)`。
        *   *注意*：探测次数是指从第一次算哈希开始，直到找到该元素为止，一共比较了几次。没冲突的比较1次，探测到第3个坑才找到的就比较3次。
    3.  **不成功查找的平均查找长度（ASL_unsuccess）**：
        *   公式：`(对所有可能的哈希值(0~12)进行查找失败的探测次数之和) / 哈希长度(13)`。
        *   *注意*：所谓“查找失败”，是指从哈希值位置出发，**线性探测到第一个空位时，所经历的步数**。因为这个表有部分位置是空的，只要遇到空位，就可以断定查找失败。
        *   例如：假设哈希值为 0，如果位置 0 被占了，位置 1 没被占，那么探一步就失败了，失败探测次数为 1。把所有 0~12 的失败探测次数加起来除以 13 即可。

### 🟨 Slide 226：单循环有序链表与“浮动指针”查找算法
*   **考点剖析**：这题考察对**单循环链表**的灵活运用。难点在于多了一个浮动的尾指针 `t`（缓存上一次查找的位置，利用**局部性原理**）。
*   **算法逻辑梳理（写代码前的思路）**：
    1.  代码头有 `LinkList &t`（引用传递）。这意味着查找函数可以直接修改链表外部指针 `t` 的位置。
    2.  **第一步（检查缓存）**：将目标值 `kval` 与 `t->key` 比较。如果相等，直接返回成功，记录比较次数为1，结束。
    3.  **第二步（判断方向）**：
        *   如果 `kval < t->key`：说明数据在最小位置 `h` 和当前位置 `t` 之间。**开始指针设为 `h`**。
        *   如果 `kval > t->key`：说明数据在当前位置 `t` 的后面（直到循环到 `h` 之前）。**开始指针设为 `t->next`**。
    4.  **第三步（循环查找）**：顺着单循环链表一直找，直到遇到 `h`（回到起点）。如果找到则返回节点指针并更新 `t`（记录查找位置）；如果没找到则返回 NULL。
*   **避坑提醒**：函数签名里有个 `int &len`，这是用来记录比较次数的，**每次进行 `->key` 比较时，必须执行 `len++`**。

### 🟨 Slide 227：判定二叉树是否为二叉排序树（BST）
*   **考点剖析**：这题的**最优解法**是利用二叉排序树的一个重要特性——**二叉排序树的中序遍历序列一定是严格单调递增的**。
*   **思路指导（伪代码思路）**：
    *   **方法 A（中序遍历+数组）**：写一个中序遍历函数，把遍历到的节点值存入一个数组。如果数组是严格递增的，返回 `true`，否则返回 `false`。
    *   **方法 B（递归边界法，不额外开辟数组，更优）**：写一个递归函数 `bool check(node, min_val, max_val)`。
        *   递归时，判断当前节点值是否在 `(min_val, max_val)` 之间。
        *   向左递归时，更新最大值为当前节点值：`check(node->left, min_val, node->val)`。
        *   向右递归时，更新最小值为当前节点值：`check(node->right, node->val, max_val)`。
        *   只要遇到违反边界的节点，直接返回 `false`。
*   *(因为题目说明了关键词均不同，所以不需要判断等于的情况，用严格 `<` 和 `>` 即可。)*

### 🟨 Slide 228：思考：为什么 512x512 矩阵转置比 513x513 慢？
这题**脱离了单纯的数据结构，触及了计算机体系结构底层的硬件知识**。
*   **核心原理**：**CPU 缓存（Cache）的组相联映射冲突（Cache Thrashing / 抖动）**。
*   **通俗解释**：
    1.  现在的 CPU 都有 **L1/L2 缓存**。内存数据是以“行（Cache Line，通常是 64 字节）”加载到缓存中的。
    2.  缓存里分了很多**“组（Set）”**。如果矩阵的列大小（比如 512）恰好是 `2的幂`，那么在按行或按列读取时，矩阵的每一行首地址在内存中的偏移量，**刚好会映射到缓存里完全相同的“组”中**。
    3.  当程序转置矩阵时，需要频繁读取不同行、不同列的数据。因为这 512 大小的矩阵连续读取时，**会不断把上一次加载到同一个 Cache Set 里的数据“踢出去”（Cache 冲突失效）**，导致 CPU 频繁去内存里拿数据，形成了“缓存颠簸”。
    4.  **为什么 513 就快？** 513 不是 2 的幂。在内存中，不同行的偏移量不再是严格对齐的，因此它们会**均匀地散落**到缓存的不同“组（Set）”里。这样缓存能容下更多正在处理的数据行，不需要频繁去内存里读取，CPU 执行效率自然暴涨。
    5.  *（延伸知识点：TLB 快表也有类似的尴尬情况，所以计算机底层的“填充（Padding）”技巧就是为了避开 2 的幂这种对齐陷阱。）*
