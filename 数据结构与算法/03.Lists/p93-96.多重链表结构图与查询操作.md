## 第一页（P93）—— 多重链表的完整结构图

### 图示解读

PPT展示了一个十字链表结构：

**顶部（列头）：** S1, S2, ..., Sm —— 每个学生一个头节点

**左侧（行头）：** C1, C2, ..., Cn —— 每门课一个头节点

**中间的节点：** 每个节点代表一条"学生 Si 选了课程 Cj"的记录

每个数据节点有**两个指针**：
- **横向指针（→）**：指向同一门课（同一行）的下一个学生
- **纵向指针（↓）**：指向同一个学生（同一列）的下一门课

### 结构示意

```
         S1        S2        ...       Sm
         ↓         ↓                   ↓
C1 → [S1,C1] → [      ] → ... → [Sm,C1] → (回到C1)
         ↓                           ↓
C2 → [S1,C2] → [      ] → ... → [Sm,C2] → (回到C2)
         ↓                           ↓
...      ...                         ...
         ↓                           ↓
Cn → [S1,Cn] → [      ] → ... → [Sm,Cn] → (回到Cn)
         ↓                           ↓
      (回到S1)                    (回到Sm)
```

注意：不是每个交叉点都有节点——只有实际存在的选课关系才有节点。图中只画出了有选课记录的位置。

### 节点结构

每个数据节点包含：

```c
struct MultiNode {
    // 数据（可选，也可以通过位置推断）
    struct MultiNode *nextInRow;    // 同一行（同一门课）的下一个节点
    struct MultiNode *nextInCol;    // 同一列（同一个学生）的下一个节点
};
```

### 头节点数组

- 课程头节点数组：`CourseHead[1..n]`，每个元素是该课程链表的入口
- 学生头节点数组：`StudentHead[1..m]`，每个元素是该学生链表的入口

## 第二页（P94）—— 循环链表 + 如何确定节点属于哪个学生

### 两个关键设计决策

**1. 所有链表都使用头节点，并且是循环的**

- 每门课的链表是循环的：最后一个节点的横向指针指回课程头节点
- 每个学生的链表是循环的：最后一个节点的纵向指针指回学生头节点

循环的好处：
- 遍历时可以判断"回到头节点"作为终止条件
- 从任何节点出发都能遍历整个链表

**2. 两个链表合并为一个结构**

每个节点同时属于两个循环链表：
- 属于某门课的"学生链表"（横向）
- 属于某个学生的"选课链表"（纵向）

### 如何确定一个节点属于哪个学生？

PPT提出了一个问题：当我们沿着课程 Cn 的链表（横向）遍历时，遇到一个节点，怎么知道它属于哪个学生？

**方法：沿着该节点的学生链表（纵向）一直走，直到回到学生头节点。**

具体过程：
1. 从课程 Cn 的头节点出发，沿横向指针走到第一个数据节点
2. 这个节点属于哪个学生？沿着它的纵向指针一直走
3. 最终会回到某个学生头节点（因为是循环链表）
4. 那个学生头节点就告诉你这个节点属于哪个学生

### 示例

```
从 Cn 出发 → 横向走到第一个数据节点 X
从 X 出发 → 纵向走：X → Y → Z → ... → 回到 S1 头节点
结论：节点 X 属于学生 S1
```

### 这种设计的代价

确定一个节点属于哪个学生需要遍历该学生的整个选课链表（纵向走到头节点）。如果学生平均选 3 门课，那么最多走 3 步就能到达头节点。代价很小。

### 替代方案

如果不想每次都遍历来确定归属，可以在每个节点中直接存储学生编号和课程编号：

```c
struct MultiNode {
    int studentID;              // 直接记录属于哪个学生
    int courseID;               // 直接记录属于哪门课
    struct MultiNode *nextInRow;
    struct MultiNode *nextInCol;
};
```

这样用空间换时间，查询归属变成 O(1)，但每个节点多了两个 int 的空间。

## 第三页（P95）—— 遍历课程链表找出所有学生

### 完整遍历过程

PPT描述了如何列出课程 Cn 的所有学生：

**步骤：**

1. 从 Cn 头节点出发，沿横向指针走到第一个数据节点
2. 确定该节点属于哪个学生（沿纵向走到学生头节点）→ 得到 S1
3. 回到 Cn 的链表，继续横向走到下一个数据节点
4. 确定该节点属于哪个学生 → 得到 S3
5. 继续横向走 → 得到 S4
6. 继续横向走 → 得到 S5
7. 横向走回到 Cn 头节点 → 遍历结束

**结果：** 课程 Cn 的学生有 S1, S3, S4, S5

### 反向查询

PPT还指出：用同样的方式，可以确定任何学生选了哪些课程。

**查询学生 Si 选了哪些课：**

1. 从 Si 头节点出发，沿纵向指针走到第一个数据节点
2. 确定该节点属于哪门课（沿横向走到课程头节点）→ 得到 C2
3. 回到 Si 的链表，继续纵向走到下一个数据节点
4. 确定该节点属于哪门课 → 得到 C5
5. 继续纵向走回到 Si 头节点 → 遍历结束

**结果：** 学生 Si 选了 C2, C5

## 第四页（P96）—— 查询操作总结

### 两种基本查询

**查询一：课程 C1 有哪些学生？**

> "To see which students are in course C1, we simply go along the first row of the multilist."

- 从 C1 头节点出发
- 沿横向指针（→）遍历
- 经过的每个节点代表一个选了 C1 的学生
- 回到 C1 头节点时结束

时间复杂度：O(k)，k 是选了 C1 的学生数

**查询二：学生 S1 选了哪些课？**

> "To see which courses student S1 is taking, we simply go down the first column of the multilist."

- 从 S1 头节点出发
- 沿纵向指针（↓）遍历
- 经过的每个节点代表 S1 选的一门课
- 回到 S1 头节点时结束

时间复杂度：O(k)，k 是 S1 选的课程数

### 与二维数组的查询对比

| 查询             | 二维数组          | 多重链表                      |
| ---------------- | ----------------- | ----------------------------- |
| C1 有哪些学生？  | 遍历一行 O(40000) | 遍历 C1 链表 O(选课人数)      |
| S1 选了哪些课？  | 遍历一列 O(2500)  | 遍历 S1 链表 O(选课数) ≈ O(3) |
| S1 是否选了 C1？ | 直接访问 O(1)     | 遍历链表 O(选课数)            |

多重链表在"列举关系"方面远优于二维数组（只访问有效数据），但在"判断特定关系是否存在"方面不如数组的 O(1) 直接访问。

### 伪代码

```c
// 列出课程 C 的所有学生
void ListStudentsInCourse(int courseID) {
    MultiNode *p = CourseHead[courseID]->nextInRow;  // 第一个数据节点
    while (p != CourseHead[courseID]) {              // 没有回到头节点
        printf("Student: %d\n", p->studentID);      // 或者沿纵向找学生头
        p = p->nextInRow;                           // 横向前进
    }
}

// 列出学生 S 选的所有课程
void ListCoursesOfStudent(int studentID) {
    MultiNode *p = StudentHead[studentID]->nextInCol;
    while (p != StudentHead[studentID]) {
        printf("Course: %d\n", p->courseID);
        p = p->nextInCol;                           // 纵向前进
    }
}
```
