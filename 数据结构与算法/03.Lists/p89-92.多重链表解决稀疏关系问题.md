这四页PPT引入了链表的另一个应用：**十字链表 / 多重链表（Multilists）**，用于解决稀疏矩阵类问题。

---

## 第一页（P89）—— 问题引入

### 实际场景

假设一所大学有：
- 40,000 名学生
- 2,500 门课程

需要记录：
- 每个学生选了哪些课
- 每门课有哪些学生

### 为什么不能用简单数组？

最直观的想法是用一个二维数组 `A[40000][2500]`：
- `A[i][j] = 1` 表示第 i 个学生选了第 j 门课
- `A[i][j] = 0` 表示没选

**问题：**
- 数组大小 = 40,000 × 2,500 = **1 亿个元素**
- 每个学生平均只选 3 门课 → 有效数据只有 40,000 × 3 = 120,000 个
- 有效数据占比：120,000 / 100,000,000 = **0.12%**
- 99.88% 的空间被浪费了

这就是典型的**稀疏矩阵（Sparse Matrix）**问题——绝大多数元素为零/空。

---

## 第二页（P90）—— 二维数组的直观表示

### 图示

PPT展示了一个学生-课程的二维表格：

```
        C1   C2   C3   C4   ...
S1    [    ][    ][    ][    ]...
S2    [    ][    ][    ][    ]...
S3    [    ][    ][    ][    ]...
S4    [    ][    ][    ][    ]...
S5    [    ][    ][    ][    ]...
...
```

- 行代表学生（S1, S2, S3, ...）
- 列代表课程（C1, C2, C3, ...）
- 每个格子表示"该学生是否选了该课程"

### 这个表格的问题

对于 40,000 × 2,500 的规模：
- 存储空间巨大（即使每个格子只占 1 字节，也需要 100MB）
- 绝大多数格子是空的
- 查询"某学生选了哪些课"需要遍历该行的 2,500 个格子
- 查询"某门课有哪些学生"需要遍历该列的 40,000 个格子

---

## 第三页（P91）—— 二维数组方案的量化分析

### 具体数据

PPT进一步量化了问题：

- 二维数组有 **1 亿个条目**（40,000 × 2,500 = 100,000,000）
- 平均每个学生选 3 门课
- 有意义的数据只有 40,000 × 3 = **120,000 条**
- 占比约 **0.1%**

### 结论

用二维数组存储这种稀疏关系是极度浪费的。需要一种只存储"有效关系"的数据结构。

### 类比

这和稀疏多项式的问题一样：
- 多项式 `x¹⁰⁰⁰ + 1` 用数组需要 1001 个位置，链表只需 2 个节点
- 学生选课关系用数组需要 1 亿个位置，链表只需 12 万个节点

## 第四页（P92）—— 多重链表的解决方案

### 核心思想

PPT提出需要两种链表：

**1. 每门课一个链表**——包含选了这门课的所有学生

```
C1 → S3 → S7 → S156 → S23001 → NULL
C2 → S1 → S3 → S500 → NULL
...
```

这样查询"C1 有哪些学生"只需遍历 C1 的链表，不需要扫描所有 40,000 个学生。

**2. 每个学生一个链表**——包含该学生选的所有课程

```
S1 → C2 → C5 → C100 → NULL
S2 → C1 → C3 → NULL
...
```

这样查询"S1 选了哪些课"只需遍历 S1 的链表，不需要扫描所有 2,500 门课。

### 多重链表（Multilist）的结构

每个节点同时属于两个链表：
- 属于某个学生的"选课链表"
- 属于某门课的"学生链表"

所以每个节点需要**两个指针**：

```c
struct EnrollNode {
    int studentID;              // 学生编号
    int courseID;               // 课程编号
    struct EnrollNode *nextStudent;  // 同一门课的下一个学生
    struct EnrollNode *nextCourse;   // 同一个学生的下一门课
};
```

### 图示理解

```
课程链表头：  C1    C2    C3
              ↓     ↓     ↓
学生链表头：
S1 →→→→→→→ [S1,C2] →→→→→ [S1,C3] → NULL
              ↓              ↓
S2 →→→→→→→ [S2,C1] → NULL
              ↓
S3 → [S3,C1] →→→→→→ [S3,C2] → NULL
       ↓              ↓
      NULL           NULL
```

每个节点 `[Si, Cj]` 有两个方向的链接：
- 横向（→）：同一个学生的下一门课（nextCourse）
- 纵向（↓）：同一门课的下一个学生（nextStudent）

这就是**十字链表（Orthogonal List / Cross List）**的结构。

### 空间效率

- 只为实际存在的选课关系创建节点
- 总节点数 = 120,000（而不是 1 亿）
- 每个节点大小约 20 字节（2个int + 2个指针）
- 总空间 ≈ 120,000 × 20 = 2.4MB（远小于 100MB）

### 查询效率

| 查询                    | 二维数组               | 多重链表                               |
| ----------------------- | ---------------------- | -------------------------------------- |
| 学生 Si 选了哪些课？    | 遍历一行：O(2500)      | 遍历 Si 的链表：O(该学生选课数) ≈ O(3) |
| 课程 Cj 有哪些学生？    | 遍历一列：O(40000)     | 遍历 Cj 的链表：O(该课程学生数)        |
| 学生 Si 是否选了课 Cj？ | 直接访问 A[i][j]：O(1) | 遍历链表：O(该学生选课数)              |

多重链表在"列出所有关系"方面效率高，但在"判断某个特定关系是否存在"方面不如数组（数组是 O(1) 直接访问）。

## 四页知识脉络总结

```
多重链表（Multilists）（P89-P92）
│
├── 问题引入（P89）
│   ├── 40,000 学生 × 2,500 课程
│   ├── 二维数组需要 1 亿个位置
│   └── 绝大多数为空（稀疏）
│
├── 二维数组的直观表示（P90）
│   └── 行=学生，列=课程，格子=是否选课
│
├── 量化分析（P91）
│   ├── 有效数据仅 120,000 条（0.1%）
│   └── 99.9% 的空间被浪费
│
└── 多重链表解决方案（P92）
    ├── 每门课一个链表（包含选课学生）
    ├── 每个学生一个链表（包含所选课程）
    ├── 每个节点同时属于两个链表（两个指针）
    ├── 空间：只存有效关系，从 100MB 降到 2.4MB
    └── 本质：十字链表（稀疏矩阵的经典存储方式）
```

3. **画出多重链表的结构图**：给定选课关系，画出十字链表
4. **与稀疏矩阵的联系**：多重链表就是稀疏矩阵的十字链表存储方式，广泛用于图的邻接表、数据库索引等场景
