## 第一页（P41）—— 动态性（Dynamic）

### 什么是"动态"？

这里的"动态"指的是数据结构在运行时可以按需调整大小，而不是在编译阶段就把大小写死。

### 数组为什么是"静态"的？

在C语言中，当你声明一个数组：

```c
int arr[100];  // 编译时就确定了：占 100 × 4 = 400 字节
```

这块内存在栈上连续分配，大小不可变。即使你只用了3个元素，剩下97个位置的内存照样被占着。如果你需要101个元素？对不起，越界了，行为未定义（undefined behavior）。

即使用 `malloc` 动态分配数组：

```c
int *arr = (int *)malloc(100 * sizeof(int));
```

虽然是在堆上分配的，但大小一旦确定，想扩容就必须用 `realloc`，而 `realloc` 的本质是：分配一块更大的新内存 → 把旧数据拷贝过去 → 释放旧内存。代价很高，时间复杂度 O(n)。

### 链表为什么是"动态"的？

链表的每个节点是独立分配的：

```c
struct Node {
    int data;
    struct Node *next;
};

// 需要新元素时，随时创建
struct Node *newNode = (struct Node *)malloc(sizeof(struct Node));
newNode->data = 42;
newNode->next = NULL;
```

- 需要一个节点就 `malloc` 一个，不需要就 `free` 掉
- 节点在内存中不需要连续存放（散落在堆的各个位置）
- 永远不会出现"空间不够需要整体搬迁"的情况

### 实际意义

想象你在做一个文本编辑器，用户随时可能输入或删除字符，你根本不知道最终文本有多长。用链表（或链表变体）来管理就很自然，不需要反复 realloc。

## 第二页（P42）—— 插入和删除效率

### 数组的插入/删除为什么慢？

数组在内存中是连续存储的，这意味着元素之间没有"空隙"。

**插入示例：** 在数组 `[1, 2, 3, 4, 5]` 的索引2处插入元素 `99`：

```
原始：[1, 2, 3, 4, 5, _]
                ↓ 先把索引2及之后的元素全部后移一位
步骤1：[1, 2, _, 3, 4, 5]
步骤2：[1, 2, 99, 3, 4, 5]  ← 把99放进去
```

代码实现：

```c
// 在位置 pos 插入 value，n 是当前元素个数
for (int i = n; i > pos; i--) {
    arr[i] = arr[i - 1];  // 逐个后移
}
arr[pos] = value;
n++;
```

最坏情况（在头部插入）：需要移动全部 n 个元素，时间复杂度 O(n)。

**删除示例：** 删除索引2的元素：

```
原始：[1, 2, 99, 3, 4, 5]
步骤：[1, 2, 3, 4, 5, _]  ← 后面的元素全部前移一位
```

同样是 O(n)。

### 链表的插入/删除为什么快？

链表节点通过指针连接，插入/删除只需要改变指针指向：

**在节点B后面插入新节点X：**

```
原始：A → B → C
目标：A → B → X → C

操作：
1. X->next = B->next;   // X 指向 C
2. B->next = X;         // B 指向 X
```

就两条语句，O(1)，不管链表有多长。

**删除节点B后面的节点C：**

```
原始：A → B → C → D
目标：A → B → D

操作：
1. temp = B->next;       // 保存C的地址
2. B->next = C->next;   // B 直接指向 D
3. free(temp);           // 释放C的内存
```

代码示例：

```c
// 在 prev 节点后插入新节点
void insertAfter(struct Node *prev, int data) {
    struct Node *newNode = (struct Node *)malloc(sizeof(struct Node));
    newNode->data = data;
    newNode->next = prev->next;
    prev->next = newNode;
}

// 删除 prev 节点后面的节点
void deleteAfter(struct Node *prev) {
    if (prev->next == NULL) return;
    struct Node *temp = prev->next;
    prev->next = temp->next;
    free(temp);
}
```

### 注意一个前提

链表插入/删除是 O(1) 的前提是：**你已经拿到了操作位置的指针**。如果你需要先"找到"那个位置（比如找第k个节点），那查找本身是 O(n) 的。所以完整来看：

- 数组：定位 O(1) + 搬移 O(n) = O(n)
- 链表：定位 O(n) + 修改指针 O(1) = O(n)

但如果你已经持有目标位置的引用（比如在遍历过程中顺便插入），链表就是纯 O(1) 的优势。

## 第三页（P43）—— 空间（存储）考量

### 链表的空间开销

每个链表节点不仅存数据，还要存一个指针：

```c
struct Node {
    int data;           // 4 字节（假设 int 是 4 字节）
    struct Node *next;  // 8 字节（64位系统上指针是 8 字节）
};
```

所以存一个 `int`，链表实际用了 12 字节（甚至因为内存对齐可能是 16 字节），而数组只用 4 字节。

**空间利用率对比：**

假设存 1000 个 int：
- 数组：1000 × 4 = 4000 字节（纯数据，零浪费）
- 链表：1000 × (4 + 8) = 12000 字节（多了 8000 字节的指针开销）

如果是双向链表（有 `prev` 和 `next` 两个指针），开销更大：1000 × (4 + 8 + 8) = 20000 字节。

### 数组的空间浪费

数组的问题在于你必须预估最大容量：

```c
#define MAX_SIZE 10000
int arr[MAX_SIZE];
int count = 0;  // 实际只用了 50 个
```

实际用了 50 个元素（200 字节），但分配了 10000 个位置（40000 字节），浪费了 99.5% 的空间。

### 什么时候链表更省空间？什么时候数组更省？

设数据元素大小为 D，指针大小为 P，数组预分配大小为 M，实际使用 N 个元素：

- 数组总空间：M × D
- 链表总空间：N × (D + P)

当 N × (D + P) < M × D 时，链表更省空间。

简化一下：当实际使用率 N/M 很低时（数组大量空闲），链表更划算。当数据本身很大（D >> P）时，指针的额外开销可以忽略不计，链表的空间效率也不错。

反过来，如果数据很小（比如存 `char`，1字节）而指针是8字节，链表的空间开销就非常夸张了——每存1字节数据要额外花8字节存指针。

## 第四页（P44）—— 时间考量

### 链表操作语句更多

PPT说"链表的大多数方法需要更多语句"，这是什么意思？

以"访问第 k 个元素"为例：

**数组：**
```c
int value = arr[k];  // 一条语句，直接计算地址：base + k * sizeof(int)
```

**链表：**
```c
struct Node *current = head;
for (int i = 0; i < k; i++) {
    current = current->next;  // 逐个跳转
}
int value = current->data;
```

链表需要一个循环，执行 k 次指针跳转。不仅代码更多，而且每次跳转都可能导致 CPU 缓存未命中（cache miss），因为节点散落在内存各处。

### 数组随机访问快（Finding and altering "in the middle"）

数组支持 O(1) 随机访问，原理是：

```
元素地址 = 数组基地址 + 索引 × 元素大小
```

这是一次简单的算术运算，CPU 一个时钟周期就能完成。而且数组元素在内存中连续存放，CPU 预取（prefetch）机制会把相邻元素提前加载到缓存中，遍历时非常快。

链表要访问中间元素，必须从头节点开始一个一个跳，O(n)。而且每跳一次，下一个节点的地址可能在内存的完全不同位置，缓存命中率很低。

### 链表中间增删快（Additions and removals "in the middle"）

这一点第二页已经详细讲过。核心是：

- 数组在中间插入/删除 → 需要搬移后续所有元素 → O(n)
- 链表在中间插入/删除 → 只改指针 → O(1)

### 综合时间复杂度对比

| 操作                 | 数组                                 | 链表                                      |
| -------------------- | ------------------------------------ | ----------------------------------------- |
| 按索引访问第k个元素  | O(1)                                 | O(n)                                      |
| 在头部插入           | O(n)（全部后移）                     | O(1)                                      |
| 在尾部插入           | O(1)（如果有空间）                   | O(n)（需遍历到尾部）或 O(1)（维护尾指针） |
| 在中间插入（已定位） | O(n)（后移元素）                     | O(1)（改指针）                            |
| 按值查找             | O(n)（无序）/ O(log n)（有序，二分） | O(n)（只能顺序查找）                      |
| 在中间删除（已定位） | O(n)                                 | O(1)                                      |

### 缓存友好性（补充知识）

PPT没有提到但很重要的一点：现代CPU有多级缓存（L1/L2/L3），数组因为内存连续，遍历时缓存命中率极高；链表节点分散在堆中各处，遍历时频繁 cache miss。所以在实际运行中，即使理论复杂度相同，数组的常数因子往往更小。

## 最终总结：如何选择？

**选数组的场景：**

- 数据量已知或变化不大
- 需要频繁随机访问（按下标取值）
- 数据元素很小（指针开销相对太大）
- 需要缓存友好的顺序遍历

**选链表的场景：**
- 数据量不确定，频繁增减
- 需要频繁在头部或中间插入/删除
- 不需要随机访问，主要是顺序遍历
- 实现栈、队列、LRU缓存等结构时

实际工程中，很多语言的标准库会结合两者优点：比如 C++ 的 `std::vector` 是动态数组（自动扩容），Java 的 `ArrayList` 也是。它们在尾部插入是均摊 O(1)，兼顾了数组的随机访问优势和一定的动态性。
