这三页PPT通过经典的**斐波那契数列 (Fibonacci Sequence)**，向我们展示了一个极其重要的算法教训：**看似优雅的数学递归定义，如果直接生搬硬套成代码，可能会导致极其低下的执行效率。**

以下为你整理的详细笔记：

### 第68页：斐波那契数列的起源 (兔子繁殖问题)

这页PPT通过著名的“斐波那契兔子问题”引出了该数列的数学定义。

- **问题模型**：

  - 第 1 个月：1对初生兔子。
  - 第 2 个月：这1对兔子长熟。
  - 第 3 个月：成熟的兔子生下1对小兔子，共2对。
  - 第 4 个月：老兔子又生1对，上个月的小兔子长熟，共3对。

- **数学推导**：

  按照这个规律，每个月的兔子总数序列为：1, 1, 2, 3, 5, 8, 13, 21, 34...

- **核心递推公式**：

  从第三项开始，每一项都等于前两项之和。即 **$F(N) = F(N-1) + F(N-2)$**。

### 第69页：斐波那契的几何可视化 (黄金螺旋)

这页PPT是一张纯视觉图，展示了斐波那契数列在几何上的直观表现。

- **图像解析**：图中的每一个正方形的边长，正好对应斐波那契数列中的数字（1, 1, 2, 3, 5, 8, 13, 21, 34）。将这些正方形拼在一起，并在其中画四分之一圆弧，就能得到一条不断向外扩张的螺旋线。
- **核心意义**：这张图直观地向学生展示了斐波那契数列的**增长速度**。越往后，正方形的面积爆炸式增大。这隐喻了接下来要讲的：计算该数列的时间复杂度也会爆炸式增长。另外，相邻两个斐波那契数的比值会越来越趋近于黄金分割率（约 1.618）。

### 第70页：递归代码实现与时间复杂度灾难 (核心考点)

这页PPT给出了斐波那契数列的直接递归C++代码，并给出了一个令人震惊的时间复杂度结论。

- **代码分析 (`Fib` 函数)**：
  - **基线条件 (Base case)**：`if (N<=1) return 1;`（处理 $F(0)$ 和 $F(1)$）。
  - **递归推进 (Recursive step)**：`return ( Fib(N-1) + Fib(N-2) );`
- **时间复杂度评判：$O((5/3)^N)$**
  - **为什么这么高？** 在上一讲中，计算阶乘 `fact(n)` 也是递归，但复杂度只有 $O(N)$，因为每次递归只调用了一次自己。但在这里，每次计算 `Fib(N)`，都要**分裂**成两次调用（`Fib(N-1)` 和 `Fib(N-2)`）。
  - **递归树的膨胀**：这会形成一棵庞大的二叉树。比如算 $F(5)$，要算 $F(4)$ 和 $F(3)$；算 $F(4)$ 又要算 $F(3)$ 和 $F(2)$... 你会发现 $F(3)$ 被重复计算了多次！数据量 $N$ 稍微大一点，这棵树的节点数就会呈指数级爆炸。
  - **关于底数 5/3**：实际上，严格的数学推导表明，这种纯递归算法的时间复杂度是 $O(\phi^N)$，其中 $\phi$ 是黄金分割率（约等于 1.618）。PPT 中使用的 $5/3$ （约等于 1.667）是为了方便教学记忆而采用的一个粗略的近似上限。在很多教材中，为了简便，也会直接将其放宽写为 **$O(2^N)$**，即典型的**指数级复杂度**。

**总结教训**：这段代码是典型的“反面教材”。它完美契合了数学公式，但因为存在海量的**重复计算 (Overlapping Subproblems)**，导致时间复杂度高达指数级 $O((5/3)^N)$。在实际工程中，求斐波那契数列绝对不能用这种纯递归，而应该用循环（迭代法）或数组记录（动态规划），那样可以把时间复杂度降到极其优秀的 $O(N)$。
