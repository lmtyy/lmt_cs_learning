这三页PPT为你提供了一套非常实用的“看代码算时间复杂度”的四大实战法则 (Rules for Running Time Calculations)。

之前我们学的都是纯数学推导，而这四条法则直接对应了我们日常编程中最常见的四种代码结构。熟练掌握它们，你就能一眼看出一段代码的复杂度。

以下为你整理的详细笔记：

### 运行时间计算法则 (Running Time Calculations)

#### 法则 1：单层 For 循环 (Rule 1 - For loops)

- **代码特征**：只有一层单纯的循环，例如 `for (i=0; i<N; i++) k++;`
- **计算规则**：循环的运行时间 = **循环内部语句的运行时间 $\times$ 循环次数**。
- **复杂度分析**：在这个例子中，内部语句 `k++` 执行一次的时间是常数级别 O(1)。外层循环执行了 N 次，所以总时间复杂度是 **O(N)**。

#### 法则 2：嵌套 For 循环 (Rule 2 - Nested for loops)

- **代码特征**：循环套着循环，例如外层循环 `i` 跑 N 次，内层循环 `j` 跑 N 次。
- **计算规则**：从里向外分析。内部语句的总运行时间 = **每一层循环大小的乘积**。
- **复杂度分析**：内层循环自己跑完需要 O(N) 的时间。而外层循环每走一步，内层循环都要完整跑一圈。所以总执行次数是 N * N，时间复杂度为 **O(N²)**。这就是前面提到的“乘法法则”在代码中的具体体现。

#### 法则 3：顺序执行的语句 (Rule 3 - Consecutive Statements)

- **代码特征**：几个不同的代码块按顺序挨个执行。PPT 中给的例子是：先执行了一个单层循环 O(N)，紧接着下面又执行了一个嵌套循环 O(N²)。
- **计算规则**：求和并保留最大项。总时间 = 各个代码块时间的总和。但是在计算大 O 符号时，我们只保留最高阶的项。
- **复杂度分析**：第一部分是 O(N)，第二部分是 O(N²)。总时间是 O(N) + O(N²)。根据之前学的加法法则，取最大值，所以这段代码的最终整体复杂度依然是 **O(N²)**。这告诉我们在顺序执行的代码中，**性能瓶颈是由最慢的那段代码决定的**。

#### 法则 4：条件分支语句 If-Else (Rule 4 - Condition Statement)

- **代码特征**：包含逻辑判断的代码，即 `if (condition) S1 else S2`。
- **计算规则**：总时间 = **判断条件的时间 + 分支中耗时较长的那个分支的时间**。
  - 用公式表示：`Time(test) + max(Time(S1), Time(S2))`
- **复杂度分析**：因为在做复杂度分析时，我们通常评估的是**最坏情况 (Worst-case)**。既然不知道代码运行时会走哪个分支，我们就假设它永远走最慢、花时间最长的那个分支。
- **结合冒泡排序例子的分析 (绿框部分)**：
  - `if` 判断条件 `Element[j -1] > Element[j]`：这是数组读取和大小比较，耗时 O(1)。
  - `if` 内部的分支（交换变量）：耗时 O(1)。
  - `else` 分支（图中没写出来，相当于什么都不做）：耗时 O(0)。
  - 所以整个 `if` 语句块的时间是：O(1) + max(O(1), O(0)) = **O(1)**。这意味着不论条件是否成立，这个 `if` 结构在单次执行时的耗时都是常数级别的。外层再套上循环，就能算出整体复杂度了。
